# 【拼多多内容】负载均衡/隔离/压测怎么做？

> JD 依据："系统高可用/扩展性"、"稳定性建设"。

## 一、多层负载均衡

```
DNS（机房入口）
  ↓
CDN/LVS（入口流量分发）
  ↓
Spring Cloud Gateway（业务网关）
  ↓
服务（Nacos+LoadBalancer 内部 RPC）
  ↓
数据（Redis Cluster/MySQL 读写分离/分片）
```

**算法**：
| 算法 | 原理 | 适用 |
|------|------|------|
| 轮询 | 依次分配 | 实例同质 |
| 加权轮询 | 按权重 | 配置不同 |
| 最少连接 | 给连接最少的 | 长连接 |
| 随机 | 随机 | 简单 |
| 一致性 hash | hash 环 | 有状态（缓存） |
| IP hash | 同 IP 同实例 | 会话保持 |

**一致性 hash**：
```
节点 hash → 环上
key hash → 顺时针找最近节点
增减节点只影响相邻段（迁移最少）
带虚拟节点（解决倾斜）
```

## 二、多层隔离

**1. 线程池隔离**（计算密集）：
```java
// 评价/直播/Feed 各自线程池
@Bean("reviewExecutor") public ThreadPoolExecutor review() { ... }
@Bean("liveExecutor") public ThreadPoolExecutor live() { ... }
// 评价慢不会拖累直播
```

**2. 信号量隔离**（IO 密集，轻量）：
```java
@SentinelResource(value = "searchES", blockHandler = "...")
// Sentinel 信号量限流（不创建线程）
```

**3. 集群/机房隔离**：
- 多机房部署（同城双活/异地多活）
- 单机房故障切流

**4. 业务隔离**：
- 评价/直播/Feed 独立服务+独立库
- 核心/非核心部署在不同集群

**5. 租户隔离**（中台）：
- 按业务线分库/表 或 行级
- 限流/监控按租户

**6. 读写隔离**：
- 主写从读
- CQRS（命令与查询分离）

## 三、全链路压测

**压测分级**：
- 基准压测：单接口（JMeter/wrk）
- 集群压测：整个服务集群
- 全链路压测：业务全链路（生产环境影子）

**全链路压测方案**：
```
1. 影子库表（压测数据隔离）
   review（生产） + review_pressure（压测）
   
2. 压测标识透传
   请求头 X-Pressure-Test: true
   → 全链路透传到 DB/Redis/MQ
   
3. 影子流量路由
   网关识别压测标识 → 路由到影子库表
   Kafka 压测 topic 单独消费
   
4. mock 外部依赖
   支付/短信等第三方接口 mock
```

**压测流程**：
```
1. 容量规划（目标 QPS/RT）
2. 数据准备（仿真实数据量）
3. 压测执行（梯度加压：10% → 50% → 100% → 120%）
4. 监控观察（CPU/内存/RT/错误率）
5. 瓶颈定位+优化
6. 容量公式（单机 QPS × 机器数 = 集群 QPS）
```

## 四、内容场景实战

**1. 直播开播压测**：
```
场景：热门主播开播，瞬时 100w 观众涌入
压测点：
  - 信令网关连接数（50w/机）
  - CDN 带宽（Gbps 级）
  - 弹幕 QPS（百万级）
  - 房间订阅树性能
目标：
  - 起播时间 <2s
  - 弹幕延迟 <1s
  - 卡顿率 <5%
```

**2. 评价查询压测**：
```
场景：双 11 商品页访问峰值
压测：
  - 评价列表查询（热点商品）
  - 多级缓存命中率
  - DB 兜底 QPS
目标：
  - RT <100ms（P99）
  - DB QPS <5k
```

**3. 评价/直播业务隔离**：
```
评价服务（独立线程池/独立库）→ 评价慢不影响直播
直播信令（独立网关）→ 不受评价流量影响
Feed 写扩散（独立池）→ 不影响实时查询
```

## 五、压测工具

- **wrk/JMeter**：单接口压测
- **TcpCopy**：复制线上流量
- **影子Master**：全链路压测平台（阿里）
- **JMeter+InfluxDB+Grafana**：压测监控

## 六、容量规划

```
日常 QPS = 1w
峰值 QPS = 10w（10 倍日常）
单机 QPS = 1k
机器数 = 峰值/单机 × 冗余系数(1.5) = 150 台

公式：
  N = (PeakQPS × SafetyMargin) / SingleMachineQPS
```

## 七、底层本质

负载均衡+隔离+压测本质是**"用均匀分发+故障隔离+容量验证构建高可用"**——负载均衡分散压力、隔离防蔓延、压测提前发现瓶颈，三者构成容量与稳定的基础。

## 常见考点
1. **一致性 hash 怎么解决倾斜**？——虚拟节点（每节点映射多个虚拟点）。
2. **线程池 vs 信号量隔离**？——线程池独立（强但有开销），信号量轻量（共享线程，限流）。
3. **全链路压测怎么不污染生产**？——影子库表+压测标识透传+独立消费组。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：负载均衡你用一致性 hash 分配请求（基于 uid），但评价服务是无状态的。为什么无状态服务要用一致性 hash？轮询不是更均匀吗？**

无状态服务确实用轮询最简单均匀。一致性 hash 用在"有状态或希望亲和"的场景。评价服务虽然逻辑无状态，但有几处"隐性状态"：1. 本地缓存（Caffeine）——同一 uid 的请求落同一实例，本地缓存命中率高，否则每个实例都要缓存该 uid 的评价。2. DB 连接池——如果按 uid 分片，同一 uid 总是打同一 DB 分片，连接复用好。但一致性 hash 的代价是"节点变动时流量重新分布"，扩容/缩容时大量请求换实例，本地缓存全失效（雪崩）。评价服务 QPS 高且缓存敏感，用一致性 hash（+虚拟节点）让同一 uid 尽量稳定在同一实例，缓存命中。如果缓存不敏感（如纯计算），用轮询更简单。本质：一致性 hash 是"用亲和换缓存效率"，轮询是"用均匀换简单"。

### 第二层：证据与定位

**Q：线上某台评价服务实例 CPU 100%，但其他实例只有 30%。你怎么定位是负载不均还是单实例有问题？**

单实例高负载的两类原因：
1. 负载不均——看 Nginx/网关的 `upstream request distribution`，如果某实例分配的请求数是其他的 3 倍，是负载均衡问题。一致性 hash 的"倾斜"（某 hash 段特别热）或 IP hash（某代理 IP 流量大）会导致。
2. 单实例问题——如果请求数均匀但 CPU 不均，是该实例自身问题。看 `arthas dashboard` 的线程 CPU，找耗 CPU 的线程（如某个 Full GC 线程、或某个慢请求卡住）。常见根因：该实例的 JVM 参数配错（堆太小频繁 GC）、或该实例跑了定时任务（如凌晨校准）、或该实例连的 DB 是慢节点（分片不均）。
3. 排查——`jstat -gcutil` 看 GC 频率，`top -Hp <pid>` 看线程，对比其他实例。

### 第三层：根因深挖

**Q：全链路压测时，评价查询的 P99 飙到 2s（正常 100ms），但单服务压测没问题。根因是什么？**

单服务 OK 但全链路炸，根因在"依赖的相互影响"：
1. 下游瓶颈——评价查询依赖 Redis（缓存）+ ES（搜索）+ MySQL（兜底）。单服务压测可能没打到下游瓶颈（Redis 缓存命中），全链路压测的流量模式（真实分布）让更多请求回源 ES/MySQL，ES 被打满（CPU 100%），查询慢。
2. 连接池耗尽——全链路压测 QPS 高，评价服务到 Redis/ES 的连接池打满，请求排队等连接，P99 涨。单服务压测 QPS 低没暴露。
3. 跨服务调用放大——评价查询调商品服务（Feign），商品服务又调库存服务，链路深。全链路压测时每层都有延迟，叠加放大（A 100ms + B 200ms + C 300ms = 600ms）。
4. 竞争资源——多个服务部署在同一 K8s 节点，CPU/内存争抢。全链路压测所有服务同时高负载，节点资源不够。
解法：全链路压测时分层监控，找出第一个"瓶颈服务"（通常是最先超时的），优化它，再压测找下一个。

### 第四层：方案权衡

**Q：你用线程池隔离评价和直播，但有人建议用信号量隔离（Hystrix/Sentinel 的信号量模式）更轻量。两者怎么选？**

线程池 vs 信号量隔离的权衡：
1. **线程池隔离**——独立线程池，强隔离（评价池满不影响直播），但开销大（每池几十个线程，线程切换成本）。适合"调用下游耗时不可控"的场景（如调外部 API）。
2. **信号量隔离**——共享主线程池，用信号量限并发（不创建线程），轻量但弱隔离（评价慢还是占用主线程）。适合"调用快且可控"的场景（如查 Redis/本地缓存）。
拼多多评价调下游（ES/DB）耗时不可控（可能慢查询），用线程池隔离；直播弹幕查 Redis（在线用户）耗时可控且快，用信号量隔离。混合策略：按"下游耗时 + 隔离需求"选，不是一刀切。核心判断：如果下游可能拖垮，必须线程池（强隔离）；如果下游快且稳定，信号量够（省资源）。

### 第五层：验证与沉淀

**Q：全链路压测怎么保证"不污染生产数据"（压测产生的评价不混入真实评价）？**

影子库表 + 标识透传是标准方案，但容易有漏洞：
1. 影子库表——压测数据的 review 表是 `review_pressure`，但代码里如果硬编码了表名（如 `reviewDao.findAll()` 没走分表路由），压测数据写进了 `review`（污染）。验证：压测后 `SELECT COUNT(*) FROM review WHERE tenant='pressure'` 应为 0。
2. 压测标识透传——请求头 `X-Pressure-Test: true` 必须全链路透传（网关→服务→Kafka→消费者）。如果某服务漏了透传（如 Kafka 消费者没读 header），压测事件被当真实事件处理。验证：压测前在各服务打日志确认标识透传。
3. 缓存隔离——压测的 Redis key 必须加前缀（`pressure:review:{id}`），否则覆盖真实缓存。验证：压测后 `redis-cli KEYS pressure:*` 应有压测数据，`review:*` 无异常增长。
4. 第三方 mock——支付/短信等外部接口不能真调，用 mock 服务。验证 mock 的返回符合契约。
沉淀：压测平台自动校验"压测标识全链路透传"（扫描各服务的 header 传递）；压测后自动巡检"生产数据无污染"（对比压测前后的 count）；影子库表定期清理。

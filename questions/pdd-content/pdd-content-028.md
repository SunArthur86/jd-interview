# 【拼多多内容】负载均衡/隔离/压测怎么做？

> JD 依据："系统高可用/扩展性"、"稳定性建设"。

## 一、多层负载均衡

```
DNS（机房入口）
  ↓
CDN/LVS（入口流量分发）
  ↓
Spring Cloud Gateway（业务网关）
  ↓
服务（Nacos+LoadBalancer 内部 RPC）
  ↓
数据（Redis Cluster/MySQL 读写分离/分片）
```

**算法**：
| 算法 | 原理 | 适用 |
|------|------|------|
| 轮询 | 依次分配 | 实例同质 |
| 加权轮询 | 按权重 | 配置不同 |
| 最少连接 | 给连接最少的 | 长连接 |
| 随机 | 随机 | 简单 |
| 一致性 hash | hash 环 | 有状态（缓存） |
| IP hash | 同 IP 同实例 | 会话保持 |

**一致性 hash**：
```
节点 hash → 环上
key hash → 顺时针找最近节点
增减节点只影响相邻段（迁移最少）
带虚拟节点（解决倾斜）
```

## 二、多层隔离

**1. 线程池隔离**（计算密集）：
```java
// 评价/直播/Feed 各自线程池
@Bean("reviewExecutor") public ThreadPoolExecutor review() { ... }
@Bean("liveExecutor") public ThreadPoolExecutor live() { ... }
// 评价慢不会拖累直播
```

**2. 信号量隔离**（IO 密集，轻量）：
```java
@SentinelResource(value = "searchES", blockHandler = "...")
// Sentinel 信号量限流（不创建线程）
```

**3. 集群/机房隔离**：
- 多机房部署（同城双活/异地多活）
- 单机房故障切流

**4. 业务隔离**：
- 评价/直播/Feed 独立服务+独立库
- 核心/非核心部署在不同集群

**5. 租户隔离**（中台）：
- 按业务线分库/表 或 行级
- 限流/监控按租户

**6. 读写隔离**：
- 主写从读
- CQRS（命令与查询分离）

## 三、全链路压测

**压测分级**：
- 基准压测：单接口（JMeter/wrk）
- 集群压测：整个服务集群
- 全链路压测：业务全链路（生产环境影子）

**全链路压测方案**：
```
1. 影子库表（压测数据隔离）
   review（生产） + review_pressure（压测）
   
2. 压测标识透传
   请求头 X-Pressure-Test: true
   → 全链路透传到 DB/Redis/MQ
   
3. 影子流量路由
   网关识别压测标识 → 路由到影子库表
   Kafka 压测 topic 单独消费
   
4. mock 外部依赖
   支付/短信等第三方接口 mock
```

**压测流程**：
```
1. 容量规划（目标 QPS/RT）
2. 数据准备（仿真实数据量）
3. 压测执行（梯度加压：10% → 50% → 100% → 120%）
4. 监控观察（CPU/内存/RT/错误率）
5. 瓶颈定位+优化
6. 容量公式（单机 QPS × 机器数 = 集群 QPS）
```

## 四、内容场景实战

**1. 直播开播压测**：
```
场景：热门主播开播，瞬时 100w 观众涌入
压测点：
  - 信令网关连接数（50w/机）
  - CDN 带宽（Gbps 级）
  - 弹幕 QPS（百万级）
  - 房间订阅树性能
目标：
  - 起播时间 <2s
  - 弹幕延迟 <1s
  - 卡顿率 <5%
```

**2. 评价查询压测**：
```
场景：双 11 商品页访问峰值
压测：
  - 评价列表查询（热点商品）
  - 多级缓存命中率
  - DB 兜底 QPS
目标：
  - RT <100ms（P99）
  - DB QPS <5k
```

**3. 评价/直播业务隔离**：
```
评价服务（独立线程池/独立库）→ 评价慢不影响直播
直播信令（独立网关）→ 不受评价流量影响
Feed 写扩散（独立池）→ 不影响实时查询
```

## 五、压测工具

- **wrk/JMeter**：单接口压测
- **TcpCopy**：复制线上流量
- **影子Master**：全链路压测平台（阿里）
- **JMeter+InfluxDB+Grafana**：压测监控

## 六、容量规划

```
日常 QPS = 1w
峰值 QPS = 10w（10 倍日常）
单机 QPS = 1k
机器数 = 峰值/单机 × 冗余系数(1.5) = 150 台

公式：
  N = (PeakQPS × SafetyMargin) / SingleMachineQPS
```

## 七、底层本质

负载均衡+隔离+压测本质是**"用均匀分发+故障隔离+容量验证构建高可用"**——负载均衡分散压力、隔离防蔓延、压测提前发现瓶颈，三者构成容量与稳定的基础。

## 常见考点
1. **一致性 hash 怎么解决倾斜**？——虚拟节点（每节点映射多个虚拟点）。
2. **线程池 vs 信号量隔离**？——线程池独立（强但有开销），信号量轻量（共享线程，限流）。
3. **全链路压测怎么不污染生产**？——影子库表+压测标识透传+独立消费组。

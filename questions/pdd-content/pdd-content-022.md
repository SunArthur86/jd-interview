# 【拼多多内容】直播系统整体架构怎么设计？

> JD 依据："直播短视频"、"高并发大流量大数据量"、"系统高可用/扩展性"。

## 一、整体架构

```
                 ┌─────────── 主播端 ──────────┐
                 │  推流 SDK (RTMP/SRT)         │
                 └─────────────┬───────────────┘
                               ▼
                 ┌─────────── 直播中心 ────────┐
                 │  接收 → 转码（多档位）→ 录制 │
                 └──────┬───────────────┬──────┘
                        │               │
                  HTTP-FLV/HLS          信令
                        │               │
                        ▼               ▼
                 ┌───────────┐   ┌──────────────┐
                 │   CDN     │   │ 信令网关      │
                 │  边缘节点  │   │ (Netty+WS)   │
                 └─────┬─────┘   └──────┬───────┘
                       │                │
                       ▼                ▼
                 ┌───────────┐   ┌──────────────┐
                 │  观众端    │   │ 业务服务     │
                 │  播放器    │   │ 弹幕/礼物/审核│
                 └───────────┘   └──────────────┘
```

## 二、流链路（音视频）

**1. 推流**：
- 主播端 SDK 按 RTMP/SRT/WebRTC 推到直播中心
- 弱网用 SRT/WebRTC（自动码率调整）

**2. 转码**：
```
源流（1080p 4Mbps）
  → 1080p（高清）/720p（标清）/480p（流畅）/360p（省流）
  → 编码 H.264/H.265/AV1
  → 协议：HTTP-FLV（流式）/HLS（切片 ts 3-10s）/WebRTC（超低延迟）
```

**3. CDN 分发**：
- 观众就近访问边缘节点
- 命中率 >95%（命中率低说明冷流，可考虑 P2P）

**4. 录制**：
- 转码旁路录制为 ts 切片 → 合成 MP4 → OSS
- 回放走点播（VOD）

## 三、信令链路（弹幕/礼物/连麦）

**长连接网关（Netty）**：
```
观众 ──WebSocket── 网关（Boss+Worker Reactor）
                    ↓
                 Kafka（按 liveId 分区）
                    ↓
              消费者（限速+聚合）
                    ↓
              网关下发（广播到房间所有连接）
```

**单房间订阅**：
```java
// 房间 → 在线用户 Set（Redis）
Set<Long> online = redis.opsForSet().members("live:online:" + liveId);
// 推送：网关按 connectionId 批量下发
```

## 四、弹幕削峰（关键）

```
三层削峰：
1. 服务端：
   - Kafka 削峰（弹幕先进 MQ）
   - 消费者限速（每房每秒 N 条）
   - 同类聚合（"哈哈哈"合并）
   - 优先级（付费/大 V 优先）

2. 网关：
   - 单连接限流（用户 N/s）
   - 房间级限流（每房 M/s 下发）

3. 客户端：
   - 节流渲染（按帧合并）
   - 队列丢弃（旧弹幕丢）
```

## 五、连麦（超低延迟）

```
主播 A ←─WebRTC─→ 媒体服务器（SFU）←─WebRTC─→ 主播 B
                       ↓
                    混流 → CDN（普通观众仍看 HLS）
```

WebRTC 走 UDP + SRTP，延迟 <500ms。

## 六、内容审核（直播实时）

```
视频流 ──抽帧（每秒 1 帧）── 视觉模型（涉黄/暴恐/广告）
音频流 ──ASR 转文字 ── NLP 模型（涉政/辱骂）
       ↓
命中 → 强下线/警告/人工复审
```

**关键**：
- 抽帧+模型推理要快（<1s，否则直播都过了）
- 超时默认通过（先播后审，错了再下架）
- 严重违规（涉政/未成年）实时阻断

## 七、礼物与计费

```
观众送礼 → 礼物服务（事务：扣虚拟币+记录流水+主播加收益）
       ↓ Kafka
   实时大屏（Flink 聚合）
   主播收益结算（T+1）
```

**强一致**：扣币+流水同事务（资金相关）。

## 八、容量与扩展

```
- 流量：CDN 带宽按需扩（自动）
- 信令网关：单机 50w 连接（Linux 调优），K8s 弹性扩
- Kafka：按 liveId 分区（保证同房间顺序）
- 转码：GPU 集群，按需弹性（按直播数）
- MySQL：按 live_id 分片
```

## 九、稳定性与监控

```
质量监控：
  - 推流码率/帧率
  - 转码耗时/失败率
  - CDN 命中率/带宽
  - 端到端延迟
  - 观众卡顿率/起播时间

业务监控：
  - 同时在线（Redis）
  - 弹幕量/礼物量（Kafka 流量）
  - 审核命中/误判
  - 主播活跃数

预案：
  - CDN 故障 → 切换备用厂商
  - 信令网关过载 → 限流+降级（弹幕显示拥挤提示）
  - 转码故障 → 降级单档位
```

## 十、底层本质

直播架构本质是**"流（音视频低延迟分发）+ 信令（实时互动削峰）+ 审核（实时合规）"**——流走 CDN 优化带宽，信令走长连接+MQ 优化并发，审核走抽帧+模型保证合规。

## 常见考点
1. **怎么扛百万并发弹幕**？——Netty 网关+Kafka 削峰+多级限流+客户端节流。
2. **直播延迟怎么降到 1s 内**？——WebRTC 替代 RTMP/HLS。
3. **直播录制和回放怎么做**？——旁路录制 ts→MP4→OSS，回放走点播 CDN。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：直播流走 CDN 分发，信令（弹幕/礼物）走自建 Netty 网关，为什么不都走 CDN？CDN 不是更能扛吗？**

CDN 擅长"一对多分发静态/流媒体"——一路推流，百万观众从最近的边缘节点拉，CDN 边缘缓存命中率高，省带宽。但 CDN 不适合"实时双向信令"——弹幕是观众产生的（多对一上行）+ 要广播给房间所有人（一对多下行），且要求毫秒级延迟。CDN 是"拉"模型（观众主动拉流），不支持服务端主动 push；CDN 的回源延迟（边缘→源站）对弹幕的实时性不够。自建 Netty 网关用 WebSocket 长连接，服务端可主动 push 弹幕，且按房间路由（同房间的连接在同一网关实例，广播本地完成）。本质：CDN 优化"分发效率"，自建网关优化"实时互动"，两者职责正交。

### 第二层：证据与定位

**Q：直播卡顿率从 3% 飙到 20%，你怎么定位是 CDN、转码、还是推流端的问题？**

沿"主播推流→转码→CDN→观众播放"链路分段查：
1. 推流端——主播 SDK 上报的推流码率/帧率，如果码率掉（主播网络上行差），源流就卡，所有人卡。查推流服务的 `publish_bitrate`。
2. 转码——转码服务的 `transcode_lag` 和 `output_frame_rate`，如果转码积压（GPU 不够），切片产出延迟，观众 buffer 空了拉不到新片。
3. CDN——`CDN 命中率` 从 95% 掉到 70%，大量回源，边缘节点拉源站慢。查 CDN 日志的 `cache_status`（HIT/MISS）。
4. 观众端——客户端上报的 `buffer_health`（播放器缓冲区健康度），如果大量用户 buffer <1s，是拉流跟不上。按 `network_type`（WiFi/4G/5G）分组看是否特定网络差。

### 第三层：根因深挖

**Q：转码服务在直播高峰（几千路同时直播）时 GPU 利用率打满，转码延迟从 1s 涨到 5s，观众端卡顿。根因和解法是什么？**

根因是 GPU 资源不足 + 转码任务调度不均。深挖：
1. 资源不足——每路直播要转 4 档（1080p/720p/480p/360p），几千路就是几万路转码任务，GPU 显存和算力打满。扩 GPU 集群成本高（一块 A10 几万块）。
2. 调度不均——某些 GPU 卡上挤了大主播（源流码率高，转码耗时翻倍），其他 GPU 空闲。
解法：
1. 动态档位——不是所有直播都转 4 档，冷门直播只转 720p+480p（够用），省 50% GPU。热门直播才转全档。
2. 硬件编码——从 CPU 软编（x264）切到 GPU 硬编（NVENC），单卡能转的路数翻 3-5 倍。
3. 弹性 GPU——用云厂商的 GPU Spot 实例（便宜），高峰拉起，低谷释放。
4. 转码降级——GPU 打满时，新直播降级为"只转源流不转多档"，观众端自适应码率（ABR）。

### 第四层：方案权衡

**Q：礼物扣费你用强一致（事务：扣虚拟币+流水+主播收益），但这会让送礼链路慢。为什么不异步（发 Kafka，下游扣）？**

送礼是资金场景，必须强一致。异步扣（发 Kafka）的风险：消息丢了或消费失败，用户送了礼但没扣币（资损），或扣了币但主播没加收益（客诉）。权衡：
1. 事务边界——扣币 + 流水 + 主播收益加在同一事务（MySQL 本地事务），保证原子性。延迟在 DB 事务（通常 <50ms），可接受。
2. 收益结算异步——主播收益"实时加"改"T+1 结算"——送礼时只记流水（事务内），主播收益凌晨批量算。这样事务只含扣币+流水（2 张表），更快。
3. 补偿兜底——即使事务，也要对账（每天对比"扣币总额 = 流水总额 = 主播收益总额"），发现不一致人工介入。
异步只适合"可容忍丢失/重试"的非资金场景（如弹幕计数、通知），资金必须强一致。

### 第五层：验证与沉淀

**Q：直播架构上线前，你怎么验证百万并发弹幕和转码的容量？**

高并发场景的容量验证靠全链路压测 + 仿真：
1. 信令网关压测——用 Netty 客户端模拟 50 万 WebSocket 长连接/机，每连接每秒发 1 条弹幕，验证网关 CPU/内存/连接数稳定，弹幕推送 P99 <200ms。
2. 转码压测——模拟 1000 路直播同时推流（用 FFmpeg 推测试流），验证转码集群的 GPU 利用率、转码延迟、输出质量。
3. CDN 压测——和 CDN 厂商联合压测，验证带宽峰值（Gbps 级）下的命中率和回源延迟。
4. 真实流量回放——TcpCopy 复制线上弹幕流量到预发环境，验证在真实流量分布下的表现。
沉淀：直播开播前 30 分钟自动扩容（HPA + CDN 带宽预留）；容量公式（单机连接数 × 机器数 = 集群连接数）写在容量规划文档；每月一次全链路压测（影子环境），发现瓶颈提前优化。

## 结构化回答




**30 秒电梯演讲：** 直播架构像电视台+电话总机——电视台负责节目分发（音视频流），总机负责观众互动（弹幕/礼物/连麦）。

**展开框架：**
1. **流链路** — 推流→转码→CDN→播放
2. **信令链路** — 长连接→Kafka→广播
3. **转码** — 多档位+录制

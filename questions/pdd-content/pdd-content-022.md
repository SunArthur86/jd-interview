# 【拼多多内容】直播系统整体架构怎么设计？

> JD 依据："直播短视频"、"高并发大流量大数据量"、"系统高可用/扩展性"。

## 一、整体架构

```
                 ┌─────────── 主播端 ──────────┐
                 │  推流 SDK (RTMP/SRT)         │
                 └─────────────┬───────────────┘
                               ▼
                 ┌─────────── 直播中心 ────────┐
                 │  接收 → 转码（多档位）→ 录制 │
                 └──────┬───────────────┬──────┘
                        │               │
                  HTTP-FLV/HLS          信令
                        │               │
                        ▼               ▼
                 ┌───────────┐   ┌──────────────┐
                 │   CDN     │   │ 信令网关      │
                 │  边缘节点  │   │ (Netty+WS)   │
                 └─────┬─────┘   └──────┬───────┘
                       │                │
                       ▼                ▼
                 ┌───────────┐   ┌──────────────┐
                 │  观众端    │   │ 业务服务     │
                 │  播放器    │   │ 弹幕/礼物/审核│
                 └───────────┘   └──────────────┘
```

## 二、流链路（音视频）

**1. 推流**：
- 主播端 SDK 按 RTMP/SRT/WebRTC 推到直播中心
- 弱网用 SRT/WebRTC（自动码率调整）

**2. 转码**：
```
源流（1080p 4Mbps）
  → 1080p（高清）/720p（标清）/480p（流畅）/360p（省流）
  → 编码 H.264/H.265/AV1
  → 协议：HTTP-FLV（流式）/HLS（切片 ts 3-10s）/WebRTC（超低延迟）
```

**3. CDN 分发**：
- 观众就近访问边缘节点
- 命中率 >95%（命中率低说明冷流，可考虑 P2P）

**4. 录制**：
- 转码旁路录制为 ts 切片 → 合成 MP4 → OSS
- 回放走点播（VOD）

## 三、信令链路（弹幕/礼物/连麦）

**长连接网关（Netty）**：
```
观众 ──WebSocket── 网关（Boss+Worker Reactor）
                    ↓
                 Kafka（按 liveId 分区）
                    ↓
              消费者（限速+聚合）
                    ↓
              网关下发（广播到房间所有连接）
```

**单房间订阅**：
```java
// 房间 → 在线用户 Set（Redis）
Set<Long> online = redis.opsForSet().members("live:online:" + liveId);
// 推送：网关按 connectionId 批量下发
```

## 四、弹幕削峰（关键）

```
三层削峰：
1. 服务端：
   - Kafka 削峰（弹幕先进 MQ）
   - 消费者限速（每房每秒 N 条）
   - 同类聚合（"哈哈哈"合并）
   - 优先级（付费/大 V 优先）

2. 网关：
   - 单连接限流（用户 N/s）
   - 房间级限流（每房 M/s 下发）

3. 客户端：
   - 节流渲染（按帧合并）
   - 队列丢弃（旧弹幕丢）
```

## 五、连麦（超低延迟）

```
主播 A ←─WebRTC─→ 媒体服务器（SFU）←─WebRTC─→ 主播 B
                       ↓
                    混流 → CDN（普通观众仍看 HLS）
```

WebRTC 走 UDP + SRTP，延迟 <500ms。

## 六、内容审核（直播实时）

```
视频流 ──抽帧（每秒 1 帧）── 视觉模型（涉黄/暴恐/广告）
音频流 ──ASR 转文字 ── NLP 模型（涉政/辱骂）
       ↓
命中 → 强下线/警告/人工复审
```

**关键**：
- 抽帧+模型推理要快（<1s，否则直播都过了）
- 超时默认通过（先播后审，错了再下架）
- 严重违规（涉政/未成年）实时阻断

## 七、礼物与计费

```
观众送礼 → 礼物服务（事务：扣虚拟币+记录流水+主播加收益）
       ↓ Kafka
   实时大屏（Flink 聚合）
   主播收益结算（T+1）
```

**强一致**：扣币+流水同事务（资金相关）。

## 八、容量与扩展

```
- 流量：CDN 带宽按需扩（自动）
- 信令网关：单机 50w 连接（Linux 调优），K8s 弹性扩
- Kafka：按 liveId 分区（保证同房间顺序）
- 转码：GPU 集群，按需弹性（按直播数）
- MySQL：按 live_id 分片
```

## 九、稳定性与监控

```
质量监控：
  - 推流码率/帧率
  - 转码耗时/失败率
  - CDN 命中率/带宽
  - 端到端延迟
  - 观众卡顿率/起播时间

业务监控：
  - 同时在线（Redis）
  - 弹幕量/礼物量（Kafka 流量）
  - 审核命中/误判
  - 主播活跃数

预案：
  - CDN 故障 → 切换备用厂商
  - 信令网关过载 → 限流+降级（弹幕显示拥挤提示）
  - 转码故障 → 降级单档位
```

## 十、底层本质

直播架构本质是**"流（音视频低延迟分发）+ 信令（实时互动削峰）+ 审核（实时合规）"**——流走 CDN 优化带宽，信令走长连接+MQ 优化并发，审核走抽帧+模型保证合规。

## 常见考点
1. **怎么扛百万并发弹幕**？——Netty 网关+Kafka 削峰+多级限流+客户端节流。
2. **直播延迟怎么降到 1s 内**？——WebRTC 替代 RTMP/HLS。
3. **直播录制和回放怎么做**？——旁路录制 ts→MP4→OSS，回放走点播 CDN。

# 【拼多多内容】Feed 推荐架构怎么设计？

> JD 依据："和算法同学挖掘业务问题"、"Feed 流"、"高并发大流量大数据量"。

## 一、推荐漏斗

```
                  亿级内容池
                       ↓
        ┌─────── 召回（多路）──────┐
        │ 协同过滤 / 向量 / 标签 / 热点 │  → 几千候选
        └──────────┬──────────────┘
                   ▼
            粗排（双塔模型）        → 几百
                   ▼
            精排（DIN/DeepFM）      → 一百（带 CTR 分）
                   ▼
            重排（多样性+广告）      → 几十（最终展示）
```

## 二、召回层（多路并行）

| 召回路 | 原理 | 用途 |
|--------|------|------|
| 协同过滤 | 相似用户看过的 | 经典 |
| Item2Vec | 内容向量相似 | 内容相关 |
| User-Vec | 用户兴趣向量 | 个性化 |
| 标签召回 | 用户标签匹配内容 | 简单稳 |
| 热点召回 | 全局热榜 | 兜底 |
| 向量召回 | ANN（Faiss/HNSW） | 大规模 |

```java
public List<Candidate> recall(User user) {
    List<Candidate> all = new ArrayList<>();
    all.addAll(cfRecall(user));         // 协同过滤
    all.addAll(vecRecall(user));        // 向量
    all.addAll(tagRecall(user));        // 标签
    all.addAll(hotRecall());            // 热点兜底
    return all;
}
```

**向量召回**（Faiss/HNSW）：
```
内容 Embedding（BERT/双塔）→ 向量库
用户 Embedding → ANN 查询 TopK 相似内容
```

## 三、粗排（快速过滤）

**双塔模型**：
```
用户塔：用户特征 → MLP → 用户向量
内容塔：内容特征 → MLP → 内容向量
              ↓ 内积
         相似度分数
```

特点：内容塔可离线预计算，在线只算用户塔+内积，速度快。

## 四、精排（CTR 预测）

**DIN（Deep Interest Network）**：
```
用户历史行为序列 → Attention（与候选 Item 相关性加权）→ 兴趣向量
+ 用户特征 + 内容特征 → MLP → CTR
```

**DeepFM**：
```
FM（一阶+二阶交叉） + DNN（高阶）→ CTR
```

```python
# 精排输入特征
features = {
    "user": {"age": ..., "gender": ..., "history": [...]},
    "item": {"category": ..., "tags": ..., "price": ...},
    "context": {"time": ..., "device": ...}
}
ctr = model.predict(features)
```

## 五、重排（业务策略）

```
1. 多样性：同 category 不连续（避免信息茧房）
2. 新颖性：插入未看过的新内容
3. 时效性：新闻/直播优先
4. 广告：按比例插入（第 3/7 位）
5. 业务规则：屏蔽已看过/黑名单
```

```java
public List<Feed> rerank(List<Feed> ranked) {
    // 多样性：滑窗保证相邻不同类
    List<Feed> result = slideWindow(ranked, 4);
    // 广告插入
    result = insertAds(result, adRatio);
    return result;
}
```

## 六、特征工程（与算法同学合作）

**特征类型**：
- 用户特征：基础（年龄/性别）+ 行为（点击/购买/停留）+ 兴趣标签
- 内容特征：基础（类目/价格）+ 质量（评分/热度）+ 标签
- 上下文：时间/地域/设备
- 交叉特征：用户×内容×上下文

**实时特征**（Flink）：
```
用户实时行为 → Flink → 实时特征库（Redis/HBase）
  例：最近 5 分钟点击的商品类目
```

## 七、冷启动

**新内容冷启动**：
- 用内容特征（类目/标签/Embedding）找相似用户
- 流量探索（EE 策略，给新内容曝光机会）
- 兜底用热度

**新用户冷启动**：
- 注册信息（性别/年龄/地域）→ 默认兴趣
- 引导选择兴趣标签
- 短期高频探索+快速建模

## 八、实时性

```
特征实时：Flink 流式聚合用户行为 → Redis（5min 窗口）
模型实时：在线学习（FTRL/PD）→ 小步快迭
召回实时：用户行为即时写入 ANN 库
```

## 九、架构组件

```
- 召回服务：Java/Python，Faiss/HNSW 向量库
- 排序服务：Python（TF/PyTorch），GPU 推理
- 特征平台：Hadoop/HBase（离线）+ Flink/Redis（实时）
- 模型平台：训练（TF/PyTorch）+ 推理（TF Serving/Triton）
- AB 实验：流量分桶+指标对比
```

## 十、底层本质

Feed 推荐本质是**"用多级漏斗+模型预测从亿级内容筛出最相关的几十条"**——召回保 recall、排序保 precision、重排保体验，特征+模型是核心，与算法同学紧密合作迭代。

## 常见考点
1. **怎么衡量推荐效果**？——CTR/停留时长/互动率/留存/GMV。
2. **AB 实验怎么做**？——流量分桶（hash uid）+ 指标显著性检验。
3. **向量召回用什么库**？——Faiss（CPU）/NGT/HNSW（图）/Milvus。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：推荐漏斗你设"召回几千→粗排几百→精排一百→重排几十"，为什么不直接精排全量候选（亿级）？精排最准不是效果最好吗？**

精排（DIN/DeepFM）是计算密集的深度模型，单次推理 50-100ms。亿级候选全精排要几万秒（不可能）。漏斗的本质是"用便宜的模型粗筛，用贵的模型精选"：召回（向量 ANN 查询，毫秒级，从亿级到几千）→ 粗排（双塔模型，内积快，从几千到几百）→ 精排（复杂模型，从几百到一百）。每一层用"成本递增 + 准确率递增"过滤，最终精排只跑 100 个候选，单次请求总耗时控制在 200ms 内。如果直接精排全量，要么延迟爆炸（用户等不了），要么模型必须极度简化（效果差）。漏斗是用"分层降本"在延迟和效果间找平衡。

### 第二层：证据与定位

**Q：推荐 CTR 从 3.5% 掉到 2.8%，你怎么定位是召回、排序、还是特征的问题？**

CTR 下降要分层定位：
1. 看召回层——召回的"覆盖率"和"多样性"是否下降。如果召回只从热点池拉（协同过滤失效），千篇一律的热门内容，用户点击欲望低。检查各召回路（协同/向量/标签/热点）的召回数量占比，是否某路召回挂了。
2. 看排序层——精排模型的 AUC（离线评测）是否下降。对比线上模型的预测 CTR 分布，是否集中在低分（模型保守，不敢推）。检查模型版本，是否最近更新后效果退化。
3. 看特征——实时特征（用户最近 5 分钟点击）是否刷新。Flink 任务挂了导致特征是几小时前的，模型用旧特征预测不准。检查特征平台的 `feature_freshness`。
4. 看重排——多样性重排是否过激（强制插入不相关内容），或广告插入比例调高了。

### 第三层：根因深挖

**Q：精排模型上线后离线 AUC 0.78（很好），但线上 CTR 没涨。这是经典的"离线好线上差"，根因是什么？**

离线/线上不一致的根因：
1. 特征不一致——离线训练用 Hive 的特征（T+1），线上预测用 Flink 的实时特征。如果实时特征和离线特征的口径不同（如"最近点击"离线是 30 天、线上是 5 分钟），模型见到的输入分布变了，预测偏差。解法：特征一致性校验（在线/离线特征对比）。
2. 样本偏差——离线训练用历史日志（过去的用户行为），线上是新流量。如果用户兴趣漂移（季节性/热点变化），模型泛化差。解法：模型在线学习（FTRL）+ 高频更新（每天/每小时重训）。
3. 位置偏差——离线日志里的点击受展示位置影响（排前面的天然点击高），模型学了"位置特征"而非"内容质量"。上线后用模型排序，位置变了，偏差暴露。解法：位置去偏（PAL 模型，把位置作为独立特征分离）。
4. E&E 问题——新模型上线后，推荐的内容和旧模型不同，用户没见过的新内容点击率天然低（探索期）。需要观察足够长（1-2 周）让模型收敛。

### 第四层：方案权衡

**Q：信息茧房（用户只看到同质内容）是推荐的痛点。你的重排层用"多样性滑窗"保证相邻不同类，但用户抱怨"我点了几个母婴商品，现在全是母婴"。怎么权衡相关性和多样性？**

多样性滑窗只解决"局部多样性"（相邻几条不同类），不解决"全局多样性"（整体分布）。权衡方案：
1. 全局多样性约束——重排时不只看相邻，而是统计"当前 Feed 流的类目分布"，如果某类目占比 >30%，强制插入其他类目。代价是可能插入低相关内容。
2. 兴趣探索（EE）——按 80/20 分配：80% 流量用模型推荐（强相关），20% 流量探索（随机/新类目）。探索的内容用户可能不点（短期 CTR 降），但长期丰富用户画像。
3. 长期收益指标——不只看单次 CTR，看"7 日留存""人均类目数"。如果为短期 CTR 而制造茧房，用户会因审美疲劳流失。
4. 用户可控——给用户"减少此类推荐"的反馈按钮，显式调整。本质是"让用户参与多样性决策"，而不是算法强加。

### 第五层：验证与沉淀

**Q：你怎么验证新推荐模型比旧模型好？上线决策依据是什么？**

推荐模型上线必须靠 AB 实验：
1. 流量分桶——`uid hash % 100`，实验组 5% 用新模型，对照组 5% 用旧模型，剩余 90% 旧模型（保底）。
2. 指标对比——实验组 vs 对照组的 CTR、CVR、人均时长、留存（7 日）。关键指标提升 + 统计显著（p <0.05）才上线。
3. 显著性检验——t 检验对比两组指标，避免"碰巧"提升。流量小（1%）时指标波动大，需更长实验周期（2 周）。
4. 分人群下钻——整体涨但某个人群（如新用户）降，要分析是否对该人群有害。
沉淀：所有推荐变更走 AB 实验（禁止直接全量上线）；实验平台支持多实验正交（互不影响）；指标看板每日更新，实验报告自动生成；模型版本管理（可回滚到旧模型）。

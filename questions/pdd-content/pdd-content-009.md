# 【拼多多内容】Redis 缓存一致性方案（评价/Feed）？

> JD 依据："稳定性建设"、"监控"。

## 一、不一致的根因

```
并发场景 1（先更新库再更新缓存）：
  T1 更新库 A=1
  T2 更新库 A=2
  T2 更新缓存 A=2
  T1 更新缓存 A=1   ← 库=2 缓存=1，脏！

并发场景 2（先删缓存再更新库）：
  T1 删缓存
  T2 读缓存未命中→读 DB（旧值 A=1）→ 写缓存 A=1
  T1 更新库 A=2   ← 库=2 缓存=1，脏！
```

## 二、主流策略：Cache Aside（旁路缓存）

```
写：先更新 DB → 再删缓存
读：查缓存未命中 → 查 DB → 写缓存
```

```java
@Transactional
public void updateReview(Review r) {
    reviewDao.update(r);
    redis.del("review:" + r.getId());    // 删缓存
}
```

**为什么删而不是更新**：删是幂等的；更新可能并发覆盖，删让下次读惰性重建。

**仍有问题**（极少）：
```
T1 读 DB（旧值）
T2 更新 DB
T2 删缓存
T1 写缓存（旧值）   ← 脏！
```
触发条件苛刻（T1 读 DB 必须比 T2 写+删慢得多，且无并发写），实际概率低。

## 三、延时双删（更稳）

```
T1 写 DB
T1 删缓存
T1 sleep 1s（等并发读完成）
T1 再删一次缓存
```

```java
public void updateReview(Review r) {
    reviewDao.update(r);
    redis.del(key);
    executor.schedule(() -> redis.del(key), 1, SECONDS);   // 延时再删
}
```

## 四、最终一致：canal 监听 binlog

```
DB → canal（伪装 slave）→ MQ → 消费者删缓存
```

```java
@KafkaListener(topics = "review-binlog")
public void onBinlog(BinlogEvent e) {
    if (e.getType() == UPDATE || e.getType() == DELETE) {
        redis.del("review:" + e.getId());
        // 同步更新 ES
        esService.update(e.toReview());
    }
}
```

**优点**：业务代码不感知缓存；DB 一致性由 binlog 保证。

## 五、内容场景实战

**评价缓存**：
```
评价页数据结构（Hash）：
  key = review:product:{productId}:page:1
  field = reviewId    value = {uid, content, score, ...}

更新策略：
  1. 用户改评价 → 写 DB → 删缓存
  2. canal 监听 binlog → 异步删缓存 + 更新 ES（搜索）
  3. 删除延时双删兜底（防并发读脏）
```

**Feed 流缓存**：
```
Feed 时间线：
  key = feed:user:{uid}
  type = ZSet   score=create_time   member=feed_id
  
更新策略：
  发布 Feed → 写 DB → canal 触发"推到粉丝收件箱"→ 异步
  读 Feed → ZSet 直接拿（避免扫全表）
```

**热点商品评价**（爆款）：
```
- 多级缓存：Caffeine（本地）+ Redis（分布式）
- 缓存预热：商品上架/活动前预热
- 失效：用户改/审核通过/新增 → 删 product:{id}:page:* + 异步重建
```

## 六、强一致（慎用）

如果必须强一致（如金融级）：
- 串行化：读写都加分布式锁（吞吐差）
- 2PC：DB 和缓存参与两阶段提交（复杂）

内容场景**不要**用强一致，最终一致足够。

## 七、底层本质

缓存一致本质是**"副本与原数据的一致性"**——Cache Aside 是基础（删优于更新），延时双删防并发窗口，canal binlog 做最终一致兜底。

## 常见考点
1. **缓存击穿/穿透/雪崩**？——击穿（互斥锁重建）/穿透（布隆+空值）/雪崩（随机 TTL）。
2. **怎么保证缓存高可用**？——Redis Cluster（主从+分片）+ Sentinel 哨兵。
3. **缓存预热怎么做**？——活动前批量灌入（脚本/任务）+ 灰度放量。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Cache Aside 你选"写库后删缓存"而不是"写库后更新缓存"，为什么不更新？更新不是读得更快吗？**

更新缓存有两个致命问题：1.并发覆盖——线程 A 先写库（值=1），线程 B 后写库（值=2），但因网络/调度，B 先更新缓存（=2），A 后更新缓存（=1），最终缓存=1 而 DB=2，脏数据。删缓存是幂等的，删两次和删一次效果一样，不存在覆盖。2.缓存利用率——很多写操作后并不会立刻被读（如用户编辑评价后关闭页面），更新缓存是无效写入。删缓存让"下次真有人读时再重建"（lazy load），天然过滤无效写入。唯一的代价是删后的第一次读要回源 DB（多一次查询），但相比脏数据风险和无效写入，这点延迟可接受。

### 第二层：证据与定位

**Q：用户反馈"修改了评价但页面上还是旧的"，你怎么定位是缓存不一致还是别的（CDN/前端/DB 主从延迟）？**

缓存不一致的定位要排除其他环节。沿着"用户→CDN→网关→应用→Redis→MySQL 主从"链路逐段查：
1. 先确认 DB 主库是新的——直接连主库 `SELECT * FROM review WHERE id=?` 看值是不是新的。如果主库是旧的，问题在写入链路（写没成功）。
2. 看从库——如果应用读的是从库，`SHOW SLAVE STATUS` 看 `Seconds_Behind_Master`，主从延迟几秒会导致"改了立刻读还是旧的"。
3. 看 Redis——`redis-cli GET review:{id}` 看缓存值。如果缓存是旧的，确认删除是否执行了：看应用日志里 `redis.del` 调用记录，或 Redis 的 MONITOR（测试环境）。
4. 看 CDN——如果评价页是 HTML 走 CDN，CDN 缓存未失效也会显示旧内容，看 `Cache-Control` 和 CDN purge。

### 第三层：根因深挖

**Q：你发现 Cache Aside 仍有极端场景会脏（读 DB 旧值→写缓存的窗口），用延时双删还是 canal binlog 补偿？两者本质区别是什么？**

两者解决的是同一个"并发窗口"问题，但机制不同：
1. **延时双删**——写库→删缓存→sleep 1s→再删一次。原理是"等并发读把旧值写进缓存后，第二次删清掉它"。代价是依赖 sleep 时长估得准（太短清不掉，太长阻塞写请求），且写请求延迟增加 1s。
2. **canal binlog 补偿**——DB 变更产生 binlog，canal 伪装 slave 消费 binlog，异步删缓存。原理是"所有 DB 变更都触发删缓存"，彻底兜底。代价是引入 canal + MQ 的运维复杂度，且有延迟（binlog 消费 + MQ 投递，通常 100ms-1s）。
内容场景推荐 canal——延时双删的 sleep 难估（不同操作耗时不同），canal 是"DB 事件驱动"，不依赖业务代码写法，一致性更稳。拼多多评价/Feed 用 canal + Kafka，binlog 到缓存删除的 P99 延迟 <500ms，业务无感。

### 第四层：方案权衡

**Q：canal 删缓存方案里，如果 Kafka 消息丢了或消费失败了，缓存就一直脏。你怎么权衡一致性和复杂度？**

消息丢失/消费失败要靠"兜底 + 幂等"：
1. 幂等——消费者删缓存天然幂等（删多次无副作用），所以重试安全。canal 的 ACK 机制保证消息至少投递一次，配合幂等就是 at-least-once。
2. 失败补偿——消费者 catch 异常后写入"重试队列"（死信队列），定时任务扫描重试。如果连续失败 N 次，告警人工介入。
3. 全量校准兜底——定时任务（如每天凌晨）对比 Redis 缓存和 MySQL，发现不一致的 key 主动删缓存重建。这是"最终一致的最终保障"，即使 canal 挂了也能在 24 小时内自愈。
4. 不用强一致——内容场景允许秒级不一致（用户改评价后 1 秒内看到旧值无感），所以不引入 2PC 或分布式锁。强一致留给金融场景。

### 第五层：验证与沉淀

**Q：你怎么验证 canal 删缓存方案真的生效，而不是"恰好没出现不一致"？**

不能用"线上没报问题"当验证，要主动测：
1. 故障注入测试——压测时主动造并发：一边改评价（写库），一边并发读，对比读到的值和最终 DB 值，统计不一致率。正常应 <0.01%（最终一致窗口内）。
2. 一致性巡检——离线任务每小时抽样 1 万个评价，对比 Redis 值和 MySQL 主库值，不一致率告警阈值 <0.1%。
3. canal 健康度——监控 canal 的 `delay`（binlog 位点滞后），`delay > 5s` 告警；监控 Kafka 消费 lag，`lag > 1000` 告警。
沉淀：所有写库 + 缓存场景必须走 canal（禁止业务代码手动删缓存，统一收口）；canal 位点定期备份；缓存 key 设计带版本号（`review:{id}:v2`）便于强制失效。

## 结构化回答


**30 秒电梯演讲：** 缓存像便签纸——DB 是账本，便签是抄录，改账本要扔掉旧便签（删缓存），不然读到旧值。

**展开框架：**
1. **Cache Aside** — 写库→删缓存（最常用）
2. **延时双删** — 写库→删缓存→延时再删
3. **最终一致** — canal 监听 binlog + 消息补偿

**收尾：** 为什么是删缓存不是更新缓存？

# 【拼多多内容】评价系统整体架构怎么设计？

> JD 依据："评价和行家社区"、"稳定性建设"、"系统架构优化"。

## 一、整体架构

```
                  ┌────────────────────────────┐
                  │     API 网关 (Spring Cloud) │
                  │   鉴权 / 限流 / 路由          │
                  └─────────────┬──────────────┘
                                │
        ┌──────────┬────────────┼─────────────┬──────────────┐
        ▼          ▼            ▼             ▼              ▼
   评价写入     评价查询     评价审核       评分聚合       商家回复
   服务        服务        服务           服务           服务
        │          │            │             │              │
        └──────────┴────────────┴─────────────┴──────────────┘
                                │
                                ▼
                        ┌──────────────┐
                        │ Kafka 事件总线│
                        └──────┬───────┘
               ┌────────┬─────┴──────┬──────────┐
               ▼        ▼            ▼          ▼
            ES 同步  统计聚合     推荐召回    通知商家
            (搜索)   (Flink)      (算法)      (消息)
                                │
                                ▼
        ┌─────────────── 存储 ──────────────────┐
        │ MySQL（主，分库分表）│ Redis（聚合缓存）│
        │ ES（搜索索引）       │ OSS（图片/视频） │
        └─────────────────────────────────────────┘
```

## 二、服务拆分

| 服务 | 职责 | 关键能力 |
|------|------|----------|
| 评价写入 | 接收评价提交 | 校验/幂等/落库 |
| 评价查询 | 列表/详情/我的 | 多级缓存 |
| 评价审核 | 机审+人审 | 规则+模型+人工 |
| 评分聚合 | 商品评分实时计算 | Redis 增量 |
| 商家回复 | 商家互动 | 同样审核 |

## 三、写入链路

```java
@PostMapping("/reviews")
public Result submit(@RequestBody ReviewDTO dto) {
    // 1. 参数校验 + 防重（orderId+uid 幂等）
    if (reviewDao.existsByOrderUid(dto.getOrderId(), dto.getUid())) {
        throw new BizException("已评价");
    }
    // 2. 反作弊前置
    if (antiCheatService.isSuspicious(dto)) {
        return Result.fail("评价异常");
    }
    // 3. 落 DB（status=0 待审）
    Review r = reviewService.create(dto);
    // 4. 发事件（异步触发审核/统计/通知）
    eventBus.publish(new ReviewSubmittedEvent(r));
    return Result.ok(r.getId());
}
```

## 四、审核链路

```
ReviewSubmittedEvent
   ↓ Kafka
机审服务（敏感词+模型）
   ├─ 通过 → status=1 → 发 ReviewApprovedEvent
   └─ 疑难 → status=2 → 入人审队列
                  ↓
              人审台决策 → 通过/拒绝 → 发事件
```

## 六、查询链路（多级缓存）

```
请求 → Caffeine（本地）→ Redis（分布式）→ ES/MySQL
        L1（秒级）       L2（分钟级）       兜底
```

**搜索走 ES**（关键词/过滤/聚合），**详情走 Redis+MySQL**。

## 七、容量与扩展

```
- MySQL：按 product_id 分库分表（256 张表）
- Redis：Cluster（评分缓存/计数/锁）
- ES：按月建索引（review-2026.07）+ 别名
- Kafka：按 product_id 分区（保证同商品顺序）
- 服务：K8s HPA 弹性扩容
```

## 八、稳定性

```
- 网关限流：单 UID/全局 QPS 限制
- 服务熔断：Sentinel（依赖挂时不拖死）
- 降级：审核挂时先入库 status=0 异步审；ES 挂时查 MySQL
- 监控：写入 QPS/审核延迟/评分一致性/反作弊命中率
```

## 九、底层本质

评价架构本质是**"分层服务 + 事件驱动 + 多存储分工"**——服务按职责拆分，事件总线解耦下游，存储各司其职（MySQL 主、Redis 缓存、ES 搜索、OSS 文件）。

## 常见考点
1. **评分怎么保证一致**？——Redis 增量+定时校准+幂等消费。
2. **审核延迟怎么降**？——机审实时+模型推理优化+人审并发+优先级队列。
3. **评价搜索怎么做**？——ES 倒排+分词+BM25+filter 过滤+聚合。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：评价架构你把"写入""审核""聚合"拆成三个独立服务，为什么不做成一个大服务（少跨进程调用不是更快）？**

拆服务的核心驱动是"故障隔离 + 独立扩容"，不是性能。评价写入是同步链路（用户提交要等结果），审核是异步链路（机审+人审耗秒到分钟），聚合是读链路（商品页读评分）。如果合并成一个服务：审核高峰（模型推理慢）会吃掉写入的线程池，用户提交评价卡顿；聚合读的缓存重建会抢占写入的 DB 连接。拆开后：写入服务保持轻量（校验+落库+发事件，P99 <100ms），审核服务可独立扩容（GPU 资源按审核量配），聚合服务走缓存（高 QPS 读不碰主库）。跨进程调用的延迟（RPC 1-3ms）相比隔离带来的稳定性收益可忽略。本质是用"拆分"换"故障不蔓延"。

### 第二层：证据与定位

**Q：评价提交后用户反馈"提交成功但商品页看不到"，你怎么定位是写入成功但审核未通过，还是缓存未更新？**

沿链路逐段查：
1. 查 DB 主库 `SELECT status FROM review WHERE id=?`——如果 status=0（待审），说明审核还没过，商品页过滤了未审评价，是正常的（等审核）。如果 status=1（通过），说明写入和审核都 OK，问题在展示。
2. 查缓存——`redis GET product:rating:{pid}` 看 count 是否 +1，`GET review:list:{pid}:page:1` 看列表是否含该 id。如果 DB 是通过但缓存没更新，是 canal 同步延迟或失败。
3. 查 ES——如果商品页评价列表走 ES 搜索，`POST /review/_search` 看是否索引了该评价。ES 异步同步有 1-2s 延迟。
4. 查审核日志——如果是机审拒绝（status=2），用户看不到是预期，但用户以为"提交成功"是因为接口返回了 id。UX 上应提示"评价审核中"。

### 第三层：根因深挖

**Q：评分聚合的增量计算（Redis hincrby）和定时校准（凌晨全量重算）偶尔打架——校准后评分变了，用户感觉"评分一天变两次"。根因和解法是什么？**

根因是"增量"和"全量"用了不同的计算口径或时间窗口。深挖：
1. 增量算的是"实时事件"（评价上架 +1），全量算的是"当前 DB 状态"（SELECT count(*) WHERE status=1）。如果增量只统计了"上架"事件但没统计"下架/删除"事件，两者口径不一致，校准时数字会跳变。
2. 校准和增量的并发——凌晨 3 点校准跑全量 SELECT（耗时几秒），这几秒内又有新评价上架，增量 hincrby +1，但校准的 SELECT 是几秒前的快照，覆盖时把增量加的又冲掉了。
解法：校准用"乐观锁"——校准前记 `version`，校准后写入时检查 `if version == current then set else abort`，被打断就重算。或校准只在"低峰且无增量"时跑（凌晨 + 临时禁增量）。

### 第四层：方案权衡

**Q：评价写入你用同步写 DB + 异步发 Kafka 触发审核。但 Kafka 消息可能延迟（秒级），用户提交后审核结果迟迟不出来的体验差。你要不要改成同步审核？**

同步审核（提交时实时调 NLP 模型）的代价是延迟从 100ms 涨到 500ms+（模型推理），且审核服务故障会拖垮写入。权衡：
1. 分级审核——规则机审同步（敏感词毫秒级，命中直接拒绝），模型机审异步（Kafka 秒级）。大部分违规（敏感词/黑名单）同步挡掉，用户即时知道被拒；模糊的（模型判断）异步，用户先看到"审核中"。
2. UX 优化——提交即返回"评价已提交，审核中"，用户不阻塞等待。审核通过后通过 WebSocket/消息推送通知用户。
3. 超时策略——模型审核 SLA 设 30s，超时默认通过（先展示，事后人审发现违规再下架）。这样 P99 用户在 30s 内看到评价。
拼多多选异步 + SLA 承诺——同步审核的延迟（500ms）在写入链路成本太高，异步的 1-2s 延迟用户可接受（评价不是实时互动）。

### 第五层：验证与沉淀

**Q：评价系统重构后（单体拆微服务），怎么验证拆分没引入回归 bug？**

微服务拆分的验证靠全链路回归：
1. 功能回归——评价的完整生命周期（提交→审核→上架→编辑→下架→申诉）跑全流程，对比拆分前后行为一致。
2. 数据一致性——拆分前后的 DB 数据对账（count、score、status 分布），ES 和 MySQL 的数据一致性巡检。
3. 性能对比——拆分后 P99 不能显著退化（RPC 调用的额外开销 <10ms 可接受）。压测对比单服务 vs 拆分后的 QPS。
4. 故障注入——kill 审核服务，验证写入仍正常（异步，不阻塞）；kill 聚合服务，验证商品页降级（显示默认评分）。
沉淀：服务拆分必须有契约测试（Pact）保证接口契约；服务间调用必须有熔断（防级联故障）；评价全链路 traceId 串联（写入→审核→聚合→ES），方便定位。

# 【拼多多内容】内容中台架构怎么设计？

> JD 依据："新媒体业务平台"、"系统架构优化"、"中台"。

## 一、为什么需要内容中台

```
无中台：
  评价组：自建生产/审核/存储
  直播组：自建生产/审核/存储
  短视频组：自建生产/审核/存储
  → 重复造轮子，审核规则不一致，数据孤岛

有中台：
  内容中台：统一生产/审核/存储/分发能力
  评价组/直播组/短视频组：调用中台 + 业务定制
  → 复用+一致+快速创新
```

## 二、能力分层

```
┌────────────── 业务前台（定制） ──────────────┐
│  评价业务 │ 直播业务 │ 短视频业务 │ 社区业务    │
└──────────────────┬──────────────────────────┘
                   │ 调用
┌──────────── 内容中台（共性能力） ────────────┐
│  生产能力 │ 审核能力 │ 存储能力 │ 分发能力 │ 分析 │
│  (UGC 编辑器)│(机审+人审)│(对象+索引)│(Feed/搜索)│(统计)│
└──────────────────┬──────────────────────────┘
                   │
┌──────────── 基础设施 ────────────────────────┐
│  MySQL │ Redis │ ES │ Kafka │ OSS │ Flink     │
└──────────────────────────────────────────────┘
```

## 三、五大核心能力

**1. 生产能力**：
- UGC 编辑器（富文本/图片/视频）
- 模板（评价/动态/短视频）
- 草稿/发布

**2. 审核能力**：
- 规则引擎+模型服务+人审台（详见 024）
- 多业务共享审核规则（敏感词/合规统一）
- 业务线定制规则（评价查重 vs 短视频查重）

**3. 存储能力**：
- 内容主数据（MySQL 分库分表）
- 文件存储（OSS/S3 + CDN）
- 搜索索引（ES）
- 缓存（Redis）

**4. 分发能力**：
- Feed 流（推/拉/混合）
- 搜索（ES）
- 推荐（召回+排序）
- 推送（消息/通知）

**5. 分析能力**：
- 内容统计（量/质/互动）
- 用户行为（点击/停留/转化）
- 实时大屏（Flink）

## 四、DDD 领域划分

```
内容域（Content）
  - 实体：Content（评价/视频/动态）
  - 值对象：Tag/Media
  - 聚合根：Content

审核域（Audit）
  - 实体：AuditTask/AuditRule
  - 领域服务：RuleEngine/ModelPredictor

用户域（User）
  - 实体：User/Author
  - 值对象：Profile/TrustScore

互动域（Interaction）
  - 实体：Like/Comment/Share
  - 领域事件：UserLikedEvent

分发域（Distribution）
  - 实体：Feed/SearchIndex
  - 领域服务：FeedPusher/Searcher
```

## 五、多租户（业务线隔离）

```java
@Entity
@Table(name = "content")
public class Content {
    @Id private Long id;
    private String tenantId;     // 业务线标识（review/live/short_video）
    private String bizType;      // 业务子类型
    private Long bizId;          // 业务 ID（评价 ID/直播 ID）
    ...
}
```

**租户隔离**：
- 数据：按 tenantId 分库/分表 或 行级隔离
- 配置：每个租户独立审核规则/分发策略
- 限流：按租户限流防互相影响
- 监控：按租户维度统计

## 六、能力开放

```
API（HTTP/gRPC）：
  POST /content/publish        发布内容
  GET  /content/{id}           查询
  POST /content/audit          触发审核
  POST /feed/push              推送到 Feed
  GET  /search                 搜索

SDK：
  Java SDK（业务线集成）
  JS SDK（前端编辑器）
```

**OpenAPI 网关**：
- 鉴权（业务线 token）
- 限流（按租户）
- 路由（灰度）
- 审计

## 七、内容主数据模型

```sql
CREATE TABLE content (
  id BIGINT PRIMARY KEY,
  tenant_id VARCHAR(32),        -- 业务线
  biz_type VARCHAR(32),         -- review/live/short_video
  biz_id BIGINT,
  author_uid BIGINT,
  title VARCHAR(200),
  content TEXT,
  media JSON,                   -- 图片/视频 URL
  tags JSON,
  status TINYINT,               -- 0待审 1通过 2人审 3下架
  create_time DATETIME,
  INDEX idx_tenant_biz(tenant_id, biz_type, biz_id),
  INDEX idx_author(author_uid, create_time)
);
```

## 八、事件驱动

```
内容生命周期事件：
  ContentPublished / ContentApproved / ContentRejected / ContentTakenDown
  
各业务订阅：
  - 评价组：监听 ContentPublished 同步商品评价列表
  - Feed 组：监听 ContentApproved 推送到粉丝收件箱
  - 搜索组：监听 ContentApproved 同步 ES
  - 推荐组：监听 ContentApproved 加入召回池
```

## 九、中台治理

```
- 能力 owner：每个能力有团队负责（开发/运维/迭代）
- SLA：能力有明确 SLA（可用性/延迟/QPS）
- 版本管理：API 版本化，兼容
- 文档：OpenAPI 文档 + 接入指南
- 监控：能力调用监控+业务反馈
```

## 十、底层本质

内容中台本质是**"把内容共性能力下沉为共享服务"**——通过 DDD 划域+多租户隔离+能力开放，让业务前台快速创新，避免重复造轮子。中台成功的关键是贴合业务+有 owner+不过度抽象。

## 常见考点
1. **中台和微服务关系**？——中台是能力复用理念，微服务是技术落地形式。
2. **怎么避免中台失败**？——贴合业务（不要先建中台再找业务）+有 owner+小步迭代。
3. **多租户怎么隔离**？——数据（库/表/行）+ 配置（规则）+ 限流 + 监控。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：内容中台你用 DDD 划域（内容/审核/用户/互动/分发），但业务线说"我们就要个简单的内容存储，搞 DDD 太重了"。你怎么反驳或认同？**

DDD 的价值取决于"业务复杂度是否会增长"。如果内容中台只服务评价（单一业务、简单 CRUD），DDD 确实过度设计。但拼多多内容中台要服务评价、直播、Feed、短视频、社区等多业务线，每条线的内容模型、审核规则、分发策略都不同且有共性。DDD 的意义在于"识别共性（内容域）vs 差异（业务定制）"，把共性沉淀为领域模型（Content 实体、AuditTask 聚合），差异通过扩展点（策略模式）实现。没有 DDD，多业务线迭代会让代码变成"if bizType == review / if bizType == live"的大泥球，复用无从谈起。反驳依据：中台的立身之本是"复用"，DDD 是保证复用不被业务定制污染的方法论。如果只是简单存储，不该叫中台，直接给个 DAO 就行。

### 第二层：证据与定位

**Q：内容中台上线后，评价组反馈"调用中台的内容发布接口比我们原来的慢 100ms"。你怎么定位是中台架构的开销还是实现问题？**

中台化必然引入抽象开销，但 100ms 偏高。排查：
1. 看链路——用 SkyWalking/traceId 看发布接口的全链路，每段耗时。中台的典型链路：鉴权 → 参数校验 → 内容落库 → 发审核事件 → 发分发事件。如果鉴权花了 30ms（每次查权限表），是实现问题（应缓存权限）。
2. 对比单业务直连——评价组原来直接写 review 表，中台走 content 表（tenant_id 隔离）+ 事件总线。多出来的开销：事件投递 Kafka（10-20ms）、多租户路由（几 ms）。100ms 远超理论值。
3. 看 DDD 抽象——是否每次发布都经过太多领域服务（ContentService → AuditDomainService → DistributionDomainService），每次跨服务调用（RPC）几 ms。如果中台把多域拆成微服务，RPC 开销累积。解法：同进程内用本地调用，跨进程才 RPC。

### 第三层：根因深挖

**Q：内容中台的多租户（tenantId 隔离）在数据量大时出问题——评价（tenant=review）有 10 亿条，短视频（tenant=short_video）才 100 万条，查询评价时大量短视频数据在索引里拖慢。根因和解法？**

根因是"行级隔离"在数据量悬殊时退化。所有租户的数据在同一张 content 表，即使 `WHERE tenant_id='review'`，B+ 树索引仍然要遍历所有 tenant_id 的节点（索引未按 tenant 物理聚簇）。解法：
1. 分区表——按 tenant_id 做 MySQL 分区（PARTITION BY LIST），评价在独立分区，查询只扫评价分区。
2. 分库分表——按 tenant_id 分到不同库/表（评价库、短视频库物理隔离）。彻底隔离，但跨租户查询难。
3. 索引优化——建 `(tenant_id, biz_type, create_time)` 联合索引，让索引先按 tenant 聚簇，查询 `WHERE tenant_id=?` 走索引快速定位。
4. 冷热分离——评价历史数据（1 年前）归档到冷存储（HBase/OSS），热表只存近期，减小索引规模。

### 第四层：方案权衡

**Q：中台的能力开放你用 HTTP/gRPC API + SDK，但业务线抱怨"每次加功能都要等中台发版"。你要不要让业务线直接操作中台的 DB（给 SQL 权限）？**

绝对不能让业务线直接操作中台 DB。权衡：
1. 直接操作 DB 的代价——业务线绕过中台的领域逻辑（审核、事件分发），数据不一致（没触发审核的内容直接可见）；中台失去演进能力（DB schema 变了影响所有业务线）；安全风险（SQL 注入、误删）。
2. 中台发版慢的根因——能力开放节奏太慢，每次小功能都要走中台全量发版。解法不是"放开 DB"，而是"降低开放成本"：
   - 插件化——中台提供扩展点（SPI/钩子），业务线实现插件注入，不用中台改代码。
   - 配置化——业务规则（审核阈值、分发策略）放配置中心，业务线自助改。
   - 低代码——中台提供 DSL/模板（如"新建内容类型"的配置化向导），业务线自助创建。
3. 平衡点——核心能力（审核、存储）由中台控制（强一致），业务定制（排序、展示）通过扩展点开放（灵活）。

### 第五层：验证与沉淀

**Q：内容中台上线后怎么验证"真的实现了复用"（而不是各业务线各自在中台上搭了一套独立逻辑）？**

中台复用的衡量指标：
1. 复用率——统计各业务线调用中台"公共能力"的比例 vs 自己写定制逻辑的比例。理想：80% 走公共能力，20% 定制。
2. 代码重复度——用工具（如 PMD/Sonar）检测业务线代码里是否有重复的内容处理逻辑（CRUD、审核触发）。如果各业务线都在写"落库 + 发审核事件"，说明中台没收口。
3. 接入成本——新业务线接入中台的"人天"数，应该递减（第 1 个业务线 30 人天，第 5 个 <10 人天，因为沉淀了 SDK/模板/文档）。
4. 能力演进——中台每季度新增的"公共能力"数 vs 业务线提的"定制需求"数，公共能力增长说明复用在深化。
沉淀：中台必须有 owner 团队（开发+运维+迭代）；能力有 SLA（可用性/延迟/QPS）；API 版本化（兼容）；接入文档 + 最佳实践 + Demo 案例；能力调用监控 + 业务反馈闭环。

## 结构化回答

**30 秒电梯演讲：** 多业务线有共性内容能力，如何下沉复用避免重复造轮子？简单说就是——内容中台是"把内容生产/审核/存储/分发能力下沉为共享服务"的架构，避免各业务线重复造轮子；用 DDD 划域+能力开放+多租户。DDD：内容/审核/用户/互动域；多租户：业务线隔离+共享。

**展开框架：**
1. **能力** — 能力：生产/审核/存储/分发/分析
2. **DDD** — DDD：内容/审核/用户/互动域
3. **多租户** — 多租户：业务线隔离+共享

# 【拼多多内容】直播系统架构与弹幕削峰？

> JD 依据："直播短视频"、"高并发大流量"、"IO/多线程/网络"。

## 一、直播全链路

```
                推流                       信令
  ┌────────┐   RTMP/SRT   ┌────────┐   WebSocket   ┌────────┐
  │ 主播端  │ ──────────▶ │ 直播中心 │ ◀──────────▶ │ 观众端  │
  │ 推流 SDK│             │ 转码/录制│              │ 播放器  │
  └────────┘             └────┬─────┘              │ 弹幕 UI │
                              │                    └────────┘
                              ▼ HTTP-FLV/HLS
                       ┌──────────┐
                       │   CDN    │ 边缘节点分发
                       │ 边缘节点  │
                       └──────────┘
```

## 二、核心组件

**1. 推流**：
- 主播端用 SDK（OBS/自研）按 RTMP 协议推流到直播中心
- 协议：RTMP（推流常用）/SRT（弱网好）/WebRTC（连麦超低延迟）

**2. 转码**：
- 一路源流 → 多档位（1080p/720p/480p/360p）适配网络
- 编码：H.264/H.265/AV1
- 协议转换：RTMP → HLS（切片 ts）/HTTP-FLV（流式）

**3. CDN 分发**：
- 观众就近访问边缘节点
- 命中率 >95% 才省钱省带宽

**4. 信令服务**（弹幕/礼物/连麦）：
- 长连接（WebSocket/TCP）
- 消息广播：单直播间百万级订阅

## 三、信令架构（百万并发）

```
观众 ────WebSocket──── 长连接网关（百万级）
                          ↓
                       消息总线（Kafka）  ← 弹幕生产者
                          ↓
                       广播服务（按 liveId 路由）
                          ↓
                       房间订阅树（Redis ZSet 存在线用户）
```

**单直播间弹幕广播**：
```java
// 1. 接收弹幕
@MessageMapping("/danmaku/{liveId}")
public void sendDanmaku(Danmaku d) {
    // 风控+敏感词过滤
    if (auditService.isBlocked(d)) return;
    // 投 Kafka
    kafkaTemplate.send("live-danmaku-" + (d.getLiveId() % 32), d);
}

// 2. 消费 + 广播
@KafkaListener(topicPattern = "live-danmaku-.*")
public void broadcast(Danmaku d) {
    Set<Long> uids = onlineUsers.get(d.getLiveId());
    // 批量推到网关，网关下发到每个连接
    gatewayPusher.broadcast(d.getLiveId(), d);
}
```

## 四、弹幕削峰（关键）

**痛点**：百万直播间×每秒千条弹幕 = 百万 QPS，网关和带宽扛不住。

**多层削峰**：
```
1. 服务端：
   - Kafka 削峰（弹幕先进 MQ）
   - 消费者按容量限速（每秒 N 条）
   - 同类聚合（"哈哈哈"合并显示）
   - 优先级（VIP/付费弹幕优先）

2. 网关：
   - 每个连接限流（用户 N/s）
   - 房间级限流（每房间 M/s 下发）

3. 客户端：
   - 节流渲染（按帧合并）
   - 队列丢弃（旧的丢，新的显）
```

```java
// 房间级限流
RateLimiter limiter = RateLimiter.create(500);   // 每房间 500/s
if (limiter.tryAcquire()) {
    gatewayPusher.broadcast(liveId, danmaku);
} else {
    // 拥挤弹幕提示（不真发）
    gatewayPusher.broadcast(liveId, NoticeMessage.crowded());
}
```

## 五、连麦（超低延迟）

```
主播 A ←─WebRTC─→ 媒体服务器（SFU）←─WebRTC─→ 主播 B
                       ↓
                    混流 → CDN 分发（普通观众看 HLS）
```

WebRTC 走 UDP + SRTP，延迟 <500ms；观众仍走 CDN（成本低）。

## 六、监控

```
质量监控：
  - 推流码率/帧率
  - 转码耗时
  - CDN 命中率
  - 观众端卡顿率/起播时间
  - 端到端延迟

业务监控：
  - 同时在线数（Redis 在线 Set）
  - 弹幕量（Kafka 流量）
  - 礼物收入（事件计数）
```

## 七、底层本质

直播本质是**"音视频流低延迟分发 + 实时互动信令削峰"**——核心是带宽（CDN）+ 延迟（协议选择）+ 并发（信令削峰）。

## 常见考点
1. **怎么降低直播延迟**？——WebRTC/SRT 替代 RTMP；LL-HLS 切片更小。
2. **百万并发弹幕怎么扛**？——长连接网关+Kafka 削峰+限流+客户端节流。
3. **直播录制怎么实现**？——转码旁路录制为切片（ts）→ 合成 MP4 → 对象存储。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：弹幕削峰你用 Kafka + 服务端限流 + 客户端节流三层，为什么不只用一层（比如服务端限流就够了）？**

单层削峰扛不住弹幕洪峰。百万直播间 × 每秒千条弹幕 = 百万 QPS，如果只靠服务端限流，Kafka 的写入和网关的接收已经被打满（带宽 + 连接数），限流是"已经到了才拒绝"，治标。三层削峰是"各司其职、逐层过滤"：
1. Kafka 削峰——弹幕先进 MQ，把同步推送解耦成异步消费，消费者按自身容量拉取，生产端的洪峰被 MQ 缓冲。
2. 服务端限流——消费者侧按房间限速（每房间 500/s），同类弹幕聚合（"哈哈哈"合并），把"实际下发量"从十万降到几千。
3. 客户端节流——即使服务端推 500/s，客户端渲染也跟不上（60fps 只能渲染 60 条/秒），客户端按帧合并丢弃旧弹幕。
三层叠加，最终屏幕显示的弹幕是"用户能看清楚的量"，而不是"把网关和带宽打爆的量"。

### 第二层：证据与定位

**Q：直播间卡顿率从 2% 飙到 15%，你怎么定位是 CDN、转码、还是观众端网络问题？**

卡顿排查沿链路分段：
1. CDN 命中率——`CDN 日志`看边缘节点的 cache hit ratio，如果从 95% 掉到 70%，大量回源，观众端拉流慢。常见根因是直播流刚开播（CDN 还没预热）或码率突增。
2. 转码延迟——转码服务监控 `transcode_p99`，如果从 2s 涨到 10s，切片产出延迟，观众端 buffer 空了拉不到新切片。
3. 观众端——客户端上报的 `bufferHealth`（播放器缓冲区），如果大量用户 buffer <1s，是拉流跟不上播放。结合 `networkType`（WiFi/4G）看是否特定网络差。
4. 推流端——主播端推流码率/帧率监控，如果推流本身卡（主播网络差），所有观众都卡。

### 第三层：根因深挖

**Q：弹幕你用 `live-danmaku-{liveId % 32}` 分 32 个 Kafka 分区，但某个超级直播间（liveId=100）的弹幕量占全站 50%，单分区被打爆。怎么解？**

按 `liveId % 32` 分区的问题是"数据倾斜"——超级直播间和普通直播间按 hash 均分，但流量天差地别。单分区被超级直播间的弹幕打爆表现为该分区消费 lag 飙升。深挖解法：
1. **大房间单独分区**——维护一个"大房间列表"（在线人数 >10 万），这些房间的 liveId 直接映射到专属分区（如 partition 0-3 专给大房间，4-31 给普通房间）。大房间动态升降级。
2. **房间内再分片**——超级直播间的弹幕按 `uid % 4` 再分到 4 个子分区，消费端聚合。代价是同一房间弹幕不再全局有序（但弹幕场景可容忍轻微乱序）。
3. **推模式下发过滤**——不是所有弹幕都要推给所有观众，按用户标签（VIP/普通）分流，VIP 弹幕高优全推，普通弹幕抽样推。

### 第四层：方案权衡

**Q：直播延迟你用 RTMP（3-10s），但竞品用 WebRTC（<500ms）。为什么不全部切 WebRTC 追求超低延迟？**

WebRTC 的代价远高于 RTMP：
1. **成本**——WebRTC 是 P2P 或 SFU 架构，每个观众要占用媒体服务器的转发资源（CPU 转码 + 带宽），百万观众意味着百万路 WebRTC 流，服务器成本是 CDN 分发的 5-10 倍。CDN 是"一路流万人拉"（多播），WebRTC 是"万人各一路"（单播）。
2. **弱网适配**——WebRTC 对网络质量敏感，弱网下丢包卡顿比 RTMP/HLS 严重。
3. **生态**——RTMP/HLS 的 CDN 基础设施成熟（阿里/腾讯 CDN 都优化到位），WebRTC 的边缘节点覆盖不足。
权衡：只在"必须超低延迟"的场景用 WebRTC——连麦（主播间互动 <500ms 必须）、答题直播（实时性强）。普通观众观看走 LL-HLS（2-5s 延迟，成本可控）。混合架构：主播间 WebRTC 连麦 → 混流 → CDN 的 HLS 分发给观众。

### 第五层：验证与沉淀

**Q：弹幕削峰上线后，你怎么验证"用户体验没退化"（弹幕量合适、不卡顿、不漏重要内容）？**

削峰的本质是"丢弃部分弹幕"，必须验证丢弃的是"可丢的"。验证维度：
1. 弹幕量合理性——服务端统计"每房间实际下发量/秒"vs"生产量/秒"，下发率应在 30-70%（太多卡 UI，太少用户觉得冷清）。客户端上报"屏幕同时显示弹幕数"，应稳定在 20-50 条。
2. 重要内容不丢——VIP/付费弹幕（礼物/提问）的送达率应 100%。监控 `vip_danmaku_drop_rate`，必须为 0。
3. 卡顿——客户端上报 `fps` 和 `danmaku_render_jank`（弹幕渲染卡顿次数），应不随弹幕量增长。
沉淀：削峰参数（房间限速、聚合窗口、VIP 优先级）接 Apollo 动态调；弹幕量监控按房间分桶（大/中/小房间分别看）；客户端节流逻辑版本化，支持灰度（先 10% 用户验证不卡顿再全量）。

## 结构化回答




**30 秒电梯演讲：** 直播像电视台——主播是记者（推流）、转码是剪辑（多档位）、CDN 是信号塔（分发）、弹幕是观众短信（信令）。

**展开框架：**
1. **推流** — RTMP/SRT → 中心
2. **转码** — HLS/FLV 多档位
3. **分发** — CDN 边缘节点

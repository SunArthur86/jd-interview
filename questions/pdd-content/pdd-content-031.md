# 【拼多多内容】AI Agent 怎么改造内容审核？

> JD 依据："和算法同学挖掘业务问题"、"评价和行家社区"、"系统架构优化"。

## 一、传统审核的局限

```
规则审核：
  - 只能匹配关键词/正则
  - 变体（谐音/拆字/暗语）漏判
  
模型审核：
  - 单任务（涉黄/广告分别训）
  - 改成本高（新违规要标数据+训练）
  - 多模态融合弱

人审：
  - 成本高
  - 慢
```

## 二、AI Agent 改造思路

```
传统：规则 → 模型 → 人工（漏斗）
Agent：规则 → Agent（LLM+工具+多模态）→ 人工（少量疑难）

Agent 能力：
  - LLM 理解语义（暗语/变体/上下文）
  - 多模态（图+文+视频一起判）
  - 工具调用（查敏感词库/黑名单/历史）
  - 推理（结合上下文判断意图）
```

## 三、Agent 架构

```
UGC 内容
   ↓
感知：提取文本/图片/视频/音频
   ↓
决策（LLM）：
  - 上下文理解（商品/用户/历史）
  - 多模态融合（图+文联合判）
  - 工具调用：
    * 查敏感词库（API）
    * 查用户黑名单（API）
    * 查相似历史违规（向量库）
    * 调小模型（涉黄/暴恐分类）
  - 输出：违规类型+置信度+理由
   ↓
执行：
  - 高置信度 → 直接决策（通过/拒绝）
  - 中置信度 → 人审
  - 低置信度 → 通过
```

## 四、Prompt 设计

```
你是拼多多内容审核 Agent。

任务：判断 UGC 内容是否违规。

违规类型：
  - 涉政/涉黄/暴恐/广告/辱骂/诱导/虚假

输入：
  - 文本：{content}
  - 图片描述：{image_caption}（视觉模型生成）
  - 上下文：商品={product}, 用户历史={history}

可用工具：
  - querySensitiveWords(text)：查敏感词库
  - queryBlacklist(uid)：查黑名单
  - searchSimilarCases(content)：查相似违规

输出（JSON）：
{
  "violated": true/false,
  "type": "广告/辱骂/...",
  "confidence": 0.0-1.0,
  "reason": "判断理由"
}

注意：
  - 暗语/谐音/拆字都算违规
  - 结合上下文判断（如商品是母婴，评价涉性是严重违规）
```

## 五、工具调用（Function Calling）

```python
tools = [
    {
        "name": "query_sensitive_words",
        "description": "查文本是否含敏感词",
        "parameters": {"text": "string"}
    },
    {
        "name": "search_similar_violations",
        "description": "向量检索相似违规案例",
        "parameters": {"content": "string", "top_k": "int"}
    }
]

# LLM 决定调哪个工具
response = llm.chat(messages, tools=tools)
# Agent 框架执行工具调用，结果返回 LLM 再决策
```

## 六、多模态审核

```
图片：
  - 视觉模型（CLIP/BLIP）生成描述
  - LLM 结合图文判断
  
视频：
  - 抽帧 → 每帧视觉模型
  - 关键帧 + 音频 ASR → LLM 联合判

音频：
  - ASR 转文字
  - 声纹识别（敏感人物）
  - LLM 判断文字
```

## 七、延迟与成本控制

**痛点**：LLM 推理慢（秒级）+贵，全量调不可承受。

**分级策略**：
```
1. 规则层（毫秒，便宜）：明显违规/正常直接判（80%）
2. 小模型层（百毫秒）：常见违规分类（15%）
3. LLM Agent（秒，贵）：仅疑难/高价值/新型（5%）
```

**优化**：
- 小模型蒸馏（学生模型）
- 批量推理（多条一起）
- 缓存（相似内容命中）
- 异步（不阻塞发布，先 status=0 后审）

## 八、防误判

```
1. 置信度阈值（>0.95 直接判，0.5-0.95 人审，<0.5 通过）
2. 对抗样本测试（红队定期攻击）
3. 人审兜底（关键场景必人审）
4. 用户申诉通道（误判可申诉）
5. 监控指标（误判率/漏判率）
```

## 九、反馈闭环

```
人审结果 → 标注库 → 
  - 加入 RAG 知识库（Agent 检索参考）
  - 微调 LLM（领域适配）
  - 规则补丁（新模式入库）
```

## 十、落地路径

```
阶段 1：LLM 辅助人审（生成审核建议，人确认）
阶段 2：LLM 处理疑难（规则+模型搞不定的）
阶段 3：LLM 替代部分模型（多模态融合）
阶段 4：全 Agent（LLM+工具+流程自动化）
```

## 十一、底层本质

AI Agent 改造审核本质是**"用 LLM 理解+推理+工具调用突破规则/单模型的局限"**——多模态融合+语义理解+工具扩展+人审兜底+反馈闭环；落地关键是分级（成本可控）+防误判+持续优化。

## 常见考点
1. **LLM 审核延迟怎么降**？——小模型蒸馏+分级调用+缓存+异步。
2. **怎么防 LLM 幻觉误判**？——置信度阈值+人审兜底+对抗样本+可解释（输出理由）。
3. **Agent 和传统模型审核怎么协同**？——规则→小模型→Agent→人审分级漏斗。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：用 AI Agent 改造审核，你说 LLM 理解语义强。但 LLM 推理贵（每次几毛钱）又慢（秒级），全量调成本爆炸。为什么还要用 LLM，规则+小模型不够吗？**

规则+小模型在"已知违规模式"上够用，但在"新型/未知变体"上乏力。规则的局限是"只能匹配已知词典"，小模型的局限是"只能识别训练过的类别"。黑产/用户每天都在造新变体（谐音/拆字/暗语/多模态组合），规则词典追不上，小模型要重新标注+训练（周期 1-2 周）。LLM 的价值在于"零样本理解"——它能理解"这个评价在母婴商品下出现性暗示是严重违规"，即使没见过这个具体变体。Agent 的落地不是"全量调 LLM"，而是分级漏斗：规则挡 80%（明显违规/正常），小模型挡 15%（常见类别），LLM 只处理 5% 的疑难/新型。5% 的量级乘以 LLM 成本（每次几分钱）是可承受的，且换来了"新型违规当天识别"的能力（不用等模型重训）。

### 第二层：证据与定位

**Q：Agent 审核上线后，整体准确率提升了，但延迟从 200ms 涨到 1.5s（LLM 推理慢）。你怎么定位是 LLM 推理本身慢还是 Agent 流程编排的开销？**

延迟拆分定位：
1. 链路埋点——Agent 编排（LangGraph）的每个节点（perceive/decide/llm_audit/tool_call）单独打点耗时。如果 LLM 节点占 1.2s，是推理慢；如果 tool_call（查敏感词/向量库）占 800ms，是工具调用慢。
2. LLM 推理监控——看 vLLM 的 `inference_latency` P99。如果单次推理 1s+，是模型大/批处理未优化。检查是否用了 KV Cache、连续批处理、量化。
3. 工具调用——如果 Agent 决策要调 3 个工具（敏感词/黑名单/相似案例），每个 100ms，串行就是 300ms。改成并行（CompletableFuture）能降到 100ms。
4. prompt 长度——如果 prompt 很长（含大量上下文/示例），首 token 延迟（TTFT）高。精简 prompt 或用 Prefix Caching。

### 第三层：根因深挖

**Q：LLM 审核出现幻觉——把正常评价判成违规（误杀），导致用户投诉。根因和解法是什么？**

LLM 幻觉的根因和防幻觉策略：
1. prompt 不清晰——"判断是否违规"太模糊，LLM 对边界 case 倾向保守（宁可判违规）。解法：prompt 里给明确判定标准 + few-shot 示例（正常/违规各几条），让 LLM 学边界。
2. 缺乏事实依据——LLM 凭"感觉"判，不看真实数据。解法：工具调用注入事实——查用户历史（是否屡犯）、查商品类目（母婴 vs 成人用品判准不同）、查相似案例（历史类似内容的判定结果）。
3. 置信度阈值不当——LLM 输出的 confidence 不准（LLM 自评置信度偏高）。解法：不让 LLM 自己报置信度，而是用"多次采样投票"（同一内容跑 3 次，2/3 一致才采纳）或"温度调低"（temperature=0.1 减少随机性）。
4. 兜底——关键场景（如评价上架）LLM 判违规必须人审确认，不直接执行。LLM 只给"建议"，人审决定。

### 第四层：方案权衡

**Q：Agent 的工具调用（查敏感词/黑名单/相似案例）让审核更准，但每个工具是一次 RPC（几十 ms）。Agent 决策要调多个工具，延迟累加。怎么权衡准确性和延迟？**

工具调用的延迟优化：
1. 并行化——独立工具（查敏感词 vs 查黑名单）并行调用，延迟从"串行 N×50ms"降到"并行 max(50ms)"。LangGraph 支持并行节点。
2. 预取——Agent 决策前，先异步预加载用户相关数据（黑名单/历史）到上下文，决策时直接读上下文不等 RPC。
3. 工具合并——把多个查询合并成一个批量接口（如 `/audit/context?uid=&content=` 一次返回敏感词命中+黑名单+相似案例）。
4. 分级调用——不是所有 case 都要调全部工具。先用 LLM 判断"需要哪些工具"（如纯文本评价不用查图片相似案例），按需调。
5. 缓存——用户黑名单/历史短时间内不变，缓存 5 分钟。

### 第五层：验证与沉淀

**Q：你怎么衡量 Agent 审核比传统（规则+模型）审核真的更好？不是"上线了就是成功"。**

Agent 审核的效果衡量：
1. 离线评测——用同一批标注数据（1000 条），对比传统（规则+模型）vs Agent 的准确率/召回率/F1。Agent 应在"新型变体"子集上显著优于传统（传统没见过变体漏判，Agent 零样本识别）。
2. 在线 AB——5% 流量走 Agent，5% 走传统，对比误判率/漏判率/人审率/延迟。Agent 的核心价值是"降低人审率"（疑难 case 自己处理）和"提升新型违规识别"。
3. 成本对比——Agent 虽然单次贵（LLM 推理），但只处理 5% 疏难，总量小。算总成本（人审工资 + 机审算力 + LLM 调用费）对比传统。
4. 持续监控——Agent 的准确率会随 LLM 版本/对抗变体变化，每周评测一次，发现退化触发 prompt/模型更新。
沉淀：Agent 审核必须有"人审兜底"（关键场景 LLM 只给建议不决策）；Agent 的每次判定记录理由（可追溯）；人审结果回流 RAG 知识库（Agent 下次检索参考）。

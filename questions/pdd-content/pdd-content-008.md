# 【拼多多内容】评价库分库分表方案？

> JD 依据："稳定性建设"、"高并发大流量大数据量"。

## 一、为什么分库分表

```
单表瓶颈：
  - 行数 >1000 万：B+ 树变高，查询慢
  - 单库 QPS >5000：磁盘 IO/CPU 吃满
  - 单库故障影响全业务
分库分表解决：
  - 容量：水平拆开，单表可控
  - QPS：分摊到多库
  - 故障：分片隔离
```

## 二、拆分方式

**垂直拆分**：
- 垂直分库：按业务（评价库、直播库、Feed 库）
- 垂直分表：按字段热度（review 基础表 + review_content 大字段表）

**水平拆分**（按分片键）：
```
review 表 → review_0, review_1, ..., review_15（16 张表，4 库×4 表）
分片键：product_id % 16 → 路由到对应表
```

## 三、分片策略

| 策略 | 做法 | 优缺点 |
|------|------|--------|
| Range | 按区间（时间/ID 段） | 易扩容，但热点 |
| Hash | `product_id % N` | 均匀，扩容痛 |
| 一致性 Hash | 节点环 | 扩容迁移少 |
| 雪花算法 | 自带分片信息 | ID 即路由 |

**评价场景**：按 `product_id` Hash 分 16 表（4 库 4 表）
**Feed 场景**：按 `uid` Hash（同一用户的 Feed 在同一分片，写入高效）

## 四、ShardingSphere 配置

```yaml
spring:
  shardingsphere:
    datasource:
      names: ds0,ds1,ds2,ds3
      ds0: { type: HikariDataSource, ... }
      ...
    rules:
      sharding:
        tables:
          review:
            actualDataNodes: ds${0..3}.review_${0..3}    # 4 库 4 表
            databaseStrategy:
              standard:
                shardingColumn: product_id
                shardingAlgorithmName: review_db_mod
            tableStrategy:
              standard:
                shardingColumn: product_id
                shardingAlgorithmName: review_table_mod
        shardingAlgorithms:
          review_db_mod:
            type: MOD
            props: { sharding-count: 4 }
          review_table_mod:
            type: MOD
            props: { sharding-count: 4 }
        keyGenerators:
          snowflake:
            type: SNOWFLAKE
            props: { worker-id: 1 }
```

## 五、难点与解法

**1. 跨片查询**：
```sql
-- 不带分片键：广播到所有分片合并（贵）
SELECT * FROM review WHERE uid = 123;          -- product_id 不在
-- 解法：建异构索引（uid → product_id 映射入 ES/HBase）
```

**2. 分页**：
```
LIMIT 100000, 10 跨片：每个分片取 100010 行→合并→取 10 行
解法：禁止深翻页；或游标（last_id）
```

**3. 分布式事务**：
- 弱一致：本地消息表 + 异步补偿
- 强一致：Seata AT/TCC

**4. 扩容（4 库 → 8 库）**：
- 双写新老库（按新规则）→ 数据同步 → 切流量 → 下老库
- 或一致性 hash 减少迁移量

**5. 全局 ID**：雪花算法（worker + 时间戳 + 序列）

## 六、内容场景实战

```
评价库分片（拼多多）：
  分片键：product_id
  分片数：64 库 × 4 表 = 256 张表
  单表数据：~500 万（可控）
  查询：商品评价页（带 product_id，直接路由）
  
Feed 库分片：
  分片键：uid
  查询：用户主页 Feed（带 uid，直接路由）

直播库分片：
  分片键：live_id
  查询：直播间详情/弹幕历史
```

## 七、底层本质

分库分表本质是**"用分片键把数据水平拆开破解容量/QPS 瓶颈"**——核心是选好分片键，难点是跨片/事务/扩容，往往配合异构索引（ES/HBase）补足。

## 常见考点
1. **分片键选错了怎么办**？——异构索引（双写 ES）+ 异步同步。
2. **怎么不停机扩容**？——双写新老库+数据回填+灰度切流+下线老库。
3. **ShardingSphere 和 MyCat 区别**？——前者客户端/中间件双模式（更轻量），后者是中间件。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：评价表按 `product_id` 分片，但"我的评价"（按 uid 查）查不了，为什么当初不按 uid 分片？**

分片键只能选一个，选 `product_id` 还是 `uid` 取决于"最高频且最重的查询"。拼多多评价页（按 product_id 查列表）是 C 端流量入口，QPS 是"我的评价"（按 uid）的 10 倍以上，且是购买决策关键路径必须低延迟。按 product_id 分片让评价页直接路由到单分片（O(1)），而"我的评价"走 ES 异构索引兜底（频率低，可容忍多一跳）。如果反过来按 uid 分片，评价页就要广播到全部分片（256 张表）合并排序，P99 直接炸。本质是"用主链路的极致优化换次链路的额外开销"。

### 第二层：证据与定位

**Q：某商品评价页偶发返回空，但 DB 里明明有这条评价，你怎么定位是不是分片路由错了？**

分片路由错会导致"数据在分片 A，查到了分片 B"。排查：
1. ShardingSphere 日志——开启 `spring.shardingsphere.props.sql-show: true`，打印实际路由到哪个库哪个表的 SQL。看路由的 `actual SQL` 是否和数据所在分片一致。
2. 反推分片——拿到 product_id，手动算 `product_id % 4`（库）和 `(product_id / 4) % 4`（表），对比数据实际所在分片。如果不一致，说明分片算法配置错了（如 ShardingSphere 的 `MOD` 算法 sharding-count 配成了 8 而建表是 4）。
3. 数据是不是被写错分片了——查所有 256 张表 grep 这个 product_id，如果数据在错误分片，说明历史数据迁移时分片规则变了但没重算。

### 第三层：根因深挖

**Q：分页查询 `LIMIT 100, 10` 在分库分表下变慢了，比单表慢 10 倍，根因是什么？**

根因是分库分表下"全局排序分页"必须归并。单表 `LIMIT 100, 10` 直接取。分库分表下，MySQL 不知道哪 10 条是全局第 100-110，只能每个分片都取 `LIMIT 0, 110`（每个分片取前 110），汇总到应用层（ShardingSphere）做内存归并排序再取第 100-110。256 个分片 × 110 = 28160 行回传 + 内存排序，比单表慢 10 倍。更深翻页更糟——`LIMIT 100000, 10` 每分片取 100010，总计 2560 万行。根治：禁止深翻页，改游标分页（`WHERE product_id=? AND create_time < ? ORDER BY create_time DESC LIMIT 10`），每个分片只取 10 条归并。

### 第四层：方案权衡

**Q：评价库要从 4 库 4 表（16 分片）扩到 8 库 4 表（32 分片），你选"双写迁移"还是"一致性 hash"？权衡是什么？**

两种扩容方案的本质区别：
1. **双写迁移**——新老库同时写，历史数据用脚本按新规则回填，校验一致后切流量到新库，下老库。优点是简单可控，缺点是迁移周期长（评价数据几十亿，回填要几小时）、双写期间要处理冲突。适合分片规则变化大的场景。
2. **一致性 hash**——分片键 hash 到环上，扩容时只有落在新节点区间的数据要迁移，迁移量约 1/N。优点是迁移少，缺点是 ShardingSphere 的 MOD 算法不支持一致性 hash，要自己实现。
拼多多评价场景选双写迁移更稳——因为分片数从 16 翻倍到 32，传统取模下几乎每条数据都要重新分布（`id % 16` ≠ `id % 32`），一致性 hash 也省不了多少迁移。双写的好处是"可灰度、可回滚"——先双写观察 1 天，校验数据一致再切流，有问题立刻切回老库。

### 第五层：验证与沉淀

**Q：扩容后你怎么验证 256 张表的数据完整、没丢没重？**

数据校验是分库分表扩容最易翻车的环节。验证：
1. 分片级 count——新老库每个分片 `SELECT COUNT(*)`，对比。但 count 慢（大表），可抽样：每分片按 `id % 100 = 0` 抽样对比。
2. 全量 checksum——`SELECT BIT_XOR(CRC32(CONCAT_WS('#', id, product_id, content))) FROM review WHERE product_id BETWEEN ? AND ?`，新老库分段算校验和对比。
3. 业务校验——切流后跑全量业务回归（评价页、我的评价、评分聚合），对比指标（评分、数量）是否和新库一致。
4. 灰度对比——10% 流量读新库 90% 读老库，对比返回结果，无差异再全切。
沉淀：分片键设计规范——必须覆盖 Top 3 查询；扩容必须双写 + 校验 + 灰度，禁止直接切换；异构索引（ES）必须同步扩容感知，否则跨片查询打错索引。

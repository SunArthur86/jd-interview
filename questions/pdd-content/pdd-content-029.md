# 【拼多多内容】预案回滚与钱学森工程理论？

> JD 依据："稳定性建设"、"系统高可用/扩展性"。

## 一、灰度发布

**金丝雀发布**：
```
新版本先发布 1-2 台机器 → 观察（监控指标）→ 逐步扩大 → 全量
  优点：影响小，发现问题快回滚
```

**蓝绿发布**：
```
两套环境（蓝/绿），新版本部署到备用环境 → 切流量 → 测试 → 切回/留新
  优点：瞬时切换+回滚快
  缺点：资源双倍
```

**滚动发布**：
```
逐台替换：旧实例下线一台 → 新实例上线一台 → 直到全替换
  优点：资源占用平稳
  缺点：滚动期间两版本并存（需兼容）
```

**流量灰度**（按维度）：
```java
// 按用户 ID hash
if (Math.abs(uid.hashCode()) % 100 < 5) {
    // 5% 流量走新版本
}
// 按地域/设备/白名单
```

## 二、回滚三层

**1. 版本回滚**（代码）：
```
K8s 一键回滚：
  kubectl rollout undo deployment/review-service --to-revision=N

CI/CD 流水线保留历史镜像：
  镜像版本号 → 一键切回旧版本
```

**2. 数据回滚**（最难）：
```
DDL（建表/改字段）往往不可逆！
策略：
  - 双写兼容（新旧字段并存一段时间）
  - 灰度上线新代码（读新写旧）
  - 确认稳定后再清理老字段
  - 出问题：禁用新代码读，回退到老字段

DML（数据改错）：
  - 备份恢复（耗时）
  - binlog 反向回放（精确）
  - 业务补偿（脚本修复）
```

**3. 配置回滚**：
```
配置中心（Apollo/Nacos）版本管理 → 一键回滚
```

## 三、预案体系

**故障分级**：
| 级别 | 影响 | 响应 |
|------|------|------|
| P0 | 全站不可用 | 全员+老板+30 分钟恢复 |
| P1 | 核心功能不可用 | 团队+1 小时恢复 |
| P2 | 部分功能不可用 | 值班+4 小时恢复 |
| P3 | 体验问题 | 工作日修复 |

**预案剧本**（每个场景一份）：
```
场景：直播信令网关挂
现象：弹幕/礼物无法发送
影响范围：所有直播
应对：
  1. 网关自动重启（K8s）
  2. 切到备用机房（DNS）
  3. 限流降级（弹幕显示拥挤）
  4. 通知主播+观众
负责人：xxx
电话/钉钉：xxx
```

**一键开关**（预案工具化）：
```java
// 配置中心控制
@Switch("live.danmaku.degrade")
private boolean danmakuDegrade = false;

if (danmakuDegrade) {
    // 弹幕降级：只显示提示，不真发
    return NoticeMessage.crowded();
}
```

## 四、演练（混沌工程）

**Chaos Engineering**（Netflix 提出）：
- 主动注入故障，验证系统韧性
- 在生产或类生产环境做

**演练类型**：
- 杀进程（K8s pod 故障）
- 断网（网络分区）
- 限 CPU/内存
- DB 慢/挂
- 依赖服务挂

**演练流程**：
```
1. 假设：依赖 A 挂，服务仍可用（降级）
2. 注入：在演练环境杀掉 A
3. 观察：是否如预期降级
4. 修复：发现的问题
5. 自动化：定期演练（GameDay）
```

## 五、钱学森工程理论

**核心思想**：
- **系统工程**：把复杂系统分解为子系统+接口，整体设计
- **可靠性设计**：从设计阶段就考虑可靠性（FMEA/容错）
- **冗余**：关键组件冗余备份（主从/多活）
- **测试充分**：仿真+实验（仿真计算→实物实验）

**应用到稳定性**：
```
系统工程：
  - 把稳定性分解为：限流/降级/熔断/隔离/监控/预案
  - 各组件设计+协同（不是堆砌）

可靠性设计：
  - FMEA（故障模式与影响分析）：提前识别单点
  - 容错设计：单组件故障不影响整体

冗余：
  - 多机房部署（异地多活）
  - 主从/集群（Redis/MySQL）
  - 服务多副本（K8s）

测试：
  - 全链路压测（仿真）
  - 混沌演练（注入故障）
  - 灰度发布（小流量验证）
```

**钱学森的话**："我们不能就事论事地解决一个问题，而应该从总体上考虑问题。" → 稳定性是系统工程，不是单点优化。

## 六、内容场景实战

**直播开播预案**：
```
场景：热门主播开播流量洪峰
预案：
  1. 提前 30 分钟扩容（HPA + 手动）
  2. 缓存预热（直播间统计/在线列表）
  3. CDN 带宽预留
  4. 限流阈值调高（活动期临时）
  5. 监控加强（值班盯盘）
  6. 出问题：限流+降级+切备用机房
```

**评价服务故障**：
```
现象：评价查询挂
影响：商品页评价区无内容
应对：
  1. 商品页降级（评价区显示"加载中"+默认评分）
  2. 故障机器重启/扩容
  3. DB/Redis 排查
  4. 修复后逐步放开
```

## 七、底层本质

预案回滚本质是**"系统工程思想下的故障预防+控制+恢复"**——灰度控制影响、回滚快速恢复、预案预先准备、演练验证韧性；钱学森理论告诉我们稳定性是系统工程，需整体设计而非单点堆砌。

## 常见考点
1. **怎么做零停机发布**？——蓝绿/滚动+灰度+健康检查+回滚预案。
2. **数据库 DDL 怎么安全变更**？——扩展（兼容旧）→ 双写 → 灰度 → 收缩。
3. **混沌工程怎么落地**？——从演练环境开始，逐步到生产；先单组件，后组合。

# 【拼多多内容】预案回滚与钱学森工程理论？

> JD 依据："稳定性建设"、"系统高可用/扩展性"。

## 一、灰度发布

**金丝雀发布**：
```
新版本先发布 1-2 台机器 → 观察（监控指标）→ 逐步扩大 → 全量
  优点：影响小，发现问题快回滚
```

**蓝绿发布**：
```
两套环境（蓝/绿），新版本部署到备用环境 → 切流量 → 测试 → 切回/留新
  优点：瞬时切换+回滚快
  缺点：资源双倍
```

**滚动发布**：
```
逐台替换：旧实例下线一台 → 新实例上线一台 → 直到全替换
  优点：资源占用平稳
  缺点：滚动期间两版本并存（需兼容）
```

**流量灰度**（按维度）：
```java
// 按用户 ID hash
if (Math.abs(uid.hashCode()) % 100 < 5) {
    // 5% 流量走新版本
}
// 按地域/设备/白名单
```

## 二、回滚三层

**1. 版本回滚**（代码）：
```
K8s 一键回滚：
  kubectl rollout undo deployment/review-service --to-revision=N

CI/CD 流水线保留历史镜像：
  镜像版本号 → 一键切回旧版本
```

**2. 数据回滚**（最难）：
```
DDL（建表/改字段）往往不可逆！
策略：
  - 双写兼容（新旧字段并存一段时间）
  - 灰度上线新代码（读新写旧）
  - 确认稳定后再清理老字段
  - 出问题：禁用新代码读，回退到老字段

DML（数据改错）：
  - 备份恢复（耗时）
  - binlog 反向回放（精确）
  - 业务补偿（脚本修复）
```

**3. 配置回滚**：
```
配置中心（Apollo/Nacos）版本管理 → 一键回滚
```

## 三、预案体系

**故障分级**：
| 级别 | 影响 | 响应 |
|------|------|------|
| P0 | 全站不可用 | 全员+老板+30 分钟恢复 |
| P1 | 核心功能不可用 | 团队+1 小时恢复 |
| P2 | 部分功能不可用 | 值班+4 小时恢复 |
| P3 | 体验问题 | 工作日修复 |

**预案剧本**（每个场景一份）：
```
场景：直播信令网关挂
现象：弹幕/礼物无法发送
影响范围：所有直播
应对：
  1. 网关自动重启（K8s）
  2. 切到备用机房（DNS）
  3. 限流降级（弹幕显示拥挤）
  4. 通知主播+观众
负责人：xxx
电话/钉钉：xxx
```

**一键开关**（预案工具化）：
```java
// 配置中心控制
@Switch("live.danmaku.degrade")
private boolean danmakuDegrade = false;

if (danmakuDegrade) {
    // 弹幕降级：只显示提示，不真发
    return NoticeMessage.crowded();
}
```

## 四、演练（混沌工程）

**Chaos Engineering**（Netflix 提出）：
- 主动注入故障，验证系统韧性
- 在生产或类生产环境做

**演练类型**：
- 杀进程（K8s pod 故障）
- 断网（网络分区）
- 限 CPU/内存
- DB 慢/挂
- 依赖服务挂

**演练流程**：
```
1. 假设：依赖 A 挂，服务仍可用（降级）
2. 注入：在演练环境杀掉 A
3. 观察：是否如预期降级
4. 修复：发现的问题
5. 自动化：定期演练（GameDay）
```

## 五、钱学森工程理论

**核心思想**：
- **系统工程**：把复杂系统分解为子系统+接口，整体设计
- **可靠性设计**：从设计阶段就考虑可靠性（FMEA/容错）
- **冗余**：关键组件冗余备份（主从/多活）
- **测试充分**：仿真+实验（仿真计算→实物实验）

**应用到稳定性**：
```
系统工程：
  - 把稳定性分解为：限流/降级/熔断/隔离/监控/预案
  - 各组件设计+协同（不是堆砌）

可靠性设计：
  - FMEA（故障模式与影响分析）：提前识别单点
  - 容错设计：单组件故障不影响整体

冗余：
  - 多机房部署（异地多活）
  - 主从/集群（Redis/MySQL）
  - 服务多副本（K8s）

测试：
  - 全链路压测（仿真）
  - 混沌演练（注入故障）
  - 灰度发布（小流量验证）
```

**钱学森的话**："我们不能就事论事地解决一个问题，而应该从总体上考虑问题。" → 稳定性是系统工程，不是单点优化。

## 六、内容场景实战

**直播开播预案**：
```
场景：热门主播开播流量洪峰
预案：
  1. 提前 30 分钟扩容（HPA + 手动）
  2. 缓存预热（直播间统计/在线列表）
  3. CDN 带宽预留
  4. 限流阈值调高（活动期临时）
  5. 监控加强（值班盯盘）
  6. 出问题：限流+降级+切备用机房
```

**评价服务故障**：
```
现象：评价查询挂
影响：商品页评价区无内容
应对：
  1. 商品页降级（评价区显示"加载中"+默认评分）
  2. 故障机器重启/扩容
  3. DB/Redis 排查
  4. 修复后逐步放开
```

## 七、底层本质

预案回滚本质是**"系统工程思想下的故障预防+控制+恢复"**——灰度控制影响、回滚快速恢复、预案预先准备、演练验证韧性；钱学森理论告诉我们稳定性是系统工程，需整体设计而非单点堆砌。

## 常见考点
1. **怎么做零停机发布**？——蓝绿/滚动+灰度+健康检查+回滚预案。
2. **数据库 DDL 怎么安全变更**？——扩展（兼容旧）→ 双写 → 灰度 → 收缩。
3. **混沌工程怎么落地**？——从演练环境开始，逐步到生产；先单组件，后组合。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：灰度发布你选金丝雀（1-2 台先上），但有人说蓝绿更安全（可瞬时切回）。为什么拼多多内容场景选金丝雀？**

蓝绿的代价是"双倍资源"（两套环境同时跑）。拼多多内容业务（评价/直播/Feed）服务多、流量大，蓝绿意味着每个服务都要双倍机器，成本不可承受。金丝雀只多 1-2 台实例（增量成本低），且通过"逐步放量 + 监控观察"达到接近蓝绿的安全性——发现问题立即下线金丝雀实例（流量自动回到老版本，K8s 滚动更新的 rollback）。蓝绿的"瞬时切回"优势在内容场景不明显（金丝雀下线也只需几十秒），而双倍资源的代价是实打实的。选金丝雀的本质是"用更长的观察时间换更低的资源成本"，内容业务变更频繁，蓝绿的双倍成本长期累积惊人。

### 第二层：证据与定位

**Q：灰度发布评价服务新版本，1% 流量观察了 1 小时指标正常，全量后却大面积报错。你怎么定位是灰度观察期漏了什么？**

灰度漏检的根因：
1. 流量分桶不均——灰度 1% 流量如果按 `uid % 100 < 1` 分桶，可能只覆盖了某类用户（如 uid 较小的老用户），他们的行为模式不代表全量。全量后新用户/特殊用户触发新代码的 bug。验证：看灰度期间的 `uid 分布`是否均匀。
2. 流量太小没暴露——1% 流量 QPS 可能只有几十，没触发并发问题（如线程竞争、连接池耗尽）。全量后 QPS 几千，并发 bug 暴露。
3. 数据依赖——新代码依赖新字段（如 `review.is_expert`），灰度期间碰巧只查了老数据（无该字段），全量后查新数据报错。
4. 时间窗口——灰度 1 小时可能没覆盖"直播高峰""审核高峰"等特定时段的业务模式。全量后高峰时段触发新代码的边界。

### 第三层：根因深挖

**Q：数据库 DDL 变更（给 review 表加字段）后，回滚代码时发现新代码已经写了新字段，回退到老代码读不到，报错。根因和解法是什么？**

这是 DDL 不可逆的经典坑。根因：代码回滚但 DDL 不能回滚（删字段会丢数据）。解法是"DDL 正向兼容"：
1. 扩展（加字段）——加字段用 `ALTER TABLE ADD COLUMN`（带默认值），老代码不读新字段也能跑（字段存在但被忽略），新代码读新字段。回滚老代码无影响。
2. 双写过渡——新代码"读新写旧"（写时同时写新旧字段），灰度验证新字段数据正确。
3. 收缩（删老字段）——确认新代码稳定后，分批删老字段。删之前必须确认无代码读老字段。
4. 回滚策略——DDL 变更的回滚不是"删新字段"，而是"代码回退到不依赖新字段的版本"。所以 DDL 变更必须配"可兼容老代码"的发布顺序：先发兼容代码（不读新字段）→ 加字段 → 发读写新字段的代码。出问题回滚到"兼容代码"，字段还在但不用，无害。

### 第四层：方案权衡

**Q：混沌工程你建议在生产做（发现真实问题），但老板担心"生产注入故障会真把业务搞挂"。你怎么权衡？**

生产混沌工程的核心原则是"可控爆炸半径"：
1. 从非核心开始——先对非核心服务（如统计、推荐侧边栏）注入故障，验证降级，不影响主交易。
2. 控制影响范围——注入故障时限定在"1 个实例 / 1% 流量"，即使降级失败也只影响小部分用户。用 K8s 的 chaos-mesh 限定 namespace/pod。
3. 监控 + 自动终止——演练时紧盯核心指标（错误率/延迟），超阈值自动终止故障注入（kill chaos job）。
4. 非生产先验证——先在预发环境跑一遍（验证降级逻辑），确认无误再上生产的小范围。
5. 业务时段选择——在低峰期（凌晨）做，减少用户影响。
权衡：生产混沌的价值是"发现生产特有的问题"（真实流量、真实依赖、真实配置），预发环境发现不了这些。风险可控的前提下（小范围 + 自动终止 + 低峰期），收益大于风险。拼多多是每周一次小范围混沌（GameDay），发现问题持续优化，逐步扩大注入范围。

### 第五层：验证与沉淀

**Q：你怎么验证预案（如"评价服务挂时降级显示默认评分"）真的能执行，而不是文档里写了但实际没接？**

预案验证靠"定期演练 + 自动化检查"：
1. 混沌演练——定期（每月）在生产或预发 kill 评价服务，验证商品页是否如预期降级（显示默认评分而非报错）。如果没降级，说明降级开关没接或 fallback 逻辑有 bug。
2. 自动化巡检——脚本定期检查"降级开关是否存在且可切换"（如 Apollo 配置项存在）、"fallback 方法是否有实现"（代码扫描 `@SentinelResource` 是否配了 `fallback` 且方法非空）。
3. 预案卡片——每个预案有"触发条件 + 应对步骤 + 负责人 + 验证记录"，每次演练后更新"上次验证时间"。
4. 灾备切换——异地多活的"DNS 切换"预案，定期（每季度）演练切流（先切 1% 流量到备机房验证）。
沉淀：预案平台化（一键执行预案 + 自动验证效果）；混沌工程纳入 CI/CD（每次发布前跑基础混沌用例）；预案文档版本化，故障复盘后必更新预案。

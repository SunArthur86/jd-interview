# 【拼多多内容】AI Infra 平台怎么建设？

> JD 依据："和算法同学挖掘业务问题"、"系统架构优化"、"稳定性建设"。

## 一、AI Infra 全景

```
┌────────────── 应用层 ──────────────┐
│ 内容审核 │ 智能客服 │ 推荐 │ 搜索  │
└──────────────────┬─────────────────┘
                   │
┌──────────── 平台层（AI Infra）─────┐
│ 训练平台 │ 推理平台 │ 数据平台      │
│ 评测平台 │ 实验平台 │ 部署平台      │
└──────────────────┬─────────────────┘
                   │
┌──────────── 基础设施 ──────────────┐
│ GPU 集群 │ K8s │ 对象存储 │ 监控   │
└────────────────────────────────────┘
```

## 二、训练平台

**GPU 调度**：
```
K8s + device plugin（NVIDIA GPU）
调度策略：
  - 抢占式（高优先级抢低）
  - 共享（多任务分时）
  - 弹性（空闲时扩）
  
资源池：
  - 训练池（A100/H100，贵，专用）
  - 推理池（A10/T4，性价比）
  - 开发池（V100/T4，实验）
```

**分布式训练**：
```
数据并行（DDP）：每卡完整模型+部分数据
模型并行：
  - 张量并行（层内切）
  - Pipeline 并行（层间切）
框架：DeepSpeed/Megatron-LM/Ray
```

**训练任务管理**：
```
- 任务队列（优先级）
- 断点续训（checkpoint）
- 超参搜索（Optuna/Ray Tune）
- 训练监控（loss/metrics）
```

## 三、推理平台

**模型服务化**：
```
框架：
  - vLLM（通用首选）
  - TensorRT-LLM（Nvidia 极致）
  - Triton（多模型多框架）
  - TF Serving（TensorFlow）

部署：
  - K8s + GPU 节点池
  - 弹性扩容（按 QPS）
  - 金丝雀/蓝绿发布
  - A/B 路由
```

**模型仓库**：
```
- 版本管理（v1.0/v1.1）
- 元数据（输入/输出/延迟/精度）
- 灰度比例
- 回滚按钮
```

## 四、数据平台

**标注平台**：
```
- 任务管理（标注/审核/复审）
- 多人标注+一致性
- 标注工具（文本/图片/视频）
- 标注质量监控
```

**特征平台**（推荐/搜索）：
```
离线：Hadoop/Hive（T-1 特征）
实时：Flink + Redis（实时特征）
特征注册+共享+版本
```

**数据版本**：
```
- 训练数据版本化（DVC）
- 模型-数据关联（追溯）
- 数据漂移检测
```

## 五、评测平台

**离线评测**：
```
benchmark 数据集（标注）
自动指标：
  - 准确率/召回率/F1
  - BLEU/ROUGE（生成）
  - CTR/停留时长（推荐）
LLM-as-Judge（强模型评弱模型）
```

**在线 AB**：
```
流量分桶（hash uid）
指标对比（CTR/解决率/满意度）
显著性检验（p-value）
决策：全量/回滚/迭代
```

**红队/对抗**：
```
- 故意构造攻击样本
- 找模型漏洞
- 持续改进
```

## 六、部署平台

**模型上线流程**：
```
训练 → 离线评测 → 灰度（5%）→ 监控 → 全量
   ↓                ↓
   失败          指标差
   回练          回滚
```

**版本管理**：
```
模型版本 + Prompt 版本 + 数据版本 + 评测版本
回滚：一键回到上版本（含所有关联）
```

## 七、实验平台

**实验管理**：
```
- 实验登记（假设/参数/结果）
- 对比（A/B/C 多组）
- 复现（环境+数据+代码快照）
- 知识库（沉淀经验）
```

**超参搜索**：
```
Optuna/Ray Tune
网格/随机/贝叶斯优化
```

## 八、内容场景应用

**1. 审核模型迭代**：
```
数据：人审标注 → 数据平台
训练：审核模型（NLP/视觉）→ 训练平台
评测：benchmark + 在线 AB → 评测平台
部署：vLLM 服务化 → 推理平台
监控：误判率/延迟 → 监控
```

**2. 推荐模型迭代**：
```
特征：用户行为实时特征 → 特征平台
训练：DIN/DeepFM → 训练平台
评测：CTR AB → 评测平台
部署：TF Serving → 推理平台
```

**3. LLM Agent 迭代**：
```
Prompt：版本管理
模型：基础模型 + 微调
评测：Agent benchmark
部署：vLLM + Agent 编排
```

## 九、监控与成本

**监控**：
```
业务：
  - 模型准确率/召回率
  - 在线指标（CTR/解决率）
  - 数据漂移
  
技术：
  - GPU 利用率（>70%）
  - 推理延迟
  - 服务可用性
  - 成本（token/GPU 时）
```

**成本控制**：
```
- GPU 共享（多任务分时）
- 模型蒸馏（小模型替代）
- 量化（INT8/INT4）
- 推理优化（vLLM）
- 分级调用（简单走小模型）
```

## 十、底层本质

AI Infra 本质是**"把模型全链路（训练/推理/数据/评测/部署）平台化"**——让算法同学专注模型，工程保证效率/质量/成本；内容场景需特别关注 GPU 调度（贵）+实时特征（推荐）+在线 AB（效果量化）。

## 常见考点
1. **GPU 怎么高效调度**？——K8s+device plugin+抢占式+共享+弹性。
2. **模型上线怎么保证质量**？——离线评测+灰度+在线 AB+回滚。
3. **特征平台怎么设计**？——离线（Hive T-1）+实时（Flink+Redis）+注册共享。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：AI Infra 平台你说"让算法专注模型"，但算法同学自己也能搭训练环境（pip install + Jupyter），为什么还需要平台？平台比个人环境强在哪？**

个人环境的痛点和平台的价值：
1. 资源——算法同学自己搭只能用单机单卡（或抢几台机器），平台调度整个 GPU 集群（几百卡），训练任务排队/抢占/共享，利用率从 30%（个人闲置）提到 70%+。
2. 复现——个人环境的依赖/数据/代码版本靠手动记录，换机器就跑不通。平台固化环境（Docker 镜像）+数据版本（DVC）+代码快照（git），一键复现。
3. 协作——个人环境的模型/特征无法共享，每个算法重复造轮子。平台的模型仓库/特征注册中心让大家复用，新同学能拉别人的 baseline 跑。
4. 上线——个人训完模型要手动部署（拷权重、写服务代码），平台一键训练→评测→部署→灰度，闭环。
5. 成本——GPU 贵（A100 单卡 2 万美元），平台统一调度+共享+抢占，把资源利用率榨干，公司省钱。
结论：小团队（几个算法）个人环境够用，团队大了（几十算法+多业务线）必须平台，否则资源浪费+协作低效+上线混乱。

### 第二层：证据与定位

**Q：推荐模型上线后 CTR 掉了 5%。你怎么用 AI Infra 的监控+实验平台定位是特征问题还是模型问题？**

分层定位：
1. 数据漂移检测——特征平台的"特征监控"看实时特征分布是否变了（如用户点击序列的平均长度从 10 变 5）。分布漂移说明上游数据问题（埋点丢/ETL 错），不是模型问题。
2. 特征一致性——离线训练特征（Hive T-1）vs 在线推理特征（Flink+Redis）是否一致。常见 bug：离线用了"未来特征"（训练时能看到当天点击，推理时看不到），上线 CTR 必掉。跑一致性校验（离线/在线同用户同时间的特征对比）。
3. 模型版本——实验平台对比当前模型 vs 上个版本，如果换模型后 CTR 掉，是模型问题（过拟合/训练数据污染）。
4. 在线 AB——AB 平台看实验组（新模型）vs 对照组（旧模型）的 CTR，p-value <0.05 说明差异显著（不是波动）。
5. case 分析——抽 100 条实验组点击率低的样本，看特征值是否异常（某特征缺失/类型错）。

### 第三层：根因深挖

**Q：GPU 利用率你监控发现只有 40%，远低于 70% 的目标。根因是什么？怎么提上去？**

GPU 利用率低的根因和对策：
1. 数据加载瓶颈——GPU 算得快但数据喂不上去（CPU 预处理慢/IO 瓶颈）。现象：GPU 等待时间长，SM 利用率低。对策：dataloader 用多进程+prefetch（PyTorch 的 `num_workers=8, pin_memory=True`），数据存本地 SSD（非网络盘）。
2. batch 太小——显存没塞满，GPU 算力没用满。对策：开梯度累积或混合精度（FP16）省显存，把 batch size 调大。
3. 任务排队空转——任务间切换有 idle。对策：vLLM 的连续批处理让 GPU 持续满载；训练用多任务共享（一卡跑多 job 分时）。
4. 模型太小——7B 模型单卡算不满（GPU 算力过剩）。对策：一卡多模型实例（MIG 切分），或多模型共享一卡（时分）。
5. 通信瓶颈——分布式训练的 all-reduce 慢（卡间带宽不够），GPU 等通信。对策：用 NVLink/NVSwitch（卡内高速互联），梯度压缩（减少通信量）。
根本：GPU 利用率 = 计算时间 / 总时间。低利用率要么"算得不够"（batch 小/模型小），要么"等得太多"（等数据/等通信/等任务切换），对症下药。

### 第四层：方案权衡

**Q：特征平台你说"离线 Hive T-1 + 实时 Flink+Redis"。但 T-1 的离线特征有一天的延迟，推荐场景用户行为实时变化，离线特征不够新鲜。怎么权衡？**

离线 vs 实时特征的权衡：
1. 离线（T-1）——准（批处理全量计算，无丢数据）、全（能算复杂特征如"30 天点击序列"）、稳。但不新鲜（昨天的事今天才更新）。
2. 实时（Flink+Redis）——新鲜（秒级更新）、响应快。但贵（Flink 常驻计算+Redis 存）、可能不准（实时流丢数据/乱序）、特征简单（算不了 30 天的）。
3. 权衡策略——长周期特征（30 天购买偏好/用户画像）用离线（变化慢，T-1 够）；短周期特征（最近 10 分钟点击/实时 session）用实时（变化快，必须新鲜）。两者拼成"用户特征向量"喂给模型。
4. 混合架构——离线特征写 Hive + 同步到 Redis（T-1 更新）；实时特征 Flink 写 Redis（秒级更新）。推理时 Redis 一次读全（离线+实时），延迟 <10ms。
5. 演进——先离线（MVP，便宜），看效果再补实时（按场景，贵的只给高价值特征）。不要一上来全实时（成本爆炸）。

### 第五层：验证与沉淀

**Q：你怎么证明 AI Infra 平台"让模型上线效率提升 3 倍"？这种工程平台的 ROI 怎么量化？**

平台 ROI 的量化维度：
1. 效率——上线周期：建平台前从训完模型到上线平均 2 周（手动部署+测试），建平台后 2 天（一键部署+灰度），效率提升 7 倍。
2. 成本——GPU 利用率：建平台前 30%（各自为政，闲置多），建平台后 70%（统一调度+共享），单位算力成本降一半。年省 X 万美元（GPU 数 × 单价 × 利用率提升）。
3. 质量——线上事故：建平台前每月 2 次（模型退化无监控），建平台后每季度 1 次（评测 gate + 监控告警），稳定性提升。
4. 复用——特征/模型复用率：建平台前每个算法重复造特征，建平台后特征注册中心，新项目复用率 60%（少写一半特征工程）。
5. 覆盖——接入业务线数：从 2 个（推荐/搜索）扩到 6 个（加审核/客服/广告/风控），平台边际成本递减。
沉淀：平台看板（上线周期/GPU 利用率/事故数/复用率）给管理层看 ROI；每个模型上线走平台（不准旁路上线），保证数据真实；平台本身的 SLO（可用性 99.9% + 部署成功率 99%）作为平台质量指标。

# 【拼多多内容】AI Infra 平台怎么建设？

> JD 依据："和算法同学挖掘业务问题"、"系统架构优化"、"稳定性建设"。

## 一、AI Infra 全景

```
┌────────────── 应用层 ──────────────┐
│ 内容审核 │ 智能客服 │ 推荐 │ 搜索  │
└──────────────────┬─────────────────┘
                   │
┌──────────── 平台层（AI Infra）─────┐
│ 训练平台 │ 推理平台 │ 数据平台      │
│ 评测平台 │ 实验平台 │ 部署平台      │
└──────────────────┬─────────────────┘
                   │
┌──────────── 基础设施 ──────────────┐
│ GPU 集群 │ K8s │ 对象存储 │ 监控   │
└────────────────────────────────────┘
```

## 二、训练平台

**GPU 调度**：
```
K8s + device plugin（NVIDIA GPU）
调度策略：
  - 抢占式（高优先级抢低）
  - 共享（多任务分时）
  - 弹性（空闲时扩）
  
资源池：
  - 训练池（A100/H100，贵，专用）
  - 推理池（A10/T4，性价比）
  - 开发池（V100/T4，实验）
```

**分布式训练**：
```
数据并行（DDP）：每卡完整模型+部分数据
模型并行：
  - 张量并行（层内切）
  - Pipeline 并行（层间切）
框架：DeepSpeed/Megatron-LM/Ray
```

**训练任务管理**：
```
- 任务队列（优先级）
- 断点续训（checkpoint）
- 超参搜索（Optuna/Ray Tune）
- 训练监控（loss/metrics）
```

## 三、推理平台

**模型服务化**：
```
框架：
  - vLLM（通用首选）
  - TensorRT-LLM（Nvidia 极致）
  - Triton（多模型多框架）
  - TF Serving（TensorFlow）

部署：
  - K8s + GPU 节点池
  - 弹性扩容（按 QPS）
  - 金丝雀/蓝绿发布
  - A/B 路由
```

**模型仓库**：
```
- 版本管理（v1.0/v1.1）
- 元数据（输入/输出/延迟/精度）
- 灰度比例
- 回滚按钮
```

## 四、数据平台

**标注平台**：
```
- 任务管理（标注/审核/复审）
- 多人标注+一致性
- 标注工具（文本/图片/视频）
- 标注质量监控
```

**特征平台**（推荐/搜索）：
```
离线：Hadoop/Hive（T-1 特征）
实时：Flink + Redis（实时特征）
特征注册+共享+版本
```

**数据版本**：
```
- 训练数据版本化（DVC）
- 模型-数据关联（追溯）
- 数据漂移检测
```

## 五、评测平台

**离线评测**：
```
benchmark 数据集（标注）
自动指标：
  - 准确率/召回率/F1
  - BLEU/ROUGE（生成）
  - CTR/停留时长（推荐）
LLM-as-Judge（强模型评弱模型）
```

**在线 AB**：
```
流量分桶（hash uid）
指标对比（CTR/解决率/满意度）
显著性检验（p-value）
决策：全量/回滚/迭代
```

**红队/对抗**：
```
- 故意构造攻击样本
- 找模型漏洞
- 持续改进
```

## 六、部署平台

**模型上线流程**：
```
训练 → 离线评测 → 灰度（5%）→ 监控 → 全量
   ↓                ↓
   失败          指标差
   回练          回滚
```

**版本管理**：
```
模型版本 + Prompt 版本 + 数据版本 + 评测版本
回滚：一键回到上版本（含所有关联）
```

## 七、实验平台

**实验管理**：
```
- 实验登记（假设/参数/结果）
- 对比（A/B/C 多组）
- 复现（环境+数据+代码快照）
- 知识库（沉淀经验）
```

**超参搜索**：
```
Optuna/Ray Tune
网格/随机/贝叶斯优化
```

## 八、内容场景应用

**1. 审核模型迭代**：
```
数据：人审标注 → 数据平台
训练：审核模型（NLP/视觉）→ 训练平台
评测：benchmark + 在线 AB → 评测平台
部署：vLLM 服务化 → 推理平台
监控：误判率/延迟 → 监控
```

**2. 推荐模型迭代**：
```
特征：用户行为实时特征 → 特征平台
训练：DIN/DeepFM → 训练平台
评测：CTR AB → 评测平台
部署：TF Serving → 推理平台
```

**3. LLM Agent 迭代**：
```
Prompt：版本管理
模型：基础模型 + 微调
评测：Agent benchmark
部署：vLLM + Agent 编排
```

## 九、监控与成本

**监控**：
```
业务：
  - 模型准确率/召回率
  - 在线指标（CTR/解决率）
  - 数据漂移
  
技术：
  - GPU 利用率（>70%）
  - 推理延迟
  - 服务可用性
  - 成本（token/GPU 时）
```

**成本控制**：
```
- GPU 共享（多任务分时）
- 模型蒸馏（小模型替代）
- 量化（INT8/INT4）
- 推理优化（vLLM）
- 分级调用（简单走小模型）
```

## 十、底层本质

AI Infra 本质是**"把模型全链路（训练/推理/数据/评测/部署）平台化"**——让算法同学专注模型，工程保证效率/质量/成本；内容场景需特别关注 GPU 调度（贵）+实时特征（推荐）+在线 AB（效果量化）。

## 常见考点
1. **GPU 怎么高效调度**？——K8s+device plugin+抢占式+共享+弹性。
2. **模型上线怎么保证质量**？——离线评测+灰度+在线 AB+回滚。
3. **特征平台怎么设计**？——离线（Hive T-1）+实时（Flink+Redis）+注册共享。

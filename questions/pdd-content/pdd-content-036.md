# 【拼多多内容】RAG 在 UGC 检索与审核的应用？

> JD 依据："分布式/缓存/消息/搜索"、"评价和行家社区"、"和算法同学挖掘业务问题"。

## 一、RAG 是什么

```
Retrieval-Augmented Generation（检索增强生成）
  用户问题 → 检索知识库 → 注入 prompt → LLM 生成（带引用）
  
vs 微调：
  RAG：外挂知识，灵活（数据变了重新索引）
  微调：内化知识，固化（改了要重训）
```

## 二、知识库构建

**1. 数据来源**（内容场景）：
- 历史 UGC（优质评价/回答）
- 平台政策/规则
- 商品信息
- 审核案例（违规样本）
- FAQ

**2. 切片**：
```
按语义切片（优于固定长度）：
  - 段落/章节切分
  - 递归切分（先大后小）
  - 重叠（前后 50 字重叠，防切断语义）
  
元数据：
  chunk = {
    text: "...",
    source: "policy.md",
    section: "退款规则",
    tags: ["退款", "售后"]
  }
```

**3. Embedding**：
```python
from sentence_transformers import SentenceTransformer
model = SentenceTransformer('bge-large-zh')
embeddings = model.encode(chunks)
```

**4. 向量库**：
- Milvus（生产首选，分布式）
- Faiss（单机，快）
- HNSW（图索引，召回好）
- ES（dense_vector，混合检索方便）

## 三、检索

**向量检索**（语义相似）：
```python
q_emb = model.encode(question)
docs = vector_db.search(q_emb, top_k=20)
```

**关键词检索**（BM25，精确匹配）：
```
question 分词 → ES BM25 查询
适合：专有名词/数字/代码
```

**混合检索**（推荐）：
```
向量召回（语义宽）+ BM25 召回（精确）
  → 合并去重
  → 重排
```

**查询改写**：
```python
# 用户问题可能口语化/模糊
# LLM 改写为多个变体提升召回
rewrites = llm.rewrite(question, n=3)
for q in rewrites:
    docs += retrieve(q)
```

## 四、重排

**Cross-Encoder**（精排）：
```python
from sentence_transformers import CrossEncoder
reranker = CrossEncoder('bge-reranker-large')
scores = reranker.predict([(question, doc.text) for doc in docs])
# 取 Top 5
```

**为什么重排**：
- 向量检索（双塔）快但粗
- Cross-Encoder（单塔）慢但准
- 召回用粗排，最终用精排

## 五、生成（带引用）

```python
prompt = f"""
基于以下资料回答问题，引用来源。

资料：
[1] {docs[0].text} (来源: {docs[0].source})
[2] {docs[1].text} (来源: {docs[1].source})
...

问题：{question}

要求：
- 只基于资料回答，不要编造
- 引用来源（如"根据[1]..."）
- 资料不足时说"暂无相关信息"
"""

answer = llm.generate(prompt)
```

## 六、内容场景应用

**1. UGC 智能问答（评价摘要）**：
```
用户："这个商品评价怎么样？"
RAG：
  - 检索商品相关评价（向量+商品 ID 过滤）
  - 重排取 Top K
  - LLM 总结（好评点/差评点/总体）
  - 带引用（"用户 A 说物流快"）
```

**2. 审核 RAG（违规案例参考）**：
```
Agent 审核时：
  - 输入：待审内容
  - RAG 检索：相似历史违规案例
  - LLM 结合案例判断（一致性）
  - 输出：违规类型+相似案例 ID
  
效果：新型违规快速识别（参考历史相似）
```

**3. 平台政策问答（客服/用户）**：
```
用户："怎么退款？"
RAG：
  - 检索退款政策
  - LLM 生成步骤+引用政策条款
```

**4. 行家内容检索**：
```
找行家深度评测：
  - 向量检索（语义相似）
  - 过滤（行家标记）
  - 重排（权威度加权）
```

## 七、防幻觉

```
1. 只基于检索资料生成（不凭空）
2. 输出带引用（可追溯）
3. 资料不足明确说（不编）
4. 事实核查（与 DB/RAG 对比）
5. 置信度阈值（低不答）
```

## 八、工程优化

**缓存**：
```
相似问题缓存（Embedding 相似度 >0.95 命中）
减少 LLM 调用
```

**增量索引**：
```
新评价/政策实时入向量库
Kafka → 消费 → Embedding → 写入
保证知识库新鲜
```

**多模态 RAG**：
```
图片 → 视觉模型生成描述 → 入库
查询时图文联合检索
```

## 九、评测

```
检索质量：
  - Recall@K（召回率）
  - MRR（倒数排名）
  
生成质量：
  - 准确率（事实对不对）
  - 引用准确率（引用对不对）
  - 用户满意度
  
端到端：
  - 解决率（用户问题解决）
  - 转人工率
```

## 十、底层本质

RAG 本质是**"用检索增强 LLM 的知识边界"**——切片+Embedding+向量检索+重排+带引用生成；内容场景特别适合（UGC 海量+审核需要案例参考+客服需要政策），比微调灵活，比纯 LLM 准确。

## 常见考点
1. **RAG 和微调怎么选**？——RAG 适合知识频繁变/私有数据，微调适合任务/风格固化。
2. **怎么提升召回**？——查询改写+混合检索（向量+BM25）+重排。
3. **切片策略怎么定**？——按语义（段落）优于固定长度，重叠防切断。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：RAG 和微调都能给 LLM 加知识，你的审核场景选了 RAG。如果违规案例更新频繁（每天新增），为什么不微调？微调不是更"内化"吗？**

微调的成本和灵活性是硬伤：
1. 更新成本——微调一次需要 GPU 训练几小时到几天，每天新增违规案例不可能每天微调。RAG 只需 Embedding + 写入向量库（秒级），知识实时更新。
2. 灵活性——微调后的知识是"固化"的（改了要重训），RAG 的知识是"外挂"的（删改向量库即可）。违规案例会修订（标错的纠正），RAG 改一条向量就行，微调无法局部改。
3. 可追溯——RAG 检索带引用（"根据案例 #1234 判违规"），审计可追溯；微调的知识融在权重里，无法说清"为什么这么判"。
4. 效果——微调擅长"风格/格式/任务模式"内化，不擅长"事实/案例"记忆（会幻觉串改）。事实型知识用 RAG 更准。
结论：违规案例是"频繁变化的事实型知识"，RAG 完胜微调。微调留给"审核风格/输出格式"这类稳定的模式。

### 第二层：证据与定位

**Q：审核 RAG 上线后，Agent 说"找不到相似违规案例"，但运营说库里明明有类似的。你怎么定位是检索召回问题还是 Agent 判断问题？**

分层定位：
1. 直接查向量库——拿待审内容做 query，`Milvus.search(query_emb, top_k=20)`，看返回结果里有没有运营说的案例。如果有但 Agent 没用，是 Agent prompt 问题（没用好检索结果）；如果没有，是检索召回问题。
2. 召回率指标——用标注集（100 条 query + 正确答案）跑 Recall@20，如果从 90% 降到 70%，是检索退化。
3. Embedding 模型——检查是否换了 Embedding 模型（如 bge-large 换 small），query 和库的向量空间不一致导致召回差。
4. 数据入库——查新案例是否真的入库了（`select count from vector_meta where create_time > today`），可能 Kafka 消费积压，案例没 Embedding 进库。
5. 切片问题——如果案例是长文档，切片把关键信息切散了，单片段召回不到。查 chunk 内容是否包含判违规的关键信息。

### 第三层：根因深挖

**Q：你用向量检索召回率 80%，但运营说很多"字面不同但语义相似"的违规没召回。Cross-Encoder 重排能解决吗？根本问题在哪？**

Cross-Encoder 重排解决不了召回率低（它只重排已召回的，不增加召回）。根本问题在向量召回的"语义覆盖"：
1. 双塔模型局限——向量检索（Bi-Encoder）把 query 和 doc 各自编码再算相似度，交互浅，对"字面不同语义同"的 case 召回不稳。
2. 查询改写——用户（这里是 Agent）的 query 可能表述和库里的案例差很远。解法：LLM 改写 query 为多个变体（"这个评价是虚假好评"→改写为"刷单/水军/诱导好评"），多路召回合并，提升语义覆盖。
3. 混合检索——向量（语义宽）+ BM25（精确词，如品牌名/数字）混合，互补。纯向量会漏掉精确匹配的。
4. 索引质量——HNSW 索引的 ef 参数调高（如 200→500），召回更全但变慢，权衡。
5. 根本——召回率 80% 不够，审核场景要 95%+（漏判合规风险）。需查询改写 + 混合检索 + 索引调优 + 定期补标（把漏召回的加进标注集持续优化 Embedding）。

### 第四层：方案权衡

**Q：RAG 检索 top_k=20 再 Cross-Encoder 重排取 top_5，延迟多 200ms。审核要求 P99 <500ms，你要砍掉重排吗？**

不砍，分级权衡：
1. 延迟拆解——向量检索 50ms + 重排 200ms + LLM 生成 200ms = 470ms。重排占 40%，是大头。
2. 重排价值——重排把"向量召回的粗排"精排，相关性提升明显（Recall@5 从 70%→90%）。砍掉重排，Agent 引用错案例，误判率升。
3. 优化——①重排模型用小 Cross-Encoder（bge-reranker-base 而非 large），200ms→80ms；②只对 top_20 重排 top_10（非全部），减少计算；③重排异步（先返回 LLM 初步结果，重排后补充引用），降感知延迟。
4. 分级——实时审核（直播）砍重排（延迟优先），异步审核（评价）保留重排（质量优先）。按 SLA 分级。
结论：用小重排模型 + 部分重排 + 场景分级，把重排延迟压到 80ms，总 P99 控制在 450ms 内，保住质量。

### 第五层：验证与沉淀

**Q：你怎么衡量审核 RAG 的效果？不能只看"检索准不准"，要从审核业务角度证明它有价值。**

效果衡量的多维指标：
1. 检索层——Recall@K（召回率，标注集评测）、MRR（相关案例的排名，越靠前越好）、引用准确率（Agent 引用的案例是否真的相关，抽样人工验）。
2. 生成层——事实准确率（Agent 判断是否和案例一致，不应幻觉）、引用覆盖率（每个判断是否都引了案例，无案例时是否明确说"暂无参考"）。
3. 业务层——新型违规识别率（RAG 上线后，历史没见过的违规类型能否参考相似案例判出，比纯规则/模型提升多少）、审核一致性（相似内容判得一致，RAG 提供统一参考）、人审采纳率（Agent 引用案例后人审认可的比例，>80% 算好）。
4. 端到端——机审率（60%→90%）、误判率（<1%）、端到端延迟（<500ms）。
沉淀：RAG 评测基准（1000 条 query + 标注答案）入版本管理，每次 Embedding/切片/检索策略变更跑基准，效果不退化才上线；人审纠正的案例回流标注库，持续扩 benchmark。

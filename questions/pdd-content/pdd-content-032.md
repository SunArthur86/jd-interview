# 【拼多多内容】LLM 智能客服 Agent 怎么设计？

> JD 依据："和算法同学挖掘业务问题"、"新媒体业务平台"。

## 一、传统客服的痛点

```
IVR（按键菜单）：体验差，找不到选项
关键词客服：只能命中预设 FAQ，复杂问题无解
人工客服：成本高，高峰排队
```

## 二、LLM 客服 Agent 架构

```
用户消息
   ↓
意图识别（LLM）：咨询/投诉/办理/闲聊
   ↓
路由：
  ├─ FAQ 咨询 → RAG 检索知识库 → LLM 生成答案
  ├─ 业务办理 → 工具调用（查订单/退款）→ LLM 总结
  ├─ 情绪化/复杂 → 转人工
  └─ 闲聊 → LLM 直接回（带人设）
   ↓
输出（带引用来源）
   ↓
满意度反馈 → 优化
```

## 三、RAG 知识库

**知识来源**：
- FAQ（常见问题）
- 平台政策（退款/售后/规则）
- 商品信息（规格/参数）
- 历史工单（优质回答）

**流程**：
```
知识库 → 切片 → Embedding → 向量库（Milvus/Faiss）
  ↓
用户问题 → Embedding → 向量检索 TopK → 重排 → LLM 生成
```

```python
def rag_answer(question):
    # 1. 检索
    docs = vector_db.search(embed(question), top_k=5)
    # 2. 重排（Cross-Encoder）
    reranked = reranker.rerank(question, docs)
    # 3. 生成
    prompt = f"基于以下资料回答：\n{reranked}\n问题：{question}"
    return llm.generate(prompt)
```

tools = [

## 五、多轮对话管理

**会话状态**：
```python
session = {
    "uid": "123",
    "history": [...],          # 最近 N 轮
    "context": {...},          # 提取的实体（订单号/商品）
    "intent": "查询物流",      # 当前意图
    "slot": {"order_id": None} # 待填充槽位
}

# 多轮：用户没给订单号 → LLM 反问 → 用户补全 → 继续工具调用
```

**长期记忆**：
- 用户画像（偏好/历史投诉）
- Redis 存最近会话摘要

## 六、人审兜底

**触发条件**：
```
1. 情绪检测（用户愤怒/抱怨）→ 转人
2. LLM 置信度低（多次澄清仍不懂）→ 转人
3. 高风险操作（大额退款/账号问题）→ 转人
4. 用户主动要求人工
```

```python
if detect_anger(user_msg) or confidence < 0.5 or is_high_risk(intent):
    transfer_to_human(session)
```

## 七、防幻觉

```
1. RAG 接真实数据（不凭空生成）
2. 输出带引用来源（"根据退款政策第 3 条..."）
3. 工具调用拿真实数据（订单/物流）
4. 置信度阈值（低不答，转人）
5. 关键场景人审（涉及资金/账号）
```

## 八、Prompt 设计

```
你是拼多多智能客服小多。

人设：友好/专业/简洁

能力：
  - 回答 FAQ（基于知识库）
  - 查询订单/物流（工具调用）
  - 协助退款/售后（工具调用）

约束：
  - 不确定时不要编，反问或转人工
  - 涉及资金/账号操作要确认
  - 用户愤怒时优先安抚+转人工
  - 回答带来源（"根据xxx政策"）

工具：query_order / query_logistics / apply_refund / ...
```

## 九、内容场景应用

**1. 评价投诉**：
```
用户："我评价被删了怎么回事？"
LLM：查评价状态 → 解释审核规则 → 申诉通道
```

**2. 直播问题**：
```
用户："主播没发货"
LLM：查订单 → 联系商家 → 退款选项
```

**3. 内容咨询**：
```
用户："这个商品评价好吗？"
LLM：RAG 检索评价摘要 → 总结（好评点/差评点）
```

## 十、效果衡量

```
业务指标：
  - 解决率（不转人工）目标 >70%
  - 转人工率 <30%
  - 首次响应时间 <2s
  - 用户满意度 CSAT >4.5/5
  
技术指标：
  - 意图识别准确率
  - RAG 召回率
  - 工具调用成功率
  - 幻觉率（抽检）
```

## 十一、底层本质

LLM 智能客服本质是**"LLM 理解+RAG 知识+工具调用+人审兜底"**——LLM 替代关键词理解意图、RAG 接真实知识、工具办业务、人审处理复杂/情绪化，是 Agent 工程化的典型场景。

## 常见考点
1. **怎么防幻觉**？——RAG+工具拿真实数据+置信度+人审+引用来源。
2. **多轮怎么管理**？——会话状态+槽位填充+历史摘要+长期记忆。
3. **RAG 怎么提升**？——切片优化+混合检索（向量+关键词）+重排+查询改写。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：智能客服你用 LLM + RAG，为什么不直接用 LLM（大模型本身知识很丰富）？加 RAG 不是更复杂吗？**

LLM 的知识有两个硬伤：1.知识截止——训练数据到某日期，之后的政策（如拼多多 2026 年新退款规则）LLM 不知道，会编旧规则。2.业务数据私有——用户的订单/物流/评价状态 LLM 完全不知道，不接系统就答不了"我的订单到哪了"。RAG 的价值是"给 LLM 接实时+私有数据"：检索退款政策（最新）注入 prompt，LLM 基于真实政策回答；调工具查订单拿真实物流。没有 RAG，LLM 只能"通用闲聊"，无法解决真实业务问题。复杂度增加换来的是"准确性"——客服场景答错（退款政策说错）会引发客诉和资损。

### 第二层：证据与定位

**Q：客服 Agent 回答用户"怎么退款"，但答案和实际退款政策不符（LLM 编了）。你怎么定位是 RAG 没检索到正确政策，还是 LLM 生成时忽略了检索结果？**

幻觉来源的分层定位：
1. 看 RAG 检索——记录检索到的文档（`docs`），检查是否包含正确的退款政策文档。如果 docs 里没有，是检索失败（召回率低）——可能是切片把政策切断了、或查询改写把"退款"改成了"退货"导致召回偏。
2. 看 LLM 生成——如果 docs 里有正确政策，但 LLM 答错了，是生成时忽略了上下文。检查 prompt 是否明确"只基于资料回答，不要编造"。或 prompt 太长（其他文档干扰），LLM 没注意到正确的那条。
3. 重排问题——如果检索召回 20 条但重排把正确的排到第 15 名（Top5 截断丢掉），LLM 没看到。检查重排模型是否准。

### 第三层：根因深挖

**Q：客服 Agent 的 RAG 召回率低——用户问"退货流程"，知识库里明明有退货政策但没检索到。根因是什么？**

RAG 召回失败的常见根因：
1. 查询与文档的语义鸿沟——用户问"退货"，政策文档写的是"售后退换货"，向量相似度低（词不匹配）。解法：查询改写（LLM 把"退货"扩展为"售后/退换货/退款"多个查询）。
2. 切片不当——退款政策是一整页，切片时把"退货流程"这段切到了上一个 chunk，而这个 chunk 的标题是"退款规则"，向量检索时"退货流程"和"退款规则"的相似度不如其他 chunk。解法：按语义切片（标题+正文一起）+ 重叠（前后 chunk 重叠 50 字）。
3. Embedding 模型不适配——通用 Embedding（如 BGE）在电商领域不如领域微调的。解法：用电商语料微调 Embedding 模型。
4. 只用向量检索——向量擅长语义但不擅长精确匹配（如订单号/商品编号）。解法：混合检索（向量 + BM25 关键词），BM25 补精确匹配。

### 第四层：方案权衡

**Q：客服 Agent 用 LLM 生成回答，但生成慢（2-3s）。用户等不及。你要不要改成模板回复（秒回）？权衡是什么？**

模板回复 vs LLM 生成的权衡：
1. 模板回复——快（<100ms），但死板，复杂问题答不了（用户问的不是模板预设的）。适合高频简单问题（如"退款多久到账"→固定答案）。
2. LLM 生成——慢（2-3s），但灵活，能理解变体/多轮。适合复杂/个性化问题。
3. 权衡方案——**分层路由**：
   - 意图识别先判断——高频简单问题（退款/物流查询）走模板或预设 FAQ 库，秒回。
   - 复杂/模糊问题走 LLM 生成，用户看到"正在为您查询..."的 loading（UX 缓解等待感）。
   - 流式输出——LLM 边生成边输出（streaming），首字延迟 <500ms，用户感知快于完整生成 2s。
4. 缓存——常见问题（"怎么退款"）的答案缓存，相似问题命中（Embedding 相似度 >0.95），秒回且省 LLM 调用。

### 第五层：验证与沉淀

**Q：你怎么衡量智能客服 Agent 的效果？业务说"解决率"但客服说"满意度"，指标冲突怎么办？**

多维度指标体系 + 权重：
1. 解决率（不转人工）——核心指标，目标 >70%。权重最高。
2. 转人工率——目标 <30%。与解决率互补。转人工率高说明 Agent 能力不足。
3. 首次响应时间——<2s（用户耐心阈值）。超时用户流失。
4. 满意度（CSAT）——用户评分（赞/踩），目标 >4.5/5。但满意度受"问题是否解决"+"等待时间"+"态度"综合影响。
5. 幻觉率——抽检（每天抽 100 条对话人工复核），<5%。幻觉高即使解决率高也有风险（答错用户以为对了）。
6. 工具调用成功率——调 query_order/apply_refund 的成功率，>95%。失败可能是接口异常或参数提取错。
指标冲突的解决：解决率优先（业务核心），满意度作为约束（不能为了解决率瞎答），幻觉率作为红线（超阈值强制人审兜底）。
沉淀：RAG 知识库每周更新（新政策/新 FAQ）；prompt 版本化管理（AB 对比）；用户反馈（踩的对话）回流优化训练集。

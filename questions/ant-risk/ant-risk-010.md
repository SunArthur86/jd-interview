# 【蚂蚁风控】亿级数据的 MySQL 怎么扩展？分库分表方案？主从延迟怎么解决？

> JD 依据："海量数据应用系统的性能和稳定性"。风控事件表日增上亿，单库扛不住。

## 一、表面层：什么时候要分库分表

**单库的瓶颈信号**：
- 单表数据 > 1000 万（B+ 树从 3 层变 4 层，IO 翻倍）
- 单库 QPS > 5000（连接数、buffer pool 不够）
- 单库磁盘 > 1TB（备份慢、DDL 卡）
- 写放大导致 binlog 同步慢

**经验阈值**：
| 数据量 | 方案 |
|--------|------|
| < 500 万 | 单库单表 + 索引优化 |
| 500 万 - 1000 万 | 读写分离（主从） |
| 1000 万 - 5000 万 | 分区表（PARTITION） |
| > 5000 万 | **分库分表** |

## 二、垂直分 vs 水平分

**垂直分（按结构）**：
- **垂直分库**：按业务拆库（user_db、trade_db、risk_db）
- **垂直分表**：按字段冷热拆表（user_basic、user_detail）

**水平分（按数据）**：
- **按范围**：1-1000万一张、1000万-2000万一张（热点问题）
- **按哈希**：`hash(uid) % N`（均匀但扩容麻烦）
- **按时间**：按月分表（适合日志类，event_202607、event_202608）
- **按一致性哈希**：扩容只迁移部分数据

**风控的选择**：事件表按时间分（每月一张，方便归档），用户画像按 uid hash 分（均匀分布）。

## 三、分片键选择（核心决策）

分片键一旦选定无法轻易换。三大原则：
1. **高频**：90% 查询都带的字段（uid）
2. **均匀**：值分布均匀，避免某分片热点（uid 通常均匀）
3. **不可变**：值不会变（避免变更时迁移数据）

**反面案例**：用手机号分片 → 用户换号 → 数据要跨分片迁移。

**分片算法**：
```java
// 范围分片（适合范围查询）
shardNo = uid / 10000000;  // 0-1000万→分片0, 1000万-2000万→分片1

// 哈希分片（均匀，但扩容要全量迁移）
shardNo = Math.abs(uid.hashCode()) % 16;

// 一致性哈希（扩容只迁移 1/N）
shardNo = consistentHash(uid, nodes);
```

## 四、全局 ID 生成

分库分表后 MySQL 自增主键失效，需要全局唯一 ID：

**雪花算法（Snowflake）**：
```
| 1 bit 符号 | 41 bit 时间戳 | 10 bit 机器ID | 12 bit 序号 |
              单位 ms，可用 69 年    1024 台机器     每毫秒 4096 个
```
- 去中心化、有序、高性能
- 依赖机器时钟（时钟回拨要处理）

**号段模式**（Leaf-Segment）：
- 中心化服务批量发号（每次发 1000 个）
- 客户端本地缓存，用完再申请
- 双 buffer 预加载，避免停顿

## 五、主从复制：读写分离

```
主库（Master）——写
   │ binlog 同步
   ├─→ 从库1（Slave）——读
   ├─→ 从库2（Slave）——读
   └─→ 从库3（Slave）——读/统计
```

**复制方式**：
| 方式 | 延迟 | 数据安全 |
|------|------|---------|
| 异步复制（默认） | 秒级 | 主挂可能丢未同步 |
| 半同步复制 | 毫秒级 | 至少一个从库收到才返回 |
| 组复制（MGR） | 强一致 | 多数派确认 |

**主从延迟问题**：异步复制有秒级延迟，读从库可能读到旧数据。

**解决方案**：
1. **关键场景强制读主**：`@Master` 注解（如交易后查余额）
2. **半同步复制**：业务可接受稍慢时
3. **客户端缓存"刚写"标记**：写后 N 秒内强制读主
4. **业务容忍**：列表页读旧数据无所谓

## 六、跨库难题与解法

**1. 分布式事务**：
- 强一致：XA（性能差，少用）
- 柔性事务：TCC、SAGA、本地消息表（最终一致）

**2. 跨库 JOIN**：
- 业务层组装：分别查多库，内存 join
- 冗余字段：把 join 字段冗余存
- 异构索引：把数据同步到 ES，在 ES 做 join

**3. 聚合统计**：
- 各库分别 count，汇总求和（count 可以）
- 但 sum/avg 跨库不准（avg 要加权）
- 异步汇总到汇总表

**4. 深分页**：
- 全局 LIMIT 1000000,10 跨库代价大
- 各库查前 N，业务层归并排序
- 或用 ES 异构索引统一查

## 七、风控实战架构

**事件表分库分表方案**：
```
risk_event 表（按月+uid 分）

物理库：risk_db_0, risk_db_1, ..., risk_db_7（8 库）
物理表：risk_event_202607_0, ..., risk_event_202607_7（每库每月1张）

路由：db = hash(uid) % 8, table = "risk_event_" + month + "_" + (hash(uid) % 8)
```

**ShardingSphere-JDBC 配置**：
```yaml
shardingsphere:
  datasource:
    names: ds0,ds1,...,ds7
    ds0: { ... }
  sharding:
    tables:
      risk_event:
        actual-data-nodes: ds${0..7}.risk_event_${202601..202612}_${0..7}
        database-strategy:
          hash:
            sharding-column: uid
            algorithm-expression: ds${uid.hashCode().abs() % 8}
        table-strategy:
          complex:
            sharding-columns: uid,created_at
            algorithm: riskEventTableAlgorithm
        key-generator:
          column: id
          type: SNOWFLAKE
```

**读写分离**：
```yaml
masterslave:
  name: ms_ds0
  master-data-source-name: ds0_master
  slave-data-source-names: [ds0_slave1, ds0_slave2]
```

## 八、底层本质：分而治之与一致性代价

分库分表本质是 **"用一致性换扩展性"**：
- 单库 ACID 强，但扩展性差
- 分库后失去跨库事务、JOIN、聚合
- 换来线性扩展能力（加机器即可加容量）

**这是分布式系统的 CAP 取舍**：
- 单库：CA（强一致 + 可用，不可扩展）
- 分库：AP（可用 + 可扩展，最终一致）

工程上的折中：**核心业务强一致（同库或柔性事务），非核心最终一致**。

**风控的实践**：
- 风险扣款：单库事务保证强一致
- 事件查询：分库分表 + ES 异构索引（最终一致）
- 实时统计：异步汇总到汇总表（容忍分钟级延迟）

## 常见考点
1. **分片键选错了怎么办**？——很难改，要么停服迁移，要么"双写新旧分片键"灰度切换。
2. **分库分表后还能加索引吗**？——能，但要每库都加（DDL 影响所有分片），用 pt-online-schema-change 平滑。
3. **ShardingSphere-JDBC 和 Proxy 区别**？——JDBC 是客户端模式（轻量、直连），Proxy 是中间层（语言无关、多一跳 RT）。

**代码示例**（柔性事务——本地消息表）：
```java
// 业务表 + 消息表在同一库（同事务）
@Transactional
public void deductAndSendEvent(long uid, long amount) {
    accountDao.deduct(uid, amount);              // 业务操作
    outboxDao.insert(new Outbox(uid, amount));   // 消息表（待发）
}
// 定时扫描 outbox 投递到 Kafka，成功后删除（最终一致）
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：风控事件表你按"时间+uid"双重分片（按月+hash(uid)%8），而不是只按 uid hash 分。决策依据是什么？为什么时间维度也要参与分片？**

因为风控事件表是"写多读多 + 时间衰减"的数据——每天新增上亿行，90% 的查询是"某用户最近 N 天的事件"。只按 uid hash 分片，每个分片会无限膨胀（单分片几年后上百亿行），B+ 树层数增加、DDL 慢、备份久。引入时间维度后，每月一张物理表（risk_event_202607），历史月份可归档/删除（冷热分离），单表容量可控。同时 uid hash 保证同一用户的数据在多个时间分片内均匀分布，查询"某用户事件"时按 uid+时间范围定位到少数几张表，避免全分片扫描。决策依据是查询模式分析——统计风控事件表的查询，95% 带 uid 条件、80% 带时间范围（最近 7/30 天），所以 uid + 时间双分片键覆盖大多数查询。

### 第二层：证据与定位

**Q：风控事件查询突然变慢，平均 RT 从 20ms 涨到 2 秒。你怎么确认是分片路由问题还是某个分片本身慢？**

两组证据区分：
1. 看分片级别的 RT——ShardingSphere 暴露了每个分片的执行指标（`shardingsphere.proxy.backend.execute.latency.{ds0..ds7}`），如果 8 个分片里有 1 个 RT 飙到 15 秒，其他正常，是单分片问题（该分片所在 MySQL 实例故障或数据倾斜）。如果所有分片 RT 都正常但整体慢，是路由到了太多分片（比如查询没带分片键，全分片扫描）。
2. 看 SQL 路由日志——ShardingSphere 的 `routing.log` 记录每条 SQL 路由到几个分片。正常按 uid 查询应该路由到 1 个分片（`actual tables: ds3.risk_event_202607_3`），如果看到 `actual tables: ds0..ds7`（全分片），说明查询没带分片键或分片键没识别到（如 SQL 用了 `WHERE uid IN (1,2,3)` 或 `WHERE merchant_id=xxx`）。全分片扫描的代价是单次查询放大 8-64 倍（8 库 × 12 月）。

### 第三层：根因深挖

**Q：你发现查询路由到了全分片——SQL 是 `WHERE merchant_id=500 AND created_at > '2026-07-01'`，merchant_id 不是分片键。根因是什么？怎么解决按 merchant 维度的查询？**

根因是分片键（uid）和查询维度（merchant_id）不一致。分片键选 uid 是因为 90% 查询按用户维度，但风控还有 10% 的查询是"某商户的所有用户事件"（商户风控报表），这种查询 merchant_id 不在分片键里，必须全分片扫。解决思路不是换分片键（换了会影响 90% 的主链路查询），而是"异构索引"——把数据按 merchant_id 维度再存一份到另一个存储。具体做法是用 Canal 监听 MySQL binlog，把事件数据同步到 Elasticsearch，在 ES 里建 `merchant_id` 索引，商户维度的查询走 ES（支持任意字段索引），用户维度的查询仍走 MySQL 分片。代价是数据最终一致（binlog 同步有 1-5 秒延迟）+ 多一份 ES 存储成本，但解决了跨维度查询。

**Q：根因是多维度查询需求。那为什么不直接用 TiDB/NewSQL 让所有维度都能查，省去异构索引的复杂度？**

TiDB 确实支持任意二级索引查询、无需人工分片。但前面提过，TiDB 的点查 RT 比 MySQL 高 2-3 倍（KV 层网络 + Raft），风控主链路（用户维度查询）要求 RT <20ms，TiDB 可能 40-60ms，吃掉决策链路预算。异构索引的复杂度（Canal + ES）换来的是"主链路用 MySQL 极快、副维度查询走 ES 也够快"，各取所长。而 TiDB 是"所有查询都中等快"。风控选了 MySQL + ES 异构，因为主链路（占 90% 流量）的极致性能比副维度查询的便利性更重要。只有当副维度查询变成主要场景（如报表型业务），才考虑直接用 TiDB/NewSQL 简化架构。

### 第四层：方案权衡

**Q：你引入了 ES 异构索引解决 merchant 维度查询。但 ES 的数据有 1-5 秒延迟，业务说"刚发生的事件立刻查不到"。怎么权衡？**

这是最终一致性的延迟权衡。先分场景：
1. 实时风控决策——查"最近 1 小时的事件"，1-5 秒延迟可接受（事件发生到决策有秒级窗口，延迟在窗口内）。
2. 事后调查/报表——查"昨天的事件"，1-5 秒延迟完全无感。
3. 强实时场景（如"3 秒前刚交易完，立刻查"）——必须强制读主库（MySQL 主库），不走 ES。用 `@Master` 注解或业务层判断"创建时间 < 5 秒的查 MySQL、> 5 秒的查 ES"。
权衡方案是"读写分离 + 强制读主"：ES 承担大部分读（历史数据、报表），MySQL 主库承担"刚写入的强实时读"。代价是应用层要区分路由（代码里判断时间窗口），但这比"全量同步等待强一致"的代价小。

**Q：为什么不直接用 MySQL 半同步复制替代异步，把从库延迟降到毫秒级，就不用 ES 了？**

半同步复制只解决"主从延迟"，没解决"按 merchant_id 查询要全分片扫"的问题。半同步保证从库数据快速同步，但从库仍然是按 uid 分片的结构，查 merchant_id 还是要全分片扫。ES 的价值不是"低延迟读"，而是"多维索引"——它的倒排索引让任意字段都能 O(1) 查询，这是 MySQL 分片结构做不到的。半同步 + MySQL 适合"同维度读写分离"（主写从读，延迟低），ES 异构适合"跨维度查询"（不同字段索引）。两者解决不同问题，不能互相替代。我们两者都用——MySQL 半同步保证主从低延迟（读从库的强实时查询），ES 异构解决跨维度查询。

### 第五层：验证与沉淀

**Q：你怎么验证分片方案上线后，查询性能真的达标？怎么量化"分片是否均匀"？**

三个指标验证：
1. 分片均匀度——用 ShardingSphere 的 `shardingsphere.proxy.backend.execute.count.{ds0..ds7}` 指标，统计每个分片的 QPS 分布。理想是各分片 QPS 接近（偏差 <20%）。如果某个分片 QPS 是其他的 5 倍，是数据热点（某个 uid 段请求集中）。同时统计每个物理表的数据量 `SELECT COUNT(*) FROM risk_event_202607_X`，各分片行数偏差应 <30%。
2. 路由命中率——统计"带分片键的查询占比"（`shardingsphere.proxy.route.hit.rate`），应 >95%。如果 <80%，说明很多查询没带分片键（全分片扫），要优化 SQL 或建异构索引。
3. 查询 RT 分位——按"带分片键"和"不带分片键"分别统计 RT。带分片键的 P99 应 <50ms（单分片点查），不带分片键的 P99 可能 2 秒（全分片扫），这是预期行为，对应的查询应迁移到 ES。

**Q：怎么让团队分库分表方案不再踩坑？**

沉淀成规范：
1. 分片键评审——任何新表的分片键必须经架构评审，提供"查询模式分析"（90% 查询带什么字段）、"数据增长预估"、"分片均匀度压测"，否则不允许上线。
2. 查询规范——所有 SQL 必须带分片键（CR 强制检查），不带分片键的查询必须走 ES 或汇总表，禁止全分片扫。用 ShardingSphere 的 SQL hint 或 audit log 检测全分片查询并告警。
3. 数据均衡监控——每周跑一次各分片数据量统计，偏差 >50% 告警，考虑重新分片或热点拆分。
4. 主从延迟监控——`Seconds_Behind_Master` 指标告警阈值 >3 秒，触发自动切读主或告警。
5. 故障复盘——把这次"merchant_id 查询全分片扫 → RT 2 秒"的 SQL 路由日志、ES 异构上线前后的 RT 对比存知识库，作为"分片键和多维查询"的标准案例。


## 结构化回答

**30 秒电梯演讲：** 聊到亿级数据的 MySQL 怎么扩展，我的理解是——单库千万级后写放大、连接数、查询延迟都会劣化，用"垂直分（按业务拆库）+ 水平分（按 hash/range 拆表）"分散到多节点，配合主从分离读写。打个比方，分库分表像把一本厚字典拆成多本——按部首拆（垂直，按业务）、按页码拆（水平，按数据）。主从像把同一本字典复印多份，主本写、副本读。

**展开框架：**
1. **垂直分库（按业务）** — 垂直分库（按业务）、垂直分表（按字段冷热）、水平分表（按 hash/range/一致性 hash）
2. **分片键选择** — 高频查询字段、分布均匀、不可变
3. **主从延迟** — 异步复制（默认）/ 半同步 / 组复制

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：分片键怎么选？您更想看哪个方向？

# 【蚂蚁风控】设计一个智能风控平台

> JD 依据："数字化风控服务及智能风控平台"。这是 JD 的核心目标。

## 一、智能风控平台的目标

**业务目标**：
- 多业务接入（支付、借贷、营销、海外）
- 实时决策（< 200ms）
- 高准确率（少漏报少误杀）
- 快速迭代（运营自助）

**技术目标**：
- 组件化（规则/模型/Agent 可插拔）
- 平台化（统一编排、自助运营）
- AI 化（Agent 决策、自然语言配置）
- 可观测（全链路监控、效果评估）

## 三、决策编排层（核心）

**双引擎**：
- **DAG 引擎**：传统顺序+并行+短路（毫秒）
- **Agent 引擎**：AI 自主编排（秒级，边缘场景）

**编排策略**：
```
所有事件 →
  ├─ 80% 简单场景 → DAG 引擎（毫秒）
  └─ 20% 边缘场景 → Agent 引擎（秒级）

DAG 引擎:
  名单 → 特征 → 规则 → 模型 → 决策

Agent 引擎:
  LLM 理解 → 调工具（特征/规则/模型/关系）→ 推理 → 决策
```

**编排可视化**（运营拖拽）：
```
运营在控制台拖拽节点:
  [名单] → [特征] → [规则] → [模型] → [决策]
  连线表示依赖
  短路条件可配
```

## 四、能力层（各组件）

**1. 名单服务**：黑白名单 + 布隆过滤器（毫秒）
**2. 特征服务**：见 ant-risk-022（特征平台）
**3. 规则引擎**：见 ant-risk-012（Drools/QLExpress）
**4. 模型服务**：LR/XGBoost/DNN/LLM
**5. 关系网络**：见 ant-risk-023（JanusGraph）
**6. AI Agent**：见 ant-risk-031（决策 Agent）
**7. 案件库**：历史决策记录 + 人工审核结果
**8. 审核平台**：人工复核工作台

**组件统一接入**：每个组件用 MCP 包装成标准工具，决策编排调用。

## 五、运营层（自助化）

**核心目标**：运营自助，开发少介入。

**功能**：
1. **规则配置**：DSL 拖拽 / 自然语言
2. **模型管理**：版本、灰度、回滚
3. **决策编排**：拖拽画布
4. **效果看板**：实时监控
5. **案件审核**：人工工作台
6. **A/B 实验**：策略对比

**自然语言配置**（AI 加持）：
```
运营: "夜间大额转账到陌生账户要拦截"
   ↓ LLM 翻译
DSL: if amount > 5000 && time ∈ [22,6] && relation == new → REJECT
   ↓ 自动生效（灰度）
```

## 六、AI Agent 在平台中的角色

**1. 决策 Agent**：
- 边缘场景自主决策
- 调用其他组件做工具

**2. 运营助手 Agent**：
- 自然语言翻译成 DSL
- 推荐规则、解释策略

**3. 复核 Agent**：
- 自动复核决策（双验证）
- 生成解释

**4. 自学习 Agent**：
- 分析错判案例
- 推荐规则/模型改进

## 七、自助接入（多业务）

**业务自助接入流程**：
```
新业务接入（如海外支付）:
  1. 申请接入（控制台）
  2. 配置业务标识（业务隔离）
  3. 选择规则集（复用 / 新建）
  4. 配置决策编排（DAG 拖拽）
  5. 灰度发布（10% → 全量）
  6. 监控效果
```

**业务隔离**：
- 独立规则集
- 独立模型
- 共享特征平台
- 决策日志按业务分库

## 八、灰度发布与 A/B 实验

**策略灰度**：
```
新规则上线:
  - 影子模式（不打扰线上，只评估）
  - 灰度 10%（10% 流量用新规则）
  - 灰度 50%
  - 全量

监控指标:
  - 拦截率变化
  - 误杀率
  - 漏报率
  - 用户投诉
```

**A/B 实验**：
```
对照: 规则集 A
实验: 规则集 B

按 uid hash 分桶（50/50）
对比拦截率、误杀率、收益
显著差异（p < 0.05）才切换
```

## 九、效果评估闭环

**决策回流**：
```
决策 → 落库 → 人工审核 → 标注
        ↓
      分析（错判原因）
        ↓
      迭代（规则/模型/Agent 改进）
```

**核心指标**：
- **召回率**（漏报率，越低越好）
- **精准率**（误杀率，越低越好）
- **决策 RT**（P99，< 200ms）
- **运营自助率**（< 80% 算合格）
- **AI 决策占比**（趋势上升）

## 十、可观测性

**业务大盘**：
- QPS、拦截率、误杀率
- 各业务对比
- 规则命中 Top
- 模型分数分布

**系统大盘**：
- 决策链路 RT 分布
- 各组件健康度
- Agent 决策 trace

**告警**：
- 拦截率骤变 → 业务异常
- P99 RT > 500ms → 性能问题
- Agent 幻觉率 > 5% → AI 异常

## 十一、底层本质：从"系统集成"到"平台编排"

传统风控 vs 智能风控平台：

| 维度 | 传统风控 | 智能风控平台 |
|------|---------|------------|
| 架构 | 单体应用 | 平台化（组件 + 编排） |
| 决策 | 硬编码规则 | DAG + Agent 双引擎 |
| 运营 | 改代码上线 | 控制台自助 |
| AI | 单点（模型） | 全链路（Agent/特征/Harness） |
| 迭代 | 周/月 | 天 |
| 多业务 | 各做各的 | 共享平台 |

**核心转变**：从"做系统"到"做平台"。

**这是软件工程的"平台化"思想在风控的应用**：
- 业务方像 SaaS 用户
- 风控平台像 PaaS
- 各组件像微服务

## 十三、落地演进路径

**Stage 1：传统风控**（现状）
- 规则 + 模型
- 改代码上线
- 单业务

**Stage 2：平台化**
- 规则/模型可配置
- 多业务共享
- 运营部分自助

**Stage 3：AI 辅助**
- LLM 做解释、提示
- 边缘场景 Agent 决策
- 自然语言配置

**Stage 4：AI 原生**
- Agent 主导决策
- 规则做硬约束
- 全自助运营
- 自学习闭环

**风控的当前阶段**：Stage 2-3，向 Stage 4 演进。

## 常见考点
1. **智能风控平台的核心价值**？——平台化让组件协同 + 运营自助 + 快速迭代；AI 化让决策更智能。
2. **DAG 和 Agent 引擎怎么选**？——简单场景 DAG（毫秒）、边缘场景 Agent（秒），分层组合。
3. **怎么衡量平台成熟度**？——决策 RT、迭代周期、运营自助率、AI 决策占比、多业务接入数。

**代码示例**（决策编排 DAG）：
```java
DecisionFlow flow = DecisionFlow.builder()
    .start("whitelist")                          // 名单
    .thenParallel("feature", "profile")          // 并行查特征画像
    .then("rule_eval")                           // 规则
    .then("model_infer")                         // 模型
    .then("fusion")                              // 融合
    .onShortCircuit("whitelist", r -> r.isFinal())
    .timeout(Duration.ofMillis(200))
    .fallback(e -> RiskResult.basicPass())       // 全挂兜底
    .build();

RiskResult result = flow.execute(event);
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：智能风控平台和"把规则引擎 + 模型 + 关系网络拼在一起"有什么本质区别？为什么要做"平台"而不是"系统"？**

本质区别是"谁来编排决策"。把组件拼在一起是"硬编码编排"——开发在代码里写死"先查名单 → 再跑规则 → 再跑模型"，改流程要发版，运营无法自助。平台的核心是"编排能力产品化"——运营在可视化界面拖拽 DAG（或用自然语言描述），平台动态执行，不发版就能调整决策流程。这就像"写死 SQL"vs"BI 平台"的区别——前者是开发活，后者是运营自助。做平台的动机是"迭代速度"——新型欺诈每周出现，如果每次调规则流程都要开发排期 + 测试 + 发版（2 周），风控响应滞后；平台化后运营自己配置（2 小时），对抗效率提升 100 倍。这是业务驱动，不是技术追新。

### 第二层：证据与定位

**Q：智能风控平台上线后，运营反馈"配置的决策流程跑不出预期结果"。你怎么定位是配置错误还是平台 bug？**

三步定位：
1. 看 DAG 执行 trace——平台必须记录每一步的输入输出（`whitelist` 步骤的输入是 event、输出是 `PASS`；`rule_eval` 步骤的输入是 features、输出是 `REJECT` 命中规则 R123）。顺着 trace 看，是哪一步的输出不符合预期。如果某一步输入对但输出错，是组件 bug；如果输入本身就不对（特征没查到），是上游配置或数据问题。
2. 看配置版本——运营改过配置吗？查配置变更历史，确认当前执行的是哪个版本的 DAG。常见错误：运营改了 DAG 但没发布（存的是草稿），或发布了但没到这个机房（灰度没覆盖）。
3. 复现验证——把这笔交易的 event 数据拷贝到"测试环境"，用相同 DAG 跑一遍，看能否复现。如果测试环境结果和线上不同，是环境差异（测试环境特征数据和线上不同）。

### 第三层：根因深挖

**Q：你定位到是"规则配置错误"——运营把"金额 > 1 万"写成了"金额 > 100"，导致大量误拦。为什么平台没拦住这种低级错误？**

根因是平台缺少"配置校验"。平台允许运营自由配置规则，但没有"合理性检查"——比如"金额阈值变动超过 10 倍"要告警、"规则变更影响预估命中率 > 5%"要二次确认。这种校验不是技术上做不到，是平台设计时没考虑"防呆"。真正的解法是三层防护：第一，配置校验——阈值变化幅度、规则冲突检测、语法检查，不通过不让保存。第二，影子验证——新规则保存后先跑影子流量（不影响线上），对比新旧规则的决策差异，差异大的告警。第三，灰度发布——规则不能一键全量，必须 1% → 10% → 100%，每阶段看拦截率和误报率，异常自动暂停。根因是"平台给了自由但没给护栏"，自由和护栏要平衡。

**Q：那为什么不限制运营的配置权限，让所有规则变更都走开发 review，避免错误？**

这违背了平台的初衷。平台的目的是"运营自助"，如果每个变更都要开发 review，就回到了"硬编码时代"，迭代速度优势消失。正确的做法不是"限制权限"而是"工具赋能"——给运营提供"规则仿真器"（输入 event 看规则会怎么判）、"影响分析"（预估变更会影响多少流量）、"A/B 实验"（新旧规则对比效果）。工具到位后，运营自己能发现"金额 > 100 会拦 80% 流量"（影响分析显示），不会犯这种错。限制权限是"因噎废食"，赋能工具才是正道。同时保留"硬约束"（涉及资损的强规则变更需架构师审批）作为底线。

### 第四层：方案权衡

**Q：决策编排引擎选 DAG（确定性流程）还是 Agent（LLM 自主编排）？为什么不是全用 Agent？**

分层使用，不是二选一。原因：DAG 是确定性的（同样的输入永远同样的流程和结果），延迟可控（毫秒），可解释（每一步都清楚）。Agent 是非确定性的（LLM 可能每次选不同工具），延迟高（秒级），解释性依赖 LLM 推理链。风控 95% 的决策是"明确场景"（黑名单、强规则、高置信度模型），用 DAG 足够，且必须确定性（监管要求"同案同判"）。只有 5% 的"边缘 case"（DAG 拿不准的）才走 Agent 做深度推理。全用 Agent 会导致：延迟不可接受（每笔交易等 LLM 秒级响应）、结果不确定（同一笔交易不同时间可能判不同）、成本爆炸（每笔都调 LLM）。所以 DAG 为主、Agent 为辅是工程最优，不是技术保守。

**Q：既然 Agent 有这么多问题，为什么不等到 LLM 更成熟（更快更准）再上，现在就做 DAG？**

因为"等"的成本是漏报新型欺诈。黑产现在就在用 LLM 生成个性化欺诈，传统 DAG 规则覆盖不住——等 LLM 成熟（可能 3-5 年），这期间新型欺诈的漏报损失是实实在在的资损。现在的策略是"小规模验证 + 逐步扩大"——Agent 只跑 1% 的边缘 case（影子模式，不影响线上决策），积累效果数据。当 Agent 的准确率和延迟达到可接受水平（比如准确率 > 90%、延迟 < 2 秒），再逐步切流。这是"边等边练"——既不盲目全量（风险大），也不纯粹等待（错失对抗新型欺诈的窗口）。技术演进期，"部分采用 + 持续评估"比"全有或全无"更务实。

### 第五层：验证与沉淀

**Q：你怎么衡量智能风控平台的成熟度，证明它真的在"进化"而不是原地踏步？**

用一组递进指标：
1. 决策 RT——P99 从 200ms → 100ms → 50ms，体现技术优化。
2. 迭代周期——新规则上线从 2 周 → 2 天 → 2 小时，体现运营自助能力。
3. 运营自助率——运营自己配置的规则占比，从 20% → 60% → 90%，体现平台易用性。
4. AI 决策占比——Agent/模型决策的流量占比，从 0% → 5% → 20%，体现智能化程度。
5. 多业务接入数——从 1 个业务 → 5 个 → 20 个，体现平台通用性。
这五个维度共同定义"成熟度阶段"（Stage 1 到 Stage 4），不是单一指标能衡量的。

**Q：智能风控平台的设计经验和组件怎么沉淀，让新业务快速接入？**

三件事：
1. 标准化接入流程——新业务接入提供"向导式"流程（定义事件 schema → 选规则模板 → 配置决策 DAG → 灰度上线），不用从零搭。
2. 组件市场——规则模板、模型、特征、决策 DAG 沉淀成可复用组件，新业务按需组合，避免重复造轮子。
3. 最佳实践库——每个已接入业务的配置案例、踩过的坑、优化经验存档，新业务参考同类业务的配置快速上手。


## 结构化回答

**30 秒电梯演讲：** 聊到设计一个智能风控平台，我的理解是——智能风控平台把"规则+模型+AI Agent+关系网络+特征平台"统一编排，让风控从"配置驱动"升级为"AI 驱动"——平台化、Agent 化、自学习。打个比方，智能风控平台像一个"AI 风控大脑"——各个组件（规则、模型、关系网络、Agent）像脑区，平台像大脑皮层统一编排，对外提供智能决策。

**展开框架：**
1. **平台组件** — 决策编排 + 规则引擎 + 模型平台 + 特征平台 + 关系网络 + AI Agent
2. **平台能力** — 自助接入、可视化配置、灰度发布、效果评估
3. **AI 加持** — Agent 编排、自然语言配置、自动迭代

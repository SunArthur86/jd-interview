# 【蚂蚁风控】用图数据库做风控关系网络怎么设计？怎么发现团伙欺诈？

> JD 依据：JD 提到"图数据库"。蚂蚁是图风控的代表，这是高级架构题。

## 一、需求拆解

**业务场景**：
- 中介养号（一个手机登录几十个账号）
- 团伙套现（多个账号给同一商户刷单）
- 黑产设备农场（同 IP/同设备批量操作）
- 资金清洗（资金多跳转账隐藏来源）

**核心难点**：
- 每个账号单独看都"正常"（有实名、有交易历史）
- 单点规则失效
- 需要从"关系"维度发现异常聚集

## 二、图模型设计

**节点（实体）**：
```
(:User {uid, age, score})
(:Device {did, type})
(:IP {ip, isp})
(:Phone {phone})
(:Card {cardno, bank})
(:Merchant {mid, type})
(:Address {addr})
```

**关系（边）**：
```
(:User)-[:LOGIN_FROM]->(:Device)      // 用户从设备登录
(:User)-[:USE_IP]->(:IP)              // 用户用 IP
(:User)-[:OWN_PHONE]->(:Phone)        // 用户绑手机
(:User)-[:PAY_WITH]->(:Card)          // 用户用卡支付
(:User)-[:TRADE_WITH]->(:Merchant)    // 用户和商户交易
(:User)-[:TRANSFER_TO]->(:User)       // 用户间转账
```

**举例**：
```
账户A ─┬─ 登录设备1
       ├─ 用 IP X
       └─ 转账给 → 账户B ─┬─ 登录设备1  ← 同设备！
                          └─ 用 IP X    ← 同 IP！

→ A、B 高度关联，疑似同一人控制
```

## 四、在线查询（毫秒级）

**实时决策时查关系风险**：

```cypher
// Cypher 查询：UID 共享设备数
MATCH (u:User {uid: $uid})-[:LOGIN_FROM]->(d:Device)<-[:LOGIN_FROM]-(other:User)
WHERE other.uid <> $uid
RETURN COUNT(DISTINCT other.uid) as shared_users,
       COLLECT(DISTINCT d.did) as shared_devices

// 结果：shared_users=15, shared_devices=3
// 一个用户和其他 15 个用户共享 3 个设备 → 高风险
```

**预计算的高风险标签**（更快）：
```java
// 离线算好后推到 Redis
RiskResult decide(Event e) {
    // 1. 查 UID 是否在高风险团伙
    Integer gangId = gangCache.get(e.uid);
    if (gangId != null) {
        return RiskResult.reject("in_gang_" + gangId);
    }
    // 2. 查共享设备数
    Integer sharedDevs = sharedDeviceCache.get(e.uid);
    if (sharedDevs > 5) {
        return RiskResult.review("shared_device_" + sharedDevs);
    }
    // 3. 查 2 跳邻居黑名单
    Integer blackNeighbors = blackNeighborCache.get(e.uid);
    if (blackNeighbors > 0) {
        return RiskResult.review("connected_to_blacklist");
    }
    return RiskResult.pass();
}
```

**性能**：
- 预计算 + Redis：3ms 内
- 在线 Cypher：30-50ms（1-2 跳）
- 超过 3 跳：必须预计算

## 五、离线图算法（团伙挖掘）

**1. 连通分量（Connected Components）**：
找出"完全连通"的团伙。
```
A-B-C-D 是一个连通分量（团伙1）
E-F-G 是另一个连通分量（团伙2）
```

**2. 社区发现（Louvain）**：
按"边密集度"挖社区，比连通分量更松。
```
社区1：节点间边多 → 强团伙
社区2：节点间边少 → 弱团伙
```

**3. 中心度（PageRank / 度数）**：
找出"枢纽节点"（高风险点）。
```
节点 X 连接 100 个账号 → X 是中介 → 标记
```

**4. 路径分析**：
查资金流转路径。
```
A → B → C → D → E（资金多跳转账）
深度 > 5 → 疑似洗钱
```

**5. 三角计数**：
找"三角关系"（强关联）。
```
A-B, B-C, A-C 三角 → 强团伙
```

## 七、增量更新（准实时）

**Flink 实时建边**：
```java
DataStream<LoginEvent> logins = env.addSource(kafka("logins"));

logins.keyBy(e -> e.did)
    // 同设备的所有登录 → 两两建边
    .process(new SameDeviceEdgeBuilder())
    // 写图存储
    .addSink(new JanusGraphSink());
```

**增量 vs 全量**：
- 增量（Flink）：秒级建新边，但算法重算成本高
- 全量（T+1）：每天重建，算法定期更新
- 实战：增量建边 + 定期全量重算算法

## 八、规模与性能

**蚂蚁的规模**：
- 节点：10 亿+（账号 + 设备 + IP + 卡 + 商户）
- 边：1000 亿+
- 图大小：1TB+（压缩存储）

**JanusGraph 选型**：
- 后端 HBase（存图数据）
- 索引 Elasticsearch（按属性查节点）
- 分布式（支持百亿节点）

**为什么不用 Neo4j**：
- Neo4j 社区版单机
- 企业版分布式但贵
- JanusGraph 开源分布式

## 九、底层本质：从"个体"到"群体"的风控范式

传统风控 vs 图风控：

| 维度 | 单点规则 | 图风控 |
|------|---------|--------|
| 视角 | 个体 | 群体 |
| 数据 | 用户特征 | 关系网络 |
| 攻防 | 黑产伪装个体即可绕 | 必须伪装整个网络（成本高） |
| 算法 | 阈值、规则 | 图算法、拓扑分析 |

**图风控的精髓**：
- 黑产可以伪装个体（实名、养号）
- 但很难伪装网络（团伙必然有共享设备/IP/资金流）
- 网络拓扑是不可消除的"指纹"

**这是"关系即风险"的哲学**：
- 个体异常可能误判
- 群体异常几乎准确
- 关系网络是"行为留下的不可磨灭痕迹"

## 常见考点
1. **十亿节点规模怎么存**？——JanusGraph + HBase 后端（按节点 ID 分片）+ ES 索引（属性查）。
2. **在线查询为什么限 2 跳**？——超过 2 跳节点数指数爆炸，必须预计算或离线算。
3. **团伙规模阈值怎么定**？——正常用户 1-3 个关联，黑产通常 10+；阈值要结合误杀率调（看大盘）。

**代码示例**（Flink 实时建同设备边）：
```java
logins.keyBy(e -> e.did)
    .window(SlidingEventTimeWindows.of(Time.hours(1), Time.minutes(10)))
    .apply((did, window, events, out) -> {
        List<LoginEvent> list = new ArrayList<>();
        for (LoginEvent e : events) list.add(e);
        // 两两建边（同设备的不同 UID）
        for (int i = 0; i < list.size(); i++) {
            for (int j = i + 1; j < list.size(); j++) {
                out.collect(new Edge(list.get(i).uid, list.get(j).uid, "same_device", did));
            }
        }
    });
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼问本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：关系网络风控你用 JanusGraph + HBase 后端，而不是 Neo4j。决策依据是什么？为什么 Neo4j 不行？**

规模决定选型。风控的关系网络有十亿级节点（亿级用户 + 设备 + IP + 手机号 + 银行卡）、百亿级边（同设备、同 IP、转账等关系）。Neo4j 社区版单机部署（数据量上限几十亿，十亿节点 + 百亿边单机扛不住），企业版才支持集群但收费昂贵（百万级 license）。JanusGraph 是开源分布式图数据库，后端用 HBase/Cassandra（存储无限扩展）+ Elasticsearch（属性索引），适配大规模。决策依据是节点规模——我们实测 Neo4j 社区版在 1 亿节点时查询 RT 从毫秒涨到秒级（单机内存放不下全图索引），JanusGraph 在 10 亿节点时邻域查询仍 <50ms（分布式存储 + 索引分片）。对十亿级图，只有分布式图数据库可行，Neo4j 适合中小规模（千万级节点）。

### 第二层：证据与定位

**Q：风控决策时查"某 UID 是否在高风险团伙"（图查询），P99 RT 突然从 30ms 涨到 3 秒。你怎么确认是图数据库慢还是图本身变大？**

两组证据定位：
1. JanusGraph 的查询指标——`janusgraph.query.latency`（查询 RT）和 `janusgraph.query.vertex.count`（查询涉及的节点数）。如果 RT 3 秒且 vertex.count 几十万，是这个 UID 的邻域太大（关联节点多，图查询爆炸）。如果 RT 3 秒但 vertex.count 几百，是底层存储（HBase）慢。
2. HBase 层——JanusGraph 的节点和边存 HBase，看 HBase 的 `regionserver.Server.Get_99th` 和 `Scan_99th`。JanusGraph 的图查询转成多次 HBase Get/Scan，如果 HBase 慢，整体慢。看 HBase 的 Region 分布，如果某 Region 超大（JanusGraph 的 vertex ID 哈希不均导致热点），单 Region 慢拖整体。
3. 查询模式——看慢查询的 Gremlin 语句，如果是 `g.V(uid).out().out().dedup().count()`（2 跳邻居去重计数），节点数指数爆炸（一跳几百、二跳几万、三跳百万）。限制跳数（在线只查 1-2 跳）是图查询的基本约束。

### 第三层：根因深挖

**Q：你发现是某个超级节点（如某个公共 Wi-Fi 的 IP 节点，关联了几百万个 UID）导致查询爆炸——这个 IP 的邻域查询要遍历几百万节点。根因是什么？怎么处理超级节点？**

根因是"超级节点"（hub node）问题，是图数据库的经典坑。公共 IP（如公司 Wi-Fi、学校网络）天然关联大量用户，查询这个 IP 的邻居要遍历几百万节点，RT 爆炸。根因是建模时把"高频共享实体"当作普通节点，没做特殊处理。解法有几种：
1. 节点分级——超级节点（度数 >1 万的 IP/Wi-Fi）不参与在线图查询，只做离线分析。在线查询时过滤掉超级节点类型（如只查设备不查公共 IP）。
2. 边采样——超级节点的边只保留"高权重"的（如同设备登录 >10 次的），丢弃低权重的（只登录 1 次的），把度数从百万降到千级。
3. 预计算——超级节点的邻域离线算好（如"这个 IP 的关联 UID 列表"存 HBase），在线查询直接查预计算结果，不实时遍历图。
风控选 1 + 3——公共 IP/Wi-Fi 这种超级节点走预计算（离线标签），设备/银行卡等"正常度数"节点走实时图查询。

**Q：根因是超级节点。那为什么不直接限制图的度数（建边时度数 >1 万就不建）？一劳永逸。**

限制建边会丢信息。超级节点（公共 IP）虽然度数高，但其中可能藏着一小撮欺诈团伙（如 100 个账号都连这个 IP 且这 100 个账号互相转账）。如果直接不建边，这 100 个账号的关联丢失，团伙检测漏。正确做法是"分层处理"——超级节点的边照建（信息不丢），但在线查询时走预计算（把"连这个 IP 的 UID 列表"预先算好存 HBase），不实时遍历。离线分析时仍用完整图（含超级节点）跑社区发现，能挖出那 100 个账号的团伙。所以"限制度数"是错的（丢信息），"分层查询"是对的（在线快、离线全）。代价是预计算的维护成本（IP 关联列表要实时更新），但这是超级节点场景的必要代价。

### 第四层：方案权衡

**Q：你做了超级节点分层处理，在线查询 RT 恢复。但业务说"团伙检测的召回率下降了"（部分团伙漏检）。怎么权衡检测效果和查询性能？**

这是图查询的经典权衡。先量化——召回率从 90% 降到 80%，漏检的团伙特征是什么？如果漏检的都是"小团伙"（5-10 人），且这些团伙的欺诈金额占比低（<5%），可接受。如果漏检的是大团伙（50+ 人），不可接受。权衡方案是分级查询：
1. 实时决策（在线，<30ms）——只查 1 跳邻居 + 预计算标签（团伙标记），覆盖 80% 的团伙（有明显关联的）。
2. 准实时复核（异步，<1 分钟）——对"中等风险"的决策（模型分数中等），异步跑 2-3 跳图查询 + 社区发现，补充检测，结果反馈到后续决策。
3. 离线全量（T+1）——全图社区发现 + 连通分量，挖出所有团伙，事后冻结。
这样实时快（80% 覆盖）、准实时补漏（+15%）、离线兜底（+5%），整体召回 99%+，但实时链路 RT 受控。漏掉的 20% 在准实时/离线补回来（事后冻结账户 + 追损）。

**Q：为什么不直接用 GNN（图神经网络）替代图算法 + 规则？GNN 能学到隐藏的团伙模式，不用人工设计。**

GNN 确实能学复杂图模式，但有局限。一是推理 RT——GNN 推理要计算节点的 embedding（聚合邻居），对十亿级图的在线推理（某 UID 的团伙判断），RT 在秒级（要聚合多跳邻居），超风控 100ms 预算。二是冷启动——新节点（新注册用户）没有邻居信息，GNN 无法推理。三是可解释性——GNN 输出"风险分"，但监管要求"为什么拦截"（可解释），GNN 是黑盒。GNN 的角色是"离线训练 + embedding 预计算"——离线用 GNN 算每个 UID 的图 embedding + 团伙标签，存 HBase，在线查询时直接查标签（毫秒），不实时跑 GNN。这和前面说的预计算团伙标签是一个思路。GNN 是更先进的预计算方法（比传统社区发现精度高），但不替代在线查询的实时性要求。

### 第五层：验证与沉淀

**Q：你怎么验证关系网络风控的有效性——团伙检测的召回率、误杀率怎么量化？**

离线标注 + 线上 AB：
1. 离线评估——用标注的欺诈团伙（已知黑产名单）作为 ground truth，跑关系网络检测，统计"检测出的团伙中有多少是真团伙"（精准率）、"真团伙有多少被检测出"（召回率）。目标召回 >85%、精准 >70%（精准低则误杀多）。
2. 线上 AB——对部分流量开启关系网络规则（对照组只用单点规则），对比两组的欺诈率（拦截后剩余的欺诈交易占比）。如果开启组的欺诈率显著低（如降 30%），证明关系网络有效。
3. 团伙规模分布——统计检测出的团伙规模分布，正常用户聚集应是 1-3 人，团伙应是 10+ 人。如果检出大量 5-10 人的"小团伙"，可能是误判（正常用户被误聚）。

**Q：怎么让团队的关系网络风控持续有效、图数据质量不退化？**

沉淀成机制：
1. 图数据质量监控——节点/边的完整性（是否有缺失实体）、时效性（边是否过期，如 1 年前的同设备关系是否还有效）、准确性（边是否建错，如同设备但实际是家人共用）。每天跑数据质量检查。
2. 图算法版本管理——社区发现算法（Louvain/Label Propagation）的参数调优（分辨率、迭代次数）影响结果，要版本化管理 + AB 验证，不随意改。
3. 团伙标签回流——离线检测的团伙标签要回流到在线（HBase），供实时查询。监控回流延迟（应 <1 小时）。
4. 超级节点巡检——每周统计度数 Top 1000 节点，识别新增超级节点，纳入预计算。
5. 故障复盘——把这次"超级节点导致图查询 3 秒"的 Gremlin 查询、度数分布、分层方案存知识库。


## 结构化回答

**30 秒电梯演讲：** 聊到用图数据库做风控关系网络怎么设计？怎么发现团伙欺诈，我的理解是——关系网络风控把"账号、设备、IP、手机号、银行卡"等实体建成图，用社区发现、中心度、路径分析挖掘团伙欺诈，是单点规则的"上帝视角"。打个比方，关系网络像地铁图——每个站点是一个实体（账号/设备），线是关系（同 IP/同设备/转账）。欺诈团伙像频繁共线的可疑站点簇，图算法能一眼揪出。

**展开框架：**
1. **实体** — 账号、设备、IP、手机、银行卡、商户
2. **关系** — 同设备登录、同 IP、转账、同 Wi-Fi、同联系人
3. **在线查询** — 邻居、度数、最短路径（图数据库 Neo4j/JanusGraph）

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：图数据库选型？您更想看哪个方向？

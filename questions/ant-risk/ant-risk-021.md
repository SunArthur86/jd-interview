# 【蚂蚁风控】设计一个亿级 QPS 的风控系统，你怎么做？

> JD 依据："前瞻性地探索新型风控系统的架构设计"。这是 P7+ 的核心架构题。

## 一、需求拆解

**业务需求**：
- 覆盖蚂蚁国际全业务（支付、借贷、营销、海外）
- 每笔交易实时决策（拦截/放行/复核）
- 风险识别准确（漏报率低、误杀率低）
- 可解释、可审计

**非功能需求**：
- **QPS**：峰值 100 万+（亿级日交易）
- **RT**：P99 < 200ms（用户支付等不及）
- **可用性**：99.99%（不能漏判风险）
- **扩展性**：新业务快速接入
- **可解释性**：每笔决策有依据

## 三、同步决策链路（核心 100ms）

```
1. 接入（5ms）
   ├─ 鉴权（API key/签名）
   └─ 限流（Sentinel：单 UID QPS + 全局 QPS）

2. 黑名单/白名单（5ms，前置短路）
   ├─ 白名单：直放（高管、可信商户）
   └─ 黑名单：直拦（已知欺诈 UID/设备/IP）
   工具：Redis 全量缓存 + 布隆过滤器（内存）

3. 特征查询（30ms，并行）
   CompletableFuture 同时拉：
   ├─ Redis 热数据（10ms）
   ├─ HBase 用户画像（30ms）
   ├─ Flink 实时特征（30ms）
   超时 30ms 全部降级缓存

4. 规则匹配（30ms）
   ├─ 前置规则（强规则，硬拦截）
   ├─ 专家规则（RETE 引擎）
   └─ 决策树（运营配置）

5. 模型推理（30ms）
   ├─ 预算分（离线算好的分，Redis 直查）
   ├─ 在线模型（动态特征推理）
   └─ 模型版本路由（A/B）

6. 决策融合（5ms）
   ├─ 规则 + 模型加权
   ├─ 按业务阈值输出决策
   └─ 输出：决策 + 命中规则 + 风险分

总 RT ≈ 100ms（并行化）
```

**并行化的关键**：
```java
public RiskResult decide(Event e) {
    // 并行查询所有特征
    CompletableFuture<Profile> profileF = supplyAsync(() -> profileService.get(e.uid), pool);
    CompletableFuture<Features> featF = supplyAsync(() -> featService.get(e.uid), pool);
    CompletableFuture<ModelScore> scoreF = supplyAsync(() -> modelService.score(e), pool);

    // 等所有结果（带超时）
    CompletableFuture.allOf(profileF, featF, scoreF).get(80, MS);

    // 决策融合
    return fuse(profileF.join(), featF.join(), scoreF.join());
}
```

## 九、底层本质：分层 + 异步 + 缓存的工程哲学

应对海量流量（日均亿级请求、峰值百万级 QPS）的本质是**"用空间换时间，用异步换吞吐，用分层换效率"**：

**1. 分层（解决决策精度 vs 时效矛盾）**：
- 实时层（毫秒）做粗筛
- 准实时层（秒）做精筛
- 离线层（天）做深度

**2. 异步（解决吞吐 vs 延迟矛盾）**：
- 同步链路做决策（必须快）
- 异步链路做补全（可以慢）
- 解耦强弱依赖

**3. 缓存（解决读 vs 计算矛盾）**：
- 热数据缓存（99% 命中 Redis）
- 预计算缓存（离线算好直接查）
- 多级缓存（本地 + Redis + DB）

**这是分布式系统的"不可能三角"破解**：
- 强一致 + 高可用 + 高性能不能同时满足
- 但可以"分层取舍"：核心强一致，外围最终一致；同步保时效，异步保吞吐

## 十、和 AI 的融合（见 jd-ai 批次）

- **模型层升级**：从规则 + 浅模型 → 大模型实时推理
- **决策层升级**：用 Agent 编排多模型决策
- **特征层升级**：LLM 生成语义特征
- **运营升级**：自然语言配置规则（AI 翻译成 DSL）

## 常见考点
1. **决策链路为什么能压到 100ms**？——并行化（CompletableFuture）+ 缓存（Redis 95% 命中）+ 短路（命中强规则即返回）+ 异步化（非关键走 Kafka）。
2. **怎么保证决策不漏**？——兜底策略 + 异步复核 + 离线全量审计；漏报是风险，但绝不漏决策（必须给个结果）。
3. **怎么评估风控系统好坏**？——核心三率：召回率（漏报）、精准率（误杀）、决策 RT；再加运营效率（规则迭代速度）。

**代码示例**（决策融合）：
```java
public RiskResult fuse(Profile p, Features f, ModelScore s) {
    int riskScore = 0;
    List<String> hits = new ArrayList<>();

    // 强规则（命中即拦）
    if (f.isBlackDevice) { return RiskResult.reject("black_device"); }

    // 专家规则加权
    for (Rule r : expertRules) {
        if (r.match(p, f)) {
            riskScore += r.getWeight();
            hits.add(r.getId());
        }
    }

    // 模型分加权
    riskScore += s.getScore() * MODEL_WEIGHT;

    // 决策
    if (riskScore >= REJECT_THRESHOLD) return RiskResult.reject(hits);
    if (riskScore >= REVIEW_THRESHOLD) return RiskResult.review(hits);
    return RiskResult.pass();
}
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼迫本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你设计亿级 QPS 风控系统把决策链路压到 100ms（并行化 + 缓存 + 短路 + 异步）。为什么不放宽到 200ms？100ms 的决策依据是什么？**

100ms 是"用户无感"的延迟阈值。用户点支付后如果等 200ms 以上，会感知到卡顿（移动端支付体验研究），可能重复点击或放弃交易，影响转化率。100ms 是支付链路的风控预算（整个支付链路 500ms，风控占 100ms、支付处理 400ms）。这个预算是从业务倒推的——产品方约定"风控决策必须 <100ms，否则影响支付体验和转化"。技术决策依据是 SLA，不是拍脑袋。我们做的并行化（CompletableFuture 并发查特征/画像）、缓存（Redis 95% 命中让特征查询 <5ms）、短路（黑名单命中即返回不跑模型）、异步化（非关键的离线复核走 Kafka），都是为了在这个预算内完成"特征 + 规则 + 模型"的融合决策。如果放宽到 200ms，技术简单（可以串行查、少用缓存），但业务损失（转化率降 1-2%）远大于技术成本。

### 第二层：证据与定位

**Q：风控决策 P99 RT 突然从 100ms 飙到 300ms，超 SLA。你怎么快速定位是哪个环节变慢？**

三步定位：
1. 看 Metrics 的链路耗时分布——风控决策的 RT 拆成几段（特征查询、规则匹配、模型推理、决策融合）。Prometheus 的 `risk.decision.duration` 按 span 标签分，看哪段涨了。如果特征查询从 5ms 涨到 200ms，是特征层问题；如果模型推理从 30ms 涨到 150ms，是模型层问题。
2. 看 SkyWalking trace 的慢请求——拉 P99 时段的 trace 列表，看 Top 10 慢请求的 span 树。如果某个 span（如 feature-service.get）耗时 200ms，定位到具体调用。
3. 对比变更时间线——RT 飙升的时间点附近有没有变更（新规则上线、模型升级、下游扩缩容）。如果和某次变更时间吻合，优先怀疑变更引入的问题。

### 第三层：根因深挖

**Q：你发现是模型推理从 30ms 涨到 150ms。模型服务的 GPU/CPU 没变，为什么推理变慢？**

模型推理慢几种根因：
1. 模型输入变大——如果特征的维度增加（如新加了 100 个特征字段），模型推理的计算量增加。看模型服务的 input size 指标，对比推理 RT 和 input size 的相关性。
2. 批处理退化——模型服务通常做 micro-batching（攒多个请求一起推理，利用 GPU 并行）。如果 QPS 波动导致 batch size 变小（每次只推理 1-2 个请求），吞吐下降、单次 RT 上升。看模型服务的 `model.batch.size` 分布，如果 P50 从 32 掉到 2，是 batching 退化。
3. 模型版本切换——如果最近发布了新模型（如从 XGBoost 换成深度学习模型），RT 本身就从 30ms 变 100ms+，是模型选型问题。看模型服务的版本标签对比 RT。
真实案例常见是第 2 种——QPS 下降导致 batch size 变小（模型服务的 dynamic batching 设了等待窗口 10ms，低 QPS 时窗口内只来 2 个请求就推理了，GPU 利用率低）。

**Q：根因是 batch size 退化。那为什么不固定 batch size（攒够 32 个再推理）？**

固定 batch size 会增加延迟。如果 QPS 低，攒够 32 个要等几十毫秒，等待时间加到 RT 里，P99 反而更高。dynamic batching 是"等待窗口（如 10ms）内来多少就攒多少"——高 QPS 时窗口内来 50 个（取 32）、低 QPS 时来 3 个。这是 RT 和吞吐的权衡。根因不是 batching 策略错，是"低 QPS 时段模型 RT 天然高"。解法是分级——高峰期用 dynamic batching（吞吐优先），低峰期 QPS 低时用单请求推理（不攒批，RT 更稳定），或保持模型常驻 GPU 预热（避免冷启动）。风控是 7x24 流量，低谷也有几万 QPS，batch size 不会掉到个位数。如果真掉到个位数，考虑缩小模型（用更轻量的模型替代，单次推理 10ms 以内，不需要 batching）。

### 第四层：方案权衡

**Q：你优化了模型推理 RT。但业务说模型准确率下降（为了快用了轻量模型）。怎么权衡 RT 和准确率？**

这是风控的核心权衡——准确率 vs 延迟。先量化：轻量模型的准确率（如 AUC 0.85）vs 重模型（AUC 0.90），差 0.05 意味着多漏判/误杀一定比例。再量化 RT——轻量模型 10ms、重模型 100ms。权衡方案是分层模型（cascade）：
1. 第一层轻量模型（10ms）——对低风险用户（历史无风险）快速判断，90% 流量在这一层 PASS，RT 10ms。
2. 第二层重模型（100ms）——对中高风险用户（轻量模型分数中等、或有风险历史）用重模型精确判断，10% 流量走这层，RT 100ms 但只影响少数。
3. 第三层人工——极高风险（重模型分数也高）走人工审核。
整体 P99 RT 由第一层决定（10ms），P999（前 1%）由第二层决定（100ms）。准确率上，高风险用户用重模型（更准），低风险用轻量（够用）。这样 RT 和准确率都达标——比"所有请求都走重模型"（RT 100ms）或"都走轻量"（准确率低）都优。

**Q：为什么不直接用 LLM 做风控决策，LLM 推理能力强，准确率更高？**

LLM 推理 RT 是秒级（几百毫秒到几秒），远超风控 100ms 预算。LLM 适合"非实时"或"对延迟不敏感"的风控场景（如事后调查、案件分析、规则生成），不适合"实时决策"（支付时的毫秒级判断）。且 LLM 的推理成本是传统模型的 100-1000 倍（GPU + token 费用），亿级 QPS 用 LLM 成本不可承受。实时风控决策仍用传统模型（XGBoost/深度学习小模型），LLM 作为辅助——离线生成规则、分析复杂案件、辅助人工审核。只有当 LLM 推理 RT 降到毫秒级（如蒸馏小模型 + 边缘部署）且成本可控时，才考虑实时链路引入 LLM。

### 第五层：验证与沉淀

**Q：你怎么验证亿级 QPS 风控系统的 SLA（RT、可用性、准确率）达标？怎么持续监控？**

三个 SLA 维度的监控：
1. RT——`risk.decision.duration` 的 P50/P99/P999，P99 <100ms 是硬指标。按"流量归一化"看（RT / QPS），消除流量波动影响。RT 超标告警（P99 >100ms 持续 5 分钟，P0 告警）。
2. 可用性——决策成功率 >99.99%（每百万请求允许 100 次失败）。失败包括超时、异常、降级。按失败原因分类统计（超时 vs 异常 vs 降级），定位短板。可用性告警（成功率 <99.99%，P0 告警）。
3. 准确率——召回率（漏报率，欺诈交易中未拦截的比例）和精准率（误杀率，拦截交易中正常的比例）。这两个指标要离线 T+1 统计（要等欺诈标签回流），实时只能看代理指标（如拦截率突增可能是规则异常、模型分数分布异常）。召回率下降（漏报多）是 P0，误杀率上升是 P1。

**Q：怎么让团队的风控系统持续达标 SLA、不退化？**

沉淀成机制：
1. SLA 仪表盘——实时展示 RT/可用性/准确率，全员可见，每日 review。
2. 容量规划——每季度做全链路压测（N 倍峰值），验证 RT 和可用性在峰值下达标。容量不够提前扩容。
3. 变更管控——核心链路（决策服务、模型）的变更走灰度（1% → 10% → 100%），每阶段监控 SLA，退化自动回滚。
4. 混沌工程——定期注入故障（杀实例、延迟下游），验证降级和兜底生效，SLA 不受影响。
5. 故障复盘——每次 SLA 破防的故障复盘，根因分析 + 改进措施 + 知识库沉淀。把这次"batch size 退化导致模型 RT 涨"的指标曲线、优化方案存知识库。


## 结构化回答

**30 秒电梯演讲：** 聊到设计一个亿级 QPS 的风控系统，你怎么做，我的理解是——风控系统设计核心是"前置拦截 + 分层决策 + 异步补全 + 弹性降级"——同步链路只做最关键的（<100ms），其他靠异步和兜底。打个比方，风控像机场安检——快速初检（黑名单，毫秒级）→ 详细检查（特征规则，几十毫秒）→ 重点盘问（人工审核，分钟级）。每个环节剔除一部分，越往后越精细化。

**展开框架：**
1. **分层决策** — 实时（毫秒，同步）/ 准实时（秒，异步）/ 离线（天，T+1）
2. **同步链路只做** — 黑名单、特征查询、规则匹配、模型推理
3. **异步补全** — 画像更新、关系网络、案件回流

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：同步链路压到 100ms 怎么做？您更想看哪个方向？

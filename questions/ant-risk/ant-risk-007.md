# 【蚂蚁风控】Spring Cloud 里的熔断和限流怎么实现的？Sentinel 原理？风控怎么用？

> JD 依据："Spring Cloud" + "保障海量数据系统的稳定性"。熔断限流是风控系统高可用的核心组件。

## 一、表面层：为什么需要熔断限流

**雪崩场景**（没有熔断）：
```
特征服务慢查询 → 决策服务调用特征超时 → 决策线程池被占满
   → 决策服务不可用 → 网关调用决策堆积 → 网关线程池满
   → 全链路雪崩
```

**熔断的作用**：特征服务变慢到阈值 → 决策侧熔断器 OPEN → 快速失败返回兜底 → 决策线程不被拖死 → 上游正常 → 故障被"切断"在依赖链路中。

## 二、熔断器三态机（核心机制）

```
        失败率/RT超阈值              定时探测
   ┌──────────────────┐      ┌──────────────────┐
   │                  ▼      │                  │
   │             ┌─────────────┐            │
   │             │   OPEN      │            │
   │             │  (熔断拒绝) │            │
   │             └─────────────┘            │
   │                  │                      │
   │       等待时间到(半开)                  │
   │                  ▼                      │
   │             ┌─────────────┐            │
   │             │ HALF_OPEN   │            │
   │             │ (放1请求试探)│            │
   │             └─────────────┘            │
   │              /       \                 │
   │       成功  /         \ 失败           │
   │            ▼           ▼               │
   │      ┌────────┐   回到 OPEN            │
   └─────│ CLOSED │                        │
         │ (正常) │                        │
         └────────┘                        
```

- **CLOSED**：正常放行，统计失败率
- **OPEN**：直接拒绝（不发起调用），保护本地资源
- **HALF_OPEN**：等待超时后放 1 个请求试探，成功→CLOSED，失败→OPEN

## 三、Sentinel 原理

Sentinel 是阿里开源的流量治理组件，核心是**滑动窗口统计 + 责任链插槽**。

**插槽链**（每个请求经过）：
```
NodeSelectorSlot → ClusterBuilderSlot → StatisticSlot → FlowSlot → DegradeSlot → ...
   选节点         构建集群节点          统计RT/QPS        限流判断    熔断判断
```

**滑动窗口统计**：
```
LeapArray (默认 1 秒，分 2 个 500ms 桶)
   ├─ Window1 [0-500ms]: QPS=200, RT=50ms
   └─ Window2 [500-1000ms]: QPS=180, RT=45ms
滑动：每过 500ms 滚动一个桶
```

**熔断策略**：
1. **慢调用比例**：RT > 阈值的比例超 maxRatio，熔断（适合外部依赖慢）
2. **异常比例**：异常/总请求 > ratio，熔断
3. **异常数**：异常数 > 阈值，熔断

## 四、Sentinel vs Hystrix

| 维度 | Hystrix | Sentinel |
|------|---------|----------|
| 隔离方式 | 线程池（强隔离，开销大） | 信号量（轻量，默认） |
| 熔断策略 | 异常比例 | 慢调用比例 / 异常比例 / 异常数 |
| 限流 | 无（只熔断） | QPS / 并发数 / 关系链路 |
| 流控效果 | 直接拒绝 | 拒绝 / Warm Up / 排队等待 |
| 系统自适应 | 无 | 有（基于 Load1/RT/线程数自适应） |
| 控制台 | 弱 | 强（实时监控+动态规则） |
| 维护状态 | 已停止维护 | 持续更新 |

**风控选 Sentinel**：信号量隔离无上下文切换开销；规则丰富；阿里系生态。

## 五、风控实战配置

**场景 1：决策服务对下游特征的熔断**
```java
@SentinelResource(value = "callFeature",
    blockHandler = "featureFallback",       // 限流熔断的兜底
    fallback = "featureExceptionFallback")  // 业务异常的兜底
public Feature queryFeature(String uid) {
    return featureService.get(uid);          // 可能慢/失败
}

public Feature featureFallback(String uid, BlockException ex) {
    return Feature.empty();                  // 返回空特征，规则用默认策略
}
```

规则（Dashboard 配置）：
- 慢调用比例：RT > 200ms 的比例 > 50% → 熔断 10s
- 最小请求数：5（避免低流量误判）
- 异常比例：> 30% → 熔断

**场景 2：网关限流（防恶意刷）**
```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: risk-invoke
          uri: lb://risk-decision
          predicates:
            - Path=/risk/invoke
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 1000   # 令牌桶填充速率
                redis-rate-limiter.burstCapacity: 2000   # 桶容量
                key-resolver: "#{@userKeyResolver}"      # 按 UID 限流
```

**场景 3：集群限流**
单机限流无法准确控全局 QPS（每台机器看到的只是局部），用 Token Server：
```
风控决策 N 台 → 都向 Token Server 申请令牌
Token Server 集中算全局 QPS，避免单机偏差
```

## 六、流控效果：拒绝 / Warm Up / 排队

| 效果 | 行为 | 适用 |
|------|------|------|
| **直接拒绝** | 超阈值直接抛 BlockException | 大多数场景 |
| **Warm Up** | 阈值从 low 缓升到 high（如 30s 内 10→1000） | 冷启动（缓存刚预热） |
| **排队等待** | 请求匀速通过（漏桶算法），超出排队 | 削峰填谷（消息类） |

**风控的 Warm Up 场景**：发布后缓存冷启动，直接放开 1000 QPS 会击穿缓存 → DB 暴击。用 Warm Up 让流量从 100 缓升到 1000。

## 七、底层本质：流量治理的"保险"机制

熔断/限流本质是给系统装"保险"：
- **熔断 = 自动断路器**：检测下游异常，自动切断保护自己
- **限流 = 流量调节阀**：控制入口流量，让系统在额定功率内运行

它们的**第一性原理**：承认分布式系统的"不可靠性"（网络抖、依赖挂、流量突），不试图消除故障，而是**让故障的影响可控**——单点故障不传染、洪峰不冲垮系统。

**和"高可用三件套"的关系**：
- **限流**：入口截流（防过载）
- **熔断**：依赖隔离（防雪崩）
- **降级**：兜底返回（保可用性）

三者配合，构成系统在高压力下的生存策略。

## 常见考点
1. **滑动窗口和漏桶/令牌桶区别**？——滑动窗口是统计（看历史），漏桶/令牌桶是控制（限流策略）。Sentinel 用滑动窗口统计 + 令牌桶（默认）或漏桶（匀速排队）。
2. **Sentinel 怎么和 Feign 集成**？——`feign.sentinel.enabled=true`，每个 Feign 客户端都可配 fallback。
3. **集群限流一定要 Token Server 吗**？——是的，需要集中算全局 QPS；但可以"伪集群限流"用每机 1/N 阈值近似。

**代码示例**（自定义 Slot 实现业务限流）：
```java
// 按 UID 维度限流（网关层）
@Bean
public KeyResolver userKeyResolver() {
    return exchange -> Mono.just(
        exchange.getRequest().getHeaders().getFirst("X-UID")
    );
}
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：风控决策服务对特征服务的熔断，你配的是慢调用比例策略（RT > 200ms 比例 > 50% 熔断 10s），而不是异常比例策略。为什么？决策依据是什么？**

因为风控链路的故障模式以"慢"为主，不是以"错"为主。特征服务依赖 Redis/HBase，它们的故障表现是 RT 飙升（主从切换、慢查询、Full GC），HTTP 状态码仍返回 200 但耗时 2 秒。如果用异常比例策略，RT 飙到 2 秒但没抛异常时熔断器不触发，决策服务的线程池照样被拖死。慢调用比例策略直接盯 RT，更早发现性能退化。决策依据是历史故障复盘——统计过去半年特征服务的故障，95% 是"慢但不报错"（RT 从 10ms 涨到 500ms+），只有 5% 是真正的 5xx 异常。所以熔断策略要盯主要矛盾。阈值 RT > 200ms 是因为决策链路总预算 200ms，特征段占 30ms，超 200ms 意味着已经超预算 6 倍。

### 第二层：证据与定位

**Q：风控决策服务突然出现大量 BlockException（熔断触发），但特征服务的监控 RT 正常（P99 < 20ms）。你怎么确认是误熔断还是真的有问题？**

三步定位：
1. 看 Sentinel Dashboard 的实时监控——进到 `callFeature` 资源的"熔断"页，看触发原因（Sentinel 区分"慢调用比例"/"异常比例"/"异常数"）。如果是慢调用比例触发，看当时的 RT 统计——如果 Sentinel 的滑动窗口记录的 RT P99 是 500ms，但特征服务自己的监控显示 20ms，这是观测口径差异（Sentinel 统计的是调用方到被调方的端到端 RT，包含网络 + 序列化）。
2. 对比时间戳——BlockException 发生时，立刻在特征服务侧查同时间段的 APM trace，看特征服务收到的请求 RT。如果特征服务侧 RT 正常，但调用方侧 RT 高，问题在网络或调用方自身（如 GC、序列化、连接池等待）。用arthas trace `com.xxx.DecisionService queryFeature '#cost>200'` 看是哪个子调用慢。
3. 检查滑动窗口的统计异常——Sentinel 的 `LeapArray` 滑动窗口在高 QPS 下可能因为时钟回拨或 GC pause 导致窗口统计错乱，用 `arthas ognl '@com.alibaba.csp.sentinel.slots.block.flow.FlowRule@manager` dump 一下当前的窗口数据看是否有异常的 RT 尖峰。

### 第三层：根因深挖

**Q：你定位到是调用方自己的连接池等待——HBase 连接池满了，每次 get 要等连接 500ms。根因是连接池，不是特征服务。那为什么熔断策略没区分"自己的问题"和"下游的问题"？**

因为 Sentinel 的熔断器是"调用方视角"——它只能统计"我调你花了多久"，无法区分这 500ms 是下游处理慢还是我本地等连接慢。这是熔断器的设计边界：它是粗粒度的保护机制，目标是"不管谁的错，先保护我自己不雪崩"。根因要靠后续的 trace 串联定位。这次根因是连接池配置（maxTotal=50）对突发流量不够，解法是调大连接池（maxTotal=200）+ 加连接池监控（`commons-pool2.numActive` 指标）。但熔断策略本身没错——它在连接池满时快速失败，避免了线程池被拖死，是正确的兜底行为。

**Q：那熔断策略就不该盯 RT 吗？既然它分不清是自己的问题还是下游的问题？**

应该盯 RT，但要配合"归因"。熔断的本质是"故障下保命"，它就该一刀切——不管谁的错先断了保命。分清责任是"故障定位"阶段做的事，不是熔断阶段。如果熔断器要等分清责任再决定断不断，判断逻辑复杂、延迟高，反而起不到保护作用。正确做法是两层：熔断层（粗粒度保命，盯 RT）+ 监控告警层（细粒度归因，区分连接池满 vs 下游慢）。熔断触发后自动触发告警，告警里带上"本次熔断期间本地资源水位（连接池 numActive、线程池 active、GC）"和"下游 RT（从 trace 拉取）"，人工或自动归因。

### 第四层：方案权衡

**Q：你熔断后 fallback 返回 Feature.empty()，业务方说空特征会让规则误判（缺特征按低风险处理，放过欺诈交易）。怎么权衡？**

这是经典的"可用性 vs 准确性"权衡。风控的准确性要求高（漏判一笔欺诈损失真金白银），不能简单返回空。权衡方案是分级的 fallback：
1. 轻度降级——熔断时返回"最近一次缓存的特征"（每个 uid 维护一个带 TTL 的本地快照），虽然不是最新但比空好。代价是特征可能滞后几十秒到几分钟，但规则模型对短时滞后有一定鲁棒性。
2. 中度降级——返回缓存特征 + 打一个"降级标记"，规则引擎看到标记后走"保守策略"（提高阈值、增加人工审核环节），不放行高风险额度的交易。
3. 重度降级——缓存也没有的 uid，返回空 + 强制走人工审核（不放行也不拒绝）。
哪种取决于业务——我们和风控策略团队定的 SLA：熔断期间漏判率不得高于正常时的 2 倍。实测"缓存特征 + 保守策略"能控制漏判率在 1.5 倍以内，可接受。直接返回 empty 会让漏判率涨到 10 倍，不可接受。

**Q：为什么不直接扩容特征服务让它不慢，而要搞这么复杂的 fallback？**

因为故障不可预测。扩容是稳态容量规划，但故障是瞬态的（一次 Redis 主从切换 30 秒、一次 HBase Full GC 5 秒），这段时间即便特征服务有 1000 个实例也无济于事——每个实例都在等同一个 Redis。扩容解决的是"流量增长"问题，不是"依赖故障"问题。fallback 解决的是"依赖瞬态故障"问题。两者是正交的：扩容应对流量、熔断+fallback 应对故障。我们会同时做——平时按 1.5 倍峰值扩容，故障时熔断+fallback 兜底。只扩容不熔断，遇到依赖故障照样雪崩。

### 第五层：验证与沉淀

**Q：你怎么证明熔断 + fallback 真的在故障时起作用，而不是规则配错了形同虚设？**

主动故障注入 + 端到端验证：
1. 在预发环境用 ChaosBlade 注入特征服务延迟（`blade create http delay --time 500 --uri /feature/get`），让特征接口 RT 涨到 500ms。
2. 观察 Sentinel Dashboard——`callFeature` 资源的 RT 统计应 5 秒内反映 500ms，慢调用比例超 50% 后熔断器转 OPEN，持续 10 秒。如果熔断器没转 OPEN，说明 minRequest 最小请求数配置过高（如配了 1000 但预发流量才 100），要调低。
3. 验证 fallback——熔断期间压测决策接口，看返回的特征是 empty（未做缓存降级）还是缓存快照（做了降级）。同时统计漏判率（对比正常时段），应 < 正常的 2 倍。
4. 验证恢复——10 秒后半开探测，ChaosBlade 撤销延迟（`blade destroy <uid>`），探测请求成功后熔断器转 CLOSED，RT 恢复正常。如果没恢复，说明半开探测的请求也失败了（延迟还在），要看超时配置。

**Q：怎么让团队所有关键依赖都有熔断保护、配置合理？**

沉淀成规范和工具：
1. 熔断规则模板——按依赖类型给模板：外部 HTTP（慢调用 RT > 500ms 比例 > 50%）、内部 RPC（RT > 100ms 比例 > 30%）、DB（RT > 50ms 比例 > 20%），超出模板要评审。所有 Feign/RPC 调用必须配 @SentinelResource + blockHandler，否则 CR 不通过。
2. fallback 规范——fallback 不能简单返回 null/empty，必须按业务定义降级策略（缓存/默认值/人工审核），降级策略要和业务方签字确认。
3. 故障演练常态化——每季度对所有核心依赖做 ChaosBlade 注入（延迟、异常、断网），验证熔断触发 + fallback 有效 + 恢复正常，形成演练报告。
4. 监控告警——Sentinel 的 `circuit_breaker.open.count` 指标告警，任何熔断器 OPEN 必须告警并自动归因（拉取本地资源水位 + 下游 trace）。


## 结构化回答

**30 秒电梯演讲：** 聊到Spring Cloud 里的熔断和限流怎么实现的，我的理解是——熔断器用"半开探测-恢复"机制防止下游故障向调用链上游蔓延；限流器在入口按 QPS/线程数截断洪峰，二者是高可用系统的"保险丝"和"水闸"。打个比方，熔断像电路保险丝——电流过大自动断电保护电器；限流像水坝泄洪闸——超过库容放掉多余的水。系统恢复时熔断"半开"先试探小流量，正常再"闭合"全放开。

**展开框架：**
1. **熔断三态** — CLOSED（正常）/ OPEN（熔断，快速失败）/ HALF_OPEN（半开探测）
2. **限流维度** — QPS、并发线程数、调用关系（链路）
3. **Sentinel 用滑动窗口统计 +** — Sentinel 用滑动窗口统计 + 信号量隔离；Hystrix 用桶计数 + 线程池隔离

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：Sentinel 和 Hystrix 区别？您更想看哪个方向？

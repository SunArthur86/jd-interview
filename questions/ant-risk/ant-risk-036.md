# 【蚂蚁风控】RAG 怎么用？风控系统的 RAG 应用

> JD 依据："大模型实践"。RAG 是 LLM 应用最广的模式。

## 一、为什么需要 RAG

**LLM 的局限**：
1. **知识截止**：训练数据有截止日期（不知道最新规则）
2. **不知道私有知识**：公司内部规则、案件不在训练集
3. **幻觉**：编造看起来合理但错误的事实
4. **不可追溯**：不知道答案从哪来

**RAG 的解法**：
- 把私有知识存向量库
- 提问时检索相关片段
- 拼到 prompt 让 LLM 基于片段回答
- 答案带引用可追溯

## 二、RAG 的基本流程

```
离线建库:
  文档 → 切分（chunk）→ Embedding → 存向量库

在线检索:
  问题 → Embedding → 向量检索（找相似 chunk）→ 拼 prompt → LLM 生成
```

**完整流程**：

```
1. 知识准备
   文档（PDF/Markdown/HTML）→ 切分成 chunk（如每段 500 字）

2. 向量化
   每个 chunk → embedding 模型 → 向量（如 1536 维）

3. 入库
   向量 + 原文存向量库（Milvus/Pinecone）

4. 查询时
   用户问题 → embedding → 向量库检索 Top-K 相似 chunk

5. 拼 prompt
   "基于以下资料回答: {retrieved_chunks}
    问题: {question}"

6. LLM 生成
   基于检索结果回答，带引用
```

## 三、向量库选型

| 向量库 | 特点 | 适用 |
|--------|------|------|
| **Milvus** | 开源、分布式、高性能 | 大规模生产 |
| **Pinecone** | 托管、易用 | 中小规模 |
| **Weaviate** | 开源、支持混合检索 | 通用 |
| **Faiss** | Facebook 开源、单机库 | 中小规模 |
| **Elasticsearch** | 8.0+ 支持向量 | 已有 ES 团队 |

**风控选 Milvus**：
- 开源、可私有部署
- 分布式、支撑亿级向量
- 阿里有增强版（Proxima）

## 四、风控的 RAG 应用

### 应用 1：相似案件检索
```
新事件 → embedding → 案件库检索 Top-5 相似历史案件
拼 prompt: "这是新事件: {event}
           历史相似案件: {cases}
           决策:"
LLM: 基于历史案件推理新事件
```

### 应用 2：规则库问答（运营助手）
```
运营: "什么场景该用规则 R045？"
→ 检索规则库（找到 R045 的描述、适用场景、案例）
→ LLM: "R045 是设备指纹关联多账号规则，适用于..."
```

### 应用 3：风控知识助手
```
客服: "这个用户为什么被拦截？"
→ 检索该用户决策日志 + 规则库
→ LLM: "用户被拦截因为命中 R001（异地登录）和 R045（设备关联）"
```

### 应用 4：欺诈话术识别
```
聊天文本 → embedding → 话术库检索
→ 命中"代付话术" → 标记风险
```

## 五、高级 RAG 技术

**朴素 RAG 的问题**：
- 检索不准（embedding 不匹配）
- chunk 切分差（重要信息被切断）
- 查询和文档语义差距

**改进技术**：

### 1. 查询改写（Query Rewriting）
```
原始问题: "这个用户有问题吗"
改写: "查询用户 A 的风险评分、历史决策、关联账号"
→ 改写后再检索，更精准
```

### 2. HyDE（Hypothetical Document Embeddings）
```
问题 → LLM 先生成"假设答案"
用假设答案的 embedding 检索（比问题本身更接近文档）
```

### 3. 重排序（Re-ranking）
```
向量检索召回 Top-50（粗排）
用 cross-encoder 重排到 Top-5（精排）
精度大幅提升
```

### 4. 多跳检索（Multi-hop）
```
问题需要多步推理:
  "用户 A 关联的中介是谁？"
  1. 查 A 的关联（找 B）
  2. 查 B 是否是中介
→ 多次检索
```

### 5. 混合检索
```
向量检索（语义）+ 关键词检索（精确）
合并结果（RR Fusion）
→ 既懂语义又精确
```

## 六、chunk 切分策略

**切分影响检索质量**：

**朴素切分**：
```
按固定长度切（如 500 字），按句号边界
问题：可能切断语义
```

**结构化切分**：
```
按文档结构（章节、段落、列表）
问题：需要解析文档结构
```

**语义切分**（推荐）：
```
用 embedding 检测语义边界
语义连贯的一段切成一个 chunk
```

**重叠切分**：
```
chunk 之间重叠 50-100 字
避免边界信息丢失
```

**风控规则库的切分**：
```
按规则单元切（每条规则一个 chunk）
含：规则 ID、描述、适用场景、案例
```

## 八、评估 RAG 系统

**检索质量**：
- **召回率**：相关文档是否被召回
- **精确率**：召回的文档是否相关
- **MRR**（Mean Reciprocal Rank）：相关文档排名

**生成质量**：
- **答案准确性**：是否基于检索结果
- **答案完整性**：是否覆盖关键信息
- **幻觉率**：是否编造未检索到的内容

**RAGAS 框架**：
- 自动化评估 RAG 系统
- 包含检索和生成的多维度指标

## 十、性能优化

**1. 检索缓存**：
- 相似问题复用结果
- Redis 缓存查询 embedding

**2. 异步预检索**：
- 用户输入时实时检索（边输入边查）
- 减少最终响应时间

**3. 增量更新**：
- 新规则/案件增量入库
- 不重建全库

**4. 多级缓存**：
- 答案缓存（同问题直接返回）
- 检索结果缓存
- Embedding 缓存

## 十一、底层本质：检索 + 生成的解耦哲学

RAG 的核心思想是**"知识的外置"**：

**传统 LLM**：知识在参数里（训练时固化）
**RAG**：知识在外部库（可动态更新）

**这是软件工程"代码 vs 数据"的分离**：
- LLM 参数 = 代码（推理逻辑）
- 向量库 = 数据（知识）
- 数据可独立更新，不用改代码（重训）

**好处**：
- 知识更新快（不用重训 LLM）
- 可控（删除某知识 = 删 chunk）
- 可追溯（答案带引用）

**代价**：
- 检索延迟（+100-500ms）
- 检索质量影响生成质量

## 十二、RAG vs 微调

| 维度 | RAG | 微调 |
|------|-----|------|
| 知识更新 | 快（更新库） | 慢（重训） |
| 适合 | 知识频繁变 | 固定模式 |
| 成本 | 低（不用训） | 高（需要数据+算力） |
| 可控性 | 高（删 chunk） | 低（重训才改） |
| 性能 | 检索 +100ms | 无额外延迟 |

**风控的选择**：
- 规则/案件：RAG（频繁更新）
- 风控决策模式：微调（固定模式，如"什么样的交易是欺诈"）

**组合**：RAG（实时知识） + 微调（基础能力）

## 常见考点
1. **RAG 检索不准怎么办**？——优化 embedding 模型、改 chunk 策略、加 re-ranking、查询改写。
2. **RAG 和微调什么时候选**？——知识更新频繁选 RAG，固定模式选微调，最佳是组合。
3. **向量库为什么用近似检索**？——精确 KNN 在亿级向量下不可行（O(n)），近似检索（HNSW/IVF）O(log n) 牺牲一点精度换百倍速度。

**代码示例**（RAG 实现）：
```python
def rag_answer(question, top_k=5):
    # 1. Embedding
    q_emb = embedding_model.encode(question)

    # 2. 向量检索
    chunks = milvus.search(
        collection="risk_knowledge",
        vector=q_emb,
        top_k=top_k * 2  # 多召回，后续重排
    )

    # 3. Re-ranking
    reranked = cross_encoder.rank(question, chunks)[:top_k]

    # 4. 构造 prompt
    context = "\n".join([c.text for c in reranked])
    prompt = f"基于以下资料回答:\n{context}\n\n问题: {question}\n回答（带引用）:"

    # 5. LLM 生成
    answer = llm.complete(prompt)
    return answer
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：风控知识库用 RAG 而不是微调 LLM，核心动机是什么？**

知识更新频率决定的。风控的规则、案件、政策是高频更新的——每周新增欺诈案件、每月调整风控规则、监管政策随时变。微调的成本：每次知识更新都要重新标注 + 训练 + 评测 + 部署，周期至少 1 周，跟不上业务节奏。RAG 的优势：知识更新只需"写入向量库"（秒级生效），LLM 推理时检索最新知识，无需重训。另外，风控要求"答案可追溯到具体文档"（监管审计），RAG 的检索结果天然带引用（"根据《XX 规则》第 3 条"），微调做不到（知识融在权重里无法追溯）。所以风控知识库的场景，RAG 是更优解。

### 第二层：证据与定位

**Q：风控 RAG 助手上线后，用户反馈"答案不准"。你怎么定位是检索召回问题还是 LLM 生成问题？**

分两步隔离：
1. 看检索环节——把用户问题和 RAG 检索到的 top-k 片段拉出来人工判断"这些片段是否包含正确答案"。如果 top-k 里有正确片段但 LLM 答错了，是生成环节问题（LLM 没用对检索结果，可能 prompt 写得差或 LLM 能力弱）。如果 top-k 里根本没有正确片段，是检索召回问题。
2. 看检索召回率——如果是检索问题，进一步看：正确片段在向量库里的相似度排名是多少？如果正确片段排名 top-50 但 top-5 没召回，是 chunk 切分或 embedding 质量问题；如果正确片段和问题的向量相似度本身就很低（cosine < 0.5），是 embedding 模型选错了或查询和文档语义不匹配（比如用户问"套现"但文档写的是"虚假交易套取资金"）。

### 第三层：根因深挖

**Q：你定位到是 chunk 切分导致召回率低——一份风控规则文档被切成 512 token 的 chunk，但一条完整规则跨了两个 chunk。为什么不用更大的 chunk（比如 2048）？**

大 chunk 引入新问题。第一，检索精度下降——2048 token 的 chunk 里可能包含 5 条规则，向量是 5 条规则的"语义平均"，用户问第 3 条规则时，这个 chunk 的相似度不如只含第 3 条的 512 chunk 高（被其他规则稀释）。第二，prompt 膨胀——top-5 召回 5 × 2048 = 10K token，prompt 太长导致 LLM 注意力分散 + 成本高。正确做法不是盲目调大 chunk，而是"语义切分"——按规则边界切（一条规则一个 chunk），而不是按固定 token 数切。用 LLM 或规则解析器识别"这是一条完整的规则"再切，保证每个 chunk 语义完整。如果一条规则确实长（> 1024 token），用"重叠切分"（overlap 100 token）保证不切断。

**Q：那为什么不直接把整篇文档作为一个 chunk，让 LLM 自己找？反正 LLM 有长上下文（128K）？**

长上下文不等于好效果。第一，"Lost in the Middle"现象——研究表明 LLM 对长 prompt 中间部分的信息利用率显著低于首尾，把整篇文档塞进去，中间的规则反而被忽略。第二，检索精度——整篇文档作为一个 chunk，向量是全文的语义摘要，用户问某条具体规则时相似度不高（被其他内容稀释），可能根本召不回这篇文档。第三，成本——每次查询都塞 128K token，token 成本爆炸（每次 0.5 元），高频场景扛不住。RAG 的核心价值就是"精准定位 + 小 prompt"，把 LLM 从"大海捞针"变成"定向查找"。所以 chunk 要小而精（语义完整的 256-512 token），用 top-k 召回最相关的几个，而非整篇塞进去。

### 第四层：方案权衡

**Q：风控 RAG 用向量检索（embedding 相似度）还是 BM25（关键词匹配）？为什么？**

混合检索（Hybrid Search）最优，不是二选一。向量检索擅长"语义匹配"——用户问"代付风险"，能召回写"第三方代偿"的文档（语义相似但词不同）。BM25 擅长"精确匹配"——用户搜特定术语"反洗钱 35 号文"，向量检索可能召回一堆语义相关但不精确的文档，BM25 能精准命中包含这个词的文档。风控知识库两者都需要：语义场景（"这算不算套现"）走向量，精确查询（"监管文件编号"）走 BM25。做法是两路检索各召回 top-k，用 RRF（Reciprocal Rank Fusion）融合排序，取最终 top-5。这比单用任一种的召回率高 15-20%。

**Q：高级 RAG 有很多技术（HyDE、查询改写、多跳检索），为什么不全用上，反而要选择性？**

每个技术有成本和适用场景，全用会"过度工程化"。HyDE（假设性文档生成）——先用 LLM 生成一个"假想答案"再用它检索，适合"查询和文档语义差距大"的场景（用户问得模糊），但每次多一次 LLM 调用（成本翻倍）。查询改写——把用户问题改写成多个子问题分别检索，适合复杂查询（"风控规则和监管政策的冲突"），但简单查询用不上。多跳检索——检索 → 读答案 → 再检索，适合需要推理链的场景（"这条规则引用了哪条上位法"），但延迟高（多次往返）。所以正确做法是"按场景路由"——简单查询走基础 RAG（快），复杂查询走高级 RAG（准），而不是所有查询都走最重的链路。判断"简单 vs 复杂"可以用规则（问题长度、关键词）或轻量分类模型。

### 第五层：验证与沉淀

**Q：你怎么评估风控 RAG 系统的质量，证明它比传统关键词搜索好？**

用 RAG 评估框架（如 RAGAS）量化：
1. 检索质量——Context Precision（召回的 chunk 里有多少相关）、Context Recall（相关的 chunk 有多少被召回）。对比向量检索 vs BM25 vs 混合检索，混合应最优。
2. 生成质量——Faithfulness（答案是否忠于检索内容，无幻觉）、Answer Relevancy（答案是否回答了问题）。对比 RAG vs 纯 LLM（无检索），RAG 的 Faithfulness 应显著高（因为基于事实）。
3. 业务指标——用户满意度（点赞/点踩）、人工标注准确率（抽 200 条答案标注对错）、解决率（用户问完没再转人工）。最终看业务指标——RAG 助手的解决率从 40%（关键词搜索）提升到 70%，才算真正有效。

**Q：RAG 系统的知识库怎么持续维护，避免"上线即巅峰，之后越来越差"？**

三件事：
1. 知识更新流程——新规则/案件上线时，自动触发"文档入库"流程（解析 → 切分 → embedding → 写向量库），保证知识实时性。下线的规则要从向量库删除（避免召回过期知识）。
2. 质量监控——统计"RAG 答案被用户点踩"的 case，定期人工 review 这些 case 的检索结果和生成答案，定位是知识缺失还是检索/生成问题，持续优化。
3. 评估回归——维护一个"标准问答集"（200 条人工标注的正确答案），每次向量库更新或 prompt 修改后跑一遍，防止"改好一个 case 弄坏一批 case"的回归。


## 结构化回答

**30 秒电梯演讲：** 聊到RAG 怎么用？风控系统的 RAG 应用，我的理解是——RAG（检索增强生成）让 LLM 基于私有知识回答——检索相关文档片段拼到 prompt，LLM 基于检索结果生成答案，解决 LLM 知识滞后和幻觉。打个比方，RAG 像 LLM 的"开卷考试"——LLM 不用死记所有知识（参数化），考前给本"参考书"（向量库），答题时翻书（检索）抄相关段落，再综合回答。

**展开框架：**
1. **RAG = 检索 + 生成**
2. **流程** — Embedding → 向量检索 → 拼 prompt → LLM 生成
3. **向量库** — Milvus/Pinecone/Weaviate

# 【蚂蚁风控】设计一个特征平台，支持亿级用户、千万特征、毫秒级查询

> JD 依据："智能化数据平台"。这是 JD 明确提到"优先"的方向。

## 一、需求拆解

**业务需求**：
- 支撑风控规则、模型、关系网络
- 千万级特征定义，亿级用户实例
- 实时特征（毫秒）、离线特征（T+1）

**技术需求**：
- 在线查询 < 30ms（决策同步链路）
- 实时特征写入延迟 < 1s
- 离线计算吞吐支持千亿数据
- 离线在线口径 100% 一致

## 二、整体架构

## 三、特征定义（统一 DSL）

```yaml
# 特征定义文件
features:
  - name: user_trade_cnt_1h
    type: REAL_TIME
    description: "用户近 1 小时交易次数"
    sql: |
      SELECT uid, COUNT(*) as cnt, HOP_START(1h, 5min) as w_start
      FROM trades
      GROUP BY uid, HOP(proctime, INTERVAL '5' MINUTE, INTERVAL '1' HOUR)
    storage:
      online: { type: HBASE, table: feat_realtime, ttl: 7d }
      offline: { type: HIVE, table: feat_realtime_hist }
    owner: risk-team
    version: 2.1.0

  - name: user_amount_sum_90d
    type: OFFLINE
    description: "用户近 90 天累计金额"
    sql: |
      SELECT uid, SUM(amount) FROM trades
      WHERE dt >= date_sub(current_date, 90)
      GROUP BY uid
    storage:
      offline: { type: HIVE, table: feat_offline }
      online: { type: HBASE, table: feat_offline_online }  # T+1 推到在线
    schedule: "0 2 * * *"   # 每天 2 点跑
    owner: risk-team
    version: 1.0.0
```

**关键设计**：
- **统一 SQL**：离线和实时用同一份逻辑（避免口径不一致）
- **声明式**：算法只定义，平台执行
- **版本化**：每个特征有版本，支持回滚

## 四、特征计算层

**离线计算（Spark）**：
- 跑 T+1 全量
- 千亿数据，Spark 数百核并行
- 输出写 Hive + 推 HBase（在线查询用）

**实时计算（Flink）**：
- 订阅 Kafka 事件流
- 滑动窗口聚合
- 写 HBase + Redis（热缓存）

**双轨一致性**：
- 同一份 SQL 在 Spark 和 Flink 双引擎跑
- 每天 4 点对账（离线 vs 实时结果）
- 偏差 > 阈值告警

## 五、特征存储层

**在线存储（决策查询）**：

| 存储 | 角色 | 数据量 | RT |
|------|------|--------|-----|
| **Redis** | 热缓存（95% 命中） | 100GB（亿级 UID × 热特征） | <10ms |
| **HBase** | 兜底 + 全量 | 100TB（亿级 × 千特征） | <30ms |

**离线存储（训练/分析）**：

| 存储 | 用途 |
|------|------|
| **Hive** | 历史特征仓库（T+1 全量） |
| **Iceberg** | 特征版本化（时间旅行） |

## 六、特征服务层（API）

**统一查询接口**：
```java
public interface FeatureService {
    // 单点查（最快）
    Map<String, Object> get(String uid, List<String> featNames);

    // 批量查（一次请求合并）
    Map<String, Map<String, Object>> batchGet(List<String> uids, List<String> featNames);

    // 模型专用（特征向量）
    FeatureVector vector(String uid, String modelName);
}
```

**实现**：
```java
public class FeatureServiceImpl implements FeatureService {
    public Map<String, Object> get(String uid, List<String> feats) {
        // 1. 先查 Redis（Pipeline 批量）
        Map<String, Object> result = redisBatchGet(uid, feats);
        List<String> missing = findMissing(result, feats);

        if (!missing.isEmpty()) {
            // 2. 缺失的查 HBase
            Map<String, Object> fromHBase = hBaseGet(uid, missing);
            result.putAll(fromHBase);
            // 3. 回填 Redis（异步）
            asyncBackfillRedis(uid, fromHBase);
        }
        return result;
    }
}
```

**性能优化**：
- Redis Pipeline：批量查 N 个特征一次 RTT
- HBase 列族：特征按访问模式分族，避免读不需要的列族
- 本地缓存：Caffeine 缓存活跃用户（命中率 60%）

## 七、特征治理

**特征目录**（Feature Catalog）：
- 所有特征可搜索、可浏览
- 描述、字段、计算逻辑、覆盖率
- 用过的模型列表（依赖关系）

**特征版本**：
- 每个变更出新版本
- 训练用历史版本（保证一致）
- 在线用最新版本（灰度切换）

**特征质量监控**：
- **覆盖率**：null 比例
- **稳定性（PSI）**：分布漂移
- **重要性**：模型权重变化
- **新鲜度**：实时特征延迟

## 九、底层本质：特征即数据资产

特征平台本质是把"特征"从代码资产升级为**数据资产**：
- 代码资产：每个算法自己写特征代码
- 数据资产：特征是平台管理的可复用资源

**这是从"手工作坊"到"工业流水线"的转变**：
- 定义标准化（DSL）
- 生产规模化（批+流）
- 流通统一（API）
- 质量可控（治理）

类比：
- 数据库时代：表是数据资产
- 数据仓库时代：表+视图是数据资产
- 机器学习时代：特征是数据资产
- LLM 时代：embedding 是数据资产

**和 AI 的关系**：
- LLM 推理需要特征（用户/物品特征作为上下文）
- LLM 训练数据本身就是"广义特征"
- 未来：特征平台 = AI 数据供给平台

## 常见考点
1. **特征平台和数据库区别**？——特征是"加工后的派生数据"，平台管"计算+存储+服务+版本+治理"，比 DB 多了计算层。
2. **离线在线怎么保证一致**？——同 SQL 双跑 + 对账 + 一致性告警。
3. **怎么处理冷启动（新用户无特征）**？——默认值、相似用户群体统计、初始规则（无特征用规则）。

**代码示例**（特征质量监控）：
```java
// 每天跑一次特征质量检查
public void checkFeatureQuality(String featName) {
    // 覆盖率
    double coverage = computeCoverage(featName);
    if (coverage < 0.8) alert("低覆盖", featName);

    // PSI（群体稳定性，对比 7 天前）
    double psi = computePSI(featName, now(), minus7d());
    if (psi > 0.25) alert("分布漂移", featName);

    // 新鲜度（实时特征延迟）
    long lag = computeLag(featName);
    if (lag > 5000) alert("实时延迟", featName);
}
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼问本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你设计的特征平台支持"亿级用户、千万特征、毫秒级查询"。千万特征这个量级是真的需求还是夸张？业务真的用到这么多特征吗？**

千万特征是"特征候选池"的量级，不是每个决策都用全部。风控的算法团队持续探索新特征（用户行为序列、设备关系网络、地理时序），历史积累 + 在研特征上千万。单次决策只用其中几百个（按场景选）——支付决策用 200 个、借贷决策用 500 个、营销决策用 100 个。平台要支持"千万候选 + 每次选几百"的灵活组合。决策依据是算法团队的特征仓库规模——实际注册的特征 800 万，加上历史版本（v1/v2/v3）和实验特征（A/B 测试），总量 1500 万。如果平台不支持千万级，算法探索受限于平台容量，是瓶颈。所以"千万特征"是前瞻性设计，匹配算法团队的迭代速度。

### 第二层：证据与定位

**Q：特征查询的 P99 RT 突然从 5ms 涨到 50ms。你怎么确认是特征平台慢、还是底层存储（HBase/Redis）慢？**

沿查询栈分层定位：
1. 特征服务层——看 `feature.query.latency`（应用层 RT）和 `feature.store.latency`（存储层 RT）。如果应用 RT 50ms 但存储 RT 5ms，差值在应用层（序列化、过滤逻辑慢）。如果两者都 50ms，是存储慢。
2. Redis 层——`redis.cmd.latency`（Redis 命令 RT），看 HGET/HMGET 的 P99。如果 Redis RT 50ms，是 Redis 慢（大 key、慢命令、网络）。用 `redis-cli --bigkeys` 找大 key、`SLOWLOG GET` 看慢命令。
3. HBase 层——`regionserver.Server.Get_99th`，如果 HBase Get P99 50ms，是 HBase 慢（BlockCache 失效、Compaction、Region 热点）。
4. 按特征维度归因——如果某个特征（如 user_trade_sequence）查询特别慢，可能 Value 太大（存了长序列），序列化/传输慢。按 feature_name 维度统计 RT，找慢特征。

### 第三层：根因深挖

**Q：你定位到是 Redis 层慢——某个特征 key（feat:uid_500:sequence）的 Value 是 2MB（存了用户最近 1000 次交易序列）。根因是什么？为什么特征存这么大？**

根因是"特征设计不当"。用户交易序列是高基数特征（每次交易追加），直接存原始序列，Value 会无限增长。Redis 是单线程模型，单次操作 2MB 的 Value 会阻塞 Redis 数毫秒（序列化 + 网络），高 QPS 下 Redis CPU 飙升、所有特征查询变慢。根因不是 Redis 慢，是特征存储设计把"大对象"塞进 Redis。正确做法是特征预聚合——把"1000 次交易的序列"在离线/实时算成"聚合特征"（交易频率、金额均值、变异系数），只存几十字节。如果模型确实需要原始序列（序列模型），用专门的序列存储（SequenceDB 或 Redis Stream）+ 分页查询，不塞进单个 key。

**Q：根因是特征太大。那为什么不直接给 Redis 扩容（内存够大就不在乎大 key）？**

扩容治标不治本。Redis 是单线程，瓶颈不是内存容量而是"单次操作耗时"。2MB 的 Value 即使内存够，单次 GET + 网络传输也要 5-10ms（千兆网络传 2MB 约 16ms），Redis 的单线程在这期间不能处理其他请求，QPS 掉到几百。扩容（加内存）不能解决单线程瓶颈——内存再大，单次操作慢就是慢。要分片（大 Value 拆成多个小 key）或换存储。风控特征查询要求 P99 <5ms，单次操作不能超过 1ms，2MB 的 Value 天然超限。根因是特征设计，扩容是错误方向。

### 第四层：方案权衡

**Q：你把序列特征改成了聚合特征（存几十字节）。但算法团队说"聚合丢失了时序信息，序列模型效果下降"。怎么权衡特征精度和存储效率？**

分层存储权衡。算法要的"时序信息"和存储要的"小 Value"不矛盾，关键是分级：
1. 实时特征（Redis，<5ms）——存聚合值（频率、均值），满足树模型、LR 等。
2. 准实时特征（HBase，<30ms）——存近期序列（最近 100 次），满足序列模型基本需求。
3. 离线特征（Hive/HDFS，秒级）——存全量历史序列，离线训练用。
序列模型在实时决策时取"近期 100 次"（HBase），离线训练时取全量（HDFS）。聚合特征（Redis）给非序列模型。算法精度（序列信息）和存储效率（小 Value）都满足，只是分级存放。代价是查询链路变复杂（模型要声明用哪级特征），但特征平台统一管理，模型配置化选特征源。

**Q：为什么不直接用向量数据库（如 Milvus）存特征？向量数据库天生支持高维特征。**

向量数据库适合"向量相似度搜索"（embedding 的 KNN 查询），不适合"结构化特征查询"。风控的特征是结构化的（交易次数=8、金额=6000），不是向量。即使序列特征转成 embedding，也是离线训练时算，在线决策时不实时算 embedding（太慢），而是查预计算的 embedding（存 Redis）。所以向量数据库的角色是"离线 embedding 计算和检索"，不是"在线特征存储"。在线特征存储仍用 Redis（快、结构化）+ HBase（大容量）。两者互补，不替代。

### 第五层：验证与沉淀

**Q：你怎么验证特征平台的质量——特征准确、新鲜、可用？怎么量化"特征健康度"？**

三个量化指标：
1. 离线在线一致性——每天对账，同特征在离线（Spark/Hive）和在线（HBase/Redis）的值，按 uid 对比，差异 >1% 的 uid 占比应 <0.1%。不一致说明实时计算有 bug 或延迟。
2. 特征新鲜度——实时特征的 `feature.update.lag`（最后更新时间 vs 当前时间），P99 <1 分钟。lag 大说明 Flink 计算延迟或写入慢。
3. 特征覆盖率——`feature.coverage`（有值的 uid 占比），核心特征应 >95%。覆盖率低可能是特征源缺失或计算 bug。

**Q：怎么让团队的特征开发规范、特征质量持续达标？**

沉淀成规范和平台：
1. 特征注册制——所有特征必须在特征平台注册（名称、SQL、类型、TTL、owner），平台统一计算和存储。禁止各团队自己算特征。
2. 特征质量门禁——新特征上线前必须通过质量检查（覆盖率 >80%、PSI <0.25、离线在线一致），不达标不发布。
3. 特征下线机制——长期不用（90 天无查询）的特征自动下线，释放存储和计算资源。
4. 监控基线——每个特征的覆盖率、新鲜度、查询 RT 上报 Prometheus，异常告警。特征质量报表每周 review。
5. 故障复盘——把这次"序列特征 2MB 大 Value 拖慢 Redis"的 Redis 慢日志、大 key 截图、聚合重构方案存知识库。


## 结构化回答

**30 秒电梯演讲：** 聊到设计一个特征平台，支持亿级用户、千万特征、毫秒级查询，我的理解是——特征平台是"特征的定义、计算、存储、服务"一站式平台，解决特征复用、口径一致、上线慢的痛点。打个比方，特征平台像中央厨房——食材（原始数据）→半成品（特征）→配送（在线服务）。各餐厅（业务）共享中央厨房，避免各自备料。

**展开框架：**
1. **特征生命周期** — 定义 → 计算 → 存储 → 服务 → 治理
2. **离线特征（Spark/Hive）** — 离线特征（Spark/Hive）+ 实时特征（Flink）双轨
3. **在线存储** — HBase（持久）+ Redis（热缓存）

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：离线在线一致性怎么保证？您更想看哪个方向？

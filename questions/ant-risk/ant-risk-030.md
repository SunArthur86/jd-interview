# 【蚂蚁风控】怎么设计一个 99.99% 可用性的亿级 QPS 风控架构？

> JD 依据："保障海量数据系统的性能和稳定性"。这是高可用的综合架构题。

## 一、需求拆解

**可用性目标**：
- 99.99%（年停 52 分钟、月停 4.3 分钟）
- 风控是金融核心，不能漏判
- 故障要在秒级到分钟级恢复

**挑战**：
- 单机房挂（电力、网络、火灾）
- 单实例挂（OOM、Bug、硬件）
- 网络抖动（延迟、丢包）
- 数据中心级灾难（地震、洪水）

## 二、高可用三要素

### 1. 冗余（Redundancy）—— 多副本
```
单点故障率: 1%
↓ 加 1 个副本（互备）
故障率: 1% × 1% = 0.01%
↓ 加 2 个副本
故障率: 1% × 1% × 1% = 0.0001%
```

**冗余层次**：
- 实例冗余：多实例（K8s 多副本）
- 机架冗余：跨机架调度
- 机房冗余：多机房部署
- 区域冗余：多区域（异地）

### 2. 隔离（Isolation）—— 爆炸半径
```
故障必然发生，但限制影响范围
```
- 业务隔离：借贷和支付独立集群
- 资源隔离：每个下游独立线程池
- 故障域隔离：单元化（按 UID 切）

### 3. 自愈（Self-Healing）—— 自动恢复
- 健康检查 + 自动重启
- 流量自动摘除故障实例
- 自动故障切换（主从切换）

## 三、多机房架构

### 同城双活（强一致）

```
上海机房 A（主）──专线── 上海机房 B（备）
   50km 内
   延迟 < 5ms（强一致可行）
```

**特点**：
- 跨机房强一致（同步复制）
- 单机房挂不影响业务
- 适合金融核心

**数据同步**：
- MySQL：半同步复制（主写 B 收到 ACK 才返回）
- Redis：Redis Cluster 跨机房
- 配置：Nacos 跨机房集群

### 异地灾备（最终一致）

```
上海主集群 ──异步── 杭州灾备集群
   1000km
   延迟 30ms（强一致不可行）
```

**特点**：
- 异步复制（秒级延迟）
- 大灾难时切换
- 数据可能丢失几秒（业务可接受）

### 三地五中心（极致）

```
上海 + 上海 + 杭州 + 深圳 + 北京
```
**蚂蚁的实践**：五副本多数派可用（CAP 中的 CP）。

## 四、单元化架构（蚂蚁的精髓）

**单元化**：按 UID 切分单元，每单元独立完整链路。

```
单元 A（UID 0-1000万）
   ├─ 风控决策服务
   ├─ 特征服务
   ├─ 规则服务
   ├─ Redis（A 的数据）
   └─ MySQL（A 的数据）

单元 B（UID 1000万-2000万）
   └─ 完整独立链路

单元 C（UID 2000万+）
   └─ 完整独立链路
```

**路由**：
```java
public Unit route(String uid) {
    int unitNo = hash(uid) % UNIT_COUNT;
    return units[unitNo];
}
```

**好处**：
- 爆炸半径到单元级（一个单元挂只影响 1/N 用户）
- 横向扩展（加单元加容量）
- 数据分散（每单元数据量小）

**挑战**：
- 跨单元调用复杂（理论上不该发生）
- 单元间数据同步（全局表）
- 单元迁移（用户跨单元）

## 五、容量冗余

**N+1 / N+2 冗余**：
```
日常 QPS 5万，单机 5000 QPS
理论需要 10 台，实际部署 11-12 台

挂 1-2 台仍能撑住日常
```

**弹性扩容**：
```yaml
# K8s HPA
minReplicas: 50
maxReplicas: 500
```

**预留 buffer**：
- CPU 利用率 < 60%
- 内存 < 70%
- 连接池 < 80%

   │ 半同步复制
   ├─ 同城从库（同步）
   └─ 异地从库（异步）

### Redis 高可用

```

```

**1. 健康检查**：

**3. 流量调度**：
```
SLB（负载均衡）→ 健康实例

```

## 八、降级链路（保核心）

**层层降级**：
```
正常路径（所有服务正常）
   ↓ 特征挂
降级 1：用缓存特征
   ↓ 模型挂
降级 2：用规则决策
   ↓ 规则挂
降级 3：用兜底规则（黑名单 + 阈值）
   ↓ 全挂
降级 4：业务优先（默认放行）或风险优先（默认拦截）
```

**风控的选择**：
- 大部分场景：风险优先（宁错杀不漏报）
- 关键场景：业务优先（用户体验）

## 九、可观测性（高可用的眼睛）

**实时监控**：
- 业务：QPS、P99、拦截率、错误率
- 系统：CPU、内存、GC、连接池
- 依赖：Redis、HBase、MySQL 健康

**告警分级**：
- P0：业务不可用 → 电话
- P1：核心指标异常 → 短信
- P2：单实例异常 → IM

**全链路追踪**：
- SkyWalking 跟踪每笔决策
- 故障秒级定位

## 十、容量规划与压测

**容量公式**：
```
日常 QPS × 峰值系数（2-5）× 冗余系数（1.5）= 设计容量
```

**风控的容量**：
- 日常 5 万 QPS
- 双 11 峰值 100 万 QPS（× 20）
- 设计容量 150 万 QPS（× 1.5 冗余）
- 单机 5000 QPS → 300 实例

**全链路压测**：
- 影子流量（不打扰用户）
- 验证容量上限
- 找出瓶颈服务
- 验证降级生效

## 十一、底层本质：可用性的数学与工程

**可用性的数学**：
```
单实例可用性: 99%
两个独立副本: 1 - (1-0.99)^2 = 99.99%
三个独立副本: 1 - (1-0.99)^3 = 99.9999%
```

**关键假设**：副本必须独立故障（不同机房、不同机架、不同硬件批次）。

**工程实现**：
- 独立性：故障域隔离（机房、机架、电源）
- 检测：监控告警
- 切换：自动故障转移
- 验证：混沌工程

**这是"用工程手段逼近数学极限"**：
- 理论：3 个 99% 副本 = 99.9999%
- 工程：监控 + 自动切换 + 验证
- 实际：99.99%（工程损耗）

**蚂蚁的实践**：
- 三地五中心（机房级冗余）
- 单元化（爆炸半径到单元）
- 自动化运维（故障秒级切换）
- 混沌工程（主动验证）

## 十二、AI 时代的高可用

**LLM 服务的高可用新挑战**：
- GPU 资源稀缺（冗余成本高）
- 推理慢（故障检测延迟）
- 模型版本切换有副作用

**对应方案**：
- 冗余：多 GPU 集群 + 多模型版本
- 降级：LLM 挂了降级到规则
- 验证：LLM 输出验证（防幻觉）

## 常见考点
1. **99.99% 怎么计算**？——年停 = 365×24×60×0.01% ≈ 52 分钟。
2. **同城和异地多活选哪个**？——强一致场景（金融）选同城双活；容忍最终一致选异地多活。
3. **单元化最难的是什么**？——跨单元调用（理论上不该有，实际业务复杂）+ 全局数据同步。

**代码示例**（健康检查）：
```java
@RestController
public class HealthController {

    @GetMapping("/health")
    public ResponseEntity health() {
        // 检查关键依赖
        if (!redisTemplate.getConnectionFactory().getConnection().isOpen()) {
            return ResponseEntity.status(503).body("redis down");
        }
        return ResponseEntity.ok("ok");
    }

    @GetMapping("/ready")
    public ResponseEntity ready() {
        // 检查是否就绪（预热完成、配置加载完）
        if (!appReady) {
            return ResponseEntity.status(503).body("not ready");
        }
        return ResponseEntity.ok("ready");
    }
}
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：99.99% 可用性意味着年停机 52 分钟，但风控漏判一笔欺诈可能损失几十万。为什么不用 99.999%（年停 5 分钟）？**

这是成本收益权衡。从 99.99% 到 99.999% 看似只提升 0.009%，但工程成本是指数级的——99.99% 靠"同城双活 + 自动切换"能实现，99.999% 需要"异地多活强一致 + 跨城 RT < 10ms"，后者要么用 Paxos/Raft 跨城共识（延迟高，风控扛不住），要么接受数据丢失（金融场景不可接受）。而风控的真实需求是"故障期间宁可降级也不要漏判"——降级返回保守策略（默认拦截高风险操作）+ 异步复核，即使停机 5 分钟，损失也是可控的资损而非系统性风险。所以 99.99% + 强降级，比追求 99.999% 的强一致更经济也更现实。

### 第二层：证据与定位

**Q：可用性指标怎么度量？你怎么知道系统现在是 99.99% 还是 99.9%？**

靠 SLI（Service Level Indicator）精确度量：
1. 定义 SLI——风控的 SLI 是"决策请求成功率"（非 5xx 且 RT < 200ms 的请求占比），用 Prometheus 统计 `sum(rate(risk_decision_success_total[5m])) / sum(rate(risk_decision_total[5m]))`。
2. 定义 SLO——每月 SLI ≥ 99.99%，用 Grafana 大盘按月统计，配 error budget（错误预算 = 0.01% × 月总分钟数）。
3. 度量维度——按接口、按机房、按业务线分别统计，定位短板。如果整体 99.95% 但某个机房 99.8%，说明那个机房是瓶颈。

### 第三层：根因深挖

**Q：你们系统去年的可用性是 99.95%，没达到 99.99% 的目标。你怎么分析差在哪？**

不是拍脑袋，要拆解故障时间：
1. 按故障类型拆——统计全年故障的总停机分钟数，按"应用变更"、"下游依赖"、"基础设施"、"网络"分类。如果应用变更占 60%，说明发布流程是主要短板，重点改进灰度发布和自动化回滚。
2. 按 MTTR 阶段拆——MTTD（发现）+ MTTI（定位）+ MTTR（恢复）。如果 MTTD 占 40%（告警慢），加监控；如果 MTTI 占 40%（定位慢），加 runbook 和诊断工具；如果 MTTR 占 40%（恢复慢），加自动切换和降级。
3. 按单次故障拆——找 TOP 3 最长故障，逐个复盘根因，这 3 次可能就占了全年 80% 的停机时间，修复它们收益最大。

**Q：假设你发现 60% 的停机是"应用变更"导致的。为什么不直接禁止变更，冻结系统不就稳定了吗？**

冻结变更会扼杀业务迭代。风控需要快速上线新规则对抗新型欺诈（黑产每周都在进化），一个月不上新规则，欺诈率就会上升。所以解法不是"禁止变更"而是"让变更更安全"——灰度发布（1% → 10% → 100%，每阶段观察错误率）、自动化回滚（错误率超阈值自动触发，不等人工）、影子流量验证（变更前先在生产环境跑影子流量）。蚂蚁的实践是：变更频率从每周 1 次提到每天 10 次，但可用性反而提升了，因为"小步快跑 + 自动防护"比"大步慢跑 + 人工把关"更安全。

### 第四层：方案权衡

**Q：同城双活和异地多活，风控选哪个？为什么不直接上异地多活做到极致容灾？**

风控选同城双活 + 异地灾备，不选异地多活。原因：异地多活的强一致代价太高。风控的特征查询依赖 Redis 和 HBase，跨城同步延迟 30-50ms（光速限制），如果用强一致同步（2PC/Paxos），每次写操作要跨城往返，RT 从 5ms 涨到 100ms，风控决策 P99 直接爆表。而同城双活（同城两个机房，距离 < 50km，RT < 2ms）可以用同步复制保证强一致，同时单机房故障时秒级切流到另一个机房，覆盖 95% 的故障场景。异地灾备只用于"整个城市毁灭"的极端场景（地震、大面积停电），接受降级运行（最终一致 + 保守策略），不强一致。这是"覆盖常见故障 + 极端故障降级"的分层策略。

**Q：那单元化架构（按 UID 切分单元）为什么不全用，还要保留同城双活？**

单元化和双活不冲突，是叠加的。单元化解决的是"爆炸半径"——把全量用户切成 N 个单元，每单元独立完整链路（网关+应用+Redis+HBase），一个单元故障只影响 1/N 用户。同城双活解决的是"机房级容灾"——每个单元都跨两个机房部署，单机房故障时该单元切到对端机房。所以正确架构是：单元化（按 UID hash 切 100 个单元）× 同城双活（每单元两机房）= 200 个独立故障域。不单元化只双活的话，一个机房挂影响 50% 用户；单元化后一个机房挂只影响该机房承载的单元对应的用户（可通过单元迁移进一步降低）。

### 第五层：验证与沉淀

**Q：你怎么验证这套高可用架构真的能扛住机房级故障，而不是纸上谈兵？**

靠真实故障注入验证，不能只靠"理论设计"：
1. 机房切换演练——每季度一次，在低峰期（凌晨）主动切断一个机房的所有流量（用 DNS 或网关配置），验证流量是否在预期时间（< 30 秒）切到另一个机房，业务是否无感知（错误率不飙升）。
2. 数据一致性验证——切换后用对账脚本比对两个机房的数据（Redis key 数量、HBase 行数），确认无数据丢失或不一致。
3. 全链路混沌演练——用 ChaosBlade 注入组合故障（机房 A 的 Redis 挂 + 机房 B 的应用 OOM），验证降级链路、熔断、兜底策略是否按预期工作，有无级联故障。

**Q：高可用架构的设计原则怎么沉淀成团队规范，避免每个新项目重新踩坑？**

三件事：
1. 架构 Review 检查项——新服务上线前必须过架构 review，检查清单包括：是否多实例部署、是否配置健康检查、是否有降级策略、是否接入监控告警、是否做过容量规划。任何一项不达标不准上线。
2. 高可用基线模板——提供标准化的脚手架（Spring Boot starter），内置健康检查、限流降级、链路追踪、监控上报，新服务直接用，不重复造轮子。
3. 故障案例库——每次真实故障复盘后写入案例库（根因、止损过程、改进项），新成员入职必读，避免"前人踩过的坑后人再踩"。

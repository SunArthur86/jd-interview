# 【蚂蚁风控】设计一个实时风控决策引擎，融合规则、模型、名单

> JD 依据："数字化风控服务及智能风控平台"。这是风控平台的核心。

## 一、需求拆解

**业务需求**：
- 输入：交易/登录/操作事件
- 输出：决策（PASS/REVIEW/REJECT）+ 解释
- 时延：< 100ms
- 准确率：高召回（少漏报）+ 高精准（少误杀）
- 可解释：每条决策可追溯

**核心矛盾**：
- 规则：可解释但僵化（难防新型欺诈）
- 模型：灵活但黑盒（解释难、监管担忧）
- 名单：粗暴但快（黑名单误杀、白名单漏判）

## 三、决策编排（DAG）

决策过程是**有向无环图**（DAG）：
```
节点：名单检查 → 特征查询 → 规则评估 → 模型推理 → 融合
边：依赖关系
短路：白名单/黑名单可短路返回
```

**用 DAG 而非线性**：因为不同节点可并行（规则和模型可并行）、可短路（名单前置）。

**实现**（用 Akka 或自研 DAG 引擎）：
```java
DecisionDAG dag = DecisionDAG.builder()
    .start("whitelist_check")
    .thenParallel("feature_query", "rule_eval", "model_inference")
    .then("fusion")
    .onShortCircuit("whitelist_check", decision -> decision.isFinal())
    .build();

RiskResult result = dag.execute(event);
```

## 四、规则决策路

**专家规则**（运营配置）：
- DSL 表达（QLExpress / Drools）
- 热加载（不重启）
- 灰度发布（10% 流量验证）
- A/B 实验

**输出**：
```java
class RuleResult {
    String ruleId;
    String decision;   // PASS / REVIEW / REJECT
    int score;         // 命中规则的风险分
    String reason;     // 命中原因（可解释）
}
```

## 五、模型决策路

**模型类型**：
| 模型 | 用途 | 推理 RT |
|------|------|---------|
| LR/XGBoost | 浅模型，结构化特征 | 5-10ms |
| DNN | 深度模型，海量特征 | 10-50ms |
| LLM | 语义特征、复杂场景 | 100ms+（异步） |

**预计算分**（提速）：
- 离线算用户基础分（信用、风险）
- 缓存 Redis，决策时 O(1) 查
- 在线只算"动态分"（基于当前事件）

**模型版本管理**：
```yaml
model:
  name: risk_model_v2
  version: 2.3.1
  traffic: 10%   # 灰度 10%
  fallback: risk_model_v1  # 挂了 fallback
  shadow: true   # 影子流量（不打扰线上，只评估）
```

## 六、融合策略（核心）

**策略 1：优先级（硬约束优先）**
```
黑名单（直接拦） > 强规则（拦） > 模型分（高/中/低） > 弱规则（提示）
```

**策略 2：加权融合**
```java
int totalScore = ruleScore * 0.4 + modelScore * 0.5 + historyScore * 0.1;
if (totalScore >= REJECT_THRESHOLD) reject();
```

**策略 3：风险分层（决策矩阵）**
```
                模型分
                低    中    高
规则    低      放行  复核  复核
强度    中      放行  复核  拦截
        高      复核  拦截  拦截
```

**策略 4：级联（节省算力）**
```
名单过滤 → 95% 流量（白名单直放）
→ 规则匹配 → 80% 流量（命中放行）
→ 模型推理 → 20% 流量（边缘案例才用模型）
```

## 十、底层本质：决策融合的"专家+数据"哲学

决策融合本质是**专家知识与数据驱动的结合**：

**专家规则的优势**：
- 可解释（命中哪条规则）
- 强约束（明确知道某种行为是风险）
- 快（毫秒匹配）

**模型的优势**：
- 学未知模式
- 综合多维特征
- 持续优化

**互补关系**：
- 规则做"硬约束"（绝对要拦的）
- 模型做"软判断"（模糊地带）
- 名单做"快速短路"（明显黑白）

**这是机器学习的"人在回路"（Human-in-the-loop）思想**：
- 专家注入先验（规则）
- 数据学习后验（模型）
- 持续反馈（决策回流→训练）

## 十一、和 LLM/AI Agent 的演进

**未来形态**：

- 模型推理从单模型 → Agent 编排多模型

## 常见考点
1. **规则和模型谁优先**？——硬约束（黑名单/强规则）优先，模型做软判断补充；最终融合按业务策略。
2. **怎么评估决策引擎效果**？——三率（召回/精准/RT）+ 业务指标（拦截率/漏报损失/误杀申诉率）。
3. **模型推理挂了怎么办**？——降级到规则；fallback 到旧模型；最终兜底（黑名单+阈值）。

**代码示例**（决策融合）：
```java
public RiskResult fuse(Event e, List<RuleResult> rules, ModelResult model) {
    // 1. 强规则硬约束
    for (RuleResult r : rules) {
        if (r.decision == REJECT && r.priority == HARD) {
            return RiskResult.reject(r.ruleId);  // 强规则命中即拦
        }
    }

    // 2. 加权融合
    int ruleScore = rules.stream().filter(r -> r.decision != PASS)
                              .mapToInt(RuleResult::getScore).sum();
    double totalScore = ruleScore * 0.4 + model.score * 100 * 0.6;

    // 3. 风险分层
    if (totalScore >= REJECT_THRESHOLD) return RiskResult.reject(allHits);
    if (totalScore >= REVIEW_THRESHOLD) return RiskResult.review(allHits);
    return RiskResult.pass();
}
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼问本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：决策引擎你设计了"名单 + 规则 + 模型"三路融合，而不是只用模型（模型最强）。为什么要保留规则和名单？决策依据是什么？**

三个原因：可解释性、冷启动、硬约束。可解释性上，监管要求"拦截决策必须给出原因"，模型（尤其深度学习）是黑盒，解释困难；规则天然可解释（"命中 R001：短时间异地登录"），监管接受。冷启动上，新型欺诈出现时（如新攻击手法），模型要收集样本 + 训练（周级），但规则可以几小时内上线（专家经验配置），第一时间止损。硬约束上，黑名单（已知欺诈者）、监管名单（制裁列表）必须硬拦截，模型给"风险分"不够（即使分低也不能放行已知欺诈者）。决策依据是治理需求——金融风控的可解释性 + 时效性要求让规则和名单不可省略，模型作为"补充检测"（发现规则没覆盖的模式）。三者分工：名单=绝对拦截、规则=专家经验快速迭代、模型=数据驱动深度检测。

### 第二层：证据与定位

**Q：风控决策的拦截率突然飙升（从 5% 涨到 30%），业务大量投诉误杀。你怎么定位是规则问题还是模型问题？**

分路归因：
1. 看每路的命中率变化——决策引擎记录"哪个规则命中、模型分多少"。统计规则命中率（`rule.hit.rate`）和模型分分布（`model.score.distribution`）。如果某条规则（如 R099）的命中率从 0.1% 涨到 25%，是规则问题（这条规则可能配错了或数据源异常）。如果模型分分布整体右移（P50 从 30 涨到 60），是模型问题（模型可能漂移或特征异常）。
2. 对比变更——拦截率飙升的时间点附近有没有规则变更、模型升级、特征源变更。如果昨天发了新规则 R099，今天命中率飙，优先怀疑 R099。
3. 抽样 case 复盘——从拦截的请求中随机抽 50 个，人工 review 是否真该拦。如果 50 个里 45 个是正常用户被误杀，是误杀（规则/模型太敏感）；如果 45 个是真欺诈，是真实风险上升（不是引擎问题，是攻击增加）。

### 第三层：根因深挖

**Q：你发现是模型分分布右移（P50 从 30 涨到 60），但模型代码没改。根因是什么？为什么模型分突然漂移？**

模型分漂移通常是特征漂移导致。看模型的输入特征分布：
1. 单特征 PSI（群体稳定性指标）——对比模型当前的输入特征分布 vs 训练时的分布，PSI >0.25 的特征是漂移的。常见是某个特征的数据源变了（如特征平台升级导致 trade_cnt_1h 的计算口径微调），模型拿到偏移的特征，输出偏移的分数。用 `feature.psi` 指标按特征维度统计。
2. 特征缺失率——如果某特征（如 device_score）突然大量缺失（数据源故障），模型用默认值填充，输出分布变化。看 `feature.missing.rate`。
3. 特征异常值——如果某特征出现异常值（如 amount 从平均 100 涨到 10000，可能是大促），模型对异常值敏感，分数飙升。看特征的 P99 是否异常。
真实案例常见是第 1 种——特征平台的某个特征（如 user_active_level）的离线在线不一致突然放大（某次 Flink Job 改了逻辑但没同步离线），在线特征值偏离训练分布，模型分漂移。

**Q：根因是特征漂移。那为什么不直接重新训练模型（用新数据）？一次性解决。**

重新训练不是立刻能做的，且有风险。一是标签延迟——风控模型的标签（是否欺诈）要等交易确认后回流（T+7 甚至 T+30，因为有些欺诈几天后才被发现），无法立即用"今天的欺诈"重训练。二是漂移可能是临时的——如果是大促期间的特殊模式（金额普遍高），大促后恢复正常，重训练反而把大促模式学进去（正常时期误杀）。正确做法是先止血（临时调整模型分阈值，如从 60 调到 80，降低敏感度），再根因分析（哪个特征漂移、是数据源 bug 还是真实分布变化），最后决定是否重训练。如果是数据源 bug（修复后特征恢复），不用重训练；如果是真实分布变化（用户行为变了），用最新数据重训练。重训练是"最后手段"，不是"第一手段"。

### 第四层：方案权衡

**Q：你临时调高了模型分阈值止损。但业务说"调高阈值会漏判真实欺诈"。怎么权衡误杀和漏判？**

这是风控的核心权衡。误杀（正常用户被拦）和漏判（欺诈通过）是跷跷板——阈值高则漏判多、阈值低则误杀多。量化权衡：
1. 算经济账——一笔误杀的损失（用户体验差 + 客诉成本 + 流失）vs 一笔漏判的损失（欺诈金额）。如果平均欺诈金额 5000 元、误杀客诉成本 50 元，漏判损失是误杀的 100 倍，应偏向"宁可误杀不漏判"，阈值偏低。
2. 分场景阈值——不同场景的权衡不同。大额交易（>1 万）的漏判损失大，阈值偏低（宁可拦了人工审）；小额交易（<100）的误杀代价大（影响体验），阈值偏高（放行少量风险）。
3. 人工审核缓冲——对"中间分数"（如 50-80）的决策走 REVIEW（人工审核）而非直接 REJECT，让人工判断。这样既不误杀（人工可以放行）也不漏判（人工可以拦截），代价是人工审核成本（审核员人力）。
风控的实践是"分层决策 + 人工缓冲"——高分直接拦、低分直接放、中间分人工审，配合分场景阈值，平衡误杀和漏判。

**Q：为什么不直接用更复杂的模型（如深度学习 + 强化学习）自动平衡误杀和漏判，不用人工调阈值？**

复杂模型确实能自动优化，但有局限。一是可解释性——深度学习模型的决策难解释，监管不允许"黑盒拦截"。二是稳定性——复杂模型对数据更敏感（漂移更频繁），维护成本高。三是不确定性——强化学习的探索可能"故意放行"一些可疑交易来探索（学习），金融场景不能拿真实交易做实验。实务做法是"模型给分 + 人工定阈值"——模型负责"评估风险"（打分），阈值由风控策略团队根据业务（误杀/漏判成本）定，可解释（"分数 >60 拦截"）、可调整（策略改阈值不动模型）、可审计（阈值变更记录）。复杂模型作为"打分器"，阈值决策权留给人，是金融风控的最佳实践。

### 第五层：验证与沉淀

**Q：你怎么验证决策引擎的融合策略合理——三路权重（名单/规则/模型）配的对不对？怎么量化"决策质量"？**

三个量化指标：
1. 召回率（漏判率）——已知欺诈交易中，决策引擎拦截了多少。目标 >95%（漏判 <5%）。从 T+7 的欺诈标签回流统计。
2. 精准率（误杀率）——拦截的交易中，多少是真欺诈。目标 >60%（误杀 <40%， REVIEW 不算误杀）。从人工审核结果统计。
3. 决策 RT——P99 <100ms。从 APM 采集。
权重合理性验证——做 AB 实验，对照组用现有权重（如规则 0.4 + 模型 0.6），实验组调权重（如规则 0.3 + 模型 0.7），对比两组的召回/精准。如果实验组召回提升且精准不降，新权重更优。

**Q：怎么让团队的决策引擎持续有效、不退化？**

沉淀成机制：
1. 决策审计——每条决策记录三路结果（名单命中、规则命中、模型分）+ 最终决策，供事后审计和申诉追溯。保留 90 天。
2. 模型监控——模型分分布（PSI）、特征 PSI、特征缺失率，每天监控，漂移告警。
3. 规则灰度——新规则上线走影子流量（不影响真实决策）→ 灰度 10% → 全量，每阶段评估召回/精准，退化回滚。
4. 策略 review——每周 review 拦截率、误杀率、客诉率，异常时调阈值或规则。月度 review 规则有效性（长期不命中的规则下线）。
5. 故障复盘——把这次"特征漂移导致模型分飙升 → 误杀"的特征 PSI 曲线、阈值调整方案、根因（数据源 bug）存知识库。


## 结构化回答

**30 秒电梯演讲：** 聊到设计一个实时风控决策引擎，融合规则、模型、名单，我的理解是——实时决策引擎融合"规则（专家知识）+ 模型（数据驱动）+ 名单（黑白名单）"三路决策，按优先级、加权、阈值融合输出最终决策，是风控的大脑。打个比方，决策引擎像法庭审判——名单是通缉令（直接抓）、规则是法条（命中即判）、模型是陪审团（综合评估），三者按法定流程给出最终裁决。

**展开框架：**
1. **三路决策** — 名单（前置短路）+ 规则（专家）+ 模型（数据）
2. **融合策略** — 优先级、加权、风险分层、级联
3. **模型版本管理** — A/B、影子流量、灰度切流

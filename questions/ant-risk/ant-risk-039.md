# 【蚂蚁风控】FDE 在风控 AI 项目中做什么？怎么落地一个 AI 风控解决方案？

> JD 依据："FDE"是 AI 时代的复合角色。这是 AI 转型的高级方向。

## 一、FDE 是什么角色

**FDE（Field Development Engineer / Full-Stack Dev for AI）**：懂技术 + 懂业务 + 懂客户的复合角色。

**核心职责**：
1. **理解客户需求**：客户说"要降欺诈率"，FDE 翻译成"召回率提升 X%"
2. **设计技术方案**：基于需求选 AI 能力、设计架构
3. **推动落地**：协调技术团队、客户、监管
4. **负责效果**：交付后跟踪 ROI、迭代优化

**对比其他角色**：

| 角色 | 侧重 | FDE 区别 |
|------|------|---------|
| 产品经理 | 用户需求 | FDE 更懂技术，能设计方案 |
| 解决方案架构师 | 技术方案 | FDE 更贴近客户、负责落地 |
| 技术销售 | 卖产品 | FDE 负责交付 |
| 算法工程师 | 模型 | FDE 负责全链路 |

## 二、为什么 AI 项目需要 FDE

**AI 项目的失败率**（行业数据）：
- 70-85% 的 AI 项目未能交付预期价值
- 主因：需求理解错、技术方案不匹配、监管不合规

**典型失败案例**：
```
客户：我们要用 AI 风控
算法团队：训了个 AUC 0.9 的模型
落地：模型黑盒，监管不接受 → 项目黄
```

**FDE 的价值**：在需求阶段就识别"监管要可解释"，设计方案时把规则 + 模型 + Agent 组合，保证可落地。

## 三、风控 FDE 的核心场景

**场景 1：AI 风控合规**
- 客户：银行，要求 AI 决策可解释
- FDE：设计"规则做硬约束 + LLM 做解释生成"方案
- 落地：决策带自然语言解释，监管审核通过

**场景 2：新型欺诈防御**
- 客户：支付公司，发现新型欺诈
- FDE：设计"Agent + 关系网络"方案，Agent 自主推理新型模式
- 落地：Agent 决策 + 人工复核闭环

**场景 3：智能审核**
- 客户：电商，案件审核人力贵
- FDE：设计"LLM 辅助审核"方案，自动总结、分类、推荐
- 落地：审核效率 +50%

**场景 4：跨境合规**
- 客户：海外业务，多国法规不同
- FDE：设计"分区域策略"方案，每区域独立规则 + 共享平台
- 落地：满足各国合规要求

## 四、FDE 的方案设计方法论

**1. 需求拆解（理解客户）**：
```
客户：我们要降低欺诈损失
   ↓ FDE 追问
具体：哪类欺诈？（账户盗用 / 套现 / 中介）
现状：当前欺诈率多少？损失多少？
目标：降到多少？投入预算多少？
约束：监管要求？时效要求？数据可用？
```

**2. 方案设计（技术选型）**：
```
基于需求，设计技术方案:

业务目标: 欺诈率从 0.5% 降到 0.2%
技术方案:
  ├─ 数据层: 整合事件流 + 用户画像 + 关系网络
  ├─ 决策层: 规则（硬约束）+ 模型（软判断）+ Agent（边缘）
  ├─ 监管层: 决策可解释 + 审计日志
  └─ 运营层: 控制台 + 灰度发布 + A/B

技术选型:
  - 规则引擎: Drools（成熟）/ QLExpress（轻量）
  - 模型: XGBoost（浅）+ LLM（深）
  - Agent: 自研编排 + LLM
  - 图: JanusGraph（大规模）
  - 推理: vLLM（高性能）
```

**3. 落地路径（分阶段）**：
```
Phase 1（1 个月）: 基础架构 + 规则接入
Phase 2（2 个月）: 模型部署 + 决策融合
Phase 3（2 个月）: Agent 接入 + 异步复核
Phase 4（持续）: 自学习闭环 + 优化
```

## 五、AI 风控合规（FDE 的硬骨头）

**监管对 AI 决策的要求**：

| 要求 | 实现方式 |
|------|---------|
| **可解释** | 决策带自然语言解释 + 命中规则/证据 |
| **可追溯** | 完整决策日志（输入/中间/输出） |
| **可审计** | 模型版本、训练数据可追溯 |
| **公平性** | 不歧视特定群体（性别、地域） |
| **安全** | 防注入、防越权、防数据泄露 |

**FDE 的合规设计**：

```
决策输出:
{
  "decision": "REJECT",
  "explanation": "新账户深夜大额转账给高风险账户",
  "rules_hit": [...],
  "model_score": 0.85,
  "evidence": [...],
  "model_version": "v2.3.1",
  "trace": "完整推理链路"
}

可追溯:
  - 决策日志保留 7 年（监管要求）
  - 模型版本可回溯（任何决策可重放）
  - 数据版本可追溯（DVC）
```

## 六、FDE 的项目管理

**关键节点**：
1. **需求确认**（Week 1-2）：和客户对齐目标、约束
2. **方案设计**（Week 3-4）：技术方案 + 评审
3. **POC 验证**（Week 5-8）：小规模验证
4. **落地交付**（Week 9-16）：分阶段上线
5. **效果跟踪**（持续）：ROI 评估、迭代

**沟通管理**：
- 客户：定期同步进度、风险
- 技术：明确分工、技术评审
- 监管：提前沟通、合规审查
- 内部：跨团队协调

**风险管理**：
```
风险: 监管不批准
缓解: 早期介入、可解释设计、灰度验证

风险: 模型效果不达标
缓解: POC 阶段验证、备选方案（规则）

风险: 数据质量差
缓解: 数据评估、清洗、采集补全
```

**4. 项目管理**（20%）：

## 十、底层本质：AI 项目的"翻译官"

FDE 的本质是**"业务-技术-客户"的翻译官**：

**业务语言**：降欺诈率、提效率、合规
**技术语言**：模型、规则、Agent、平台
**客户语言**：投入产出、风险、价值

FDE 在三者之间翻译，让"技术能做的"变成"客户能用且有价值的"。

**这是 AI 时代的"系统化思维"**：
- 不只看技术（工程师视角）
- 不只看业务（PM 视角）
- 不只看客户（销售视角）
- 全链路负责（FDE 视角）

**类比传统软件**：
- 传统：项目经理 + 架构师 + 产品经理
- AI 时代：FDE（三合一）

## 十一、FDE 与 Agent 的协同（未来）

**未来形态**：FDE 用 Agent 辅助做方案设计。

```
FDE: "客户要降低欺诈率，预算 500 万，监管严"
   ↓ Agent 辅助
Agent:
  - 生成方案初稿（架构图 + 选型）
  - 评估技术可行性
  - 预估工期和成本
  - 识别合规风险
FDE: review + 调整 + 客户沟通
```

**Agent 不替代 FDE**：
- Agent 擅长生成、分析
- FDE 擅长判断、沟通、决策
- 二者协同

## 十二、案例：完整的 AI 风控项目交付

**客户**：某城商行，要求 AI 风控升级

**需求**：
- 现状：纯规则风控，欺诈率 0.5%
- 目标：降到 0.2%，时效 < 200ms，可解释

**FDE 方案**：
```
架构:
  ├─ 规则引擎（保留，硬约束）
  ├─ XGBoost 模型（决策融合）
  ├─ LLM 解释生成（监管要求）
  └─ Agent（边缘场景异步复核）

合规:
  - 决策带自然语言解释
  - 完整审计日志
  - 模型版本管理

落地:
  Phase 1（2 月）: 模型部署 + 决策融合
  Phase 2（1 月）: LLM 解释生成
  Phase 3（持续）: Agent + 自学习
```

**交付效果**：
- 欺诈率 0.5% → 0.18%（达标）
- RT < 150ms（达标）
- 监管审核通过（可解释）
- 客户满意度 NPS 9/10

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：FDE 这个角色为什么在 AI 时代变得重要？传统软件时代为什么没有 FDE？**

因为 AI 项目的"不确定性"远高于传统软件。传统软件的需求是明确的（"做一个支付接口"，输入输出清晰），技术团队按需求交付就行。AI 项目的需求是模糊的（"降低欺诈率"），且技术方案有不确定性（LLM 能不能达到 90% 准确率？需要实验验证）。客户不懂 AI 的能力边界（以为 LLM 什么都能做），技术不懂客户的业务上下文（不知道风控的监管约束）。这种"双向信息鸿沟"导致 70% 的 AI 项目失败——不是技术做不到，是需求理解错了。FDE 的价值是"翻译和桥接"——把客户语言翻译成技术需求，把技术约束讲给客户，在"需求 - 方案 - 落地"全链路对齐。传统软件不需要这个角色因为鸿沟没这么大，AI 时代的技术复杂度和业务不确定性放大了鸿沟，FDE 应运而生。

### 第二层：证据与定位

**Q：一个银行客户的风控 AI 项目交付后，客户反馈"效果不好"。你怎么定位是方案设计问题、实施问题还是客户期望问题？**

分层定位：
1. 看"效果不好"的具体定义——是欺诈率没降（召回不够）、误拦太多（精准不够）、还是延迟太高（体验问题）？不同问题根因不同，不能笼统说"效果不好"。让客户量化：原来欺诈率 2%，现在多少？期望多少？
2. 对比方案设计和实际落地——方案设计时承诺的指标（召回率 95%、P99 < 100ms）和实际指标的差距。如果设计承诺 95% 但实际只有 80%，是实施问题（数据质量差、模型没调好）；如果实际达到了 95% 但客户还说不好，是期望管理问题（客户期望 99%）。
3. 看客户的使用情况——客户是否按方案接入？有没有绕过 AI 决策（人工干预太多）？有没有正确配置规则？很多"效果不好"是客户没用对，不是方案不行。

### 第三层：根因深挖

**Q：你定位到是"期望管理问题"——方案承诺召回率 95%，客户期望 99%。为什么前期没对齐期望？根因在哪？**

根因是 FDE 在需求阶段没有做"可行性边界分析"。AI 的能力是有边界的——95% 召回率是当前技术 + 数据条件下的合理水平，99% 需要更多数据（标注样本）或更强模型（成本更高）。前期没对齐的原因：第一，FDE 怕丢单子，不敢说"做不到 99%"，含糊承诺"尽量高"。第二，客户不懂 AI 边界，以为"加钱就能更高"。第三，没有用数据说话——如果前期做一个小规模 POC（proof of concept），用客户真实数据测出 95% 的基线，客户就知道"原来 95% 是当前上限"，不会期望 99%。根因是"销售导向压过了工程导向"，解法是把 POC 作为售前的必经环节，用数据锚定期望。

**Q：那为什么不直接承诺保守一点（说只能到 90%），然后交付时给个惊喜（做到 95%）？**

这是"低承诺高交付"的策略，但在 to B 项目里有风险。第一，竞品可能承诺 95%，你承诺 90% 直接输掉标——销售现实就是"高承诺拿单，低承诺出局"。第二，客户会拿你的承诺去和内部汇报（"供应商说能做到 90%"），后续交付 95% 虽然是惊喜，但客户内部可能已经基于 90% 做了业务规划（比如砍掉一部分人工审核团队），惊喜变成"规划混乱"。所以正确做法不是"低承诺"，而是"数据驱动的承诺"——POC 测出基线，基于基线给区间承诺（"90%-95%，取上限需额外数据标注投入"），把不确定性讲清楚。客户更尊重"诚实 + 有依据"的承诺，而非"拍胸脯"。

### 第四层：方案权衡

**Q：风控 AI 项目里，客户要"可解释的决策"（监管要求），但 LLM 是黑盒。你怎么权衡"AI 智能"和"可解释性"？**

组合方案，不是二选一。具体策略：
1. 决策层用规则 + 结构化模型（可解释）——黑名单命中、规则命中、模型分，这些都是可解释的（"因为命中规则 R123：金额 > 1 万且夜间交易"）。这部分覆盖 80% 的决策，满足监管的"同案同判 + 可追溯"。
2. LLM 做"解释增强"而非"决策"——LLM 不直接决定拦/放，而是把规则和模型的结构化决策翻译成自然语言解释（"这笔交易被拦截，因为：1）金额 1.2 万超过阈值；2）发生在凌晨 3 点；3）收款方是新开户账户"）。LLM 的输出是解释不是决策，即使 LLM 有幻觉，影响的只是"解释文案"而非"决策结果"。
3. 证据链留痕——每个决策附带证据（命中的规则 ID、模型分、图谱路径），监管审计时可追溯。这样既用了 AI 的语义能力（生成解释），又保证了决策的可解释和可审计。

**Q：既然 LLM 只做解释不做决策，那为什么要用 LLM？传统模板生成解释不就行了？**

传统模板（"因为命中规则 {rule_id}"）的解释太生硬，客户和监管看不懂。风控规则 ID（"R123"）对业务人员没意义，他们需要的是"人话解释"——"这笔交易疑似夜间大额异常转账到新账户"。LLM 的价值是把结构化信号翻译成业务语言，让非技术人员（客服、合规官、用户本人）能理解。模板做不到这种"语义化翻译"——模板只能填变量，不能"重新组织语言让外行听懂"。而且不同受众需要不同解释深度：用户要简短（"风险交易已拦截"）、客服要详细（可申诉的点）、合规官要完整（所有命中规则 + 证据）。LLM 能根据受众生成不同粒度的解释，模板要为每个受众写一套。所以 LLM 做解释是有价值的，只是要限定在"解释层"不进"决策层"。

### 第五层：验证与沉淀

**Q：你怎么衡量 FDE 在 AI 项目中的价值，证明这个角色不可替代？**

三个维度：
1. 项目交付率——有 FDE 的项目按期交付率应 > 80%，没有 FDE 的项目（纯技术团队对接客户）通常 < 50%（需求变更频繁、期望不对齐）。对比同类项目的交付率差异。
2. 客户 NPS（净推荐值）——FDE 负责的项目客户满意度（NPS > 8/10），因为没有 FDE 时客户抱怨"技术团队听不懂业务需求"。
3. 业务 ROI——FDE 不仅交付系统，还负责"效果达标"。跟踪项目上线后客户的业务指标改善（欺诈率下降 X%、审核效率提升 Y%），换算成金额。如果 FDE 的方案帮客户年省 1000 万，FDE 的成本（年薪百万级）ROI 是 10 倍。

**Q：FDE 的经验和方案怎么沉淀成组织能力，而不是个人英雄主义？**

三件事：
1. 方案模板库——把每个行业的 AI 风控方案（银行、支付、电商）沉淀成模板（需求分析框架、技术架构、合规清单、实施计划），新 FDE 不用从零设计。
2. 案例 Playbook——每个交付完的项目写 Playbook（客户痛点、方案选型、踩过的坑、最终效果），新 FDE 面对类似客户时参考。
3. FDE 认证体系——建立 FDE 能力模型（业务理解、技术深度、项目管理、客户沟通）和培训路径，让 FDE 能力可复制，不依赖个别"明星 FDE"。


## 结构化回答

**30 秒电梯演讲：** 聊到FDE 在风控 AI 项目中做什么，我的理解是——FDE（Full-Stack Development Engineer for AI / 解决方案工程师）从客户需求到落地的全链路——理解业务、设计方案、推动落地、负责效果，是 AI 时代"懂技术 + 懂业务 + 懂客户"的复合角色。打个比方，FDE 像建筑师——客户说"要一栋楼"（需求），FDE 出图纸（方案）、找施工队（技术团队）、协调审批（合规）、最终交付钥匙（落地）。

**展开框架：**
1. **FDE 角色** — 需求理解、方案设计、技术选型、推动落地、效果负责
2. **风控 FDE 场景** — AI 风控合规、可解释、监管对接
3. **关键能力** — 业务理解 + 技术深度 + 沟通 + 项目管理

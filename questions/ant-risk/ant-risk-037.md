# 【蚂蚁风控】AI Infra 怎么设计？支撑大模型全生命周期

> JD 依据："智能化数据平台" + "大模型"。AI Infra 是 AI 时代的基础设施。

## 一、AI Infra 是什么

**AI Infra = 大模型的"操作系统 + 数据库 + 中间件"**

支撑大模型全生命周期：
```
数据采集 → 数据标注 → 模型训练 → 模型评估 → 模型部署 → 推理服务 → 监控评估 → 迭代
   ↑________________________________________________________________________________↓
                                          闭环
```

**对比传统软件**：
| 传统软件 | AI 应用 |
|---------|---------|
| IDE | Notebook / ML 平台 |
| 数据库 | 数据平台 + 特征平台 |
| 应用服务器 | 推理服务 |
| 监控 | 评估 + 漂移监控 |
| CI/CD | 训练流水线 |

## 二、AI Infra 的四层架构

### 1. 数据层

**数据采集**：
- 业务数据（风控决策日志）
- 公开数据（规则文档、法规）
- 标注数据（人工审核结果）

**数据标注**：
- 平台：Label Studio、Doccano
- 标注：人工 + 主动学习（模型辅助）
- 质量：多人标注 + 投票

**特征平台**（见 ant-risk-022）：
- 特征定义、计算、存储、服务

### 2. 训练层

**分布式训练**：

| 并行方式 | 原理 | 适用 |
|---------|------|------|
| **数据并行** | 每张卡算一部分数据，梯度同步 | 模型放得下单卡 |
| **张量并行** | 把模型矩阵切到多卡 | 模型放不下单卡 |
| **Pipeline 并行** | 把模型层分到多卡 | 超大模型（GPT-3 级） |

**训练框架**：
- PyTorch DDP（数据并行）
- DeepSpeed（ZeRO 优化）
- Megatron-LM（大模型）

**checkpoint 管理**：
```python
# 定期保存 checkpoint（容错）
for epoch in range(epochs):
    train_one_epoch()
    if epoch % 5 == 0:
        save_checkpoint(model, optimizer, epoch)

# 训练中断可恢复
load_checkpoint(latest)
```

**超参管理**：
- Weights & Biases / MLflow
- 实验追踪（参数、metrics、模型）

### 3. 推理层（见 ant-risk-034）

**模型部署**：
- 模型仓库（版本管理）
- 灰度发布（影子流量 → 灰度 → 全量）
- 蓝绿部署（一键切流）

**推理优化**：
- vLLM/PagedAttention
- 量化（INT8/INT4）
- 批处理

**GPU 调度**：
```
GPU 集群（K8s + GPU 调度）
  ├─ 训练任务（高优先级，A100）
  ├─ 推理服务（中优先级，T4/A10）
  └─ 批处理（低优先级，闲时跑）
```

### 4. 监控评估层

**在线监控**：
- 推理性能（延迟、QPS、错误率）
- 业务指标（决策准确率、拦截率）
- 数据漂移（特征分布变化）

**离线评估**：
- 测试集准确率
- A/B 对比
- 人工标注抽检

**模型迭代**：
```
决策回流 → 标注 → 增量训练 → 评估 → 上线
周期：周/月
```

## 四、风控 AI Infra 的特点

**1. 数据敏感**：
- 私有部署（数据不出境）
- 加密存储、访问审计
- 训练数据脱敏

**2. 实时性**：
- 推理延迟 < 500ms
- 决策链路集成
- 异步复核（秒级）

**3. 监管要求**：
- 决策可追溯（带证据）
- 模型可解释
- 审计日志保留

**4. 规模大**：
- 亿级用户、千亿特征
- 千万级事件/天
- 持续训练

## 五、训练流水线（风控场景）

```python
# 端到端训练流水线
@pipeline
def risk_model_pipeline():
    # 1. 数据准备
    data = load_decision_logs(start='-30d')
    labeled = label_with_audit(data)  # 用人工审核结果标注

    # 2. 特征工程
    features = feature_store.get_features(labeled['uid'])

    # 3. 训练
    model = train(
        data=labeled,
        features=features,
        hyperparams={'lr': 1e-4, 'epochs': 10}
    )

    # 4. 评估
    metrics = evaluate(model, test_set)
    if metrics.auc < 0.75:
        alert("模型质量下降")
        return

    # 5. 部署（灰度）
    deploy(model, traffic=0.1)  # 10% 灰度
    monitor(model)
```

## 六、GPU 资源调度

**问题**：GPU 稀缺且贵，需要高效调度。

**策略**：
```
集群分级:
  - 训练池（A100，高优先级）
  - 推理池（T4/A10，常驻服务）
  - 弹性池（闲时共享）

调度:
  - 工作时间：训练 + 推理
  - 夜间：批处理 + 训练
  - GPU 复用（多模型共享）
```

**蚂蚁的实践**：
- 自研 GPU 调度系统
- 多租户隔离
- 弹性扩缩容

## 八、模型漂移监控

**漂移类型**：
- **数据漂移**：特征分布变化（PSI > 0.25 告警）
- **概念漂移**：标签和特征关系变化（欺诈模式变了）
- **预测漂移**：模型输出分布变化

**监控**：
```python
def check_drift(model):
    # 数据漂移
    psi = compute_psi(current_features, baseline_features)
    if psi > 0.25:
        alert(f"数据漂移: PSI={psi}")

    # 概念漂移
    recent_auc = evaluate(model, recent_labeled)
    if recent_auc < model.baseline_auc - 0.05:
        alert(f"模型效果下降: AUC={recent_auc}")

    # 触发重训
    if psi > 0.25 or recent_auc < threshold:
        trigger_retrain()
```

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：AI Infra 和传统大数据平台（Hadoop/Spark）的核心区别是什么？为什么要单独建 AI Infra 而不是复用大数据平台？**

核心区别是"算力类型和工作负载"。大数据平台处理的是 CPU 密集型批处理（ETL、报表），AI Infra 处理的是 GPU 密集型训练和推理。两者对调度、存储、网络的要求完全不同：训练要 GPU 池化 + 高带宽互联（NVLink/RoCE）+ 大文件存储（模型权重几十 GB），推理要低延迟服务 + 弹性扩缩容。大数据平台的 YARN/K8s 调度器不理解 GPU 拓扑（GPU 亲和性、显存），强行跑训练会导致跨节点通信慢 10 倍。所以 AI Infra 是专门为"GPU + 大模型"设计的基础设施，用 K8s + GPU operator + Ray + vLLM 这套栈，和大数据平台的 Hadoop + Spark 是两套独立系统。复用会导致两边都跑不好。

### 第二层：证据与定位

**Q：风控模型训练任务从 4 小时涨到 12 小时，你怎么定位是数据问题、代码问题还是 GPU 资源问题？**

分层定位：
1. 看 GPU 利用率——`nvidia-smi` 或 DCGM exporter，看训练期间 GPU 利用率。如果利用率 < 50%，说明 GPU 空转（可能是数据加载瓶颈，CPU/IO 跟不上喂给 GPU），不是 GPU 不够。如果利用率 > 95% 但训练仍慢，是计算量真大了（数据量增加或模型变大）。
2. 看数据管道——训练日志里每个 epoch 的耗时分解。如果 `data_loading` 占 60%，是数据预处理慢（可能没用 num_workers 或数据存在慢盘）。如果 `forward_backward` 占 80%，是计算瓶颈。
3. 看资源分配——K8s 的 GPU 分配记录，确认这次训练分到了几块卡、是不是被其他任务抢占（`kubectl describe pod` 看 GPU alloc）。如果从 4 卡降到 2 卡（资源不足），训练时间翻倍合理。

### 第三层：根因深挖

**Q：你定位到 GPU 利用率只有 40%，是数据加载瓶颈。为什么不直接加 num_workers（多进程加载）解决？**

加 num_workers 是对症但不是治本。根因可能是数据存储位置——如果训练数据在远端存储（HDFS/OSS），每次读取都走网络，num_workers 再多也会被网络带宽限制。真正的解法是"数据本地化"——训练前把数据预下载到 GPU 节点的本地 NVMe SSD，训练时从本地盘读（带宽 3GB/s vs 网络 10GB/s，差 30 倍）。或者用"内存缓存"——把高频访问的数据集缓存在内存（RAM），配合 `DataLoader(pin_memory=True)` 让数据直接传到 GPU 显存。加 num_workers 只在"数据已在本地、CPU 是瓶颈"时有效，存储瓶颈时无效。所以先定位瓶颈是 CPU 还是 IO，再选解法。

**Q：那为什么不直接把所有训练数据放内存？内存比 SSD 快多了。**

内存容量有限且贵。一个风控训练数据集可能是 100GB（几亿样本 × 特征），一个 GPU 节点的内存通常 256-512GB，放数据后留给模型和其他进程的空间就紧张了。而且内存是易失的，节点重启数据要重新加载。实际策略是分层缓存：热数据（当前 epoch 的 batch）放内存，温数据（本次训练要用的）放本地 SSD，冷数据（历史归档）放 OSS/HDFS。用缓存系统（如 Alluxio）自动管理分层，训练代码无感。盲目全放内存会导致 OOM 或成本爆炸（要买大内存机器），分层才是工程最优解。

### 第四层：方案权衡

**Q：训练用大模型（A100/H100）还是推理用大模型？GPU 资源有限时，训练和推理怎么分配？**

训练和推理用不同型号的 GPU，不争抢。训练用 A100/H100（大显存 80GB，高带宽，适合大规模矩阵运算），推理用 T4/A10/L20（性价比高，显存够用，适合服务化）。两者不混用——训练卡贵且少（A100 一卡难求），用来做推理是大材小用；推理卡显存小（T4 只有 16GB），跑不了大模型训练。分配策略是物理隔离：训练集群（A100 池）和推理集群（T4 池）独立，K8s 用 nodeSelector 或 taint 隔离。只有在资源极度紧张时，才用"白天推理、夜间训练"的分时复用（推理白天流量大、训练用夜间空闲 GPU），但这需要调度系统支持抢占式调度，且训练 checkpoint 要频繁保存（防被抢占）。

**Q：既然 GPU 资源紧张，为什么不直接用云端 API（GLM-4 API）做推理，省掉推理 GPU 的投入？**

取决于数据敏感度和成本曲线。如果风控数据可以出公网（脱敏后），用 API 确实省事——不用管 GPU 采购、部署、运维。但蚂蚁的风控数据涉及用户隐私（交易记录、行为数据）和商业机密（风控规则），不能出内网，必须私有部署。成本上，当 QPS > 50 并发时，自建推理的边际成本远低于 API（自建约 0.001 元/次，API 约 0.02 元/次）。按风控日均亿级请求算，自建每天省百万级。所以判断标准是：数据能出公网 + 低 QPS → 用 API；数据敏感 + 高 QPS → 自建。风控属于后者，必须自建推理 GPU 集群。

### 第五层：验证与沉淀

**Q：AI Infra 平台的 ROI 怎么算？怎么证明这个平台值得投入而不是每个 AI 项目自己搭一套？**

对比"有平台"和"没平台"的三个维度：
1. 开发效率——没平台时，每个 AI 项目自己搭训练脚本、数据管道、推理服务、监控，平均 2 人月。有平台后，这些变成标准化组件，新项目接入 1 人周。如果一个团队一年做 10 个 AI 项目，节省 10 × (8 周 - 1 周) = 70 人周。
2. 资源利用率——没平台时各项目独占 GPU（利用率 30%），有平台后统一调度（利用率 70%+），同样 GPU 总量能支撑 2 倍任务。GPU 是稀缺资源，这是硬省。
3. 质量保障——平台提供统一的评估、监控、版本管理，避免各项目"各自为政"导致的质量参差。线上故障率应下降 50%（统一的质量流程）。

**Q：AI Infra 的能力怎么沉淀成平台，让算法工程师和业务工程师都能用？**

三件事：
1. 标准化流水线——提供从数据到部署的模板化流水线（Kubeflow/Argo），算法工程师只写训练代码，数据准备、模型打包、部署、监控自动完成。
2. 抽象层——提供 SDK 抽象掉底层复杂性（GPU 调度、分布式训练、推理优化），算法工程师不用学 K8s/Ray/vLLM 的细节，像调函数一样用。
3. 自助门户——提供 Web 界面让业务方自助管理模型（上传数据、触发训练、看评估、灰度上线），不依赖平台团队人工介入，平台团队只维护基础设施，不参与具体业务。


## 结构化回答

**30 秒电梯演讲：** 聊到AI Infra 怎么设计？支撑大模型全生命周期，我的理解是——AI Infra 是支撑大模型全生命周期的底层设施——数据（采集/标注/管理）、训练（分布式/优化）、推理（部署/服务）、监控（评估/迭代），相当于 AI 时代的"数据库 + 中间件"。打个比方，AI Infra 像传统软件的"数据库+应用服务器+监控"——上层应用（Agent/LLM）跑在它上面，它管数据、训练、部署、监控的脏活累活。

**展开框架：**
1. **数据层** — 采集、标注、版本、特征
2. **训练层** — 分布式训练、优化器、checkpoint
3. **推理层** — 部署、服务、优化（vLLM）

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：训练和推理 GPU 怎么调度？您更想看哪个方向？

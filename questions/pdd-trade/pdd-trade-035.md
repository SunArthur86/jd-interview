# 【拼多多交易】AI 代码助手怎么落地？

> JD 依据："AI 代码助手"。

## 一、能力矩阵

| 能力 | 场景 | 成熟度 |
|------|------|--------|
| 代码补全 | 行内/函数补全 | 高（30% 提效） |
| 代码生成 | 自然语言→代码（CRUD/DTO） | 高 |
| 代码审查 | PR 自动审/找 bug | 中 |
| 单测生成 | 自动生成 JUnit | 中 |
| 文档生成 | 注释/API 文档 | 高 |
| 重构建议 | 提性能/简化 | 低（需人工） |

## 二、架构

```
IDE 插件（VS Code/IDEA）
    ↓（上下文：当前文件/光标/选中）
LLM 网关 → 路由（简单补全/复杂生成）
    ├─ 通用模型（代码补全）
    └─ 企业 RAG（查内部代码规范/框架用法）
    ↓
安全过滤（敏感词/密钥检测）→ 返回 IDE
```

**企业代码 RAG**：
```
代码库向量化 → 检索相似实现/规范
  例：写 OrderService → 检索团队规范（命名/异常处理/日志格式）
```

## 三、安全合规

```
1. 私有化部署：模型在内网，代码不出公司
2. 权限管控：按代码库权限过滤 RAG（不能跨部门查）
3. 敏感检测：自动拦截密钥/密码/个人信息
4. 审计日志：记录每次调用（谁/查了什么/生成什么）
5. 水印追溯：生成代码可追溯（防恶意注入）
```

## 四、落地路径

```
阶段 1：补全（样板代码）→ 提效 20%，无风险
阶段 2：生成（CRUD/DTO/单测）→ 提效 30%，需人工审
阶段 3：审查（PR 自动 review）→ 提效 10%，辅助
阶段 4：重构（复杂业务）→ 实验性，需人工决策
```

**衡量指标**：
```
代码提交频次 +20%
开发周期 -25%
缺陷率 持平或下降（AI 生成的单测覆盖）
开发者满意度 4.5/5
```

## 五、拼多多实战

- **内部 Copilot**：IDE 插件，支持 Java/Go/Python
- **代码规范 RAG**：检索公司框架（Spring Cloud/中间件）用法
- **PR 机器人**：自动审代码规范/潜在 bug
- **单测生成**：根据业务代码生成 JUnit，覆盖率提 20%

## 六、底层本质

AI 代码助手本质是**"用 LLM 自动化重复模式+RAG 接入企业上下文"**——补全样板、生成 CRUD、审 bug，决策和架构仍靠人。短期提效 30-50%，长期演进为"AI 辅助研发流"。

## 常见考点
1. **AI 生成代码有 bug 怎么办**？——人工审查（PR 必经）+ 静态分析+ 单测兜底。
2. **企业代码怎么不外泄**？——私有化部署+权限过滤+不训练用户代码。
3. **怎么衡量 ROI**？——开发周期/缺陷率/满意度，A/B 对比（用 vs 不用）。

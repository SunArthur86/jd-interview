# 【拼多多交易】用 AI Agent 怎么改造交易系统？

> JD 依据："用 AI Agent 改造交易系统"。

## 一、分层改造策略

```
核心交易（下单/支付/库存）→ 保持规则化（确定性、性能）
长尾场景（客服/退款/异常）→ Agent 化（灵活、自动化）
```

| 场景 | 传统 | Agent 化 | 收益 |
|------|------|----------|------|
| 客服咨询 | IVR/人工 | LLM 对话+查单工具 | 人力降 60% |
| 退款审核 | 人工审 | Agent 审+人工复核高额 | 处理提速 10x |
| 异常订单 | 人工排查 | Agent 自主查+处置 | MTTR 降 70% |
| 风控复核 | 人工 | Agent 辅助 | 效率提升 |

## 二、Agent 架构

```
用户输入 → LLM（推理）→ Function Call（调工具）→ 结果 → LLM 总结 → 输出
            ↑                                              ↓
            └──── 记忆（对话历史/用户画像）←──────────────┘
```

**工具定义**（封装交易 API）：
```java
@Tool(name = "queryOrder", desc = "查询用户订单状态")
public Order queryOrder(@Param("orderId") Long id) {
    return orderService.get(id);
}

@Tool(name = "refundOrder", desc = "退款，需额度校验")
public RefundResult refundOrder(@Param("orderId") Long id, @Param("amount") BigDecimal amt) {
    if (amt.compareTo(MAX_AUTO_REFUND) > 0) {
        return RefundResult.needHumanReview();  // 大额转人工
    }
    return refundService.refund(id, amt);
}
```

**Agent 主循环**（ReAct 模式）：
```python
def agent_loop(user_input):
    messages = [{"role": "user", "content": user_input}]
    for step in range(MAX_STEPS):
        response = llm.chat(messages, tools=TOOLS)
        if response.is_final:
            return response.content
        tool_result = execute_tool(response.tool_call)
        messages.append({"role": "tool", "content": tool_result})
    return "需人工介入"
```

## 三、安全护栏

```
1. 工具校验：每个 tool 内置参数校验+权限校验
2. 额度限制：单 Agent 单日退款 < 1 万元
3. 高风险转人工：退款 > 500 元/异常订单触发审核
4. 审计日志：每次 tool call 留痕，可回溯
5. 兜底：Agent N 步无解 → 转人工
```

## 四、落地优先级

```
P0 客服（Q&A + 查单）：低风险，数据积累充分
P1 退款审核：中等风险，工具+额度兜底
P2 异常订单处置：高风险，Agent 建议+人工确认
P3 风控辅助：Agent 出建议，人工决策
```

## 五、拼多多场景

- **百亿补贴客服**：Agent 自动解释规则、查补贴进度
- **拼团异常**：未成团自动退款、咨询自动应答
- **物流异常**：Agent 主动联系物流+通知用户+补偿券

## 六、底层本质

Agent 改造本质是**"把规则无法穷举的长尾场景用 LLM 推理+工具调用自动化"**——核心保确定性（规则），长尾求灵活性（Agent），安全靠工具校验+额度+人工兜底。

## 常见考点
1. **Agent 和传统 RPA 区别**？——RPA 固定流程，Agent 自主决策（LLM 推理）。
2. **怎么防 Agent 乱退款**？——tool 内额度校验+大额人工+审计回溯。
3. **Function Calling 怎么实现**？——LLM 输出结构化 JSON（tool_name+args），工程层解析调用。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你坚持"核心交易走规则，长尾走 Agent"，为什么核心交易不能用 Agent？Agent 也能调退款 API，看起来更灵活。**

核心交易的诉求是"确定性 + 性能"，Agent 两条都不满足。性能上，Agent 是 LLM 推理（秒级响应），下单要求 P99 <300ms，Agent 的 LLM 调用就要 1-3 秒，直接超时。确定性上，LLM 有幻觉——同样的输入可能输出不同的决策（如某次推理错误多退了款），核心交易"每笔必须按规则精确处理"，不能接受概率性错误。规则的动机是"资金安全必须 100% 可预测"，规则引擎对"满 100 减 20"的判定是确定性的，Agent 推理可能因 prompt 不同给出不同结果。所以核心交易（扣款/扣库存/状态流转）必须规则，Agent 只在"规则难穷举的长尾"（如"用户描述的问题是什么意图""异常订单怎么处置")发挥灵活推理能力。分层是确定性和灵活性的权衡。

### 第二层：证据与定位

**Q：Agent 处理客服时，用户反馈"退款没到账"，Agent 查了订单说"已退款"，但实际用户没收到钱，你怎么定位是 Agent 判断错了，还是真的没到账？**

按 Agent 的 tool call 链路核查。三步：
1. 看 Agent 的执行 trace——Agent 调了哪些工具（`queryOrder`/`queryRefund`）、工具返回什么。如果 `queryRefund` 返回"退款状态=成功"，Agent 据此回答"已退款"，是 Agent 基于工具数据的推理没错。
2. 验证工具数据准确性——直接查退款服务 DB，这笔退款的真实状态。如果 DB 显示"退款失败"但工具返回"成功"，是工具的数据问题（如查询了缓存的旧状态、或查询条件错）。这是 Agent 被工具的错误数据误导。
3. 如果退款 DB 确实成功但用户没收到——是通道侧（银行/支付宝）延迟，Agent 的回答"已退款"是对的（平台侧已退），只是用户银行到账延迟。Agent 应该进一步查"退款到通道的时间 + 通道预计到账时间"再回答。重点是区分"Agent 推理错"（LLM 幻觉）、"工具数据错"（系统 bug）、"业务真异常"（通道延迟），三者解法不同。

### 第三层：根因深挖

**Q：你发现 Agent 偶尔会"幻觉"调用不存在的工具或编造订单号（如返回一个查不到的 orderId），根因是什么？光是把 prompt 写得更清楚就行吗？**

调 prompt 是治标。根因是"LLM 的生成式本质 + 工具约束不足"：
1. LLM 生成式本质——LLM 是概率生成，可能"编造"看似合理但不存在的 orderId（尤其当 prompt 没明确"必须从工具返回值取 orderId"时）。治本是"工具结果强制约束"——Agent 不能凭空生成 orderId，必须从 `queryOrder` 的返回值里取。工程上校验"Agent 传给 `refundOrder` 的 orderId 必须在之前 `queryOrder` 的结果集里"，否则拒绝并提示 LLM 重新推理。
2. 工具 schema 不严——如果工具的参数没有 enum/正则约束，LLM 可能传非法值。治本是 JSON Schema 严格定义参数（如 orderId 必须是 19 位数字、且在白名单）。
3. 更根本——Agent 的 ReAct 循环要"工具结果反馈+纠错"。LLM 编造 orderId 时，工具返回"orderId 不存在"，Agent 应该基于反馈重新推理（而非坚持编造）。加"工具校验失败重试"机制。根因是"没把 LLM 当不可信组件约束"，治本是"输入输出全程校验 + 失败反馈纠错"。

**Q：那为什么不直接用规则系统做客服（FAQ 匹配 + 关键词路由），而要用 LLM Agent？规则没有幻觉。**

规则的客服系统的局限：覆盖不了长尾问题。用户问"我上周买的那个红色的，还没发货，能不能换蓝色的"——这种"自然语言理解 + 多步操作（查单/查库存/换货）"靠规则（关键词匹配/决策树）做不好，要么匹配不到（用户转人工）、要么理解错（答非所问）。LLM 的优势是"自然语言理解 + 灵活推理 + 多工具编排"，能处理规则穷举不了的表述多样性。幻觉的代价通过"工具校验+人工兜底"控制。拼多多客服有几十万种用户表述，规则覆盖 70%（常见 FAQ），剩余 30% 长尾靠 Agent 自动化（省人力）。规则和 Agent 不是替代，是"规则兜常见 + Agent 兜长尾 + 人工兜极复杂"的分层。

### 第四层：方案权衡

**Q：你的安全护栏是"大额退款（>500 元）转人工"，但这样 Agent 自动化的价值打折（大额恰恰是需要效率的），你怎么权衡"自动化率"和"安全"？**

权衡方案是"风险评估 + 分级自动化"：
1. 不是所有大额都转人工——结合"用户信用 + 退款原因 + 商户风险"。老用户、低风险原因（如商品瑕疵）、正常商户的大额退款可以自动；新用户、高风险原因（如批量退款）、问题商户的大额转人工。用风控模型评估单笔退款的风险分，高风险转人工，低风险自动。
2. 渐进信任——Agent 先处理小额（<100 元）积累成功数据，验证无误后逐步放开额度（100→500→1000）。每放开一档监控"Agent 自动退款的差错率"，达标才继续。
3. 人工抽检——即使自动处理的，也按比例（如 5%）抽检审计，发现差错调整阈值。权衡是"按风险动态决定自动化边界"，不是"金额一刀切"。低风险高额度可以自动（效率），高风险小额也转人工（安全）。

**Q：为什么不让 Agent 处理所有退款（不设额度上限），完全自动化，效率最高？**

完全自动化风险不可控。LLM 的幻觉/误判没有上限兜底时，一次错误（如把 1000 元退成 10000 元、或给非退款场景误退）就是资损。即使概率低（如 0.1%），拼多多每天百万级退款，0.1% 就是上千笔资损。额度上限是"单次错误的最大损失"控制——Agent 最多误退 500 元，超出的转人工。这是金融系统的"风险敞口"管理。完全自动化要等 LLM 可靠性（如差错率 <0.001%）+ 校验机制（实时风控+事后审计+对账兜底）都足够成熟，才能逐步放开。当前 LLM 的可靠性还撑不住"无上限自动退款"，必须设限。

### 第五层：验证与沉淀

**Q：你怎么验证 Agent 在真实业务中不出资损？LLM 的输出是概率性的，传统单测覆盖不了。**

Agent 的测试要专门设计：
1. 黄金集回归——人工标注一批"问题-正确处理"的黄金数据（如"用户要求退款 X 元，正确操作是 queryOrder→checkEligibility→refund X"），每次 Agent 版本更新跑黄金集，断言"工具调用序列正确+参数正确"。这是 Agent 的"单测"。
2. 影子流量验证——Agent 上线前，真实用户请求同时跑"Agent"和"人工"，对比 Agent 的决策和人工决策的一致率。一致率达标（如 >95%）才上线。影子流量让 Agent 在"不执行真实操作"的前提下积累真实数据验证。
3. 在线审计——上线后所有 Agent 的 tool call 进审计日志，按比例（如 10%）人工抽检，`agent_error_rate`（Agent 决策错误的比例）监控。资损相关的高危操作（退款/发券）100% 审计。拼多多这种规模，Agent 错误率必须量化监控，不能靠"没投诉"。

**Q：Agent 改造怎么沉淀成平台，让多个业务（客服/退款/异常处置）复用，而不是每个场景各建一个 Agent？**

沉淀成"Agent 平台"：
1. 统一 Agent 框架——封装 ReAct 循环/工具调用/记忆/安全护栏，业务侧只定义"工具集 + prompt 模板 + 额度规则"。新场景接入（如售后 Agent）复用框架，不重写。
2. 工具市场——把交易系统能力（查单/退款/发券/查物流）封装成标准工具，注册到工具市场，各 Agent 按需调用。工具的权限/校验/审计统一在平台层，业务 Agent 不自己实现。
3. 评测平台——所有 Agent 的"黄金集/影子流量/在线审计"在统一平台跑，标准化评测指标（准确率/资损率/自动化率）。新 Agent 上线必须过评测。
4. 模型层抽象——底层 LLM 可插拔（如换 GLM-5/GPT），Agent 框架与具体模型解耦。模型升级时所有 Agent 受益。把 Agent 从"每场景一个项目"变成"平台 + 配置"，这是 AI 工程化的核心。

## 结构化回答

**30 秒电梯演讲：** 交易系统有大量长尾场景靠人工，如何用 AI 自动化又不影响核心确定性？简单说就是——用 AI Agent 改造交易系统是"把规则硬编码的业务决策换成 LLM 推理+工具调用"。Agent = LLM + 工具 + 记忆；安全：工具校验+额度+人工兜底。

2. **Agent** — Agent = LLM + 工具 + 记忆

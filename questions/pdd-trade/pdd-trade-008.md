# 【拼多多交易】消息队列怎么用？数据同步怎么做？

> JD 依据："消息中间件、数据同步平台"。

## 一、MQ 三大作用

```
1. 解耦: 订单事件 → 库存/物流/积分/风控（各订阅）
2. 削峰: 秒杀洪峰 → Kafka → 稳定消费
3. 数据同步: MySQL binlog → Canal → Kafka → ES/Redis/数仓
```

## 二、Kafka vs RocketMQ

| 维度 | Kafka | RocketMQ |
|------|-------|----------|
| 吞吐 | 极高（百万/秒） | 高（十万/秒） |
| 事务消息 | 弱 | 原生支持 |
| 顺序 | partition 内 | 队列内 |
| 适用 | 大数据/日志 | 电商事务 |

拼多多交易选 RocketMQ（事务消息）+ Kafka（数据同步）。

## 三、数据同步（Canal + Kafka）

```
MySQL → Canal（伪装从库）→ Kafka → 消费者
   ├→ ES（搜索索引）
   ├→ Redis（删缓存）
   └→ 数仓（离线分析）
```

业务代码无感知，binlog 驱动最终一致。

## 四、消息可靠性

```properties
# 生产
acks=all
enable.idempotence=true

# Broker
replication.factor=3
min.insync.replicas=2

# 消费
enable.auto.commit=false  # 手动提交
```

消费端幂等（业务唯一键）。

## 五、订单事件流（交易核心）

```
下单 → topic: order-created
   ├→ 库存服务：冻结库存
   ├→ 营销服务：核销券
   ├→ 风控服务：异步复核
   └→ 数仓：实时 GMV

支付 → topic: order-paid
   ├→ 库存：确认扣减
   ├→ 商家：通知发货
   └→ 积分：加分
```

## 六、底层本质

MQ 本质是**"生产者消费者的解耦中间件"**——把同步强依赖转异步弱依赖，让系统更松耦合、更弹性。

## 常见考点
1. **Kafka 为什么快**？——顺序写磁盘 + 零拷贝（sendfile）+ 批量压缩 + 分区并行。
2. **怎么保证消息顺序**？——同 key 进同 partition，单 partition 单消费者有序。
3. **消息积压**？——扩消费者 + 排查慢消费 + 临时跳过堆积。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：订单事件你既用了 RocketMQ（事务消息）又用了 Kafka（数据同步），为什么不统一用一个？**

两者擅长的场景不同。订单事件通知（库存冻结/券核销/积分）需要"事务消息"——订单落库和消息发送必须原子，RocketMQ 的半消息+回查机制原生支持，且电商场景下消息可靠性 > 极致吞吐。而数据同步（binlog→ES/Redis/数仓）是 Canal 生产、批量大、容忍秒级延迟，Kafka 的顺序写+零拷贝在大数据吞吐上优势明显，且数仓/实时计算生态（Flink/Spark）对 Kafka 支持更好。统一用一个的代价是：用 Kafka 做订单事件要自己实现事务消息（本地消息表），复杂；用 RocketMQ 做数据同步吞吐不如 Kafka，且 Flink 对接要额外适配。混合用的动机是"各取所长"，代价是运维两套 MQ。

### 第二层：证据与定位

**Q：订单服务的 RocketMQ 积压了 500 万条消息，你怎么定位是生产者发太快，还是消费者消费太慢？**

看生产速率 vs 消费速率的对比。三步：
1. 看 RocketMQ Dashboard 的 `topic TPS`（生产）和 `consumer TPS`（消费）——如果生产 1 万/s、消费 5000/s，明显是消费者慢。如果生产 5 万/s、消费 5 万/s 但历史积压还在，说明之前有段时间生产远超消费，现在追平了但积压未消。
2. 看消费者的 `consume latency`（消费单条耗时）——如果从 5ms 飙到 200ms，是消费者处理逻辑慢了。jstack 看消费线程在干嘛：是不是在等下游 RPC（库存服务慢）、或 DB 慢查询、或 GC 停顿。
3. 看消费者实例数 vs partition 数——如果消费者实例数 >= partition 数，加消费者也没用（多余的抢不到 partition）。这是配置问题，不是消费慢。重点是区分"消费能力不足"（加机器/优化逻辑）和"配置不合理"（partition 太少）。

### 第三层：根因深挖

**Q：你定位到是消费者处理慢，jstack 显示消费线程都阻塞在调库存服务的 RPC 上，根因是什么？光是把库存服务扩容就行吗？**

扩容库存服务可能治标，根因要看库存服务为什么慢：
1. 库存服务本身的 DB 慢——查库存服务的 APM，看 `inventoryDao.freeze` 的 SQL 耗时。`EXPLAIN` 看是不是全表扫（缺索引），或行锁等待（`SHOW PROCESSLIST` 看 `Lock_wait`）。
2. 如果库存 DB 没问题，是库存服务线程池满了——查 `ThreadPoolExecutor` 的 `activeCount == maxPoolSize`，队列堆积。根因可能是线程池配置小，或下游（如 Redis/其他 RPC）慢拖累。
3. 更深一层：订单消息批量到达（大促积压），消费者串行调库存 RPC，每条 50ms × 1 万条 = 500 秒。治本可以是"批量消费"——RocketMQ 支持 `consumeMessageBatchMaxSize=50`，一次拉 50 条批量处理，RPC 调用次数减少。但前提是业务允许批量（50 条订单一起冻结库存）。

**Q：那为什么不直接在消费端用多线程并行消费每条消息，加快速度？**

多线程并行消费会破坏消息顺序和限流语义。RocketMQ 单 partition 单消费者是为了"顺序消费"——同一订单的 create/pay/refund 必须按顺序处理，多线程并行可能导致 pay 比 create 先执行。即使不考虑顺序，多线程会让下游（库存 DB）瞬时承受 N 倍压力，本来 DB 就是瓶颈，加并发反而雪崩。正确扩容方式是增加 partition 数（让多个消费者实例并行消费不同 partition），而不是单消费者内部多线程。如果业务确实无序且下游扛得住，可以用"并发消费+幂等"模式（RocketMQ 的 `MessageListenerConcurrently`），但要在业务层保证幂等，且限流保护下游。

### 第四层：方案权衡

**Q：你用 Canal 监听 binlog 同步到 ES，但如果某条 binlog 解析失败或写 ES 失败，这条消息丢了，数据就不一致了。你怎么权衡"实时性"和"可靠性"？**

Canal 同步的可靠性靠"失败重试 + 对账补偿"。权衡方案：
1. Canal 消费端的失败处理——解析失败的消息进死信队列（DLQ），不阻塞后续 binlog 消费，同时告警人工介入。写 ES 失败的重试，重试 N 次仍失败进 DLQ。
2. 全量对账补偿——每天低峰跑一次 `MySQL vs ES` 的对账（按 update_time 扫描当天变更的记录，逐条对比），不一致的重新同步。这是兜底，覆盖所有"实时同步漏掉"的情况。
3. 实时性 vs 可靠性的取舍——对搜索场景（ES）容忍秒级延迟，所以可以"先消费成功 + 异步重试 + 对账兜底"，不阻塞主链路。如果是资金相关（如账户余额同步），必须同步双写 + 强对账，不能用 Canal 异步。

**Q：为什么不直接业务代码双写（写 MySQL 同时写 ES），而要用 Canal 监听 binlog？**

业务双写有三个问题：一是每个写 MySQL 的地方都要记得写 ES，漏一个就不一致，几十个 service 方法维护成本高；二是双写的事务问题——MySQL 成功 ES 失败怎么办？要么引入分布式事务（重），要么接受短暂不一致；三是历史数据迁移——业务双写只能覆盖新增，老数据要单独脚本刷。Canal 监听 binlog 的优势是"业务无感知"（binlog 是 MySQL 的副作用，业务代码不改）+ 顺序可靠（binlog 本身有序）+ 全量增量统一（Canal 既能消费实时 binlog，也能做全量初始化）。代价是引入 Canal 组件（运维成本）+ 秒级延迟。对于"不需要强一致的数据同步"场景，Canal 是更优雅的解耦方案。

### 第五层：验证与沉淀

**Q：你怎么验证消息可靠性配置（acks=all/手动 offset/幂等消费）真的生效，消息不丢不重？**

必须做故障注入测试，覆盖三个环节：
1. 生产端可靠性——压测期间 kill broker 节点（模拟 Broker 故障），验证 `acks=all` 下消息不丢（`min.insync.replicas=2` 保证两个副本写入才算成功）。对比生产消息数和 broker 收到的消息数，`message_loss_count = 0`。
2. 消费端幂等——人为让消费者抛异常触发重试（同一条消息消费 3 次），验证业务侧（如积分表）只生效一次，靠业务唯一键幂等。
3. offset 提交——验证手动提交模式下，消费到一半进程挂了，重启后从上次提交的 offset 继续，不丢不重。`auto.offset.reset=earliest` 防止 offset 失效时从头消费导致重复（配合幂等兜底）。每个环节都要有明确的断言数字，不能靠"没报错"。

**Q：消息积压/丢失这种事怎么沉淀成机制，而不是每次大促都手动救火？**

沉淀成监控+预案+容量规划：
1. 积压告警自动化——`consumer_lag`（积压量）接入 Prometheus，超过阈值（如 10 万条）自动告警 + 触发扩消费者（K8s HPA 按 lag 自动扩缩）。大促前按预估流量预扩容，不等积压才扩。
2. 容量规划基线——每次大促后记录"流量峰值/积压量/消费 TPS/机器数"，建立容量模型，下次大促按模型预估机器数。
3. 降级预案——积压到临界值（如影响业务）时，自动触发降级：非核心消费者（如数仓同步）暂停消费让出资源、或跳过堆积消息（业务能容忍的话）先恢复实时消费。把"救火动作"沉淀成可自动执行的预案，而不是靠人半夜爬起来处理。

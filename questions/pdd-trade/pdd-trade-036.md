# 【拼多多交易】RAG 在交易怎么用？

> JD 依据："RAG 在交易"。

## 一、RAG 流程

```
文档库 → 切块（chunk）→ embedding → 向量库
                                        ↓（检索）
用户问题 → embedding → 相似检索 → rerank → 拼 prompt → LLM 生成
```

## 二、文档处理

**切块策略**：
```python
# 按语义切块（标题/段落）+ 重叠窗口
chunks = semantic_chunk(
    doc,
    max_tokens=500,
    overlap=50,          # 防切断上下文
    split_on=["h1","h2","段落"]
)
```

**文档类型**：
```
商品详情：标题/规格/参数/详情页（每商品独立索引）
售后规则：退货政策/退款流程/争议处理（按政策版本）
FAQ：高频问题+答案（客服自助）
订单政策：拼团/预售/百亿补贴规则
```

## 三、检索+重排

```java
public List<Chunk> retrieve(String query, String docType) {
    // 1. 向量召回（粗排，top50）
    List<Chunk> vec = vectorStore.search(embed(query), topK=50);
    // 2. 关键词召回（BM25，补精确匹配）
    List<Chunk> bm25 = keywordStore.search(query, topK=50);
    // 3. 合并 + 去重
    List<Chunk> merged = merge(vec, bm25);
    // 4. Rerank（精排，cross-encoder，top5）
    return reranker.rerank(query, merged, topK=5);
}
```

**Hybrid 必要性**：
- 纯向量：语义相似但关键词错（"iPhone 15" 检出 "iPhone 14"）
- 纯 BM25：精确但不懂同义（"退款" 匹配不到 "退货"）
- 混合：语义+精确兼顾

## 四、Query 改写

用户问得口语化/模糊，先改写：
```
用户："买的那个还没到"
改写：订单状态查询 → "订单 [订单号] 物流状态 预计送达"
+ 查用户最近订单上下文
```

## 五、GraphRAG（关系增强）

商品/类目/品牌有强关系，传统向量检索忽略结构。

```
知识图：商品 -[属于]-> 类目 -[关联]-> 配件
       商品 -[品牌]-> 品牌 -[同档]-> 竞品
```

GraphRAG：向量召回 + 图扩展（找关联商品/品牌）。

**拼多多导购**：
```
"推荐 iPhone 配件" →
  向量召回：配件商品
  图扩展：iPhone 关联的充电壳/膜/数据线
```

## 六、工程化

**增量索引**：
```java
@KafkaListener(topics = "product-update")
public void onProductUpdate(ProductEvent e) {
    List<Chunk> chunks = chunker.chunk(e.getProduct());
    List<Vector> vectors = embedder.embed(chunks);
    vectorStore.upsert(e.getProductId(), vectors);  // 增量更新
}
```

**缓存**：
```
热门问题（"怎么退款"）→ 缓存答案，命中 60% 流量
```

**评测**：
```
标注集 1000 问 → 召回率/准确率/满意度
新版检索/切块 → 回归评测 → 上线
```

## 七、底层本质

RAG 本质是**"用检索给 LLM 补外挂知识"**——切块平衡精度和上下文，Hybrid+Rerank 提召回质量，GraphRAG 增强关系，增量索引保时效。

## 常见考点
1. **切块多大合适**？——512-1024 token，太大稀释相似度，太小丢上下文，按文档类型调。
2. **Rerank 为什么必要**？——向量召回快但不精（双塔），Rerank（cross-encoder）精排提准确。
3. **RAG 和微调区别**？——RAG 外挂知识（即时更新），微调内化能力（风格/格式），可叠加。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：交易场景的知识（商品/售后规则/订单政策）用 RAG 外挂，为什么不直接微调模型把这些知识训进去？微调后模型直接知道答案，省了检索这一步不是更快？**

微调做不到"知识频繁更新"。交易场景的知识是高变的——商品每天上下架几十万、促销规则每天变（双 11 和平时不一样）、退货政策按活动调整。微调一次模型几小时到几天，且每次知识变都要重训，成本和时效都不可行。RAG 的动机是"知识外挂可即时更新"——商品/规则变更触发增量 re-embed 入库（秒级），下次检索就是新知识。微调擅长内化"稳定的能力"（如回答风格/输出格式/领域术语），RAG 擅长挂载"变化的知识"。交易场景的知识是高频变的，必须 RAG；如果要统一回答风格/格式，再叠加微调。两者互补不是二选一。

### 第二层：证据与定位

**Q：客服反馈"RAG 召回了完全不相关的商品文档"，你怎么定位是切块问题、embedding 问题，还是检索策略问题？**

按 RAG 链路逐层排查。三步：
1. 看召回结果——把这次 query 的 topK=50 召回全部打印。如果召回的全是不相关文档，是 embedding 或索引问题；如果召回里有相关的但排在 top5 之外，是 rerank 没排好。
2. 看切块——检查召回文档所在的 chunk，是不是把不相关内容切一起了（如商品 A 的规格和商品 B 的描述在同一 chunk，因切块没按商品边界）。如果 chunk 内容混杂，是切块策略问题。
3. 看 embedding——用 `embed(query)` 和 `embed(相关文档)` 算 cosine 相似度，如果本应高相关的两者相似度低，是 embedding 模型不适合（如通用模型对电商术语理解差）。三组证据定位：召回全错→embedding/索引；召回有但排序差→rerank；chunk 内容混杂→切块。不同根因不同修法。

### 第三层：根因深挖

**Q：你定位到是切块问题——一个 chunk 把多个商品的参数表混在一起（因为参数表是表格密集，切块按固定 500 token 切，把商品 A 的末尾和商品 B 的开头切一起了）。根因是固定 token 切块不适合表格，还是更深层？**

根因是"固定 token 切块忽视了文档结构"。深挖：
1. 结构盲切——固定 500 token 切块不考虑文档语义边界（标题/段落/表格行），把强相关内容（如商品 A 的全部参数）切断，或把不相关内容（商品 A 和 B）拼一起。治本是"结构化切块"——按文档的 DOM 结构（h1/h2/段落/表格行）切，保证一个 chunk 内是同一商品/同一语义单元。
2. 表格特殊处理——参数表是"字段-值"对，固定 token 切会破坏结构。要把每行参数（如"屏幕：6.1 英寸"）作为独立单元，或把整个表格转成 KV 文本再切，保证字段完整。
3. 元数据附加——每个 chunk 带元数据（商品 ID/类目/品牌），检索时按元数据过滤（如只召回指定商品的 chunk）。根因是"切块必须尊重文档结构"，固定 token 切只适合纯文本，结构化文档（表格/列表/多商品页）要语义化切。

**Q：那为什么不按"每个商品一个 chunk"切，最简单不就保证不混了吗？**

单商品单 chunk 太粗会丢精度。一个商品详情页有标题/规格/详情图描述/评价摘要，信息量大（可能 2000+ token）。如果一个商品塞一个 chunk：
1. 相似度稀释——query 问"iPhone 15 的屏幕尺寸"，chunk 是整个 iPhone 15 详情（含电池/相机/价格），向量是全内容的平均，和"屏幕尺寸"的相似度被其他内容稀释，可能召回不准。
2. prompt 膨胀——召回 top5 商品就是 10000+ token，prompt 爆炸。
3. 正确做法是"分层切"——商品级（粗）+ 字段级（细），如"iPhone 15/屏幕/6.1 英寸"是一个细 chunk，带商品 ID 元数据。检索时既保结构（不混商品）又保精度（字段级匹配）。单商品单 chunk 是简单但牺牲精度，分层切才能兼顾。

### 第四层：方案权衡

**Q：你的检索用 Hybrid（向量 + BM25），为什么要混两个，而不是只用一个最强的？哪个权重该大？**

单一检索都有盲区：
1. 纯向量（语义）——擅长同义（"退款"匹配"退货"），但弱精确匹配（"iPhone 15" 可能召回 "iPhone 14"，语义相近但型号错）。电商场景型号/规格错是硬伤。
2. 纯 BM25（关键词）——擅长精确（"iPhone 15" 精确匹配），但不懂同义（"退款"匹配不到"退货"），口语化 query 召回差。
3. Hybrid 权重——不是固定 50:50，按 query 类型调：精确查询（型号/SKU）BM25 权重大（如 0.7 向量 + 0.3 BM25 反着来），语义查询（"怎么退货"）向量权重大。或用 RRF（Reciprocal Rank Fusion）自动融合两路排序，不依赖固定权重。Hybrid 的本质是"语义 + 精确互补"，覆盖单路的盲区。

**Q：为什么不直接用一个更强的 embedding 模型（如最新的大参数模型），让它同时搞定语义和精确，不就不用 BM25 了吗？**

更强 embedding 仍做不到"语义+精确"双优。原因：
1. embedding 是"语义压缩"——把文本压成向量不可避免丢精确信息（如具体型号数字），再强的模型也有这个本质局限。BM25 是"字面匹配"，精确信息不丢。
2. 成本权衡——最强 embedding 模型（如大参数）推理慢且贵，BM25 是倒排索引查表，毫秒级且几乎免费。Hybrid 用中等 embedding（快）+ BM25（免费）组合，效果接近最强 embedding 但成本低得多。
3. 实测验证——在电商标注集上，Hybrid（中等向量 + BM25）的 Recall@10 通常优于单一强向量模型，因为补上了精确匹配盲区。用一个最强模型替代 Hybrid 在效果和成本上都不优，Hybrid 是工程上的最优解。

### 第五层：验证与沉淀

**Q：你怎么持续评估 RAG 的检索质量？线上效果差的时候怎么知道是检索的问题还是生成的问题？**

建立"检索 + 生成"分层评测：
1. 检索单独评——标注集 1000 问，每问标注"正确文档"。跑检索后看 Recall@K（正确文档是否在 topK）和 MRR（正确文档排名）。Recall@5 < 90% 是检索问题，和生成无关。这个评测纯检索，不跑 LLM，快且便宜。
2. 端到端评——跑完整 RAG（检索 + 生成），看回答准确率。如果检索 Recall 高但端到端准确率低，是生成问题（LLM 没遵循检索结果）；如果检索 Recall 低，是检索问题。
3. 在线监控——线上采集"用户负反馈率"（点踩/转人工），和离线 Recall/准确率关联。负反馈飙升时对比检索指标，定位是哪层。分层评测把"检索"和"生成"解耦，才能精准定位问题层，而非笼统说"RAG 效果差"。

**Q：RAG 怎么沉淀成交易场景的平台能力，让客服/导购/售后都复用？**

沉淀成"RAG 平台"：
1. 统一文档处理 pipeline——切块/embedding/入库标准化，业务侧只提供文档（商品/规则/FAQ），平台自动处理入索引。
2. 统一检索 API——Hybrid 检索 + rerank 封装，业务侧调 `retrieve(query, docType)` 即可，不碰底层。
3. 多租户索引——客服/导购/售后各自的文档索引隔离，但共享检索引擎和评测能力。
4. 增量更新总线——商品/规则的 binlog 变更统一接到 Kafka，平台自动 re-embed 更新索引，业务无感。
5. 评测平台——所有场景的标注集/回归/线上监控统一管理，新场景接入即享评测。把 RAG 从"每场景自建"变"平台 + 配置"，这是 RAG 工程化的必然路径。

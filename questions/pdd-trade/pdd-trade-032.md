# 【拼多多交易】LLM 在订单/客服怎么应用？

> JD 依据："LLM 在订单/客服应用"。

## 一、客服场景分层

```
L1 自助（FAQ）：物流/退款规则/优惠说明 → RAG 直接答（80% 流量）
L2 AI 客服（查单）：查订单状态/解释异常 → LLM + 工具（15%）
L3 人工（投诉/复杂）：情绪安抚/特殊处理 → 人工兜底（5%）
```

## 二、RAG 架构

```
用户问题 → 向量化 → 向量库检索（FAQ/规则）→ 拼 prompt → LLM 生成 → 回答
                         ↑
                   业务知识库（售后规则/FAQ/政策）
```

**知识库构建**：
```
售后政策（退货/换货/退款规则）
商品 FAQ（常见问题）
订单状态说明（拼团/预售/百亿补贴）
物流异常处理 SOP
```

**检索+生成**：
```java
public String answer(String question, long uid) {
    // 1. 向量检索相关 FAQ
    List<Doc> docs = vectorStore.search(question, topK=5);
    // 2. 查用户订单（工具）
    List<Order> orders = orderService.listByUid(uid);
    // 3. 拼 prompt
    String prompt = buildPrompt(question, docs, orders);
    // 4. LLM 生成
    String answer = llm.chat(prompt);
    // 5. 低置信转人工
    if (confidence(answer) < 0.7) return routeToHuman();
    return answer;
}
```

## 三、订单状态解释（LLM 加值）

传统：返回"已发货"（用户不懂细节）。
LLM：
```
你的订单已发货，顺丰单号 SF123，预计 7/10 送达。
当前在【上海转运中心】，正常时效。
如超时可回复"延期"申请补偿。
```

## 四、降幻觉手段

1. **RAG 约束**：检索最新政策喂 prompt，避免过时知识
2. **Prompt 约束**：明确"只基于提供的资料回答，不知道就说不知道"
3. **引用来源**：回答附政策链接，用户可核实
4. **Function Calling**：实时数据（订单/物流）走工具，不靠 LLM 记忆

## 五、拼多多实战

- **百亿补贴客服**：LLM 解释补贴规则、查补贴进度
- **拼团咨询**：解释成团机制、未成团自动退款流程
- **物流异常**：LLM 主动联系物流+安抚用户+发补偿券
- **多轮对话**：支持"退款进度→退款到哪→预计到账"连续追问

## 六、底层本质

LLM 客服本质是**"理解+检索+生成三件套"**——LLM 理解意图，RAG 提供准确知识，Function Calling 查实时数据，组合起来覆盖 80% 自助+15% AI 客服，5% 留人工兜底。

## 常见考点
1. **RAG 怎么选向量模型**？——中文用 BGE/M3E，召回优先用向量+关键词混合检索。
2. **多轮对话怎么记上下文**？——session 记历史 message，超出窗口用摘要压缩。
3. **LLM 回答违规怎么办**？——安全分类器前置+关键词黑名单+prompt 约束+人工抽审。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：客服场景你用 RAG（检索增强生成）而非让 LLM 直接回答，RAG 的检索+生成比直接生成多一步，为什么多此一举？**

LLM 直接回答的"幻觉"在业务场景是灾难。用户问"百亿补贴的退货规则"，LLM 直接答可能编造一个看似合理但错误的规则（如"7 天无理由退货"，实际百亿补贴可能是"15 天"），用户照做发现被骗，投诉更严重。RAG 的动机是"把权威知识检索出来喂给 LLM，让回答基于事实而非 LLM 的记忆"。检索 + 生成比直接生成多一步，但这步是"事实约束"——LLM 基于检索到的真实政策回答，幻觉率从 20%+ 降到 <5%。客服场景"准确性 > 速度"，多 200ms 的检索延迟换"回答基于真实政策"是值得的。直接生成只在"创意/开放性"场景（如写文案）合适，事实性问答必须 RAG。

### 第二层：证据与定位

**Q：用户反馈"客服回答的政策是错的"（如说退货期 7 天，实际是 15 天），你怎么定位是 RAG 检索错了，还是 LLM 生成时没遵循检索结果？**

按 RAG 链路逐步查。三步：
1. 看检索结果——查这次请求的 `vectorStore.search` 返回了哪些文档（topK=5）。如果检索到的文档本身就是旧的"7 天退货"政策（知识库没更新），是知识库问题。如果检索到的是正确的"15 天"，但 LLM 回答了 7 天，是生成问题。
2. 看生成 prompt——把检索结果和用户问题拼成的 prompt 打印出来，看检索到的"15 天"是否在 prompt 里。如果在 prompt 里但 LLM 仍回答 7 天，是 LLM 没遵循 prompt（可能 prompt 指令不够强、或上下文太长 LLM 忽略了）。
3. 看 LLM 输出的引用——如果回答附了引用来源（如"详见政策链接 X"），验证链接 X 的内容是否真是 15 天。如果链接是 15 天但回答 7 天，明显是生成 bug。重点是区分"检索阶段错（知识库/召回）"和"生成阶段错（LLM 不遵循）"，前者优化检索，后者优化 prompt 或换更强模型。

### 第三层：根因深挖

**Q：你查到是知识库里的政策过时了（活动改了退货期但知识库没同步），根因是什么？光是手动更新知识库就行吗？**

手动更新是治标且易漏。根因是"知识库更新机制和业务系统脱节"：
1. 政策变更没触发知识库更新——运营在业务系统改了"退货期 15 天"，但知识库（向量库）还是旧的"7 天"。治本是"业务系统变更 → 自动触发知识库重建"——如政策表用 Canal 同步，binlog 变更时重新 embedding 入向量库。或定时（每小时）从业务系统拉最新政策重建索引。
2. 知识库的多版本管理——活动期间政策可能临时变（如双 11 退货期延长），活动后恢复。知识库要支持"时间维度"的版本（如某政策 11/1-11/11 生效），检索时按当前时间过滤。
3. 校验机制——定时（每天）对比知识库内容和业务系统源数据，`policy_mismatch_count` 告警。根因是"知识库是派生数据，必须和源数据同步且可校验"，不能独立维护。

**Q：那为什么不直接让 LLM 实时查业务系统的政策表（而非先建知识库再检索），不就没有同步问题了吗？**

实时查政策表（Function Calling）和 RAG 检索知识库的适用场景不同：
1. 结构化数据（如"某 SKU 的退货期"）适合 Function Calling——`queryPolicy(skuId)` 直接查 DB，返回精确值，无幻觉。这种用工具调用更好。
2. 非结构化知识（如"退货流程的详细说明""百亿补贴的复杂规则")适合 RAG——这些是长文本，LLM 需要理解语义后回答，Function Calling 返回的原始文本太长塞 prompt 也行，但检索更精准（只召回相关片段）。
3. 实际是混合——结构化字段（退货天数/金额）走 Function Calling 保证精确，非结构化说明走 RAG 保证语义理解。完全不建知识库全走 Function Calling，会把"长文本政策"全塞 prompt（token 爆炸）或让 LLM 自己读（理解不准）。知识库 + 工具的混合架构是兼顾精确性和语义理解的方案。

### 第四层：方案权衡

**Q：你的客服分层是"L1 自助 80% / L2 AI 客服 15% / L3 人工 5%"，但 L1 自助（FAQ RAG）和 L2 AI 客服（LLM+工具）边界模糊，怎么权衡该走哪层？**

分层路由的依据是"问题复杂度 + 风险"，而非固定比例：
1. 意图分类前置——用户问题先进 LLM 意图分类（轻量模型），判定"FAQ 可答/需查单/需人工"。FAQ 类（如"退货流程"）走 L1 RAG 直接答；需实时数据（如"我的订单到哪了"）走 L2 LLM+工具；情绪激动/复杂投诉走 L3 人工。
2. 置信度降级——L1 RAG 回答的置信度低（检索没命中强相关 FAQ），自动升级到 L2（LLM 推理）；L2 的置信度低或涉及高风险操作（退款/投诉），升级到 L3 人工。是"逐级升级"而非"硬切"。
3. 动态调整——监控各层的"解决率"和"用户满意度"，如果 L1 的解决率掉（用户转人工多），说明 FAQ 覆盖不够，补知识库；如果 L2 错误多，加强 prompt 或转人工阈值。权衡是"尽量自助（成本低）+ 兜底人工（质量保）+ 按数据动态调边界"。

**Q：为什么 L1 用 RAG 而不用更便宜的"关键词匹配 FAQ"？关键词没有 LLM 成本，更经济。**

关键词匹配的局限是"覆盖不了同义表述"。用户问"买的东西多久能退"，关键词 FAQ 可能只匹配"退货"标准问题，但用户的真实意图是"退货时效"，如果 FAQ 库里是"退货流程"和"退货时效"两条，关键词可能匹配错。RAG 的语义检索能理解"多久能退 ≈ 退货时效"，召回正确的 FAQ。且 RAG 能处理"我上周买的那个还没到怎么办"这种含代词和上下文的问题，关键词做不到。LLM 成本虽高，但 L1 RAG 用轻量模型（如小参数 embedding + 小模型生成），单次成本可控（毫分级），相比"关键词匹配不到转人工"的人力成本，RAG 更经济。只有"完全标准化的短问题"（如"运费多少"）适合关键词，大多数客服问题语义多变，RAG 召回更准。

### 第五层：验证与沉淀

**Q：你怎么评估 LLM 客服的回答质量？自动指标（BLEU/ROUGE）够吗，还是要人工？**

自动指标不够，要"自动 + 人工 + 业务指标"三层：
1. 自动指标（初筛）——RAG 召回的 Recall@K（相关 FAQ 是否被检索到）、回答的引用准确率（引用链接内容是否支持回答）。这些可自动化跑，成本低。
2. 人工标注（精评）——抽样（如每天 500 条）人工评估"回答是否正确/是否解决用户问题/是否有幻觉"，标注准确率。这是金标准但成本高，只抽检。
3. 业务指标（最终）——用户满意度（回答后用户是否转人工/是否好评）、一次性解决率（FCR，用户是否需要追问）、客服人工转接率。这些反映"真实效果"，是最终评判。BLEU/ROUGE 是机器翻译指标，不适合客服（客服回答只要正确，不需要和标准答案字面一致）。重点是"业务指标 + 人工抽样"为主，自动指标为辅。

**Q：LLM 客服怎么沉淀成平台，让多个业务线（订单/物流/售后）复用，而不是各建一套？**

沉淀成"客服 AI 平台"：
1. 统一 RAG 框架——知识库管理（入库/版本/更新）、向量检索（混合检索）、prompt 管理，业务侧只提供"知识 + 场景配置"。
2. 工具市场——查单/查物流/退款/发券等能力封装成标准工具，各场景的 LLM 按需调用，工具的权限/校验/审计统一。
3. 多租户——订单客服/物流客服/售后客服共享平台但知识库和工具隔离，数据不串。
4. 评测平台——所有场景的回答质量在统一平台评估（自动+人工），标准化指标。新场景接入即享有评测能力。
5. 模型层——底层 LLM 可插拔（GLM-5/GPT/自研），按场景选合适模型（简单 FAQ 用小模型省成本，复杂咨询用大模型）。把 LLM 客服从"每个业务线自建"变成"平台 + 配置"，这是 AI 工程化的必然路径。

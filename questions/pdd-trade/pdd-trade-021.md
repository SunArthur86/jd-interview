# 【拼多多交易】订单中心怎么设计？

> JD 依据："订单/用户业务"、"交易系统技术升级"。

## 一、整体架构

```
             ┌─ 创单 → MySQL（分库分表，写）
交易网关 ───→ ├─ 查询 → ES（多维查）/ HBase（明细）
             ├─ 状态机驱动 → Kafka 事件
             └─ 冷热分离 → 归档到 HBase/OSS
```

## 二、分库分表（核心）

**分片键**：`order_id`（一致性 hash 路由到库表）

**基因法**（让 order_id 含 UID 信息，支持按 UID 反查）：
```java
// 生成 order_id：时间戳 + UID 后 4 位（基因）+ 序列号
long orderId = (timestamp << 20) | (uid & 0xFFFF) << 4 | seq;
// 按 UID 查：从 order_id 提取基因反推库表（或扫全表+二级索引）
```

**分片策略**：
```
库数：64 库 × 64 表 = 4096 分片
单分片：≈ 2500 万订单（亿级 / 4096）
扩容：2 倍扩容法，分片位预留
```

## 三、CQRS（读写分离）

```java
// 写：MySQL（事务一致性）
@Transactional
public void create(Order o) {
    orderDao.insert(o);
    outboxDao.insert(new Outbox("OrderCreated", o));  // 同事务事件
}

// 同步：Binlog/Canal → ES（多维查询）
// 读：ES（按 UID/状态/时间/商户任意维度）
public Page<Order> search(OrderQuery q) {
    return esOrderDao.search(q);  // 聚合/分页/模糊
}
```

## 四、订单状态机 + 领域事件

```
待支付 → 已支付 → 已发货 → 已完成
   ↓       ↓        ↓
 已取消  退款中    退款中

状态变更同事务发 Outbox → Kafka → 下游订阅
  履约：发货
  营销：发券/积分
  财务：分账
```

## 五、冷热数据分离

| 数据 | 存储 | 查询频次 |
|------|------|----------|
| 近 3 月 | MySQL 热库 | 高（用户/客服） |
| 3 月-2 年 | HBase | 中（客服/对账） |
| 2 年+ | OSS + 索引 | 低（合规/审计） |

归档任务：每天凌晨 T+1 把"已完结 90 天"订单迁到 HBase，MySQL 删除。

## 六、底层本质

订单中心本质是**"用分片+读写分离+冷热分层应对规模增长"**——分库分表扛写、ES 扛多维读、HBase 扛冷存，用 Outbox 保证事件可靠。

## 常见考点
1. **基因法怎么实现**？——UID 后几位嵌入 order_id，分片路由时用基因定位。
2. **ES 数据怎么和 MySQL 一致**？——Binlog/Canal 增量同步 + 定时全量对账。
3. **订单归档怎么不停服**？——双写过渡期 + 灰度切读 + 验证一致后下线旧库。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你用基因法把 UID 后几位嵌入 order_id，为什么不直接用 UID 做分片键（前面分库分表题是 UID 分片）？**

这是订单中心和交易架构视角的差异。交易架构（pdd-trade-012）按 UID 分片，是因为"买家查我的订单"是最高频查询。但订单中心要服务更多场景：支付按 order_id 查、财务按 order_id 对账、客服按 order_id 查，这些场景 order_id 是入口。如果按 UID 分片，按 order_id 查要先"从 order_id 反查 UID"（要么扫全表、要么维护 order_id→UID 二级索引），多一次查询。基因法的动机是"让 order_id 自带 UID 路由信息"——分片键是 order_id，但 order_id 内嵌 UID 基因，按 UID 查时提取基因定位分片，按 order_id 查时直接用 order_id 路由。两种查询都单分片命中。代价是 order_id 生成逻辑复杂，且基因位数有限（UID 后 4 位只能区分 65536 个分片）。

### 第二层：证据与定位

**Q：客服反馈"按 order_id 查订单偶尔查不到，但 MySQL 里确实有这条订单"，你怎么定位是分片路由错，还是 ES/查询层的问题？**

按数据流逐层验证。三步：
1. 直接连 MySQL 查——算 order_id 的分片位置（基因法提取库表号），直连对应分片 `SELECT * FROM orders_X WHERE id=?`。如果 MySQL 有但接口查不到，是查询层（ES 或路由）问题。如果 MySQL 也没有，是写入问题（订单没落库或落错库）。
2. 如果 MySQL 有，查 ES——`GET /order/_doc/{orderId}`，如果 ES 没有这条，是 Canal 同步延迟或失败（查 Canal 日志和 outbox 投递状态）。如果 ES 有但接口查不到，是查询条件/权限问题。
3. 验证分片路由——用代码复算 `route(orderId)` 应该得到的库表号，和订单实际存储的库表号对比。如果不一致，是基因法实现 bug（如位运算错误导致基因提取错）。重点是按"MySQL → ES → 接口"逐层验证数据存在性，定位断点。

### 第三层：根因深挖

**Q：你发现是基因法的位运算有 bug——UID 后 4 位提取错误，导致部分 order_id 路由到错的分片，根因是什么？光是改位运算就行吗？**

改位运算只解决新订单，老订单已经落错库了。根因是"路由算法变更导致历史数据位置不一致"。治本：
1. 历史数据迁移——算出所有"路由错的订单"实际在哪个分片，建立"order_id → 实际分片"的映射表（纠错表）。查询时先查纠错表，命中则去实际分片查。或离线脚本把错位订单搬到正确分片（代价大，要停服或灰度）。
2. 防止再错——基因法的位运算写成独立函数，单元测试覆盖边界（UID=0、UID=最大值、orderId=边界值），且测试"生成 orderId → 提取基因 → 路由分片"的闭环一致。
3. 更深层根因——"分片路由逻辑分散在多处"（生成 orderId 时算基因、查询时提基因、扩容时重算），任何一处不一致就错乱。治本是"路由逻辑单一职责"，封装成 `ShardingService`，所有地方调同一个方法。根因是"算法不集中 + 测试不全"，改位运算只是表象。

**Q：那为什么不直接维护一个 order_id → 分片位置 的全局索引表，不用基因法，不是更直观吗？**

全局索引表的代价是"每次查询多一次索引查询"——按 order_id 查要先查索引表拿分片位置，再去对应分片查订单，RT 翻倍且索引表本身要高可用。基因法的优势是"路由信息内嵌在 key 里，零额外查询"——order_id 自带分片信息，路由是纯计算（位运算），无 IO。全局索引表还引入"索引表的一致性问题"（订单写了但索引没写、或反过来）。基因法适合"主查询维度明确（order_id/UID）"的场景，全局索引表适合"查询维度多样且无法预编码基因"的场景。拼多多订单的主查询维度（order_id/UID）明确，基因法更高效。如果查询维度极其多变（任意字段都可能查），用全局索引或 ES 兜底。

### 第四层：方案权衡

**Q：你的冷热分离是"90 天后归档到 HBase"，但用户查 1 年前的订单要去 HBase，HBase 查询比 MySQL 慢，体验差，你怎么权衡"存储成本"和"查询体验"？**

权衡方案是"分层查询 + 透明路由"：
1. 分层存储——热数据（90 天内）MySQL，P99 <20ms；温数据（90 天-2 年）HBase，P99 <100ms；冷数据（2 年+）OSS+索引，按需拉取。大多数用户查近期订单（热），秒回。
2. 透明路由——查询接口内部判断时间范围，自动路由到对应存储，用户不感知。查"1 年前"的请求自动走 HBase，用户体感"稍慢但能查到"（100ms 仍可接受）。
3. 体验优化——对 HBase 查询加 Redis 缓存（用户查过的历史订单缓存几分钟），重复查秒回。或前端引导"查看更多历史订单"用异步加载（先返回近期，历史异步查）。本质是"按查询频率分层存储，低频的容忍稍慢"，不是"所有数据都放最快存储"（成本爆炸）。

**Q：为什么不全部放 MySQL（不分冷热），现在存储便宜，多加机器就行？**

全部放 MySQL 的代价不是"存储成本"，而是"查询性能 + 运维成本"。MySQL 是 B+ 树索引，单表数据量越大，索引深度越深、查询越慢（即使有索引，千万级表 P99 也比百万级高）。且亿级订单全在 MySQL 热库，DB 的内存（buffer pool）装不下，频繁磁盘 IO，整体性能下降。冷数据（2 年前的订单）几乎不被查（访问频率 <0.01%），却占用热库的索引和 buffer pool 资源，拖慢热数据查询。冷热分离的核心是"让热库只装热数据，性能最优；冷数据用廉价存储（HBase/OSS）兜底"。存储便宜不代表"把所有数据放高性能存储"，内存和索引资源是瓶颈。

### 第五层：验证与沉淀

**Q：订单中心分库分表 + 冷热分离这么复杂，你怎么验证数据迁移（归档）不丢数据、查询路由正确？**

数据迁移是高危操作，必须有双重验证：
1. 归档一致性——每次归档任务执行后，对比"MySQL 删除的订单数 == HBase 新增的订单数"，且抽样逐条对比字段（`MD5(concat(all_fields))` 比对）。`archive_diff_count` 必须为 0。归档前先在 HBase 写入，验证写入成功后再删 MySQL（失败则回滚不删）。
2. 查询路由验证——压测覆盖各种时间范围的查询，断言"90 天内的查 MySQL、90 天前的查 HBase、跨界的合并结果无缺漏"。用 traceId 追踪每个查询实际访问的存储，验证路由正确。
3. 全量对账——定期（每周）跑全量对账：`MySQL 热表订单数 + HBase 冷表订单数 == 全局订单总数`，且无重复（order_id 唯一）。任何不一致告警。拼多多亿级订单，归档错了就是数据丢失，必须靠对账兜底。

**Q：订单中心怎么沉淀成平台，让新业务（如拼多多海外站）快速复用，而不是重新搭一套？**

沉淀成"订单中台"：
1. 存储与路由抽象——分库分表/冷热分离/基因法封装成基础设施，业务侧只定义"订单模型"和"查询模式"，路由和存储由中台处理。新业务接入不改底座。
2. 状态机引擎——订单状态流转用可配置的状态机（如 BPMN 或 DSL），不同业务（普通/拼团/海外）的状态图不同，但引擎复用。新业务配置状态图而非重写流转逻辑。
3. 事件总线——订单状态变更的事件（OrderCreated/Paid/Shipped）统一发布到 Kafka，下游业务（履约/营销/财务）订阅。新业务的事件自动接入生态。
4. 多租户隔离——中台支持多业务（国内/海外/新业务）的订单数据隔离（分库或分 namespace），但共享基础设施。这是大厂中台的核心价值：一次建设，多业务复用，避免重复造轮子。

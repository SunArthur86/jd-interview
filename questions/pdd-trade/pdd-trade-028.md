# 【拼多多交易】负载均衡/隔离/压测怎么做？

> JD 依据："高并发/高可用"。

## 一、多层负载均衡

```
DNS（地域） → SLB（机房） → Nginx（集群） → Gateway（路由） → Ribbon（实例）
```

| 层 | 算法 | 用途 |
|----|------|------|
| DNS | 轮询/地理位置 | 多机房分流 |
| SLB | 最小连接 | L4 转发 |
| Nginx | 轮询/IP hash | L7 负载 |
| Ribbon | 轮询/一致性 hash | 微服务实例选择 |

**一致性 hash**（会话保持）：
```java
// 同 UID 路由到同实例（本地缓存有效）
int hash = consistentHash(uid.hashCode(), instances);
Server target = instances.get(hash);
```

## 二、隔离（多维度）

**线程池隔离**（核心与非核心）：
```java
@Bean("createOrderPool")
public ThreadPoolExecutor createOrderPool() { ... }  // 核心创单

@Bean("commentPool")
public ThreadPoolExecutor commentPool() { ... }  // 评论，独立池
```
评论挂了不会耗尽创单线程。

**信号量隔离**（轻量，无队列）：
```java
@HystrixCommand(
    commandProperties = @HystrixProperty(
        name = "execution.isolation.strategy", value = "SEMAPHORE"))
public Result query() { ... }
```

**集群隔离**：核心交易集群 vs 营销/推荐集群，物理机分开。

**大促租户隔离**：大商户（苹果/小米）独立集群，防互相影响。

## 三、全链路压测

```
1. 影子库表：压测流量写到 shadow_order 表，不影响真实
2. 压测标透传：header X-Pressure=true 全链路识别
3. mock 外部：支付通道/物流 API 用影子账户
4. 容量预估：压到 RT 拐点 → 推算单机 QPS × 实例数 = 总容量
```

**压测脚本**：
```java
// 模拟双 11 峰值
PressureScenario scenario = PressureScenario.builder()
    .qps(100_000)               // 目标 QPS
    .duration(Duration.ofMinutes(30))
    .rampUp(Duration.ofMinutes(5))  // 爬坡
    .shadowTable(true)          // 影子表
    .build();
pressureEngine.run(scenario);
```

**容量模型**：
```
单机 QPS = 压测拐点 QPS × 安全系数 0.7
总容量 = 单机 QPS × 实例数
扩容目标 = 峰值 QPS / 单机 QPS + 冗余 30%
```

## 四、拼多多双 11 实战

- **LB**：DNS 地域→机房→Nginx 集群→Gateway
- **隔离**：交易/支付/库存核心集群独立（千台级），评论/推荐降级
- **压测**：双 11 前全链路压测（百万 QPS），影子库表隔离，演练降级预案

## 五、底层本质

LB/隔离/压测本质是**"均摊风险+划舱防沉+摸清家底"**——LB 让流量均匀，隔离让故障不传染，压测让容量可量化，是稳定性的预防工程。

## 常见考点
1. **一致性 hash 怎么避免数据倾斜**？——虚拟节点（每实例 150 虚拟节点）。
2. **线程池隔离缺点**？——上下文切换开销+队列内存，适合重 IO 不适合高频轻量。
3. **压测怎么不打扰真实用户**？——影子库表+独立压测集群+夜间低峰压。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你强调隔离（线程池/集群/物理）是稳定性关键，但隔离意味着资源不共享（每个隔离单元独立资源），这浪费机器，为什么还要隔离？**

隔离的动机是"故障不传染"。如果不隔离，评论服务（非核心）调用慢会占用创单服务（核心）的线程池/连接池，导致核心业务也慢——一个边缘服务的故障拖垮核心交易。拼多多这种规模，任何服务都可能出 bug 或被流量打爆，不隔离就是"一根绳上的蚂蚱"。资源浪费的代价是可控的（多备一些机器），但"核心交易被非核心拖垮"的代价是 GMV 损失（每分钟几百万）。隔离是把"核心"和"非核心"放在不同的"舱"（水密舱理念），非核心舱进水不沉核心舱。机器浪费换的是"核心业务的高可用"，值得。大促时甚至把大商户（如苹果旗舰店）独立集群，防互相影响。

### 第二层：证据与定位

**Q：线上发现核心交易集群的 CPU 突然飙高，但交易 QPS 没涨，你怎么定位是非核心流量泄漏到了核心集群，还是核心集群被别的拖累？**

按流量来源和调用链定位。三步：
1. 看核心集群的流量来源——APM 看核心交易服务的调用方，如果有非核心服务（如评论/推荐）在调核心交易接口（不合理），是流量泄漏（配置错误把非核心请求路由到核心集群）。
2. 看核心集群内各服务的 CPU——如果是交易服务自身 CPU 高但 QPS 没涨，可能是"GC 频繁"（jstat 看 FGC）或"死循环/大计算"（jstack 看热点线程）。如果有慢查询占 CPU，是 DB 问题传导。
3. 看隔离配置——Ribbon/Nginx 的路由规则是否正确隔离了核心和非核心。如果某非核心服务和核心服务注册到同一个 Ribbon 列表（没做集群标签隔离），调用方可能误调到核心实例。重点验证"隔离的边界是否真的隔离了"（流量染色 + 集群标签校验）。

### 第三层：根因深挖

**Q：你发现是线程池隔离没做好——评论服务和创单服务共用一个线程池，评论慢拖垮了创单，根因是什么？光是拆线程池就行吗？**

拆线程池是治标。根因是"隔离的粒度设计不合理"：
1. 共用线程池——历史代码可能为了省事用一个 `@Async` 默认池，评论和创单都往里塞。治本是按业务重要性拆池（创单独立池、评论独立池），且强制规范"所有 `@Async` 必须指定具名线程池，禁用默认池"。
2. 即使拆了池，如果下游（评论依赖的 DB/Redis）和创单共用，评论慢导致 DB 慢，创单查 DB 也慢。治本是"全链路隔离"——不仅线程池隔离，DB 连接池、Redis 实例也要按重要性分（核心交易用独立 DB 实例，非核心用另一个）。
3. 更深层——服务依赖的隔离。核心服务不依赖非核心服务（创单不该调评论）。如果创单同步调评论（不合理），评论慢拖垮创单，这要重构依赖关系。根因是"隔离只在某层（线程池）做了，其他层（连接池/DB/依赖关系）没隔离"，治本是"多层端到端隔离"。

**Q：那为什么不把所有服务都物理隔离（每服务独立机器集群），不就彻底隔离了？**

物理隔离（每服务独立机器）是彻底但浪费的方案。问题：
1. 资源浪费——非核心服务（如评论）平时流量低，独立集群的机器大部分时间闲着（利用率 <10%）。拼多多几百个微服务，全物理隔离要几倍机器。
2. 弹性差——物理隔离的集群各自容量固定，某服务突发流量时不能借用其他服务的空闲资源。共享集群（按优先级调度）更灵活。
3. 折中方案——"核心服务物理隔离（交易/支付独立集群），非核心服务共享集群（评论/推荐共享，但内部逻辑隔离）"。核心业务值得物理隔离保可用，非核心用逻辑隔离（线程池/信号量）省资源。拼多多交易核心是千台级独立集群，评论推荐共享百台级集群。这是成本和隔离性的权衡，不是全隔离或全共享。

### 第四层：方案权衡

**Q：全链路压测你用影子库表（压测流量写 shadow_order 表），但这要求所有服务都识别压测标并路由到影子表，改造成本高，你怎么权衡"压测真实性"和"改造成本"？**

影子库表的改造成本是"全链路透传压测标 + 各存储层路由影子表"，确实高。权衡方案是分级压测：
1. 单机/单服务压测——不需要影子表，直接压单个服务，用 mock 屏蔽下游。验证单服务容量，成本低。
2. 核心链路压测——只对交易核心链路（下单/支付）做影子表改造，非核心（评论/推荐）用 mock。改造范围缩小到核心，成本可控。
3. 全链路压测——大促前才做，影子库表 + 压测标全链路透传 + 影子账户（支付通道）。这是最真实但最贵的，一年做几次（大促前）。日常用单机/核心链路压测，大促用全链路。压测真实性 vs 改造成本的权衡是"按频次和重要性分级"，不是所有时候都上最重的全链路压测。

**Q：为什么不直接在生产压测真实流量（不用影子表），数据最真实？**

生产压真实流量会污染真实数据——压测的"假订单"进了真实订单表，影响对账、报表、用户视图。且压测流量打爆真实 DB 会导致真实用户受影响。影子表的意义是"压测数据物理隔离"（shadow_order 和 order 是两张表），压测完直接清空 shadow 表，不污染真实数据。生产压测必须用影子表/影子队列/影子账户，把"压测的副作用"隔离。真实流量压测的替代方案是"引流压测"（把真实用户流量复制一份到压测集群，用真实数据但隔离环境），但这也有数据安全风险（压测集群拿真实数据）。影子表是最安全的"生产环境模拟"，改造成本值得。

### 第五层：验证与沉淀

**Q：你怎么验证压测得到的容量模型是准的，而不是"压测时能扛 10 万 QPS，大促真实流量 10 万却崩了"？**

压测和真实流量的差异是容量模型失准的主因。验证：
1. 压测流量模型要"仿真"——不能只压"下单"单接口，要按真实流量分布（下单:查询:详情 = 1:50:100），否则压测没覆盖查询接口对共享资源（DB/Redis）的竞争，大促时查询拖垮核心。
2. 对比压测结果 vs 真实大促数据——每次大促后复盘"预估容量 vs 实际峰值 vs 实际表现"，修正容量模型（如发现"查询接口的 DB 压力被低估"，下次压测加权查询流量）。模型是迭代修正的，不是一次定终身。
3. 拐点定义——压测时"拐点"（RT 突然上升的 QPS）要明确，且容量预估用拐点 × 安全系数（0.7）。如果大促真实流量没到拐点却崩了，说明压测没覆盖某些瓶颈（如缓存击穿、GC 停顿），要补测。容量模型的准确性靠"压测 + 大促复盘"闭环验证。

**Q：负载均衡/隔离/压测怎么沉淀成持续的能力，而不是每次大促临时搞？**

沉淀成"稳定性工程平台"：
1. 容量管理平台——每个服务的容量（单机 QPS/总容量）自动采集（基于监控），容量模型自动维护（定期压测更新）。大促容量预估自动出（基于流量预测 × 容量模型）。
2. 隔离配置自动化——集群标签/线程池/路由规则的配置在平台管理，新服务上线自动按模板配隔离，不依赖人工。
3. 压测常态化——核心链路每周自动压测（回归容量），全链路压测每月一次（验证隔离/降级），大促前专项压测。压测是持续工程，不是一次性。
4. 故障演练平台——自动注入故障（kill 实例/慢下游/网络分区），验证隔离/限流/降级生效。稳定性是"练出来的"，不是"配出来的"。把稳定性从"大促救火"变成"日常工程"，这是大厂的核心能力。

# 【拼多多交易】支付路由怎么设计？

> JD 依据："交易系统技术升级"、"基础电商业务架构"。

## 一、支付链路

```
用户支付 → 网关 → 支付路由 → [微信/支付宝/银行/余额] → 回调 → 订单状态
                       ↑
                  路由决策（规则+模型）
```

## 二、路由决策

```java
public class PayRouter {
    // 通道实时成功率（滑动窗口 5 分钟）
    private Map<String, RateWindow> successRate;
    // 通道配置（费率/限额/配额）
    private Map<String, ChannelConfig> channels;

    public String route(PayRequest req) {
        List<String> candidates = channels.values().stream()
            .filter(c -> c.supports(req.getAmount(), req.getPayType()))
            .filter(c -> c.hasQuota(req.getUid()))
            .sorted(Comparator.comparingDouble(c -> score(c, req)))
            .map(ChannelConfig::getName)
            .collect(Collectors.toList());
        return candidates.get(0);  // 最优通道
    }

    // 综合评分：成本（低）+ 成功率（高）+ 当前负载（低）
    private double score(ChannelConfig c, PayRequest req) {
        double costScore = c.getFeeRate();                      // 越低越好
        double succScore = 1 - successRate.get(c.getName()).get(); // 失败率
        double loadScore = c.getCurrentQps() * 1.0 / c.getMaxQps();
        return costScore * 0.4 + succScore * 0.5 + loadScore * 0.1;
    }
}
```

## 三、实时成功率统计

```java
// 滑动窗口统计（每个通道）
class RateWindow {
    private Window<Long> success = new SlidingWindow(5, MINUTES);
    private Window<Long> total   = new SlidingWindow(5, MINUTES);
    public void record(boolean ok) { total.add(1L); if (ok) success.add(1L); }
    public double get() { return success.sum() * 1.0 / Math.max(1, total.sum()); }
}
```

成功率低于阈值（如 95%）自动降权或熔断切其他通道。

## 四、对账系统（T+1）

```
1. 下载各通道 T 日账单
2. 与本地支付流水比对
   ├─ 平台有/通道无 → 长款（可能通道延迟）
   └─ 通道有/平台无 → 短款（可能掉单）
3. 差错处理：补单/退款/人工介入
```

## 五、拼多多特色

- **多通道并存**：微信/支付宝/银联/苹果支付/余额
- **大额分期**：路由到支持分期的通道
- **下沉市场**：覆盖货到付款/线下代收
- **资金归集**：各通道资金 T+1 归集到主账户

## 六、底层本质

支付路由本质是**"多目标优化（成功率最高+成本最低+风险最小）"**——用规则保证基础正确，用模型动态学习，用对账兜底资金正确。

## 常见考点
1. **路由决策为什么放服务端**？——统一控制、可热切换、防客户端被破解。
2. **掉单（用户扣钱订单未支付）怎么处理**？——主动查询通道+对账兜底+退款。
3. **大促通道限额怎么办**？——预热配额+多通道分流+限流保护。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你的路由评分是 `cost*0.4 + failRate*0.5 + load*0.1`，为什么成功率权重 0.5 比成本 0.4 高？支付公司不是最看重成本吗？**

成功率优先于成本，因为"支付失败"的隐性成本远高于通道费率差异。一次支付失败，用户体验差（可能放弃购买）、客服投诉、订单流失（GMV 损失）。拼多多客单价假设 50 元，通道费率差 0.1% 是 0.05 元，但支付失败导致订单流失是 50 元 GMV 损失。所以"高成功率"比"低费率"价值大得多。权重 0.5 vs 0.4 是"成功率略优先但仍考虑成本"的权衡。极端情况下（某通道费率极低但成功率 80%），路由会选成功率高的而非便宜的。权重不是拍脑袋，是算"失败成本 vs 费率差"的经济模型。

### 第二层：证据与定位

**Q：线上整体支付成功率从 98% 掉到 92%，你怎么定位是某个通道拉低的，还是路由决策把流量分错了？**

按通道拆解成功率。三步：
1. 分通道看成功率——`success_rate by channel`，如果微信 97%、支付宝 96%、某银行通道 70%，明显是银行通道拉低。这时要看路由是不是把过多流量分给了银行通道（可能是它费率低被 score 偏向）。
2. 看路由的流量分配——`traffic_share by channel`，如果银行通道分到了 30% 流量（异常高），是路由决策问题。查 score 计算是否正确：银行通道的 `failRate` 是不是没实时更新（还显示历史高成功率），或 `cost` 权重让它胜出。
3. 看 `RateWindow` 的实时统计——如果银行通道的成功率统计窗口（5 分钟）样本太少（如只有 10 笔），统计不置信，路由可能误判。治本是"样本不足时用历史成功率兜底，而非实时"。重点是区分"通道真的变差"（通道问题）和"路由误判流量分配"（路由问题），两者解法不同。

### 第三层：根因深挖

**Q：你查到是某银行通道的实时成功率统计窗口样本太少（低峰期只有几笔），导致成功率虚高（偶然几笔都成功就显示 100%），路由误判分流过多，根因是什么？**

根因是"小样本统计不置信"。治本：
1. 样本下限——成功率统计要求最少 N 笔（如 100 笔）才置信，不足时用历史成功率（昨日全天）兜底。`RateWindow` 增加 `if total < 100 return historicalRate`。
2. 置信区间——不只看成功率点估计，看置信区间（如 Wilson 区间），95% 置信下限作为路由依据。10 笔全成功的下限可能只有 70%，不会盲目高分。
3. 探索与利用（Explore-Exploit）——对新通道或低样本通道，保留一定"探索流量"（如 5% 强制分配给它）积累样本，而非完全按 score。这叫多臂老虎机策略，平衡"用已知好通道"和"探索未知通道"。根因是"统计方法不严谨"，治本要用统计学的置信度而非拍脑袋阈值。

**Q：那为什么不直接固定一个最优通道（如永远用微信），简单且不用维护路由？**

单通道无法应对"通道故障 + 限额 + 成本优化"。微信支付也会故障（网络抖动/接口维护），单通道意味着它挂了全站支付瘫痪。大促时单通道有配额/限额（微信单商户日限额），超了支付失败。多通道还能按成本优化（A 通道费率 0.6%、B 通道 0.4%，分流省成本）。固定单通道是"把命脉交给一个第三方"，风险极高。路由的价值是"多通道冗余 + 智能选择"，既高可用又优成本。只有业务量极小（不值得接多通道）才固定单通道，拼多多这种规模必须多通道路由。

### 第四层：方案权衡

**Q：你的路由是主通道失败自动切备通道（重试），但这样用户可能被扣两次（主通道实际成功了但回包慢，路由切备又扣一次），你怎么权衡"高可用"和"防重复扣"？**

重试导致重复扣是支付路由的高危场景。权衡方案是"幂等键 + 查询前置"：
1. 全局幂等键——每笔支付用同一个 `outTradeNo`（订单维度唯一），切通道重试时用相同的 outTradeNo。通道侧（微信/支付宝）对同一 outTradeNo 只处理一次，第二次返回"已支付"。这保证"切通道不会真扣两次"。
2. 切换前查询——切备通道前，先查主通道状态（`orderQuery` 接口），确认主通道确实失败（而非"成功但回包慢"）才切。如果主通道显示已成功，不切，直接走主通道的成功流程。
3. 重试限制——最多切 1-2 次（不是无限重试），且每次切换间隔退避（避免连续打爆通道）。超过重试次数返回"支付失败请稍后重试"，让用户主动重试。本质是"用幂等保证不重复扣，用查询确认不误切"，重试是手段但不能盲目。

**Q：为什么不让用户自己选通道（前端展示微信/支付宝按钮），而不是服务端路由？用户选最直观。**

用户自选和服务端路由不冲突——拼多多的前端确实让用户选"微信/支付宝"，但服务端路由在"用户选的大类"内做精细化路由。比如用户选"支付宝"，支付宝背后可能有多个收单机构（费率/成功率不同），服务端路由选最优的收单机构，用户无感。另外，用户选错了（如选了不支持的分期通道），服务端要兜底路由到支持的。完全让用户选的问题是"用户不知道哪个通道成功率高/费率低"，且无法应对通道实时故障（用户选了挂掉的通道，支付失败）。服务端路由是"用户选大类 + 服务端选具体通道"的分层设计。

### 第五层：验证与沉淀

**Q：你怎么验证支付路由在各种通道故障下仍能高可用，而不是"平时最优，故障就乱"？**

必须有通道故障注入测试：
1. 模拟通道故障——用 Mock 或流量染色，让某通道的支付接口返回失败/超时，验证路由是否自动降级到备通道，且用户支付最终成功。`channel_failover_count` 应该增加，`final_success_rate` 应该保持 >95%。
2. 模拟限额——让某通道返回"超限额"，验证路由是否切到其他通道，且不重复扣款（幂等键验证）。
3. 对账验证——T+1 对账的 `reconcile_diff_count`，在故障注入测试期间应该为 0（切通道没有导致资金错乱）。重点测试"主通道成功但回包慢 + 路由切备"的边界，验证幂等键确实防止了重复扣。

**Q：支付路由怎么沉淀，让新接入的通道（如新增 Apple Pay）快速上线，且自动参与路由？**

沉淀成"支付路由平台"：
1. 通道接入标准化——每个通道实现统一接口（`pay/query/refund/notify`），接入 SDK 自动接入路由。通道的差异（签名/协议/限额）在适配层屏蔽。新通道接入只需实现接口 + 配置费率/限额。
2. 路由规则配置化——通道的权重/熔断阈值/配额通过配置中心动态下发，运营或系统自动调整，不改代码。新通道上线后自动参与路由（先小流量灰度验证成功率，达标后提权）。
3. 自动化监控——每个通道的成功率/费率/流量份额自动监控，异常（如成功率掉）自动降权或熔断，无需人工。对账差错自动归类（已知类型自动调账，未知类型告警人工）。把路由从"人工运维"变成"自治理平台"。

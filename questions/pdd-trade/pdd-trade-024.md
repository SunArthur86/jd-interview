# 【拼多多交易】对账系统怎么设计？

> JD 依据："交易系统技术升级"、"基础电商业务架构"。

## 一、对账流程

```
T+1 凌晨：
  1. 下载通道账单（微信/支付宝/银行 SFTP/API）
  2. 抽取平台流水（订单表/支付表）
  3. 双向比对（按订单号/支付号）
  4. 差错处理（自动/人工）
  5. 出对账报表
```

## 二、三方对账

```
平台订单 ──┐
           ├─→ 比对引擎 ──→ 差错表 ──→ 处理
支付流水 ──┤
通道账单 ──┘
```

| 类型 | 表现 | 可能原因 | 处理 |
|------|------|----------|------|
| 平账 | 三方一致 | 正常 | 无 |
| 长款 | 平台有/通道无 | 通道延迟/重复记账 | 挂账观察 |
| 短款 | 通道有/平台无 | 掉单 | 补单/查通道 |
| 金额错 | 金额不一致 | 退款/部分退款 | 退款流水核对 |

## 三、核心代码

```java
public class ReconcileJob {
    public void run(LocalDate day) {
        // 1. 加载三方数据
        List<Order> orders = orderDao.listByDay(day);
        List<PayRecord> pays = payDao.listByDay(day);
        List<ChannelBill> bills = channelDao.listByDay(day);

        // 2. 按支付号 join
        Map<String, PayRecord> payMap = pays.stream()
            .collect(toMap(PayRecord::getTradeNo, p -> p));

        for (ChannelBill bill : bills) {
            PayRecord pay = payMap.get(bill.getTradeNo());
            if (pay == null) {
                saveError(SHORT, bill);  // 短款
            } else if (!pay.getAmount().equals(bill.getAmount())) {
                saveError(AMOUNT_MISMATCH, bill);
            } else {
                markMatched(pay);        // 平账
            }
        }
        // 3. 平台有通道无（长款）
        for (PayRecord pay : pays) {
            if (!bills.contains(pay.getTradeNo())) {
                saveError(LONG, pay);
            }
        }
        // 4. 自动平账规则
        autoResolve();
    }
}
```

## 四、自动平账规则

```
长款 < 1 元：自动忽略（噪声）
长款 1-100 元：挂账 3 天，仍无通道记录则退款
短款：立即查通道 API 确认，确认后补单
金额错：触发退款流水核对
```

## 五、拼多多特色

- **多通道对账**：微信/支付宝/银联/Apple Pay 各有账单格式，统一适配器
- **退款对账**：退款和正向订单分开对（避免混淆）
- **商户分账**：平台抽佣+商户结算，三方（平台/商户/通道）核对
- **大促小时对**：双 11 改成每小时增量对账，T+1 全量兜底

## 六、底层本质

对账本质是**"用离线全量比对兜底实时一致性"**——实时可能有掉单/重复，离线 T+1 全量核对保证资金最终正确。这是分布式系统 CAP 之外的"业务级最终一致"保障。

## 常见考点
1. **大促对账怎么做**？——每小时增量对账+T+1 全量，分布式任务（ElasticJob）扛规模。
2. **对账数据量大怎么办**？——按通道/时间分片并行+Spark 大数据对账。
3. **差错处理 SLA**？——长款挂账 3 天/短款实时告警人工介入/金额差错 1 小时内确认。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：对账是 T+1 离线做的，但资金问题（如掉单）等到第二天才发现，用户已经投诉了，为什么不实时对账？**

实时对账做不到"全量可靠"。实时链路的问题是"数据可能还没到齐"——通道账单是 T+1 才出（微信/支付宝的账单是次日提供），实时拿不到通道侧的权威数据；且实时对账要在支付链路里做比对，增加 RT 和复杂度。T+1 的动机是"等所有数据落齐后做全量权威比对"——通道账单次日到，平台流水也稳定了，双向比对能发现所有差异。为了提前发现问题，拼多多在大促期间改成"每小时增量对账"（用通道的实时查询 API 代替账单，虽然不全量但能抓大促期间的高频问题），T+1 仍是全量兜底。实时和离线结合，不是非此即彼。

### 第二层：证据与定位

**Q：对账系统跑完后，发现 `reconcile_diff_count`（差错数）从平时的几十飙升到几千，你怎么定位是通道账单有问题，还是平台流水有问题，还是对账逻辑 bug？**

按差错类型拆解。三步：
1. 看差错类型分布——长款（平台有通道无）/短款（通道有平台无）/金额错的比例。如果绝大多数是"长款"，可能是通道账单延迟（账单还没出全，平台记录通道侧没有）。等几小时或拿次日补充账单重跑，看是否消失。
2. 抽样核对——挑几条差错，人工查平台 DB 和通道的商户后台，确认哪边的数据是对的。如果平台有支付流水但通道后台查不到这笔，是"平台多记账"（可能是测试数据或 bug）；如果通道有但平台没有，是"掉单"（支付回调丢失）。
3. 对账逻辑验证——如果差错集中在某个通道或某个时间段（如某小时集中出错），可能是对账代码处理那个通道/时段的 bug（如时区转换、金额单位、字段映射）。查最近对账代码的改动，`git log` 看是否改过那个通道的适配器。重点是"抽样核对真实数据"而非只看代码，数据会说话。

### 第三层：根因深挖

**Q：你抽样发现大部分差错是"短款"（通道有支付记录但平台没有），即掉单，根因是什么？光是补单就行吗？**

补单是治标。根因是"支付回调丢失导致平台没记录这笔支付"。深挖：
1. 查掉单的回调日志——这些订单的支付回调是否到达平台。如果回调根本没来（通道侧没发或网络丢包），是通道问题或网络问题。平台应该有"主动查询"机制（定时用 orderQuery 查通道，补充漏掉的回调）。
2. 如果回调来了但处理失败——查 `pay_notify` 日志，看这些回调是不是抛异常了（如 DB 写入失败、幂等校验逻辑 bug）。治本是修处理逻辑 + 回调重试。
3. 量化掉单率——`notify_loss_rate`（回调丢失率），如果持续 >0.01%，是系统性问题。拼多多量级下 0.01% 就是每天上百笔掉单，必须靠"主动查询补偿"+"T+1 对账"双保险。根因可能是"只依赖回调，没有主动查询兜底"，治本是补上主动查询任务。

**Q：那为什么不直接每次支付都同步查通道确认（而非等回调），这样就不会掉单？**

同步查通道会拖垮支付链路。每次支付成功后同步调通道 `orderQuery` 确认，增加几百 ms RT，且通道的查询 API 有 QPS 限制（不可能支撑每笔支付都查）。正确的架构是"回调为主 + 定时主动查询为辅 + T+1 对账兜底"三层防护：
1. 回调——实时性最好，但可能丢失（网络），是主路径。
2. 定时主动查询——每分钟扫"支付中超过 N 分钟未回调"的订单，主动查通道补结果。覆盖回调丢失，延迟几分钟。
3. T+1 对账——全量兜底，覆盖前两层漏掉的。三层叠加，掉单率趋近 0。全靠同步查询既慢又压通道，分层防护是性能和可靠性的平衡。

### 第四层：方案权衡

**Q：你的自动平账规则是"长款 <1 元自动忽略"，但 1 元的阈值怎么定？定高了会漏真实差错，定低了噪声多，你怎么权衡？**

阈值取决于"业务影响 + 噪声成本"。权衡依据：
1. 业务影响——长款 <1 元（如 0.01 元的利息/手续费尾差）通常是通道结算精度差异（如通道按分结算，平台按厘记账），不是真实差错，忽略无资损。但 >1 元的长款可能是真实问题（重复记账/通道延迟），要挂账观察。
2. 噪声成本——阈值太低（如 0.01 元），每天几万条噪声差错，人工核查成本高。阈值太高（如 100 元），可能漏掉累积的小额差错（如某 bug 每笔多记 0.5 元，单笔忽略但每天几千笔累积是大问题）。
3. 动态调整——不只看单笔阈值，还要监控"小额差错的累积"。如果某天小额差错总数异常（如平时 100 条今天 1 万条），即使单笔都 <1 元，也要告警（可能是系统性 bug）。阈值是"单笔忽略 + 累积监控"的组合，不是单维度。

**Q：为什么不所有差错都人工处理，而要自动平账？人工最准确。**

拼多多日均亿级订单，即使万分之一差错率也是每天上万条，全人工核查不现实。自动平账的价值是"把已知类型的差错用规则批量处理"，人工只处理"未知/复杂"的。比如"通道延迟的长款"（次日补充账单后消失）是已知类型，自动挂账 3 天自动平；"重复扣款"是已知类型，自动退款。人工只看"金额错且核对不上""短款且通道查无记录"这种异常。权衡是"自动处理高置信度差错（省人力）+ 人工处理低置信度差错（保准确）"，自动规则的阈值和类型要经过财务确认，不能开发拍脑袋。

### 第五层：验证与沉淀

**Q：对账系统本身怎么验证正确性？对账系统算错了，差错就掩盖了，怎么发现？**

对账系统是"审计的审计"，必须有元验证：
1. 注入测试——构造已知差异的数据（如人工造一笔平台有通道无的长款），验证对账系统能检出并归类正确。这是"对账系统本身的单元测试"。
2. 平衡性校验——对账跑完后，`平台流水总额 == 通道账单总额 + 差错净额`。如果不平衡，说明对账漏了某些记录（对账逻辑 bug）。这个恒等式是元校验。
3. 交叉对账——定期用两套对账逻辑（如 Spark 和 SQL 两种实现）跑同一批数据，对比结果差异。如果两套结果不一致，至少一套有 bug。或抽样人工核对自动对账的结果，`audit_error_rate`（人工发现对账系统漏判的比例）要趋近 0。

**Q：对账系统怎么沉淀，让新通道（如新增银行）快速接入对账？**

沉淀成"对账中台"：
1. 通道适配器标准化——每个通道实现统一接口（`downloadBill/parseBill/queryOrder`），账单格式差异在适配器屏蔽。新通道接入只需写适配器 + 配置对账规则。
2. 对账规则配置化——差错类型/自动平账阈值/SLA 通过配置，不同通道可不同规则，不改代码。财务可调阈值。
3. 统一差错平台——所有通道的差错统一进差错表，统一大盘监控（差错率/未平账金额/SLA 达标率），自动告警。运维不分散在各通道。
4. 退款对账分离——正向和退款分开对账（逻辑不同），新通道接入时明确"正向对账 + 退款对账"两套规则。对账中台让"接新通道"从"开发一个月"变成"配置一周"，这是中台对业务的支撑价值。

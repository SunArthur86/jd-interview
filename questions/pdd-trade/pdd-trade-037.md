# 【拼多多交易】AI Infra 怎么搭？

> JD 依据："AI Infra"。

## 一、整体架构

```
业务方（交易/客服/搜索/推荐）
    ↓ 统一 API
AI 网关（路由/鉴权/限流/计费）
    ↓
┌─────────────────────────────────┐
│ 推理平台          训练平台       │
│ vLLM/Triton       任务队列       │
│ 自动扩缩           资源隔离       │
└─────────────────────────────────┘
    ↓
GPU 池（H100/A100 集群）
    ↓ 调度
K8s + Volcano/KubeFlow
```

## 二、GPU 池化与调度

**池化**：
```
物理 GPU → MIG 切片（1 卡切 7 实例，小模型用）
        → 任务级复用（推理完切训练）
统一池 → 按业务优先级分配
```

**优先级调度**（Volcano）：
```yaml
# 在线推理（高优先级）
scheduling:
  priorityClassName: high-priority
  queue: inference
# 离线训练（低优先级，可抢占）
scheduling:
  priorityClassName: low-priority
  queue: training
  preemptionPolicy: PreemptLowerPriority
```

**分时**：
```
白天 9-23：80% GPU 给推理（大促 100%）
夜间 23-9：GPU 给训练（微调/对齐）
```

## 三、推理平台

```yaml
# 模型服务（KServe/vLLM）
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: trade-llm
spec:
  predictor:
    minReplicas: 2
    maxReplicas: 50
    scaleTarget: 100    # QPS 触发扩容
    containers:
      - image: vllm:v0.6
        args: ["--model=qwen2-7b", "--tensor-parallel-size=2"]
        resources:
          limits: {nvidia.com/gpu: 2}
```

**自动扩缩**：
```
QPS 涨 → HPA 扩副本
GPU 利用率低 → 缩容（成本）
大促预热 → 提前手动扩
```

## 四、训练平台

```
任务队列（训练/微调/评测）→ 资源调度 → GPU 执行 → 日志/指标
```

**微调流水线**：
```python
@pipeline
def finetune_pipeline(dataset, base_model):
    # 1. 数据预处理
    train_data = preprocess(dataset)
    # 2. LoRA 微调
    adapter = lora_finetune(base_model, train_data)
    # 3. 评测
    score = evaluate(adapter, eval_set)
    if score < threshold:
        return reject()
    # 4. 合并+部署
    merged = merge_adapter(base_model, adapter)
    deploy(merged, stage="canary")
```

## 五、模型仓库

```
模型中心（MLflow）：
  - 版本化（v1/v2/canary）
  - 元数据（训练数据/指标/prompt）
  - 灰度发布（5%→20%→100%）
  - 一键回滚
```

## 六、计费与监控

```
按 token 计费：业务调用多少 token 算多少钱
按 GPU 时：训练任务按卡时
监控：
  GPU 利用率（目标 > 70%）
  推理 RT/吞吐
  业务成本归因（哪个接口花多少）
```

## 七、拼多多实战

- **GPU 集群**：千卡级 H100，推理/训练分池
- **弹性**：大促白天推理扩到 80%，夜间切训练
- **多模型路由**：简单走 1.5B（便宜），复杂走 14B（贵但准）
- **成本看板**：每个 AI 接口的 GPU 成本，业务方按需优化

## 八、底层本质

AI Infra 本质是**"把 GPU 当云资源来管"**——池化提利用率，调度保公平，平台化降低业务门槛，弹性降成本。

## 常见考点
1. **GPU 利用率怎么提**？——池化+分时+MIG 切片+Continuous Batching。
2. **推理扩容有什么坑**？——模型加载慢（分钟级），需预热+流量预热。
3. **训练和推理冲突怎么办**？——分池+优先级抢占+Volcano 调度。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：AI Infra 做的是"GPU 池化 + 调度 + 平台化"，但每个业务线自己买 GPU 跑自己的模型，不是更简单、不用等调度吗？为什么要统一池化？**

自建的根本问题是"GPU 利用率低 + 成本爆炸"。原因：
1. 潮汐效应——客服 LLM 白天 QPS 高、夜间几乎零，自建的话夜间 GPU 闲置（利用率 < 10%）。训练任务夜间跑、白天闲，自建白天 GPU 闲置。池化后白天 GPU 给推理、夜间给训练，同一批 GPU 24 小时满载，利用率从 20% 提到 70%+。
2. 弹性诉求——大促时推理 QPS 涨 10 倍，自建要常备峰值 GPU（平时浪费），池化可按需抢占低优先级资源。统一池化的动机是"用共享 + 调度把 GPU 利用率拉满，降总成本"，自建看似简单实际是资源浪费。GPU 单卡数万美元，千卡集群池化省的是亿元级成本。

### 第二层：证据与定位

**Q：大促前业务方反馈"推理 RT 飙到 10 秒"，你怎么定位是 GPU 不够、调度问题，还是推理引擎配置问题？**

按推理链路拆证据。三步：
1. 看 GPU 池水位——`nvidia-smi` + K8s metrics 看 GPU 利用率和显存。如果利用率 100%、显存快满，是容量不够（要扩容或抢占训练资源）；如果利用率低（< 60%）但 RT 高，是调度或引擎问题（如 batch 没拼满、请求排队）。
2. 看调度日志——Volcano/K8s 看推理 pod 是否被训练任务抢占、是否有 pending pod（资源不足没调度上）。如果有 pending，是资源分配问题；如果 pod 都 running 但 RT 高，是引擎层。
3. 看推理引擎 metrics——vLLM 的 `num_requests_waiting`（排队）、`time_to_first_token`、`time_per_output_token`。如果排队多，是 QPS 超过单 pod 容量（扩副本）；如果单 token 生成慢，是 KV Cache 或显存压力。三组证据定位"容量/调度/引擎"哪层瓶颈。

### 第三层：根因深挖

**Q：你定位到大促 RT 飙升的根因是"扩容时模型加载慢"——HPA 触发扩副本，但新 pod 加载 14B 模型要 3 分钟，期间请求堆积。根因是 HPA 不适合 LLM 推理，还是更深层？**

根因是"LLM 推理扩容的冷启动"和传统微服务不同。深挖：
1. 模型加载是分钟级——14B 模型几十 GB，从镜像仓库拉到 GPU 显存要 1-3 分钟，期间 pod 是 running 但不能服务，HPA 按 QPS 扩容但新 pod 还没 ready，请求继续堆积。
2. HPA 机制滞后——HPA 按 QPS 触发扩容是"已过载才扩"，对突发流量（如大促零点）必然滞后。根因是"被动扩容 + 长冷启动"的双重延迟。
3. 治本——预热池（常备几个已加载模型的 warm pod，扩容时直接接入流量，省加载时间）+ 主动扩容（大促零点前按预案提前扩到峰值，不等 HPA 触发）+ 模型镜像预拉（镜像提前分发到节点，省拉取时间）。根因是"LLM 推理扩容要主动 + 预热"，不能依赖传统微服务的被动 HPA。

**Q：那为什么不常备峰值 GPU（如大促期间一直开 50 个 pod），不就没有冷启动问题了吗？**

常备峰值是成本爆炸。大促峰值要 50 个 pod，平时只要 5 个，常备 50 意味着 45 个 GPU 平时空闲（利用率 < 10%）。GPU 单卡数万美元，千卡集群这种浪费是亿元级。正确做法是"分级预热"：
1. 常备基线（如 5 pod）+ warm 池（如额外 10 warm pod，已加载模型待命）——warm 池覆盖突发，成本可控（10 个 GPU 的闲置成本远低于 50 个）。
2. 大促按预案主动扩——提前 1 小时扩到预估峰值，不靠实时 HPA。
3. 分时复用——非大促期间，warm 池的 GPU 切给训练（低优先级），不浪费。常备峰值是"用钱换稳定但不经济"，分级预热 + 分时复用是"低成本保稳定"的平衡。

### 第四层：方案权衡

**Q：推理和训练共享 GPU 池靠优先级抢占（推理高、训练低），但训练被抢占会丢 checkpoint（训练进度），这个权衡怎么处理？**

抢占丢进度是真问题。权衡方案：
1. 高频 checkpoint——训练任务每 N 步存 checkpoint（如每 500 步），被抢占时从最近 checkpoint 恢复，最多丢 500 步（几分钟），而非从头来。这是用存储成本换恢复速度。
2. 优雅抢占——调度器通知训练任务"将被抢占"，训练 pod 有 60 秒优雅期存 checkpoint 再退出，而非直接 kill。Volcano 支持这种 graceful preemption。
3. 抢占阈值——不是推理一缺资源就抢训练，而是推理资源低于阈值（如 20% 余量）才触发，避免频繁抢占。权衡是"训练的可恢复性 + 推理的优先级"，通过 checkpoint + 优雅退出把抢占的代价降到可接受。完全不抢占（分池）会降利用率，无脑抢占会浪费训练，这个机制是平衡点。

**Q：为什么不直接物理分池（推理一批 GPU、训练一批 GPU，互不干扰），不就没有抢占问题了吗？**

物理分池的代价是利用率低。推理和训练的潮汐正好相反（推理白天忙、训练夜间忙），物理分池意味着两组 GPU 各自有闲置时段，整体利用率降 30-40%。共享池通过分时复用把利用率拉到 70%+，省的是亿元级 GPU 成本。实际是"部分分池 + 部分共享"的混合：
1. 核心推理（交易/客服，SLA 严）——专用池，不被抢占，保证稳定。
2. 弹性推理（非实时，如批量生成）+ 训练——共享池，分时 + 优先级抢占。这样核心稳定、弹性高利用率。纯物理分池是简单但浪费，纯共享是利用率高但抢占痛，混合是工程最优解。

### 第五层：验证与沉淀

**Q：你怎么证明 AI Infra 的池化真的降本了，而不是把成本藏到"共享池"里算不清？**

建立成本归因体系：
1. 按 token 计费——每次推理调用记 token 数 × 单价，精确到业务接口（如"客服接口用了多少 token = 多少 GPU 时"），业务方看到自己的成本。
2. 按 GPU 时分摊——训练任务按占用卡时分摊，推理按 QPS × 单次 GPU 时分摊。共享池的成本按各业务实际用量分摊，不藏在"公共池"里。
3. 对比基线——池化前（各业务自建）的总 GPU 成本 vs 池化后的总成本 + 平台运维成本，净降多少。关键是"分摊透明"——每个业务能看到自己用了多少、花了多少，而不是一笔糊涂账。池化的降本要可量化（如"GPU 利用率从 25% 提到 72%，单位 token 成本降 60%"），不是口号。

**Q：AI Infra 怎么沉淀成团队能力，而不是每次新模型上线都靠平台组手工支持？**

沉淀成"自助式 AI 平台"：
1. 自助上模型——业务方通过平台 portal 上传模型（或选模型仓库的），配并发/SLA，平台自动部署（vLLM + HPA + 监控），不需要平台组介入。
2. 标准推理镜像——vLLM/Triton + 监控 + 计费 agent 封装成标准镜像，业务声明模型和参数即可，不碰底层。
3. 自助训练 pipeline——数据上传 → 选基座模型 → LoRA 微调 → 评测 → 部署，全流程平台化，业务自助跑。
4. 监控/计费看板——每个业务看自己的 RT/QPS/GPU 利用率/成本，异常自愈（如自动扩容）。
5. 文档 + onboarding——新业务接入有标准流程和文档，不靠平台组手把手。把 AI Infra 从"平台组手工支持"变"业务自助"，这是规模化 AI 的必然路径。

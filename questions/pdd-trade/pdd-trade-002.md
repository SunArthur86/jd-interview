# 【拼多多交易】支付系统怎么设计？怎么保证资金安全？

> JD 依据："交易系统核心功能开发"。

## 一、支付全流程

```
下单 → 生成支付单 → 调第三方支付 → 异步回调 → 更新订单 → 清分 → 结算
                                         ↓
                                    幂等处理（防重复）
```

## 二、支付编排

```java
public PayResult pay(PayReq req) {
    // 1. 幂等校验（防重复支付）
    if (payOrderDao.exists(req.getOutTradeNo())) {
        return payOrderDao.getByOutTradeNo(req.getOutTradeNo()).toResult();
    }
    // 2. 创建支付单
    PayOrder order = createPayOrder(req);
    // 3. 调第三方（微信/支付宝）
    ThirdPayResult third = thirdPayService.unifiedOrder(order);
    // 4. 返回支付参数（前端拉起支付）
    return PayResult.of(third.getPayUrl());
}
```

## 三、回调幂等（关键）

第三方支付机构可能多次回调同一笔支付：
```java
public void onNotify(PayNotify notify) {
    // 唯一索引保证幂等
    PayOrder order = payOrderDao.lockByTradeNo(notify.getTradeNo());
    if (order.status == PAID) return;  // 已处理，幂等返回

    order.status = PAID;
    outboxDao.insert(new Outbox("PaySuccessEvent", order));  // 同事务
}
```

## 四、异步清结算

支付成功后异步分账（拼多多平台/商家/服务商）：
```
PaySuccessEvent → 清分服务
   ├─ 商家账户 += 商品金额 - 佣金
   ├─ 平台账户 += 佣金
   └─ 服务商账户 += 分润
```

## 五、T+1 对账

```
拼多多支付流水 vs 微信/支付宝账单
   ↓ 比对
一致 → 平账
差异（长款/短款）→ 人工核查
```

## 六、底层本质

支付系统本质是**"资金流的强一致管理"**：
- 强一致：支付成功/失败明确（不能中间态）
- 幂等：重复回调不重复扣
- 对账：多源交叉验证兜底

这是金融级系统的最高要求——容错不容灾（宁可降级不可错账）。

## 常见考点
1. **支付回调为什么可能重复**？——网络重试、第三方兜底重发，必须幂等。
2. **支付和订单怎么一致**？——本地消息表（支付成功事件同事务）+ 订单订阅。
3. **对账差异怎么处理**？——自动调账规则（已知类型）+ 人工核查（复杂）。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你给支付回调加了幂等，但为什么不干脆让第三方支付机构别重发？**

因为第三方重发是兜底机制，不是 bug。微信/支付宝的回调协议就是"如果你不回 SUCCESS，我会按 15s/30s/1min/5min/30min 的退避策略重发最多 8 次"，这是为了应对我们自己的服务网络抖动、重启、GC 停顿——如果哪天我们回调处理挂了，没有重发机制，这笔钱就成了"用户扣了款但订单没更新"的资损。所以幂等不是防御第三方的 bug，是配合第三方的容错设计。我们必须保证"同一笔支付被回调 N 次，只处理一次"。

### 第二层：证据与定位

**Q：监控发现 `pay_callback_duplicate_rate`（回调重复率）从平时的 0.1% 飙到 5%，你怎么定位是正常的第三方重发，还是我们自己的回调处理变慢了？**

看回调处理耗时和重复回调的时间间隔。三步：
1. 看 APM 里 `onNotify` 接口的 P99——如果从 50ms 飙到 3s，说明我们处理慢了，导致第三方超时重发。重点查 `payOrderDao.lockByTradeNo` 是不是行锁等待（看 DB 慢日志的 `lock_wait_time`）。
2. 看重复回调的时间分布——如果是同一批 tradeNo 在很短时间内密集重发（比如 15s 内 5 次），且我们的处理耗时正常，那是第三方那边的问题（比如它没收到我们的 SUCCESS 回包），查我们回包的 HTTP 状态码是不是都是 200。
3. 看是否有单笔大额支付集中回调——大促期间本身就是回调高峰，重复率会自然上升，这种是容量问题不是 bug。重点是区分"处理慢导致重发"和"回包丢失导致重发"。

### 第三层：根因深挖

**Q：假设你定位到是 `lockByTradeNo` 行锁等待导致回调处理慢，根因是什么？光是加索引就能解决吗？**

不一定是索引问题。`trade_no` 上有唯一索引，单行锁等待大概率是"同一笔支付的多个回调并发到达，都在等同一把行锁"。这是正常的——第三方重发的多个回调可能在很短时间内同时打过来。根因是锁的粒度和持有时间：
1. 持有时间过长——锁住之后还做了 `outboxDao.insert`、发 MQ、RPC 通知，把整个事务撑到几百 ms。
2. 解法是把锁内只做"状态判断 + 状态变更 + outbox 插入"（同事务，快），把发 MQ、RPC 通知挪到事务外（outbox 投递由定时任务异步做），锁持有时间从 300ms 降到 20ms。

**Q：那为什么不直接用 Redis 分布式锁代替 DB 行锁，性能不是更好吗？**

因为支付幂等的正确性必须以 DB 状态为准，Redis 锁只是"门禁"，真正的"已处理"判断还是看 `order.status == PAID`。如果用 Redis 锁，会出现锁过期（TTL 到了）但事务还没提交，下一个回调拿到锁、查 DB 还是 PENDING、又处理一遍——双重扣款。DB 行锁虽然慢，但它和状态变更在同一个事务里，天然原子。支付场景宁可慢不能错，Redis 锁适合"防并发重复计算"（如限流），不适合"防资金重复扣"。

### 第四层：方案权衡

**Q：你的幂等是靠 `trade_no` 唯一索引 + 状态机校验实现的，但有些场景第三方回包的 trade_no 可能变（比如关单重支付），你怎么权衡？**

幂等键的选择是支付系统的核心设计。`trade_no`（第三方流水号）做幂等键有个前提：第三方保证同一个 outTradeNo 的 trade_no 不变。但实际上微信会在"关单后重新支付"时换一个新的 prepay_id 和 trade_no，这时如果还用 trade_no 做幂等键，就会把两笔支付都处理成成功。权衡方案是双键幂等：业务幂等键用 `outTradeNo`（我们的下单流水号），第三方幂等键用 `trade_no`。两个都建唯一索引，处理前先查 `outTradeNo` 对应的订单状态——如果订单已经是 PAID 且 trade_no 不匹配，说明是关单重支付，应该拒绝（提示用户已支付）。绝不能两个 trade_no 都置成功。

**Q：为什么不在支付回调里直接同步清分结算，而要异步？同步不是更不容易出错吗？**

同步清分会让回调处理时间从 50ms 飙到秒级（清分要算多商家分账、写多个账户表），第三方等不及会重发，反而增加重复率。异步的核心是"把强一致的支付结果和最终一致的清结算解耦"——支付回调只负责"确认钱到了"，清结算在 `PaySuccessEvent` 后异步跑（通常秒级到分钟级）。出错的风险用 T+1 对账兜底。如果同步清分，一笔支付的失败会拖累整个回调链路，且清分本身的复杂逻辑（多账户事务）会拖垮支付成功率。

### 第五层：验证与沉淀

**Q：你怎么验证支付幂等真的有效，而不是"没出事是因为运气好"？**

必须有故障注入测试。写一个压测脚本：对同一笔 outTradeNo，并发发起 100 次回调（模拟第三方重发），断言最终 `pay_order` 表只有一条 PAID 记录、`account_flow` 只有一条扣款流水、`outbox` 只有一条事件。再在生产上跑对账验证：每天 T+1 对账的 `reconcile_diff_count`（对账差错数）如果是 0，说明没有重复扣或多扣。拼多多量级下，哪怕 0.001% 的幂等失效也是每天上百笔资损，必须靠对账数字兜底，不能靠"没投诉"。

**Q：怎么沉淀这套支付幂等的设计，让新接手的支付通道（比如新增银联）不踩坑？**

沉淀成支付接入规范而不是靠口口相传：
1. 幂等 SDK——把"查状态 + 唯一索引 + outbox 事务"封装成 `@IdempotentCallback` 注解，新通道接入只要打注解，不用自己写幂等逻辑。
2. 接入 checklist——每个新通道上线前必须过：幂等键是什么？回包格式？重发策略？对账文件格式？由支付域 owner review。
3. 对账规则库——每个通道的对账差异类型（长款/短款/时间差/金额差）沉淀成自动调账规则，新通道的已知差异自动处理，只有未知类型才人工。这样人力只花在新问题上。

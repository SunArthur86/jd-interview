# 【拼多多交易】AI Harness 怎么工程化？

> JD 依据："AI Harness 工程化"。

## 一、Harness 整体

```
开发 → Prompt 管理（版本）→ 离线评估（标注集）→ 灰度上线 → 监控 → 回滚
                              ↑                                    ↓
                              └────── 反馈数据回流 ←────────────────┘
```

## 二、Prompt 管理

**模板化+版本化**：
```yaml
# prompts/refund_v3.yaml
name: refund_advisor
version: v3
template: |
  你是拼多多退款助手。根据以下规则判断是否可退款：
  {{policy_docs}}
  用户订单：{{order}}
  用户问题：{{question}}
  只回答"可退/不可退/转人工"+理由，引用规则编号。
variables: [policy_docs, order, question]
```

```java
Prompt prompt = promptManager.get("refund_advisor", "v3");
String rendered = prompt.render(Map.of(
    "policy_docs", rag.retrieve("refund"),
    "order", order,
    "question", q
));
```

## 三、评估体系

**离线评估**（标注集）：
```python
eval_set = load_eval_set("refund_1000.json")  # 1000 标注样本
results = []
for case in eval_set:
    pred = agent.run(case.input)
    results.append({
        "accuracy": llm_judge(pred, case.expected),  # LLM-as-judge
        "latency": pred.latency,
        "cost": pred.tokens * PRICE
    })
print(f"准确率: {mean(r.accuracy for r in results):.2%}")
```

**在线 A/B**：
```
新版 prompt 灰度 10% 流量：
  对照组（v3）：满意度 85%、转人工率 15%
  实验组（v4）：满意度 88%、转人工率 12%
显著提升 → 全量；下降 → 回滚
```

**业务指标**：
- 客服：满意度、转人工率、首问解决率
- 退款：自动通过率、申诉率、错退率

## 四、监控告警

```
指标：
  RT（p99 < 3s）
  成本（单次 < 0.1 元）
  幻觉率（人工抽检 < 1%）
  调用失败率（< 1%）
  用户负反馈率（< 5%）

异常 → 自动切稳定版+告警
```

## 五、LLMOps 流程

```
1. 改 prompt → 单测（标注集）→ 代码评审
2. 预发评估（影子流量）→ 离线报告
3. 灰度上线（5%→20%→100%）
4. 监控异常 → 一键回滚
5. 反馈数据回流 → 迭代下一版
```

## 六、拼多多实战

- **Prompt 仓库**：200+ prompt 版本化，团队协作
- **评估平台**：客服标注集 10 万+，每周回归
- **灰度系统**：LLM 应用像业务一样灰度
- **成本看板**：每天百万次调用，成本精确到接口

## 七、底层本质

AI Harness 本质是**"把 LLM 应用当软件工程来管"**——prompt 是代码（版本化），评估是测试（标注集），灰度是发布，监控是运维，让"玄学调 prompt"变"工程化迭代"。

## 常见考点
1. **LLM-as-judge 可靠吗**？——比人工便宜但有限，关键场景仍需人工，且需校准（与人工一致性）。
2. **prompt 改动怎么评估**？——离线标注集回归+在线 A/B 双验证。
3. **LLM 成本怎么控**？——模型分级（简单用小模型）+缓存（同问题直接返回）+token 优化。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说 AI Harness 要做 prompt 版本化管理、灰度、监控、回滚——可 prompt 就是一段文本，改了直接发不就行了，为什么非要套软件工程的流程？**

prompt 改动的"爆炸半径"比代码大得多。一段 Java 代码改了有编译器+单测兜底，prompt 改了一个词（如把"可退/不可退"改成"判断是否可退"），输出分布可能整体漂移，而代码层面零报错。线上跑着百万次调用，一次 prompt 上线可能让客服满意度从 85% 掉到 70%，且因为 LLM 输出不确定，肉眼 review prompt 看不出来。Harness 的动机不是"流程仪式"，是"让 prompt 改动可回归（标注集）、可灰度（5% 先试）、可回滚（一键切稳定版）、可监控（幻觉率/满意度告警）"。不套这套流程的团队，必然踩"改一个词线上炸一次"的坑。

### 第二层：证据与定位

**Q：线上客服 Agent 的满意度从 85% 掉到 70%，你怎么定位是 prompt 的问题，还是 RAG 检索的问题，还是模型本身的问题？**

按链路拆证据。三步：
1. 看变更记录——最近哪一版 prompt/知识库/模型上线了？时间点和掉点是否吻合。如果是 v3→v4 上线当天掉点，锁定变更。
2. 看离线回归——把 v4 跑在 10 万标注集上，对比 v3 的准确率。如果离线准确率也掉，是 prompt/模型问题；如果离线不掉但线上掉，是线上数据分布（如双 11 期间问题类型变了）或 RAG 检索问题。
3. 看 bad case——抽样线上 500 条低分回答，人工分类根因：是"检索没召回正确文档"（RAG 问题）、还是"召回了但 prompt 指令让 LLM 答错"（prompt 问题）、还是"LLM 理解能力不够"（模型问题）。三类根因对应不同修复方向，不能混为一谈。

### 第三层：根因深挖

**Q：你定位到是 v4 的 prompt 把"只回答可退/不可退"改成了"详细解释退款理由"，导致 LLM 长篇大论但关键结论反而不准。根因是 prompt 写法问题，还是更深层的？**

根因不是"prompt 写得不好"，是"prompt 改动没有约束式评估就上线"。深挖：
1. 缺乏约束式回归——v4 改了输出格式（从分类变长文），但离线评估只看了"准确率"整体指标，没单独看"关键结论是否还在"。治本是评估要分维度（准确率/完整性/格式合规/安全性），不是只看一个总分。
2. 缺乏 prompt 评审规范——prompt 改动不像代码有 review 规范，谁都能改。要建立"prompt 改动需双评审 + 标注集回归 + 灰度"的规范，重大改动（如输出格式）必须过评审。
3. prompt 与业务的耦合——"只回答可退/不可退"是为了下游系统解析（客服工单系统按结论分流），改成"详细解释"破坏了下游解析。根因是 prompt 改动没考虑下游消费方，要把 prompt 当 API 契约管，格式改动要通知下游。

**Q：那为什么不直接锁死 prompt 格式，禁止任何人改输出格式，不就杜绝这类问题了吗？**

锁死格式会扼杀迭代。LLM 应用的核心价值就是"通过调 prompt 持续优化效果"，锁死等于不优化。正确做法是"约束变更流程"而非"禁止变更"：
1. 把 prompt 的"输出格式"和"业务指令"分离——格式部分（如 JSON schema）用强约束（如 Function Calling / structured output），业务指令（如语气/策略）可灵活调。
2. 变更分级——格式变更属"重大变更"（过评审 + 下游联调 + 灰度），业务指令调优属"小变更"（标注集回归即可上线）。
3. 契约测试——prompt 输出格式有自动校验（如 JSON schema 校验），格式不对自动告警。锁死是因噎废食，分层约束才能既安全又迭代。

### 第四层：方案权衡

**Q：你的评估用 LLM-as-judge（让大模型打分），但 LLM 打分本身也可能错（幻觉/偏好），这个评估可信吗？怎么权衡 LLM-as-judge 和人工标注？**

LLM-as-judge 是"成本和质量的权衡"。它便宜（自动跑 10 万样本几分钱）、快（分钟级），但有三类偏差：
1. 位置偏好——judge 倾向打第一个/最后一个答案高分，需随机打乱顺序消除。
2. 冗长偏好——judge 倾向给长答案高分，需控制长度或归一化。
3. 自我偏好——GPT 当 judge 偏好 GPT 风格，需用不同家族模型交叉判。
正确用法是"LLM-as-judge 初筛 + 人工精标校准"——离线评估全量跑 LLM-as-judge 找差异样本，差异大的（如 judge 给高分但人工看是错的）抽人工复核，定期校准 judge 和人工的一致性（一致率 > 85% 才可信）。关键场景（退款/投诉）必须人工，非关键（FAQ）LLM-as-judge 足矣。

**Q：为什么不全部用人工标注，人工不是最准吗？**

全部人工不可行。一是成本——10 万样本人工标注一次几万元，prompt 每周迭代跑不动。二是速度——人工跑一周，prompt 迭代节奏跟不上。三是规模——线上百万次调用，人工只能抽检 0.1%。LLM-as-judge 的价值是"让全量评估成为可能"，把人工从"全量标注"解放到"抽检校准 + 关键场景兜底"。工程上必须用自动化评估换迭代速度，人工只在最关键的 5% 兜底。

### 第五层：验证与沉淀

**Q：你做了离线评估（标注集）和在线 A/B，但线上效果有时和离线评估不一致（离线好但线上差），怎么验证 prompt 改动真的有效？**

离线/线上不一致的根因是"评估集和线上分布漂移"。验证要做三件事：
1. 标注集要"活"——离线集不能是几个月前的静态集，要定期从线上真实流量回流（如每周加 1000 条线上 case 进标注集），保持分布新鲜。否则离线集答得好但线上新问题答不好。
2. 在线 A/B 是最终判据——离线只是初筛，线上 5% 灰度跑 3-7 天看业务指标（满意度/转人工率/错退率），统计显著（p<0.05）才全量。离线和线上矛盾时，以线上为准。
3. shadow traffic 预演——灰度前先跑影子流量（线上请求复制一份给新版，不影响用户），看新版的输出和旧版差异分布，提前发现风险。验证不是单一指标，是"离线回归 + shadow + 灰度 A/B"三层递进。

**Q：怎么把这套 Harness 沉淀成团队级平台，而不是每个 LLM 应用各搞一套？**

沉淀成"LLM 应用平台"：
1. 统一 prompt 仓库——所有 prompt 版本化在中心仓库，团队协作有评审记录，不是散在各服务代码里。
2. 评估平台——标注集管理 + LLM-as-judge + 人工标注协同，所有应用共享评估能力，新应用接入即享回归测试。
3. 灰度发布平台——LLM 应用像业务应用一样灰度（5%→20%→100%），自动对比业务指标，异常自动切回。
4. 监控看板——RT/成本/token/幻觉率/满意度统一采集，按应用维度看，成本精确到接口。
5. 反馈闭环——线上 bad case 自动回流到标注集，形成"上线→监控→回流→迭代"闭环。把 Harness 从"每个团队自建"变"平台 + 配置"，这是 LLMOps 的工程化必然。

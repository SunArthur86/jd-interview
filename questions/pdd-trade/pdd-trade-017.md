# 【拼多多交易】交易系统怎么用设计模式？

> JD 依据："软件设计原则、设计模式"。

## 一、策略模式（促销）

```java
interface PromoStrategy { BigDecimal calc(Order o); }
// 满减/打折/直降各一实现，Map 注入消除 if-else
strategies.get(order.promoType).calc(order);
```

## 二、状态模式（订单状态机）

```java
interface OrderState { void next(Order o); }
class PaidState implements OrderState { public void next(Order o) { o.setState(new ShippedState()); } }
```

## 三、观察者（事件通知）

```java
order.pay();  // 发布 OrderPaidEvent
@EventHandler void on(OrderPaidEvent e) { stock.deduct(); points.add(); }
```

## 四、责任链（风控过滤）

```java
chain.add(new BlacklistHandler()).add(new RateLimitHandler()).add(new RiskHandler());
```

## 五、模板方法（下单流程）

```java
abstract class OrderTemplate {
    void create() { validate(); calcPrice(); deductStock(); save(); notify(); }
    abstract void calcPrice();  // 普通/拼团/预售覆写
}
```

## 六、底层本质

模式封装变化——策略封装算法变化、工厂封装创建、观察者封装联动。SOLID 落地工具。

## 常见考点
1. **单例怎么写**？——双重检查（volatile+synchronized）或静态内部类。
2. **策略 vs 状态**？——策略客户端选；状态对象内部自动切换。
3. **过度设计**？——简单 CRUD 不用，Rule of Three。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你用策略模式消除促销的 if-else，但促销规则运营三天两头改，策略模式真的比 if-else 好维护吗？**

策略模式的价值不是"少写 if-else"，而是"开闭原则（OCP）"——加新促销类型时，新增一个 PromoStrategy 实现类并注册到 Map，不改主流程代码。if-else 的问题是"加一个分支要改 createOrder 的方法体"，违反 OCP（对修改开放）。促销规则多变正是策略模式的适用场景：每条规则独立成一个类，运营要加"跨店满减"，开发新增 `CrossStoreDiscountStrategy`，主流程 `strategies.get(type).calc(order)` 一行不变。维护性体现在"改一个规则不影响其他规则"（类隔离）和"加规则不改主流程"（OCP）。代价是类变多，但相比"改 if-else 引入 bug 影响整条链路"，可接受。

### 第二层：证据与定位

**Q：线上有个新促销"百亿补贴直降"上线后，所有用了满减的订单金额都算错了，你怎么定位是策略模式注册错，还是满减策略本身的 bug？**

按策略的执行路径定位。三步：
1. 看策略注册表——Spring 启动时 `Map<String, PromoStrategy>` 的内容（打印或 actuator beans），确认"满减 type → FullReductionStrategy""百亿补贴 type → SubsidyStrategy"映射正确。常见 bug 是 type 字符串拼错（如 `FULL_REDUCTION` vs `full_reduction`）导致 get 返回 null 或错误策略。
2. 如果映射正确，看满减策略的代码——是不是"百亿补贴"和"满减"共用了某个计算逻辑（如 `BasePromo.calc`），改了公共逻辑影响满减。git diff 看这次上线改了哪些文件，重点看是否动了满减相关的类。
3. 复现——构造一个满减订单，单步调试 `strategies.get("FULL_REDUCTION").calc(order)`，看返回金额错在哪一步。策略模式的好处是定位清晰：每个策略独立类，断点直接打到对应策略，不用在巨型 if-else 里翻。

### 第三层：根因深挖

**Q：你查到是"百亿补贴策略"和"满减策略"都继承 `BasePromo`，改 BasePromo 的公共方法影响了满减，根因是什么？光是把 BasePromo 改回去就行吗？**

根因是"继承的脆弱性"——父类 BasePromo 的任何修改影响所有子类。治本：
1. 合成优于继承——把公共逻辑抽成工具类或服务（`PromoCalcHelper`），策略类组合使用而非继承。这样改 Helper 不影响不调用它的策略。
2. 如果必须共享逻辑，用模板方法但要明确"哪些可覆写、哪些固定"——`BasePromo` 的 `calc()` 是 final（骨架），子类只覆写 `doCalc()`（差异部分），改骨架要评审所有子类。
3. 测试保护——每个策略都有单元测试覆盖，改 BasePromo 后跑全部策略测试，任何策略金额变化都能立即发现。根因是"改公共代码没评估影响范围"，治本是"合成替代继承 + 测试保护"。

**Q：那为什么不把所有促销逻辑写在一个类里用 if-else 分支，至少改的时候能一眼看到所有影响？**

巨型 if-else 的问题是"改一个分支可能碰其他分支"（如改满减时不小心删了打折的某行），且分支多了方法几百行不可读。策略模式把"修改的影响范围"限制在单个策略类——改满减只动 FullReductionStrategy，不会误碰打折。至于"一眼看到所有影响"，靠的是测试覆盖（每个策略有测试）和 Code Review，而不是"都写在一起"。拼多多几十种促销类型，if-else 会是几千行的方法，根本无法维护。策略模式的类隔离是规模化下的必然选择。Rule of Three 之外，"类型多且会增长"也是用策略模式的信号。

### 第四层：方案权衡

**Q：你用观察者模式（事件通知）解耦下单和下游（库存/积分/风控），但事件是异步的，如果积分服务消费失败，用户没加上积分怎么权衡"解耦"和"可靠"？**

观察者模式的异步解耦必然牺牲"同步可靠"。权衡方案：
1. 事件存储保证可靠——用本地消息表（outbox），下单事务里同时插事件记录，事件由定时任务可靠投递，积分消费失败重试，保证最终一致。这是"用存储换可靠"。
2. 关键事件可同步——如果积分对用户体验关键（如"下单送积分立即可用"），把积分改成同步调用（下单时直接调积分服务），失败则下单失败回滚。牺牲解耦换一致。
3. 分级处理——强一致需求（钱/库存）走同步或 TCC，弱一致需求（积分/通知）走观察者异步。拼多多积分是"延迟到账"可接受的，走异步 + outbox 兜底；库存是强一致，走 TCC。观察者适合"允许延迟、允许重试、解耦价值大"的场景。

**Q：为什么不用响应式/Reactor（Mono/Flux）代替观察者模式，不是更现代吗？**

响应式编程和观察者模式不冲突——Reactor 的 `Flux` 本质就是观察者（发布订阅），只是加了"背压、流式操作、线程调度"能力。如果系统已用 Reactor（如 WebFlux），用 `Flux<DomainEvent>` 自然；如果是传统 Servlet 架构，用 Spring 的 `ApplicationEvent` 或直接 MQ 更简单。设计模式是思想，Reactor 是实现之一。拼多多的交易系统以同步为主（强一致要求），事件用 Spring Event + outbox 异步化，不一定上 Reactor。选型看整体技术栈，不是"谁更现代"。引入 Reactor 的学习成本和调试复杂度高，没有明确收益（如非阻塞 IO 的高并发优势）时，不必为了现代而现代。

### 第五层：验证与沉淀

**Q：你怎么验证策略模式的正确性——每个促销类型算出的金额都对？促销组合（满减+打折叠加）怎么测？**

促销是资损敏感场景，必须有完整测试：
1. 每个策略单测——构造各种订单（不同金额/商品数），断言 calc 返回的优惠金额符合规则（如满 100 减 20）。覆盖边界（满 99 不减、满 100 减 20、满 200 减 40）。
2. 组合测试——多个促销叠加（满减 + 打折 + 券）时，测试策略链（Chain of Responsibility）的执行顺序和互斥规则（如"打折和满减不叠加"）。这是策略模式 + 责任链的组合，要测"叠加后的最终金额"。
3. 回归测试——每次促销规则变更，跑全量历史订单的回归（用历史订单数据重算，对比新旧金额），任何不一致都要解释清楚。拼多多促销规则复杂，靠人工算不过来，必须自动化回归。

**Q：设计模式怎么沉淀成团队规范，避免"该用不用"或"过度设计"？**

靠"原则 + 评审 + 反模式清单"：
1. Rule of Three——代码重复三次才抽象成模式，不预先设计。CR 时如果发现"为未来可能的需求"加了策略/工厂，要求砍掉（YAGNI）。
2. 模式应用场景清单——文档明确"什么场景用什么模式"：促销/支付用策略、订单流转用状态、风控过滤用责任链、下单骨架用模板方法。新人按清单选，不自创。
3. 反模式清单——禁止"上帝类"（一个类几千行）、"深继承"（继承超 3 层用合成）、"过度工厂"（简单 new 包成工厂无意义）。CI 用设计度量（如类的圈复杂度、继承深度）告警。
4. DDD 战术落地——结合 DDD 的聚合/领域事件，让模式成为领域建模的自然产物，而非为模式而模式。

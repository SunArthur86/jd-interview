# 【拼多多交易】稳定性四板斧：限流/降级/超时/重试？

> JD 依据："高并发/高可用"。

## 一、限流

**令牌桶**（允许突发）：
```java
RateLimiter limiter = RateLimiter.create(1000);  // 每秒 1000 令牌
if (!limiter.tryAcquire()) {
    throw new RateLimitException();  // 拒绝
}
```

**Sentinel 多维限流**：
```java
@SentinelResource(value = "createOrder",
    blockHandler = "blockHandler",    // 限流降级
    fallback = "fallback")            // 异常降级
public CreateOrderResp createOrder(CreateOrderReq req) { ... }
```

限流维度：
- 单 UID（防刷）
- 全局 QPS（保总容量）
- 接口维度（核心接口单独配额）

## 二、降级

```
大促优先级：
  P0 核心：下单/支付/创单（绝不降级）
  P1 重要：查询/物流（限流保容量）
  P2 边缘：评论/推荐/历史（直接降级返回兜底）
```

```java
public List<Comment> getComments(Long pid) {
    if (switchManager.isDowngrade("comment-service")) {
        return Collections.emptyList();  // 降级：返回空
    }
    return commentService.list(pid);
}
```

## 三、超时（分级）

```
DB 查询：100ms
RPC 调用：200ms
HTTP 网关：3s
全链路：5s（用户可接受）
```

```yaml
feign:
  client:
    config:
      default:
        connect-timeout: 1000
        read-timeout: 2000
ribbon:
  ReadTimeout: 2000
```

**Hystrix 熔断**（失败率/RT 触发）：
```java
@HystrixCommand(commandProperties = {
    @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", value = "20"),
    @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", value = "50"),
    @HystrixProperty(name = "execution.isolation.thread.timeoutInMilliseconds", value = "500")
}, fallbackMethod = "fallback")
```

## 四、重试

**指数退避+幂等**：
```java
@Retryable(value = {SQLException.class, SocketTimeoutException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 100, multiplier = 2))
public Order pay(Long orderId) {
    // 必须幂等：带 requestId
    return payService.pay(orderId, requestId);
}
```

注意：
- 只对**临时故障**（网络抖动）重试
- 必须**幂等**（防重复扣款）
- **有限次**（3 次以内）+ 指数退避（避免重试风暴）

## 五、拼多多实战

- **大促预热**：限流配额提前调（核心接口 10 倍）
- **熔断兜底**：支付通道失败率 > 50% 熔断切通道
- **预案演练**：每月演练降级/限流，验证兜底有效

## 六、底层本质

四板斧本质是**"用可控的局部失败换整体存活"**——限流挡外部、降级舍内部、超时不等死、重试抗抖动，组合起来构成故障容错网。

## 常见考点
1. **Sentinel 和 Hystrix 区别**？——Sentinel 流量维度（QPS/并发），Hystrix 故障维度（失败率/超时）。
2. **重试风暴怎么防**？——指数退避+随机抖动+有限次+断路器。
3. **降级和熔断区别**？——降级是主动放弃（配置），熔断是被动保护（失败率触发）。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：限流会让一部分用户"下单失败"，这伤害体验，为什么还要限流？不限流让系统自己扛不行吗？**

不限流的结果是"系统被打爆后全部用户失败"。系统的容量是有上限的（DB 几万 QPS、应用 CPU 满载），超过容量后请求堆积、响应变慢、最终雪崩（超时连锁反应），所有用户都失败。限流的动机是"主动牺牲少部分（超阈值的请求），保住大部分（阈值内的请求）"。与其被动全崩，不如主动拒绝超额流量。拼多多大促如果没限流，峰值流量可能是平时 50 倍，系统崩了 100% 用户失败；限流后保住核心容量的 80% 用户成功，牺牲 20%。这是"局部失败换整体存活"，是高可用的底线。

### 第二层：证据与定位

**Q：线上 `rate_limited_count`（限流拒绝数）突然飙升，你怎么定位是真实流量涨了，还是限流配额配低了，还是有恶意流量？**

按流量来源拆解。三步：
1. 看总 QPS 和来源——如果总 QPS 从 10 万飙到 50 万，是流量涨了。看流量的来源分布（用户端/App/爬虫/恶意），如果某 IP 或某 UID 段集中异常，是恶意流量（刷单/爬虫），走风控/黑名单而非普通限流。
2. 看限流配额——如果 QPS 没涨（还是 10 万）但 `rate_limited_count` 飙升，是配额被调低了（运营误操作或配置错误）。查配置中心（Nacos/Apollo）的限流规则变更历史。
3. 看限流维度——如果是"单 UID 限流"触发多，可能是某 UID 在刷（恶意）；如果是"全局 QPS"触发多，是整体流量超容量。区分维度能定位是"刷子攻击"还是"容量不足"。恶意流量要挡在风控/网关层（黑名单/验证码），正常流量超容量要扩容或提配额。

### 第三层：根因深挖

**Q：你发现限流后系统还是慢（被限流的请求虽然拒绝了，但没被限流的请求也慢），根因是什么？限流不是应该保护系统吗？**

根因是"限流的位置不对"或"限流的粒度不够"。深挖：
1. 限流太靠后——如果在应用层（Service 方法）限流，请求已经经过了网关、鉴权、参数解析等逻辑，这些逻辑本身消耗资源。限流应该在最前端（网关层），把超额请求挡在业务逻辑之前。如果限流在 Service 层，网关到 Service 的链路（Tomcat 线程、鉴权）已经被占满，系统还是慢。
2. 限流粒度不够——如果只限了"下单接口"，但其他接口（查询、详情）没限流，整体资源（DB/Redis）被查询类请求占满，下单还是慢。要按"共享资源"限流（如 DB 连接池维度的限流），而非只按接口。
3. 下游没限流——即使本服务限流了，下游服务（库存/支付）没限流被打爆，本服务调用下游超时，还是慢。全链路都要限流。根因是"限流没覆盖到真正的瓶颈资源"，治本是"在瓶颈资源前限流 + 全链路限流"。

**Q：那为什么不直接把限流阈值设得很高（如百万 QPS），不就基本不限流了？**

阈值设高了等于没限流，失去保护意义。限流阈值应该设"系统能承受的最大容量"（如 DB 扛 5 万 QPS，限流设 4.5 万留 buffer）。设成百万 QPS，真实流量超过容量时限流不生效，系统照样崩。阈值的依据是"压测得到的容量上限 × 安全系数"，不是拍脑袋。如果业务确实需要更高 QPS，要扩容（加机器/分片）提升容量，再相应提阈值，而不是不改容量只提阈值（阈值超过了实际容量就是虚假的保护）。拼多多大促前先扩容（把容量从 5 万提到 50 万），再把限流阈值设到 45 万，两者匹配。

### 第四层：方案权衡

**Q：你的重试策略是"指数退避 + 3 次"，但如果下游真的故障了，3 次重试都失败，用户还是失败，而且重试加重了下游负担，你怎么权衡"重试换可用性"和"重试加重雪崩"？**

重试是双刃剑，权衡方案是"重试 + 熔断配合"：
1. 重试只在"偶发故障"有效——网络抖动/单次超时，重试一次可能成功。但如果下游真的挂了（持续返回 500），3 次重试只是 3 倍流量打向已经挂掉的下游，加重雪崩。
2. 熔断保护——当下游错误率超阈值（如 50%），熔断器打开，直接快速失败不再重试，让下游有时间恢复。重试和熔断配合：正常时重试提升可用性，故障时熔断避免雪崩。
3. 重试的策略优化——"有界重试"（最多 3 次）+"指数退避 + 随机抖动"（避免重试同步化）+"只对幂等接口重试"（避免重复扣款）+"对错误类型选择性重试"（连接超时可重试，业务错误不重试）。权衡是"正常时小重试提升成功率，故障时熔断兜底"，不是无脑重试。

**Q：为什么不用"快速失败不重试"，下游恢复后用户自己重新发起，不是更简单吗？**

不重试的代价是"偶发抖动也失败"，用户体验差。网络抖动（几十毫秒的瞬时问题）在日常很常见，如果不重试，用户遇到一次抖动就下单失败，要自己重试，体验差且可能导致用户放弃购买。重试能把"瞬时抖动"透明处理（用户无感）。只在"持续故障"时才该快速失败，而"持续 vs 瞬时"的判断由熔断器做（错误率阈值）。所以正确组合是"瞬时抖动靠重试（用户无感）+ 持续故障靠熔断（快速失败）"，两者协作。完全不重试是把"瞬时可恢复的"也当成"故障"处理，牺牲了可用性。

### 第五层：验证与沉淀

**Q：你怎么验证四板斧（限流/降级/超时/重试）在故障时真的有效，而不是"配置了但没生效"或"阈值错了"？**

必须做故障演练 + 阈值验证：
1. 限流验证——压测超过阈值的流量，验证 `rate_limited_count` 在阈值处线性增长，且后端实际 QPS 稳定在阈值（说明限流生效）。如果后端 QPS 也跟着涨，限流没生效。
2. 降级验证——手动触发降级开关（关闭评论服务），验证评论接口返回兜底（空列表），且下单主流程不受影响。验证降级的"时效性"（开关切换后几秒生效，而非几分钟）。
3. 熔断验证——人为让下游错误率超 50%，验证熔断器在阈值处打开（`circuit_breaker_state=OPEN`），打开后请求快速失败（不再等超时）。半开（HALF_OPEN）后探测下游恢复自动关闭。
4. 超时验证——人为让下游响应慢（5 秒），验证本服务在配置的超时（如 2 秒）处断开，而不是等 5 秒。每个板斧都要有明确的"触发条件 + 生效断言"，不能配置了就以为生效。

**Q：四板斧的配置怎么沉淀成规范，避免每个服务乱配（如重试 10 次、超时 30 秒）？**

靠"治理平台 + 强制规范"：
1. 限流/降级配置中心化——所有服务的限流/降级规则在统一平台（如 Sentinel 控制台）配置，有审计和审批，不允许业务侧随意改阈值。大促前统一调整。
2. 超时模板——按服务类型分模板（核心交易：读超时 500ms/写超时 1s；查询类：2s），业务用模板不自设。全链路超时层级校验（下游 < 上游）。
3. 重试规范——SDK 层强制"最多 3 次 + 指数退避 + 幂等校验"，业务侧用 `@Retryable` 但不能自己设 maxAttempts=10。CI 扫描重试配置，超限告警。
4. 演练常态化——每月一次全链路故障演练（限流/降级/熔断/超时），验证各板斧生效。演练失败的服务要整改。四板斧是"生命线"，配置必须规范且经过验证，不能靠运气。

## 结构化回答

**30 秒电梯演讲：** 流量超过系统容量，如何保整体不崩？简单说就是——限流（保护不被打死）+ 降级（保核心弃边缘）+ 超时（不等死）+ 重试（临时所以障重试）是稳定性四板斧，本质是"用可控失败换整体存活"。降级：核心保交易，非核心弃；超时：分级（DB<服务<网关）。

**展开框架：**
1. **限流** — 限流：令牌桶/漏桶/Sentinel
2. **降级** — 降级：核心保交易，非核心弃
3. **超时** — 超时：分级（DB<服务<网关）

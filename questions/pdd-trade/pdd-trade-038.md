# 【拼多多交易】智能交易中台怎么搭？

> JD 依据："智能交易中台"。

## 一、中台分层

```
应用层：商品/订单/营销/风控/客服（业务定制）
   ↑
服务层：推荐/搜索/风控/对话/画像（API 化能力）
   ↑
模型层：召回/排序/GBDT/深度学习/LLM（版本化）
   ↑
特征层：实时（Flink）+离线（Spark）特征仓库
   ↑
数据层：埋点/订单/行为/商品（数据中台）
```

## 二、能力服务化

```java
// 推荐服务（中台 API）
public interface RecommendService {
    List<Item> recommend(long uid, String scene, int n);
}

// 业务调用（场景定制）
@RestController
public class ProductPage {
    @GetMapping("/you-may-like")
    public List<Item> youMayLike(long uid) {
        return recommendSvc.recommend(uid, "product_detail_sidebar", 10);
    }
}
```

**能力清单**：
```
推荐：商品/类目/关联
搜索：召回/排序/相关推荐
风控：反作弊/限购/风险评分
对话：客服 LLM/导购 Agent
画像：用户/商品/商户标签
```

## 三、特征平台

```
实时（Flink）：当前会话浏览/点击/下单 → Redis（毫秒级）
离线（Spark）：历史购买/偏好 → HBase（T+1）
特征服务：统一 API，模型按需取
```

```java
public class FeatureService {

        Features f = new Features();
        for (String name : names) {
            f.put(name, featureStore.get(uid, name));  // 路由到 Redis/HBase
        }
        return f;
    }
```

## 四、AB 实验平台

```

实验配置：对照组 50 桶 / 实验组 50 桶
指标统计：点击率/转化率/客单价
显著检验：t-test，p<0.05 显著
决策：实验组显著优 → 全量
```

## 五、业务接入

```
低代码：
  - 选能力（推荐/搜索）
  - 配场景（页面/位置）
  - 选模型（默认/定制）
  - 上线 + AB 实验
SDK：
  - Java/Go/Python SDK
  - 自动埋点+实验
  - 降级兜底（中台挂走默认）
```

## 六、治理

```
模型治理：版本化+灰度+回滚+评测
效果归因：哪个模型贡献多少 GMV
成本归因：各业务用了多少 GPU/特征
SLA：中台 RT < 50ms，可用性 99.99%
```

## 七、拼多多实战

- **商品中台**：千万级商品统一打标/画像，喂推荐/搜索
- **用户中台**：亿级用户画像，千人千面
- **营销中台**：券/补贴/拼团统一规则引擎
- **AI 中台**：推荐/搜索/风控/对话模型统一管理，AB 实验每日千个

## 八、底层本质

智能交易中台本质是**"把 AI 能力从项目沉淀为平台"**——数据/特征/模型/服务分层，业务低代码接入，AB 实验驱动迭代。中台成功的关键是"开放+标准+SLA"，避免成为瓶颈。

## 常见考点
1. **中台怎么避免变重**？——能力下沉通用部分，业务定制留给上层；严控接入 SLA。
2. **特征平台怎么保证实时**？——Flink 流算 + Redis 缓存 + 预聚合。
3. **AB 实验怎么防互扰**？——正交流量分桶（每个实验独立维度），互不影响。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：智能交易中台把推荐/搜索/风控/对话能力都收归中台，但每个业务线（商品/订单/营销）的场景差异很大，统一收口会不会"中台不懂业务"，反而拖慢迭代？**

中台的价值是"通用能力复用 + 业务快速定制"，不是"中台包办一切"。中台提供的是底层能力（推荐召回/排序算法、搜索索引、风控模型、LLM 对话框架），这些跨业务共性高，每个业务自建是重复造轮子。业务特有的部分（如商品详情页的推荐策略 vs 营销页的推荐策略）由业务方在上层定制（配 prompt/规则/场景参数），中台提供 API 和配置能力。中台不懂业务是正常的，它懂的是"能力"；业务方不懂算法也是正常的，它懂的是"场景"。中台 + 业务定制的分工，比"每个业务线自建全套 AI 团队"省几倍人力。关键是中台 API 要足够通用 + 配置灵活，让业务能定制而非事事找中台。

### 第二层：证据与定位

**Q：商品详情页的"猜你喜欢"推荐点击率突然从 5% 掉到 3%，你怎么定位是中台推荐模型的问题，还是业务接入的问题，还是特征的问题？**

按中台链路逐层拆证据。三步：
1. 看模型层——中台推荐模型的离线评测指标（AUC/NDCG）是否下降。如果最近模型版本上线后离线指标掉，是模型问题；如果离线指标稳但线上 CTR 掉，是线上数据/特征/接入问题。
2. 看特征层——`FeatureService` 返回的特征是否缺失。如果实时特征（如"当前会话浏览"）因 Flink 延迟/Redis 故障缺失，模型输入残缺，推荐质量掉。查 `feature_missing_rate` 指标。
3. 看接入层——业务方调 `recommend(uid, scene, n)` 的 scene 参数是否变了（如从"product_detail_sidebar"误改成别的），或 AB 实验路由异常（流量被误分到对照组）。三组证据交叉定位"模型/特征/接入"哪层问题，而非笼统说"中台推荐不行"。

### 第三层：根因深挖

**Q：你定位到是实时特征缺失——Flink 流算延迟导致"当前会话浏览"特征滞后 5 分钟，推荐基于过期特征排序。根因是 Flink 性能问题，还是更深层？**

根因是"实时特征的时效性和成本权衡"。深挖：
1. Flink 延迟——大促期间流量暴涨，Flink 反压导致特征写入 Redis 延迟 5 分钟。表面是 Flink 性能，但根因是"特征算子资源没跟上流量"。
2. 特效时效要求分级——"当前会话浏览"要求秒级（影响实时推荐），"近 7 天购买偏好"要求小时级。所有特征都走 Flink 秒级算会浪费资源。根因是"没按时效分级，全走最高档"。
3. 治本——特征分级 SLA：秒级特征（会话浏览）走 Flink + Redis（毫秒），分钟级特征（实时点击率）走 Flink + 预聚合，T+1 特征（历史偏好）走 Spark + HBase。按时效分配算力，不把所有特征都压到秒级。Flink 大促前按峰值预估扩算子资源。根因是"特征平台要分级 + 弹性"，不是一味求快。

**Q：那为什么不把所有特征都走实时（Flink + Redis），保证最新，不就没有滞后问题了吗？**

全实时不可行。原因：
1. 成本爆炸——亿级用户 × 几百特征全实时算，Flink 算子和 Redis 容量翻几十倍，成本不可承受。
2. 没必要——"近 30 天购买频次"这种特征一天变一次，全实时算和 T+1 算结果几乎一样，浪费算力。
3. 噪声——过度实时反而引入噪声（如"最近 10 秒浏览"波动大，不如"最近 1 小时"稳定）。正确做法是按特征价值 × 时效需求分级：高价值 + 强时效（会话/实时点击）走实时，低时效（历史/偏好）走批。全实时是"用钱换不必要的精度"，分级是工程最优。

### 第四层：方案权衡

**Q：中台的 AB 实验平台，业务方想跑"新推荐模型"实验，但中台已有 5 个实验在跑，流量怎么分才不互扰？为什么不简单按比例分（如每个实验 16%）？**

简单按比例分会互扰。原因：6 个实验各 16%，用户同时进 6 个实验，效果交叉污染，无法归因哪个实验贡献了提升。正确做法是"正交分桶"：
1. 每个实验用独立的 hash 维度——如实验 A 按 `uid % 100` 分桶（0-49 对照 / 50-99 实验），实验 B 按 `(uid/100) % 100` 分桶（另一维度），两维度正交，用户在 A 是实验组、在 B 是对照组的概率独立。
2. 流量层互不重叠——每个实验独占一层流量（如层 1 跑推荐模型实验，层 2 跑 UI 实验），同层实验互斥（用户只进一个），跨层正交（互不影响）。
3. 为什么要正交——正交后每个实验的效果可独立归因（对照组和实验组在其他实验维度上分布均匀），非正交分桶会因 Simpson 悖论得出错误结论。简单按比例分是"看似公平实则污染"，正交分桶是 AB 平台的工程必需。

**Q：为什么不一个一个跑实验（串行），跑完一个再下一个，不就没有互扰问题了吗？**

串行太慢。中台每日千个实验，串行每个跑 7-14 天才显著，一年跑不完几十个。互联网的竞争力是"快速实验 + 快速迭代"，串行会让实验吞吐成瓶颈。正交分桶的价值是"并行跑几十个实验且可归因"，把实验吞吐从"月级几十个"提到"日级千个"。正交的设计成本（流量分层 + 统计）是一次性工程，换来的是实验效率数量级提升。串行是简单但扼杀迭代速度，正交是工程复杂但支撑规模化 AB，这是中台的必然选择。

### 第五层：验证与沉淀

**Q：你怎么证明中台的价值（而非每个业务自建）？中台建设投入大，ROI 怎么衡量？**

中台 ROI 从三个维度量化：
1. 重复建设省的人力——没中台时，商品/订单/营销各建推荐团队（如各 10 人 = 30 人），中台后统一 15 人支撑全部，省 50% 人力。
2. 能力复用率——中台能力被多少业务调用（如推荐 API 被商品/营销/搜索页调用 = 复用 3 次），复用率越高摊薄成本越低。
3. 迭代速度——中台后新业务接入 AI 能力的周期（从立项到上线），如从 2 个月降到 1 周（配置 + SDK）。ROI 不是单一数字，是"人力省多少 + 复用多少 + 迭代快多少"的综合。中台失败的特征是"复用率低（各业务还是自建）+ SLA 差（业务绕过中台）"，要持续度量这些指标。

**Q：怎么避免中台变成瓶颈（"中台不中"），即业务都要等中台支持，反而拖慢业务？**

避免瓶颈的关键是"开放 + 自治 + SLA"：
1. 能力自助化——中台提供自助 portal / SDK，业务配置即用，不等中台人工支持。新需求中台评估后纳入 roadmap，不让业务无限等。
2. 可绕过机制——中台 SLA 不达标（如 RT > 50ms）时，业务有降级兜底（如走默认推荐/缓存结果），而非死等中台。中台挂了业务还能跑。
3. 治理边界清晰——中台只做"通用能力"，业务定制（prompt/规则/场景）由业务负责，中台不越界包办。避免中台变成"什么都接"的臃肿团队。
4. 竞争机制——允许业务自建（如果中台不够好），用脚投票倒逼中台提升。中台的成功靠"能力好 + SLA 强 + 接入易"吸引业务，而非行政强制。把中台当"内部产品"运营，业务是"客户"，而非"垄断供应商"。

# 【拼多多交易】预案/回滚怎么做？钱学森工程思维？

> JD 依据："高可用/多活容灾"、"钱学森工程理论"。

## 一、预案（分级 SOP）

```
P0 致命（下单/支付全挂）：1 分钟发现，5 分钟止血
  预案：切机房/降级非核心/扩容
P1 严重（部分功能不可用）：5 分钟发现，30 分钟止血
  预案：限流/熔断/切通道
P2 一般（性能下降）：10 分钟发现，2 小时止血
  预案：扩容/调参
P3 轻微（不影响用户）：观察
```

每个预案标准化：
```
触发条件：RT > 1s 或 错误率 > 1%
执行步骤：1. 限流 50% → 2. 切备库 → 3. 通知值班
回滚条件：恢复后 30 分钟稳定 → 取消限流
负责人：on-call
演练频次：每月 1 次
```

## 二、回滚机制

**发布回滚（蓝绿/灰度）**：
```
蓝绿：蓝集群（v1）←→ 绿集群（v2），切流量秒级回滚
灰度：1%→10%→100%，每阶段观察，异常立即停
```

```java
// 灰度发布卡点
if (errorRate > 1% || rt > threshold) {
    featureFlag.rollback();  // 一键关 feature
    alert("灰度异常，已回滚");
}
```

**配置回滚**：
- 配置中心（Apollo/Nacos）版本化，秒级回滚
- 大促配置打 snapshot，异常一键还原

**数据回滚**（最复杂）：
- 灰度小流量验证（10% 用户），发现数据错立刻停
- Binlog 反向补偿（执行 reverse SQL）
- 严重时备份恢复（RPO/RTO 提前定）

## 三、钱学森系统工程思维

**还原论**（拆解）：把交易系统拆为订单/支付/库存/履约子系统，每个独立分析容量/故障。
**整体论**（涌现）：子系统各自 OK 不代表整体 OK，涌现性如级联故障、雪崩。
**系统工程**（定量化）：
- 容量定量化（压测给数字）
- 故障概率定量化（MTBF/MTTR）
- 预案有效性定量化（演练通过率）

**钱学森综合集成方法**：专家经验 + 数据 + 模型 → 定量决策。
```
混沌工程演练 → 积累故障数据 → 训练故障预测模型 → 自动化预案匹配
```

## 四、自动化预案系统

```
监控告警 → 故障画像（哪个服务/什么症状）→ 预案匹配 → 自动执行
                                          ↓ 否
                                       人工介入
```

```java
public class AutoPlanExecutor {
    public void onAlert(Alert alert) {
        Plan plan = planMatcher.match(alert);  // 匹配预案
        if (plan.isAutoExecutable()) {
            plan.execute();  // 自动限流/切流/扩容
        } else {
            notifyOnCall(plan);  // 人工审批
        }
    }
}
```

## 五、拼多多实战

- **双 11 预案**：200+ 预案覆盖各场景，大促前全量演练
- **多活切流**：机房故障 30 秒切流到备机房
- **混沌工程**：ChaosBlade 定期注入故障验证预案有效性
- **回滚演练**：每次大版本发布必做回滚演练

## 六、底层本质

预案/回滚本质是**"把故障应对从救火式升级为系统工程"**——还原论拆解，整体论看涌现，定量化预案，自动化执行。钱学森思维强调"从定性到定量、从专家到工程"。

## 常见考点
1. **蓝绿和灰度区别**？——蓝绿全量切换（秒级回滚），灰度渐进（验证成本高）。
2. **混沌工程怎么不"乱"**？——爆炸半径控制（小流量/单机房/工作日白天）+ 一键中止。
3. **数据回滚最痛的点**？——已写库的数据难回滚，靠灰度前置+Binlog 补偿+备份恢复组合。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你提到钱学森"还原论 + 整体论"，每个子系统都测过没问题（还原论），为什么还要整体演练（整体论）？子系统 OK 不就整体 OK 吗？**

整体不是部分的简单相加，有"涌现性"。每个子系统（订单/支付/库存）单独压测都达标，但全链路组合时会出现子系统级不会出现的问题：
1. 级联故障——订单服务慢导致支付服务调用超时，支付服务的熔断器打开，连锁导致履约也挂。这种"故障传染"在子系统单测中不会出现（单测时下游是 mock 的）。
2. 资源竞争——各子系统单独压测时 DB/Redis 是独占的，全链路时多个子系统共享 DB 连接池/Redis 集群，资源竞争导致大家都慢。
3. 容量叠加——单服务能扛 10 万 QPS，但全链路（下单要调 5 个服务）可能总容量只有 2 万（受限于最弱的共享资源）。整体演练的动机是"发现涌现性故障"，这是还原论（拆开看）发现不了的。钱学森的整体论强调"系统级的行为不能从部分推导"，必须整体测试。

### 第二层：证据与定位

**Q：线上 P0 故障（下单全挂），预案系统匹配了一个预案但执行后没恢复，你怎么定位是预案不对，还是执行失败，还是故障原因判断错了？**

按预案执行链路逐步核查。三步：
1. 看预案执行日志——`auto_plan_executor` 是否真的执行了预案步骤（如限流 50%、切备库）。如果执行了但没恢复，预案本身不对（措施没对症）。
2. 看故障画像——预案匹配的依据是"故障画像"（哪个服务/什么症状）。如果故障画像判断错（如把"支付通道挂"误判为"订单 DB 慢"），匹配的预案当然不对。回看监控数据，确认故障的真正根源。
3. 如果预案执行失败（如想切备库但切流失败），是执行层问题——看切流脚本/API 的执行日志，可能是备库也不健康、或切流权限/网络问题。重点是区分"预案逻辑错"（措施不对症）、"故障判断错"（画像不准）、"执行失败"（操作没成功），三者解法完全不同。

### 第三层：根因深挖

**Q：你发现是故障画像判断错了——把"Redis 集群挂"误判为"应用 CPU 高"，导致预案做了"扩容应用"而非"切 Redis"，根因是什么？光是加更多监控指标就行吗？**

加指标是治标。根因是"故障诊断逻辑不够智能"：
1. 单一指标误判——CPU 高可能是因为"Redis 挂导致缓存 miss，请求都打应用计算"，CPU 高是表象，根因是 Redis。仅看 CPU 会误判。治本是"多指标交叉诊断"——CPU 高 + Redis 错误率高 + 缓存命中率掉，综合判断为"Redis 故障"。
2. 因果链推理——故障诊断需要"因果链"（Redis 挂 → 缓存 miss → 应用 CPU 高 → 下单慢），而非单点指标。这需要拓扑知识（哪个服务依赖 Redis）和推理引擎。自动预案系统要内置"故障传播图"。
3. 更根本——预案系统应该"多预案并行尝试"而非"单预案押注"。诊断不确定时，同时执行多个可能的预案（如同时切 Redis 备用 + 扩容应用），观察哪个有效。根因是"诊断和预案绑死（诊断错则预案错）"，治本是"诊断容错 + 多预案试探 + 快速反馈修正"。

**Q：那为什么不让人工来判断和执行预案，人比机器更聪明，能看懂因果？**

人工慢且易错。P0 故障要求 5 分钟止血，但人发现告警→登系统→看监控→判断原因→执行预案，整个流程 10-30 分钟，早就超 SLA。且人在高压下容易误判（如慌乱中切错机房）或犹豫（不敢执行破坏性预案如切流）。自动化预案的价值是"1 分钟内匹配并执行"，比人快一个数量级。正确模式是"机器做快速预案 + 人做复杂决策"——常见故障（有 SOP 的）机器自动执行，复杂/未知故障机器告警人工介入。机器不是替代人，是处理"已知的、重复的"故障，让人聚焦"未知的、复杂的"故障。钱学森的"从定性到定量"就是把可定量的部分（已知预案）自动化，不可定量的（新故障）留给人。

### 第四层：方案权衡

**Q：你的回滚策略里，数据回滚（已写库的数据）最难，你提到"Binlog 反向补偿"，但如果数据已经被下游消费（如订单已通知履约发货），反向补偿会不一致，你怎么权衡？**

数据回滚是高危操作，权衡方案是"前置预防 > 事后补偿"：
1. 前置预防（最优）——灰度发布（小流量验证），发现数据错立即停（只影响 10% 用户，数据少）。或"延迟生效"（数据写入后不立即对下游可见，观察窗口期无误后再通知下游）。前置预防避免"数据已扩散"。
2. 事后补偿（兜底）——如果数据已经写库且通知了下游，单纯反向 SQL 不够，要"业务级回滚"。如错误创建了订单，不只删订单记录，还要：通知库存回滚扣减、通知支付退款、通知履约取消发货。这是"业务事务的补偿"，不是"数据行的回滚"。
3. 严重场景的备份恢复——如果数据错乱范围大（如批量 bug 影响百万订单），靠补偿修不过来，只能"时间点恢复（PITR）"恢复到故障前的备份。代价是丢失备份后的所有数据（RPO）。权衡是"前置预防成本低但要预判 / 补偿修复精准但复杂 / 备份恢复快但丢数据"，按故障影响选。

**Q：为什么不禁止所有"不可回滚的变更"（如 DDL 改字段类型），这样就不用做数据回滚了？**

禁止不可回滚变更会阻塞业务迭代。很多业务需求涉及 DDL（如给订单加字段、改字段类型），如果一刀切禁止，业务没法演进。正确的做法是"分级管理不可回滚变更"：
1. 高危变更（如删字段/改类型）——必须可回滚才上线，或用"扩展优于修改"（加新字段而非改老字段，老字段保留，灰度切读到新字段，确认后删老字段）。
2. 中风险变更（如加字段/加索引）——本身可回滚（删字段/删索引），但要评估影响（加索引可能锁表）。用在线 DDL 工具（如 gh-ost/pt-osc）避免锁表。
3. 低风险变更（如加配置项）——正常发布。权衡是"不阻塞迭代 + 高危变更特殊处理"，而非一刀切禁止。工程上要建立"变更分级 + 审批 + 回滚预案"流程，让"可回滚"成为默认，"不可回滚"要特批且充分预案。

### 第五层：验证与沉淀

**Q：你怎么验证预案真的能在故障时生效，而不是"预案写了但演练时发现执行不了"？**

预案必须常态化演练，不能等真故障：
1. 混沌工程注入——用 ChaosBlade 等工具定期注入故障（kill 实例/慢下游/网络分区/Redis 挂），验证预案系统的"检测→匹配→执行"全链路。每个预案至少每季度演练一次，过期的预案视为失效。
2. 演练的爆炸半径控制——先在单机房/小流量/工作日白天演练，验证预案执行正确且不扩散。逐步扩大范围（多机房/大流量），确认可控后纳入"可自动执行"白名单。
3. 演练指标——`plan_execution_success_rate`（预案执行成功率）、`time_to_recover`（故障恢复时间，应 <SLA）。演练失败的预案要整改。拼多多双 11 前 200+ 预案全量演练，演练通过才允许大促。

**Q：预案和回滚怎么沉淀成工程能力，而不是依赖个别老员工的经验？**

沉淀成"稳定性工程平台 + 知识库"：
1. 预案平台——预案的配置（触发条件/执行步骤/回滚条件）在平台管理，标准化模板（不依赖个人经验），版本化（每次演练更新）。故障画像自动匹配预案，自动执行或人工审批。
2. 回滚能力建设——蓝绿/灰度发布平台（一键回滚）、配置中心（版本回滚）、Binlog 反向工具（数据补偿）、备份恢复（PITR）。这些是基础设施能力，团队都能用，不依赖某个"回滚专家"。
3. 故障复盘知识库——每次故障的根因/预案有效性/改进措施归档，新故障参考历史（相似故障用相似预案）。钱学森的"专家经验 + 数据 + 模型"，专家经验沉淀进知识库，数据是故障监控，模型是自动匹配。
4. 演练文化——定期演练是硬要求（KPI），不是"有空再做"。把稳定性从"救火英雄"变成"工程体系"，这是大厂和小厂的分水岭。

# 【拼多多交易】用户画像系统怎么设计？

> JD 依据："用户平台研发"。

## 一、标签体系

```
基础标签: 年龄/性别/地区/注册时长
行为标签: 购买频次/客单价/活跃时段/浏览品类
偏好标签: 偏好类目/价格区间/品牌偏好
时序标签: 最近活跃/购买周期/流失风险
```

## 二、计算架构

```
离线（Spark T+1）: 全量历史算长期标签
实时（Flink 毫秒）: 当前行为算短期标签（如近 1 小时浏览）
存储: HBase（全量亿用户）+ Redis（活跃用户热缓存）
服务: 推荐/营销/风控调用，RT < 20ms
```

## 三、推荐应用

```java
public List<Product> recommend(long uid) {
    UserProfile profile = profileService.get(uid);  // 拉画像
    // 用偏好类目 + 价格区间 + 协同过滤召回
    return recallByProfile(profile);
}
```

## 四、底层本质

画像是"把行为数据压缩成可用特征"——支持个性化（推荐/营销）和智能化（风控/预测）。

## 常见考点
1. **实时和离线怎么一致**？——同 SQL 双跑 + 对账。
2. **冷启动**？——默认标签 + 相似用户 + 快速学习。
3. **隐私合规**？——脱敏存储 + 授权采集 + 可关闭。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：用户画像你用"离线 Spark T+1 + 实时 Flink"双轨，为什么不全部实时（Flink），实时不是更准吗？**

全实时不可行也不划算。长期标签（如"年均客单价""偏好类目"）需要全量历史数据计算，Flink 流式处理窗口有限（存全量历史成本高），且这类标签变化慢（客单价不会一天巨变），T+1 算一次够用。实时标签（如"近 1 小时浏览品类""当前购物车品类"）变化快，影响即时推荐，必须 Flink 毫秒更新。双轨的动机是"按标签时效性分级计算"——慢变标签离线批（省资源、算得全）、快变标签实时流（贵但即时）。全实时会让 Flink 承担全量历史计算，资源开销巨大且无必要（年客单价没必要每秒更新）。双轨是成本和时效的权衡。

### 第二层：证据与定位

**Q：推荐系统反馈"用户画像读延迟偶尔 200ms+"，你怎么定位是 Redis 热缓存没命中回源 HBase 慢，还是画像服务本身有问题？**

按读取链路定位。三步：
1. 看缓存命中率——`redis_hit_rate` 指标，如果某个 uid 的画像缓存 miss，回源 HBase，HBase 的 RT 可能几十到几百 ms（随机读 HBase 不如 Redis 快）。如果命中率从 95% 掉到 70%，是缓存问题（预热不够或 TTL 太短）。
2. 看 HBase 的 RT——`hbase_get_latency` P99，如果本身 200ms，是 HBase 慢（可能是 RegionServer 负载高、或 hot region、或 GC）。`hbase hbck` 检查 region 均衡，`status 'simple'` 看 RegionServer 负载。
3. 看画像服务的 RT 分布——如果 Redis 命中的请求 RT 都 <5ms，但 miss 的请求 RT 200ms，确认是"回源 HBase 慢"。解法是"预热活跃用户画像到 Redis"（定时把高频 uid 的画像刷入 Redis），让 99% 请求命中缓存。关键是区分"缓存层慢"（命中率低）和"存储层慢"（HBase RT 高）。

### 第三层：根因深挖

**Q：你查到是 HBase 某个 RegionServer 负载高（hot region），根因是什么？光是手动 balance 就行吗？**

手动 balance 是临时缓解。根因要看 hot region 的成因：
1. 数据热点——如果 rowkey 设计是 `uid`（顺序写入），活跃用户（大 uid）集中在某些 region，导致写入/读取热点。治本是 rowkey 加盐（`hash(uid) + uid`）打散，或反转 uid 前缀。
2. 读热点——某个超级用户（如大商家账号）被高频查询，所有请求打到同一 region。治本是"客户端缓存"（Redis 多副本）或"读副本"（HBase region replicas）。
3. Region 不均衡——`hbase balancer` 没跑或 region 数不均。治本是开自动 balance（定期重平衡）。拼多多亿级用户，rowkey 设计是关键，设计错了所有机器都救不了。根因定位要看是"设计问题"（rowkey 热点）还是"运维问题"（balance 没跑），治本要对症。

**Q：那为什么不直接全部存 Redis（不用 HBase），延迟不就最低吗？**

Redis 存不下亿级用户的全量画像。亿级用户 × 每用户几 KB 画像 = 几百 GB 到 TB 级，Redis 全内存成本是 HBase（磁盘存储）的几十倍。且画像的读取模式是"活跃用户高频、长尾用户低频"——20% 活跃用户贡献 80% 读取，这 20% 放 Redis（热缓存），80% 长尾放 HBase（冷存储），成本最优。全 Redis 是"用最贵的存储装最低频的数据"，浪费。HBase + Redis 的分层是经典的热冷数据分离，活跃的进 Redis（RT <5ms），长尾的查 HBase（RT 几十 ms，但访问频率低，整体可接受）。

### 第四层：方案权衡

**Q：你的实时标签（Flink）和离线标签（Spark）可能不一致（如"购买频次"实时算 10 次、离线算 9 次），你怎么权衡这种"双轨数据不一致"？**

双轨不一致是已知问题，权衡方案是"明确语义 + 对账校正"：
1. 语义区分——实时标签是"截至当前的实时值"（含最新行为），离线标签是"T+1 快照值"（截止昨天）。业务侧明确用哪个：推荐用实时（要新）、报表用离线（要稳）。不要混用。
2. 对账校正——每天离线结果出来后，用离线值校正实时值（离线是"全量准确"基准，实时可能因窗口/延迟有误差）。实时标签带版本/时间戳，离线覆盖时更新。
3. 容忍度——关键标签（如"会员等级"）以离线为准（避免实时抖动影响权益），辅助标签（如"近 1 小时浏览"）只用实时（离线无意义）。按业务影响分级处理一致性，不是所有标签都要强一致。

**Q：为什么不只用实时（Flink 算所有标签），这样不就没有双轨不一致问题了吗？**

全 Flink 的问题：一是长期标签计算需要全量历史，Flink 的状态存储（RocksDB state backend）存全量历史成本高且管理复杂；二是 Flink 作业重启/升级时状态恢复风险大，长期标签算错影响大；三是离线批（Spark）跑全量重算简单可靠（SQL 一把梭），Flink 流式增量计算的累积误差难排查。双轨的"不一致"是可控的（对账校正），而全 Flink 的"状态管理复杂 + 重算难"是工程灾难。拼多多这种规模，离线批是"基准和兜底"，实时流是"增量补充"，两者协作而非互斥。

### 第五层：验证与沉淀

**Q：你怎么验证用户画像标签的准确性？标签算错了会直接导致推荐/营销错位。**

标签准确性是画像系统的生命线，必须有评估机制：
1. 离线评估——抽样人工标注（如给一批用户的"偏好类目"标签，人工核对实际购买记录），算准确率/召回率。关键标签（如"高价值用户"）准确率 >95% 才上线。
2. 在线 A/B 测试——新标签上线时，小流量用新标签推荐、对照组用老标签，对比 CTR/转化率。如果新标签推荐效果差，标签可能有问题。
3. 业务反馈闭环——运营/客服反馈"这个用户标错标签了"（如把 inactive 标成 active），归因到计算逻辑，修复后回归。标签错了用户最直接的反馈是"推荐的东西我不感兴趣"，监控推荐 CTR 下降可能是画像质量问题。

**Q：用户画像怎么沉淀成平台能力，让推荐/营销/风控各业务线都能用而不重复建设？**

沉淀成"画像中台"：
1. 统一标签中心——所有标签统一定义（id/名称/计算逻辑/更新频率/owner），各业务线查同一个标签是同一份数据，避免"推荐算一个购买力、风控算另一个购买力"的不一致。标签有"数据字典"。
2. 统一服务 API——`ProfileService.get(uid, tags)` 统一入口，内部路由到 Redis/HBase，业务侧不感知存储。API 有 SLA（RT <20ms、可用性 99.9%）。
3. 标签生命周期管理——标签的创建（业务申请）、上线（评估准确率）、监控（命中率/质量）、下线（无人用清理）。避免标签无限膨胀（几千个没人用的标签拖垮计算）。
4. 隐私合规——所有标签记录"是否敏感""授权状态"，敏感标签（如收入）查询要审计。GDPR/个保法要求用户可查询/删除画像，中台统一支持。

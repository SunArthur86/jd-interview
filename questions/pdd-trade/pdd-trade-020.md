# 【拼多多交易】交易全链路架构怎么设计？

> JD 依据："交易系统技术升级改造"、"基础电商业务架构"。

## 一、全链路七段

```
用户 → [商品/购物车] → [下单] → [风控] → [创单] → [库存] → [支付] → [履约] → [结算]
       └─ 商品中心 ─┘  └ 交易 ┘ └ 风控 ┘ └ 订单 ┘ └ 库存 ┘ └ 支付 ┘ └ 物流 ┘ └ 财务 ┘
```

| 阶段 | 服务 | 关键操作 | 一致性 |
|------|------|----------|--------|
| 下单 | 交易 | 校验/锁价/算优惠 | 最终一致 |
| 风控 | 风控 | 反作弊/限购 | 强一致（拦截） |
| 创单 | 订单 | 生成订单号/状态机 | 强一致 |
| 库存 | 库存 | 扣减/预占 | 强一致 |
| 支付 | 支付 | 路由/回调 | 强一致 |
| 履约 | 物流 | 发货/配送 | 最终一致 |
| 结算 | 财务 | 对账/分账 | 最终一致 |

## 二、下单核心链路（创单+扣库存+锁价）

```java
@Transactional
public CreateOrderResp createOrder(CreateOrderReq req) {
    // 1. 幂等校验（防止重复下单）
    if (orderDao.existsByRequestId(req.getRequestId())) {
        return orderDao.getByRequestId(req.getRequestId());
    }
    // 2. 风控前置
    riskService.check(req.getUid(), req.getSkuList());
    // 3. 锁价（商品价格快照，防下单中变价）
    PriceSnapshot price = priceService.lock(req.getSkuList());
    // 4. 扣库存（TCC Try，预占）
    inventoryService.deduct(req.getSkuList());
    // 5. 创单（状态机：待支付）
    Order order = orderService.create(req, price);
    // 6. 发领域事件（异步通知履约/营销/积分）
    eventBus.publish(new OrderCreatedEvent(order));
    // 7. 延时关单（30 分钟未支付自动取消）
    mqSender.sendDelayed("order-close", order.getId(), 30, MINUTES);
    return resp(order);
}
```

## 三、分布式事务选型

```
资金强一致（扣库存+创单）：TCC
  Try   → 预占库存 + 生成待支付订单
  Confirm → 支付成功后确认扣减
  Cancel → 支付超时/取消时回滚库存

长流程（履约+物流）：Saga
  正向：创单→发货→配送→签收
  补偿：签收失败→拦截配送→召回

最终一致（积分/营销）：本地消息表
  创单事务同表写 Outbox → 定时投递 Kafka → 下游幂等消费
```

## 四、异步事件驱动

```java
// 订单服务发事件
@Transactional
public void paySuccess(Long orderId) {
    orderDao.updateStatus(orderId, PAID);
    outboxDao.insert(new Outbox("OrderPaid", payload));  // 同事务保证不丢
}
// Kafka 投递器扫描 outbox 推到 MQ
// 下游幂等消费
@KafkaListener(topics = "OrderPaid")
public void onPaid(OrderPaidEvent e) {
    if (consumeLog.isConsumed(e.getId())) return;  // 幂等
    fulfillmentService.ship(e.getOrderId());
}
```

## 五、拼多多特色（拼团/百亿补贴）

- **拼团**：下单→待成团→（满员）已成团→走支付；超时未成团→自动退款
- **百亿补贴**：补贴金额独立账本，结算时财务单独对账
- **多端（小程序/App/H5）**：统一网关+不同端 SDK

## 六、底层本质

交易全链路本质是**"把一笔交易分解为有序的强一致+最终一致阶段"**——资金相关强一致（TCC/Saga），业务辅助最终一致（MQ），用幂等保证重试安全，用状态机保证流程合法。

## 常见考点
1. **下单到支付挂了怎么办**？——订单待支付态+延时 MQ 超时关单+TCC Cancel 回库存。
2. **库存超卖怎么防**？——Redis 预扣 + DB 乐观锁 `WHERE stock>=n` + 分桶。
3. **全链路压测怎么做**？——影子库表+压测标识透传+全链路 mock 外部依赖。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：全链路你把"创单+扣库存"用 TCC 强一致，但"积分/营销"用本地消息表最终一致，为什么不全部 TCC 保证全部强一致？**

全部 TCC 会让链路极其脆弱且慢。TCC 的代价是"每个参与者实现 Try/Confirm/Cancel 三方法 + 防悬挂 + 幂等"，业务侵入大；且 TCC 的全局事务在 TC（协调器）协调下，任何一环失败全局回滚，链路越长越容易失败回滚。积分、营销这种"晚几秒到账用户无感"的场景，用 TCC 是杀鸡用牛刀——强一致带来的延迟和复杂度换不来业务价值。分级的动机是"按业务语义匹配一致性成本"：资金/库存（超卖是资损）必须 TCC，积分/通知（延迟可接受）用最终一致。把所有环节强一致，既慢又脆弱，违背"按需付费"原则。

### 第二层：证据与定位

**Q：用户投诉"下单扣了库存但订单没创建出来"，你怎么定位是创单失败回滚了库存但没回滚干净，还是创单成功但订单查询有问题？**

按 traceId 追全链路。三步：
1. 查链路追踪（SkyWalking/Pinpoint）——用用户提供的 uid 和时间找到 traceId，看创单请求的每个 span。如果创单 span 有 `status=ERROR`，是创单失败。
2. 看库存服务的 TCC Cancel 日志——创单失败应该触发库存 Cancel（解冻）。查 `inventory_cancel_log`，如果 Cancel 成功，库存已回滚（用户看到的"扣库存"可能是 Try 阶段的"预占"，Cancel 后已释放，用户误以为扣了）。如果 Cancel 没执行或失败，是 TCC 协调问题（TC 没通知到，或 Cancel 接口异常）。
3. 如果创单成功（订单表有记录）但用户查不到，是查询问题——查订单服务的查询接口（ES 同步延迟、或分库分表路由错误）。关键是顺着 traceId 看每一步的成败，而不是猜"哪个环节错了"。全链路追踪是这种问题的唯一解。

### 第三层：根因深挖

**Q：你查到是创单失败（如订单 DB 主键冲突），TCC 应该 Cancel 库存，但库存没回滚（用户库存被占），根因是什么？**

根因是 TCC 的 Cancel 没执行或执行失败。可能：
1. TC（事务协调器）没下发 Cancel——创单失败后，TC 应感知并通知库存 Cancel。如果 TC 和创单服务之间网络问题/TC 自己挂了，Cancel 下发不了。查 TC 的日志（Seata Server 的 `undo_log` 和事务状态），看全局事务是 `Rollbacking` 还是卡在 `Begin`。
2. Cancel 执行失败——库存服务的 Cancel 接口抛异常（如 DB 死锁、冻结记录被删了找不到）。查库存服务日志的 Cancel 异常。治本是 Cancel 接口幂等 + 重试（即使第一次失败，TC 会重试下发）。
3. 防悬挂问题——Cancel 比 Try 先到（网络乱序），Cancel 记录了"空回滚"，之后 Try 才到又冻结了库存，但这个 Try 永远不会被 Confirm/Cancel（因为全局事务已 Rollback）。防悬挂表要记录"已 Cancel 的 xid"，Try 到了发现已 Cancel 直接拒绝。根因是 TCC 的异常路径（Cancel 失败/悬挂）没处理好。

**Q：那为什么不用 Saga（补偿事务）代替 TCC，Saga 有补偿逻辑不是也能回滚？**

Saga 和 TCC 的区别：TCC 是"资源预留"（Try 冻结，Confirm/Cancel 原子），Saga 是"正向执行+反向补偿"（先真扣库存，失败则补偿加回去）。Saga 的问题是"中间状态可见"——扣库存成功但创单失败期间，库存是真的少了（其他人看到少库存），补偿后才加回。TCC 的 Try 是"冻结"（其他人不可见扣减），中间状态对业务透明。库存这种"不能中间态暴露"的场景用 TCC 更合适。Saga 适合"长流程且容忍中间态"（如履约：发货→配送→签收，中间态本就是业务的合法状态）。资金/库存强一致用 TCC，长流程业务用 Saga，各有所长。

### 第四层：方案权衡

**Q：你的全链路用 Kafka 事件驱动解耦下游（履约/积分），但如果下游消费慢导致 outbox 堆积，创单接口会受影响吗？你怎么权衡"解耦"和"背压"？**

好的设计是"创单不受下游影响"。权衡方案：
1. outbox 异步投递隔离——创单事务只负责"写订单 + 写 outbox 记录"（同事务，快），outbox 的 MQ 投递由独立的定时任务/Worker 做。下游消费慢只会导致 outbox 表堆积（或 MQ 堆积），不影响创单接口的 RT。这是"读写分离"——创单是同步快路径，事件投递是异步慢路径。
2. outbox 堆积的告警——监控 outbox 表的 `PENDING` 记录数，超过阈值告警。堆积说明下游消费能力不足，扩消费者或优化消费逻辑。
3. 极端背压保护——如果 outbox 堆积到影响 DB（表太大拖慢），要有"降级"策略（如暂时丢弃低优先级事件、或批量合并投递）。但创单主链路不能因下游慢而挂。解耦的意义就是"下游问题不传染上游"，架构上要保证这点。

**Q：为什么不直接同步调用下游（创单时同步调履约/积分），简单直接，不用维护 outbox？**

同步调用的三个问题：一是 RT 爆炸——创单同步调履约（发货准备）、积分、营销、风控复核，每个几十 ms，串行下来创单 RT 几百 ms，且任何下游慢都拖垮创单；二是强耦合——履约挂了创单也挂，一个下游故障导致全链路不可用；三是扩展性差——加一个新下游（如新的营销服务）要改创单代码。事件驱动的核心价值是"解耦 + 异步"——创单只发事件，下游各订阅各处理，互不影响。outbox 是保证"事件可靠投递"的代价（一张表 + Worker），相比同步调用的"高耦合 + 高 RT"，值得。拼多多几十个下游订阅创单事件，同步调用根本不可行。

### 第五层：验证与沉淀

**Q：你怎么验证全链路在各种故障下都能最终一致？比如"支付成功但履约没发货"这种边界？**

必须做全链路混沌测试，覆盖关键故障：
1. 各环节注入故障——kill 订单服务、支付服务、库存服务、MQ broker、DB，分别验证全链路的行为：订单是否回滚、库存是否 Cancel、支付是否幂等、履约是否补偿。每个故障场景断言"最终业务状态一致"（如订单 PAID 的，库存扣减、积分到账、履约发货）。
2. 全链路对账——每天 T+1 跑全链路对账：订单数 = 库存扣减数 = 支付成功数 = 履约发货数（业务不变式）。任何环节的 `reconcile_diff_count` >0 告警。对账是最终一致的兜底验证。
3. 故障恢复演练——不仅测故障发生时，还测故障恢复后（如 MQ 恢复）积压的消息能否正确消费、状态是否收敛。拼多多这种复杂链路，故障组合爆炸，必须靠自动化混沌测试 + 对账，不能靠人工。

**Q：交易全链路这么复杂，怎么沉淀成新业务能复用的平台，而不是每个业务（普通/拼团/百亿补贴）各写一遍？**

沉淀成"交易中台"：
1. 流程编排抽象——把"下单→风控→创单→库存→支付→履约"抽象成可配置的流程编排（如 BPMN 或自研 DSL），不同业务（普通/拼团/预售）的差异通过"扩展点"实现（如拼团在创单后插入"待成团"环节）。主流程复用，差异点扩展。
2. 统一事务框架——TCC/本地消息表封装成 SDK（`@TccAction`/`@TransactionalWithOutbox`），业务侧不感知分布式事务细节。幂等键生成、状态机、事件发布都是 SDK 能力。
3. 统一可观测——全链路 traceId 透传、统一监控大盘（每环节的成功率/RT/对账差异）、统一故障演练平台。新业务接入即享有可观测性。
4. 业务接入规范——新业务（如跨境）接入交易中台，走标准化的接入流程（流程配置/扩展点实现/压测/灰度），不重新造轮子。这是大厂"中台化"的核心价值——把复杂能力沉淀成可复用基础设施。

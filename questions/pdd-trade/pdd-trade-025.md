# 【拼多多交易】交易和风控怎么联调？

> JD 依据："交易系统技术升级"、"基础电商业务架构"。

## 一、联调节点

```
用户 → [领券] → [下单] → [支付] → [退款]
       风控     风控     风控     风控
       异步     同步     同步     异步
       （发券） （拦）   （拦）   （复核）
```

| 节点 | 同步/异步 | 决策 | 阈值 |
|------|-----------|------|------|
| 下单 | 同步 | 拦截/放行/挑战（验证码） | 100ms |
| 支付 | 同步 | 拦截（高风险） | 100ms |
| 领券 | 异步 | 延迟撤销 | 秒级 |
| 退款 | 异步 | 人工复核 | 分钟 |

## 二、同步决策

```java
public class RiskGateway {
    @HystrixCommand(fallbackMethod = "defaultPass", commandProperties = {
        @HystrixProperty(name = "execution.isolation.thread.timeoutInMilliseconds", value = "100")
    })
    public RiskResult check(RiskReq req) {
        // 1. 规则引擎（毫秒级）
        RiskResult ruleResult = ruleEngine.eval(req);
        if (ruleResult.isBlock()) return ruleResult;
        // 2. 模型评分（< 50ms，特征预热）
        double score = modelService.score(req);
        return score > THRESHOLD ? RiskResult.block("模型拦截") : RiskResult.pass();
    }
    // 降级：风控挂了默认放行（不阻塞下单）
    private RiskResult defaultPass(RiskReq req) {
        asyncRecheck(req);  // 异步补救
        return RiskResult.pass();
    }
}
```

## 三、规则引擎 + 模型

**规则（白名单/黑名单/限购）**：
```
IF uid IN blacklist THEN block
IF 同 IP > 10 单/分钟 THEN challenge  // 验证码
IF 同设备 > 5 单/天 THEN block
IF 新用户首单 > 1000 元 THEN review
```

**模型（GBDT/深度学习）**：
- 特征：用户画像/设备指纹/行为序列/图关系
- 输出：作弊概率 0-1
- 阈值：> 0.9 拦截，0.7-0.9 挑战，< 0.7 放行

## 四、多级降级

```
风控正常 → 同步拦截
风控慢（>80ms）→ 跳过模型，仅规则
风控挂 → 默认放行 + 异步复核（事后退款/封号）
大促预热 → 已知白名单用户跳过风控
```

## 五、数据回流闭环

```
交易行为（订单/支付/退款）→ Kafka → 风控特征平台
  → 更新用户画像/设备图
  → 离线训练新模型
  → 上线（A/B + 灰度）
```

## 六、底层本质

风控联调本质是**"用同步决策+多级降级平衡反作弊和体验"**——核心拦在交易前（防损），失败宁可放过不阻塞正常用户（事后追责）。

## 常见考点
1. **风控决策 RT 怎么保证 < 100ms**？——规则引擎前置+模型特征预热+本地缓存。
2. **风控和交易的事务关系**？——风控独立服务，交易调风控失败降级，不影响创单事务。
3. **模型怎么不停服上线**？——A/B 灰度（5%→20%→100%）+ 影子流量验证 + 一键回滚。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：风控联调你在下单/支付节点做同步拦截，但领券/退款做异步复核，为什么不全链路同步？同步不是更安全吗？**

全同步会拖垮交易体验和风控系统。下单/支付是"资金流出的关键卡口"——不拦住作弊，钱或货就出去了（资损），必须同步拦。领券/退款做异步的原因不同：
1. 领券——券的发放是营销侧，作弊领券的损失（多领几张券）远小于下单资损，且领券量大（每人每次活动都领），全同步会让风控 QPS 承压。异步"先发后复核"，发现作弊再撤销券，RT 无感。
2. 退款——退款本身是用户主动发起且金额已知，高风险的（如大额/频繁退款）走人工复核（分钟级可接受），低风险的自动通过。退款不是"实时拦截"场景（钱已经在平台），是"事后审查"。分级（同步拦资金流出 / 异步复核低风险）是成本和安全的权衡。

### 第二层：证据与定位

**Q：风控的同步决策 RT 突然从 50ms 飙到 300ms，导致下单被风控超时拖慢，你怎么定位是规则引擎慢，还是模型评分慢？**

按决策链路分段。三步：
1. 看风控内部的 metrics——`rule_engine_duration` 和 `model_score_duration` 分别多少。如果规则 10ms 但模型 280ms，是模型慢。如果规则 250ms，是规则引擎慢。
2. 模型慢的原因——模型评分要拉特征（用户画像/设备指纹/行为序列），如果特征查询（查 Redis/HBase）从 5ms 飙到 200ms，是特征服务慢（HBase hot region 或 Redis miss）。看 `feature_fetch_duration`。
3. 规则慢的原因——规则引擎如果用复杂脚本（如 Groovy/JS）且规则数量多，单次 eval 可能慢。看 `rule_count` 和 `rule_eval_per_rule_time`，优化规则（合并相似规则/用更高效的规则引擎如 Drools 决策表）。重点区分"特征查询慢"（存储问题）和"模型计算慢"（CPU 问题）和"规则引擎慢"（引擎效率问题）。

### 第三层：根因深挖

**Q：你查到是模型评分慢，根因是每次评分都要实时查 HBase 拉用户行为序列特征（200ms），光是预热特征就行吗？**

预热是治标。根因是"特征查询模式和高 RT 矛盾"：
1. 用户行为序列（如最近 100 次操作）存在 HBase，每次评分查一次，200ms。治本是"特征预热到 Redis"——风控决策前，把高频用户的特征提前加载到 Redis（定时刷新或事件驱动更新），评分时只查 Redis（5ms）。
2. 特征版本化——行为序列不需要"绝对实时"，用"近 5 分钟的快照"足够（作弊者不会 5 分钟内突变行为）。所以特征可以缓存（Redis TTL 5 分钟），用"稍微旧的特征"换"200ms → 5ms"的 RT 优化。
3. 更深层——模型本身（如果是在线推理）可能也慢（如复杂深度学习模型）。如果模型推理 100ms，要优化模型（蒸馏/量化/减小特征数）或用"轻量模型（GBDT）打分 + 重风险用深度模型复核"。根因要区分"特征慢"和"模型慢"，治本路径不同。

**Q：那为什么不直接把风控决策改成异步（下单时不拦，事后退款/封号），不就没有 RT 问题了吗？**

事后追责的代价是"资损已发生"。下单不拦，作弊者拿到货/券/补贴，事后发现再退款——但货可能已转手、券已使用、补贴已提现，追不回来。拼多多百亿补贴如果被黄牛薅，事后退款根本追不回（黄牛用小号提现跑路）。同步拦截的 RT 代价（100ms）远小于资损（每笔几十到几百元）。异步只适合"损失可追回"的场景（如多领的券可撤销），不适合"资金/实物已流出"的场景。所以下单/支付必须同步拦，宁可付出 RT 代价。降级（风控挂了放行 + 异步复核）是"极端情况的保底"，不是常态。

### 第四层：方案权衡

**Q：你的降级策略是"风控超时默认放行 + 异步复核"，但这等于"风控挂了就不拦"，黄牛可能趁机爆发，你怎么权衡"可用性"和"反作弊"？**

这是经典权衡。方案是"分级降级 + 白名单兜底"：
1. 超时降级但有上限——风控超时时，对"低风险用户"（历史清白、老用户）放行，对"高风险用户"（新号/黑名单关联）仍拦截（即使风控挂了，黑名单是本地缓存的，能查）。不是无差别放行。
2. 大促白名单——大促前预热"白名单用户"（历史清白的高价值用户），风控挂时白名单用户直接放行（跳过风控），非白名单限流或拒绝。这样既保住核心用户体验，又限制可疑流量。
3. 异步复核加强——降级期间的订单全量异步复核（而非抽样），发现作弊立即冻结/退款。降级时间窗（如 5 分钟）内的异常订单重点审查。权衡是"风控挂了不能让全站下单瘫痪（可用性），但用本地黑名单 + 白名单 + 强异步复核把风险降到可控"，不是非黑即白。

**Q：为什么不让风控规则全走本地（不下发到风控服务），交易服务本地直接 eval，不就没有"风控服务挂"的问题？**

本地 eval 规则的问题：
1. 规则更新复杂——风控规则频繁变（每天加新规则/调阈值），如果规则在交易服务本地，每次更新要重新发交易服务（或搞本地热加载，一致性难保证）。集中式风控服务用配置中心热更新，所有调用方立即生效。
2. 规则复杂度——复杂规则（如"同设备多账号""图关系"）需要集中数据（设备库/关系图），交易服务本地没有这些数据，查不了。
3. 模型推理——深度学习模型推理需要专门的推理服务（GPU/大内存），不可能在交易服务本地跑。所以风控必须是独立服务，交易服务调用它。降级策略（本地黑名单兜底）保留极简的本地能力（查黑名单），但完整规则和模型在风控服务。集中式是为了能力和灵活性，降级是为了可用性，两者结合。

### 第五层：验证与沉淀

**Q：你怎么验证风控联调在各种故障下既不漏拦（资损）也不误拦（影响正常用户）？**

风控的两种错误都要验证：
1. 误拦（正常用户被拦）——压测正常用户流量，验证 `block_rate`（拦截率）在合理范围（如 <0.1%）。误拦率 = 正常用户被拦截的比例，高了对业务伤害大（用户流失）。用历史"已确认正常"的订单回流测试，断言不被拦。
2. 漏拦（作弊者放过）——用历史"已确认作弊"的订单（标注数据）测试，断言被拦截。漏拦率 = 作弊订单放过的比例，高了是资损。这是风控模型的核心评估指标（召回率）。
3. 降级场景测试——kill 风控服务，验证交易服务按降级策略（放行/白名单/本地黑名单）行为正确，且降级期间的异步复核能抓回作弊。`degrade_duration`（降级持续）和 `post_degrade_recall_rate`（降级期事后追回率）要监控。

**Q：风控联调怎么沉淀，让新的交易场景（如新业务线）快速接入风控，而不重复对接？**

沉淀成"风控网关 SDK + 决策平台"：
1. 统一 SDK——所有业务线通过 `RiskGateway.check(scene, req)` 接入，scene 标识场景（下单/支付/领券），SDK 内部路由到对应规则和模型。业务侧不感知风控细节。
2. 决策平台配置化——风控运营在平台配置规则/模型/阈值/降级策略，按场景生效，不改业务代码。新场景接入只需"定义场景 + 配规则"。
3. 数据回流标准化——所有业务的行为数据统一格式回流风控特征平台（schema 统一），新业务的特征自动进入模型训练。闭环自动迭代。
4. 灰度机制——新规则/模型先影子流量（只 eval 不拦截，验证不影响业务），再小流量灰度，全量上线。把风控从"每接一个业务都要开发"变成"配置化接入"，这是中台对反作弊能力的沉淀。

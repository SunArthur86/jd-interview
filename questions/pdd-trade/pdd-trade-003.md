# 【拼多多交易】多活容灾怎么设计？核心交易怎么保证不挂？

> JD 依据："核心交易链路多活容灾"。

## 一、多活架构

```
同城双活（上海 A + 上海 B，50km）
  ├─ 强一致（专线半同步复制）
  └─ 单机房挂自动切另一机房

异地灾备（杭州/深圳）
  ├─ 异步复制（秒级延迟）
  └─ 大灾难切流

单元化（按 UID 切）
  ├─ 单元 A: UID 0-5000万（完整链路）
  ├─ 单元 B: UID 5000万-1亿（完整链路）
  └─ 全局服务（GZS）：跨单元（如全局黑名单）
```

## 二、同城双活强一致

```
写请求 → 主库（上海 A）
   ↓ 半同步
备库（上海 B）ACK → 返回客户端成功
（备库没 ACK 则降级异步 + 告警）
```

专线 50km 延迟 < 5ms，强一致可行。

## 三、单元化（爆炸半径控制）

```
单元 A（服务 UID hash 0-49%）
  ├─ 交易服务
  ├─ 订单 DB（A 的数据）
  └─ Redis（A 的缓存）

单元 B（服务 UID hash 50-100%）
  └─ 完整独立链路
```

**好处**：单元挂只影响 50% 用户（而非全站）；横向扩展加单元。

**跨单元**：全局数据（如全局黑名单）用 GZS，理论上业务不该跨单元调用。

## 四、流量切换

```
机房故障检测 → SLB/DNS 切流到备机房
  ├─ 4xx/5xx 率突增 → 告警
  ├─ 心跳失败 → 摘除
  └─ 灰度切流（10% → 50% → 100%）验证
```

## 五、底层本质

多活容灾本质是**"用冗余换可用性"**：
- 单机房故障率 1% → 两机房 0.01%（独立故障域）
- 单元化限制爆炸半径
- 自动切流秒级恢复

这是金融核心系统的硬要求——拼多多核心交易必须 99.99%+ 可用。

## 常见考点
1. **同城和异地多活选哪个**？——强一致场景（交易）同城双活；容忍最终一致选异地多活。
2. **单元化最难的是什么**？——跨单元调用（不该有）+ 全局数据同步 + 单元迁移。
3. **多活的成本**？——机房+专线+数据同步延迟，通常核心业务才值得。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：核心交易为什么要做单元化按 UID 切分，而不是所有机房共享一套全量数据？**

因为全量数据共享意味着每次写都要在所有机房间同步强一致，跨机房延迟（同城 5ms、异地 30ms+）会把单次写入的 RT 拉高一个数量级，核心交易 P99 从 50ms 飙到 300ms，下单成功率直接掉。单元化的动机是"把跨机房同步变成单机房内闭环"——UID 落在哪个单元，这个 UID 的所有数据（订单/账户/购物车）都在那个单元，正常读写不跨机房。只有全局数据（黑名单/库存）才用 GZS。本质是用"数据分片"换"延迟可控"。

### 第二层：证据与定位

**Q：监控发现同城双活的写延迟 P99 从 8ms 飙到 50ms，你怎么确认是半同步复制在等备库 ACK，而不是应用本身变慢了？**

看 DB 侧的 `semi_sync_net_avg_wait_time`（半同步平均等待时间）和 `Rpl_semi_sync_master_net_wait_time`。三步定位：
1. 登主库 `SHOW GLOBAL STATUS LIKE 'Rpl_semi_sync_master%'`——如果 `Rpl_semi_sync_master_net_avg_wait_time` 从 2ms 飙到 40ms，说明时间花在等备库 ACK 上。
2. 看备库 `SHOW SLAVE STATUS` 的 `Seconds_Behind_Master` 和 `Slave_IO_Running`——如果备库 SQL 线程慢了（比如大事务回放），ACK 自然慢。
3. 看专线质量——`ping` 主备专线，如果 RTT 从 2ms 飙到 20ms，是专线抖动（光衰/路由切换），找网络组。如果 `semi_sync_master_yes_timout` 增加，说明半同步在退化成异步，要查是不是备库 ACK 超时阈值（`rpl_semi_sync_master_timeout`）设低了。

### 第三层：根因深挖

**Q：你查到是备库有大事务在回放导致 ACK 慢，根因是什么？光是把那个大事务杀掉就行吗？**

杀掉只是治标。根因是业务有"批量大事务"——比如运营一次性 UPDATE 几百万行订单状态做营销活动，这种事务在备库回放时阻塞后续所有 ACK。治本：
1. 业务侧拆事务——批量操作改成每次 1000 行的小事务循环，间隔 50ms，让备库追得上。
2. 大事务检测——DB 侧开 `long_transaction_time` 告警，超过 1s 的事务直接告警拦截。
3. 关键链路和批量任务隔离——批量任务走只读库或独立实例，不污染主备复制的核心交易链路。

**Q：那为什么不直接把半同步改成异步复制，延迟不就没了吗？**

异步复制延迟确实没了，但主库挂了的瞬间，已 ACK 给用户的订单可能还没同步到备库——切流后这些订单"用户已支付但订单不存在"，就是资损。核心交易宁可慢（50ms 延迟）不能错（数据丢失）。半同步的意义就是"至少有一个备库收到了才告诉用户成功"，这是金融场景的底线。异步只适合异地灾备（容忍秒级丢失），核心同城双活必须半同步。如果半同步太慢，应该治大事务和专线，而不是降级复制强度。

### 第四层：方案权衡

**Q：单元化按 UID hash 切分，但如果某个 UID 是超级大 V（比如商家账号操作量是普通用户 1000 倍），数据倾斜严重，你怎么权衡？**

纯 hash 会倾斜，因为业务上流量本就不均匀（大商家/热门 SKU）。权衡方案是"路由键分层"：
1. C 端用户按 UID hash 均匀切——C 端流量大且均匀，hash 能打散。
2. B 端商家单独成单元或按 merchantId 范围分片——商家少但每个量大，单独管理，必要时单独扩容。
3. 热点商家（如百亿补贴店铺）用"子单元"——把一个超热商家的数据再按 orderId hash 拆到多个子单元，读时聚合。代价是路由表复杂、跨子单元查询要合并。本质是承认"业务不均匀"，用分层路由而非纯 hash 来消化倾斜。

**Q：为什么不直接用一致性哈希解决倾斜和扩容问题？**

一致性哈希解决的是"扩容时数据迁移量少"，不解决"单 key 热点"。一致性哈希下，一个超级 UID 还是落在单个节点，单点 QPS 爆炸。单元化的扩容其实是"加新单元 + 按 UID 范围迁移"，迁移量可控（只迁一段 UID）。而热点问题必须靠"key 分桶"（如把库存拆成 stock:1~stock:10 十个 key）或"子单元"解决，和一致性哈希是两个维度。单元化选范围分片而不是一致性哈希，是因为交易场景需要"UID 范围 → 单元"的明确映射，便于路由和容灾（整段 UID 一起切流）。

### 第五层：验证与沉淀

**Q：你怎么验证切流真的能救场，而不是切过去发现备机房数据不全又切回来？**

必须有真实的容灾演练，不能等真故障才发现问题：
1. 每月做一次"机房破坏演练"——在低峰期把主机房流量按 10%→50%→100% 切到备机房，监控 `order_create_success_rate`、`pay_success_rate`、`reconcile_diff_count`，任何指标异常立刻切回。
2. 数据完整性校验——切流前用 `pt-table-checksum` 对比主备数据差异，`diff_count` 必须为 0 才允许切流。
3. 演练要覆盖"半同步退化"场景——故意让备库 ACK 超时，验证系统是否正确降级异步 + 告警，而不是卡死。拼多多这种量级，切流失败一次就是全站故障，必须演练到形成肌肉记忆。

**Q：单元化之后，新来的 RD 经常写出跨单元调用的代码（比如查另一个单元的订单），你怎么从机制上防住？**

靠 Code Review 挡不住，必须工具化：
1. RPC 框架层拦截——所有 RPC 调用必须带 `targetUid`，框架根据 UID 判断是否跨单元，跨单元调用默认拒绝（抛异常），白名单（GZS 服务）才放行。
2. CI 静态扫描——扫描代码里直接拼 orderId 查询的 SQL（没带 UID 路由条件的），CI 直接挂掉。
3. 单元化设计文档 + 新人必考——入职第一周必须通过"单元化路由"的考试，知道哪些数据在本地、哪些走 GZS。把架构约束固化成框架规则和 CI 规则，而不是靠人记。

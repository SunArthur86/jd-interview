# 【拼多多交易】订单状态机怎么设计？

> JD 依据："交易系统技术升级改造"。

## 一、核心状态机

```
        ┌─────────┐  支付   ┌─────────┐  发货  ┌─────────┐  确认  ┌────────┐
        │ 待支付  │ ──────▶ │ 已支付  │──────▶│ 已发货  │──────▶│ 已完成 │
        └────┬────┘         └────┬────┘        └─────────┘        └────────┘
             │ 取消               │ 退款
             ▼                    ▼
        ┌─────────┐         ┌─────────┐
        │ 已取消  │         │ 退款中  │
        └─────────┘         └─────────┘
```

**状态转移表**（合法转移白名单）：
```java
Map<Status, Set<Status>> TRANSITIONS = Map.of(
    PENDING_PAY, Set.of(PAID, CANCELLED),
    PAID, Set.of(SHIPPED, REFUNDING),
    SHIPPED, Set.of(COMPLETED, REFUNDING),
    REFUNDING, Set.of(REFUNDED, REJECTED)
);

public void transit(Order order, Status target) {
    if (!TRANSITIONS.get(order.status).contains(target))
        throw new IllegalStateTransitionException();
    order.status = target;
}
```

## 二、拼多多扩展态

**拼团订单**：
```
待支付 → 待成团（N 人未满）→ 已成团（继续发货流程）
                          → 未成团（超时退款）
```

**预售订单**：
```
待支付定金 → 待支付尾款 → 已支付全款 → 发货
```

## 三、事件驱动

```java
public class Order extends AggregateRoot {
    public void pay() {
        transit(this, PAID);
        DomainEvents.publish(new OrderPaidEvent(id));  // 触发库存扣减/通知
    }
    public void ship() {
        transit(this, SHIPPED);
        DomainEvents.publish(new OrderShippedEvent(id));
    }
}
```

## 四、状态变更一致性

状态变更 + 业务 + 事件同事务：
```java
@Transactional
public void pay(Long orderId) {
    orderDao.updateStatus(orderId, PAID);
    outboxDao.insert(new Outbox("OrderPaid", payload));  // 同事务
}
```

## 五、底层本质

订单状态机本质是**"用规则约束保证流程合法"**——资金相关不允许状态错乱。状态机是交易系统的骨架，所有业务（支付/库存/物流）围绕状态流转。

## 常见考点
1. **状态机怎么防并发**？——DB 乐观锁（version 字段）或 `UPDATE ... WHERE status=原状态`。
2. **拼团成团怎么判定**？——定时扫描超时团 + 实时计数满团触发。
3. **状态机和领域事件**？——状态变更发事件，下游订阅做后续（发货/通知/结算）。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你为什么坚持订单流转必须用状态机，而不是直接在订单表上加个 status 字段由各业务方自己改？**

订单涉及资金和库存，status 一旦错乱就是资损。直接改字段没有任何约束，支付服务把"待支付"改成"已完成"也能成功，但发货流程没跑、库存没扣，这就是漏单。状态机本质是把"合法转移白名单"固化在代码里（`TRANSITIONS` Map），任何非法转移直接抛 `IllegalStateTransitionException`，把人能犯的错用编译期/运行期规则挡掉。动机不是优雅，是止损——拼多多日均亿级订单，靠人肉 review 挡不住。

### 第二层：证据与定位

**Q：线上监控发现某天突然有 200 单从"已支付"直接跳到"已完成"，跳过了"已发货"，你怎么定位是状态机没拦住，还是别的原因？**

先怀疑状态机被绕过，而不是状态机失效。三步定位：
1. 查 `order_status_log`（状态变更流水表，每次变更记 from/to/operator/source）——看这 200 单的 source 是哪个接口、哪个服务改的。
2. 查代码——看是不是有人新加了个"后台一键完成"的管理接口，直接 `order.setStatus(COMPLETED)` 没走 `transit()` 方法。这种情况状态机根本没生效。
3. 如果 source 都走了 `transit()`，那是状态机配置漏了——查 `TRANSITIONS` 是不是被改过，git blame 看谁加了 `PAID → COMPLETED` 这条非法边。
99% 的情况是"绕过状态机直接 setStatus"，而不是状态机本身没拦住。

### 第三层：根因深挖

**Q：假设你确认是有人绕过状态机直接 setStatus，那根因是什么？光是把那个人骂一顿就行吗？**

根因是状态机的约束没有下沉到聚合根。订单状态被暴露成 public setter，任何拿到 Order 对象的人都能改，这是设计漏洞。骂人没用，治本是让状态只能通过聚合根的方法改——把 `setStatus` 设为 private，对外只暴露 `pay()/ship()/confirm()` 等业务方法，方法内部调 `transit()`。这样即使有人想绕，编译都过不了。另一个根因是 Code Review 没卡住——状态机相关的改动必须要求订单域 owner review。

**Q：那如果业务说"我就是需要一个后台一键把异常订单标记完成"的接口，为什么不直接给他加？**

直接加就是在状态机里开个口子，等于没状态机。要区分"业务正常流转"和"运营人工干预"两条路：正常流转走状态机严格约束；人工干预走独立的 `OrderAdjustService`，单独鉴权（需要运营主管审批）、单独记录 `adjust_log`（谁、为什么、审批链）、且只允许特定终态（比如只允许从"已发货"调到"已完成"，不允许跳态）。把绕过行为"显式化、可审计"，而不是堵死或放任。

### 第四层：方案权衡

**Q：你现在用 `Map<Status, Set<Status>>` 的转移表实现状态机，如果订单状态从 6 个变成 20 个（加上拼团/预售/跨境各种态），这张表会爆炸，你怎么权衡？**

二维表在状态少时清晰，状态多了维护成本高（N² 组合）。权衡方案是按业务域拆状态机——普通订单、拼团订单、预售订单是三个独立状态机，各自有自己的转移表，而不是塞进一张大表。订单创建时按业务类型选对应状态机。这样每张表还是 6-8 个状态，可维护。代价是状态机之间不能共享状态（比如拼团的"已成团"和普通的"已支付"是不同枚举），但这正是业务想要的——不同业务的语义本就不同。

**Q：为什么不直接上 Spring StateMachine 这种框架，而要自己用 Map 实现？**

Spring StateMachine 功能全（嵌套状态/守护条件/事件监听），但它是为复杂状态机设计的，配置重、调试难、启动慢。拼多多的订单状态机其实是"扁平 + 简单转移"，20 行代码的 Map 转移表完全够用，且单元测试好写（直接断言 `TRANSITIONS` 内容）。引入框架是为了解决复杂度，没有那个复杂度就不该引入。只有当出现嵌套状态（如"退款中"内部还有"待商家同意/待买家退货/待商家收货"子状态）且子状态转移逻辑复杂时，才考虑 Spring StateMachine。

### 第五层：验证与沉淀

**Q：你怎么验证状态机的正确性？单元测试怎么写才能覆盖所有非法转移？**

不能只测合法转移，更关键是测"所有非法转移都被拒绝"。用参数化测试遍历状态笛卡尔积：对每一对 (from, to)，如果不在 `TRANSITIONS` 里，断言抛 `IllegalStateTransitionException`；如果在，断言成功。这样一旦有人偷偷加了条非法边，测试会失败。再补一个并发测试：起 100 个线程同时对同一订单调 `pay()`，断言只有一个成功（靠 DB 乐观锁 `WHERE status=PENDING_PAY`），其余抛异常——验证状态机在并发下不漏。

**Q：状态机上线后怎么防止下次故障重演？**

沉淀两件事：
1. CI 强制检查——用 ArchUnit 或自定义注解扫描，禁止任何地方出现 `order.setStatus(`（除聚合根内部），编译或 CI 阶段直接挂掉。
2. 状态变更全量留痕——`order_status_log` 表记录 from/to/bizNo/operator/traceId，接 ELK，一旦线上再出现非法跳态，按 traceId 一查到底。拼多多这种量级，靠人 review 挡不住，必须靠机制和工具。

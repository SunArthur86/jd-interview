# 【拼多多交易】百亿补贴秒杀怎么设计？

> JD 依据："高并发系统开发"。

## 一、层层削峰架构

```
用户 → CDN（静态资源）
   ↓ 动态请求
网关（限流 + 黑名单）
   ↓
Redis（预扣库存，Lua 原子）
   ↓ 成功
MQ（异步下单，削峰）
   ↓ 消费
订单服务（DB 落库）
   ↓
用户轮询查结果
```

## 二、前端优化

- 按钮防抖（点击后置灰）
- 答题验证（防机器人）
- 页面静态化（CDN 缓存）
- 倒计时（不到点按钮不可点）

## 三、Redis 预扣库存（防超卖核心）

```lua
-- Lua 原子：判断 + 扣减
local stock = tonumber(redis.call("get", KEYS[1]))
local bought = tonumber(redis.call("sismember", KEYS[2], ARGV[1]))  -- 是否买过
if bought == 1 then return -1 end  -- 已抢过
if stock and stock >= 1 then
    redis.call("decr", KEYS[1])
    redis.call("sadd", KEYS[2], ARGV[1])  -- 标记已买
    return 1  -- 成功
end
return 0  -- 售罄
```

## 四、MQ 异步下单

```
Redis 扣减成功 → 发 MQ 消息（含 uid/skuId）
   ↓ 消费者稳定处理（如 1万/s）
订单服务：创建订单 + DB 落库 + 通知用户
```

用户端：抢到"排队中" → 轮询查订单结果（通常 1-3 秒）。

## 五、防黄牛

- UID 限购（每人 1 件，Redis Set 标记）
- 设备指纹（一设备 N 账号拒绝）
- 答题验证（增加机器人成本）
- IP 频控

## 六、底层本质

秒杀本质是**"瞬时洪峰的层层削峰"**——每层过滤一部分，把百万 QPS 转成后端可承受的稳定流。关键是不超卖（Redis Lua 原子）和不压垮（异步削峰）。

## 常见考点
1. **怎么保证不超卖**？——Redis Lua 原子扣减（判断+扣减原子）+ DB 兜底校验。
2. **MQ 消息丢了怎么办**？——Redis 扣减成功但 MQ 丢 → 定时对账补偿。
3. **热点 key 怎么处理**？——分桶（库存拆多份）+ 本地缓存。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你用 Redis 预扣库存 + MQ 异步下单，为什么不直接让用户请求打到 DB 扣库存？加个分布式锁不也能防超卖吗？**

DB 扛不住百万瞬时并发。单 MySQL 的 QPS 上限也就几万，秒杀瞬时百万 QPS 直接打过去，连接池打满、行锁排队、最终雪崩。分布式锁更糟——Redis 分布式锁让扣库存串行化，一秒只能处理几百个，排队到用户超时。Redis 预扣的动机是"把热点库存的扣减逻辑搬到内存数据库"，Redis 单实例 10 万+ QPS，Lua 脚本保证"判断+扣减"原子（不用锁），再通过 MQ 把"异步落库"的速率降到 DB 能承受的 1 万/s。本质是"内存层削峰 + 异步落库"，DB 只需要处理稳定流。

### 第二层：证据与定位

**Q：秒杀活动结束后，发现有 100 件商品超卖了（DB 库存变 -100），你怎么定位是 Redis Lua 没生效，还是异步落库出问题了？**

先看超卖发生在哪一层。三步定位：
1. 看 Redis 的库存 key——活动结束后 `GET stock:sku123` 应该是 0，如果是负数，说明 Lua 脚本的"判断 stock >= 1"被绕过（比如有人直接 `DECR` 没走 Lua），查代码是不是有别的入口改了库存。
2. 看 DB 订单数 vs 实际库存——如果 Redis 库存是 0 但 DB 订单数 > 库存，是异步落库的问题。查 MQ 消费日志，是不是消费失败了重试导致重复下单（消费者没做幂等）。
3. 看热点 key 的 QPS 监控——如果某个库存 key 的 QPS 远超预期（比如预期 1 万实际 50 万），可能是流量没被前端/网关削峰挡住，Redis 扛住了但 DB 落库积压，消费延迟导致用户重复抢（前端轮询接口没幂等）。重点是区分"Redis 层超卖"（Lua 失效）和"DB 层超卖"（落库重复/积压）。

### 第三层：根因深挖

**Q：假设你定位到是 Redis Lua 脚本本身没问题，但库存 key 是分桶的（stock:1~stock:10），某个桶先卖完了用户还能抢到，根因是什么？**

分桶库存的经典问题：分桶是为了分散热点，但扣减逻辑要保证"只在还有库存的桶扣"。根因是分桶后的"售罄判定"没做全局协调——10 个桶各 100 件，桶 1 卖完后用户请求打到桶 1 应该返回售罄，但如果路由逻辑是固定 hash（UID%10），用户被绑死在桶 1，桶 1 没了也去不了桶 2。治本：
1. 售罄桶重路由——Lua 脚本里如果当前桶 stock=0，遍历其他桶找还有库存的扣（脚本内循环，原子）。
2. 或全局库存兜底——所有桶扣减后再校验全局库存 key（`stock:total`），全局为 0 直接拒绝。代价是 Lua 脚本变复杂、耗时增加，但保证不超卖。

**Q：那为什么不直接用一个库存 key（不分桶），Lua 原子扣减不是能扛 10 万 QPS 吗？**

10 万 QPS 是单 key 的理论上限，但秒杀瞬时 QPS 可能百万级，单 key 会成为瓶颈——所有请求打到同一个 Redis slot，CPU 打满、响应变慢。分桶是为了把单 key 的 10 万 QPS 分散到 10 个 key（10 个 slot），合计 100 万 QPS。另一个问题是"热点 key 让 Redis 集群倾斜"——单 key 所在的节点 CPU 100%，其他节点闲着。分桶是把热点打散。所以分桶的动机是"突破单 key 性能瓶颈 + 均衡集群负载"，不是为了好看。如果秒杀 QPS 只有几万，单 key 完全够用，不用分桶。

### 第四层：方案权衡

**Q：你用 UID 限购（Redis Set 标记每人 1 件）防黄牛，但黄牛用 1000 个小号批量抢，你怎么权衡"防刷"和"正常用户买不到"？**

纯 UID 限购挡不住养号黄牛，因为他们的 UID 都是真实的。权衡方案是"多维信号 + 实时风控"：
1. 账号质量分——拼多多有用户画像（注册时长/历史消费/活跃度），秒杀只对"高质量账号"开放，新号/低活号直接拒绝或要求更严验证。
2. 设备指纹聚类——同一设备登录 N 个账号参与秒杀，直接封设备。黄牛一台手机刷 1000 个号的特征很明显。
3. 实时行为风控——秒杀点击到下单的速度（<500ms 可疑）、请求来源 IP 集中度、是否有自动化工具特征（请求头/JS 执行环境）。
4. 兜底是答题验证/滑动验证码——增加机器人的成本，真正用户觉得麻烦但能接受，机器人成本飙升可能放弃。本质是用"多维信号"把黄牛账号和正常账号区分开，而不是靠单一 UID 规则。

**Q：为什么不直接用"先到先得"的队列模式（所有人请求排队，按顺序处理），这样最公平？**

队列模式（所有请求进 FIFO 队列按顺序处理）听起来公平，实际上：一是百万请求进队列本身就是巨大开销（MQ 堆积、消费延迟），用户体验是"等了 30 秒告诉你没货"，比"立即返回售罄"更差；二是队列模式还是要把库存扣减放到消费端，消费端依然面临"高并发扣减"（只是从同步变异步），没解决根本问题。Redis 预扣 + 立即返回的本质是"快速失败"——抢到的立即知道（排队中），没抢到的立即知道（售罄），体验最好。公平性靠"防黄牛"保证（不让机器人抢占名额），而不是靠队列顺序。

### 第五层：验证与沉淀

**Q：你怎么验证秒杀不超卖？压测时怎么制造真实的百万并发来验证 Lua 脚本和分桶的正确性？**

必须有专项压测，不能等线上大促验证：
1. 库存边界压测——设库存=100，压 10 万并发请求，断言：成功抢到的订单数 = 100（不多不少），Redis 库存最终 = 0，DB 订单数 = 100。重复跑多次确认稳定。
2. 分桶正确性压测——10 个桶各 100 件，压百万并发，断言总订单 = 1000，且售罄桶的重路由生效（没有用户因为落到空桶而误判售罄，但实际别的桶还有货）。
3. 异步落库一致性——压测后等所有 MQ 消费完，对账 `Redis 扣减总数 == DB 订单总数`，`reconcile_diff_count = 0`。任何不一致就是 bug。这种压测要在隔离环境跑，用真实的百万并发工具（如 pts/自研压测平台），不能靠 JMeter 单机。

**Q：秒杀系统怎么沉淀，让下次活动配置化上线，而不是每次都改代码？**

沉淀成"秒杀活动平台"而不是每次硬编码：
1. 活动配置化——运营在后台配置活动（SKU/库存/开始时间/限购规则/分桶数），平台自动预热库存到 Redis、生成 Lua 脚本、挂载限流规则，无需 RD 改代码。
2. 标准化链路——所有秒杀走同一套"Redis 预扣 + MQ 异步 + 轮询查询"的模板，活动差异只在前端展示和风控规则。新业务线接入秒杀只需配置，不用重新开发。
3. 复盘数据沉淀——每次活动的抢购率/超卖检测/黄牛拦截率/系统指标自动归档，下次活动配置时参考（比如库存多少该分几个桶）。把秒杀从"每次重做"变成"配置上线"，这才是平台化的价值。

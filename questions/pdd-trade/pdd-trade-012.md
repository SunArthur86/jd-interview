# 【拼多多交易】亿级订单怎么分库分表？

> JD 依据："大规模业务系统"。

## 一、分片策略

```
分片键: uid（买家）
库数: 16（ds0..ds15）
表数: 每库 16 张（order_0..order_15）
路由: hash(uid) % 16 → 库; hash(uid) / 16 % 16 → 表

容量: 16 库 × 16 表 × 千万 = 25 亿订单
```

```yaml
shardingsphere:
  tables:
    orders:
      actualDataNodes: ds${0..15}.orders_${0..15}
      databaseStrategy: { hash: { column: uid } }
      keyGenerator: { type: SNOWFLAKE, column: id }
```

## 二、异构（卖家视角）

uid 分片后，卖家查自己订单要跨所有分片，性能差。**异构到 ES**：

```
订单写入 → MySQL（uid 分片）+ binlog → Canal → Kafka → ES（sellerId 索引）
卖家查询 → ES（按 sellerId 过滤）
```

## 三、冷热分离

```
热订单（近 1 年）: MySQL（分库分表）
冷订单（1 年前）: HBase（归档）

查询: 先查 MySQL，没有再查 HBase（透明路由）
```

## 四、全局订单 ID（雪花）

```
| 1 bit | 41 bit 时间 | 10 bit 机器 | 12 bit 序号 |
```
- 全局唯一、有序（按时间）
- 不依赖 DB

## 五、跨库难题

- **跨库 JOIN**：业务组装（多查内存 join）/冗余字段/ES
- **聚合统计**：异步汇总表 / 数仓离线
- **分布式事务**：本地消息表/TCC

## 六、底层本质

分库分表本质是**"用一致性换扩展性"**——单库 ACID 强但扩展差，分库后失去跨库事务/JOIN，换线性扩展。

## 常见考点
1. **为什么按 uid 不按订单 id 分**？——买家查自己订单最高频，uid 分片同买家订单在一起。
2. **卖家怎么查**？——异构 ES（sellerId 索引），牺牲一致换查询性能。
3. **分片键能改吗**？——极难（停服迁移/双写灰度），选错代价大。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你选 uid 作为分片键，但订单表里还有 sellerId、orderId、createTime 等维度，为什么 uid 优先级最高？**

因为业务查询频率决定分片键。拼多多的订单查询，买家查"我的订单"是绝对高频（每天几亿次），且每个买家一辈子可能几千单，按 uid 分片让"同买家的订单物理相邻"，查"我的订单"只命中单分片，性能最优。而卖家查（商家后台）、按时间统计（报表）频率低得多，用异构（ES）或汇总表解决。分片键只能选一个，选最高频的查询维度。如果选 orderId 分片，买家查"我的订单"要扫所有分片（WHERE uid=xxx 不带分片键），性能灾难。选错分片键等于把最高频查询变成全分片扫，万劫不复。

### 第二层：证据与定位

**Q：线上某个买家反馈"我的订单"列表很慢（3 秒），但其他买家正常，你怎么定位是这个买家的数据问题，还是分片路由问题？**

先看是不是这个 uid 落到的分片有问题。三步：
1. 算这个 uid 的分片位置——`hash(uid) % 16` 得到库号、`hash(uid)/16%16` 得到表号，确认数据在 `dsX.orders_Y`。
2. 直连那个分片查 `SELECT count(*) FROM orders_Y WHERE uid=?`——如果这个买家在这个分片有几十万条订单（异常多，比如黄牛账号或脚本刷单），单分片单表数据倾斜，查询慢是正常的（LIMIT 翻页深了慢）。看 `EXPLAIN` 是不是走 uid 索引但还是慢（数据量大）。
3. 如果数据量正常（几千条），但查询仍慢，看那个分片库的负载——`SHOW PROCESSLIST` 看是否有大查询/锁等待，或这个分片所在物理机 IO 高。单买家慢大概率是"该分片数据倾斜"或"该分片物理机故障"，不是全局问题。关键是定位到具体分片，再具体分析。

### 第三层：根因深挖

**Q：你查到是这个买家（黄牛）单分片有 50 万条订单导致深分页慢（查第 1000 页要 3 秒），根因是什么？光是限制他查询深度就行吗？**

限制查询深度是治标。根因要看为什么慢：
1. MySQL 深分页慢的根因是 `LIMIT offset, size` 要扫描 offset+size 行——`LIMIT 100000, 20` 扫 100020 行，即使有索引。解法是用"游标分页"（`WHERE id > lastId ORDER BY id LIMIT 20`，基于主键游标），每次只扫 20 行，深分页也快。
2. 50 万订单的单表本来就偏大——虽然分库分表了，但单分片 50 万不算多（千万级才到瓶颈），真正的瓶颈是"深分页扫描"。游标分页能解决 99%。
3. 如果业务真的需要"跳页"（直接看第 1000 页），可以限制最大翻页深度（如最多看前 100 页），超过的引导用户用时间范围筛选。光限制深度而不优化查询方式，是把问题推给用户。

**Q：那为什么不干脆给这个黄牛的订单单独分一个表，避免影响其他买家？**

单买家单独分表是反模式——分片策略应该是"数据均匀分布"，给特殊买家开特例会让路由表爆炸（要维护"哪些 uid 在特殊表"）。而且黄牛的特征是"账号多"，你给他单独分表，他用 1000 个小号继续倾斜。正确的解法不是"为单个倾斜 key 单独分片"，而是"解决查询本身的高效性"（游标分页 + 合理的 LIMIT）。如果真的有超级商家（如百亿补贴店铺）单 sellerId 的订单上亿，那是 sellerId 维度的查询走 ES 异构，不影响 uid 分片。数据倾斜靠"查询优化 + 异构"解决，不靠"分片特例"。

### 第四层：方案权衡

**Q：你用 ES 异构解决卖家查询（sellerId 索引），但 ES 数据有秒级延迟（binlog 同步），卖家刚发货后立刻查可能查不到，你怎么权衡"实时性"和"查询能力"？**

异构必然牺牲实时性，权衡方案是"分级查询"：
1. 卖家后台默认查 ES（秒级延迟可接受，因为卖家看的是订单列表，刚发的订单稍微延迟显示无感）。
2. 卖家对"刚操作的订单"的即时反馈走 MySQL——比如卖家点"发货"后，前端立即显示成功是基于操作返回值（不依赖查询），不需要查 ES。只有"回查订单列表"才查 ES。
3. 强一致场景（如卖家对账、财务核对）走 MySQL 直查（按 sellerId 跨分片扫，慢但准），或走专门的 sellerId 汇总表（实时双写）。权衡是"日常查询走 ES（快但延迟）、关键对账走 MySQL（慢但准）"，按场景选数据源。完全追求实时会失去 ES 的查询能力，完全用 ES 会丢强一致，分级是必然。

**Q：为什么不用"双写 MySQL（uid 分片）+ MySQL（sellerId 分片）"两套 MySQL，而要引入 ES 异构？**

双写两套 MySQL 有三个问题：一是 sellerId 维度的查询除了按 sellerId，还可能要按时间/状态/金额组合查询，MySQL 的索引灵活性不如 ES（ES 天然支持任意字段倒排）；二是双写要保证一致性（要么分布式事务重，要么接受不一致），而 binlog 异构是"单写源 + 异步派生"，一致性更可控；三是 sellerId 分片的 MySQL 依然面临分库分表的扩容难题（卖家增长也要重新分片）。ES 的优势是"查询灵活 + 横向扩展 + 异步派生"，代价是秒级延迟和 ES 运维成本。对于"卖家后台多维查询"这种场景，ES 比 sellerId 分片的 MySQL 更合适。如果查询模式简单（只按 sellerId），sellerId 分片的 MySQL 也能用，但拼多多卖家后台查询复杂，ES 更值。

### 第五层：验证与沉淀

**Q：分库分表上线后，你怎么验证数据路由正确、没有"订单写错库"或"查询漏数据"？**

必须有路由正确性和数据完整性的验证：
1. 路由正确性测试——写一个 uid 对应订单的用例，断言订单确实落在 `ds{hash(uid)%16}.orders_{hash(uid)/16%16}`（直接连那个库查）。用 ShardingSphere 的 SQL 日志（`sql.show=true`）验证路由的库表号正确。覆盖边界 uid（如 uid=0、uid=最大值、uid 跨库临界值）。
2. 数据完整性对账——分库分表后，跑一个全量对账：`SUM(各分片订单数) == 全局订单总数`，且无重复（雪花 ID 唯一）。每天增量对账（当天新增的订单数 vs 各分片新增数之和）。
3. 异构一致性——MySQL vs ES 的对账，`reconcile_diff_count` 每天 T+1 跑，差异率要在阈值内（如 <0.01%）。任何完整性问题都说明分片/异构逻辑有 bug，必须靠对账发现，不能靠"用户没投诉"。

**Q：分片键选错的代价极大（迁移要停服），怎么沉淀机制让团队选对分片键、且未来业务变化时不踩坑？**

沉淀成"分片键评审 + 演进预案"：
1. 分片键强制评审——任何新表的分片键选择必须由架构组评审，评估"未来 3 年的查询模式"，产出评审文档（为什么选这个键、其他维度怎么查、扩容方案）。拼多多这种规模，分片键选错就是灾难。
2. 分片数预留扩容空间——初始 16 库 16 表，但路由算法用"一致性 hash"或"2 的幂扩容"（16→32 时只需迁移一半数据），预留扩容路径。不要选死板的 `hash%16`（扩容要全量迁移）。
3. 异构兜底——即使主分片键（uid）满足不了新查询，靠 ES/HBase 异构补，而不是改分片键。新业务接入时，先看能否用异构解决，实在不行才考虑分片键迁移（双写灰度，周期长）。本质是"分片键视为不可变，靠异构层演进"。

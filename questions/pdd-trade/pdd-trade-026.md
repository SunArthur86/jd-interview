# 【拼多多交易】高并发四件套：池化/缓存/扩容/异步队列？

> JD 依据："高并发/高可用"。

## 一、池化

**连接池（HikariCP）**：
```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20          # 经验值：CPU 核心数 * 2 + 有效磁盘数
      minimum-idle: 10
      connection-timeout: 3000
      idle-timeout: 600000
```

**线程池**（创单隔离）：
```java
@Bean("createOrderExecutor")
public ThreadPoolExecutor createOrderExecutor() {
    return new ThreadPoolExecutor(
        50, 100, 60, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(1000),
        new ThreadFactoryBuilder().setNameFormat("create-order-%d").build(),
        new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略：让调用方降速
    );
}
```

## 二、多级缓存

```
请求 → Caffeine（本地，ms）→ Redis Cluster（分布式）→ DB
        L1（秒级失效）       L2（分钟级）            兜底
```

```java
@Cacheable(value = "product", key = "#id", cacheManager = "multiLevel")
public Product getProduct(Long id) {
    return productDao.findById(id);
}
```

**三防**：
- 击穿（热 key 失效）：互斥锁重建
- 穿透（不存在的 key）：布隆过滤器 + 缓存空值
- 雪崩（批量失效）：过期时间加随机 `ttl + random(60s)`

## 五、拼多多双 11 实战

- **预热**：商品/库存/价格预热到 Redis
- **限流**：网关 Sentinel 单 UID 限流
- **削峰**：创单走 Kafka，消费按容量限速
- **扩容**：HPA + 提前手动扩容（核心服务 5 倍）

## 六、底层本质

四件套本质是**"用空间/资源换时间和稳定"**——池化换创建开销、缓存换 DB 压力、扩容换吞吐、队列换峰值冲击。

## 常见考点
1. **缓存一致性怎么保证**？——Cache Aside（先写库再删缓存）+ 延时双删。
2. **线程池满了怎么办**？——拒绝策略（CallerRuns 反压/Abort 抛异常）+ 上游限流。
3. **Kafka 积压怎么处理**？——临时加消费者+扩分区+跳过历史（先恢复实时）。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说池化/缓存/扩容/异步是"四件套"，但这四个不是孤立的，它们之间有什么关系？是不是上了缓存就不用扩容？**

四者关系是"分层防御、互补而非替代"。缓存挡读压力（80% 读走缓存，DB 只扛 20%），但缓存解决不了"写压力"和"计算压力"——下单的计算（算价/扣库存）还是要 CPU 和 DB，这时要扩容（加机器分摊计算）和池化（复用连接/线程减少开销）。异步队列削峰是把"瞬时高并发写"转成"稳定速率消费"，保护 DB 不被打爆。上了缓存仍要扩容——缓存miss时（如穿透/击穿）请求落到 DB，还是要足够机器扛住。四件套的动机是"应对不同瓶颈"：缓存（读瓶颈）、扩容（计算瓶颈）、池化（资源创建瓶颈）、队列（写峰值瓶颈）。组合使用才能全方位扛高并发，单一手段不够。

### 第二层：证据与定位

**Q：下单接口 P99 飙高，你怎么定位是该加缓存（DB 慢），还是该扩容（CPU 满），还是该改异步（链路长）？**

按资源瓶颈定位。三步：
1. 看 DB 负载——`SHOW PROCESSLIST` 看活跃连接数、慢查询。如果 DB CPU 高、慢查询多、连接池（HikariCP）的 `active == max`，是 DB 瓶颈，加缓存（挡读）或优化 SQL。
2. 看应用 CPU——如果应用 CPU 接近 100%、但 DB 负载正常，是计算瓶颈（应用自身算价/扣库存逻辑重），扩容（加 Pod）或优化算法。
3. 看链路耗时——APM 的 trace 看下单各 span 耗时，如果串行调 5 个下游（库存/风控/营销/积分/优惠券）各 50ms，总 250ms，是链路长，改异步（非核心的并行或异步化）。关键是用资源指标（DB CPU/应用 CPU/网络）和链路 trace 定位"瓶颈在哪层"，再决定用哪件套——缓存解 DB 瓶颈、扩容解 CPU 瓶颈、异步解链路瓶颈。

### 第三层：根因深挖

**Q：你发现是缓存击穿（热 key 过期瞬间大量请求打 DB），根因是什么？光是给热 key 加互斥锁就行吗？**

互斥锁是治标。根因是"热点 key 的失效时机集中"：
1. 热点 key 失效——某个商品详情缓存 TTL 到期，瞬时几万请求同时发现 miss，都去查 DB 重建缓存，DB 被打爆。互斥锁让"只有一个请求查 DB 重建，其他等待"，缓解但不治本。
2. 治本方案——热点 key "永不过期"（逻辑过期）：缓存的 value 带逻辑过期时间，物理上不设 TTL，由后台异步刷新。读时发现逻辑过期，返回旧值 + 触发异步刷新（不阻塞用户）。
3. 更根本——热点 key 的识别和预热。大促前用历史数据预测热销商品，提前预热缓存且 TTL 设得足够长（大促期间不过期）。`hot_key_detection`（热 key 检测，如基于访问频次）自动识别并特殊处理（加副本/分桶）。根因是"没识别热点 + 失效策略粗暴"，治本是"热点识别 + 永不过期 + 预热"。

**Q：那为什么不直接把所有缓存 TTL 设得很长（如 24 小时），不就没有失效问题了？**

TTL 长的问题是"数据陈旧"。商品价格/库存变了，缓存 24 小时不过期，用户看到旧价格/超卖库存。TTL 是"新鲜度和 DB 压力的权衡"——TTL 短（数据新鲜但 DB 压力大）、TTL 长（DB 压力小但数据陈旧）。正确策略是分级：高频且少变的（如商品基础信息）TTL 长（1 小时）；低频或易变的（如实时库存）TTL 短（10 秒）或不缓存（直接查）。热点 key 用"永不过期 + 主动刷新"（数据变更时主动更新缓存，不依赖 TTL）。一刀切设长 TTL 会导致数据陈旧（用户投诉价格不对），不是缓存设计的正道。一致性要求高的数据，配合"写库后删缓存 + binlog 兜底更新"，而非依赖 TTL。

### 第四层：方案权衡

**Q：你的多级缓存是 Caffeine（本地）+ Redis（分布式），但本地缓存在多实例下数据不一致（A 实例更新了本地，B 实例还是旧的），你怎么权衡"本地缓存性能"和"一致性"？**

本地缓存不一致是已知代价。权衡方案：
1. 本地缓存只存"低频变更"的数据——如商品基础信息（名称/图片）、配置类数据。这些数据即使各实例短暂不一致（几秒），业务影响小（用户看到稍旧的名称无感）。
2. 高频变更的数据（如实时价格/库存）不放本地缓存，只放 Redis——Redis 是单源，一致性更好。本地缓存的性能优势（亚毫秒）用在不敏感的数据上。
3. 本地缓存失效广播——数据变更时，通过 MQ/Redis Pub-Sub 广播"失效消息"，所有实例收到后清本地缓存。近似一致（广播有延迟，但秒级收敛）。适合"变更不频繁但需要及时生效"的数据。权衡是"本地缓存换性能 + 接受秒级不一致 + 广播收敛"，适合读多写少且容忍短暂不一致的场景，强一致场景别用本地缓存。

**Q：为什么不用纯 Redis 缓存（不要本地），一致性不是更好吗？**

纯 Redis 的问题是"每次读都要网络 RTT"。Caffeine 本地缓存是进程内（纳秒级），Redis 是网络（毫秒级）。对于"读极高频"的数据（如热门商品被每秒几十万次查），每次查 Redis 是几十万次网络调用，Redis 集群压力大且 RT 累加。本地缓存挡掉这几十万次（大部分命中本地），Redis 只在 miss 时查，QPS 降两个数量级。一致性上，本地缓存适合"不变或慢变"数据，配合"短 TTL + 失效广播"，短暂不一致可接受。纯 Redis 适合"一致性好但读频次可控"的场景。两者是性能和一致性的权衡，不是非此即彼。

### 第五层：验证与沉淀

**Q：你怎么验证四件套的配置在大促峰值下真的有效，而不是"平时够用，大促崩"？**

必须全链路压测 + 容量验证：
1. 压测——用预估的大促峰值 QPS 压下单链路，验证各层是否达标：缓存命中率（>95%）、DB QPS（在容量内）、应用 CPU（<70%）、Kafka 消费无积压。任何一层不达标，针对性优化。
2. 容量模型——记录"各层的容量上限"（DB 扛 X QPS、Redis 扛 Y QPS、应用单实例扛 Z QPS），按峰值预估所需实例数/分片数。大促前按模型预扩容。
3. 故障注入——压测中 kill 部分 Redis/DB/应用实例，验证四件套的容错（缓存降级、扩容接管、队列削峰）。光压正常峰值不够，要压"峰值 + 故障"场景。拼多多大促复盘的数据沉淀成容量基线，下次大促按基线准备。

**Q：四件套的配置怎么沉淀成规范，避免每个服务各自摸索（如缓存 TTL 乱设、线程池乱配）？**

靠"基础设施 + 规范"：
1. 缓存 SDK——封装多级缓存（Caffeine+Redis）的统一 SDK，业务侧用 `@Cacheable` 注解，TTL/一致性策略由 SDK 按数据类型默认配置（如"商品类默认 30 分钟 + 失效广播"）。业务不自己设 TTL。
2. 线程池规范——pdd-trade-014 题提到的线程池工厂 SDK，强制有界队列 + 命名 + 监控。池化（连接池）用统一配置模板（按服务类型分）。
3. 扩容基线——每个服务的 HPA 配置（min/max/阈值）按容量模型预设，大促前手动调高 min。不允许业务侧随意改 max（避免过度扩容浪费或不足）。
4. 队列使用规范——异步队列的接入（什么场景用 Kafka 削峰、什么用延时队列关单）标准化，消费端的幂等/限流/死信处理由 SDK 兜底。四件套是基础设施能力，业务侧"配置而非编码"地使用，这才是平台化的沉淀。

## 结构化回答

**30 秒电梯演讲：** 单机 QPS 有上限，如何让系统支撑突发大流量？简单说就是——池化（连接/线程）+ 缓存（多级）+ 扩容（弹性）+ 异步队列（削峰）是扛高并发四件套——用空间换时间、用队列换稳定。缓存：Caffeine + Redis 多级；扩容：无状态 HPA + 有状态分片。

**展开框架：**
1. **池化** — 池化：连接池/线程池/对象池
2. **缓存** — 缓存：Caffeine + Redis 多级
3. **扩容** — 扩容：无状态 HPA + 有状态分片

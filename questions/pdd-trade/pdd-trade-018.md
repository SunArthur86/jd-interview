# 【拼多多交易】ES 怎么做订单/商品搜索？

> JD 依据："熟悉 ES"。

## 一、倒排索引

```
分词 → 词项 → posting list（文档 ID + 词频 + 位置）
搜索"红色连衣裙"：红色 ∩ 连衣裙 → 命中文档
```

## 二、订单搜索实战

```json
POST /order/_search {
  "query": {
    "bool": {
      "must": [{"match": {"title": "手机"}}],
      "filter": [{"term": {"buyer_uid": 500}}, {"range": {"create_time": {"gte": "now-30d"}}}]
    }
  },
  "sort": [{"create_time": "desc"}]
}
```

## 三、Canal 同步

```
MySQL → Canal → Kafka → ES
订单变更自动刷 ES（秒级延迟，业务无感）
```

## 四、深分页

`from+size` 深分页差，用 `search_after`（游标）。

## 五、底层本质

倒排是"用空间换搜索效率"——O(n) LIKE → O(命中数) 倒排。

## 常见考点
1. **ES 比 MySQL LIKE 快**？——倒排按词定位，LIKE 全表扫。
2. **分片设计**？——单分片 < 50GB；按数据量分。
3. **近实时**？——写入 1s（refresh interval）后可搜。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：订单搜索你用 ES 而不是 MySQL，但订单数据本来就在 MySQL，多同步一份到 ES 不是增加一致性成本吗？为什么不让 MySQL 直接扛搜索？**

MySQL 扛不住多维搜索。订单搜索的查询模式是"全文（商品标题）+ 多条件过滤（买家/时间/状态）+ 排序+分页"，MySQL 的 `LIKE '%手机%'` 是全表扫不走索引，百万级订单单次查询秒级；多条件组合 MySQL 要建多个联合索引，组合爆炸。ES 的倒排索引让"手机"直接定位到命中文档（O(命中数)），且支持任意字段组合过滤（每个字段独立倒排）。同步成本（Canal + 秒级延迟 + 对账）是值得的代价，换取"搜索从秒级到毫秒级"。动机是"MySQL 擅长事务存储，ES 擅长搜索分析，各司其职"。如果只是按主键/单索引查，MySQL 够用，不需要 ES；多维搜索才是 ES 的战场。

### 第二层：证据与定位

**Q：订单搜索的 P99 从 50ms 飙到 2 秒，你怎么定位是 ES 查询本身慢，还是 ES 集群压力大？**

看 ES 的慢查询和节点负载。三步：
1. 开 ES 慢查询日志（`indices.search.slowlog.threshold.query.warn: 1s`）——看是不是有查询超过 1 秒，记录的 query 是什么。如果某个复杂 bool query（多 should/must + 聚合）频繁触发慢日志，是查询本身重。
2. 看 ES 节点监控（`_cat/nodes?v` 的 cpu/heap/disk）——如果 CPU 或 heap 接近 100%，是集群压力大（查询并发太多或 GC）。`_cat/thread_pool/search?v` 看 search 线程池的 queue 和 rejected，rejected 说明查询被拒绝（线程池满）。
3. 用 `_nodes/hot_threads` 看热点线程——如果都在做 query phase 或 fetch phase，定位到具体索引/分片。常见根因：深分页（from+size 的 from 大）、大聚合、或分片数据倾斜（某分片特别大）。重点是区分"单查询慢"（优化 query）和"集群满"（扩容或限流）。

### 第三层：根因深挖

**Q：你定位到是订单搜索的深分页（from=10000）导致慢，根因是什么？光是限制最大页数就行吗？**

ES 深分页慢的根因是"from+size 要协调节点从每个分片取 from+size 条，再合并排序取 size 条"——from=10000 size=10 意味着每个分片取 10010 条，N 个分片合计 N×10010 条在协调节点内存排序，CPU 和内存都爆。治本：
1. 用 `search_after`（游标分页）——每次查询带上一页最后一条的 sort 值，ES 从那个位置取下一页，O(size) 而非 O(from+size)。代价是不能跳页（只能顺序翻）。
2. 用 `scroll`（滚动查询）——适合"导出全部数据"场景，维护快照游标，但不适合实时搜索（快照会过期、占内存）。
3. 业务上限制最大 from——如只允许看前 1000 条（from < 1000），超过的引导用时间范围/条件筛选缩小结果集。限制页数是兜底，search_after 是技术解。拼多多订单搜索"我的订单"用 search_after（用户顺序翻），后台导出用 scroll。

**Q：那为什么不直接用 MySQL 的游标分页（WHERE id > lastId），非要 ES？**

MySQL 游标分页能解决"按主键翻页"，但订单搜索不是"按 id 翻页"，是"按相关性排序+多条件过滤"。比如搜"红色连衣裙"，结果按 BM25 相关性排序，没有单调递增的 id 可以游标（相关性分数相同的文档顺序不定）。ES 的 search_after 用"排序字段的 tuple"做游标（如 `[score, _id]`），保证游标的单调性，这是 ES 特有能力。另外 MySQL 的多条件过滤依然要扫大量行（即使有索引，组合查询也要回表），ES 的倒排让过滤在索引层完成。MySQL 游标分页只适合"按主键或单字段排序"的简单翻页，复杂搜索场景还是 ES。

### 第四层：方案权衡

**Q：你用 Canal 把 MySQL 订单同步到 ES，但 Canal 有秒级延迟，用户刚下的单立刻搜不到，你怎么权衡"实时性"和"解耦"？**

权衡方案是"分级查询 + 强一致兜底"：
1. 用户"我的订单"列表——容忍秒级延迟（刚下的单 1-2 秒后出现），走 ES。用户体验上"下单成功"是基于下单接口返回（不依赖搜索），回查列表稍延迟可接受。
2. 刚下的单的即时查询——按订单 id 查（主键查询走 MySQL，不走搜索），实时准确。用户下单成功后跳转"订单详情"是按 id 查，不依赖 ES。
3. 关键业务（如风控核对、对账）——不走 ES，走 MySQL 直查或 T+1 离线对账，保证强一致。ES 只服务"C 端搜索/列表"这种容忍延迟的场景。实时性和解耦的权衡是"按场景选数据源"，搜索走 ES（快但延迟）、关键查询走 MySQL（慢但准），不追求 ES 实时。

**Q：为什么不用"双写 MySQL + ES"（业务代码同时写两个），实时性不是更好吗？**

双写的问题前面说过：一是事务性难保证（MySQL 成功 ES 失败怎么办），二是每个写订单的地方都要记得写 ES（漏一个就不一致），三是双写的 ES 索引构建在业务请求链路里，增加下单 RT。Canal 监听 binlog 的优势是"业务无感知 + 顺序可靠 + 异步不阻塞主链路"。实时性的劣势（秒级延迟）用"按 id 查走 MySQL"兜底。双写换来的"实时"在大多数搜索场景没必要（搜索本就容忍延迟），反而引入一致性和性能问题。只有"搜索结果必须毫秒级实时"（如刚上架的商品立即搜到）才考虑双写或缩短 refresh interval（`index.refresh_interval=500ms`），但要权衡 ES 写入性能。

### 第五层：验证与沉淀

**Q：你怎么验证 MySQL 到 ES 的数据一致性，而不是"搜索结果偶尔不对但没人发现"？**

必须有对账机制：
1. 全量对账——定期（每天）跑 `MySQL count vs ES count`，以及抽样逐条对比（按订单 id 查 MySQL 和 ES，对比字段）。`reconcile_diff_count` 接入告警，差异率超阈值（如 >0.01%）告警。
2. 增量对账——Canal 同步链路加监控，binlog 消费延迟（`binlog_lag`）超过阈值告警。延迟大说明同步堆积，搜索结果会大面积"过期"。
3. 字段级一致性——重点对账"金额/状态"等关键字段，这些错了是资损（如搜索显示已支付但实际未支付）。数值型字段对比用 hash 比对（`MD5(concat(字段))` 批量比对）。ES 的 mapping 变更（如某字段从 text 改 keyword）也要测试，避免类型变更导致查询结果变化。

**Q：ES 的使用怎么沉淀成团队规范，避免"该用 ES 没用"或"滥用 ES"（如把 ES 当主存）？**

靠"选型规范 + 数据流规范"：
1. 选型清单——什么时候用 ES：全文搜索、多条件组合过滤、聚合分析。什么时候不用：主键查询（用 MySQL/Redis）、强一致读写（用 MySQL）、事务性（MySQL）。ES 是"派生存储"，不是主存。
2. 数据流规范——MySQL 是唯一写入源（Single Source of Truth），ES 只能通过 Canal/binlog 同步，禁止业务直接写 ES（避免双源不一致）。ES 的数据都可从 MySQL 重建（reindex）。
3. 索引设计规范——分片数（按数据量，单分片 <50GB）、mapping（字段类型/分词器/是否索引）、refresh_interval（按实时性需求）。新人按规范建索引，不随意。
4. 容量规划——ES 集群容量（节点数/heap/磁盘）按数据量预估，大促前预扩容。ES 是重资源组件，不规范使用会拖垮集群影响所有业务。

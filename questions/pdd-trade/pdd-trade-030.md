# 【拼多多交易】可观测性怎么治理跨团队？

> JD 依据："高可用"、"稳定性治理"。

## 一、三支柱

```
Metrics（指标）: Prometheus + Grafana，聚合数字（QPS/RT/错误率）
Logs（日志）   : ELK/Loki，明细事件（错误日志/业务流水）
Traces（链路） : SkyWalking/Jaeger，跨服务因果（一次请求全链路）
```

三支柱互补：
- Metrics 快速发现（告警触发）
- Traces 定位因果（哪个服务慢）
- Logs 查明细（错误堆栈）

## 二、统一协议（跨团队关键）

**traceId 全链路透传**：
```java
// HTTP 入口（网关）
String traceId = req.getHeader("X-Trace-Id");
MDC.put("traceId", traceId);

// 跨服务调用（Feign）
@Bean
public RequestInterceptor traceInterceptor() {
    return template -> template.header("X-Trace-Id", MDC.get("traceId"));
}

// 跨 Kafka
@KafkaListener(topics = "order")
public void on(ConsumerRecord<String, String> record) {
    MDC.put("traceId", record.headers().lastHeader("X-Trace-Id").value());
}
```

**日志格式统一**：所有团队 JSON 结构化日志，含 traceId/spanId/uid/接口。
```json
{"ts":"2026-07-08T12:00:00","traceId":"abc123","uid":"123","api":"createOrder","level":"ERROR","msg":"库存不足"}
```

## 三、SLO 驱动治理

**SLO 定义**（核心接口）：
```
下单接口：可用性 99.99%，p99 RT < 300ms
支付接口：可用性 99.99%，p99 RT < 500ms
```

**错误预算**：
```
30 天 99.99% 可用性 → 允许 4.3 分钟不可用
错误预算耗尽 → 冻结新功能发布，全力稳定性
```

**跨团队责任划分**：
- SLO 达标：负责团队 OK
- SLO 不达标：必须修复，不可推诿
- 跨团队故障：用 traceId 定位根因，按 SLO 协议定责

## 四、告警治理（防风暴）

```
原始告警（10万/天）
  ├─ 聚合（同根因合并）
  ├─ 降噪（已知问题/维护期）
  ├─ 分级（P0 电话/P1 钉钉/P2 邮件）
  └─ 值班（轮值，避免单人）
→ 有效告警（100/天）
```

```java
// AlertManager 告警收敛规则
groups:
  - name: trade-rt
    rules:
      - alert: TradeRtHigh
        condition: rt_p99 > 500ms for 2m
        annotations:
          summary: "交易 RT 高，traceId 见关联链路"
```

## 五、可观测性平台化

```
业务团队 → 接入 SDK（自动埋点）→ 可观测性平台
                                  ├─ Metrics 看板
                                  ├─ Logs 查询
                                  ├─ Traces 链路
                                  └─ 告警/预案联动
```

标准化：所有 Java 服务引 SDK 自动埋点，业务无感接入。

## 六、拼多多实战

- **全链路追踪**：SkyWalking，一次请求跨 30+ 服务，秒级定位慢点
- **SLO 平台**：核心接口 200+ SLO，错误预算驱动版本节奏
- **告警收敛**：从 10 万/天降到 100/天，值班不再麻木
- **大促作战室**：实时大盘（订单/支付/RT/错误率），异常秒级告警

## 七、底层本质

可观测性本质是**"用外部表现反推内部状态"**——三支柱覆盖数字/明细/因果，跨团队靠统一协议（traceId/SLO），治理靠错误预算和告警收敛。本质是把"黑盒系统"变"白盒"。

## 常见考点
1. **Metrics 和 Logs 区别**？——Metrics 聚合数字（便宜，适合告警），Logs 明细（贵，适合查因）。
2. **traceId 怎么保证全局唯一**？——网关生成（雪花/UUID），下游透传，Kafka header 携带。
3. **SLO 和 SLA 区别**？——SLO 内部目标（99.99%），SLA 对外承诺（赔款条款）。

## 苏格拉底式面试追问

> 这组追问不背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：可观测性你强调"跨团队统一协议（traceId/SLO）"，但每个团队技术栈和习惯不同，强制统一阻力大，为什么必须统一？**

跨团队故障定位时，不统一的代价是"无法关联"。用户下单失败，traceId 在交易团队是 abc123，到支付团队变成了 xyz789（各团队自己生成），没法把跨团队的日志/链路串起来，定位要靠人肉协调（交易找支付查、支付找库存查），耗时从分钟变小时。统一 traceId 透传的动机是"让一次请求在所有团队的系统里可追溯"——交易发订单事件带 traceId，支付/库存/履约都透传同一个 traceId，故障时一个 traceId 查全链路。这要求所有团队遵循同一协议（traceId 生成规则 + 透传方式 + 日志格式），是"基础设施级"的统一。阻力通过"平台 SDK 自动埋点"化解——业务团队引 SDK，traceId 自动透传，不感知细节，这样统一成本低。

### 第二层：证据与定位

**Q：用户投诉"下单失败"，但前端只返回了"系统繁忙"，你怎么用可观测性定位是哪个服务的问题？**

按 traceId 追链路。三步：
1. 拿到 traceId——如果前端有上报（请求头/响应体带 traceId），直接用。如果没有，按"用户 uid + 时间范围"在网关日志里反查 traceId。
2. 用 traceId 查链路追踪（SkyWalking/Jaeger）——看这次请求跨了哪些服务、每个 span 的 status 和 duration。如果某 span 是 `status=ERROR`（如库存服务返回失败），定位到库存。如果某 span duration 很长（如支付 3 秒），是支付慢导致整体超时。
3. 用 traceId 查日志（ELK）——在库存服务的日志里按 traceId 过滤，看具体的错误堆栈（如 `InsufficientStockException`）。三支柱协作：Metrics 发现"错误率涨了"（告警）、Traces 定位"哪个服务出错"（链路）、Logs 查"具体什么错"（明细）。没有 traceId 贯穿，这三步无法关联，定位效率极低。

### 第三层：根因深挖

**Q：你发现告警太多（每天 10 万条），值班麻木了真故障也被淹没，根因是什么？光是把告警阈值调高就行吗？**

调高阈值会漏掉真告警。根因是"告警缺乏收敛和分级"：
1. 同一根因的告警重复发——库存服务挂了，依赖它的 20 个服务都报"调用库存超时"，产生 20 条告警，实际是 1 个根因。治本是"告警聚合"——按根因（或关联服务）合并，只发 1 条"库存服务故障，影响 20 个下游"。
2. 告警不分级——P0（下单全挂）和 P3（某边缘接口 RT 略高）用同样渠道（都发钉钉），值班被打扰到麻木。治本是"分级通知"——P0 电话叫醒、P1 钉钉、P2 邮件、P3 只记录不通知。
3. 已知问题反复告警——某个已知 bug 在修复中，但告警每天发。治本是"告警静默"（维护期/已知问题静默）。根因是"告警治理缺失"，治本是"聚合 + 分级 + 静默"，把 10 万条降到 100 条有效告警，值班才能专注真问题。

**Q：那为什么不直接只对"用户影响"告警（如 SLO 违规），而非对"系统指标"（CPU/RT）告警？**

两种告警都要，职责不同：
1. SLO 告警（用户影响）——如"下单成功率 <99.9%"，这是"最终结果"，直接关系用户体验，最高优先级。但 SLO 告警是"滞后"的——等 SLO 违规时，用户已经受影响了。
2. 系统指标告警（前兆）——如"CPU >80%""RT P99 上升"，这是"前兆信号"，在 SLO 违规之前就预警，给处理时间。如果只看 SLO，等告警时已经晚了。
3. 分层告警——系统指标告警作为"早期预警"（P2，关注但不紧急），SLO 告警作为"业务影响"（P0，立即处理）。两者配合：指标异常时先排查（防患未然），SLO 违规时立即救火。只留一种都会失衡——只 SLO 太滞后，只指标太噪声。关键是告警的"信噪比"和"分级"，而非"留哪种"。

### 第四层：方案权衡

**Q：你的 SLO 是"下单可用性 99.99%"，错误预算驱动"预算耗尽则冻结发布"，但这会阻塞业务迭代，业务方肯定反对，你怎么权衡"稳定性"和"迭代速度"？**

错误预算的本质就是"权衡稳定性和迭代"的工具：
1. 错误预算是"定量化的权衡"——99.99% 可用性 = 每月 4.3 分钟可容忍不可用。这 4.3 分钟是"预算"，可以"花"在发布新功能（发布可能引入 bug 导致故障）上。预算充足时鼓励迭代（快速发布），预算紧张时收敛（冻结发布保稳定）。
2. 与业务对齐——如果业务有大促/重要节点，提前预留预算（大促前一个月冻结非必要发布）。平时迭代快一点（消耗预算），关键节点保守。这是"按业务节奏分配预算"，不是"永远冻结"。
3. 错误预算的激励——团队有动力减少故障（省预算给迭代），而非"反正能发布就发布"。如果某团队频繁故障耗尽预算，被冻结是"自然后果"，业务方也会支持（谁也不想发布一个挂掉的功能）。权衡是"用预算这个量化工具，让稳定性和迭代可视化、可协商"，而非"稳定性团队 vs 业务团队"的对立。

**Q：为什么不把 SLO 定到 100%（绝对不挂），这样不就最稳吗？**

100% SLO 不可行也不经济。99.99% 意味着每月 4.3 分钟不可用，要做到 100% 要付出指数级成本——更多冗余、更多多活、更多人工 review，且软件系统总有未知 bug 和外部依赖故障（第三方支付/网络），不可能 100%。即使 Google 也只承诺 99.99%。100% 的追求会导致：成本爆炸（机器冗余几十倍）、迭代停滞（任何变更都有风险，干脆不变更）、过度工程（为 0.001% 的故障做巨大投入）。99.99% 是"经济最优"的权衡点——剩余的 0.01% 用"快速恢复（预案/回滚）"兜底，而非"绝对预防"。SLO 的设计是"可接受的故障率 + 快速恢复"，不是"零故障"。

### 第五层：验证与沉淀

**Q：你怎么验证可观测性系统本身能工作（"眼睛"没瞎）？如果监控系统自己挂了，故障时双盲。**

监控系统自身要有冗余和监控：
1. 监控的监控——Prometheus/ELK 本身有指标上报（采集成功率/查询延迟/存储健康），如果 Prometheus 采集失败（exporter 挂），要有"元监控"告警（监控另一个 Prometheus 监控这个 Prometheus）。
2. 冗余——监控系统多副本部署，数据多副本存储。采集层有缓冲（采集失败先本地缓存，恢复后补传）。
3. 降级预案——监控挂了怎么办？核心服务有"本地日志兜底"（即使集中式日志系统挂，本地文件日志还在，手动查）。告警通道冗余（钉钉挂了走电话/短信）。监控系统的 SLO 应该比业务系统更高（它是眼睛，不能先瞎）。

**Q：可观测性怎么沉淀成"所有团队自动接入"的能力，而不是每个团队自己搞？**

沉淀成"可观测性平台 + 强制 SDK"：
1. 统一 SDK——所有 Java/Go/Python 服务引官方 SDK，自动埋点（HTTP/RPC/DB/Redis 的 metrics + trace），业务无感。日志用统一格式（JSON + traceId）。不引 SDK 的服务不允许上线（CI 校验）。
2. 平台化能力——Metrics（Prometheus + Grafana 模板，按服务自动生成看板）、Logs（ELK 统一采集 + 按团队隔离的查询权限）、Traces（SkyWalking 自动拓扑）。业务团队只管"用"，不管"建"。
3. SLO 平台——核心接口的 SLO 在平台定义，自动计算错误预算，自动告警 SLO 违规，自动驱动发布门禁（预算耗尽冻结）。SLO 的定义由业务和稳定性团队共建，平台自动执行。
4. 告警治理平台——所有团队的告警进统一平台，自动聚合/分级/降噪，按值班表路由。告警规则版本化，变更可审计。这是大厂"平台化稳定性能力"的核心——可观测性是基础设施，不是每个团队的自留地。

## 结构化回答



**30 秒电梯演讲：** 可观测性像医院的"监护系统"——Metrics 是心率血压（数字）、Logs 是病历（事件明细）、Traces 是会诊记录（跨科室因果链）。

**展开框架：**
1. **三支柱** — Metrics（Prometheus）/Logs（ELK）/Traces（SkyWalking）
2. **跨团队** — 统一 traceId 透传 + SLO 协议 + 告警收敛
3. **SLO** — 错误预算驱动优先级

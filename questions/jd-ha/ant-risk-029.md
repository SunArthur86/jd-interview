# 【蚂蚁风控】高可用——预案、回滚、应急响应怎么做？

> JD 依据："保障海量数据系统的稳定性"。这是故障发生后的关键。

## 一、预案（Contingency Plan）

**作用**：提前规划"什么场景怎么应对"，让故障响应标准化而非凭经验。

**预案分级**：

| 级别 | 场景 | 响应方式 | 风险等级 |
|------|------|---------|---------|
| **P0** | 决策服务不可用、风控全挂 | 电话叫全员，5 分钟响应 | 紧急 |
| **P1** | 拦截率骤变、误杀大面积 | 短信通知 oncall | 高 |
| **P2** | 单实例异常、P99 RT 升高 | IM 通知 | 中 |
| **P3** | 日志报错、监控告警 | 自动化处理 | 低 |

**预案文档结构**：
```markdown
# 预案 R-001：风控决策服务不可用

触发条件：
  - 决策接口 5xx 错误率 > 10%
  - 决策 P99 RT > 2s 持续 3 分钟

应急步骤：
  1. 立即通知 oncall（电话）
  2. 检查大盘，确认是否全量受影响
  3. 紧急降级：打开"全部走兜底规则"开关
  4. 排查：jstack/jmap 看是否死锁/OOM
  5. 若无法立即修复，回滚最近发布版本
  6. 恢复后观察 30 分钟
  7. 复盘

责任人：risk-oncall
预案版本：1.2.0
```

## 二、回滚（Rollback）

**回滚类型**：

### 1. 代码回滚
最简单，发布新版本发现 bug，立即发布旧版本。

```bash
# K8s 一键回滚
kubectl rollout undo deployment/risk-decision
```

**前提**：发布用蓝绿/金丝雀，不直接覆盖。

### 2. 配置回滚
配置中心（Nacos）有版本管理，一键切回。

```bash
# Nacos 配置历史，一键回滚
nacos config rollback --dataId=risk-rules --version=42
```

### 3. 数据回滚（最难）
代码回滚了但数据已经按新逻辑改了，怎么办？

**风控的案例**：
- 新规则误杀了一批用户（错误拦截）
- 数据库里这些用户的状态被改了（如风控分）
- 回滚代码不能恢复已改的数据

**解法**：
- **数据备份**：变更前备份影响的数据
- **审计日志**：记录每次变更（binlog）
- **逆向 SQL**：根据变更日志生成逆操作

### 4. 流量切换
切到备份机房/集群。

```bash
# DNS 切换（流量切到备机房）
dns update risk.alipay.com --ip=backup-vip
```

## 三、回滚的坑（风控特别注意）

**1. 代码回滚 + 数据兼容**
```
版本 2.0 改了数据库 schema（加字段）
回滚到版本 1.0（不认识新字段）→ 报错

→ 回滚前必须验证数据兼容性
```

**2. 规则变更影响历史决策**
```
新规则把一批用户判为高风险
这些用户已经被限制（不能交易）
回滚规则不能自动解除限制

→ 历史决策要"冻结"，回滚只影响未来决策
→ 被误杀的用户需要批量申诉恢复
```

**3. 模型灰度切流**
```
新模型 30% 流量，发现有问题
要立即切回旧模型

→ 流量切回后，新模型的决策记录要标记
→ 不能直接覆盖（影响审计）
```

## 四、应急响应流程

**五步法**：
```
1. 发现（Detect）
   - 监控告警触发
   - 用户反馈
   - 巡检发现

2. 通知（Notify）
   - 自动告警（按级别通知）
   - 创建事故群
   - 召集相关人

3. 定位（Locate）
   - 看监控大盘（哪个环节）
   - 看链路追踪（哪个服务慢/错）
   - 看日志（具体错误）

4. 止损（Mitigate）
   - 优先止损而非根治
   - 降级、回滚、切流
   - 5 分钟内必须有动作

5. 恢复（Recover）
   - 验证业务正常
   - 观察 30 分钟
   - 关闭事故群

6. 复盘（Postmortem）
   - 5Why 根因分析
   - 改进措施
   - 责任（无指责文化）
```

**应急响应原则**：
- **5 分钟内止损**：宁可有损也要快速恢复
- **优先恢复业务**：不是找 bug，是让用户能用
- **统一指挥**：一个 incident commander 决策
- **透明沟通**：定期同步状态（用户/老板/团队）

## 五、SRE 黄金指标

| 指标 | 含义 | 优化目标 |
|------|------|---------|
| **MTTD**（Mean Time to Detect） | 平均发现时间 | < 1 分钟（自动化告警） |
| **MTTI**（Mean Time to Identify） | 平均定位时间 | < 5 分钟（看监控） |
| **MTTF**（Mean Time to Fix） | 平均修复时间 | < 30 分钟 |
| **MTTR**（Mean Time to Recover） | 平均恢复时间 | < 5 分钟（止损） |
| **MTBF**（Mean Time Between Failures） | 平均故障间隔 | 越长越好 |

**SRE 的核心目标**：
- 减少 MTT*（更快响应）
- 增加 MTBF（更少故障）
- 错误预算（Error Budget）：99.99% 可用性 = 每月 4.3 分钟不可用

## 六、风控的实战预案

**预案 1：决策服务全挂**
```
触发：决策接口 5xx > 50%
动作：
  1. 自动切流到备机房（30 秒）
  2. 启动降级开关（全部走兜底规则）
  3. 排查根因
恢复标准：连续 5 分钟无错误
```

**预案 2：模型推理全挂**
```
触发：模型推理错误率 > 30%
动作：
  1. 自动熔断模型服务
  2. 决策融合退化为规则决策
  3. 排查模型服务（GPU 是否满）
恢复标准：模型推理错误率 < 1%
```

**预案 3：规则误杀大面积**
```
触发：拦截率突增 > 50%
动作：
  1. 立即回滚最近规则变更（Nacos 版本）
  2. 通知运营评估影响范围
  3. 被误杀用户批量恢复
恢复标准：拦截率恢复正常
```

## 七、混沌工程（Chaos Engineering）

**主动制造故障验证预案有效**：

```
1. 杀实例演练（验证 K8s 自愈）
   - 随机 kill pod，验证自动重启

2. 网络隔离演练（验证多机房容灾）
   - 模拟机房断网，验证切流

3. 依赖故障演练（验证降级生效）
   - 注入下游延迟，验证熔断

4. 资源耗尽演练（验证限流保护）
   - 模拟 CPU 满，验证限流
```

**Netflix Chaos Monkey** 是经典工具，蚂蚁有"故障演练平台"。

## 八、监控与告警

**告警设计原则**：
- **信噪比高**：少而准（避免告警疲劳）
- **分级明确**：P0/P1/P2/P3 不同通知方式
- **可执行**：告警带 runbook（怎么处理）
- **去重聚合**：相关告警合并

**风控告警示例**：
```yaml
- alert: RiskDecisionDown
  expr: rate(risk_decision_total[1m]) == 0
  for: 2m
  labels: { severity: page }
  annotations:
    summary: "风控决策全停"
    runbook: "https://wiki/.../R-001"

- alert: RiskInterceptAnomaly
  expr: |
    rate(risk_decision_total{result="reject"}[5m]) /
    rate(risk_decision_total[5m]) > 0.5
  for: 5m
  labels: { severity: page }
  annotations:
    summary: "拦截率异常 >50%"
    runbook: "https://wiki/.../R-003"
```

## 九、底层本质：从"避免故障"到"快速恢复"

传统运维思维：**避免故障**（严防死守）
SRE 思维：**接受故障，快速恢复**（错误预算）

**核心转变**：
- 100% 可用性不可能（成本无限）
- 设定 SLO（如 99.99%）
- 在错误预算内可以发新版（创新）
- 超出预算停止发布（稳定）

**风控的特殊性**：
- 金融场景要求更高（钱的事）
- 但也不能 100%（无法迭代）
- 错误预算更紧（如 99.995%）

**回滚和预案的本质**：
- 不是"避免出问题"
- 而是"出问题时如何快速恢复"
- 把响应标准化（预案）+ 可执行（回滚）

## 十、AI 时代的演进

**LLM 服务的应急预案新挑战**：
- LLM 推理慢、不可预测（幻觉）
- 模型版本切换有副作用
- token 成本超支

**对应方案**：
- 预案：LLM 服务故障降级到规则
- 回滚：模型版本一键切回（影子流量验证后再切）
- 应急：token 消耗告警（防成本爆炸）

## 常见考点
1. **代码回滚和数据回滚冲突怎么办**？——评估兼容性；不兼容用灰度恢复；必要时数据修复脚本。
2. **预案怎么保证有效**？——定期演练（chaos engineering）、runbook 详细、自动化执行。
3. **风控误杀怎么恢复**？——回滚规则、批量恢复用户、补偿（如误扣的钱退回）。

**代码示例**（自动化回滚触发器）：
```java
@Scheduled(every = "10s")
public void autoRollback() {
    if (errorRateMonitor.isHigh()) {
        // 错误率超阈值，自动触发回滚
        DeployVersion current = deployClient.getCurrent();
        DeployVersion previous = deployClient.getPrevious();
        if (previous != null) {
            log.warn("Auto rollback: error rate high, rolling back to {}", previous);
            deployClient.rollback(previous);
            alertClient.sendP1("自动回滚已触发: " + current + " -> " + previous);
        }
    }
}
```

# 【蚂蚁风控】怎么设计一个 99.99% 可用性的亿级 QPS 风控架构？

> JD 依据："保障海量数据系统的性能和稳定性"。这是高可用的综合架构题。

## 一、需求拆解

**可用性目标**：
- 99.99%（年停 52 分钟、月停 4.3 分钟）
- 风控是金融核心，不能漏判
- 故障要在秒级到分钟级恢复

**挑战**：
- 单机房挂（电力、网络、火灾）
- 单实例挂（OOM、Bug、硬件）
- 网络抖动（延迟、丢包）
- 数据中心级灾难（地震、洪水）

## 二、高可用三要素

### 1. 冗余（Redundancy）—— 多副本
```
单点故障率: 1%
↓ 加 1 个副本（互备）
故障率: 1% × 1% = 0.01%
↓ 加 2 个副本
故障率: 1% × 1% × 1% = 0.0001%
```

**冗余层次**：
- 实例冗余：多实例（K8s 多副本）
- 机架冗余：跨机架调度
- 机房冗余：多机房部署
- 区域冗余：多区域（异地）

### 2. 隔离（Isolation）—— 爆炸半径
```
故障必然发生，但限制影响范围
```
- 业务隔离：借贷和支付独立集群
- 资源隔离：每个下游独立线程池
- 故障域隔离：单元化（按 UID 切）

### 3. 自愈（Self-Healing）—— 自动恢复
- 健康检查 + 自动重启
- 流量自动摘除故障实例
- 自动故障切换（主从切换）

## 三、多机房架构

### 同城双活（强一致）

```
上海机房 A（主）──专线── 上海机房 B（备）
   50km 内
   延迟 < 5ms（强一致可行）
```

**特点**：
- 跨机房强一致（同步复制）
- 单机房挂不影响业务
- 适合金融核心

**数据同步**：
- MySQL：半同步复制（主写 B 收到 ACK 才返回）
- Redis：Redis Cluster 跨机房
- 配置：Nacos 跨机房集群

### 异地灾备（最终一致）

```
上海主集群 ──异步── 杭州灾备集群
   1000km
   延迟 30ms（强一致不可行）
```

**特点**：
- 异步复制（秒级延迟）
- 大灾难时切换
- 数据可能丢失几秒（业务可接受）

### 三地五中心（极致）

```
上海 + 上海 + 杭州 + 深圳 + 北京
```
**蚂蚁的实践**：五副本多数派可用（CAP 中的 CP）。

## 四、单元化架构（蚂蚁的精髓）

**单元化**：按 UID 切分单元，每单元独立完整链路。

```
单元 A（UID 0-1000万）
   ├─ 风控决策服务
   ├─ 特征服务
   ├─ 规则服务
   ├─ Redis（A 的数据）
   └─ MySQL（A 的数据）

单元 B（UID 1000万-2000万）
   └─ 完整独立链路

单元 C（UID 2000万+）
   └─ 完整独立链路
```

**路由**：
```java
public Unit route(String uid) {
    int unitNo = hash(uid) % UNIT_COUNT;
    return units[unitNo];
}
```

**好处**：
- 爆炸半径到单元级（一个单元挂只影响 1/N 用户）
- 横向扩展（加单元加容量）
- 数据分散（每单元数据量小）

**挑战**：
- 跨单元调用复杂（理论上不该发生）
- 单元间数据同步（全局表）
- 单元迁移（用户跨单元）

## 五、容量冗余

**N+1 / N+2 冗余**：
```
日常 QPS 5万，单机 5000 QPS
理论需要 10 台，实际部署 11-12 台

挂 1-2 台仍能撑住日常
```

**弹性扩容**：
```yaml
# K8s HPA
minReplicas: 50
maxReplicas: 500
```

**预留 buffer**：
- CPU 利用率 < 60%
- 内存 < 70%
- 连接池 < 80%

## 六、数据高可用

### MySQL 高可用
```
主库（Master）
   │ 半同步复制
   ├─ 同城从库（同步）
   └─ 异地从库（异步）

故障切换：MHA / Orchestrator 自动选新主
```

### Redis 高可用
```
Redis Cluster
   ├─ 主节点（slot 0-16383 分片）
   ├─ 从节点（每主一从）
   └─ 哨兵（自动故障检测和切换）
```

### HBase 高可用
```
HBase RegionServer 多副本
HDFS 三副本（数据不丢）
WAL 保证写不丢
```

## 七、服务高可用

**1. 健康检查**：
```yaml
livenessProbe:    # 存活探针（挂了重启）
  httpGet:
    path: /health
  initialDelaySeconds: 30
  periodSeconds: 10

readinessProbe:   # 就绪探针（不就绪摘流量）
  httpGet:
    path: /ready
```

**2. 优雅上下线**：
```java
@PreDestroy
public void shutdown() {
    // 1. 通知注册中心下线（不再接新流量）
    nacosClient.deregister();
    // 2. 等待存量请求处理完
    Thread.sleep(10000);
    // 3. 关闭资源
    dataSource.close();
}
```

**3. 流量调度**：
```
SLB（负载均衡）→ 健康实例
   ↓ 实例挂
自动摘除（健康检查）
```

## 八、降级链路（保核心）

**层层降级**：
```
正常路径（所有服务正常）
   ↓ 特征挂
降级 1：用缓存特征
   ↓ 模型挂
降级 2：用规则决策
   ↓ 规则挂
降级 3：用兜底规则（黑名单 + 阈值）
   ↓ 全挂
降级 4：业务优先（默认放行）或风险优先（默认拦截）
```

**风控的选择**：
- 大部分场景：风险优先（宁错杀不漏报）
- 关键场景：业务优先（用户体验）

## 九、可观测性（高可用的眼睛）

**实时监控**：
- 业务：QPS、P99、拦截率、错误率
- 系统：CPU、内存、GC、连接池
- 依赖：Redis、HBase、MySQL 健康

**告警分级**：
- P0：业务不可用 → 电话
- P1：核心指标异常 → 短信
- P2：单实例异常 → IM

**全链路追踪**：
- SkyWalking 跟踪每笔决策
- 故障秒级定位

## 十、容量规划与压测

**容量公式**：
```
日常 QPS × 峰值系数（2-5）× 冗余系数（1.5）= 设计容量
```

**风控的容量**：
- 日常 5 万 QPS
- 双 11 峰值 100 万 QPS（× 20）
- 设计容量 150 万 QPS（× 1.5 冗余）
- 单机 5000 QPS → 300 实例

**全链路压测**：
- 影子流量（不打扰用户）
- 验证容量上限
- 找出瓶颈服务
- 验证降级生效

## 十一、底层本质：可用性的数学与工程

**可用性的数学**：
```
单实例可用性: 99%
两个独立副本: 1 - (1-0.99)^2 = 99.99%
三个独立副本: 1 - (1-0.99)^3 = 99.9999%
```

**关键假设**：副本必须独立故障（不同机房、不同机架、不同硬件批次）。

**工程实现**：
- 独立性：故障域隔离（机房、机架、电源）
- 检测：监控告警
- 切换：自动故障转移
- 验证：混沌工程

**这是"用工程手段逼近数学极限"**：
- 理论：3 个 99% 副本 = 99.9999%
- 工程：监控 + 自动切换 + 验证
- 实际：99.99%（工程损耗）

**蚂蚁的实践**：
- 三地五中心（机房级冗余）
- 单元化（爆炸半径到单元）
- 自动化运维（故障秒级切换）
- 混沌工程（主动验证）

## 十二、AI 时代的高可用

**LLM 服务的高可用新挑战**：
- GPU 资源稀缺（冗余成本高）
- 推理慢（故障检测延迟）
- 模型版本切换有副作用

**对应方案**：
- 冗余：多 GPU 集群 + 多模型版本
- 降级：LLM 挂了降级到规则
- 验证：LLM 输出验证（防幻觉）

## 常见考点
1. **99.99% 怎么计算**？——年停 = 365×24×60×0.01% ≈ 52 分钟。
2. **同城和异地多活选哪个**？——强一致场景（金融）选同城双活；容忍最终一致选异地多活。
3. **单元化最难的是什么**？——跨单元调用（理论上不该有，实际业务复杂）+ 全局数据同步。

**代码示例**（健康检查）：
```java
@RestController
public class HealthController {

    @GetMapping("/health")
    public ResponseEntity health() {
        // 检查关键依赖
        if (!redisTemplate.getConnectionFactory().getConnection().isOpen()) {
            return ResponseEntity.status(503).body("redis down");
        }
        return ResponseEntity.ok("ok");
    }

    @GetMapping("/ready")
    public ResponseEntity ready() {
        // 检查是否就绪（预热完成、配置加载完）
        if (!appReady) {
            return ResponseEntity.status(503).body("not ready");
        }
        return ResponseEntity.ok("ready");
    }
}
```

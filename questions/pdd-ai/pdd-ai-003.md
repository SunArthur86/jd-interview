# 【拼多多 AI 中台】JVM GC 原理与模型服务调优怎么做？

> JD 依据："Java + 微服务、模型服务、消费者服务策略算法中台"。

## 一、GC 基础：找活对象

**可达性分析**——从 GC Root 出发，可达即活：
```
GC Roots:
  - 虚拟机栈中的局部变量
  - 方法区的静态字段
  - 方法区的常量
  - 本地方法栈 JNI 引用
  - 活动线程
  - 同步锁持有的对象
```

## 二、分代模型

```
┌──────────────────────────────────────┐
│ 堆 Heap                              │
│  ┌──────────────────┐ ┌────────────┐ │
│  │ 新生代 (1/3)     │ │ 老年代 2/3 │ │
│  │ ┌────┬────┬────┐ │ │            │ │
│  │ │Eden│ S0 │ S1 │ │ │            │ │
│  │ │ 8  │ 1  │ 1  │ │ │            │ │
│  │ └────┴────┴────┘ │ │            │ │
│  └──────────────────┘ └────────────┘ │
└──────────────────────────────────────┘
```

- 新生代：复制算法（Eden 满 → 存活进 S，清空 Eden）
- 老年代：标记-清除/标记-整理（防碎片）

**晋升条件**：年龄 ≥ 阈值（默认 15）/ 大对象直接进老年代 / S 区放不下。

## 三、收集器演进

| 收集器 | 算法 | STW | 适用 |
|--------|------|-----|------|
| Serial | 复制/整理 | 全程 | 单核小应用 |
| Parallel | 复制/整理 | 全程（吞吐优） | 批处理 |
| CMS | 标记清除 | 初始/重新标记 | 老年代低延迟（已弃） |
| **G1** | 整堆 Region 化 | 可预测暂停（200ms） | **默认（9+），通用** |
| **ZGC** | 染色指针+并发整理 | <10ms | 大堆低延迟（11+） |
| Shenandoah | 读屏障并发整理 | <10ms | RedHat |

**G1 关键**：堆分 2048 个 Region（每个 1-32MB），优先回收"垃圾最多"的 Region（Garbage First）。

## 四、模型服务 GC 调优实战

**场景**：LLM 推理网关，请求大（prompt 几 KB），并发高，对 P99 敏感。

**JVM 参数**：
```bash
java -Xms8g -Xmx8g \                   # 堆固定（避免动态扩缩 GC 抖动）
     -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=100 \         # 目标暂停 100ms
     -XX:G1HeapRegionSize=16m \
     -XX:InitiatingHeapOccupancyPercent=45 \  # 老年代占用 45% 触发并发标记
     -XX:G1NewSizePercent=30 \          # 新生代下限
     -XX:G1MaxNewSizePercent=50 \
     -XX:+ParallelRefProcEnabled \
     -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/data/dumps/ \
     -jar inference-gateway.jar
```

**调优思路**：
1. **堆固定 Xms=Xmx**：避免堆动态变化导致频繁 GC。
2. **大堆（>32G）用 ZGC**：G1 在大堆暂停会上升。
3. **新生代比例**：模型请求对象偏大，新生代要够（30-50%），否则快速晋升老年代触发 Full GC。
4. **大对象**：超 RegionSize/2 直接进老年代，监控避免大批大对象。

## 五、排查 Full GC 频繁

```
1. jstat -gcutil <pid> 1000  → 看 O(老年代) 是否持续上涨
2. jmap -histo:live <pid>    → 看对象 top
3. jmap -dump:format=b,file=heap.hprof <pid> → MAT 分析
4. 常见根因：
   - 内存泄漏（静态集合只增不减）
   - 大对象（缓存无上限、批次过大）
   - 新生代太小（频繁晋升）
   - System.gc() 被调用（加 -XX:+DisableExplicitGC）
```

## 六、底层本质

GC 本质是**"用 CPU 换内存安全 + 用 STW 换一致性"**——分代让回收频率匹配对象生命周期，并发收集把 STW 压到最小。模型服务对延迟敏感，要选 G1/ZGC + 堆固定 + 新生代足够大，避免大对象冲击。

## 常见考点

1. **对象什么时候进老年代**？——年龄达阈值/大对象/S 区放不下/动态年龄判断。
2. **CMS 为什么被替换**？——碎片严重 + 并发失败（Concurrent Mode Failure）退化 Full GC。
3. **怎么减少 GC**？——对象池化（避免建销）、缓存有上限、大对象预分配、Stream 懒求值。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你们 LLM 推理网关为什么选 G1 而不是 ZGC？推理服务对延迟敏感，ZGC 的 STW 不到 1ms，不是更合适吗？**

因为我们的堆才 8G，G1 在这个规模下 Young GC 停顿已经能压到 40ms 以内，满足 P99 < 200ms 的 SLA。ZGC 的优势在大堆（>32G）下停顿不随堆增长，但它的吞吐代价是 10-15%——同样 QPS 要多 15% 机器。我们做过实测：8G 堆下 G1 和 ZGC 的 P99 延迟差距只有 20ms，但 ZGC 的单卡 QPS 低 12%。决策依据不是"哪个更新更先进"，是看 SLO 要求（P99 < 200ms，G1 够）、堆规模（8G，G1 不吃力）、当前 GC 日志的 FGC 频率（治理后几乎为 0）。只有当堆要扩到 32G+ 或 SLA 收紧到 P99 < 50ms 时，才考虑切 ZGC。

### 第二层：证据与定位

**Q：推理网关 P99 从 80ms 飙到 600ms，你怎么确认是 GC 问题，而不是 Triton 推理变慢或网络抖动？**

三组证据交叉验证。第一，`jstat -gcutil <pid> 1000` 连续采样，看 YGC（Young GC 次数）和 YGCT（Young GC 总耗时）列在 P99 抖动时段是否同步跳变——如果 YGCT 单秒增加 200ms，且时间点和 P99 抖动吻合，初步锁定 GC。第二，看 GC 日志（`-Xlog:gc*:file=gc.log:time`），用 GCEasy 工具分析，确认是否有单次 Young GC 停顿超过 100ms 的长尾。第三，看 APM trace——P99 抖动期间，同一请求的 `triton_invoke_ms` 和 `network_ms` 是否正常，如果下游耗时正常但应用自身 `self_process_ms` 变长，排除下游问题，GC 嫌疑最大。第四，看 `jstat -gccause` 的最后一列（LGCC——最近一次 GC 原因），如果是 `Allocation Failure` 且频率高，就是 GC 压力。

### 第三层：根因深挖

**Q：你发现 Young GC 频率从每秒 0.5 次变成每秒 3 次，每次停顿 80ms。根因是什么？单纯是 Eden 区太小吗？**

不一定是 Eden 太小。`jstat -gcutil` 看 E（Eden 使用率）——如果每次 GC 后 E 从 100% 降到 0% 又快速涨回 100%，且 S0/S1（Survivor）使用率高、O（Old）持续增长，说明是"大对象或长生命周期对象快速晋升老年代"。LLM 推理网关的典型根因是：每个推理请求会构造一个大的 `InferRequest` 对象（prompt token 序列 + features，可能几百 KB 到几 MB），这些对象在 Eden 区分配，如果请求处理时间跨过一次 Young GC，对象被复制到 Survivor 再晋升 Old，很快撑爆老年代触发 Full GC。用 `jmap -histo:live <pid> | head -20` 看对象 Top，如果 `byte[]` 和 `InferRequest` 占大头，就是这个原因。

**Q：那为什么不直接把 Eden 区调到 4G？Eden 大了 Minor GC 频率不就低了吗？**

Eden 调大能降频率，但有上限和代价。第一，堆固定 8G，Eden 调到 4G 就挤压 Old 区，老年代变小反而更容易 Full GC（Full GC 停顿秒级，比 Young GC 严重得多）。第二，Eden 越大，单次 Minor GC 要复制的存活对象越多，停顿时间变长（G1 的 `MaxGCPauseMillis=100ms` 是软目标，Eden 太大会被打破）。正确做法是先治本——给 `InferRequest` 对象做池化复用（避免每次 new 大对象），或把 prompt token 序列改成 `ByteBuffer` 直接内存分配（不进堆），从源头减少 Eden 压力。Eden 比例调到 30-40%（`-XX:G1NewSizePercent=30 -XX:G1MaxNewSizePercent=50`）是经验值，不是越大越好。

### 第四层：方案权衡

**Q：你把 InferRequest 改成对象池复用，Young GC 频率降了，但同事 review 时说"对象池会增加内存占用且有线程安全问题"。这个权衡你怎么看？**

对象池的代价是"常驻内存"（池里预建 N 个对象一直占堆）和"并发安全开销"（借还对象要加锁或 CAS）。对 InferRequest 这种"大对象 + 高频创建 + 生命周期短"的场景，池化收益大于代价：省下的 GC 停顿（每次 80ms × 每秒 3 次 = 240ms/s 的 CPU 时间）远超池化本身的开销（CAS 借还纳秒级）。但如果对象很小（比如一个 Integer），池化反而亏——池的管理对象比被池对象还大。判断标准：对象创建成本 > 10us 或对象 > 1KB 才值得池化。线程安全方面，用 Netty 的 Recycler（无锁 + ThreadLocal 缓存）比传统 Apache Commons Pool 的加锁池性能好 10 倍。

**Q：为什么不直接换 ZGC 或 Shenandoah 彻底消除 GC 停顿？这样对象池也不用做了。**

换 ZGC 是治标不治本。对象在堆里指数级增长，即使 ZGC 停顿低，堆也会 OOM。而且 ZGC 的 10-15% 吞吐代价意味着同样的 QPS 要多买 GPU 机器——LLM 推理的瓶颈是 GPU 不是 CPU，但网关是 CPU 密集型，多 15% 网关机器成本也不低。对象池是"从源头减少垃圾"，ZGC 是"让垃圾回收更快"，前者治本后者治标。只有当对象池改造的成本太高（历史代码动不了）或堆规模真的超过 32G 时，才优先考虑换收集器。

### 第五层：验证与沉淀

**Q：你怎么证明这次 GC 调优真的有效，而不是当天流量低导致的假象？**

两个对比。第一，时间对比——上线前 7 天 vs 上线后 7 天，取同一时段（工作日 10-12 点高峰）的 `jstat` 采样，对比 YGC 频率、YGCT、FGC 次数、P99 延迟。第二，流量归一化——把 YGCT（Young GC 总耗时）/ 请求 QPS，得到"每千请求的 GC 耗时"，如果这个比值下降，证明是调优效果而非流量因素。第三，A/B——网关按 uid 哈希，50% 流量走旧 JVM 参数（对照组），50% 走新参数（实验组），同时段对比 P99 和 GC 指标，最严谨。第四，长期观察——上线后连续 4 周监控 Full GC 频率，确认稳定在 0 次/天。

**Q：怎么让团队避免以后再写出"大对象狂 new"的代码？**

沉淀三件事。第一，Code Review 检查项——"请求处理路径上的大对象（>1KB）是否池化或用直接内存"，未达标 review 不过。第二，JVM 参数模板化——推理网关统一用 `-Xms8g -Xmx8g -XX:+UseG1GC -XX:MaxGCPauseMillis=100 -XX:G1NewSizePercent=30 -XX:+HeapDumpOnOutOfMemoryError`，新人不用自己配。第三，GC 监控告警——`jstat` 采集 YGC/FGC 频率到 Prometheus，`fgc_count_per_hour > 0` 告警，`ygc_pause_ms_p99 > 100` 告警，把 GC 治理变成日常 SRE 动作而不是等故障。

# 【拼多多 AI 中台】训练平台架构怎么设计？GPU 调度怎么做？

> JD 依据："大模型训练框架开发维护、GPU/CPU 资源调度"。

## 一、训练平台要解决的问题

**痛点**：
```
算法：要训 70B 模型 → 需要 1024 卡 + 张量并行 + 数据并行 + 流水并行
手工：找运维要卡 → 配置分布式 → 训练挂了从头来 → 效率极低
```

**平台目标**：提交作业（Docker + 启动脚本 + 数据），平台自动调度 + 容错 + 监控。

## 三、GPU 调度

### 1. 集群管理（K8s + GPU Operator）
```yaml
# Pod 申请 GPU
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: trainer
    image: my-training:latest
    resources:
      limits:
        nvidia.com/gpu: 8       # 8 卡
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: training-data
```

### 2. 拓扑感知调度
```
任务需要 N 卡时：
  - 优先同机（NVLink 互联，通信快）
  - 其次同机架（IB 互联）
  - 最后跨机架（带宽低）

调度器查询节点拓扑，把 Pod 调到最优位置
```

### 3. 优先级与抢占
```
优先级队列：
  - 高优（生产训练）：可抢占低优
  - 中优（实验）
  - 低优（超参搜索/批量）

抢占机制：
  - 高优任务排队等待
  - 调度器驱逐低优任务（checkpoint 后退出）
  - 高优任务接管 GPU
```

### 4. 弹性（在线/离线混部）
```
GPU 池：
  - 在线推理（高优，QPS 敏感）
  - 离线训练（中优，吞吐）
  - 实验/批量（低优）

低峰期：离线任务用满 GPU
高峰期：在线扩容，挤压离线
```

## 四、分布式训练并行策略

### 1. 数据并行（DP）
```
每卡完整模型 + 不同 batch → 算梯度 → AllReduce
简单但显存压力大
```

### 2. 张量并行（TP）
```
单层内切（如 Attention 多头分到多卡）
每层前向/反向要 AllReduce
机内并行（NVLink 高带宽）
```

### 3. 流水并行（PP）
```
按层切（不同卡不同层）
前向/反向相邻层间发激活
跨机并行（IB）
```

### 4. 3D 并行（千亿模型标配）
```
机内 TP（8 卡 NVLink）
+ 跨机 DP（数据并行）
+ 跨机 PP（流水并行）

示例（1024 卡训 175B）：
  TP=8（单机 8 卡）× PP=16（16 个流水段）× DP=8（8 路数据并行）= 1024 卡
```

### 5. ZeRO（DeepSpeed）
```
ZeRO-1：优化器状态分片（省 4x 显存）
ZeRO-2：+ 梯度分片
ZeRO-3：+ 参数分片（最省显存，通信多）
```

### 6. MoE（专家并行）
```
N 个专家分布到多卡
token 路由到对应专家
All-to-All 通信（复杂）
```

## 五、容错与恢复

### 1. Checkpoint
```python
# 定期保存
if step % 100 == 0:
    save_checkpoint({
        'step': step,
        'model': model.state_dict(),
        'optimizer': optimizer.state_dict(),
        'rng_state': torch.get_rng_state(),
    }, path)

# 异常自动恢复
if os.path.exists(latest_ckpt):
    load_checkpoint(latest_ckpt)
```

### 2. 异常检测
```
- 进程崩溃（NCCL timeout）
- GPU 故障（ECC 错误、Xid 错误）
- 网络中断（IB 断连）
- Loss 异常（NaN/爆炸）
- 节点失联
```

### 3. 自动恢复
```
1. 检测异常
2. 通知调度器
3. 从最近 checkpoint 重启（重新调度 Pod）
4. 续训（数据已处理部分跳过）
```

### 4. 弹性训练（PyTorch Elastic）
```python
import torch.distributed.elastic as elastic

# 节点失败自动重组，不重启整个任务
elastic.launch(config)(
    train_fn, args
)
```

## 六、监控

### 性能
```
- GPU 利用率（目标 >80%）
- GPU 显存（防 OOM）
- 通信占比（目标 <30%）
- 数据加载（避免 IO 瓶颈）
- 吞吐（samples/s 或 tokens/s）
```

### 训练指标
```
- loss/gradient norm（收敛/爆炸检测）
- 学习率
- 梯度/激活统计
- 评估指标（AUC/PPL）
```

### 告警
```
- loss NaN
- GPU 利用率 < 50% 持续 10min
- 任务挂掉
- checkpoint 失败
```

```yaml
        - --pipeline-model-parallel-size=16

        - --hidden-size=8192
        - --bf16
        - --use-flash-attn
        - --tokenizer-type=Llama2Tokenizer
```


## 九、拼多多实战

```
集群规模：万卡级 A100/H100
网络：机内 NVLink（400GB/s）+ 机间 IB NDR（400Gbps）

监控：自研 GPU 监控（利用率/温度/显存）+ WandB 集成

关键挑战：
- 千卡训练稳定性（每周 N 次故障）→ checkpoint + 自动恢复
- 通信开销大 → 3D 并行 + 通信压缩
- 资源利用率 → 在线/离线混部 + 弹性抢占
```

## 十、底层本质

训练平台本质是**"GPU 集群 + 调度 + 分布式编排 + 容错"**——K8s 管理硬件，调度器拓扑感知分配 GPU，训练框架（Megatron/DeepSpeed）编排 3D 并行，checkpoint + 自动恢复保证长任务稳定。是大模型时代 AI 基础设施的核心。

## 常见考点

1. **怎么诊断 GPU 利用率低**？——profiler 看通信/数据加载占比，调 batch/并行度/预取。
2. **3D 并行度怎么调**？——TP 受机内 GPU 数限制（8），PP 按层均分，DP = 总卡数 / (TP×PP)。
3. **训练和推理怎么混部**？——在线推理高优 + 离线训练低优，高峰期挤压离线；GPU 实例按时间片抢占。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：训练平台要支持"千卡训练任务"。但 K8s 原生的 Pod 调度只管 CPU/内存，不管 GPU 拓扑。为什么不直接用原生 K8s，非要引入 Volcano + 自研拓扑感知调度？**

原生 K8s 调度 GPU 有三个致命问题。第一，**不管拓扑**——K8s 的 `nvidia.com/gpu` 只看"节点有没有 GPU"，不管 GPU 之间的互联（NVLink vs PCIe）。如果调度把 8 卡任务分到"4 卡在 node-A + 4 卡在 node-B"，跨节点走 IB（400Gbps）比同节点 NVLink（900GB/s）慢 20 倍，训练通信开销剧增。第二，**不管 Gang Scheduling**——K8s 默认调度是"逐 Pod 调度"，8 卡任务的 8 个 Pod 可能只调度了 3 个（其他 5 个节点资源不够），3 个 Pod 启动了但训练任务需要 8 个全启动才能 begin（AllReduce 要所有 rank 就位），结果 3 个 Pod 卡住等，资源被锁死。第三，**不管抢占公平性**——多团队共享集群，原生 K8s 的抢占逻辑（priority class）粗糙，可能导致"高优任务无限抢占低优任务"，低优任务饿死。Volcano 专门解决批处理任务的这三个问题：拓扑感知（同节点优先）、Gang Scheduling（全或无）、公平共享（DRF 算法）。不用 Volcano，千卡训练调度会一团糟。

### 第二层：证据与定位

**Q：用户提交了一个 32 卡训练任务，等了 30 分钟还没开始（Pending 状态）。你怎么定位是资源不够、调度策略问题，还是任务配置问题？**

分三步排查。第一，**看 Pod 的事件**——`kubectl describe pod <pod-name>` 看 Events，如果显示 `FailedScheduling: 0/100 nodes are available: 32 nvidia.com/gpu requested, but only 10 free`，是资源不够（集群只有 10 卡空闲，不够 32 卡）。第二，**看 Volcano 的调度队列**——`vcctl queue info` 或 Volcano Dashboard 看 PodGroup 的状态，如果是 `Pending` 且 reason 是 `gang scheduling: not enough pods to satisfy`，说明 Gang Scheduling 正在等（其他 Pod 还没调度成功）。如果是 `Inqueue`，说明在排队（有更高优先级任务在跑）。第三，**看拓扑约束**——如果任务配了"必须同机架"（`podAntiAffinity` 或 Volcano 的拓扑策略），但空闲卡分布在不同机架，调度器找不到满足约束的节点，一直 Pending。`kubectl get pod -o yaml | grep -A 20 affinity` 看拓扑约束。定位逻辑：资源不够（加卡/等）vs 排队（调优先级）vs 拓扑约束松（去掉机架约束）。

### 第三层：根因深挖

**Q：你发现集群 GPU 利用率才 40%（大量卡闲置），但用户的训练任务一直 Pending（说"资源不够"）。为什么卡闲置但任务调不上？**

典型的"资源碎片化"问题。第一，**大任务找不到整块资源**——集群有 100 卡，但分散在 20 个节点（每节点 5 卡），空闲卡的总数够（100 卡），但找不到"8 卡同节点"（NVLink 拓扑）的整块资源。用户任务要 8 卡同节点（TP 并行要 NVLink），调度器找不到匹配节点，任务 Pending，但集群"总空闲卡数"显示够。第二，**小任务占了大节点的部分卡**——节点有 8 卡，被一个 2 卡的小任务占了 2 卡，剩 6 卡。大任务（要 8 卡）调度不上，但集群统计"空闲 6 卡"——实际这 6 卡因为"不成整"用不了。第三，**解法**——Volcano 的 binpack 调度（把同任务的所有 Pod 调到同一节点/机架，减少碎片）、或"资源整理"（定期迁移小任务，腾出整节点）、或"分级调度"（小任务只调度到"非整节点"，把整节点留给大任务）。监控 `gpu_fragmentation_ratio`（碎片化程度 = 空闲卡数中"无法组成整块"的比例），> 30% 说明调度策略要优化。

**Q：那为什么不让用户都用"小任务"（1-2 卡）？这样就不会有大任务找不到整块资源的问题。**

"任务大小"由模型规模决定，不是用户能随便选的。第一，**模型规模决定并行度**——训 7B 模型用数据并行，单卡放得下（14GB 权重），可以用 1 卡任务（但太慢，训练 1 个月）。训 70B 模型必须张量并行（TP=8，8 卡同节点），单卡放不下（140GB 权重）。用户不能"把 70B 训练改成 2 卡"，物理上不可能。第二，**训练效率**——即使 7B 能单卡训，用 32 卡数据并行能加速 30 倍（训练时间从 1 月到 1 天）。用户为了快，都会用多卡。第三，**集群设计的挑战**——就是要支持"大小任务混合"（有人训 7B 用 8 卡，有人训 70B 用 64 卡），调度器要智能分配，不能要求"大家都用小任务"。正确做法是分级资源池——"大任务池"（整节点，64+ 卡的节点专供大任务）+ "小任务池"（碎片化节点，小任务用），互不干扰。类比：停车场分"大车位"（卡车）和"小车位"（轿车），不能让轿车停大车位（浪费）。

### 第四层：方案权衡

**Q：训练任务挂了（loss NaN / GPU Xid），平台自动从 checkpoint 恢复。但有时 checkpoint 损坏（恢复后还 NaN）。怎么保证 checkpoint 的可靠性？**

checkpoint 可靠性要从"写入 + 校验 + 备份"三层保障。第一，**写入原子性**——checkpoint 要么完整写成功，要么完全不写（不能写一半挂了）。PyTorch 的 `torch.save` 不是原子的（写到一半进程挂了，文件损坏）。解法：先写到临时文件（`checkpoint.tmp`），写完后 rename 成正式文件（`checkpoint.pt`），rename 是原子操作。恢复时只读正式文件，临时文件忽略。第二，**校验完整性**——写入后计算 checksum（MD5/SHA），存到 checkpoint 的 metadata。恢复时先校验 checksum，不匹配则丢弃（用上一个好的 checkpoint）。第三，**多版本备份**——不只存"最新 checkpoint"，存最近 N 个（如最近 5 个），每个 step 的 checkpoint 都保留，损坏一个用前一个。第四，**分布式存储**——checkpoint 存到 Lustre/对象存储（多副本），不存本地 SSD（节点挂了本地数据丢）。拼多多训练平台：每 500 步 checkpoint，异步上传对象存储，保留最近 10 个版本，恢复时从对象存储拉最新好的 checkpoint，可靠性 99.9%+。

**Q：那为什么不每一步都 checkpoint？这样挂了恢复损失最小（最多丢 1 步）。**

每步 checkpoint 的开销大于收益。第一，**写入开销**——70B 模型的 checkpoint（权重 + 优化器状态）约 1.5TB，写到 Lustre 要 10-30 分钟（Lustre 吞吐 1-5GB/s）。每步 checkpoint 意味着每步训练后等 10 分钟写 checkpoint，训练效率降 90%+（训练和 checkpoint 写入串行）。第二，**存储成本**——每步 checkpoint 1.5TB，训练 10 万步 = 150PB，存储成本不可承受。即使只保留最近 5 步，也要频繁删旧的（GC 开销）。第三，**异步 checkpoint**——可以减少开销（训练继续，后台写 checkpoint），但每步触发异步写入，IO 压力大（Lustre 带宽被 checkpoint 占满，影响训练数据读取）。第四，**合理频率**——经验值是每 500-2000 步 checkpoint（根据"可接受的恢复损失"决定）。500 步 checkpoint，恢复最多丢 500 步（约 30-60 分钟训练），可接受。每步 checkpoint 是"过度保守"，实际故障率（千卡每周 3 次）不值得每步保。

### 第五层：验证与沉淀

**Q：你怎么证明"拓扑感知调度 + Gang Scheduling"把集群 GPU 利用率从 40% 提升到了 70%？**

三个指标对比。第一，**GPU 利用率**——监控 `dcgm_gpu_utilization`（集群平均），优化前 40%（大量卡闲置等调度/碎片化），优化后 70%（调度优化让更多任务跑起来）。第二，**任务排队时间**——`pending_time_p50` 和 `pending_time_p99`，优化前 P50=30 分钟（等资源）、P99=2 小时，优化后 P50=5 分钟、P99=30 分钟（Gang Scheduling 减少死锁、拓扑感知减少碎片）。第三，**MFU（训练效率）**——优化前 MFU=35%（拓扑差导致通信开销大），优化后 MFU=50%（同节点 NVLink 通信快，训练效率高）。三个指标从不同维度（资源利用率 + 调度效率 + 训练效率）证明调度优化生效。A/B 验证：新旧调度策略各跑一批任务，对比上述指标。

**Q：训练平台的调度策略怎么沉淀，让新任务/新用户不用手动调？**

三件事。第一，**任务模板**——按模型规模预设模板（7B 训练：8 卡 + TP1 + DP8 + checkpoint/500 步；70B 训练：64 卡 + TP8 + PP4 + DP2 + checkpoint/1000 步），用户选模板提交，不手动配并行策略。第二，**自动并行策略推荐**——平台根据"模型大小 + 集群拓扑"自动推荐最优并行策略（类似 Megatron 的 autodistributor），用户不用懂 TP/PP/DP 怎么配。第三，**调度策略可观测**——用户提交任务后，能看到"为什么 Pending"（资源不够/拓扑约束/排队），不是黑盒等待。平台提供调度 dashboard（任务状态、资源利用率、排队原因），让用户自助排查。训练平台的目标是"用户只关心模型，不关心 GPU 调度"，把分布式训练的复杂性屏蔽在平台层。

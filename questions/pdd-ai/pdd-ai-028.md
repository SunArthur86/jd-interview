# 【拼多多 AI 中台】负载均衡和隔离怎么做？GPU 隔离和压测？

> JD 依据："高并发、模型服务、GPU 资源调度"。

## 一、负载均衡算法

| 算法 | 适用 | 特点 |
|------|------|------|
| 轮询（Round Robin） | 同质实例 | 简单 |
| 加权轮询 | 异构实例 | 按性能分权 |
| 最少请求 | 长请求 | 避免长尾堆积 |
| 最少连接 | 长连接 | 平衡活跃连接 |
| 一致性哈希 | 有状态（缓存） | 节点变动影响小 |
| 随机 | 简单 | 类似轮询 |
| P2C（Power of Two） | 通用 | 选两个比好的，避免最差 |
| GPU 利用率 | LLM | 选利用率低的实例 |

## 二、负载均衡层

```
四层（LVS）：基于 IP/端口，吞吐高（百万 QPS）
七层（Nginx/Gateway）：基于 HTTP/gRPC，灵活（路由/限流/灰度）
服务发现（Nacos）：客户端负载均衡（Dubbo/Feign）
```

### 实战（Spring Cloud LoadBalancer）
```java
@Bean
public ReactorLoadBalancer<ServiceInstance> lb(LocalRegistries registries, Environment env) {
    return new GPUAwareLoadBalancer(registries, env);   // 自定义：按 GPU 利用率选
}
```

### LLM 路由特殊
```
传统：轮询/最少请求足够
LLM：
  - 按 GPU 利用率（避免把请求打到 90% 的卡）
  - 按 KV Cache 命中率（共享 prefix 路由）
  - 按显存（防 OOM）
  - 按模型亲和（已加载模型的实例）
```

## 三、隔离（防故障扩散）

### 1. 隔离层次
```
集群级：在线/离线集群分开
实例级：不同业务不同 Pod 集合
容器级：CPU/内存/GPU limit
进程级：不同模型不同进程
线程级：线程池隔离（在线/离线）
```

### 2. 关键隔离实践
```java
// 线程池隔离（在线/离线）
ThreadPoolExecutor onlinePool = ...;     // 在线推理
ThreadPoolExecutor offlinePool = ...;    // 离线任务
// 避免离线大任务阻塞在线

// 信号量隔离（不同模型）
Semaphore model1Slots = new Semaphore(8);
Semaphore model2Slots = new Semaphore(4);
```

### 3. GPU 隔离（关键）
```
痛点：多个模型共享 GPU，互相抢资源

方案 1：MIG（Multi-Instance GPU，A100/H100）
  - 硬件分区：把一张卡切成多个独立实例
  - 完全隔离（显存/算力/缓存）
  - 重启才能改配置

方案 2：MPS（Multi-Process Service）
  - 软件共享：多进程共享 GPU
  - 算力按比例分配
  - 隔离弱（一个崩可能影响其他）

方案 3：时分复用（vLLM/Triton）
  - 同进程内多模型共享 GPU
  - 由引擎调度（Continuous Batching）
  - 隔离最弱但利用率最高

方案 4：多卡独占
  - 每模型独占卡
  - 隔离最好但浪费
  - 适合核心模型
```

## 四、压测（找容量边界）

### 1. 压测类型
```
基准压测：单接口 QPS 极限
负载压测：日常/峰值负载下表现
压力压测：超负载看怎么挂（恢复能力）
稳定性压测：长时间（24h+）找内存泄漏
全链路压测：模拟真实用户行为，覆盖完整链路
```

### 2. 压测流程
```
1. 准备
   - 压测环境（独立/UAT 或影子流量）
   - 压测数据（脱敏生产数据）
   - 压测脚本（JMeter/Gatling/Locust/自研）

2. 梯度加压
   - 100 → 500 → 1000 → 2000 → 5000 QPS
   - 每阶段稳定 10 分钟观察

3. 监控
   - QPS / RT / 错误率
   - CPU / 内存 / 网络
   - GPU 利用率 / 显存（LLM）
   - DB / Redis / Kafka 水位

4. 找瓶颈
   - CPU 100%？→ 加 CPU/优化算法
   - RT 突增？→ 找依赖（DB/下游）
   - OOM？→ 内存泄漏 / 加内存
   - GPU 满？→ 加卡/量化/batching

5. 容量规划
   - 单实例 QPS：1000
   - 峰值 QPS：50000
   - 副本数：50000 / 1000 × 1.3（buffer）= 65 副本
```

### 3. 全链路压测（关键）
```
影子流量方案：
  - 生产环境注入压测流量
  - 压测流量打影子库/影子缓存（带特殊标记）
  - 不污染生产数据

拼多多双 11 前必做：
  - 模拟双 11 峰值流量
  - 全链路（推荐/搜索/下单/支付）
  - AI 中台：LLM/推荐/搜索全压
  - 找瓶颈 + 调容量 + 验预案
```

### 4. LLM 压测特殊
```
- 输入 prompt 多样化（不要固定 prompt，否则缓存命中失真）
- 输出长度多样（短/中/长）
- 流式 vs 非流式分别压
- GPU 利用率 + 显存 + KV Cache 占用
- 首 token 延迟（TTFT）+ 每 token 延迟（TPOT）
```

## 八、底层本质

负载均衡 + 隔离 + 压测本质是**"分散流量 + 防扩散 + 知极限"**——负载均衡把流量均匀分散，多层隔离（线程/进程/容器/集群/GPU MIG/MPS）防故障扩散，全链路压测找系统容量边界。LLM 场景特别要注意 GPU 隔离（多模型共享卡资源争抢）和 GPU 利用率/显存维度的负载均衡。

## 常见考点

1. **一致性哈希是什么**？——节点和 key 都映射到哈希环，顺时针找节点；加减节点只影响相邻段；加虚拟节点解决倾斜。
2. **MIG 和 MPS 怎么选**？——MIG 强隔离（独立硬件分区）适合多租户，MPS 软共享（高利用率）适合内部多模型。
3. **压测怎么不打挂生产**？——影子流量 + 影子库 + 限流保护 + 降级开关 + 低峰期 + 监控实时止损。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：LLM 推理服务要做负载均衡。但 GPU 实例的性能不均（有的 H100、有的 L4），传统的"轮询"会把请求平均分到所有实例，导致 L4 实例积压、H100 闲置。怎么设计 GPU 异构场景的负载均衡？**

GPU 异构场景要用"加权负载均衡"而非轮询。第一，**性能差异**——H100 的 FP16 算力是 989 TFLOPS，L4 是 120 TFLOPS，差 8 倍。同样 batch 的推理，H100 100ms、L4 500ms。轮询（round-robin）会把请求平均分，H100 处理 1 个（100ms 完成）、L4 处理 1 个（500ms 完成），L4 积压 5 个请求时 H100 已经处理完 5 个（闲置）。第二，**加权方案**——按 GPU 算力加权（H100 权重 8、L4 权重 1），H100 分到 8/9 的请求、L4 分到 1/9，两者的"完成速率"一致（H100 8 个请求 × 100ms = 800ms 完成 8 个；L4 1 个 × 500ms = 500ms 完成 1 个，比例 8:1 匹配）。第三，**更优方案：最少请求 + 算力权重**——网关实时知道每实例的"in-flight 请求数"和"GPU 类型"，路由时选"（in-flight + 1）/ 权重"最小的实例，动态平衡。第四，**LLM 特殊考虑**——不同 GPU 的显存不同（H100 80GB、L4 24GB），能跑的模型/最大 batch 不同，路由时还要考虑"模型是否加载"（L4 跑不了 70B 模型，只跑 7B）。拼多多推理网关：按"模型类型 + GPU 算力 + 实时负载"三维加权路由，H100 集群跑 70B、L4 集群跑 7B，各自内部按算力加权。

### 第二层：证据与定位

**Q：压测时发现"系统在 5000 QPS 时正常，6000 QPS 时 P99 从 200ms 飙到 5 秒"。这个"拐点"怎么定位是哪个组件成为瓶颈？**

逐层分析请求链路。第一，**拆解链路**——请求路径：网关 → 推理服务 → Redis（特征查询）→ vLLM（推理）→ 返回。在每跳埋点（TraceId 透传），看 5000 vs 6000 QPS 时各跳的耗时。如果"网关 → 推理服务"从 5ms 升到 50ms（网络/排队），是网关层瓶颈；如果"Redis 查询"从 1ms 升到 100ms，是 Redis 瓶颈（连接池满/慢查询）；如果"vLLM 推理"从 100ms 升到 3 秒，是 vLLM 瓶颈（batch 满/GPU 算力到极限）。第二，**看资源监控**——6000 QPS 时，哪个资源的利用率"打满"（CPU 90%+、GPU 95%+、网络带宽 80%+、连接数接近上限）。打满的资源对应的组件是瓶颈。第三，**看队列长度**——vLLM 的 `vllm:request_waiting_time` 在 5000 QPS 时 10ms，6000 QPS 时 2 秒，说明 vLLM 的 batch 满了，请求排队。Redis 的 `connected_clients` 在 6000 QPS 时到 10000（Redis 单实例连接上限），说明连接池耗尽。第四，**根因验证**——针对嫌疑瓶颈"扩容"（vLLM 加 2 个实例、或 Redis 加连接池），如果扩容后 6000 QPS 的 P99 降到 200ms，验证了瓶颈定位。

### 第三层：根因深挖

**Q：你发现瓶颈是"GPU 显存满了"（vLLM 的 KV Cache 池占用 95%），新请求排队等显存。为什么 GPU 显存会成为瓶颈，而不是算力？**

LLM 推理是"显存密集型"而非"算力密集型"。第一，**显存占用分析**——7B 模型权重 14GB + KV Cache 池（随 batch 增长）+ 激活（随 seq 增长）。A100 80GB，权重占 14GB，剩 66GB 给 KV Cache。每请求的 KV Cache（seq=2K）约 1.6GB，66GB / 1.6GB ≈ 40 个并发请求。超过 40 个并发，KV Cache 池满，新请求 preempt（等显存释放）。第二，**算力不是瓶颈**——7B 模型单次前向 0.05 TFLOPS，A100 算力 312 TFLOPS，算力利用率才 0.016%（远未满载）。GPU 的"算力"和"显存"是两个独立资源，LLM 推理卡在显存。第三，**为什么 PagedAttention 也不够**——PagedAttention 把碎片降到 5%，利用率 95%，但"总显存量"是物理限制。95% 利用率意味着 40 个并发已是极限（66GB / 1.6GB × 95% = 39）。第四，**解法**——（1）量化（INT4 让 KV Cache 省 4 倍，并发提到 160）；（2）MQA/GQA（减少 KV Cache 大小）；（3）扩容（加 GPU 实例，线性提升并发）；（4）PD 分离（Decode 实例独立扩显存）。显存是 LLM 推理的"硬约束"，必须针对性优化。

**Q：那为什么不用"更大显存的 GPU"（H100 80GB → H200 141GB）？显存翻倍，并发翻倍，不就解决了？**

大显存 GPU 能缓解但不治本，且有成本问题。第一，**显存翻倍 vs 成本翻倍**——H200 比 H100 贵 30-50%，显存从 80GB 到 141GB（1.76 倍）。但"显存增 1.76 倍"换"成本增 30-50%"是划算的（单位显存成本降）。但 H200 供应紧张（2025 年产能受限），不一定买得到。第二，**显存需求也在增长**——模型从 7B（14GB）升级到 70B（140GB），显存需求翻 10 倍。即使 H200 的 141GB，跑 70B 模型也只够权重（140GB），KV Cache 几乎没空间。第三，**长上下文趋势**——LLM 的 context 从 2K 涨到 128K（Llama3/Qwen2），KV Cache 随 seq 线性增长，128K context 的 KV Cache = 100GB/请求。再大的显存也不够。第四，**根本解法**——不能只靠"加显存"，要"提升显存效率"：（1）KV Cache 量化（INT8/FP8 省 2 倍）；（2）前缀缓存（共享 system prompt 省 N 倍）；（3）滑动窗口（只保留最近 N token 省 N 倍）；（4）PD 分离（Decode 用小实例、Prefill 用大实例，各自优化）。拼多多：H100 跑 70B（权重 140GB INT4 + KV Cache INT8），用优化把并发做到 50+，而不是等 H200。

### 第四层：方案权衡

**Q：压测要用"影子流量"（复制线上流量到测试环境），不直接压生产。但影子流量怎么保证"和真实流量一致"？如果流量分布不一样，压测结果就不准。**

影子流量的"真实性"要靠"全量复制 + 时间对齐"。第一，**全量复制**——用流量镜像（如 Nginx 的 mirror 模块、或 K8s 的 Istio mirror）把线上请求"复制"一份到压测环境，不是"采样"。采样的流量分布会偏（只复制某些请求），压测不准。第二，**时间对齐**——压测环境"同时刻"收到和线上一样的流量（不是"录播"后慢慢回放）。如果线上 10:00 有 5000 QPS 峰值，压测环境也要 10:00 收到 5000 QPS，不能延迟到 11:00 回放。第三，**数据一致性**——压测环境的数据（Redis/DB）要和线上一致（用快照恢复），否则"压测请求查不到数据"，行为不一致。第四，**影子流量局限**——即使全量复制，压测环境的"硬件规模"可能小于线上（线上 100 卡、压测 10 卡），压测时要"按比例放大"（10 卡扛 10% 流量 = 100 卡的等效负载）。拼多多压测：流量镜像到独立压测集群（10% 线性规模），用快照恢复数据，时间对齐跑 1 小时，测"等比例负载下的表现"，推算全量负载。

**Q：那为什么不直接压生产（线上环境）？生产环境的数据/规模最真实，压测结果最准。影子流量还要维护独立环境，麻烦。**

压生产的风险远大于收益。第一，**压挂生产 = 业务中断**——如果压测流量超出系统容量，生产服务 OOM/超时，真实用户受影响（客服 LLM 挂了，用户投诉）。一次压测事故可能导致"GMV 损失/用户流失"，代价百万级。第二，**数据污染**——压测请求会写入生产 DB/Redis（如压测的"下单"请求真的创建了订单），污染业务数据（虚假订单影响财务/库存）。清理压测数据成本高（要精确区分压测数据 vs 真实数据）。第三，**压测的替代方案足够好**——影子流量（流量镜像 + 独立环境）能 95% 还原生产场景，剩下 5% 差异（硬件规模/数据量）可以通过"按比例推算"补足。不值得为了 5% 的精度冒"压挂生产"的风险。第四，**例外场景**——只有"无法用影子流量"的场景才压生产（如测试"真实用户的网络延迟分布"），且必须有"限流保护 + 降级开关 + 实时监控 + 止损预案"，在低峰期（凌晨）小流量压，随时停止。默认不压生产，是稳定性红线。

### 第五层：验证与沉淀

**Q：你怎么证明"压测发现的瓶颈"（GPU 显存）修复后，系统能扛住 10000 QPS？**

三层验证。第一，**压测前/后对比**——压测发现 6000 QPS 时显存瓶颈（P99=5s），优化（KV Cache 量化 INT8）后重压测，看 6000 QPS 时 P99 是否降到 200ms（显存占用从 95% 降到 60%），再压到 10000 QPS 看是否能扛（P99 < 1s）。如果 10000 QPS 稳定，证明优化生效。第二，**线上灰度验证**——优化版本灰度到 10% 流量（1000 QPS），观察 1 周的 `gpu_memory_usage_p99` 和 `latency_p99`，确认稳定（无 OOM/P99 无飙升），再全量。第三，**容量推算**——压测是在 10% 规模（10 卡）上跑 1000 QPS，推算全量（100 卡）能扛 10000 QPS（线性扩展）。但要注意"非线性瓶颈"（如 Redis 连接数、网络带宽），压测时也要验证这些维度。三个验证（压测 + 灰度 + 推算）一致，才能确认"能扛 10000 QPS"。

**Q：负载均衡和压测的经验怎么沉淀，让新服务的容量规划不用每次从零做？**

三件事。第一，**容量画像库**——按服务类型（LLM 推理/Web/搜索）记录"单实例 QPS 上限 + 瓶颈资源 + 扩容策略"，新服务查同类画像，预估自己的容量。如"7B vLLM 单 H100：QPS=100，瓶颈是显存，扩容线性"。第二，**压测自动化**——把压测做成"一键化"（`pdd-benchmark --service xxx --qps 5000 --duration 1h`），自动生成压测报告（QPS/P99/错误率/瓶颈分析），新服务每周自动压测，监控容量变化。第三，**容量预警**——线上监控 `qps / capacity_qps`（实际 QPS / 容量 QPS），> 70% 告警（要扩容了），> 90% 紧急告警（可能要挂）。基于压测结果设容量水位线，提前扩容。负载均衡和压测是"容量管理"的两面，要"持续压测知道容量 + 智能均衡用满容量"，把容量管理工程化。

# 【拼多多 AI 中台】负载均衡和隔离怎么做？GPU 隔离和压测？

> JD 依据："高并发、模型服务、GPU 资源调度"。

## 一、负载均衡算法

| 算法 | 适用 | 特点 |
|------|------|------|
| 轮询（Round Robin） | 同质实例 | 简单 |
| 加权轮询 | 异构实例 | 按性能分权 |
| 最少请求 | 长请求 | 避免长尾堆积 |
| 最少连接 | 长连接 | 平衡活跃连接 |
| 一致性哈希 | 有状态（缓存） | 节点变动影响小 |
| 随机 | 简单 | 类似轮询 |
| P2C（Power of Two） | 通用 | 选两个比好的，避免最差 |
| GPU 利用率 | LLM | 选利用率低的实例 |

## 二、负载均衡层

```
四层（LVS）：基于 IP/端口，吞吐高（百万 QPS）
七层（Nginx/Gateway）：基于 HTTP/gRPC，灵活（路由/限流/灰度）
服务发现（Nacos）：客户端负载均衡（Dubbo/Feign）
```

### 实战（Spring Cloud LoadBalancer）
```java
@Bean
public ReactorLoadBalancer<ServiceInstance> lb(LocalRegistries registries, Environment env) {
    return new GPUAwareLoadBalancer(registries, env);   // 自定义：按 GPU 利用率选
}
```

### LLM 路由特殊
```
传统：轮询/最少请求足够
LLM：
  - 按 GPU 利用率（避免把请求打到 90% 的卡）
  - 按 KV Cache 命中率（共享 prefix 路由）
  - 按显存（防 OOM）
  - 按模型亲和（已加载模型的实例）
```

## 三、隔离（防故障扩散）

### 1. 隔离层次
```
集群级：在线/离线集群分开
实例级：不同业务不同 Pod 集合
容器级：CPU/内存/GPU limit
进程级：不同模型不同进程
线程级：线程池隔离（在线/离线）
```

### 2. 关键隔离实践
```java
// 线程池隔离（在线/离线）
ThreadPoolExecutor onlinePool = ...;     // 在线推理
ThreadPoolExecutor offlinePool = ...;    // 离线任务
// 避免离线大任务阻塞在线

// 信号量隔离（不同模型）
Semaphore model1Slots = new Semaphore(8);
Semaphore model2Slots = new Semaphore(4);
```

### 3. GPU 隔离（关键）
```
痛点：多个模型共享 GPU，互相抢资源

方案 1：MIG（Multi-Instance GPU，A100/H100）
  - 硬件分区：把一张卡切成多个独立实例
  - 完全隔离（显存/算力/缓存）
  - 重启才能改配置

方案 2：MPS（Multi-Process Service）
  - 软件共享：多进程共享 GPU
  - 算力按比例分配
  - 隔离弱（一个崩可能影响其他）

方案 3：时分复用（vLLM/Triton）
  - 同进程内多模型共享 GPU
  - 由引擎调度（Continuous Batching）
  - 隔离最弱但利用率最高

方案 4：多卡独占
  - 每模型独占卡
  - 隔离最好但浪费
  - 适合核心模型
```

## 四、压测（找容量边界）

### 1. 压测类型
```
基准压测：单接口 QPS 极限
负载压测：日常/峰值负载下表现
压力压测：超负载看怎么挂（恢复能力）
稳定性压测：长时间（24h+）找内存泄漏
全链路压测：模拟真实用户行为，覆盖完整链路
```

### 2. 压测流程
```
1. 准备
   - 压测环境（独立/UAT 或影子流量）
   - 压测数据（脱敏生产数据）
   - 压测脚本（JMeter/Gatling/Locust/自研）

2. 梯度加压
   - 100 → 500 → 1000 → 2000 → 5000 QPS
   - 每阶段稳定 10 分钟观察

3. 监控
   - QPS / RT / 错误率
   - CPU / 内存 / 网络
   - GPU 利用率 / 显存（LLM）
   - DB / Redis / Kafka 水位

4. 找瓶颈
   - CPU 100%？→ 加 CPU/优化算法
   - RT 突增？→ 找依赖（DB/下游）
   - OOM？→ 内存泄漏 / 加内存
   - GPU 满？→ 加卡/量化/batching

5. 容量规划
   - 单实例 QPS：1000
   - 峰值 QPS：50000
   - 副本数：50000 / 1000 × 1.3（buffer）= 65 副本
```

### 3. 全链路压测（关键）
```
影子流量方案：
  - 生产环境注入压测流量
  - 压测流量打影子库/影子缓存（带特殊标记）
  - 不污染生产数据

拼多多双 11 前必做：
  - 模拟双 11 峰值流量
  - 全链路（推荐/搜索/下单/支付）
  - AI 中台：LLM/推荐/搜索全压
  - 找瓶颈 + 调容量 + 验预案
```

### 4. LLM 压测特殊
```
- 输入 prompt 多样化（不要固定 prompt，否则缓存命中失真）
- 输出长度多样（短/中/长）
- 流式 vs 非流式分别压
- GPU 利用率 + 显存 + KV Cache 占用
- 首 token 延迟（TTFT）+ 每 token 延迟（TPOT）
```

## 五、典型故障隔离案例

```
场景：客服 LLM 和代码助手共享集群
风险：代码助手大 prompt 拖垮客服响应

隔离方案：
1. 集群隔离：客服（在线，H100）+ 代码助手（在线，A100）
2. 实例隔离：不同 Deployment
3. 限流隔离：客服 1000 QPS / 代码助手 100 QPS
4. 资源配额：客服 80% GPU / 代码助手 20%

效果：代码助手挂了不影响客服
```

## 八、底层本质

负载均衡 + 隔离 + 压测本质是**"分散流量 + 防扩散 + 知极限"**——负载均衡把流量均匀分散，多层隔离（线程/进程/容器/集群/GPU MIG/MPS）防故障扩散，全链路压测找系统容量边界。LLM 场景特别要注意 GPU 隔离（多模型共享卡资源争抢）和 GPU 利用率/显存维度的负载均衡。

## 常见考点

1. **一致性哈希是什么**？——节点和 key 都映射到哈希环，顺时针找节点；加减节点只影响相邻段；加虚拟节点解决倾斜。
2. **MIG 和 MPS 怎么选**？——MIG 强隔离（独立硬件分区）适合多租户，MPS 软共享（高利用率）适合内部多模型。
3. **压测怎么不打挂生产**？——影子流量 + 影子库 + 限流保护 + 降级开关 + 低峰期 + 监控实时止损。

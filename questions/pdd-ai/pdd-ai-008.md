# 【拼多多 AI 中台】实验平台（A/B Test）怎么设计？

> JD 依据："实验平台、消费者服务策略算法中台"。

## 一、为什么需要实验平台

**问题**：算法同学训练了个新推荐模型，离线 AUC 0.85 → 上线全量，结果 GMV 掉了 3%，损失千万。

**实验平台价值**：
- 用 1% 流量小范围验证 → 失败只损失 1%
- 数据驱动决策 → 不靠拍脑袋
- 多策略并行实验 → 快速迭代

## 二、核心机制：分流

```java
// 基础分流：hash + 取模
public String getVariant(String userId, String experiment, Map<String, Integer> variants) {
    String key = userId + ":" + experiment;
    int bucket = Math.abs(MurmurHash.hash(key)) % 100;   // 0-99 桶
    // 按配置分配：对照组 50 桶，实验组 50 桶
    return findVariant(bucket, variants);
}
```

**要求**：
- **均匀**：每桶用户数/画像分布一致（MurmurHash 比 JDK hash 分散性好）
- **稳定**：同一用户每次进同一桶（哈希一致）
- **可控**：可调整流量比例（灰度）

## 三、互斥 vs 正交实验

**互斥实验**（同一层）：用户只能进一个实验
```
用户 → 桶 0-49 进实验 A
用户 → 桶 50-99 进实验 B
A 和 B 不能同时影响同一用户
```

**正交实验**（不同层）：用户可同时进多个实验
```
层 1（推荐算法）：桶 0-49 用新模型 / 50-99 用旧模型
层 2（UI 主题）：桶 0-49 红色 / 50-99 蓝色
同一用户：可能是"新模型 + 红色"
层间用 hash(uid + layer) 独立分桶，保证正交（互不干扰）
```

**拼多多规模**：同时跑上千实验，必须正交分层。

## 四、指标体系

| 类型 | 例子 |
|------|------|
| 核心指标 | GMV、CTR、转化率（决定实验成败） |
| 护栏指标 | 留存、卸载率、客诉率（防负面） |
| 辅助指标 | 人均时长、点击数（解释现象） |

**指标计算**：实时（Flink）+ 离线（Hive）双链路。

## 五、显著性检验

```
零假设 H0：实验组和对照组无差异
备择 H1：实验组优于对照组

T 检验算 p 值：
  p < 0.05 → 拒绝 H0 → 显著（策略有效）
  p ≥ 0.05 → 不能拒绝 H0 → 不显著（可能是噪声）
```

**坑**：
- **效应大小**：p 值显著但提升 0.01% 没业务价值，看绝对提升和置信区间
- **样本量**：流量太小检测不出差异，要先算 MDE（最小可检测效应）
- **多重比较**：同时看 100 个指标，5 个会假阳性，要 Bonferroni 校正
- **辛普森悖论**：分桶不均可能整体结论和分组相反

## 六、实验流程

```
1. 立项：假设 + 指标 + 预期提升 + 流量
2. 分流配置：对照组/实验组比例
3. 灰度发布：先 1% → 5% → 20%，监控异常
4. 数据积累：通常 7-14 天（覆盖周期波动）
5. 分析：指标对比 + 显著性 + 分群下钻
6. 决策：放量 / 调整 / 放弃
7. 沉淀：实验报告存档，可复用
```

## 七、AI 中台的实验场景

- **模型迭代**：新推荐模型 vs 旧模型，看 GMV
- **Prompt 实验**：不同 prompt 模板对比 LLM 输出质量
- **推荐策略**：召回/排序/重排各环节实验
- **RAG 实验**：不同召回策略对比准确率
- **定价/促销**：不同折扣策略对比转化

**Interleaving（穿插评估）**：传统 A/B 要 2 周，Interleaving 把新旧模型结果混排展示，1-2 天就能从点击率看胜负，是谷歌/拼多多等公司加速实验的法宝。

## 八、底层本质

实验平台本质是**"用随机分流 + 对照度量 + 显著性检验做科学决策"**——核心是控制变量、随机分配、统计检验。规模化的关键是正交分层支持多实验并行，加速策略迭代。AI 中台场景下，每个模型/Prompt/策略上线都要过实验平台，是"算法效果裁判"。

## 常见考点

1. **分流不均怎么发现**？——监控每桶的 SRM（样本比失衡）指标，比如对照组 49% 实验组 51% 说明分流有问题。
2. **实验结果冲突怎么办**？——看长期 vs 短期，看分群（新用户 vs 老用户），多指标综合，必要时重跑。
3. **怎么加速实验**？——Interleaving 穿插评估（1-2 天），方差缩减（CUPED 用历史协变量降低方差）。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你们实验平台为什么用 MurmurHash 而不是 JDK 自带的 String.hashCode 做分流？hashCode 不是也能把 uid 打散到 0-99 桶吗？**

JDK 的 `String.hashCode()` 分布性不如 MurmurHash。JDK hashCode 是简单的 `s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]`，对某些 uid 模式（比如全是数字、或有固定前缀的 uid）会产生聚集——大量 uid 哈希到相邻的几个桶，导致实验组/对照组样本量不均。MurmurHash 是非加密哈希但设计上分布更均匀（雪崩效应好，输入变化 1 bit 输出变化约 50%）。我们做过实测：1000 万拼多多 uid，JDK hashCode 分到 100 桶后，最大桶 12.3 万、最小桶 8.7 万（偏差 23%）；MurmurHash3 最大 10.1 万、最小 9.9 万（偏差 2%）。分流不均会直接导致 SRM（样本比失衡），实验结论不可信。

### 第二层：证据与定位

**Q：一个推荐模型 A/B 实验跑了 14 天，核心指标 GMV 实验组比对照组 +2%，p=0.04 显著。但业务方质疑"用户可能本来就在涨"。你怎么排除这个干扰？**

三步验证。第一，看 SRM（样本比失衡）——对照组配置 50% 实验组 50%，实际样本量比应该是 1:1。算 `(control_n - treatment_n) / (control_n + treatment_n)`，如果偏差 > 1%，分流有 bug，实验作废。第二，看指标趋势——把 14 天的每日 GMV 画成曲线，如果实验组和对照组从第 1 天就平行（差距稳定），说明实验组确实更优；如果前 7 天重合、后 7 天才分开，可能是外部因素（比如实验组在第 7 天碰上了某个热点事件）。第三，CUPED 方差缩减——用实验前的历史协变量（用户过去 7 天的 GMV）调整指标：`Y_adjusted = Y - theta * (X - mean(X))`，theta = cov(Y,X)/var(X)。调整后 p 值通常更小（样本量不变但方差降了），如果调整后仍显著，结论更可信。

### 第三层：根因深挖

**Q：SRM 检查发现对照组 49.8 万、实验组 50.2 万，偏差 0.4%，超过了 0.1% 的阈值。根因可能是什么？**

SRM 的常见根因有四个。第一，分流哈希本身的偏差——MurmurHash 应该没问题，但如果用的是 JDK hashCode 且 uid 模式聚集，会偏差。第二，分流后某些用户被"过滤"了——比如实验组的新模型加载慢，部分实验组用户因为超时被 fallback 到默认逻辑（没进实验统计），导致实验组样本量偏低。第三，埋点上报问题——实验组的某些用户因为新模型的 bug 导致埋点丢失（比如新模型返回了非预期格式，埋点 SDK 报错没上报）。第四，分流服务的缓存——如果分流结果被缓存（按 uid 缓存 variant），但实验配置中途变更（实验组从 50% 扩到 60%），缓存的旧 variant 会导致短期样本比失衡。排查手段：抽样 100 个用户，手动算 `MurmurHash(uid) % 100`，对比线上记录的 variant，确认哈希逻辑正确；然后查实验组的错误日志，看有没有"用户进了实验组但没产生指标数据"的 case。

**Q：那为什么不直接用 100% 流量做"前后对比"（上线前 vs 上线后），而不是费劲做 A/B 对照？**

前后对比无法控制外部变量。上线前 7 天 GMV 100 万，上线后 7 天 120 万，涨了 20%——但可能是因为同期平台做了促销活动、或竞品缺货导致用户流入、或季节性波动。前后对比把这些外部因素和新模型的效果混在一起，无法归因。A/B 同期对照的核心价值是"随机分流消除一切外部变量"——实验组和对照组在同期经历同样的促销、季节、竞品影响，唯一的差别是新模型，因此指标差异可以归因到模型。这是因果推断（causal inference）的基本要求，不是"费劲"，是科学决策的最低门槛。

### 第四层：方案权衡

**Q：你们同时跑上千个实验，用正交分层保证互不干扰。但某个实验的业务方坚持说"我们的指标被另一个实验污染了"。你怎么判断是真污染还是推卸责任？**

正交分层理论上保证独立，但实际"污染"可能来自业务逻辑耦合。第一，检查两个实验是否在同一层——如果同层且流量互斥，不会污染（用户只能进一个）；如果跨层正交，统计上独立但业务上可能耦合（比如层 1 的推荐实验提升了用户活跃度，层 2 的搜索实验的 CTR 因为活跃度提升而虚高）。第二，做"关停实验"——临时关停被怀疑污染源的那个实验（层 1 的推荐实验），观察层 2 搜索实验的指标是否变化。如果关停后层 2 指标回归，确认污染；如果不变，是推卸责任。第三，看指标定义——如果两个实验的核心指标都是 GMV，且 GMV 的统计口径包含对方影响的链路（推荐实验提升了推荐位的 GMV，搜索实验的 GMV 统计也包含推荐位的贡献），就是指标定义耦合，不是分流问题。解法是拆细指标（推荐位 GMV vs 搜索位 GMV 分开看）。

**Q：为什么不干脆一次只跑一个实验，跑完再上下一个？这样不就没有干扰问题了？**

大厂不可能串行实验。拼多多同时有推荐/搜索/风控/营销/定价等上百个团队在做策略迭代，每个团队每月可能上 3-5 个实验，串行的话单团队一年才能验证 12 个策略，迭代速度完全跟不上业务。正交分层 + 多层并行是规模化的唯一解——理论上 N 层可以同时跑 N 组独立实验，每个实验组在自己的层内 100% 流量都可用。代价是管理复杂度（要维护层的语义、防止业务耦合的实验放进正交层）和统计分析成本（要控制多重比较的假阳性）。但相比串行，收益是 10-100 倍的迭代速度，值得。

### 第五层：验证与沉淀

**Q：你怎么证明实验平台的分流是"真的随机"且"真的正交"？有没有可能 MurmurHash 实现有 bug 导致长期偏斜？**

三层验证。第一，分流均匀性测试——每天抽样 100 万 uid，算每桶（0-99）的用户数，卡方检验（chi-square test）p 值 > 0.05 表示均匀。长期监控 `bucket_imbalance_ratio`，超 1% 告警。第二，正交性验证——取两个层的分流结果，算 2x2 列联表（层 1 实验组 × 层 2 实验组），卡方检验 p 值 > 0.05 表示独立。第三，AA 实验——同样的模型/策略，随机分成"伪实验组"和"伪对照组"跑 14 天，如果指标出现显著差异（p < 0.05），说明分流或统计链路有系统性偏差。AA 实验应该定期跑（每月一次），是实验平台的"回归测试"。

**Q：怎么让实验结论不再被业务方质疑？**

沉淀两件事。第一，实验报告标准化——每个实验结论必须包含：假设、指标（核心+护栏）、样本量、SRM 检验结果、p 值、效应大小（Cohen's d）、置信区间、分群下钻结果，缺一不可。第二，引入"实验评审委员会"——高影响实验（影响 GMV > 1% 的）上线前由数据科学团队 review，检查实验设计（指标定义、流量、时长）是否合理，结论是否可靠。第三，建立"实验知识库"——每个实验的假设、结果、是否放量，存档可查，避免重复实验和"同样的实验得出相反结论"的混乱。

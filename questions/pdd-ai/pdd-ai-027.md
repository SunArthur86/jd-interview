# 【拼多多 AI 中台】限流降级超时重试怎么做？LLM 网关怎么限流？

> JD 依据："高并发、模型服务、消费者服务策略算法中台"。

## 一、限流算法

### 1. 计数器（固定窗口）
```
每秒请求数 < 100
简单但有边界突刺（窗口切换瞬间 2x）
```

### 2. 滑动窗口
```
把窗口切成小格（如 1 秒分 10 个 100ms 格）
滑动统计，平滑无突刺
Sentinel 默认实现
```

### 3. 漏桶（Leaky Bucket）
```
请求进桶（容量有限），按固定速率出桶
强制平滑速率
适合保护下游（保证匀速）
```

### 4. 令牌桶（Token Bucket）
```
固定速率发令牌，桶满丢弃
请求拿令牌才能处理
允许突发（桶里存令牌）
适合限速但不死板
Guava RateLimiter 实现
```

## 二、限流维度

```
1. QPS（每秒请求数）
2. 并发数（同时处理数）
3. token/s（LLM 特殊，按 token 限）
4. 成本（每小时花费上限）
5. 用户维度（单 uid 限流）
6. IP 维度（防刷）
7. 模型维度（按 model_id 限）
8. 接口维度（不同接口不同阈值）
```

### LLM 网关限流（关键）
```java
@Component
public class LlmRateLimiter {

    // 多维度组合限流
    public void check(String uid, String model, String prompt) {
        int estTokens = estimateTokens(prompt) + 1000;  // 估算输入+输出

        // 1. 全局 QPS
        rateLimiter.tryAcquire("llm_global_qps", 1000);

        // 2. 单 uid QPS（防刷）
        rateLimiter.tryAcquire("llm_uid:" + uid, 10);

        // 3. 单 uid token/s（防超长输出）
        tokenBucket.acquire("llm_uid_tokens:" + uid, estTokens, 10000);

        // 4. 模型维度（贵模型限更严）
        int modelQps = MODEL_LIMIT.get(model);  // gpt-4 类 → 100 QPS
        rateLimiter.tryAcquire("llm_model:" + model, modelQps);

        // 5. 成本维度（每小时累计成本）
        double cost = calcCost(model, estTokens);
        if (costLedger.tryDeduct(uid, cost) == 0) {
            throw new RateLimitException("超出成本配额");
        }
    }
}
```

## 三、降级策略

```
1. 返回兜底（默认结果）
   "系统繁忙，稍后再试" / 推荐默认商品列表

2. 切小模型
   LLM：72B → 7B（响应快成本低）
   推荐：精排 → 粗排

3. 关闭非核心
   大促时关个性化推荐、关评论展示

4. 简化流程
   跳过 RAG（直接 LLM）
   跳过重排（只召回+精排）

5. 静态化
   动态页面 → CDN 静态页
```

## 四、超时设置

```
分层超时：
- 连接超时（connect timeout）：1s
- 读超时（read timeout）：30s（LLM 慢，要长）
- 整体超时（request timeout）：60s
- 流式首 token 超时（TTFT）：5s

LLM 特殊：
- 首 token 必须快（用户等不了）
- 后续 token 可慢
- 总长度有上限（防爆 token）
```

## 五、重试策略

```
什么情况重试：
- 网络抖动（连接失败/超时）
- 临时错误（5xx）
- 限流（429，需退避）

什么情况不重试：
- 业务错误（4xx，参数错）
- 持续错误（熔断打开）
- 非幂等操作（部分场景）

重试算法：
- 指数退避：1s → 2s → 4s → 8s
- 抖动：避免重试风暴（+ random jitter）
- 上限：最多 3 次
- 熔断：错误率超阈值停止重试
```

### LLM 重试特别注意
```
LLM 推理可能部分成功（流式输出到一半失败）
- 重试要重新计算 token（已输出部分）
- 或保存已输出部分，从中断点续
- 大部分场景直接整体重试（用户重发）
```

### 重试 + 熔断（Hystrix/Sentinel）
```java
@Retryable(maxAttempts = 3, backoff = @Backoff(delay = 1000, multiplier = 2))
public String callLlm(String prompt) {
    return llmClient.invoke(prompt);
}

@Recover
public String recover(Exception e, String prompt) {
    return "系统繁忙";  // 重试用尽兜底
}
```

## 六、四板斧协同

```
请求进入
   ↓
限流（QPS/uid/token/成本）──超→ 拒绝/排队
   ↓
调用 LLM
   ├ 超时 → 重试（指数退避，3 次内）
   ├ 异常 → 降级（小模型/兜底）
   └ 持续故障 → 熔断（直接降级，不再调用）
   ↓
返回结果
```

## 七、LLM 网关限流特殊设计

### 1. Token-based 限流
```
不是 QPS，而是 tokens/s
原因：
  - 长 prompt 占用资源多
  - 长 generation 占 GPU 久
  - 按 QPS 限流会被长请求绕过

实现：
  - 输入 token 立即扣
  - 输出 token 实时扣（流式时累计）
  - 超额拒绝/降级
```

### 2. 成本维度限流
```
按 $ / ¥ 计费：
  - 72B 模型每千 token $0.01
  - 单用户每小时上限 $10
  - 全局每小时上限 $10000

防：恶意刷 → 成本爆
防：bug 死循环 → 成本爆
```

### 3. 优先级
```
高优先级：付费用户、生产业务
中优先级：内部员工、实验
低优先级：免费用户、批量任务

资源紧张时优先服务高优
```

## 九、底层本质

四板斧本质是**"控制流量 + 快速失败 + 兜底保活"**——限流防过载（按 QPS/并发/token/成本多维），超时防雪崩（连接/读/整体分层），降级保核心（兜底/小模型/关非核心），重试应对偶发（退避+上限+幂等）。LLM 场景特别要按 token 和成本限流，并做好流式降级。

## 常见考点

1. **令牌桶和漏桶区别**？——令牌桶允许突发（存令牌），漏桶强制匀速；网关限流用令牌桶，保护下游用漏桶。
2. **重试风暴怎么避免**？——指数退避 + 随机抖动 + 熔断 + 限制总重试率（如不超过 QPS 10%）。
3. **LLM 流式输出超时怎么办**？——首 token 超时立即失败，后续 token 间歇超时容忍（已生成部分返回 + 标记截断）。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试宫层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：LLM 网关的限流维度要支持"按 token 限流"（不只按 QPS）。为什么 LLM 要单独搞"token 限流"？传统 Web 的 QPS 限流不够吗？**

QPS 限流对 LLM 不够，因为"一个请求的资源消耗差异巨大"。第一，**请求成本差异**——Web 请求（如查订单）的资源消耗固定（查一次 DB，1ms），QPS 限流够了（1000 QPS = 1000 次查询）。LLM 请求的资源消耗取决于"输出长度"——一个"写一首诗"的请求可能输出 100 token（消耗 0.1 秒 GPU），一个"写一篇 5000 字文章"的请求可能输出 5000 token（消耗 5 秒 GPU）。两者 QPS 都是 1，但 GPU 消耗差 50 倍。第二，**恶意场景**——用户（或攻击者）发"写一篇 10 万字的小说"，单个请求消耗 100 秒 GPU（一个 H100 卡 100 秒），100 个这样的请求就能占满 1 个 GPU 10000 秒。QPS 限流（100 QPS）无法防御（100 个请求都是合法的，但每个都"超长输出"）。第三，**token 限流**——按"每分钟输出的 token 总数"限流（如每用户 10000 token/分钟），超长输出请求会被限制（输出 5000 token = 消耗 5000 token 配额）。token 限流本质是"按资源消耗限流"，比 QPS（按请求次数限流）更公平。

### 第二层：证据与定位

**Q：限流规则配了"每用户 100 QPS"，但监控显示某用户发了 500 QPS，限流没生效。怎么定位是限流规则没配对、还是 Sentinel 的 bug？**

分三步排查。第一，**看限流规则的生效范围**——Sentinel 的规则可以是"resource + limitApp + strategy"，如果规则配了"limitApp=appA"（只对 appA 生效），但请求来自 appB，限流不生效。检查 Sentinel Dashboard 的规则配置。第二，**看限流的统计窗口**——Sentinel 的滑动窗口默认 1 秒，如果用户在 1 秒内发了 100 个请求（窗口内没超限），第 2 秒又发 100 个（新窗口），总体 200 QPS 但每秒都不超 100。监控的"500 QPS"可能是"5 秒内的累计"（不是单秒），实际每秒 100（限流生效）。第三，**看 Sentinel 的数据源**——如果规则存在 Nacos（动态配置），Nacos 推送有延迟（1-3 秒），规则修改后没及时推送，旧规则不生效。检查 Sentinel 日志的 `RuleManager: rules updated` 时间戳。第四，**看限流维度**——如果规则是"按 uid 限流"（热点参数限流），但请求的 uid 在 header 里没传（Sentinel 拿不到 uid），限流退化成"全局 QPS"（所有用户共享 100 QPS，单个用户能占全部）。检查 `ParamFlowArgument` 是否正确提取 uid。

### 第三层：根因深挖

**Q：你发现"重试导致雪崩"——下游 LLM 服务偶发超时，上游重试（3 次），结果流量翻 3 倍，把下游打挂。重试本来是"应对偶发故障"，为什么反而放大了故障？**

根因是"重试放大效应"。第一，**重试的乘数效应**——如果每个请求重试 3 次（成功率 90% 的服务，3 次重试后成功率 99.9%），正常情况下重试只占 10% 的额外流量。但当服务"开始变慢"（P99 从 100ms 升到 500ms），更多请求超时触发重试，重试请求又加重服务负载，服务更慢，更多超时，更多重试——正反馈循环。极端情况下，重试流量可达原始流量的 3-10 倍（雪崩）。第二，**重试风暴的触发点**——当服务的"超时率"超过某个阈值（如 30%），重试流量 = 原始流量 × 30% × 3 = 90% 额外流量，总流量 1.9 倍，服务扛不住，超时率升到 50%，重试流量 = 1.5 倍，总流量 2.5 倍，继续恶化。第三，**解法**——（1）**限制总重试率**：网关统计"当前重试请求比例"，超过 10% 停止重试（直接失败）；（2）**指数退避 + 抖动**：重试间隔指数增长（1s/2s/4s）+ 随机抖动（避免重试请求同步到达）；（3）**熔断兜底**：错误率 > 50% 时熔断（停止所有请求，给下游恢复时间）；（4）**重试预算**：设"重试配额"（如最多 10% 的请求可以重试），超配额不重试。

**Q：那为什么不直接禁止重试？没有重试就不会有重试风暴。偶发超时直接失败，用户重试（人工）不就行了？**

禁止重试的代价是"用户体验差 + 成功率高时浪费"。第一，**偶发超时的比例**——正常情况下，LLM 服务的超时率 0.1-1%（偶发网络抖动、GPU 调度延迟）。如果不重试，这 1% 的用户直接看到"请求失败"，要手动重发（体验差）。重试能把失败率从 1% 降到 0.01%（3 次重试后），用户体验好。第二，**重试的收益**——99% 的重试是"成功"的（偶发故障后重试就成功），只有 1% 的重试是"徒劳"（服务真的挂了）。禁止重试 = 为了 1% 的极端情况牺牲 99% 的正常情况。第三，**正确做法**——不是"禁止重试"，是"智能重试"。（1）只在"偶发故障"时重试（超时/连接错误），不在"业务错误"时重试（400 Bad Request 重试也是 400）；（2）限制重试次数（1-2 次够了，3 次边际收益低）；（3）限制总重试率（< 10%）；（4）熔断兜底（服务真挂了就停止重试）。重试是"双刃剑"，用好是容错利器，用不好是雪崩推手，关键是"限流 + 退避 + 熔断"三件套配合。

### 第四层：方案权衡

**Q：LLM 推理超时设 30 秒（因为 LLM 生成慢，可能 20 秒才出完）。但 30 秒超时太长，用户等不了。你设 30 秒还是 10 秒？**

分层超时 + 流式缓解。第一，**LLM 的延迟特征**——TTFT（首 token）通常 100-500ms，TPOT（每 token）50-100ms。输出 200 token 的总时间 = 500ms + 200×80ms = 16.5 秒。如果超时设 10 秒，长回答会被截断（用户看到不完整）。设 30 秒，用户等 30 秒才看到结果（体验差）。第二，**流式输出是关键**——用 SSE（Server-Sent Events）流式返回，用户看到"首 token"后（500ms），后续 token 逐个显示，体验是"打字机效果"（感知快）。即使总耗时 20 秒，用户因为"看到在动"而不觉得慢。第三，**分层超时**——（1）首 token 超时 2 秒（TTFT < 2s，否则失败，用户立即知道"出错了"）；（2）后续 token 间歇超时 5 秒（两个 token 之间超 5 秒才失败，正常间隔 80ms，5 秒说明卡住了）；（3）总超时 60 秒（绝对上限，防止无限生成）。这样"快失败"（首 token 超时）+ "流式体验"（用户看到输出）+ "兜底"（总超时）。拼多多客服 LLM：首 token < 1s、TPOT < 100ms、总超时 30s（流式），用户体验好。

**Q：那为什么不设"无限超时"（等 LLM 输出完为止）？反正是异步流式，用户边看边等。**

无限超时的风险是"资源耗尽 + 恶意占用"。第一，**资源耗尽**——LLM 推理占用 GPU + 连接（HTTP 连接/线程）。如果允许无限超时，一个"生成 10 万 token"的请求（耗时 10000 秒）会占用一个 GPU 连接 3 小时。1000 个这样的请求 = 1000 个连接挂起（连接池耗尽），新请求无法接入。第二，**恶意占用**——攻击者故意发"写一篇 100 万字的小说"的请求，生成 100 万 token（100000 秒 = 27 小时），占着 GPU 不放。无限超时 = DoS 漏洞。第三，**超长的无意义**——LLM 输出超过 2000 token 后，质量通常下降（重复/跑题），用户不需要"无限长"。设 max_tokens=2000（硬限制输出长度）+ 总超时 60 秒（兜底），既保证质量又防资源耗尽。生产必须设超时上限，"无限"是系统设计的大忌。

### 第五层：验证与沉淀

**Q：你怎么证明"token 限流 + 智能重试"防止了雪崩，而不只是"幸亏那天流量不大"？**

压力测试验证。第一，**混沌测试**——主动注入故障（让下游 LLM 服务的延迟升到 2 秒，模拟故障），观察"无限流 + 无限重试"vs"token 限流 + 智能重试"的表现。前者：流量翻 3 倍，服务 OOM，全挂（雪崩）；后者：重试率限制在 10%，超限请求快速失败，服务稳定（QPS 降到正常的 70%，但没挂）。第二，**监控指标对比**——故障期间对比"重试请求比例"（前者 200%，后者 10%）、"错误率"（前者 80% 全挂，后者 30% 部分失败）、"恢复时间"（前者 30 分钟人工干预，后者 5 分钟自动恢复）。第三，**历史故障复盘**——回顾上一次真实雪崩（重试导致），对比"如果当时有智能限流会怎样"（模拟），证明限流策略能避免同类故障。三个验证（混沌 + 监控 + 复盘）证明策略有效，不是"幸亏没出事"。

**Q：限流降级的经验怎么沉淀，让新服务不踩"重试风暴"的坑？**

三件事。第一，**限流降级模板**——按服务类型（LLM 推理/Web 服务/搜索）预设限流策略模板（LLM：token 限流 + 首 token 超时 + 流式；Web：QPS 限流 + 1 秒超时 + 快速失败），新服务套模板。第二，**重试规范**——公司级规范："所有重试必须带指数退避 + 随机抖动，总重试率不超过 10%，必须配熔断"。Code Review 时检查重试逻辑是否合规，不合规不许上线。第三，**混沌演练常态化**——每月做一次混沌测试（注入延迟/故障），验证限流降级策略是否生效。把"故障应对"当成"需要持续验证的能力"，不是"配一次就不管"。监控 `retry_storm_risk_score`（重试率 × 服务负载，综合指标），> 阈值告警，提前干预。限流降级是"系统稳定性底线"，必须工程化、标准化、持续验证。

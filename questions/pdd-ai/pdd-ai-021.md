# 【拼多多 AI 中台】AI 中台整体架构怎么设计？

> JD 依据："消费者服务策略算法中台、规则中台、特征平台、实验平台、模型服务"。

## 一、AI 中台要解决的问题

**传统痛点**：
```
业务 A：自建特征平台 + 推荐模型 + A/B
业务 B：自建特征平台 + 风控模型 + A/B
业务 C：自建特征平台 + 搜索模型 + A/B
→ 重复造轮子，特征定义不一致，资源浪费
```

**中台目标**：沉淀共性能力（特征/规则/模型/实验/LLM），统一标准化对外。

## 二、分层架构

```
┌────────────────────────────────────────────────────┐
│ 应用层（业务）                                      │
│ 推荐召回 | 搜索排序 | 风控决策 | 智能客服 | 代码助手 │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────▼───────────────────────────────┐
│ 服务层（推理网关 + API）                            │
│ - 模型推理服务（Triton/vLLM/TRT-LLM）              │
│ - 规则引擎服务（Drools/QLExpress）                 │
│ - LLM 服务（Chat/RAG/Agent）                       │
│ - 决策编排（多模型/规则/LLM 串联）                  │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────▼───────────────────────────────┐
│ 模型层（训练 + 管理）                               │
│ - 传统 ML：Spark MLlib / XGBoost                   │
│ - 深度学习：PyTorch / TensorFlow                   │
│ - 大模型：预训练 / SFT / RLHF / LoRA               │
│ - 模型注册中心（MLflow）：版本/血缘/部署            │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────▼───────────────────────────────┐
│ 特征层（特征平台）                                  │
│ - 离线特征仓（Hive/HDFS）                          │
│ - 实时特征（Flink + Redis）                        │
│ - 在线查询 API                                     │
│ - 特征注册中心（血缘/版本/质量）                    │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────▼───────────────────────────────┐
│ 数据层                                              │
│ 业务库 CDC → Kafka → 实时/离线计算 → ODS/DWD/DWS   │
└────────────────────────────────────────────────────┘

横切：
┌────────────────────────────────────────────────────┐
│ 实验平台（A/B + 灰度 + 指标）                       │
│ 资源调度（GPU/CPU 池化 + K8s）                      │
│ 监控治理（延迟/QPS/效果/成本）                      │
│ 安全合规（脱敏/审计/PII）                           │
└────────────────────────────────────────────────────┘
```

## 三、各层详解

### 1. 数据层
```
业务库（MySQL）→ CDC（Canal/Debezium）→ Kafka
                                       → Flink 实时
                                       → Hive 离线

数仓分层：ODS（原始）→ DWD（明细）→ DWS（汇总）→ ADS（应用）
特征工程在 DWS 之上构建
```

### 2. 特征层（见专题）
- 离线：Spark/Hive T+1
- 实时：Flink 流 + Redis
- 查询：Feature Store API

### 3. 模型层
```
传统 ML：
  - 训练：Spark MLlib/XGBoost
  - 部署：PMML/ONNX → Triton

深度学习：
  - 训练：PyTorch/TF
  - 部署：ONNX/TRT → Triton

大模型：
  - 预训练（中心化团队）
  - 微调：SFT/LoRA/RLHF
  - 部署：vLLM/TensorRT-LLM

模型注册中心（MLflow）：
  - 模型元数据/版本/血缘
  - 模型 → 服务自动部署
```

### 4. 服务层
```
推理网关：
  - 鉴权/路由/限流/熔断
  - A/B 实验分流
  - 协议转换（HTTP/gRPC）

推理后端：
  - Triton（CV/NLP/推荐）
  - vLLM/TRT-LLM（LLM）
  - Drools/QLExpress（规则）

决策编排：
  - 多模型/规则/LLM 编排
  - 工作流引擎（如自研 DAG）
```

### 5. 应用层
各业务方按需调用中台能力，专注业务逻辑。

## 四、横切能力

### 实验平台
- A/B 分流（正交分层）
- 灰度发布
- 指标度量（实时 + 离线）
- 显著性检验

### 资源调度
```
GPU 池：
  - 在线推理（高优）
  - 离线训练（中优）
  - 实验/批量（低优）
K8s + GPU operator 调度
显存/CPU/带宽隔离
```

### 监控治理
- 性能：延迟/QPS/错误率（Prometheus）
- 效果：模型 AUC/CTR/GMV（业务指标）
- 成本：GPU 利用率/单次推理成本
- 漂移：特征/预测分布漂移告警

### 安全合规
- 数据脱敏（PII）
- 模型审计（谁/何时/改了什么）
- 输出过滤（敏感词/合规）

## 五、大模型时代的演进

新增 **LLM 工程层**：
```
LLM 训练：
  - 预训练（中心化，万亿 token）
  - 微调（SFT/LoRA，业务侧）
  - 对齐（RLHF/DPO）

LLM 推理：
  - 推理引擎（vLLM/TRT-LLM）
  - KV Cache 管理
  - 量化/PD 分离

LLM 应用：
  - RAG（向量库 + 检索）
  - Agent（工具调用/编排）
  - Prompt 管理（版本/AB）

LLMOps：
  - Prompt/模型评估（自动+人工）
  - 灰度/回滚
  - 监控（延迟/幻觉/成本）
```

## 六、典型业务接入流程

```
1. 业务接入申请（产品 + 技术对接）
2. 数据接入（数据源/特征需求）
3. 模型选型（直接用平台模型 vs 自训）
4. 部署（推理服务自动起）
5. 实验设计（A/B 指标）
6. 灰度放量（1%→5%→20%→100%）
7. 监控运维（效果/性能/成本）
8. 迭代（新版本/调优）
```

## 七、中台建设原则

1. **业务驱动**：按实际需求沉淀，不预设能力
2. **标准化**：接口/协议/数据格式统一
3. **可演进**：架构分层清晰，方便加新能力（如 LLM 层）
4. **可复用**：能力沉淀一次，多业务复用
5. **不过度**：小步快跑，业务先用起来再说

## 八、拼多多 AI 中台实践

```
能力沉淀：
  - 特征平台（用户/商品/行为特征，亿级 QPS）
  - 模型服务（推荐/搜索/风控，多模态）
  - 实验平台（千实验并行）
  - 规则中台（营销/风控规则）
  - LLM 中台（Chat/RAG/Agent）

技术栈：
  - Java（业务网关/服务编排）
  - Python（模型训练/推理）
  - Flink（实时计算）
  - Spark/Hive（离线）
  - Triton/vLLM（推理引擎）
  - K8s（部署调度）
```

## 九、底层本质

AI 中台本质是**"AI 能力的标准化 + 平台化 + 复用化"**——把分散在业务的特征/规则/模型/实验/LLM 能力沉淀成平台，业务像调水电一样用 AI。分层架构（数据/特征/模型/服务/应用）+ 横切治理（实验/监控/调度/安全）+ 大模型时代新增 LLM 工程层。是公司级 AI 降本提效的核心基础设施。

## 常见考点

1. **中台和业务怎么分工**？——中台提供能力（特征/模型/实验），业务专注逻辑（调用 + 决策）；中台不写业务代码。
2. **怎么评估中台价值**？——复用度（多少业务接入）、降本（避免重复建设的费用）、提效（业务上线周期缩短）。
3. **大模型时代传统 ML 还需要吗**？——需要。LLM 适合自然语言/泛化场景；推荐/风控等强数据驱动场景，传统 ML 仍是性价比首选。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说中台的核心价值是"复用"。但拼多多有几十个业务线（电商/多多买菜/多多视频/多多游戏），每个业务的 AI 需求差异很大（电商要推荐/CTR，视频要内容审核/标签）。这种"业务差异大"的情况下，中台怎么复用？是不是每个业务还是得自建？**

中台的复用是"能力复用"而非"逻辑复用"。第一，**复用的是"能力平台"**——特征平台（所有业务都算特征/存特征/取特征）、模型服务（所有业务都要部署模型/做推理）、实验平台（所有业务都要 A/B 测试）。这些"平台能力"是通用的，业务方接入后用"自己的数据和逻辑"。第二，**不复用的是"业务策略"**——推荐算法（商品推荐的排序逻辑）、风控规则（什么交易拦截），这些是业务特定的，由业务团队自己写，中台不碰。第三，**分层复用**——底层基础设施（GPU 集群、K8s 调度）全公司共享；中层平台（特征/模型/实验）跨业务复用；上层业务逻辑各业务自建。类比：阿里云是"基础设施中台"，所有公司用同一套云，但每家公司跑自己的业务。拼多多 AI 中台也是"内部云"，业务方租用能力，不自建。衡量标准：**复用度 = 接入中台的业务数 / 总业务数**，目标是 > 70%。

### 第二层：证据与定位

**Q：业务方抱怨"中台的能力不够灵活，我们要的功能平台没有，还是得自建"。你怎么定位是"中台能力缺失"还是"业务方需求不合理"？**

分情况判断。第一，**看需求频次**——如果这个需求（比如"自定义推荐召回逻辑"）只有 1 个业务提，是"个性化需求"，中台不做（做了不通用，维护成本高）；如果 3+ 业务都提，是"共性需求"，中台应该做。判断标准：需求是否可抽象为通用能力。第二，**看替代方案**——业务方要的"功能"能否用中台现有能力组合实现？比如"自定义召回"可以拆成"特征查询（中台有）+ 规则引擎（中台有）"，业务方组合即可。如果中台提供了"原子能力"，组合不出来是业务方的能力问题（不会用），不是中台缺失。第三，**看接入数据**——统计业务方的"中台 API 调用量"，如果调用量高（说明大量用中台）但抱怨多，可能是"用了但不够灵活"，要增强；如果调用量低（说明没怎么用），抱怨可能是"没接入就吐槽"，要沟通。数据说话，不凭主观判断。

### 第三层：根因深挖

**Q：你发现"中台越来越重，新业务接入慢"——业务方要接入特征平台，结果要学 3 个 DSL（特征定义/特征查询/特征监控）、对接 4 个系统（Flink/Spark/Redis/HBase）。接入周期 2 周。根因是什么？**

根因是"中台的抽象泄漏"——底层实现的复杂性暴露给了业务方。第一，**DSL 过多**——特征定义有 DSL（Spark 算），特征查询有 DSL（Feature Store API），监控有 DSL（Prometheus rule）。业务方为了加一个特征要学 3 套语言，认知负担重。解法：**统一 DSL**——一套 Feature DSL 覆盖"定义 + 计算 + 查询"，业务方只写一次，中台编译成各系统的代码。第二，**系统耦合**——业务方要对接 Flink（实时特征）、Spark（离线特征）、Redis（在线查询），每个系统有自己的 API。解法：**一站式平台**——业务方在 Web 控制台填表（特征名/计算逻辑/更新频率），中台自动生成 Flink/Spark 作业 + Redis schema，业务方不碰底层系统。第三，**接入文档/SDK 缺失**——业务方第一次接入没人指导。解法：**自助接入工具 + 新手引导**——提供 CLI 脚本（自动生成项目骨架 + 示例代码）+ onboarding 文档（5 分钟跑通 hello world）。目标：接入周期从 2 周降到 2 天。

**Q：那为什么不做一个"全图形化"的中台？业务方拖拖拽拽（低代码）就能定义特征/模型/实验，不用写代码。这样不是最简单吗？**

低代码适合"简单场景"，复杂场景还是要代码。第一，**低代码的表达力有限**——特征计算的复杂逻辑（多表 JOIN、窗口聚合、时间序列处理）用拖拽组件表达，组件会爆炸（几十种组件），且组合起来难调试。代码（SQL/Python）表达力强，算法同学更熟悉。第二，**版本管理/Code Review**——低代码的"配置"存在数据库里，不能像代码一样 Git 管理（diff、review、回滚）。生产级特征/模型定义要经过 Code Review（防止错误逻辑上线），低代码做不到。第三，**混合方案**——简单场景（加一个计数特征）用低代码（Web 控制台填表），复杂场景（多流 JOIN 的实时特征）用代码（DSL + Git）。拼多多特征平台：80% 简单特征用 Web 控制台（自助），20% 复杂特征用 Feature DSL（算法同学写）。不是"全低代码"或"全代码"，而是"按复杂度分级"。

### 第四层：方案权衡

**Q：中台要支持 LLM 应用（推理/RAG/Agent）。但 LLM 是新领域，中台团队对 LLM 的理解不如业务团队深。中台应该"等业务验证了再跟进"还是"主动投入 LLM 能力"？**

两种策略的风险不同。**"等业务验证再跟进"（跟随策略）**——优点是稳妥（不浪费投入）、中台做的是"已被证明的需求"；缺点是滞后（业务等不及中台，会先自建 LLM 能力，中台后续整合难），且 LLM 迭代快，等验证完可能技术已变。**"主动投入 LLM 能力"（引领策略）**——优点是抢占先机（业务方一开始就用中台的 LLM 能力，不重复造轮子）；缺点是风险（可能投入了错误的技术方向，比如中台押注 TGI 但市场选了 vLLM）。**拼多多实践**——"部分引领 + 部分跟随"。基础设施（推理平台/LLMOps 监控）主动投入（这些是通用的，不会错）；业务层能力（RAG/Agent 框架）跟随业务（业务方先验证，中台把验证过的方案抽象成平台能力）。判断标准：**技术方向不确定的跟随，基础设施确定的引领**。LLM 的"推理平台/监控"是确定的（都要），主动做；"Agent 框架"是不确定的（每家做法不同），等业务验证。

**Q：大模型时代，中台会不会被 LLM 颠覆？LLM 能做特征工程/实验分析/模型部署，是不是中台就没用了？**

LLM 不会颠覆中台，而是"升级中台"。第一，**LLM 改变的是"交互方式"**——以前业务方写 SQL/DSL 算特征，未来可以用自然语言（"帮我算一个近 7 天的购买频次特征"），LLM 翻译成 DSL。但"特征计算引擎/存储/查询"还是中台，LLM 只是前端。第二，**LLM 改变的是"能力边界"**——以前中台只做 ML（特征/模型/实验），现在要加 LLM 工程（推理/RAG/Agent）。中台的范畴扩大，不是被颠覆。第三，**LLM 不能替代"基础设施"**——LLM 自己也要跑在推理平台上（GPU 调度/监控），这个推理平台就是中台。LLM 是"中台服务的一个新模型类型"，不是"中台的替代品"。结论：中台会演进（加 LLM 能力），不会消亡。传统 ML 中台（特征/模型/实验）继续存在（推荐/风控还用），LLM 中台（推理/RAG/Agent）是新增。两个中台融合（统一调度/监控），是趋势。

### 第五层：验证与沉淀

**Q：你怎么证明中台真的创造了价值（复用/降本/提效），而不是"增加了中间层、拖慢了业务"？**

三个维度的量化指标。第一，**复用度**——统计"接入中台的业务数 / 总业务数"和"中台 API 日调用量"。如果 30 个业务接入、日均 50 亿次 API 调用，说明中台被广泛使用。如果调用量低（如 100 万/天），说明业务没用中台（自建了），中台价值存疑。第二，**降本**——对比"自建成本"和"中台成本"。如果没有中台，30 个业务各自建特征平台，每个 5 人 × 1 年 = 150 人年。中台团队 20 人维护 1 个平台 = 20 人年，省 130 人年。量化成钱：130 人 × 50 万年薪 = 6500 万/年。第三，**提效**——对比"业务接入中台 vs 自建"的上线周期。接入中台：2 天（调 API）。自建：2 个月（搭 Flink/Spark/Redis）。提效 30 倍。三个指标（复用度 + 降本 + 提效）一起看，证明中台价值。如果三个指标都差，中台确实没价值，应该裁撤。

**Q：中台的演进经验怎么沉淀，避免"做成大而全的怪物"？**

三件事。第一，**能力边界明确**——中台只做"通用能力"（特征/模型/实验/推理），不做"业务逻辑"（推荐策略/风控规则）。边界清晰的中台才不会膨胀。定期 review：哪些能力是"通用"的（保留），哪些是"某个业务的特殊需求"（剥离）。第二，**按需沉淀**——不预设能力（"我觉得业务会需要 X 所以先做"），而是等 3+ 业务有同样需求再沉淀。避免"投入了没人用"的浪费。第三，**解耦设计**——中台的每个能力（特征/模型/实验）独立部署、独立迭代，业务方按需接入（不用整个中台全接入）。解耦让中台"轻量化"，业务方"按需取用"。衡量中台健康度的指标：**能力使用率**（每个能力有多少业务用）、**接入时长**（新业务接入要多久）、**稳定性**（中台故障影响多少业务）。三个指标都健康，中台才可持续。

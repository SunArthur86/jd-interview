# 【拼多多 AI 中台】AI 中台整体架构怎么设计？

> JD 依据："消费者服务策略算法中台、规则中台、特征平台、实验平台、模型服务"。

## 一、AI 中台要解决的问题

**传统痛点**：
```
业务 A：自建特征平台 + 推荐模型 + A/B
业务 B：自建特征平台 + 风控模型 + A/B
业务 C：自建特征平台 + 搜索模型 + A/B
→ 重复造轮子，特征定义不一致，资源浪费
```

**中台目标**：沉淀共性能力（特征/规则/模型/实验/LLM），统一标准化对外。

## 二、分层架构

```
┌────────────────────────────────────────────────────┐
│ 应用层（业务）                                      │
│ 推荐召回 | 搜索排序 | 风控决策 | 智能客服 | 代码助手 │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────▼───────────────────────────────┐
│ 服务层（推理网关 + API）                            │
│ - 模型推理服务（Triton/vLLM/TRT-LLM）              │
│ - 规则引擎服务（Drools/QLExpress）                 │
│ - LLM 服务（Chat/RAG/Agent）                       │
│ - 决策编排（多模型/规则/LLM 串联）                  │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────▼───────────────────────────────┐
│ 模型层（训练 + 管理）                               │
│ - 传统 ML：Spark MLlib / XGBoost                   │
│ - 深度学习：PyTorch / TensorFlow                   │
│ - 大模型：预训练 / SFT / RLHF / LoRA               │
│ - 模型注册中心（MLflow）：版本/血缘/部署            │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────▼───────────────────────────────┐
│ 特征层（特征平台）                                  │
│ - 离线特征仓（Hive/HDFS）                          │
│ - 实时特征（Flink + Redis）                        │
│ - 在线查询 API                                     │
│ - 特征注册中心（血缘/版本/质量）                    │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────▼───────────────────────────────┐
│ 数据层                                              │
│ 业务库 CDC → Kafka → 实时/离线计算 → ODS/DWD/DWS   │
└────────────────────────────────────────────────────┘

横切：
┌────────────────────────────────────────────────────┐
│ 实验平台（A/B + 灰度 + 指标）                       │
│ 资源调度（GPU/CPU 池化 + K8s）                      │
│ 监控治理（延迟/QPS/效果/成本）                      │
│ 安全合规（脱敏/审计/PII）                           │
└────────────────────────────────────────────────────┘
```

## 三、各层详解

### 1. 数据层
```
业务库（MySQL）→ CDC（Canal/Debezium）→ Kafka
                                       → Flink 实时
                                       → Hive 离线

数仓分层：ODS（原始）→ DWD（明细）→ DWS（汇总）→ ADS（应用）
特征工程在 DWS 之上构建
```

### 2. 特征层（见专题）
- 离线：Spark/Hive T+1
- 实时：Flink 流 + Redis
- 查询：Feature Store API

### 3. 模型层
```
传统 ML：
  - 训练：Spark MLlib/XGBoost
  - 部署：PMML/ONNX → Triton

深度学习：
  - 训练：PyTorch/TF
  - 部署：ONNX/TRT → Triton

大模型：
  - 预训练（中心化团队）
  - 微调：SFT/LoRA/RLHF
  - 部署：vLLM/TensorRT-LLM

模型注册中心（MLflow）：
  - 模型元数据/版本/血缘
  - 模型 → 服务自动部署
```

### 4. 服务层
```
推理网关：
  - 鉴权/路由/限流/熔断
  - A/B 实验分流
  - 协议转换（HTTP/gRPC）

推理后端：
  - Triton（CV/NLP/推荐）
  - vLLM/TRT-LLM（LLM）
  - Drools/QLExpress（规则）

决策编排：
  - 多模型/规则/LLM 编排
  - 工作流引擎（如自研 DAG）
```

### 5. 应用层
各业务方按需调用中台能力，专注业务逻辑。

## 四、横切能力

### 实验平台
- A/B 分流（正交分层）
- 灰度发布
- 指标度量（实时 + 离线）
- 显著性检验

### 资源调度
```
GPU 池：
  - 在线推理（高优）
  - 离线训练（中优）
  - 实验/批量（低优）
K8s + GPU operator 调度
显存/CPU/带宽隔离
```

### 监控治理
- 性能：延迟/QPS/错误率（Prometheus）
- 效果：模型 AUC/CTR/GMV（业务指标）
- 成本：GPU 利用率/单次推理成本
- 漂移：特征/预测分布漂移告警

### 安全合规
- 数据脱敏（PII）
- 模型审计（谁/何时/改了什么）
- 输出过滤（敏感词/合规）

## 五、大模型时代的演进

新增 **LLM 工程层**：
```
LLM 训练：
  - 预训练（中心化，万亿 token）
  - 微调（SFT/LoRA，业务侧）
  - 对齐（RLHF/DPO）

LLM 推理：
  - 推理引擎（vLLM/TRT-LLM）
  - KV Cache 管理
  - 量化/PD 分离

LLM 应用：
  - RAG（向量库 + 检索）
  - Agent（工具调用/编排）
  - Prompt 管理（版本/AB）

LLMOps：
  - Prompt/模型评估（自动+人工）
  - 灰度/回滚
  - 监控（延迟/幻觉/成本）
```

## 六、典型业务接入流程

```
1. 业务接入申请（产品 + 技术对接）
2. 数据接入（数据源/特征需求）
3. 模型选型（直接用平台模型 vs 自训）
4. 部署（推理服务自动起）
5. 实验设计（A/B 指标）
6. 灰度放量（1%→5%→20%→100%）
7. 监控运维（效果/性能/成本）
8. 迭代（新版本/调优）
```

## 七、中台建设原则

1. **业务驱动**：按实际需求沉淀，不预设能力
2. **标准化**：接口/协议/数据格式统一
3. **可演进**：架构分层清晰，方便加新能力（如 LLM 层）
4. **可复用**：能力沉淀一次，多业务复用
5. **不过度**：小步快跑，业务先用起来再说

## 八、拼多多 AI 中台实践

```
能力沉淀：
  - 特征平台（用户/商品/行为特征，亿级 QPS）
  - 模型服务（推荐/搜索/风控，多模态）
  - 实验平台（千实验并行）
  - 规则中台（营销/风控规则）
  - LLM 中台（Chat/RAG/Agent）

技术栈：
  - Java（业务网关/服务编排）
  - Python（模型训练/推理）
  - Flink（实时计算）
  - Spark/Hive（离线）
  - Triton/vLLM（推理引擎）
  - K8s（部署调度）
```

## 九、底层本质

AI 中台本质是**"AI 能力的标准化 + 平台化 + 复用化"**——把分散在业务的特征/规则/模型/实验/LLM 能力沉淀成平台，业务像调水电一样用 AI。分层架构（数据/特征/模型/服务/应用）+ 横切治理（实验/监控/调度/安全）+ 大模型时代新增 LLM 工程层。是公司级 AI 降本提效的核心基础设施。

## 常见考点

1. **中台和业务怎么分工**？——中台提供能力（特征/模型/实验），业务专注逻辑（调用 + 决策）；中台不写业务代码。
2. **怎么评估中台价值**？——复用度（多少业务接入）、降本（避免重复建设的费用）、提效（业务上线周期缩短）。
3. **大模型时代传统 ML 还需要吗**？——需要。LLM 适合自然语言/泛化场景；推荐/风控等强数据驱动场景，传统 ML 仍是性价比首选。

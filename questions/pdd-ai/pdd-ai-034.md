# 【拼多多 AI 中台】LLM 推理优化进阶（投机解码/前缀缓存）怎么做？

> JD 依据："推理加速、KV Cache"。

## 一、投机解码（Speculative Decoding）

### 原理
```
传统 Decode（串行）：
  token1 → token2 → token3 → ... → tokenN
  每步都要读全部权重，慢

投机解码（并行验证）：
  1. Draft 模型（小，快）快速生成 k 个候选 token：t1, t2, ..., tk
  2. Target 模型（大，准）一次 forward 并行验证这 k 个
  3. 接受前 j 个（分布匹配的），从第 j+1 个重新生成
  4. 若 j=k（全对），额外免费生成 1 个（target 自己产）

效果：单次 forward 处理 k 个 token，2-3x 加速
```

### 数学保证
```
接受/拒绝策略保证最终分布等同 Target 模型：
- Draft 概率 ≤ Target：接受
- Draft 概率 > Target：按比例拒绝（避免分布偏移）
- 拒绝后从调整分布重新采样

→ 投机解码不改变输出分布（无损）
```

### 实现框架
- **Medusa**：Target 模型加多个解码头，并行预测
- **EAGLE**：用 Target 的 hidden state 训 Draft，准确率高
- **vLLM/TensorRT-LLM**：内置投机解码支持

### 代码示例（vLLM）
```bash
vllm serve Qwen/Qwen2-72B \
    --speculative-model Qwen/Qwen2-1.5B \   # draft 模型
    --num-speculative-tokens 5 \             # 每次猜 5 个
    --speculative-draft-tensor-parallel-size 1
```

### 适用条件
```
- Draft 准确率高（>50%）→ 否则浪费算力
- Draft 和 Target 分布相近（同家族模型，如 Qwen 系列）
- 长生成场景（短生成收益小）
- 用户：Draft 用 0.5B/1.5B，Target 用 7B/72B
```

## 二、前缀缓存（Prefix Caching）

### 原理
```
大量请求共享 system prompt：
"你是拼多多客服，请按规则回答用户问题..."
+
用户 A 输入：A 的问题
用户 B 输入：B 的问题
用户 C 输入：C 的问题

传统：每个请求都重新计算 system prompt 的 KV（重复浪费）
前缀缓存：system prompt 的 KV 算一次，多请求复用
```

### 实现
```
1. 识别请求的前缀（按 token 序列哈希）
2. 第一次请求：算 KV，缓存（带版本）
3. 后续相同前缀请求：直接用缓存 KV（跳过 prefill）
4. 只算用户输入部分的 KV

效果：
- 共享 system prompt 场景吞吐 2-5x
- TTFT 显著降低（不用重算 prompt）
```

### vLLM 启用
```bash
vllm serve Qwen/Qwen2-72B --enable-prefix-caching
```

### 命中条件
- 完全相同前缀（token 序列）
- 顺序敏感（中间不能差）
- 缓存淘汰策略（LRU/大小上限）

### 进阶：自动前缀缓存
- 不只 system prompt，对话历史也可缓存
- 跨请求复用（同用户多轮对话）
- 智能识别可缓存片段

## 三、模型蒸馏（Distillation）

### 原理
```
Teacher（大模型）→ Student（小模型）

方法 1：硬标签蒸馏
  - Teacher 生成回答，Student 模仿
  - 学生学老师的输出

方法 2：软标签蒸馏
  - 用 Teacher 的 logits（概率分布）
  - KL 散度损失
  - 学生学老师的"思考过程"

方法 3：特征蒸馏
  - 中间层特征对齐
```

### 效果
```
72B Teacher → 7B Student：
  - 任务：客服对话
  - 准确率：保持 95%+
  - 推理快 10x，成本低 10x
```

### 工具
- LLMLingma/PandaLM
- 自研（基于 HuggingFace Trainer）

## 四、剪枝（Pruning）

### 类型
```
权重剪枝：去不重要的权重（接近 0 的）
结构剪枝：
  - 头剪枝（Attention 头去冗余）
  - 层剪枝（去不重要层）
  - 通道剪枝

稀疏度：50% 稀疏（一半权重为 0）
硬件支持：稀疏矩阵乘需要专门支持
```

### 效果
- 模型变小（显存省）
- 推理快（如果硬件支持稀疏）
- 精度损失（要重新校准）

### 工具
- Wanda/SparseGPT（学术）
- TensorRT Model Optimizer

## 五、Medusa / EAGLE（多头投机）

### Medusa
```
Target 模型上加多个解码头（额外训练）：
  - Head 1：预测下一 token
  - Head 2：预测下下 token
  - ...

一次 forward 产 k 个候选，并行验证
优势：不用单独 Draft 模型
劣势：要训 Medusa Head
```

### EAGLE
```
基于 Target 的 hidden state 训 Draft：
  - Draft 看 Target 的隐状态
  - 准确率更高（70%+）
  - 加速更明显（3x+）
```

## 九、底层本质

LLM 推理优化进阶本质是**"打破 Decode 串行瓶颈 + 复用计算 + 压缩模型"**——投机解码把串行变并行（小猜大验），前缀缓存复用共享 KV（system prompt 不重算），蒸馏/剪枝降模型大小。这些技术组合能把 72B 模型推理成本降到原 1/4-1/5，是 LLM 大规模生产的必经之路。

## 常见考点

1. **投机解码为什么不改变输出分布**？——接受/拒绝策略保证最终概率等同 Target，Draft 准则只是加速器。
2. **前缀缓存命中率怎么提高**？——固定 system prompt + 模板化 + 跨请求复用 + 智能片段识别。
3. **蒸馏和微调区别**？——蒸馏是让 Student 模仿 Teacher（保留能力降规模），微调是让模型适应特定任务（不改变规模）。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试宫层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：投机解码用"小模型生成草稿 + 大模型验证"。但小模型会出错（生成错误的候选），大模型要"拒绝"错误候选重算。如果小模型准确率低（拒绝率高），投机解码反而比直接用大模型慢。小模型的准确率要多高才划算？**

投机解码的收益取决于"接受率"（accept rate）。第一，**接受率与加速比的关系**——投机解码的加速比 = `1 / (1 - acceptance_rate × draft_length)`。如果接受率 90%、draft 长 4 token，加速比 = 1/(1-0.9×4)... 这个公式不对，实际是"每个验证 step 平均接受的 token 数"。接受率 90% 意味着平均接受 4 个候选中的 3.4 个（每步生成 3.4 个 token），加速比约 2-3 倍（大模型验证 4 个 token 是并行的，比串行生成 4 个快）。接受率 50%，平均接受 1.5 个，加速比约 1.3 倍（收益小）。接受率 < 30%，加速比 < 1（比直接用大模型还慢，因为多了小模型的开销）。第二，**接受率的影响因素**——（1）小模型和大模型的"分布相似度"：如果小模型是"同家族的小版本"（如 Qwen-7B 和 Qwen-72B 同源），分布相似，接受率高（80%+）；如果不同家族（GPT-2 验证 Llama），接受率低（30%）。（2）任务的确定性：数学/代码任务（答案确定）接受率高，创意写作（随机性大）接受率低。（3）draft 长度：draft 短（2-3 token）接受率高（小错误少），draft 长（8 token）接受率低（更容易错一个就全拒）。第三，**生产选择**——用"同家族小模型"（Qwen-7B 做 Qwen-72B 的 draft）、draft 长度 3-5 token、接受率 > 70% 才开启投机解码。拼多多客服（Qwen-72B）：用 Qwen-1.8B 做 draft，接受率 82%，加速 1.8 倍。

### 第二层：证据与定位

**Q：投机解码上线后，你发现 TPOT（每 token 延迟）不降反升（从 80ms 升到 120ms）。你怎么定位？**

看接受率和 draft 开销。第一，**看接受率**——vLLM 暴露 `spec_decode_acceptance_rate` 指标，如果接受率才 40%（低于预期 70%+），说明 draft 模型的输出经常被拒绝，每次拒绝要"大模型串行重算"，比不用投机解码还慢。接受率低的根因：（1）draft 模型和大模型不同源（分布差异大）；（2）任务不适合（创意写作，大模型随机性强，小模型猜不准）。第二，**看 draft 开销**——即使接受率高，draft 模型的生成时间也是开销。如果 draft 模型生成 4 token 要 30ms，大模型验证 4 token 要 50ms（并行），总 80ms；但直接用大模型生成 1 token 要 80ms（串行）。投机解码生成 4 token 80ms（20ms/token），直接生成 1 token 80ms（80ms/token），投机快 4 倍。但如果 draft 模型慢（生成 4 token 要 80ms），投机解码总 130ms（draft 80 + verify 50），比直接 80ms 慢。第三，**解法**——（1）换更小的 draft 模型（Qwen-1.8B 比 Qwen-7B 快）；（2）降低 draft 长度（从 5 token 降到 3，减少 draft 时间）；（3）如果接受率持续低（< 50%），关闭投机解码（不适配这个任务）。

### 第三层：根因深挖

**Q：你发现前缀缓存（Prefix Caching）的命中率只有 30%（预期 80%+）。system prompt 是固定的，为什么命中率这么低？**

前缀缓存的命中率取决于"实际请求的前缀重合度"。第一，**system prompt 的"固定"是相对的**——如果不同业务（客服/搜索/代码）用不同的 system prompt，缓存里有多套 prompt，单套命中率低。第二，**用户 query 的位置影响**——如果请求是 `system_prompt + few_shot + user_query`，前缀是 `system_prompt + few_shot`（固定），命中率高。但如果请求是 `system_prompt + user_history + user_query`（多轮对话，history 变化），前缀不固定（history 每轮变），命中率低。第三，**缓存淘汰**——vLLM 的 KV Cache 池有限，如果并发请求多，旧的 prefix 被淘汰（LRU），新请求来时缓存没了，命中率低。第四，**排查**——看 vLLM 的 `prefix_cache_hit_blocks` 和 `prefix_cache_miss_blocks`，对比 hit/miss 的 prompt。如果 miss 的 prompt 前缀各不相同（history 变化），是"前缀不固定"问题；如果前缀相同但 miss，是"缓存被淘汰"问题。第五，**解法**——（1）统一 system prompt（跨业务用同一套，提升单套命中率）；（2）多轮对话用"session 亲和"（同会话路由到同实例，复用已有 KV）；（3）增大 KV Cache 池（降 gpu_memory_utilization 留更多 block）。拼多多客服：统一 system prompt + session 亲和路由，前缀命中率从 30% 提升到 75%。

**Q：那为什么不缓存"整个对话历史"（不只 system prompt），这样多轮对话的后续轮次都能命中？**

缓存整个对话历史有"管理复杂度"。第一，**KV Cache 的细粒度是 block**——vLLM 的 PagedAttention 以 block（16 token）为单位管理 KV Cache。缓存"整个对话历史"要存"所有历史 token 的 KV"，多轮对话的历史不断增长（每轮加几百 token），缓存膨胀。第二，**缓存复用条件严格**——前缀缓存只在"完全匹配前缀"时复用。如果多轮对话的历史是 [系统] + [用户1] + [助手1] + [用户2]，第二轮请求的前缀是 [系统] + [用户1] + [助手1]，要缓存这部分。第三轮请求的前缀是 [系统] + [用户1] + [助手1] + [用户2] + [助手2]，更长。每轮的前缀都不同（递增），缓存要存"每轮的前缀 KV"。第四，**vLLM 的实现**——vLLM 的 Prefix Caching 确实支持"任意前缀匹配"，会自动复用"请求 prompt 开头能匹配到缓存 block"的部分。如果多轮对话的同会话路由到同实例，前缀缓存会复用（因为同实例有之前的 KV）。第五，**生产实践**——多轮对话用"session 亲和"（同会话同实例）+ Prefix Caching，每轮只算"新增 token 的 KV"，省 80%+ 的 Prefill。但跨会话/跨实例不共享 KV（缓存隔离），所以"session 亲和"是前提。拼多多：多轮对话强制 session 亲和路由，配合 Prefix Caching，第 3 轮的 TTFT 从 200ms 降到 50ms（省了前 2 轮的 KV 重算）。

### 第四层：方案权衡

**Q：模型蒸馏（用大模型蒸馏小模型）能降本，但蒸馏的小模型能力不如原大模型。你怎么决定"蒸馏到多小"？7B 够用还是必须蒸馏到 1.5B？**

蒸馏规模按"业务效果 + 成本"平衡。第一，**效果基线**——用大模型（72B）在业务的黄金集（500 条）跑准确率（如 88%），这是"上限"。蒸馏的 7B 可能 85%、1.5B 可能 78%。如果业务 SLO 要求准确率 > 82%，7B 够用（85% > 82%），1.5B 不够（78% < 82%）。第二，**成本对比**——7B 推理成本是 72B 的 1/10，1.5B 是 1/50。如果 7B 的准确率（85%）满足 SLO，用 7B（性价比最高），不蒸馏到 1.5B（牺牲准确率换成本不划算）。如果 7B 不满足（准确率 80% < SLO 82%），要么用 72B（贵但准），要么优化蒸馏（用更多数据/更好方法，让 7B 达到 82%）。第三，**蒸馏的优化**——（1）用更多蒸馏数据（大模型生成 100 万条高质量回答，蒸馏给小模型）；（2）分阶段蒸馏（先蒸馏到 7B，再用 7B 蒸馏到 1.5B，级联）；（3）针对任务蒸馏（只蒸馏业务相关的数据，小模型在"客服任务"上接近大模型，其他任务不管）。第四，**拼多多实践**——客服 LLM 用 72B 蒸馏到 7B（准确率 85%，成本 1/10），SLO 达标。1.5B 只用于"简单 FAQ"（准确率 80%，超低成本），复杂查询路由到 7B。分级模型，按任务难度选规模。

**Q：那为什么不直接用量化（INT4）降成本，而要费事蒸馏？量化一个 72B 模型到 INT4，成本也降 4 倍，不用训练。**

量化和蒸馏的"降本路径"不同，适用场景不同。第一，**量化的效果**——72B INT4 的显存从 140GB 降到 35GB，能放 1 张 H100（vs FP16 要 2 张），硬件成本降 2 倍。但推理速度（throughput）提升有限（INT4 主要省显存/带宽，算力瓶颈场景提速少）。第二，**蒸馏的效果**——蒸馏到 7B，显存 14GB（vs 72B INT4 的 35GB），推理速度快 5-10 倍（7B 的算力需求远低于 72B）。蒸馏降本更激进（5-10 倍 vs 2 倍）。第三，**效果损失**——72B INT4 几乎无损（准确率 87% vs FP16 的 88%），因为只是精度降低，模型容量不变。蒸馏到 7B 有明显损失（85%），因为模型容量降了（参数少）。第四，**适用场景**——"效果敏感 + 成本适中"用量化（72B INT4，准且便宜）；"成本极致 + 效果可接受"用蒸馏（7B，便宜但略差）。第五，**组合方案**——蒸馏到 7B + 量化 7B 到 INT4，双重降本。7B INT4 显存 3.5GB，一张 L4（24GB）能跑 6 个实例，推理成本是 72B FP16 的 1/50。拼多多客服：72B INT4 用于复杂查询（效果优先），7B INT4 用于简单 FAQ（成本优先），分级部署。

### 第五层：验证与沉淀

**Q：你怎么证明"投机解码"真的把延迟降了 1.8 倍，而不是"流量小所以快"？**

控制变量对比。第一，**离线 benchmark**——固定 workload（1000 个 prompt，固定 batch），对比"开启投机解码"vs"关闭投机解码"的 TPOT。开启：TPOT 45ms，关闭：TPOT 80ms，加速 1.8 倍。流量固定，差异来自投机解码。第二，**线上 A/B**——开启和关闭各 50% 流量，同时段对比 `vllm:time_per_output_token_p99`。开启 P99=50ms，关闭 P99=85ms。第三，**接受率监控**——`spec_decode_acceptance_rate` 持续 > 80%（预期范围），证明 draft 模型适配（不是偶然高）。第四，**成本对比**——投机解码多了 draft 模型的 GPU 开销（1.8B 模型占 5% GPU），但大模型（72B）的吞吐提升 1.8 倍，净收益是"吞吐提升 1.7 倍"（1.8 / 1.05）。综合：延迟降 1.8 倍 + 吞吐升 1.7 倍，证明投机解码生效。

**Q：推理优化的经验怎么沉淀，让新模型快速应用优化？**

三件事。第一，**优化矩阵**——维护"优化技术 × 效果 × 适用场景"的表（前缀缓存省 Prefill 50%、投机解码加速 1.8 倍需要同源小模型、量化省显存 2-4 倍几乎无损、蒸馏降本 5-10 倍有效果损失），新模型查表选优化组合。第二，**自动 benchmark**——每次新模型上线，自动跑 benchmark（开启/关闭各种优化），对比 throughput/TTFT/TPOT/显存，选最优配置。不手动调参。第三，**优化效果监控**——线上监控 `prefix_cache_hit_rate`、`spec_decode_acceptance_rate`、`quantization_error_proxy`，优化效果退化（命中率降/接受率降）告警。推理优化不是"一次性配置"，要持续监控效果（新版本的 vLLM 可能改变优化行为），定期重新 benchmark 确认最优配置。把推理优化当"持续工程"，不是"配一次就不管"。

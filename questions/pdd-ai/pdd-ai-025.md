# 【拼多多 AI 中台】实验平台架构怎么设计？A/B + 灰度怎么做？

> JD 依据："实验平台、消费者服务策略算法中台"。

## 二、分流引擎

### 基础分流
```java
public String assign(String userId, String layerName, LayerConfig config) {
    // 正交：每层用不同 hash salt
    String key = layerName + ":" + userId;
    int bucket = Math.abs(MurmurHash3.hash(key)) % 10000;   // 10000 桶

    // 按 config 分配到实验组
    for (Variant v : config.getVariants()) {
        if (bucket < v.getRangeEnd()) return v.getName();
    }
    return "default";
}
```

### 正交分层（关键）
```
分层目的：让多实验互不影响

层 1（推荐算法）：用户分桶 0-99
  桶 0-49 → 算法 A
  桶 50-99 → 算法 B

层 2（UI 主题）：用 hash(uid + "layer2") 重新分桶
  桶 0-49 → 红色
  桶 50-99 → 蓝色

由于两层 hash salt 不同，桶分配独立
→ 同一用户可能是"算法 A + 红色"或"算法 A + 蓝色"
→ 两实验正交（不互相干扰）

支持层内互斥（同层只能进一个实验）
层间正交（不同层独立分流）
```

### 流量互斥（同层）
```
层 1：5 个实验
  实验 1：桶 0-19（20%）
  实验 2：桶 20-39（20%）
  实验 3：桶 40-59（20%）
  对照：桶 60-99（40%）
同层用户只能进一个实验（互斥）
```

## 四、显著性检验

```
T 检验：
  H0: 实验组均值 = 对照组均值
  H1: 实验组均值 ≠ 对照组均值

算 t 值 → 查 p 值
  p < 0.05 → 拒绝 H0 → 显著（策略有效）

效应大小（Cohen's d）：
  d = (mean_exp - mean_ctrl) / pooled_std
  d > 0.2 才算有业务意义（不只看 p 值）

置信区间：
  mean ± 1.96 × se
  不包含 0 即显著
```

## 五、加速实验（拼多多/Google 等用）

### 1. Interleaving（穿插评估）
```
传统 A/B：
  组 A 看模型 A 结果，组 B 看模型 B 结果
  对比 CTR，需 1-2 周

Interleaving：
  同一用户混排两模型结果（A1 B1 A2 B2 ...）
  看用户更点谁的（隐式反馈）
  → 1-2 天分胜负
  → 速度 10-100x

适合：排序/推荐（搜索结果/商品流）
不适合：UI/定价（无法穿插）
```

### 2. CUPED（方差缩减）
```
用预实验协变量 X（历史指标）调整：
  Y_adjusted = Y - theta × (X - mean(X))
  theta = cov(Y, X) / var(X)

方差减小 → 同样样本量检测更小效应
→ 实验时长缩短 30-50%
```

### 3. 序贯检验（Sequential Testing）
```
边跑边检验（不固定样本量）
达显著性即可停
避免"跑完才发现样本不够"
```

## 六、灰度发布

```
新模型/策略上线流程：
1. 沙箱测试（历史数据回放）
2. 内部白名单（员工先用）
3. 小流量灰度（1% → 5%）
4. 放量（20% → 50%）
5. 全量（100%）
6. 监控异常 → 秒级回滚

灰度维度：
- 按用户百分比
- 按用户分群（新老/地域/设备）
- 按城市逐步开放
```

## 七、典型实验流程

```
1. 立项
   - 假设：用新推荐模型能提升 GMV 2%
   - 指标：GMV（核心）+ 留存（护栏）
   - 流量：10%（5% 实验 + 5% 对照）
   - 时长：14 天（覆盖周期）

2. 分流配置
   - 在层 X 配置实验组/对照组
   - 设置白名单（员工）
   - 互斥检查（同层其他实验）

3. 灰度启动
   - 0.1% → 1% → 5% → 10%
   - 每阶段监控异常

4. 数据积累
   - 实时：Flink 算核心指标（每小时更新）
   - 离线：Hive T+1 全量

5. 分析
   - 显著性检验
   - 效应大小
   - 分群下钻（新用户/老用户/不同城市）
   - 长期影响（次日/7 日留存）

6. 决策
   - 显著正向 → 放量
   - 显著负向 → 放弃/调整
   - 不显著 → 加流量/换指标/放弃

7. 沉淀
   - 实验报告存档
   - 结论可复用
   - 模型注册更新
```

## 十、底层本质

实验平台本质是**"分流 + 度量 + 检验 + 加速"**——正交分层支持千实验并行，双链路指标保证实时+离线，显著性检验做科学裁判，Interleaving/CUPED 加速迭代。是 AI/算法/营销决策的"裁判"，每个策略上线都过平台。

## 常见考点

1. **SRM（样本比失衡）是什么**？——实验组/对照组实际流量比 ≠ 配置比，说明分流有 bug 或样本选择偏差，要告警。
2. **辛普森悖论怎么避免**？——分群分析（新/老用户分开看），保证分流在子群体内也均匀。
3. **实验结果和长期效果冲突怎么办**？——短期可能负向（用户适应期）但长期正向，需更长周期或 holdout 验证。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：实验平台用"正交分层"支持千实验并行。但正交分层的原理是"每层独立分流"，意味着同一用户在不同层会被分到不同实验组。这不会"互相污染"吗？比如用户在推荐层是实验组（看到新策略），在搜索层是对照组（看到旧策略），体验不矛盾吗？**

正交分层的设计是"各层策略互不相关"，不是"用户体验一致"。第一，**正交的数学保证**——每层用独立的 hash 种子（`hash(uid, layer_id)`），不同层的分组结果统计独立。这意味着用户在推荐层的"实验/对照"和在搜索层的"实验/对照"无关（4 种组合各占 25%）。第二，"污染"的定义是"实验 A 影响了实验 B 的指标"——如果推荐层的新策略（实验 A）提升了点击率，这个点击行为会影响到搜索层的实验（因为用户点击了推荐商品后，搜索行为变化）。这就是"溢出效应"。第三，**缓解措施**——（1）同层互斥（同一层内的多个实验互斥，用户只参加一个，避免本层叠加）；（2）跨层假设独立（假设推荐和搜索的影响可叠加，用线性模型分析）；（3）对"强关联的层"不放正交（如果推荐和搜索强耦合，合并成一层做实验）。拼多多：推荐层、搜索层、广告层正交（假设独立），商品详情层、购物车层同层互斥（强关联）。正交不是"完美无污染"，是"在可控假设下的工程妥协"。

### 第二层：证据与定位

**Q：一个 A/B 实验跑了 7 天，实验组 CTR 比对照组高 3%（p 值 0.04，显著）。但业务方质疑"会不会是 SRM（样本比失衡）导致的偏差"。你怎么排查 SRM？**

SRM 检测是"看实验组和对照组的实际样本量是否等于配置比例"。第一，**算 SRM 卡方检验**——配置是 50/50，实际实验组 51000 人、对照组 49000 人（比例 51:49，不是 50:50）。用卡方检验算 `χ² = (51000-50000)²/50000 + (49000-50000)²/50000 = 4`，p 值 0.046，如果 < 0.01 说明有 SRM（实际样本比 ≠ 配置比）。第二，**SRM 的根因**——（1）分流 bug（hash 函数不均匀）；（2）实验组体验差（用户流失多，留下来的是"幸存者"，样本偏倚）；（3）日志丢失（实验组的日志丢了部分，统计到的样本少）。第三，**定位方法**——按维度（设备/地区/新老用户）拆分看 SRM。如果"老用户"无 SRM 但"新用户"有 SRM（实验组新用户少），可能是实验组的新用户引导差（新用户流失）。如果所有维度都有 SRM，是分流 bug（查 hash 函数）。第四，**处理**——发现 SRM 后，即使 p 值显著，也不能下结论（样本有偏）。先修 SRM（修 bug / 延长实验到样本均衡），再重新评估。

### 第三层：根因深挖

**Q：你发现实验结果"实验组 CTR +3%"在老用户成立（+5%），但在新用户不成立（-2%）。这是辛普森悖论吗？怎么处理？**

这是"分群效果不一致"，辛普森悖论的极端情况是"总体正向但各分群都负向"。第一，**判断辛普森悖论**——总体 +3%，老用户 +5%、新用户 -2%，不是辛普森悖论（分群不都负向）。但"新用户负向"说明策略对新用户有害，总体 +3% 是被老用户的大基数（占比 80%）掩盖。第二，**根因分析**——新用户 CTR -2%，可能是策略（比如"个性化推荐"）对新用户无效（新用户没有历史行为，个性化算不准，反而不如老策略的"热门推荐"）。老用户 CTR +5%，个性化基于历史行为，效果好。第三，**处理策略**——（1）分群发布：老用户上新策略（+5%）、新用户保持老策略（不 -2%）；（2）策略优化：新用户用"冷启动策略"（热门推荐）+ 老用户用个性化，融合策略；（3）继续实验：如果新用户样本少（方差大），-2% 可能不显著，延长实验到新用户样本够（看 p 值）。第四，**警示**——如果不做分群分析，直接看总体 +3% 全量上线，会损害新用户体验（虽然总体涨）。实验分析必须"分群下钻"，不能只看总体。

**Q：那为什么不把实验跑久一点（比如 30 天），样本足够大，p 值就准了？**

延长实验时间不一定解决问题，且有副作用。第一，**样本量的边际收益递减**——统计学上，p 值的显著性取决于"效应大小 × √N"。效应大小固定（比如 +3%），样本翻倍（7 天→14 天），p 值的 z 分数 √2 倍。但样本翻 4 倍（7 天→28 天）z 分数才 2 倍。延长实验的收益递减（不是线性提升显著性）。第二，**Novelty Effect（新奇效应）**——新策略上线初期，用户因为"新鲜感"点击多（CTR 暂时高），过几天新鲜感消退，CTR 回落。7 天的实验可能"高估"了效果（包含新奇效应），30 天的实验更准（新奇效应消退）。但 30 天成本高（实验占了流量 30 天，其他实验等不了）。第三，**时间成本**——互联网迭代快，7 天实验已经长了（要求"周级迭代"），30 天实验拖慢决策。第四，**生产选择**——用 CUPED（方差缩减）缩短实验周期（7 天的 CUPED 等效于 14 天的普通实验）、Interleaving（穿插测试，1-2 天出结果）。不是"无脑延长"，是"用统计方法加速"。

### 第四层：方案权衡

**Q：实验结果显著但业务方说"3% 的提升太小，不值得全量"。你怎么判断一个实验值不值得全量？**

值不值得是"收益 vs 成本 + 风险"的综合判断。第一，**收益量化**——3% 的 CTR 提升，按"日 PV 10 亿 × CTR 5% × 3% = 1500 万次额外点击"，每次点击的商业价值（如广告收入 0.1 元）= 年增收 5.5 亿。如果收益 > 成本，值得。第二，**成本**——新策略的维护成本（代码复杂度、服务器开销）。如果新策略要额外 10 台机器（年 50 万），收益 5.5 亿 >> 成本 50 万，值得。如果新策略要额外 1000 张 GPU（年 5000 万），成本占比 10%，要评估。第三，**风险**——新策略的"长期风险"（是否损害用户体验、是否合规、是否有护城河）。即使短期 +3%，如果长期"让用户疲劳"（点击多但转化少），不值得。第四，**决策框架**——（1）收益 > 成本 × 3 且风险可控 → 全量；（2）收益 > 成本但风险高 → 小范围灰度（如 20% 流量持续观察）；（3）收益 < 成本 → 不全量（实验报告归档）。不是"显著就全量"，是"显著 + 值得 + 低风险"才全量。

**Q：为什么不直接全量上线新策略（如果离线评估好），而要先 A/B 实验？A/B 占用流量、拖慢迭代，离线评估不是更简单吗？**

离线评估无法替代 A/B 实验，有三个根本局限。第一，**离线数据的分布偏差**——离线评估用历史数据，但"新策略会影响用户行为"（比如新推荐策略让用户点不同的商品），产生新的数据分布，离线数据无法反映。只有在线 A/B 能看到"用户对新策略的真实反应"。第二，**离线指标 ≠ 业务指标**——离线评估看 AUC/NDCG（排序质量），但业务看 CTR/转化率/GMV。AUC 提升不一定 CTR 提升（可能有 position bias、展示因素）。只有在线 A/B 能直接测业务指标。第三，**长期效应**——离线评估是"单次推理"（快照），看不到"新策略上线后用户的长期行为变化"（疲劳、适应、流失）。A/B 实验（7-14 天）能捕捉中期效应。结论：离线评估是"必要但不充分"——用于筛掉明显差的策略（离线差的不用 A/B），好的策略必须 A/B 验证。拼多多的流程：离线评估（筛 50%）→ 小流量 A/B（5%，验 3 天）→ 大流量 A/B（20%，验 7 天）→ 全量。每层有 gate，不跳过。

### 第五层：验证与沉淀

**Q：你怎么证明实验平台的分流是"均匀的"（每个实验组的用户特征分布一致）？**

A/A 实验验证。第一，**A/A 实验**——配置两个"完全相同"的策略（A 和 A），分流到两组用户，理论上两组的指标应该无显著差异（p 值 > 0.05）。如果 A/A 实验出现显著差异（p < 0.05），说明分流不均匀（有 SRM 或 hash bug）。第二，**多次 A/A**——跑 100 次 A/A 实验，统计"出现假阳性（p < 0.05）的比例"。如果假阳性率是 5%（符合统计预期），分流均匀；如果假阳性率 15%，分流有问题（某些用户群被系统性分到某组）。第三，**分群均衡性**——A/A 实验中，按"新老用户/设备/地区"拆分，每组内 A 组 vs A 组的样本量应均衡（卡方检验 p > 0.05）。如果不均衡（如某地区 A 组多 10%），是分流在该维度有偏。拼多多实验平台：每周自动跑 20 个 A/A 实验，监控假阳性率和分群均衡性，异常告警。分流引擎是"实验可信的基础"，必须持续验证。

**Q：实验平台的经验怎么沉淀，让新实验不踩坑（SRM/辛普森/短期效应）？**

三件事。第一，**实验配置模板**——按实验类型（UI 改版/算法策略/推荐召回）预设模板（样本量计算/实验周期/分群维度/护栏指标），新实验套模板，不重复设计。模板里包含"必须检查 SRM、必须分群分析、必须设护栏指标（如留存率/投诉率防负面）"。第二，**自动化分析报告**——实验结束后，自动生成分析报告（总体效果 + 分群效果 + SRM 检验 + 护栏指标 + 显著性），不是实验员手动算。报告里有"红黄绿灯"（绿灯全量、黄灯灰度、红灯不全量），辅助决策。第三，**实验知识库**——每个实验的结论（效果/分群/风险）记录在知识库，新实验设计时参考历史（"同类策略的历史效果如何"）。沉淀"实验方法论"（怎么算样本量、怎么选指标、怎么处理辛普森），新同学学方法论不重复踩坑。把实验平台当成"需要持续迭代的产品"，不是"配置一次就完事的工具"。

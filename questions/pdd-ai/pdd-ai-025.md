> JD 依据："实验平台、消费者服务策略算法中台"。

## 二、分流引擎

### 基础分流
```java
public String assign(String userId, String layerName, LayerConfig config) {
    // 正交：每层用不同 hash salt
    String key = layerName + ":" + userId;
    int bucket = Math.abs(MurmurHash3.hash(key)) % 10000;   // 10000 桶

    // 按 config 分配到实验组
    for (Variant v : config.getVariants()) {
        if (bucket < v.getRangeEnd()) return v.getName();
    }
    return "default";
}
```

### 正交分层（关键）
```
分层目的：让多实验互不影响

层 1（推荐算法）：用户分桶 0-99
  桶 0-49 → 算法 A
  桶 50-99 → 算法 B

层 2（UI 主题）：用 hash(uid + "layer2") 重新分桶
  桶 0-49 → 红色
  桶 50-99 → 蓝色

由于两层 hash salt 不同，桶分配独立
→ 同一用户可能是"算法 A + 红色"或"算法 A + 蓝色"
→ 两实验正交（不互相干扰）

支持层内互斥（同层只能进一个实验）
层间正交（不同层独立分流）
```

### 流量互斥（同层）
```
层 1：5 个实验
  实验 1：桶 0-19（20%）
  实验 2：桶 20-39（20%）
  实验 3：桶 40-59（20%）
  对照：桶 60-99（40%）
同层用户只能进一个实验（互斥）
```

## 三、指标体系

| 类型 | 例子 | 计算链路 |
|------|------|---------|
| 核心指标 | GMV/CTR/转化 | 实时 + 离线 |
| 护栏指标 | 留存/卸载/客诉 | 实时 + 离线 |
| 辅助指标 | 时长/点击数 | 离线 |
| 业务质量 | 延迟/错误率 | 实时 |

**指标定义元数据**：
```yaml
metric: gmv_per_user
type: ratio
numerator: SUM(order_amount)
denominator: COUNT(DISTINCT user_id)
window: 7d
direction: higher_better
```

## 四、显著性检验

```
T 检验：
  H0: 实验组均值 = 对照组均值
  H1: 实验组均值 ≠ 对照组均值

算 t 值 → 查 p 值
  p < 0.05 → 拒绝 H0 → 显著（策略有效）

效应大小（Cohen's d）：
  d = (mean_exp - mean_ctrl) / pooled_std
  d > 0.2 才算有业务意义（不只看 p 值）

置信区间：
  mean ± 1.96 × se
  不包含 0 即显著
```

## 五、加速实验（拼多多/Google 等用）

### 1. Interleaving（穿插评估）
```
传统 A/B：
  组 A 看模型 A 结果，组 B 看模型 B 结果
  对比 CTR，需 1-2 周

Interleaving：
  同一用户混排两模型结果（A1 B1 A2 B2 ...）
  看用户更点谁的（隐式反馈）
  → 1-2 天分胜负
  → 速度 10-100x

适合：排序/推荐（搜索结果/商品流）
不适合：UI/定价（无法穿插）
```

### 2. CUPED（方差缩减）
```
用预实验协变量 X（历史指标）调整：
  Y_adjusted = Y - theta × (X - mean(X))
  theta = cov(Y, X) / var(X)

方差减小 → 同样样本量检测更小效应
→ 实验时长缩短 30-50%
```

### 3. 序贯检验（Sequential Testing）
```
边跑边检验（不固定样本量）
达显著性即可停
避免"跑完才发现样本不够"
```

## 六、灰度发布

```
新模型/策略上线流程：
1. 沙箱测试（历史数据回放）
2. 内部白名单（员工先用）
3. 小流量灰度（1% → 5%）
4. 放量（20% → 50%）
5. 全量（100%）
6. 监控异常 → 秒级回滚

灰度维度：
- 按用户百分比
- 按用户分群（新老/地域/设备）
- 按城市逐步开放
```

## 七、典型实验流程

```
1. 立项
   - 假设：用新推荐模型能提升 GMV 2%
   - 指标：GMV（核心）+ 留存（护栏）
   - 流量：10%（5% 实验 + 5% 对照）
   - 时长：14 天（覆盖周期）

2. 分流配置
   - 在层 X 配置实验组/对照组
   - 设置白名单（员工）
   - 互斥检查（同层其他实验）

3. 灰度启动
   - 0.1% → 1% → 5% → 10%
   - 每阶段监控异常

4. 数据积累
   - 实时：Flink 算核心指标（每小时更新）
   - 离线：Hive T+1 全量

5. 分析
   - 显著性检验
   - 效应大小
   - 分群下钻（新用户/老用户/不同城市）
   - 长期影响（次日/7 日留存）

6. 决策
   - 显著正向 → 放量
   - 显著负向 → 放弃/调整
   - 不显著 → 加流量/换指标/放弃

7. 沉淀
   - 实验报告存档
   - 结论可复用
   - 模型注册更新
```

## 九、AI 中台典型实验场景

```
- 推荐模型迭代：新召回/排序模型 vs 旧模型
- Prompt 实验：不同 LLM prompt 模板对比
- RAG 实验：不同召回策略对比
- 推荐策略：召回→排序→重排各环节实验
- 定价/促销：不同折扣策略
- 客服 LLM：不同模型版本/温度对比
- 代码助手：不同补全策略对比开发者效率
```

## 十、底层本质

实验平台本质是**"分流 + 度量 + 检验 + 加速"**——正交分层支持千实验并行，双链路指标保证实时+离线，显著性检验做科学裁判，Interleaving/CUPED 加速迭代。是 AI/算法/营销决策的"裁判"，每个策略上线都过平台。

## 常见考点

1. **SRM（样本比失衡）是什么**？——实验组/对照组实际流量比 ≠ 配置比，说明分流有 bug 或样本选择偏差，要告警。
2. **辛普森悖论怎么避免**？——分群分析（新/老用户分开看），保证分流在子群体内也均匀。
3. **实验结果和长期效果冲突怎么办**？——短期可能负向（用户适应期）但长期正向，需更长周期或 holdout 验证。

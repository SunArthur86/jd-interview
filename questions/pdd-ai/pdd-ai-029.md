# 【拼多多 AI 中台】预案和回滚怎么做？钱学森工程理论怎么应用？

> JD 依据："稳定性治理、模型服务"。

## 一、为什么需要预案

**反面案例**：
```
某 LLM 服务挂 → 值班同学慌 → 临时找运维 → 找研发 → 改配置 → 重启 → 30 分钟才恢复
→ 损失百万

教训：故障期才想解决方案必然慢，必须预案化。
```

**预案价值**：
- 故障时秒级响应（一键执行）
- 减少决策成本（不用现场想）
- 可演练验证（平时证明有效）

## 二、预案体系

### 1. 按场景分类

| 场景 | 预案 |
|------|------|
| 流量突增 | 限流 + 扩容 + 削峰 |
| 服务过载 | 限流 + 降级（小模型/兜底） |
| 依赖故障 | 熔断 + 降级 |
| 单实例挂 | 自动摘除 + 流量切换 |
| 集群挂 | 切灾备机房（多活） |
| 数据异常 | 数据回滚 + 缓存清理 |
| 模型效果差 | 回滚到旧版本 |
| 大促保障 | 提前扩容 + 关非核心 |

### 2. 预案模板
```yaml
name: LLM 推理过载
trigger:
  metric: gpu_utilization
  threshold: "> 95%"
  duration: 60s
actions:
  - type: rate_limit
    target: llm-gateway
    config: { qps: 5000 }    # 全局限流到 5000
  - type: degrade
    target: llm-inference
    config: { model: small-7b }  # 切小模型
  - type: scale
    target: llm-vllm
    config: { replicas: 50 }     # 扩容到 50
notifications:
  - 钉钉群
  - 值班手机
rollback:
  auto_when: gpu_utilization < 70% for 5min
  actions: [恢复限流, 切回大模型]
owner: AI 中台
```

### 3. 自动化触发
```java
@Component
public class AutoMitigation {

    @Scheduled(fixedDelay = 10000)
    public void check() {
        double gpuUtil = metrics.get("gpu_utilization");
        double errorRate = metrics.get("error_rate");

        if (gpuUtil > 0.95) {
            executor.execute(Playbook.LLM_OVERLOAD);    // 自动执行预案
        }
        if (errorRate > 0.1) {
            executor.execute(Playbook.ROLLBACK_LATEST);  // 自动回滚
        }
    }
}
```

## 三、回滚机制

### 1. 版本回滚（模型/代码）
```
传统：发版前备份 → 出问题恢复（分钟级）

蓝绿发布：
  - 蓝环境跑 v1，绿环境跑 v2
  - 切流量到绿，验证 OK
  - 出问题秒级切回蓝

金丝雀发布：
  - 新版本先 1% → 5% → 20% → 100%
  - 每阶段监控，异常自动回滚

K8s 原生：
  kubectl rollout undo deployment/llm-vllm
  → 秒级回滚到上一个版本
```

### 2. 配置回滚
```
所有配置走配置中心（Nacos/Apollo）：
  - 每次变更存版本
  - 出问题秒级切回旧版本
  - 灰度发布（5% → 100%）
```

### 3. 流量回滚
```
多机房部署：
  - 切流量到备用机房（DNS/SLB）
  - 单机房故障秒级切换

灰度切流量：
  - 新版本异常 → 流量切回旧版本
```

### 4. 数据回滚（最复杂）
```
特征/数据变更要回滚：
  - DDL 变更前备份
  - 软删除（不真删）
  - 双写迁移（新旧并行验证）
  - 限速变更（避免大规模错误）
```

## 四、钱学森工程理论应用

钱学森倡导系统工程方法（出自《工程控制论》），三原则：

### 1. 预先研究（Pre-research）
```
- 关键技术提前研究（不临阵磨枪）
- AI 中台案例：
  - 大促前 3 个月预测流量，预先扩容
  - 新模型上线前沙箱回放历史流量
  - 新硬件（H100/FP8）提前适配
```

### 2. 仿真验证（Simulation）
```
- 不直接上线试，先仿真
- AI 中台案例：
  - 全链路压测（影子流量模拟）
  - 故障注入（Chaos Engineering 混沌工程）
  - A/B 实验沙箱回放
  - 灾难演练（拔网线/关机）
```

### 3. 反复迭代（Iterative）
```
- 不一次定型，小步快跑
- AI 中台案例：
  - 模型灰度发布（1% → 5% → 20% → 100%）
  - 预案定期演练（每季度）
  - 持续监控 + 反馈 + 优化
```

## 五、混沌工程（Chaos Engineering）

预案的反向验证：主动注入故障看预案是否生效。

```
Netflix Chaos Monkey 思路：
- 随机 kill Pod（看自愈能力）
- 注入网络延迟（看超时/重试）
- 模拟 GPU 故障（看切流）
- 拔机房（看多活）

拼多多：
- 每月一次"故障日"演练
- 注入各种故障 → 验证预案自动触发
- 找出预案漏洞（不生效/动作错）
- 改进预案
```

## 六、典型预案案例

### 案例 1：LLM 模型效果突降
```
触发：A/B 实验组 CTR 跌 20%
预案：
1. 自动切流量回旧版本（100% → 0%）
2. 通知值班
3. 排查新版本（prompt 变更/数据漂移）
4. 修复后重新灰度
```

### 案例 2：双 11 流量突增 10 倍
```
触发：QPS 超 50000（预测流量）
预案：
1. 自动扩容到预案规模（提前预热）
2. 限流阈值动态调整
3. 关闭非核心功能（个性化推荐）
4. LLM 切小模型（响应快）
```

### 案例 3：GPU 集群挂
```
触发：单 AZ GPU 故障率 50%
预案：
1. 流量切到备用 AZ（多活）
2. 自动扩容其他 AZ
3. 降级（限流 + 兜底）
```

## 九、底层本质

预案回滚本质是**"故障前预演 + 故障时自动 + 故障后迭代"**——按场景准备预案（限流/降级/扩容/切流），自动化触发（监控驱动），定期演练验证。钱学森系统工程的"预先研究 + 仿真验证 + 反复迭代"是其理论基础，AI 中台这种复杂系统尤其需要。混沌工程是反向验证预案有效性的关键。

## 常见考点

1. **预案自动化到什么程度**？——重复性高/低风险的全自动（限流/扩容），高风险的半自动（切流量/回滚，需确认）。
2. **回滚和重放区别**？——回滚是切回旧版本/状态，重放是重新执行历史操作（验证）。
3. **故障复盘重点是什么**？——找根因（不只是表象）+ 改进措施（防再次发生）+ 不指责人（指责流程/系统）。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试宫层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说预案要"自动化"。但 LLM 推理服务的故障类型多样（GPU OOM/模型幻觉/延迟飙升/依赖 Redis 挂），每种故障的应急动作不同。自动化怎么覆盖这么多场景？是不是所有故障都要自动化？**

不是所有故障都自动化，要按"频次 + 风险"分级。第一，**高频低风险 → 全自动**——如"GPU 利用率 > 90%"（频次高），自动触发"扩容 2 个实例"（风险低，扩容不会让情况更糟）。这类故障 80% 的场景，自动化能快速响应（30 秒内扩容），人工干预反而慢（值班响应 5 分钟）。第二，**低频高风险 → 半自动（人工确认）**——如"模型效果退化"（幻觉率突增），自动检测到异常，但"回滚模型版本"是高风险操作（可能影响其他正常用户），应该"告警 + 建议动作 + 人工一键确认"。自动化准备好的脚本（`pdd-rollback --model xxx --to v1.2`），值班同学确认后执行。第三，**未知故障 → 不自动化**——如"新类型的故障"（没预案的），自动化无法处理（没有预设动作）。这类要"告警 + 人工排查"，排查后把新预案沉淀到系统（下次遇到就自动化）。第四，**自动化原则**——"可逆操作自动"（扩容/限流/降级，出问题能回滚）、"不可逆操作半自动"（删数据/回滚/切流量，要确认）。LLM 推理的 80% 故障是"扩容/限流"（可逆），适合自动化。

### 第二层：证据与定位

**Q：凌晨 3 点告警"LLM 推理错误率从 0.1% 飙到 15%"。值班同学被叫醒，怎么在 10 分钟内定位根因？**

标准化排查 SOP。第一，**看告警的上下文**——告警不止"错误率 15%"，还有"哪些错误"（超时/500/模型报错）。如果错误是"timeout"（请求超时），是"延迟飙升"类故障（可能 GPU 过载/依赖服务慢）；如果是"OOM"（显存不足），是"资源耗尽"类故障；如果是"model error"（模型推理异常），是"模型/数据"类故障。第二，**看时间线**——错误率什么时候开始涨的（3:00），3:00 前后有什么变更（部署/配置/流量）。如果有 2:55 的 vLLM 版本升级，可能是升级引入的 bug。看变更日志（K8s 的 Deployment events、CI/CD 的部署记录）。第三，**看监控指标**——错误率 + GPU 利用率 + 显存 + QPS + 下游依赖（Redis/Kafka）一起看。如果"错误率涨 + GPU 利用率 100% + 显存 95%"，是 GPU 过载（流量突增或 batch 配置错）；如果"错误率涨 + Redis 延迟飙升"，是 Redis 故障（推理查不到特征）；如果"错误率涨 + GPU/Redis 正常"，可能是"模型 bug"（某类 prompt 触发模型崩溃）。第四，**快速验证**——手动发一个测试请求，看返回什么（是超时还是错误信息），错误信息会提示根因（如 `CUDA out of memory` = 显存问题）。

### 第三层：根因深挖

**Q：你定位到根因是"2:55 的 vLLM 版本升级"——新版本的 Continuous Batching 策略变了，导致 batch 过大、显存 OOM。为什么升级会引入这个 bug？测试没发现吗？**

根因是"测试环境和生产环境的 workload 不同"。第一，**测试环境的局限**——测试环境用标准 benchmark（固定 prompt 分布、固定并发），benchmark 跑下来 throughput 提升 10%（看起来更好），但没覆盖"生产的长尾请求"（5K token 的超长 prompt）。新版本的 batch 策略更激进（batch 更大，吞吐高），但遇到超长 prompt 时，batch 里的 KV Cache 暴增（5K token 的 KV Cache 是普通请求的 10 倍），显存 OOM。第二，**回归测试的盲区**——CI/CD 的回归测试只测"功能正确"（模型输出对不对），不测"显存边界"（生产流量下会不会 OOM）。压测在测试环境做（10 卡规模），没复现生产的"高峰流量 + 长尾 prompt"组合。第三，**新版本策略差异**——vLLM 的版本说明（changelog）提到"改进 Continuous Batching 策略"，但没有显式说明"对长 prompt 的显存影响"。开发同学升级时看了 changelog，认为"性能提升是好事"，没预判"激进 batching 的副作用"。第四，**教训**——（1）升级前要在"生产影子流量"上跑 24 小时（覆盖全天流量模式，包括长尾）；（2）新版本的"激进策略"要配置保守参数（`max_num_batched_tokens` 调小）；（3）监控 vLLM 的 `gpu_memory_usage_p99`，新版本上线后持续观察。

**Q：那为什么不让所有升级都"全量压测"（模拟生产流量）后再上线？这样就不会漏掉 bug。**

全量压测每次升级都做，成本和时间不可承受。第一，**时间成本**——全量压测要 24-48 小时（覆盖全天流量模式 + 各种长尾场景），每次升级都压测意味着"每周只能升 1 次"（压测 2 天 + 部署 1 天 + 观察 4 天），迭代速度太慢。LLM 推理引擎迭代快（vLLM 每月有新版本），不升级会落后。第二，**资源成本**——全量压测要一个和生产等规模的集群（100 卡），压测期间这个集群不能跑生产（或跑影子流量），GPU 成本翻倍。第三，**平衡方案**——（1）**canary 发布**：新版本先灰度到 1 个实例（5% 流量），观察 2-4 小时（覆盖一个流量周期），无异常再扩大到 20%、50%、100%。canary 能在"小流量"阶段发现问题（5% 流量的 OOM 不会全挂），比全量压测轻量。（2）**关键指标告警**：canary 期间监控 `oom_count`、`error_rate`、`latency_p99`，超阈值自动回滚（不用人工判断）。（3）**变更冻结期**：大促期间（双 11/618）冻结升级（风险高），平时允许 canary 升级。拼多多：vLLM 升级走 canery（1% → 20% → 100%，每阶段 2 小时），关键指标自动告警回滚，不依赖全量压测。

### 第四层：方案权衡

**Q：故障时"切流量到备用集群"是最快的恢复手段。但备用集群要常备（成本）。你怎么决定"建多大规模的备用集群"？**

备用集群的规模按"RTO（恢复时间目标）"决定。第一，**RTO 定义**——业务能容忍的最长恢复时间。客服 LLM 的 RTO=5 分钟（5 分钟内恢复，否则用户大量投诉），搜索广告的 RTO=1 分钟（1 分钟内恢复，否则广告收入损失大）。第二，**备用集群的作用**——主集群挂了，切到备用集群（DNS/网关切换），5 分钟内恢复。但"切到备用"的前提是备用集群"能接住全部流量"。如果主集群 100 卡、备用集群 10 卡（只够 10% 流量），切换后 90% 用户还是用不了。第三，**规模决策**——（1）**冷备**（备用集群不跑，模型镜像就绪）：切换前要"启动实例 + 加载模型"（5-10 分钟），RTO 长，但平时零成本（不占 GPU）。适合 RTO > 10 分钟的场景。（2）**热备**（备用集群常开，跑 10-20% 流量）：切换快（秒级切 DNS），但平时占 10-20% GPU 成本。适合 RTO < 1 分钟的关键业务。（3）**混合**：核心业务（客服/搜索）热备（20% 规模常开），非核心（内部工具）冷备。拼多多客服 LLM：热备 20% 规模（10 卡），主挂了切热备（秒级），热备扛不住时紧急扩容（再起 30 卡，5 分钟 ready）。

**Q：那为什么不用"多活"（同城双活/异地多活），所有集群都跑生产流量？这样故障时自动切流量，不用"备用集群"。**

多活是最高可用方案，但成本和复杂度最高。第一，**多活的成本**——"同城双活"要 2 倍资源（2 个集群都跑全量流量），"异地多活"要 3+ 倍资源。LLM 推理的 GPU 资源昂贵（100 卡集群，双活 = 200 卡，年成本多 5000 万）。第二，**多活的复杂度**——多活要解决"数据同步"（用户 session、对话历史要在多集群间同步）、"流量分配"（DNS/gateway 按地域/负载分流）、"一致性"（多集群写入如何冲突解决）。LLM 推理相对简单（无状态推理），但依赖的 Redis/DB 要多活（Redis Cluster 跨机房同步，复杂）。第三，**故障概率权衡**——LLM 推理服务的"集群级故障"概率低（K8s 集群挂的概率 < 0.1%/年），主要是"实例级故障"（单 Pod 挂，K8s 自动重启）。多活防御的是"机房级/地域级故障"（概率极低），ROI 不高。第四，**生产选择**——核心业务（支付/交易）做多活（故障代价 > 多活成本），LLM 推理做"热备 + 快速扩容"（够用）。拼多多 LLM 推理：单集群 + 热备 + 自动扩容（非多活），可用性 99.9%（年宕机 < 9 小时），够业务用。多活是"最后手段"，不是默认方案。

### 第五层：验证与沉淀

**Q：你怎么证明"预案自动化"真的能在故障时快速恢复，而不只是"纸面预案"？**

定期演练。第一，**混沌工程**——定期（每月）主动注入故障（如"杀掉 30% 的 vLLM 实例"模拟实例故障），观察自动化预案是否触发、是否正确执行、恢复时间是否达标（RTO < 5 分钟）。如果演练发现"预案没触发"（告警阈值不对）或"触发了但执行失败"（脚本 bug），及时修正。第二，**故障复盘指标**——每次真实故障后，统计 `time_to_detect`（发现时间，告警到人）、`time_to_mitigate`（恢复时间，人到恢复）、`time_to_resolve`（解决时间，恢复到根因修复）。目标是 `time_to_mitigate < RTO`（5 分钟），如果超了，说明预案不够自动化（人工干预慢）。第三，**预案覆盖率**——统计"故障类型 × 是否有预案"，覆盖率 = 有预案的故障类型 / 总故障类型。目标 > 90%（10% 的未知故障靠人工）。每次遇到"无预案的新故障"，复盘后补预案，覆盖率持续提升。拼多多 LLM 推理：预案覆盖率从 60%（初期）提升到 95%（2 年迭代），`time_to_mitigate` P50 从 15 分钟降到 2 分钟。

**Q：预案和故障复盘的经验怎么沉淀，让团队"越用越稳"？**

三件事。第一，**预案库**——所有预案（限流/扩容/切流/降级/回滚）存成"可执行脚本"（不是文档），值班同学一键执行。预案库按"故障类型"分类（GPU OOM/延迟飙升/依赖故障/模型 bug），每个预案有"触发条件 + 执行动作 + 验证指标 + 回滚方法"。第二，**故障复盘标准化**——每次故障用标准模板复盘：时间线（故障发生 → 发现 → 恢复）、根因（5 why 分析）、影响（用户/业务）、改进措施（含负责人和 deadline）。复盘重点是"改进措施"（防止再犯），不是"追责"。第三，**改进措施跟踪**——复盘的改进措施（如"升级前跑影子流量"）录入任务系统（Jira），设 deadline 和 owner，定期 review 进度。如果同类故障再发生，检查"上次的改进措施做了吗"，没做要追责（这次是追责流程执行，不是追责人）。预案和复盘是"知识沉淀"——把"个人的经验"变成"团队的资产"，让系统越用越稳定。

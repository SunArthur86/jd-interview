# 【拼多多 AI 中台】预案和回滚怎么做？钱学森工程理论怎么应用？

> JD 依据："稳定性治理、模型服务"。

## 一、为什么需要预案

**反面案例**：
```
某 LLM 服务挂 → 值班同学慌 → 临时找运维 → 找研发 → 改配置 → 重启 → 30 分钟才恢复
→ 损失百万

教训：故障期才想解决方案必然慢，必须预案化。
```

**预案价值**：
- 故障时秒级响应（一键执行）
- 减少决策成本（不用现场想）
- 可演练验证（平时证明有效）

## 二、预案体系

### 1. 按场景分类

| 场景 | 预案 |
|------|------|
| 流量突增 | 限流 + 扩容 + 削峰 |
| 服务过载 | 限流 + 降级（小模型/兜底） |
| 依赖故障 | 熔断 + 降级 |
| 单实例挂 | 自动摘除 + 流量切换 |
| 集群挂 | 切灾备机房（多活） |
| 数据异常 | 数据回滚 + 缓存清理 |
| 模型效果差 | 回滚到旧版本 |
| 大促保障 | 提前扩容 + 关非核心 |

### 2. 预案模板
```yaml
name: LLM 推理过载
trigger:
  metric: gpu_utilization
  threshold: "> 95%"
  duration: 60s
actions:
  - type: rate_limit
    target: llm-gateway
    config: { qps: 5000 }    # 全局限流到 5000
  - type: degrade
    target: llm-inference
    config: { model: small-7b }  # 切小模型
  - type: scale
    target: llm-vllm
    config: { replicas: 50 }     # 扩容到 50
notifications:
  - 钉钉群
  - 值班手机
rollback:
  auto_when: gpu_utilization < 70% for 5min
  actions: [恢复限流, 切回大模型]
owner: AI 中台
```

### 3. 自动化触发
```java
@Component
public class AutoMitigation {

    @Scheduled(fixedDelay = 10000)
    public void check() {
        double gpuUtil = metrics.get("gpu_utilization");
        double errorRate = metrics.get("error_rate");

        if (gpuUtil > 0.95) {
            executor.execute(Playbook.LLM_OVERLOAD);    // 自动执行预案
        }
        if (errorRate > 0.1) {
            executor.execute(Playbook.ROLLBACK_LATEST);  // 自动回滚
        }
    }
}
```

## 三、回滚机制

### 1. 版本回滚（模型/代码）
```
传统：发版前备份 → 出问题恢复（分钟级）

蓝绿发布：
  - 蓝环境跑 v1，绿环境跑 v2
  - 切流量到绿，验证 OK
  - 出问题秒级切回蓝

金丝雀发布：
  - 新版本先 1% → 5% → 20% → 100%
  - 每阶段监控，异常自动回滚

K8s 原生：
  kubectl rollout undo deployment/llm-vllm
  → 秒级回滚到上一个版本
```

### 2. 配置回滚
```
所有配置走配置中心（Nacos/Apollo）：
  - 每次变更存版本
  - 出问题秒级切回旧版本
  - 灰度发布（5% → 100%）
```

### 3. 流量回滚
```
多机房部署：
  - 切流量到备用机房（DNS/SLB）
  - 单机房故障秒级切换

灰度切流量：
  - 新版本异常 → 流量切回旧版本
```

### 4. 数据回滚（最复杂）
```
特征/数据变更要回滚：
  - DDL 变更前备份
  - 软删除（不真删）
  - 双写迁移（新旧并行验证）
  - 限速变更（避免大规模错误）
```

## 四、钱学森工程理论应用

钱学森倡导系统工程方法（出自《工程控制论》），三原则：

### 1. 预先研究（Pre-research）
```
- 关键技术提前研究（不临阵磨枪）
- AI 中台案例：
  - 大促前 3 个月预测流量，预先扩容
  - 新模型上线前沙箱回放历史流量
  - 新硬件（H100/FP8）提前适配
```

### 2. 仿真验证（Simulation）
```
- 不直接上线试，先仿真
- AI 中台案例：
  - 全链路压测（影子流量模拟）
  - 故障注入（Chaos Engineering 混沌工程）
  - A/B 实验沙箱回放
  - 灾难演练（拔网线/关机）
```

### 3. 反复迭代（Iterative）
```
- 不一次定型，小步快跑
- AI 中台案例：
  - 模型灰度发布（1% → 5% → 20% → 100%）
  - 预案定期演练（每季度）
  - 持续监控 + 反馈 + 优化
```

## 五、混沌工程（Chaos Engineering）

预案的反向验证：主动注入故障看预案是否生效。

```
Netflix Chaos Monkey 思路：
- 随机 kill Pod（看自愈能力）
- 注入网络延迟（看超时/重试）
- 模拟 GPU 故障（看切流）
- 拔机房（看多活）

拼多多：
- 每月一次"故障日"演练
- 注入各种故障 → 验证预案自动触发
- 找出预案漏洞（不生效/动作错）
- 改进预案
```

## 六、典型预案案例

### 案例 1：LLM 模型效果突降
```
触发：A/B 实验组 CTR 跌 20%
预案：
1. 自动切流量回旧版本（100% → 0%）
2. 通知值班
3. 排查新版本（prompt 变更/数据漂移）
4. 修复后重新灰度
```

### 案例 2：双 11 流量突增 10 倍
```
触发：QPS 超 50000（预测流量）
预案：
1. 自动扩容到预案规模（提前预热）
2. 限流阈值动态调整
3. 关闭非核心功能（个性化推荐）
4. LLM 切小模型（响应快）
```

### 案例 3：GPU 集群挂
```
触发：单 AZ GPU 故障率 50%
预案：
1. 流量切到备用 AZ（多活）
2. 自动扩容其他 AZ
3. 降级（限流 + 兜底）
```

## 七、演练体系

```
桌面演练：值班同学按预案走流程（季度）
实战演练：在隔离环境注入故障（半年）
全链路演练：双 11 前全链路压测 + 故障注入
无预警演练：随机时间真实注入（高级）

目标：
- 验证预案有效性
- 训练值班同学
- 找系统隐藏问题
- 持续改进
```

## 八、拼多多稳定性治理

```
组织：
- 稳定性负责人（每业务线）
- 值班轮换 7×24
- 故障复盘文化（无指责 postmortem）

机制：
- SLO 定义（99.95% 可用性）
- 错误预算（超支停发版）
- 变更审批（高风险变更需评估）
- 自动化预案（监控触发）

工具：
- 监控告警（Prometheus + Grafana）
- 自动预案平台
- 全链路压测
- 混沌工程平台
```

## 九、底层本质

预案回滚本质是**"故障前预演 + 故障时自动 + 故障后迭代"**——按场景准备预案（限流/降级/扩容/切流），自动化触发（监控驱动），定期演练验证。钱学森系统工程的"预先研究 + 仿真验证 + 反复迭代"是其理论基础，AI 中台这种复杂系统尤其需要。混沌工程是反向验证预案有效性的关键。

## 常见考点

1. **预案自动化到什么程度**？——重复性高/低风险的全自动（限流/扩容），高风险的半自动（切流量/回滚，需确认）。
2. **回滚和重放区别**？——回滚是切回旧版本/状态，重放是重新执行历史操作（验证）。
3. **故障复盘重点是什么**？——找根因（不只是表象）+ 改进措施（防再次发生）+ 不指责人（指责流程/系统）。

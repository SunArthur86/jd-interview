# 【拼多多 AI 中台】LLM 量化（INT8/INT4/AWQ/GPTQ）怎么选？

> JD 依据："量化、推理加速、混合精度"。

## 一、为什么量化

**显存压力**：
```
LLaMA-70B FP16 = 140GB → 单 8×H100（640GB）才能放
LLaMA-70B INT4 = 35GB  → 单 H100（80GB）轻松放
```

**带宽压力**（Decode 阶段更明显）：
```
FP16：每 token 读 140GB 权重 → 70ms（2TB/s HBM）
INT4：每 token 读 35GB   → 17ms（4x 加速）
```

## 二、量化分类

### 1. 按时机
- **PTQ**（Post-Training Quantization，训练后量化）：训完直接量化，快速通用
- **QAT**（Quantization-Aware Training，量化感知训练）：训练时模拟量化，精度更好但成本高

### 2. 按比特
- INT8：通用、几乎无损
- INT4：激进、需保护（AWQ/GPTQ）
- FP8：H100 支持，介于 INT8/FP16 之间

### 3. 按粒度
- per-tensor：整个张量一组 scale（粗）
- per-channel：每通道一组（中）
- per-group：每 N 个连续元素一组（细，AWQ/GPTQ 用）

### 4. 对称/非对称
- 对称：[-max, max] 映射到 [-127, 127]（INT8）
- 非对称：[min, max] 映射到 [0, 255]，加 zero-point

## 三、基础 PTQ（RTN，Round-To-Nearest）

最简单：直接四舍五入。

```python
scale = max(|W|) / 127
W_int8 = round(W / scale)
# 反量化
W_recovered = W_int8 * scale
```

**问题**：粗暴，INT4 掉点明显（异常大值拉高 scale，小值精度差）。

## 四、GPTQ：基于二阶信息的逐层量化

**核心思想**：量化一个权重时，用 Hessian 矩阵（二阶信息）调整其他权重补偿误差。

```
对每层权重 W（OUT × IN）：
1. 计算 Hessian H = X·X^T（X 是校准数据激活）
2. 逐列量化 W[:, j]：
   - 量化 W[:, j] → ŵ
   - 误差 e = W[:, j] - ŵ
   - 调整未量化列：W[:, rest] -= e · H[j, rest] / H[j, j]
3. 累积误差被后续列补偿
```

**优点**：精度好，INT4 几乎无损。
**缺点**：量化过程慢（分钟到小时级），需校准数据集（128-1024 样本）。

## 五、AWQ：激活感知保护重要权重

**核心观察**：不是所有权重都同样重要，少数"显著"权重（对应大幅激活）量化后掉点多。

**思路**：
```
1. 用校准数据找"重要"权重（激活幅度大）
2. 给重要权重乘缩放因子 s（>1）放大
3. 整体量化（重要权重相对误差小）
4. 推理时反缩放
```

```python
# AWQ 简化
salient = find_salient_weights(W, calibration_data)
s = compute_scale(salient)              # 重要权重缩放
W_scaled = W * s
W_int = quantize(W_scaled)
# 推理：output = dequantize(W_int) * x / s
```

**优点**：比 GPTQ 快（秒级）、精度相当、推理更快。
**缺点**：仍需少量校准数据。

## 六、对比表

| 方法 | 比特 | 精度 | 量化速度 | 推理速度 | 适用 |
|------|------|------|----------|----------|------|
| FP16 | 16 | 满分 | - | 基准 | 高端卡 |
| RTN | 8/4 | INT8 好/INT4 差 | 快 | 快 | 简单 |
| GPTQ | 8/4 | 优 | 慢（分钟级） | 快 | 高精度需求 |
| AWQ | 8/4 | 优 | 中（秒级） | 最快 | 生产首选 |
| SmoothQuant | 8 | 优 | 中 | 快 | 激活+权重量化 |
| FP8（H100） | 8 | 接近 FP16 | 快 | 极快 | 新硬件 |

### Java 业务侧
业务侧不直接量化，但要：
- 监控量化后效果（业务指标对比）
- 评估成本（INT4 减半显存 → 单卡跑更大模型）
- 兜底方案（量化出问题切 FP16）

## 八、量化为什么能加速

```
Decode 阶段每 token 都要读全部权重
权重越大，访存时间越长（带宽瓶颈）
量化后权重大小减半（INT8）或 1/4（INT4）
访存时间相应减少 → Decode 提速 2-4 倍

注意：Prefill 阶段算力瓶颈，量化收益小（甚至矩阵乘 INT8 反量化有开销）
```

## 九、量化陷阱

1. **INT4 掉点**：数学/代码任务可能掉 3-5%（重要任务用 INT8）
2. **异常值**：少数极端权重拉高 scale，导致整体精度差（用 per-group 或 AWQ）
3. **KV Cache 量化**：权重之外还要考虑 KV 量化
4. **硬件支持**：INT4 需要 GPU 支持（A100/H100 支持，老卡可能不支持 INT4 kernel）
5. **校准数据**：要和业务分布一致（中文场景用中文校准集）

## 十、底层本质

量化本质是**"用更少比特表示权重，关键权重重点保护"**——直接舍入（RTN）粗暴掉点多，GPTQ/AWQ 通过校准数据 + 二阶信息/激活感知智能量化，INT4 几乎无损。对 Decode 阶段（带宽瓶颈）提速明显，是 LLM 推理降本最直接有效的手段。

## 常见考点

1. **INT8 和 INT4 怎么选**？——通用首选 INT8（几乎无损），显存极紧/极致成本选 INT4（数学/代码任务谨慎）。
2. **量化为什么对 Prefill 加速小**？——Prefill 是算力瓶颈，量化甚至增加反量化开销；Decode 是带宽瓶颈，量化直接减带宽。
3. **KV Cache 怎么量化**？——INT8/FP8，配合 sliding window 控制长序列；注意长序列误差累积。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：量化能把 70B 模型从 140GB（FP16）压到 35GB（INT4），4 张 H100 就能跑。但为什么不直接用更小的模型（如 7B FP16 = 14GB）？7B 推理更快、成本更低，为什么要费力量化大模型？**

"量化大模型"和"用小模型"的效果完全不同。第一，**能力上限不同**——70B INT4 保留的是 70B 的"知识和推理能力"，只是精度略降（perplexity 涨 < 1%）；7B 是"小模型"，能力本就不如 70B（复杂推理、代码生成差距大）。类比：70B INT4 是"聪明的博士偶尔犯小错"，7B 是"高中生很精确但懂的不多"。第二，**业务场景**——客服问答、简单摘要用 7B 够（规则性强）；代码生成、数学推理、多轮复杂对话必须 70B（7B 会胡说）。第三，**量化是"压缩已有能力"，不是"降低能力"**——70B INT4 在 GSM8K（数学）准确率 85%，7B FP16 只有 50%，差距是"模型能力"不是"精度"。所以"能用小模型就用小模型（省成本）"，但"业务需要大模型能力时，量化是降成本的唯一手段"。生产实践：简单任务用 7B FP16（快+便宜），复杂任务用 70B INT4（强+省显存）。

### 第二层：证据与定位

**Q：70B 模型用 AWQ INT4 量化部署后，客服场景的用户满意度从 88% 降到 82%。你怎么定位是"量化导致的退化"还是"Prompt 调整/流量变化"导致的？**

控制变量对比。第一，**A/B 实验隔离量化变量**——FP16 和 INT4 各部署一套（同模型、同 Prompt、同数据），网关 50/50 分流，同时段对比 `user_satisfaction`（点赞率）和 `task_completion_rate`（任务完成率）。如果 INT4 的满意度持续低 6%，是量化退化。第二，**perplexity 基线对比**——用同一套测试集（1000 条客服问答），跑 FP16 和 INT4 的 perplexity 和准确率，FP16 PPL=3.2、准确率 88%，INT4 PPL=3.5（涨 9%）、准确率 82%。PPL 涨幅和满意度降幅一致，证明是量化。第三，**bad case 分析**——抽样 INT4 的差评 case，看是"事实错误"（量化导致的权重偏差）还是"格式错误"（LLM 输出格式不规范）。如果 bad case 多是"事实错误"（退款金额算错、政策引用错），是量化对"记忆型知识"的影响大（INT4 对低频权重敏感）；如果是"格式错误"，和量化无关（Prompt 问题）。

### 第三层：根因深挖

**Q：你发现 INT4 量化对"事实型问答"（退款金额、时间）退化大，但对"对话型"（闲聊）影响小。为什么量化对不同任务的影响不一样？**

根因是"不同任务依赖的权重分布不同"。第一，**事实型任务依赖"特定权重"**——模型回答"退款 7 天到账"时，激活的是"退款/时间"相关的少数神经元，这些神经元的权重如果被 INT4 量化"抹平"（权重差异在量化后丢失），模型会混淆"7 天"和"3 天"。INT4 只有 16 个离散值，对"接近的小数值"区分度差。第二，**对话型任务依赖"广泛权重"**——闲聊时激活大量神经元（分散表达），单个权重的量化误差被"平均掉"，整体输出影响小。第三，**AWQ 的保护机制**——AWQ 在量化时识别"重要权重"（激活值大的），用混合精度保护（重要权重 FP16、其他 INT4），但"重要性"是按校准集算的，如果校准集没覆盖"事实型问答"，AWQ 保护的不准。解法：第一，**校准集要对齐业务**——AWQ 量化时用客服问答数据做校准集（不是通用数据），让 AWQ 保护"事实相关权重"。第二，**关键层保精度**——embedding 层、output projection 层用 FP16（这两层对"知识记忆"最敏感），中间层 INT4。第三，**INT4 不够就 INT8**——INT8 有 256 个离散值，对事实型任务退化 < 0.5%，成本只多 2 倍（vs INT4）。

**Q：那为什么不全部用 QAT（量化感知训练）？QAT 在训练时就模拟量化，效果比 PTQ（训练后量化）好，不是根本解决吗？**

QAT 效果好但成本高，不适合大多数场景。第一，**QAT 要重新训练**——要在训练时插入"伪量化"节点（forward 时模拟 INT4 的精度损失），让模型"学会"在低精度下工作。70B 模型的 QAT 要重新训几千 step（几天到几周），消耗大量 GPU（千卡级），成本百万级。第二，**QAT 需要"原始训练数据"**——QAT 要用预训练数据或 SFT 数据，但开源模型（Llama/Qwen）的预训练数据不公开，只能用 SFT 数据做 QAT（效果受限）。第三，**PTQ 的性价比**——AWQ/GPTQ 这种 PTQ 方法只需几百条校准数据、1 张 GPU、几小时，就能把 70B 量化到 INT4，精度损失 < 1%。对大多数业务（客服/推荐/摘要），PTQ 的精度够用。QAT 只在"PTQ 精度不够且业务敏感"时才用（如金融分析、医疗问答），且要有原始模型训练能力（基础模型团队）。拼多多用 AWQ/GPTQ 做 PTQ（覆盖 90% 场景），只在关键业务的定制模型上用 QAT。

### 第四层：方案权衡

**Q：INT8 和 INT4 怎么选？INT8 几乎无损但省 2 倍，INT4 省 4 倍但有退化。客服场景用哪个？**

按"显存约束 + 精度需求"选。第一，**显存约束**——70B FP16=140GB（8×H100），INT8=70GB（4×H100），INT4=35GB（2×H100）。如果只有 2 张 H100，必须 INT4；如果有 4 张，INT8 更稳。第二，**精度需求**——INT8 的 perplexity 涨 < 0.5%（几乎无感），INT4 涨 1-3%（事实型任务有退化）。客服场景的判断：如果客服问答是"开放对话"（用户问"怎么退货"，答案不唯一），INT4 够（泛化强）；如果是"精确政策查询"（"退款金额=订单金额×0.8"），INT8 更稳（避免算错）。第三，**生产选择**——**默认 INT8**（通用首选，精度安全），**显存紧张时 INT4**（配合 AWQ 保护重要权重 + 业务校准集），**关键业务 FP16**（支付/法务相关）。拼多多客服用 INT8（7B 模型）或 INT4（72B 模型），根据显存和精度需求决定。不是"越低比特越好"，是"够用就好"。

**Q：为什么不直接用 FP8？FP8 是 H100 原生支持的（硬件加速），比 INT8 更快，精度也够。**

FP8 确实是 H100 的新特性，但有生态和兼容性问题。第一，**硬件限制**——FP8 只有 H100/H200/B200 支持（Hopper/Ada 架构），A100/V100 不支持。如果集群是 A100，FP8 用不了。第二，**推理引擎支持**——vLLM 从 v0.5+ 支持 FP8（`--kv-cache-dtype fp8`），但早期版本不成熟（有 bug），TensorRT-LLM 的 FP8 支持更成熟（NVIDIA 亲儿子）。如果用 vLLM，要确认版本支持 FP8 且稳定。第三，**量化工具**——FP8 的量化工具链（如 NVIDIA AMMO、AutoFP8）不如 AWQ/GPTQ 成熟（后者有 2 年生态积累），新手上手难。第四，**精度对比**——FP8（E4M3）精度介于 INT8 和 INT4 之间，比 INT8 稍差（动态范围大但尾数少），但硬件加速后吞吐比 INT8 高 1.5-2 倍。生产选择：**H100 集群优先 FP8**（硬件原生 + 高吞吐），**A100 集群用 INT8/INT4**（AWQ/GPTQ）。FP8 是趋势（H100 普及后主流），但当前（2026 年）A100 还在大量使用，INT8/INT4 仍是主力。

### 第五层：验证与沉淀

**Q：你怎么证明 AWQ INT4 量化后模型质量"几乎无损"？用哪些指标证明？**

三个层次的评估。第一，**perplexity（语言模型困惑度）**——用 WikiText-2 或业务数据集测 PPL，FP16 基线 PPL=3.2，INT4 PPL=3.5（涨 9%）。如果涨幅 < 15%，算"几乎无损"（业界标准）。第二，**下游任务 benchmark**——用 GSM8K（数学）、HumanEval（代码）、MMLU（知识）测准确率，FP16 vs INT4 的差距应 < 2%。如果 GSM8K 从 85% 降到 83%，可接受；如果降到 75%，退化太大。第三，**业务指标 A/B**——线上 FP16 vs INT4 各 50% 流量，对比 `user_satisfaction`（点赞率）、`task_completion_rate`（任务完成率）、`human_escalation_rate`（转人工率），退化 < 2% 才全量。三个层次：PPL（模型层）→ benchmark（任务层）→ 业务指标（价值层），层层验证。不能只看 PPL（PPL 涨 9% 但业务指标可能涨 5% 也可能涨 15%），要业务指标拍板。

**Q：量化部署的经验怎么沉淀，让新模型上线不用每次重新评估？**

三件事。第一，**量化模型库**——维护一个"模型 × 量化方法 × 精度指标"的表（Llama2-7B AWQ INT4 PPL=5.2 准确率 75%；Qwen2-72B AWQ INT4 PPL=3.5 准确率 88%），新模型选量化方法时查同类模型的基准。第二，**量化 SOP**——标准化量化流程：选校准集（对齐业务）→ AWQ/GPTQ 量化 → PPL 测试 → benchmark 测试 → 线上灰度 → A/B 验证，每步有 checklist。第三，**量化效果监控**——线上推理时监控 `quantization_error_proxy`（如输出长度异常率、重复率，量化退化的信号），异常告警。量化不是"一次性部署"，要持续监控业务指标，发现退化及时调整（换 INT8 或换校准集重新量化）。

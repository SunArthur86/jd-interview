# 【拼多多 AI 中台】FDE 解决方案（大模型项目落地）怎么做？

> JD 依据："FDE 解决方案、大模型项目落地"。

## 一、FDE 角色定位

FDE（Full-stack Development Engineer / Field Development Engineer）= 大模型项目落地的全栈解决方案工程师。

```
传统研发：写代码（一个环节）
算法：训模型（一个环节）
FDE：业务理解 + 架构 + 算法应用 + 工程 + 交付 + 运营（端到端）

核心价值：
- 翻译：业务需求 ↔ 技术方案
- 桥接：算法/工程/数据/运维多团队
- 推进：从需求到上线全流程负责
```

## 二、大模型项目落地的坑

```
坑 1：选错场景
  "用 LLM 做订单计算" → 精度不够（LLM 不擅长精确计算）
  应选：客服对话/内容生成/数据分析等

坑 2：效果不可控
  demo 时好，上线幻觉频发 → 没有评估/治理

坑 3：成本爆
  72B 模型全量上线 → GPU 成本月百万 → 没预算

坑 4：业务不接受
  算法觉得好，业务觉得"不如规则" → 没量化业务价值

坑 5：数据/工程跟不上
  模型有，但 RAG 数据没准备好/系统集成难 → 上不了线
```

## 三、FDE 解决方案流程

```
1. 需求理解
   ↓
2. 场景筛选（适合 LLM 吗）
   ↓
3. 方案设计（架构/技术栈/ROI）
   ↓
4. POC（小规模验证）
   ↓
5. 立项（投资/资源/排期）
   ↓
6. 开发（数据/模型/工程/集成）
   ↓
7. 上线（灰度/监控）
   ↓
8. 运营（迭代/优化/度量）
```

## 四、需求理解与场景筛选

### 场景适配判断
| 场景 | 适合 LLM | 不适合 LLM |
|------|----------|-----------|
| 客服对话 | ✅ 泛化/对话 | — |
| 内容生成（文案/详情） | ✅ 创造性 | — |
| 数据分析报告 | ✅ 推理 + 文本 | — |
| 代码辅助 | ✅ 模式识别 | — |
| 知识问答 | ✅ RAG | — |
| 精确计算（订单/支付） | ❌ | ✅ 规则 |
| 强一致事务 | ❌ | ✅ 传统 |
| 高频小操作 | ❌（贵） | ✅ 规则 |

### 筛选标准
```
LLM 适合的场景特征：
- 输入/输出非结构化（自然语言/图）
- 任务有泛化性（不能枚举规则）
- 容忍概率性（80% 准确够用）
- 价值高（值得 LLM 成本）

四问：
1. 必须 LLM 吗？（规则能不能做）
2. 价值多大？（提效/降本/增收）
3. 数据 ready 吗？
4. 成本可接受吗？
```

## 五、方案设计

### 架构示例（客服 LLM）
```
业务方（客服前端）
    ↓
LLM 网关（鉴权/限流/路由）
    ↓
RAG（检索知识库）
    ↓
LLM 推理（Qwen-72B，vLLM）
    ↓
后处理（敏感词/合规/引用）
    ↓
返回（流式）

监控：延迟/幻觉率/满意度/成本
```

### 技术栈选型
```
模型：
  - 通用：Qwen2.5-72B（私有部署）
  - 垂直：基于通用微调
  - 小模型：7B 用于简单任务

推理：vLLM（通用）或 TensorRT-LLM（极致）
RAG：Milvus + bge-large-zh + bge-reranker
网关：Spring Cloud Gateway
LLMOps：自研 + LangSmith
```

### ROI 估算
```
成本：
  - GPU（H100 × N，24h）
  - 数据/标注
  - 研发投入
  - 运维

收益：
  - 提效（替代人工客服 × FTE × 工资）
  - 降本（替代传统 IVR/小模型）
  - 增收（提升转化/留存）
  - 体验（满意度提升间接价值）

ROI = 收益 / 成本 > 1（最好 > 3）
```

## 六、POC（关键）

```
目标：低成本验证可行性 + 量化效果

POC 流程：
1. 选典型 case（100-1000 条）
2. 准备数据（历史对话/知识库）
3. 接入模型（API/小规模部署）
4. 评估（自动 + 人工）
5. 成本估算（token 消耗 → 生产成本）
6. 输出报告（效果 + 成本 + 风险）

POC 成功标准：
- 效果达标（如准确率 > 85%）
- 成本可控（ROI > 3）
- 业务接受（业务方点头）

POC 失败：
- 效果不达标 → 换模型/加 RAG/调 Prompt
- 成本太高 → 量化/小模型
- 业务不接受 → 重新对齐指标
```

## 七、开发与上线

### 开发
```
1. 数据准备
   - RAG 知识库（ETL + 切块 + embedding）
   - 训练数据（SFT/RLHF，按需）
   - 评估集（黄金集）

2. 模型层
   - 选基座
   - Prompt 工程
   - 微调（如需）
   - 量化（INT4/INT8）

3. 工程层
   - 推理服务（vLLM）
   - 网关（限流/路由/AB）
   - RAG 服务（向量库）
   - 集成（业务系统）

4. 监控
   - 性能（延迟/QPS）
   - 效果（幻觉率/满意度）
   - 成本（token/GPU）
```

### 上线
```
1. 沙箱测试（历史流量回放）
2. 内部白名单（员工先用）
3. 灰度（1% → 5% → 20% → 50% → 100%）
4. 每阶段监控异常
5. 监控指标 + 兜底预案
6. 全量后持续运营
```

## 八、运营与度量

### 关键指标
```
效果：
- 任务完成率（如客服问题解决率）
- 准确率（事实性）
- 满意度（点赞/点踩）
- 幻觉率（抽样标注）

性能：
- TTFT（首 token 延迟）
- TPOT（每 token 延迟）
- QPS
- 错误率

成本：
- 单次推理成本
- 每月总成本
- ROI
```

### 持续迭代
```
- 收集 bad case（用户反馈 + 抽样）
- 分析根因（Prompt/RAG/模型）
- 改进（调 Prompt/加 RAG/微调）
- 评估（黄金集回归）
- 灰度上线
- 监控效果
```

## 九、典型项目案例

### 案例 1：客服 LLM
```
需求：客服转人工率 40%，要降
FDE 流程：
1. 场景筛选：客服对话适合 LLM（泛化/对话）✅
2. 方案：RAG（知识库）+ Qwen-72B + 工具调用（订单查询）
3. POC：1000 条历史对话，准确率 88%，ROI 5
4. 开发：RAG + Prompt + 网关 + 集成客服系统
5. 上线：灰度 5% → 100%
6. 度量：转人工率从 40% → 25%，满意度 +15%

ROI：年省人力成本 ¥X 万，模型成本 ¥Y 万，净收益明显
```

### 案例 2：商品详情自动生成
```
需求：商家上货时自动生成详情描述
FDE：
1. 场景：内容生成适合 LLM ✅
2. 方案：商品图 VLM 提属性 + LLM 生成文案
3. POC：100 商品，商家满意度 85%
4. 上线：商家后台集成
5. 度量：商家上货效率 +30%，详情质量提升
```

### 案例 3：数据分析助手
```
需求：业务方自助查数据（不用找数据同学）
FDE：
1. 场景：Text-to-SQL + 分析适合 LLM ✅
2. 方案：自然语言 → SQL（LLM）+ 执行 + LLM 解读
3. 难点：SQL 准确率/数据库权限/数据安全
4. 上线：业务方 BI 工具集成
5. 度量：业务方查询效率 +5x，数据同学释放 30%
```

## 十、跨团队协作

```
FDE 是桥梁：
- 业务方：理解需求 + 价值对齐
- 算法：选模型 + 微调 + 评估
- 工程：架构 + 开发 + 上线
- 数据：数据准备 + RAG
- 运维：部署 + 监控
- 风控/合规：安全审查

FDE 不写所有代码，但要协调所有角色保证交付。
```

## 十一、关键能力

```
1. 业务理解：懂业务痛点/价值/流程
2. 架构设计：懂 LLM/工程/数据架构
3. 算法应用：懂模型能力边界（不一定要训模型）
4. 工程能力：懂开发/部署/监控
5. 交付能力：项目管理/沟通/推进
6. 度量思维：用数据说话（效果/成本）
7. 学习能力：跟上 LLM 快速演进
```

## 十二、底层本质

FDE 本质是**"大模型项目落地的全栈桥梁"**——懂业务+懂技术+懂交付，端到端负责从需求到上线。核心方法论：场景选对（别硬塞 LLM）+ 方案合理（架构/技术栈/ROI）+ POC 验证（小成本试错）+ 工程化（成本/稳定）+ 持续度量（效果/ROI）。是 LLM 从实验室到生产的关键角色。

## 常见考点

1. **怎么判断场景是否适合 LLM**？——四问：必须 LLM 吗？价值多大？数据 ready 吗？成本可接受吗？适合场景：泛化/对话/创造/非结构化。
2. **POC 怎么说服业务**？——量化指标（效果/成本/ROI）+ 业务语言（不说 AUC 说转化率）+ 对照（vs 现状）。
3. **项目最大风险是什么**？——效果不稳定（幻觉）+ 成本爆 + 业务不接受，要在 POC 阶段提前暴露和解决。

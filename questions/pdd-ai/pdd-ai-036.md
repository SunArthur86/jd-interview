# 【拼多多 AI 中台】RAG 在中台怎么落地？

> JD 依据："RAG 工程、消费者服务策略算法中台"。

## 一、RAG 全流程

```
用户问题
   ↓
[Query 处理]（改写/扩展/意图识别）
   ↓
[检索]（向量 + 关键词 混合）
   ↓
[重排]（Cross-Encoder 精排 top-k）
   ↓
[Prompt 构造]（检索结果 + 问题 + 模板）
   ↓
[LLM 生成]（基于检索结果回答）
   ↓
[引用标注]（来源追溯）
   ↓
回答
```

## 二、数据准备：切块（Chunking）

切块质量直接决定 RAG 效果。

### 策略

| 策略 | 描述 | 适用 |
|------|------|------|
| 固定大小 | 每 N token 切一块 | 简单（不推荐） |
| 语义切分 | 按段落/章节/句子 | 文档 |
| 递归切分 | 先按大单位再按小 | 通用 |
| 父子切分 | 检索小块，返回父块（更多上下文） | 高级 |
| 按 markdown 结构 | 按标题层级 | 技术文档 |
| 按代码结构 | 按函数/类 | 代码 |

### 实战（推荐：语义 + 父子）
```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,          # 每块 512 token
    chunk_overlap=64,        # 重叠 64（边界信息保留）
    separators=["\n\n", "\n", "。", "！", "？", "，", " "]
)
chunks = splitter.split_text(doc)
```

**避坑**：
- 切太碎：语义丢失，检索不准
- 切太大：embedding 不准，token 浪费
- 一般 256-1024 token，加 10-20% overlap

## 三、Embedding

### 选型
```
中文：bge-large-zh / m3e-large / Qwen3-Embedding
英文：text-embedding-3（OpenAI）/ bge-large-en
多模态：CLIP / bge-m3（图文）

部署：
- 商业 API（OpenAI/阿里/智谱）
- 私有部署（推荐，成本低可控）
- ONNX/TensorRT 加速
```

### 要点
```
- 向量维度（768/1024/1536），权衡精度和存储
- 归一化（cosine 相似度）
- 批量 embedding（吞吐优化）
- 缓存（同文本不重复 encode）
```

## 四、向量数据库

| 库 | 特点 | 适用 |
|----|------|------|
| **Milvus** | 大规模、分布式、生态全 | 千万-亿级文档 |
| **Qdrant** | Rust 写，性能好 | 中等规模 |
| **Weaviate** | 内置模型 | 快速上线 |
| **PGVector** | PostgreSQL 扩展 | 已有 PG，小规模 |
| **Elasticsearch** | 8.0+ 支持向量 | 已有 ES |
| **Faiss** | 库（不是服务） | 单机、研究 |

### 索引
```
- Flat：暴力搜索，准确但慢
- IVF：聚类加速，近似
- HNSW：图索引，查询快（推荐）
- PQ：乘积量化，省内存
- HNSW + PQ：常用组合
```

## 五、检索策略

### 1. 纯向量检索
```
query embedding → 向量库 KNN → top-k 文档
优势：语义相似
劣势：精确匹配弱（如人名/产品编号）
```

### 2. 关键词检索（BM25）
```
query → 倒排索引 → top-k
优势：精确匹配
劣势：语义弱
```

### 3. 混合检索（推荐）
```
向量检索 top-50 + 关键词检索 top-50 → 融合（RRF）→ top-k
兼顾语义 + 精确
```

### 4. Query 改写（提升召回）
```
原 query："怎么退款"
改写：
- 同义扩展："退款流程 / 申请退款 / 退货退款"
- HyDE：先让 LLM 假设答案，用假设答案检索
- Multi-Query：LLM 生成多个表述

每条都检索 → 合并 → 重排
```

## 六、重排（Rerank）

检索召回的 top-50 可能不准，用 Cross-Encoder 精排。

```
Bi-Encoder（embedding 阶段）：
  query 和 doc 独立编码 → 算相似度
  快但糙（适合召回）

Cross-Encoder（重排阶段）：
  [query, doc] 一起进 transformer → 输出相关度
  准但慢（只用于 top-k 重排）

重排模型：bge-reranker-large / Cohere Rerank / 自训
```

## 七、生成

### Prompt 构造
```
你是拼多多客服，根据以下知识回答用户问题。

知识：
[1] {chunk1}（来源：退款政策文档）
[2] {chunk2}（来源：FAQ）
...

规则：
1. 只用提供的知识回答
2. 引用来源（如 [1]）
3. 知识中没有则说"未找到相关信息"
4. 简洁准确

用户问题：{question}
```

### 引用追溯
```
输出：
"您可以在订单详情页申请退款 [1]，款项 7 个工作日内到账 [2]"

[1] 退款政策文档 / section 3
[2] FAQ / 退款时间
```

提升可信度 + 方便人工审核。

## 八、RAG 工程化

### 1. 数据管道
```
数据源（文档/网页/数据库）→ ETL → 清洗 → 切块 → embedding → 入向量库
定期更新（增量 + 全量）
```

### 2. 版本管理
```
- 向量库版本（数据更新时新版本）
- Prompt 版本
- 模型版本
- A/B 实验（不同版本对比）
```

### 3. 评估
```
离线：
- 召回率（top-k 是否含正确答案）
- 准确率（生成答案是否正确）
- 引用准确率（引用是否对）
- 拒答率（不该答的拒绝）

在线：
- 用户满意度（点赞/点踩）
- 转人工率
- 任务完成率
```

### 4. 监控
```
- 检索质量（召回 top-k 是否相关）
- 生成质量（幻觉率/引用准确）
- 性能（检索延迟/生成延迟）
- 数据新鲜度（向量库更新延迟）
```

## 九、中台 RAG 应用场景

### 1. 智能客服
```
知识库：商品规则/退款政策/FAQ/操作指南
用户："怎么申请退款"
RAG：检索退款流程 → LLM 生成步骤 → 引用政策
```

### 2. 内部知识助手
```
知识库：技术文档/产品文档/会议纪要/历史决策
员工："特征平台怎么接入"
RAG：检索文档 → LLM 回答 + 引用
```

### 3. 数据分析助手
```
知识库：指标定义/业务规则/历史分析
分析师："GMV 为什么跌"
RAG：检索相关分析 → LLM 给出可能原因
```

### 4. 代码助手（Code RAG）
```
知识库：代码库（见 035 题）
```

### 5. 多模态 RAG
```
知识库：图文混合（商品图 + 描述）
检索：CLIP 统一编码，跨模态
生成：VLM 回答
```

## 十、GraphRAG（进阶）

传统 RAG：文本块独立检索，缺关联。

GraphRAG（微软提出）：构建知识图谱 + RAG。

```
1. 抽取实体/关系（LLM）→ 构建知识图谱
2. 社区发现（Leiden）→ 形成主题聚类
3. 检索时：
   - Local：实体 + 邻居
   - Global：社区摘要
4. 生成时结合图结构推理
```

适合复杂多跳问答（"X 公司股东投资了哪些公司？"）。

## 十一、拼多多实战

```
客服 RAG：
- 知识库：千万级文档（商品/政策/FAQ/对话历史）
- 向量库：Milvus（亿级向量，HNSW 索引）
- Embedding：bge-large-zh（私有部署）
- 检索：混合（向量 + BM25）+ Query 改写
- 重排：bge-reranker
- 生成：Qwen2.5-72B
- 引用：每答案带来源

效果：
- 召回 top-5 准确率 92%
- 答案准确率 88%
- 用户满意度提升 25%
- 转人工率降 30%
```

## 十二、底层本质

RAG 本质是**"检索知识 + 增强生成"**——切块/embedding/检索/重排/生成全链路，让 LLM 基于外部权威知识回答，解决幻觉和知识更新。中台落地要工程化（数据管道/版本/评估/监控）。GraphRAG 是处理复杂关联问答的进阶方向。

## 常见考点

1. **RAG 为什么比微调好**？——RAG 知识可动态更新/可追溯/成本低；微调适合固定能力/风格。两者可结合。
2. **怎么提高 RAG 准确率**？——好切块 + 混合检索 + 重排 + Query 改写 + 强 Prompt（强制引用/拒答）。
3. **向量库 HNSW 和 IVF 区别**？——HNSW 图索引查询快但内存大，IVF 聚类索引内存小但查询稍慢，大规模用 HNSW。

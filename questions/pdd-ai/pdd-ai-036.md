# 【拼多多 AI 中台】RAG 在中台怎么落地？

> JD 依据："RAG 工程、消费者服务策略算法中台"。

## 一、RAG 全流程

```
用户问题
   ↓
[Query 处理]（改写/扩展/意图识别）
   ↓
[检索]（向量 + 关键词 混合）
   ↓
[重排]（Cross-Encoder 精排 top-k）
   ↓
[Prompt 构造]（检索结果 + 问题 + 模板）
   ↓
[LLM 生成]（基于检索结果回答）
   ↓
[引用标注]（来源追溯）
   ↓
回答
```

## 二、数据准备：切块（Chunking）

切块质量直接决定 RAG 效果。

### 策略

| 策略 | 描述 | 适用 |
|------|------|------|
| 固定大小 | 每 N token 切一块 | 简单（不推荐） |
| 语义切分 | 按段落/章节/句子 | 文档 |
| 递归切分 | 先按大单位再按小 | 通用 |
| 父子切分 | 检索小块，返回父块（更多上下文） | 高级 |
| 按 markdown 结构 | 按标题层级 | 技术文档 |
| 按代码结构 | 按函数/类 | 代码 |

### 实战（推荐：语义 + 父子）
```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,          # 每块 512 token
    chunk_overlap=64,        # 重叠 64（边界信息保留）
    separators=["\n\n", "\n", "。", "！", "？", "，", " "]
)
chunks = splitter.split_text(doc)
```

**避坑**：
- 切太碎：语义丢失，检索不准
- 切太大：embedding 不准，token 浪费
- 一般 256-1024 token，加 10-20% overlap

## 三、Embedding

### 选型
```
中文：bge-large-zh / m3e-large / Qwen3-Embedding
英文：text-embedding-3（OpenAI）/ bge-large-en
多模态：CLIP / bge-m3（图文）

部署：
- 商业 API（OpenAI/阿里/智谱）
- 私有部署（推荐，成本低可控）
- ONNX/TensorRT 加速
```

### 要点
```
- 向量维度（768/1024/1536），权衡精度和存储
- 归一化（cosine 相似度）
- 批量 embedding（吞吐优化）
- 缓存（同文本不重复 encode）
```

## 四、向量数据库

| 库 | 特点 | 适用 |
|----|------|------|
| **Milvus** | 大规模、分布式、生态全 | 千万-亿级文档 |
| **Qdrant** | Rust 写，性能好 | 中等规模 |
| **Weaviate** | 内置模型 | 快速上线 |
| **PGVector** | PostgreSQL 扩展 | 已有 PG，小规模 |
| **Elasticsearch** | 8.0+ 支持向量 | 已有 ES |
| **Faiss** | 库（不是服务） | 单机、研究 |

### 索引
```
- Flat：暴力搜索，准确但慢
- IVF：聚类加速，近似
- HNSW：图索引，查询快（推荐）
- PQ：乘积量化，省内存
- HNSW + PQ：常用组合
```

## 五、检索策略

### 1. 纯向量检索
```
query embedding → 向量库 KNN → top-k 文档
优势：语义相似
劣势：精确匹配弱（如人名/产品编号）
```

### 2. 关键词检索（BM25）
```
query → 倒排索引 → top-k
优势：精确匹配
劣势：语义弱
```

### 3. 混合检索（推荐）
```
向量检索 top-50 + 关键词检索 top-50 → 融合（RRF）→ top-k
兼顾语义 + 精确
```

### 4. Query 改写（提升召回）
```
原 query："怎么退款"
改写：
- 同义扩展："退款流程 / 申请退款 / 退货退款"
- HyDE：先让 LLM 假设答案，用假设答案检索
- Multi-Query：LLM 生成多个表述

每条都检索 → 合并 → 重排
```

## 六、重排（Rerank）

检索召回的 top-50 可能不准，用 Cross-Encoder 精排。

```
Bi-Encoder（embedding 阶段）：
  query 和 doc 独立编码 → 算相似度
  快但糙（适合召回）

Cross-Encoder（重排阶段）：
  [query, doc] 一起进 transformer → 输出相关度
  准但慢（只用于 top-k 重排）

重排模型：bge-reranker-large / Cohere Rerank / 自训
```

## 七、生成

### Prompt 构造
```
你是拼多多客服，根据以下知识回答用户问题。

知识：
[1] {chunk1}（来源：退款政策文档）
[2] {chunk2}（来源：FAQ）
...

规则：
1. 只用提供的知识回答
2. 引用来源（如 [1]）
3. 知识中没有则说"未找到相关信息"
4. 简洁准确

用户问题：{question}
```

### 引用追溯
```
输出：
"您可以在订单详情页申请退款 [1]，款项 7 个工作日内到账 [2]"

[1] 退款政策文档 / section 3
[2] FAQ / 退款时间
```

提升可信度 + 方便人工审核。

## 十、GraphRAG（进阶）

传统 RAG：文本块独立检索，缺关联。

GraphRAG（微软提出）：构建知识图谱 + RAG。

```
1. 抽取实体/关系（LLM）→ 构建知识图谱
2. 社区发现（Leiden）→ 形成主题聚类
3. 检索时：
   - Local：实体 + 邻居
   - Global：社区摘要
4. 生成时结合图结构推理
```

适合复杂多跳问答（"X 公司股东投资了哪些公司？"）。

## 十一、拼多多实战

```
客服 RAG：
- 知识库：千万级文档（商品/政策/FAQ/对话历史）
- 向量库：Milvus（亿级向量，HNSW 索引）
- Embedding：bge-large-zh（私有部署）
- 检索：混合（向量 + BM25）+ Query 改写
- 重排：bge-reranker
- 生成：Qwen2.5-72B
- 引用：每答案带来源

效果：
- 召回 top-5 准确率 92%
- 答案准确率 88%
- 用户满意度提升 25%
- 转人工率降 30%
```

## 十二、底层本质

RAG 本质是**"检索知识 + 增强生成"**——切块/embedding/检索/重排/生成全链路，让 LLM 基于外部权威知识回答，解决幻觉和知识更新。中台落地要工程化（数据管道/版本/评估/监控）。GraphRAG 是处理复杂关联问答的进阶方向。

## 常见考点

1. **RAG 为什么比微调好**？——RAG 知识可动态更新/可追溯/成本低；微调适合固定能力/风格。两者可结合。
2. **怎么提高 RAG 准确率**？——好切块 + 混合检索 + 重排 + Query 改写 + 强 Prompt（强制引用/拒答）。
3. **向量库 HNSW 和 IVF 区别**？——HNSW 图索引查询快但内存大，IVF 聚类索引内存小但查询稍慢，大规模用 HNSW。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你们客服系统用 RAG。但客服场景的退款规则、FAQ 这些，传统搜索引擎（ES BM25）也能做精确匹配，为什么要叠一层 RAG？BM25 召回 + 模板回复不就够了吗？**

传统 BM25 + 模板的问题在"泛化和理解"。第一，用户表达多样——"我想把钱退回来"和"退款流程"BM25 的文本重叠度低（"退钱"和"退款"字面不同），BM25 召回不到，但语义一样，向量检索能覆盖。第二，模板回复死板——用户问"我昨天买的手机能退吗"，模板只能返回标准流程，但 RAG + LLM 能结合上下文（手机类目、7 天无理由、具体订单）生成个性化回答。第三，FAQ 维护成本——传统模板要人工维护每个问题的标准答案，RAG 只需维护知识文档，LLM 生成回答。RAG 的核心价值是"语义召回 + 自然语言生成"，不是简单替代 BM25，而是在 BM25 召回不准的场景补位。

### 第二层：证据与定位

**Q：客服 RAG 上线后，用户反馈"答非所问"的概率 15%。你怎么定位是检索环节的问题还是生成环节的问题？**

分两段排查。第一，给每个问题打 `retrieval_recall@5` 标签——人工标注 top-5 召回里有没有正确答案的文档。如果召回率才 70%（top-5 里 30% 的问题找不到正确文档），根因在检索环节；如果召回率 95% 但答案准确率才 85%，根因在生成环节（LLM 没用对召回内容）。第二，看召回质量的具体分布——如果某类问题（比如"退款金额计算"）召回率特别低（50%），是知识库缺这类文档或切块把关键信息切碎了。第三，用 Arthas 或 trace 追踪单次请求——看检索返回的 top-5 chunk 内容，肉眼判断"这个 chunk 能不能回答用户问题"，再看 LLM 的 prompt 里有没有包含正确的 chunk。区分"召回不到"和"召回了但没引用"。

### 第三层：根因深挖

**Q：你发现召回率才 70%，排查发现是"长文档被切碎导致信息丢失"。比如退款政策是一整段（"7 天无理由退货，但生鲜类目除外，激活的数码产品除外"），切块后分成 3 个 chunk，用户问"数码产品能退吗"，检索召回了第 1 个 chunk（7 天无理由）但漏了第 3 个（数码除外）。怎么解决？**

根因是"固定 token 切块破坏了语义完整性"。解法有四个层次。第一，**改切块策略**——从固定 token 切（chunk_size=512）改成"父子切块"（检索小块，返回父块）：把退款政策整段作为父块（1024 token），切小了检索（256 token），但检索命中后返回整个父块，保证上下文完整。第二，**语义切分**——用句号/段落/标题切，而不是硬按 token 数切，保证"7 天无理由 + 除外条款"在同一段。第三，**补充元数据**——给每个 chunk 打 `category` 标签（退款/物流/售后），检索时按用户意图过滤到对应类目，减少噪声。第四，**Query 改写**——用户问"数码产品能退吗"，用 LLM 改写成"数码产品 退货政策 7 天无理由 除外条款"，增加召回覆盖。优先级：父子切块（根本解）> 语义切分 > Query 改写。

**Q：那为什么不干脆不切块，整个退款政策文档作为一个 chunk？反正也就几 KB。**

不切块在"大规模 + 多文档"场景有代价。第一，embedding 不准——向量编码是把文本压缩到固定维度，文本越长信息越稀释，整段 1024 token 的 embedding 会"稀释"关键信息（除外条款的权重被淹没），检索时反而召不准。研究显示 256-512 token 的 chunk embedding 精度最高。第二，token 浪费——LLM 的 context window 有限（比如 32K），如果 top-5 召回每个都是 1024 token，prompt 就 5K token，成本高且可能超 context。第三，检索效率——向量库检索 1024 token 的 chunk 和 512 token 的 chunk 速度一样，但 512 能召回更精准的片段。正确做法是"检索用小块（精准）、生成用大块（上下文）"——父子切块就是这个思路。

### 第四层：方案权衡

**Q：你用了混合检索（向量 + BM25 + RRF 融合）。但向量检索召回 top-50，BM25 也 top-50，融合后重排 top-5。这个 pipeline 有 4 步，延迟要 200ms。客服场景要求 < 500ms 总延迟（含 LLM 生成）。检索就占 200ms 太多了，怎么优化？**

分步骤优化。第一，**并行化**——向量检索和 BM25 检索并行执行（用 CompletableFuture），从串行 100ms+100ms 降到并行 max(100,100)=100ms，省 100ms。第二，**减少召回量**——向量 top-50 降到 top-20（HNSW 的 ef_search 从 200 降到 64），BM25 也 top-20，融合后 40 个候选，重排 top-5，延迟降 40%。实测 top-20 和 top-50 的召回率差 < 3%。第三，**重排模型量化**——bge-reranker-large 用 FP16 或 ONNX 部署，40 个候选的重排从 80ms 降到 30ms。第四，**缓存**——高频问题（"怎么退款"这类 top-100 query）的检索结果缓存（Redis，TTL 1h），命中直接返回，省整个检索 pipeline 的 200ms。优化后：并行 100ms + 重排 30ms = 130ms，缓存命中 0ms。留 370ms 给 LLM 生成。

**Q：为什么不直接用 ColBERT 或 SPLADE 这种"同时做向量 + 稀疏"的模型，一步搞定混合检索？不就省了维护两套索引的复杂度吗？**

ColBERT/SPLADE 确实能融合稠密+稀疏，但生产可用性不够成熟。第一，ColBERT 的存储成本高——它是 token-level 的多向量（每 token 一个向量），一篇 512 token 文档存 512 个向量，存储是单向量模型的 512 倍，亿级文档存不起。第二，SPLADE 的查询延迟高——它生成稀疏向量要过一次 BERT，比 BM25 倒排索引慢 5-10 倍，对延迟敏感场景不划算。第三，生态——Milvus/Qdrant 对标准向量和 BM25 的支持成熟（HNSW + 倒排索引），但 ColBERT 要专用索引（如 RAGatouille），社区和运维工具不如 Milvus 完善。工程上"两套成熟系统 + RRF 融合"比"一套前沿系统"更稳。ColBERT 适合小规模研究或对召回质量要求极高（忽略成本）的场景。

### 第五层：验证与沉淀

**Q：你怎么证明父子切块真的提升了召回率，而不是只是流量变化导致的偶然波动？**

三层验证。第一，**离线评估集回归**——维护一个"黄金集"（500 个标注好正确文档的问题），切块策略改之前跑一次（recall@5=70%），改之后跑一次（recall@5=88%），对比同一数据集的指标变化，消除流量变量。第二，**线上 A/B**——新老切块策略各部署一套 RAG，网关 50/50 分流，同时段对比 `recall_proxy`（用户点踩率，点踩说明没答对）和 `first_contact_resolution`（一次解决率），连续 1 周看是否稳定提升。第三，**bad case 溯源**——收集点踩的 case，人工复盘"是检索没召回还是 LLM 没引用"，如果改切块后"检索没召回"的占比从 60% 降到 20%，证明切块优化生效。

**Q：RAG 系统长期运营怎么避免效果退化？**

沉淀三件事。第一，**数据新鲜度监控**——知识库每天更新，但向量库 embedding 是异步的，监控 `vector_db_lag_seconds`（向量库最新文档 vs 知识库最新文档的时间差），超过 1 小时告警，保证用户问"今天的退款政策"能检索到。第二，**bad case 闭环**——每天抽取点踩 case（采样 100 条）人工标注，分类（检索问题/生成问题/知识缺失），每周迭代（补文档/调切块/改 Prompt），形成"反馈 → 分析 → 优化 → 验证"闭环。第三，**回归测试**——每次改切块策略/Prompt/模型，都跑黄金集回归，指标不降才上线。把 RAG 当成"需要持续训练的系统"而不是"上线就不动的基础设施"。

## 结构化回答


**30 秒电梯演讲：** 像考试开卷——LLM 是考生（生成能力），检索系统是课本（找答案依据），开卷考试比闭卷（纯 LLM）准确得多。

2. **切块策略** — 固定/语义/递归/父子
3. **检索：向量 ** — 向量 + 关键词 + 混合

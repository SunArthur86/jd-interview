# 【拼多多 AI 中台】AI Harness 工程化（LLMOps 全链路）怎么做？

> JD 依据："大模型训练框架开发维护、AI Harness 工程化"。

## 一、LLMOps 全景

```
开发 → 评估 → 灰度 → 上线 → 监控 → 迭代
  ↑                                      │
  └──────────────────────────────────────┘

管理对象：
  - Prompt（模板/版本/变量）
  - 模型（基座/微调/版本）
  - 数据（训练/评估/反馈）
  - RAG（向量库/检索策略/重排序）
  - Agent（工具/编排/记忆）
```

## 二、Prompt 管理

### 版本化
```
prompt_id: customer_service_v1
content: |
  你是拼多多客服，回答用户问题：{question}
  规则：礼貌、准确、不超过 100 字
variables: [question]
model: qwen-72b
temperature: 0.7
created_at: 2026-01-01
author: zhangsan
tags: [客服, v1]
```

### Prompt 工程实践
```
1. 模板化（Jinja2/Mustache）：变量化
2. 版本管理（Git 风格）：diff/branch/merge
3. A/B 实验：多版本对比
4. 灰度：新 Prompt 5% → 20% → 100%
5. 回滚：效果掉秒级切回旧版本
6. 评估：自动指标 + 人工评分
```

### 工具
- LangSmith（LangChain）
- PromptHub/Promptfoo
- 自研（Java + 配置中心）

## 三、评估体系

### 1. 离线评估（上线前）

#### 自动指标
```
- BLEU/ROUGE（文本相似度）
- 准确率（事实性问题）
- 完整性（关键信息覆盖）
- 一致性（同问题多次回答一致）
```

#### LLM-as-Judge
```
用 GPT-4/Claude 当裁判评估：
prompt: |
  你是评估员，给以下回答打分（1-5）：
  问题：{question}
  回答：{answer}
  标准：准确性/完整性/礼貌
  返回 JSON: {score, reason}

优势：规模化（不用人工逐条）
劣势：裁判模型也可能错
```

#### 人工评估
```
- 抽样（每版本 100-1000 条）
- 标注员打分（多维：准确/流畅/有用）
- 多人交叉（一致性检验）
- 黄金集（已知答案，回归测试）
```

#### 业务指标
```
- 任务完成率（如客服问题解决率）
- 用户满意度（评分/点踩）
- 转化率（如推荐点击）
```

### 2. 在线评估（上线后）

```
A/B 实验：
  对照组（旧 Prompt/模型）vs 实验组（新）
  指标：满意度/转化率/任务完成率
  显著性检验

监控：
  - 实时指标（满意度/点踩率）
  - 业务指标（GMV/CTR）
  - 安全指标（违规率/敏感词）
```

## 四、监控

### 性能
```
- TTFT（首 token 延迟）
- TPOT（每 token 延迟）
- QPS
- 错误率
- GPU 利用率/显存
```

### 效果
```
- 幻觉率（抽样标注 + 规则校验 + 用户反馈）
- 满意度（点踩率/评分）
- 任务完成率
- 安全合规（违规率）
```

### 成本
```
- 单次推理成本（GPU 时）
- token 单价
- 缓存命中率
- 模型/Prompt 维度成本归因
```

### 漂移
```
- 输入分布漂移（用户问题类型变化）
- 输出分布漂移（模型行为变化）
- 数据漂移（特征/知识变化）
```

## 五、灰度发布

```
模型/Prompt 灰度流程：
1. 沙箱测试（历史数据回放）
2. 内部白名单（员工先用）
3. 小流量灰度（1% → 5%）
4. 放量（20% → 50%）
5. 全量（100%）
6. 监控异常 → 秒级回滚

灰度维度：
- 用户百分比
- 用户分群（新老/地域）
- 业务线
```

## 六、回滚

```
传统软件回滚：代码版本
LLM 回滚：
  - 模型版本（基座/微调）
  - Prompt 版本
  - RAG 配置（向量库版本/检索策略）
  - Agent 工具配置

实现：
  - 所有配置版本化（配置中心）
  - 一键回滚脚本
  - 监控触发自动回滚（错误率/满意度掉）
```

## 七、Harness（全链路编排）

LLM 应用是"模型 + Prompt + RAG + Agent"的组合，Harness 把这些编排起来。

```
用户请求
   ↓
[Router]（意图识别 → 路由到不同场景）
   ↓
[Context Builder]
  - RAG 检索（向量库）
  - 特征查询（中台特征）
  - 历史对话（记忆）
   ↓
[Prompt Builder]
  - 模板填充
  - few-shot 选择
   ↓
[LLM Call]
  - 模型选择（大/小模型）
  - 参数（temperature/max_tokens）
   ↓
[Guardrail]
  - 输入过滤（敏感词/Prompt 注入）
  - 输出过滤（合规/安全）
  - 事实校验（RAG 引用验证）
   ↓
[Post-process]
  - 格式化
  - 工具调用执行（Function Calling）
  - 缓存写入
   ↓
响应（流式）
```

每一环节都版本化、可监控、可灰度、可回滚。

## 九、拼多多 LLMOps 实战

```
平台能力：
- Prompt 管理（版本/A-B/灰度）
- 模型管理（注册/部署/版本）
- 评估（离线自动+人工，在线 A/B）
- 监控（性能/效果/成本/漂移）
- Harness（多模型/Prompt/RAG/Agent 编排）

关键指标：
- 上线前：准确率（黄金集）/满意度（人工）
- 上线后：任务完成率/点踩率/GPU 成本

挑战：
- 评估难（人工贵，LLM-as-Judge 有偏差）
- Prompt 效果不稳定（同 Prompt 不同 case 表现差大）
- 多组件协同（Prompt 改了影响 RAG/Agent）
- 成本控制（72B 推理贵，要量化和缓存）
```

## 十、底层本质

LLMOps/Harness 本质是**"把 LLM 应用当成传统软件一样工程化"**——Prompt/模型/数据版本管理（像代码版本），离线+在线评估（像测试），监控+灰度+回滚（像发布）。Harness 把模型/Prompt/RAG/Agent 编排成完整应用，全链路可观测、可演进。是 LLM 应用从 demo 到产品的关键工程层。

## 常见考点

1. **LLM 评估为什么难**？——输出概率性（同输入不同输出），任务多样（无统一指标），人工贵，LLM-as-Judge 可能错。
2. **Prompt 工程的核心是什么**？——清晰指令（角色/任务/约束）+ few-shot 示例 + 变量化 + 版本管理 + 持续评估。
3. **怎么监控幻觉**？——抽样人工标注 + 规则校验（关键事实）+ RAG 引用验证 + 用户反馈（点踩率）+ 同问题多次一致性。

## 苏格拉底式面试追问

> 这组追问不背答案，模拟面试宫层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：LLMOps 要监控"Prompt 版本"。但 Prompt 不是代码吗？用 Git 管理不就行了，为什么要专门的"Prompt 版本管理"？**

Prompt 的迭代模式和代码不同，Git 不够。第一，**Prompt 迭代快**——代码一周改几次，Prompt 一天可能改几十次（调措辞/加示例/改约束）。Git 的 commit 粒度太粗（每次 commit 要写 message），Prompt 调试要"快速试错 + A/B 对比"，Git 流程太重。第二，**Prompt 的效果是"概率性"的**——代码改了要么对要么错（确定），Prompt 改了效果"可能好可能坏"（取决于 LLM 的随机性）。需要"版本对比"（v1 vs v2 在同一测试集上的效果），Git 不原生支持"效果对比"。第三，**Prompt 和"模型版本 + 温度 + 其他参数"耦合**——同一个 Prompt 在 Qwen-72B（temperature=0.7）和 Qwen-7B（temperature=0.3）上效果不同，版本管理要"记录完整的推理配置"，不只是 Prompt 文本。第四，**生产需求**——Prompt 管理平台要支持：（1）版本管理（存历史版本 + diff）；（2）A/B 测试（同时跑多个版本对比）；（3）灰度发布（5% 流量用新版）；（4）效果监控（每个版本的准确率/满意度）。LangSmith/LangFuse 等专门做 LLMOps 的平台，Prompt 管理是核心功能，Git 是基础（存文本）但不充分。

### 第二层：证据与定位

**Q：客服 LLM 上线后，你发现"同一问题问 10 次，答案不一致"（如"退款几天到账"，有的说 7 天有的说 3 天）。怎么定位是 Prompt 问题、RAG 问题还是模型问题？**

分层定位。第一，**看 RAG 召回的一致性**——同一问题问 10 次，每次召回的知识库文档是否一致？如果 10 次召回的文档不同（有时召回"7 天政策"，有时召回"3 天政策"），是 RAG 检索不稳定（向量相似度接近，随机性导致召回不同）。解法：提高检索确定性（降 temperature for embedding、调 top-k 或 RRF 权重）。第二，**看 LLM 生成的确定性**——如果 10 次召回相同的文档（都是"7 天政策"），但 LLM 生成不同（有的说 7 天有的说 3 天），是 LLM 的 temperature 太高（随机性大）或 Prompt 没约束清楚。解法：temperature 降到 0（贪心解码，输出确定）、Prompt 明确约束"只基于提供的知识回答，不要编造"。第三，**看知识库本身**——如果知识库里"7 天政策"和"3 天政策"都存在（不同类目不同政策），是知识库矛盾。解法：知识库治理（标注类目，"服装 7 天、生鲜 3 天"），RAG 召回时带类目信息。第四，**验证**——用 TraceId 关联"某次错误回答的 RAG 召回 + LLM 输入 + LLM 输出"，看是哪个环节引入了错误。拼多多客服：退款问题不一致的根因 60% 是知识库矛盾（不同类目政策混在一起），30% 是 RAG 召回不稳，10% 是 LLM 幻觉。

### 第三层：根因深挖

**Q：你定位到是"知识库矛盾"——退款政策文档里"7 天"和"3 天"并存（不同类目）。但知识库是业务方维护的，他们觉得"没矛盾啊，服装 7 天生鲜 3 天，写的很清楚"。为什么 RAG 还是召错？**

根因是"文档结构没表达类目维度"。第一，**文档结构问题**——业务方写的文档是"7 天无理由退货政策"（一段）+ "生鲜特殊政策"（另一段），两段分开。用户问"生鲜退款几天到账"，RAG 可能召回第一段（"7 天"，因为"退款"匹配）而不是第二段（"生鲜 3 天"，因为"生鲜"在第二段但相似度低）。第二，**解法：元数据过滤**——给每个文档打 metadata（`category: 服装/生鲜/3C`），RAG 检索时先识别用户意图的类目（"生鲜退款"→ category=生鲜），再在该类目下检索，避免召回其他类目的政策。第三，**文档结构化**——把"退款政策"重写成结构化形式（`IF 类目=生鲜 THEN 3天 ELIF 类目=服装 THEN 7天`），RAG 召回整个结构化文档（包含所有类目），LLM 根据用户意图（"生鲜"）解析正确答案。第四，**Prompt 约束**——在 Prompt 里明确"退款天数按类目区分，必须先判断用户问的类目再回答"，LLM 会先提取类目再查政策。第五，**生产选择**——元数据过滤（最有效，从源头避免召回错）+ Prompt 约束（兜底，LLM 二次判断）。拼多多：退款类问题用"类目元数据过滤"，召回准确率从 70% 提升到 92%。

**Q：那为什么不直接用微调让 LLM"记住"退款政策？微调后 LLM 直接知道"生鲜 3 天、服装 7 天"，不用 RAG。**

微调"记忆知识"有三个问题。第一，**知识更新**——退款政策会变（双 11 期间延长到 15 天），微调一次要几天 + 成本，更新不及时。RAG 只需更新知识库文档（分钟级），LLM 立即用新政策。第二，**可追溯**——客服场景要求"回答能引用依据"（监管/审计），微调把政策融进权重（黑盒，无法引用"依据是退款政策第 3 条"）。RAG 能标注"依据：生鲜退款政策"。第三，**幻觉风险**——微调后 LLM 可能"编造"政策（权重里的知识和训练数据混，边界模糊）。比如"3C 退款几天"如果训练数据里没有明确，LLM 可能编"7 天"（实际是 15 天）。RAG 强制"基于检索结果回答"，可控。第四，**微调适合"能力/风格"而非"知识"**——让 LLM 学"拼多多客服的语气"（风格）用微调（稳定），让 LLM 知道"退款政策"（知识）用 RAG（可更新）。生产实践：微调风格（让回答更符合客服调性）+ RAG 补知识（退款政策），两者结合。纯微调记知识是"用错误工具解决问题"，微调不是万能。

### 第四层：方案权衡

**Q：LLM 评估用"LLM-as-Judge"（用一个强 LLM 评估另一个 LLM 的输出）。但 LLM-as-Judge 本身会出错（评估 LLM 判断错），你怎么保证评估的可靠性？**

LLM-as-Judge 的可靠性要"多重校验 + 人工校准"。第一，**LLM-as-Judge 的局限**——评估 LLM（如 GPT-4）判断"客服回答好不好"，可能：（1）位置偏见（倾向于选"第一个"答案）；（2）长度偏见（倾向于选更长的答案）；（3）自身能力限制（评估 LLM 不懂某专业领域，判断错）。第二，**缓解偏见**——（1）随机化答案顺序（A/B 随机，消除位置偏见）；（2）用"绝对评分"（1-5 分）而非"相对比较"（A vs B，消除相对偏见）；（3）用多个评估 LLM（GPT-4 + Claude + Qwen 投票，降低单模型偏差）。第三，**人工校准**——抽样 100 条，让"LLM-as-Judge"和"人工标注"对比，算一致率。如果一致率 85%+，LLM-as-Judge 可用；如果 60%，不可靠（要换更强的评估 LLM 或加人工）。第四，**混合评估**——LLM-as-Judge 做"大规模初筛"（10000 条，快），人工做"精标注"（500 条，准）。用人工标注校准 LLM-as-Judge 的偏差，持续校准。拼多多客服评估：LLM-as-Judge（Qwen-72B）初筛 + 人工抽检校准，一致率 88%，可用于趋势监控（不用于精确决策）。

**Q：那为什么不直接用"人工评估"全部标注？人工最准，不用纠结 LLM-as-Judge 的可靠性。**

人工评估的成本和速度不可承受。第一，**成本**——日均 100 万次客服对话，全人工标注要 1000 个标注员（每人 1000 条/天），年薪 10 万 × 1000 = 1 亿/年。LLM-as-Judge 的成本是 100 万 × 0.01 元 = 1 万元/天（年 365 万），比人工便宜 30 倍。第二，**速度**——人工标注有延迟（标注员要时间），实时监控不可能（当天的问题下周才知道）。LLM-as-Judge 实时（每条对话 2 秒评估），可以即时告警"幻觉率飙升"。第三，**一致性**——人工标注有主观性（不同标注员对"好"的标准不同），一致性 70-80%。LLM-as-Judge 的一致性 100%（同模型同输入同输出），适合"趋势监控"（虽然绝对值可能有偏，但趋势可信）。第四，**生产选择**——人工评估做"黄金集"（500 条精标注，用于模型上线决策），LLM-as-Judge 做"在线监控"（全量实时，用于发现问题）。两者配合：LLM-as-Judge 实时告警 + 人工确认 + 黄金集回归。不是"二选一"，是"各取所长"。

### 第五层：验证与沉淀

**Q：你怎么证明"Prompt 优化 + RAG 治理"把客服 LLM 的准确率从 75% 提升到了 88%？**

三层评估。第一，**黄金集回归**——500 个标注好的"问题 → 正确答案"，优化前跑一次（准确率 75%），优化后跑一次（准确率 88%），对比同一数据集的指标变化，消除流量变量。黄金集要覆盖各类问题（退款/物流/售后）和各类表述（口语/书面/错别字），保证评估全面。第二，**线上 A/B**——新旧 Prompt/RAG 各 50% 流量，对比 `user_satisfaction`（点踩率）和 `first_contact_resolution`（一次解决率）。优化前点踩率 15%，优化后 8%。第三，**人工抽检**——每天抽样 100 条线上对话，人工标注准确率。优化前日均 75%，优化后日均 88%，趋势稳定。三个评估（黄金集 + A/B + 抽检）一致，证明优化生效。注意：黄金集是"离线静态"（可能过时），A/B 是"在线动态"（反映真实流量），两者都要，互补。

**Q：LLMOps 的经验怎么沉淀，让新 Prompt/新模型上线不踩坑？**

三件事。第一，**Prompt 评估流水线**——每次改 Prompt，自动跑黄金集（准确率/延迟/成本），指标不降才上线。流水线集成到 CI/CD，Prompt 改动触发评估，自动化 gate。第二，**监控看板**——实时监控 `hallucination_rate`（幻觉率，通过 LLM-as-Judge + 人工抽检）、`user_satisfaction`（点踩率）、`cost_per_request`（token 成本）、`latency_p99`。异常（幻觉率突增/点踩率飙升）告警。第三，**知识库治理**——定期（每周）review 知识库（补充缺失文档、修正矛盾、更新过期内容），把"bad case → 知识补充"做成闭环。LLMOps 的核心是"持续评估 + 持续优化"，LLM 的效果会随时间/数据变化漂移（用户问的问题变了、知识库过时了），必须持续监控和迭代，不是"上线就不动"。把 LLMOps 当成"像 DevOps 一样的工程实践"，标准化、自动化、持续化。

## 结构化回答


**30 秒电梯演讲：** 像电视剧制作——剧本（Prompt）+ 演员（模型）+ 排练（评估）+ 试播（灰度）+ 正式播出（上线）+ 收视率监控（监控），每环节都管控。

**展开框架：**
1. **Prompt/模** — Prompt/模型/数据版本管理
2. **评估** — 离线（自动+人工）+ 在线（A/B）
3. **监控** — 延迟/幻觉/满意度/成本

**收尾：** Prompt 怎么版本管理？

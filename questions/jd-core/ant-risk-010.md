# 【蚂蚁风控】亿级数据的 MySQL 怎么扩展？分库分表方案？主从延迟怎么解决？

> JD 依据："海量数据应用系统的性能和稳定性"。风控事件表日增上亿，单库扛不住。

## 一、表面层：什么时候要分库分表

**单库的瓶颈信号**：
- 单表数据 > 1000 万（B+ 树从 3 层变 4 层，IO 翻倍）
- 单库 QPS > 5000（连接数、buffer pool 不够）
- 单库磁盘 > 1TB（备份慢、DDL 卡）
- 写放大导致 binlog 同步慢

**经验阈值**：
| 数据量 | 方案 |
|--------|------|
| < 500 万 | 单库单表 + 索引优化 |
| 500 万 - 1000 万 | 读写分离（主从） |
| 1000 万 - 5000 万 | 分区表（PARTITION） |
| > 5000 万 | **分库分表** |

## 二、垂直分 vs 水平分

**垂直分（按结构）**：
- **垂直分库**：按业务拆库（user_db、trade_db、risk_db）
- **垂直分表**：按字段冷热拆表（user_basic、user_detail）

**水平分（按数据）**：
- **按范围**：1-1000万一张、1000万-2000万一张（热点问题）
- **按哈希**：`hash(uid) % N`（均匀但扩容麻烦）
- **按时间**：按月分表（适合日志类，event_202607、event_202608）
- **按一致性哈希**：扩容只迁移部分数据

**风控的选择**：事件表按时间分（每月一张，方便归档），用户画像按 uid hash 分（均匀分布）。

## 三、分片键选择（核心决策）

分片键一旦选定无法轻易换。三大原则：
1. **高频**：90% 查询都带的字段（uid）
2. **均匀**：值分布均匀，避免某分片热点（uid 通常均匀）
3. **不可变**：值不会变（避免变更时迁移数据）

**反面案例**：用手机号分片 → 用户换号 → 数据要跨分片迁移。

**分片算法**：
```java
// 范围分片（适合范围查询）
shardNo = uid / 10000000;  // 0-1000万→分片0, 1000万-2000万→分片1

// 哈希分片（均匀，但扩容要全量迁移）
shardNo = Math.abs(uid.hashCode()) % 16;

// 一致性哈希（扩容只迁移 1/N）
shardNo = consistentHash(uid, nodes);
```

## 四、全局 ID 生成

分库分表后 MySQL 自增主键失效，需要全局唯一 ID：

**雪花算法（Snowflake）**：
```
| 1 bit 符号 | 41 bit 时间戳 | 10 bit 机器ID | 12 bit 序号 |
              单位 ms，可用 69 年    1024 台机器     每毫秒 4096 个
```
- 去中心化、有序、高性能
- 依赖机器时钟（时钟回拨要处理）

**号段模式**（Leaf-Segment）：
- 中心化服务批量发号（每次发 1000 个）
- 客户端本地缓存，用完再申请
- 双 buffer 预加载，避免停顿

## 五、主从复制：读写分离

```
主库（Master）——写
   │ binlog 同步
   ├─→ 从库1（Slave）——读
   ├─→ 从库2（Slave）——读
   └─→ 从库3（Slave）——读/统计
```

**复制方式**：
| 方式 | 延迟 | 数据安全 |
|------|------|---------|
| 异步复制（默认） | 秒级 | 主挂可能丢未同步 |
| 半同步复制 | 毫秒级 | 至少一个从库收到才返回 |
| 组复制（MGR） | 强一致 | 多数派确认 |

**主从延迟问题**：异步复制有秒级延迟，读从库可能读到旧数据。

**解决方案**：
1. **关键场景强制读主**：`@Master` 注解（如交易后查余额）
2. **半同步复制**：业务可接受稍慢时
3. **客户端缓存"刚写"标记**：写后 N 秒内强制读主
4. **业务容忍**：列表页读旧数据无所谓

## 六、跨库难题与解法

**1. 分布式事务**：
- 强一致：XA（性能差，少用）
- 柔性事务：TCC、SAGA、本地消息表（最终一致）

**2. 跨库 JOIN**：
- 业务层组装：分别查多库，内存 join
- 冗余字段：把 join 字段冗余存
- 异构索引：把数据同步到 ES，在 ES 做 join

**3. 聚合统计**：
- 各库分别 count，汇总求和（count 可以）
- 但 sum/avg 跨库不准（avg 要加权）
- 异步汇总到汇总表

**4. 深分页**：
- 全局 LIMIT 1000000,10 跨库代价大
- 各库查前 N，业务层归并排序
- 或用 ES 异构索引统一查

## 七、风控实战架构

**事件表分库分表方案**：
```
risk_event 表（按月+uid 分）

物理库：risk_db_0, risk_db_1, ..., risk_db_7（8 库）
物理表：risk_event_202607_0, ..., risk_event_202607_7（每库每月1张）

路由：db = hash(uid) % 8, table = "risk_event_" + month + "_" + (hash(uid) % 8)
```

**ShardingSphere-JDBC 配置**：
```yaml
shardingsphere:
  datasource:
    names: ds0,ds1,...,ds7
    ds0: { ... }
  sharding:
    tables:
      risk_event:
        actual-data-nodes: ds${0..7}.risk_event_${202601..202612}_${0..7}
        database-strategy:
          hash:
            sharding-column: uid
            algorithm-expression: ds${uid.hashCode().abs() % 8}
        table-strategy:
          complex:
            sharding-columns: uid,created_at
            algorithm: riskEventTableAlgorithm
        key-generator:
          column: id
          type: SNOWFLAKE
```

**读写分离**：
```yaml
masterslave:
  name: ms_ds0
  master-data-source-name: ds0_master
  slave-data-source-names: [ds0_slave1, ds0_slave2]
```

## 八、底层本质：分而治之与一致性代价

分库分表本质是 **"用一致性换扩展性"**：
- 单库 ACID 强，但扩展性差
- 分库后失去跨库事务、JOIN、聚合
- 换来线性扩展能力（加机器即可加容量）

**这是分布式系统的 CAP 取舍**：
- 单库：CA（强一致 + 可用，不可扩展）
- 分库：AP（可用 + 可扩展，最终一致）

工程上的折中：**核心业务强一致（同库或柔性事务），非核心最终一致**。

**风控的实践**：
- 风险扣款：单库事务保证强一致
- 事件查询：分库分表 + ES 异构索引（最终一致）
- 实时统计：异步汇总到汇总表（容忍分钟级延迟）

## 常见考点
1. **分片键选错了怎么办**？——很难改，要么停服迁移，要么"双写新旧分片键"灰度切换。
2. **分库分表后还能加索引吗**？——能，但要每库都加（DDL 影响所有分片），用 pt-online-schema-change 平滑。
3. **ShardingSphere-JDBC 和 Proxy 区别**？——JDBC 是客户端模式（轻量、直连），Proxy 是中间层（语言无关、多一跳 RT）。

**代码示例**（柔性事务——本地消息表）：
```java
// 业务表 + 消息表在同一库（同事务）
@Transactional
public void deductAndSendEvent(long uid, long amount) {
    accountDao.deduct(uid, amount);              // 业务操作
    outboxDao.insert(new Outbox(uid, amount));   // 消息表（待发）
}
// 定时扫描 outbox 投递到 Kafka，成功后删除（最终一致）
```

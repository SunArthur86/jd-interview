# 【蚂蚁风控】JD 提到"数据隔离、系统解耦"，你怎么理解？风控平台怎么做的？

> JD 依据：原话"攻克各种高并发、数据隔离、系统解耦等方面的技术难关"。这是 P7 必考的架构思维。

## 一、表面层：为什么风控需要隔离和解耦

**故障传染案例**：
```
风控平台同时服务：借贷、支付、营销、海外

共享一套数据库：
- 海外业务某天流量暴涨 → 库 IO 满
- 借贷业务本来正常 → 也变慢 → 风控决策超时
- 支付业务也受影响 → 全平台告警

根因：所有业务共享同一物理资源，一个业务的故障传染给所有。
```

**解法**：
- **数据隔离**：每个业务独立库（爆炸半径限制在单业务）
- **系统解耦**：业务间用消息队列通信（一个挂了不影响其他）

## 二、数据隔离的 4 个级别

按隔离强度从强到弱：

### 1. 实例级隔离（最强）
```
借贷: MySQL 实例 A + 独立机器
支付: MySQL 实例 B + 独立机器
营销: MySQL 实例 C + 独立机器
```
- 优点：完全独立，故障绝对隔离
- 缺点：成本最高
- 适用：高价值业务、监管要求

### 2. 库级隔离
```
同实例，不同 database：
risk_db_loan, risk_db_pay, risk_db_marketing
```
- 优点：成本中等，库间故障不传染
- 缺点：共享实例资源（CPU/IO）

### 3. Schema 隔离
```
同库，不同 schema（PostgreSQL 概念，MySQL 中等价于库）
```

### 4. 行级隔离（最轻）
```
所有数据同表，用 tenant_id 区分：
CREATE TABLE risk_event (id, tenant_id, ...);
SELECT * FROM risk_event WHERE tenant_id = 'loan';
```
- 优点：成本最低
- 缺点：共享表，一个租户大查询拖死全部
- 适用：SaaS 多租户、租户数多但规模小

**风控的选择**：核心业务（借贷/支付）实例级隔离；外围业务（营销）库级隔离。

## 三、垂直隔离：按敏感级别

风控数据按敏感度分：
```
高敏数据（身份证、银行卡）：独立库 + 字段加密 + 访问审计
中敏数据（手机、地址）：同库 + 字段加密
低敏数据（用户偏好）：明文
```

**字段加密**：
```java
// 用 MyBatis TypeHandler 透明加解密
@TableName("user")
public class User {
    @FieldEncrypt  // 加密注解
    private String idCard;
    private String name;
}
// 写库前自动加密，读出自动解密
```

## 四、水平隔离：按租户/地区

**蚂蚁国际风控的多区域**：
```
中国大陆业务：cn-cluster（数据不出境，监管要求）
东南亚业务：sea-cluster（独立部署）
欧洲业务：eu-cluster（GDPR，独立部署）
```

每个区域是完整的风控平台实例，互不影响，数据不互通（合规要求）。

## 五、系统解耦：从强依赖到弱依赖

**强依赖的痛点**：
```
风控决策 → 同步调用 → 用户画像服务
              ↓ 慢
         决策线程池被拖死
```

**解耦手段**：

### 1. 消息队列解耦
```
风控决策 → 异步发事件 → Kafka → 用户画像更新、统计、告警
（决策主链路不依赖这些）
```

### 2. 读写分离解耦
```
写：风控决策写决策结果到主库
读：报表/统计读从库
（写不受读慢影响）
```

### 3. 接口契约解耦
```
决策服务只依赖画像服务的接口契约（不依赖实现）
画像服务挂了 → 决策走降级（用缓存画像）
```

### 4. 资源池隔离
```java
// 每个下游独立线程池
ThreadPoolExecutor profilePool = ...;   // 画像调用专用
ThreadPoolExecutor rulePool = ...;      // 规则调用专用
// 画像慢不会拖死规则调用
```

### 5. 限流熔断解耦
```
决策服务对每个下游配独立熔断规则
下游慢 → 熔断 → 不传染
```

## 六、单体 → 微服务的拆分原则

**按业务边界拆（DDD）**：
```
风控平台单体
   ↓
风控决策服务（评估）
风控规则服务（规则匹配）
风控特征服务（特征查询）
风控画像服务（用户画像）
风控案件服务（人工审核）
风控离线服务（T+1 计算）
```

**拆分原则**：
1. **高内聚低耦合**：同业务的代码聚一起，跨业务用接口
2. **独立数据**：每个服务独占自己的库
3. **弱依赖**：服务间尽量异步（MQ）
4. **领域驱动设计（DDD）**：按领域而非技术分层

## 七、最终一致性的保证

解耦后失去强一致，需要保证最终一致：

**1. 本地消息表**（最常用）：
```java
@Transactional
public void decideAndPost(Event e) {
    decisionDao.insert(decision);              // 业务表
    outboxDao.insert(new Outbox(decision));    // 消息表（同事务）
}
// 定时扫 outbox 投递 Kafka，成功删除
```

**2. TCC**（Try-Confirm-Cancel）：
- Try：预留资源
- Confirm：确认提交
- Cancel：失败回滚

**3. SAGA**：
- 长事务拆多个本地事务
- 失败时执行补偿事务

**4. 重试 + 幂等**：
- 消费方去重（业务幂等 token）
- 死信队列兜底

## 八、底层本质：隔离与解耦的工程哲学

**隔离**的本质是**限制爆炸半径**：
- 故障一定会发生，但能限制在哪个范围
- 用成本（独立资源）换安全（故障隔离）

**解耦**的本质是**降低传染概率**：
- 系统是网络，节点越紧耦合故障越传染
- 解耦让节点变成"软连接"——一个挂了其他能降级运行

**这是 CAP 在工程层面的体现**：
- 强一致（共享）= 强耦合 = 故障传染
- 最终一致（隔离）= 松耦合 = 故障隔离

**金融/风控的极端选择**：
- 不惜成本做物理隔离（监管 + 安全）
- 牺牲强一致换可用性（业务可容忍秒级延迟）
- 严格的爆炸半径控制（单业务故障不影响其他）

## 九、和 AI 时代的演进

**AI Agent 时代的解耦**：
- 传统：服务间 API 调用
- AI：Agent 编排（用 LLM 做意图理解 + 工具调用）
- 解耦方式：Agent 之间用自然语言/JSON 协议通信

**数据隔离在 AI 时代的新挑战**：
- 训练 AI 用了哪些数据？合规追溯
- AI 推理时数据按租户隔离（多租户模型）
- 敏感数据脱敏后才能喂给 LLM

## 常见考点
1. **多租户怎么选隔离级别**？——按租户规模和价值。租户少价值高（金融）→ 库/实例隔离；租户多价值低（SaaS）→ 行级隔离。
2. **解耦和事务一致性的矛盾**？——解耦牺牲强一致，靠最终一致补偿（本地消息表/TCC）。
3. **服务拆多细合适**？——按业务能力拆，过细会引入网络开销和管理复杂度。一个团队负责一个服务（康威定律）。

**代码示例**（资源隔离 + 熔断）：
```java
// 不同下游用独立线程池 + 独立熔断规则
@Async("profilePool")  // 独立线程池
@SentinelResource(value = "callProfile", blockHandler = "fallback")
public Future<Profile> callProfile(String uid) {
    return CompletableFuture.completedFuture(profileService.get(uid));
}

public Profile fallback(String uid, BlockException ex) {
    return profileCache.get(uid);  // 降级走缓存
}
```

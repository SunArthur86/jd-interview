# 【蚂蚁风控】设计一个亿级 QPS 的风控系统，你怎么做？

> JD 依据："前瞻性地探索新型风控系统的架构设计"。这是 P7+ 的核心架构题。

## 一、需求拆解

**业务需求**：
- 覆盖蚂蚁国际全业务（支付、借贷、营销、海外）
- 每笔交易实时决策（拦截/放行/复核）
- 风险识别准确（漏报率低、误杀率低）
- 可解释、可审计

**非功能需求**：
- **QPS**：峰值 100 万+（亿级日交易）
- **RT**：P99 < 200ms（用户支付等不及）
- **可用性**：99.99%（不能漏判风险）
- **扩展性**：新业务快速接入
- **可解释性**：每笔决策有依据

## 二、整体架构（分层 + 异步）

```
┌────────────────────────────────────────────────────────────────┐
│                       接入层（万兆网络）                       │
│   SDK / 网关 / 事件订阅 → 风控入口（HTTP/Dubbo/Kafka）         │
└────────────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────────────┐
│            同步决策层（<100ms，亿级 QPS）                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ 名单服务 │→│ 特征服务 │→│ 规则引擎 │→│ 模型服务 │      │
│  │ 布隆过滤 │  │ Redis缓存│  │ 决策树   │  │ 预算分   │      │
│  │ Bloom    │  │ HBase查  │  │ RETE匹配 │  │ 在线推理 │      │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘      │
└────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────┐
│            异步补全层（秒级，准实时）                         │
│   特征回流（Kafka→Flink→特征库）                              │
```

## 三、同步决策链路（核心 100ms）

```
1. 接入（5ms）
   ├─ 鉴权（API key/签名）
   └─ 限流（Sentinel：单 UID QPS + 全局 QPS）

2. 黑名单/白名单（5ms，前置短路）
   ├─ 白名单：直放（高管、可信商户）
   └─ 黑名单：直拦（已知欺诈 UID/设备/IP）
   工具：Redis 全量缓存 + 布隆过滤器（内存）

3. 特征查询（30ms，并行）
   CompletableFuture 同时拉：
   ├─ Redis 热数据（10ms）
   ├─ HBase 用户画像（30ms）
   ├─ Flink 实时特征（30ms）
   超时 30ms 全部降级缓存

4. 规则匹配（30ms）
   ├─ 前置规则（强规则，硬拦截）
   ├─ 专家规则（RETE 引擎）
   └─ 决策树（运营配置）

5. 模型推理（30ms）
   ├─ 预算分（离线算好的分，Redis 直查）
   ├─ 在线模型（动态特征推理）
   └─ 模型版本路由（A/B）

6. 决策融合（5ms）
   ├─ 规则 + 模型加权
   ├─ 按业务阈值输出决策
   └─ 输出：决策 + 命中规则 + 风险分

总 RT ≈ 100ms（并行化）
```

**并行化的关键**：
```java
public RiskResult decide(Event e) {
    // 并行查询所有特征
    CompletableFuture<Profile> profileF = supplyAsync(() -> profileService.get(e.uid), pool);
    CompletableFuture<Features> featF = supplyAsync(() -> featService.get(e.uid), pool);
    CompletableFuture<ModelScore> scoreF = supplyAsync(() -> modelService.score(e), pool);

    // 等所有结果（带超时）
    CompletableFuture.allOf(profileF, featF, scoreF).get(80, MS);

    // 决策融合
    return fuse(profileF.join(), featF.join(), scoreF.join());
}
```

## 四、容量估算

**业务量**（参考蚂蚁双 11 量级）：
- 日均请求：1-2 亿次
- 日常峰值 QPS：5-10 万
- 双 11 峰值 QPS：100 万+（注：蚂蚁双 11 全平台峰值约 58 万笔/秒，风控链路因多场景叠加可达百万级）

**同步层容量**：
```
单实例：5000 QPS（4C8G）
峰值 100万 QPS → 200 实例（K8s 弹性扩缩）

Redis 集群：100G 内存（缓存热数据）
HBase 集群：100TB（亿级用户 × 千字段）
Kafka 集群：1000 万 QPS 入口
Flink 集群：1000 个 slot
```

**削峰设计**：
- 接入层限流（按 UID 限）
- 缓存命中率 > 95%（Redis）
- 慢请求异步化（非关键路径走 Kafka）

## 五、高可用设计（4 个 9）

**1. 多机房部署**：
- 同城双活（上海主 + 上海备）
- 异地灾备（杭州）
- 流量按机房就近路由（cluster-name）

**2. 服务降级**：
- 特征挂 → 用缓存特征（30 分钟前的）
- 规则挂 → 用兜底规则（黑名单 + 阈值）
- 模型挂 → 退化为规则决策
- 全部挂 → 兜底放行（业务优先）或拦截（风险优先）

**3. 限流保护**：
- 接入层：全局 + 单 UID
- 服务层：每个下游独立熔断
- 数据层：连接池上限

**4. 容灾切换**：
- 注册中心感知实例健康
- 故障实例自动摘除
- 机房级故障自动切流

## 六、数据一致性

**强一致**（决策结果）：
- 决策落库用 MySQL 主库（强一致）
- 决策结果缓存 Redis（最终一致）

**最终一致**（特征、画像）：
- Flink 实时特征 → HBase（秒级延迟）
- 用户画像 → 异步更新（分钟级延迟）

**异步事件**：
- 决策完发 Kafka 事件（异步下游消费）
- 本地消息表保证不丢（业务+消息表同事务）

## 九、底层本质：分层 + 异步 + 缓存的工程哲学

应对海量流量（日均亿级请求、峰值百万级 QPS）的本质是**"用空间换时间，用异步换吞吐，用分层换效率"**：

**1. 分层（解决决策精度 vs 时效矛盾）**：
- 实时层（毫秒）做粗筛
- 准实时层（秒）做精筛
- 离线层（天）做深度

**2. 异步（解决吞吐 vs 延迟矛盾）**：
- 同步链路做决策（必须快）
- 异步链路做补全（可以慢）
- 解耦强弱依赖

**3. 缓存（解决读 vs 计算矛盾）**：
- 热数据缓存（99% 命中 Redis）
- 预计算缓存（离线算好直接查）
- 多级缓存（本地 + Redis + DB）

**这是分布式系统的"不可能三角"破解**：
- 强一致 + 高可用 + 高性能不能同时满足
- 但可以"分层取舍"：核心强一致，外围最终一致；同步保时效，异步保吞吐

## 十、和 AI 的融合（见 jd-ai 批次）

- **模型层升级**：从规则 + 浅模型 → 大模型实时推理
- **决策层升级**：用 Agent 编排多模型决策
- **特征层升级**：LLM 生成语义特征
- **运营升级**：自然语言配置规则（AI 翻译成 DSL）

## 常见考点
1. **决策链路为什么能压到 100ms**？——并行化（CompletableFuture）+ 缓存（Redis 95% 命中）+ 短路（命中强规则即返回）+ 异步化（非关键走 Kafka）。
2. **怎么保证决策不漏**？——兜底策略 + 异步复核 + 离线全量审计；漏报是风险，但绝不漏决策（必须给个结果）。
3. **怎么评估风控系统好坏**？——核心三率：召回率（漏报）、精准率（误杀）、决策 RT；再加运营效率（规则迭代速度）。

**代码示例**（决策融合）：
```java
public RiskResult fuse(Profile p, Features f, ModelScore s) {
    int riskScore = 0;
    List<String> hits = new ArrayList<>();

    // 强规则（命中即拦）
    if (f.isBlackDevice) { return RiskResult.reject("black_device"); }

    // 专家规则加权
    for (Rule r : expertRules) {
        if (r.match(p, f)) {
            riskScore += r.getWeight();
            hits.add(r.getId());
        }
    }

    // 模型分加权
    riskScore += s.getScore() * MODEL_WEIGHT;

    // 决策
    if (riskScore >= REJECT_THRESHOLD) return RiskResult.reject(hits);
    if (riskScore >= REVIEW_THRESHOLD) return RiskResult.review(hits);
    return RiskResult.pass();
}
```

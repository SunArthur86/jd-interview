# 【生物医药 AI】AI Coding 团队规范怎么落地（Cursor/Claude Code）？

> JD 依据："AI Coding 提效；团队管理；工程化。"

## 一、为什么需要团队规范

```
无规范：
  各人用不同 AI 工具 → 风格不一
  AI 乱写代码 → 质量参差/安全隐患
  盲信 AI 合代码 → bug/架构腐化
  不度量 → 不知道到底提效没

有规范：
  统一工具+规则 → 风格一致
  必经审查 → 质量可控
  度量指标 → 提效有数
  人机分工 → 效率最大化
```

## 二、工具统一与规则注入

### 1. 统一工具
- 团队统一选型（如 Cursor + Claude Code）。
- 配置同步（VSCode settings/extensions）。

### 2. 规则文件（关键）
```
.cursorrules / CLAUDE.md / AGENTS.md（项目根目录）
内容：
  - 技术栈与版本（Python3.11/FastAPI/PG）
  - 编码规范（命名/格式/类型注解）
  - 架构约束（分层/目录结构/依赖方向）
  - 禁止项（不用某库/不直接操作 DB）
  - 测试要求（每功能带单测）
  - 安全红线（不硬编码 secret/输入校验）
→ AI 每次生成都遵守这些规范
```

## 三、人机分工（合理边界）

| 任务 | AI 适合 | 人适合 |
|------|---------|--------|
| 样板代码/CRUD | ✅ 快 | 慢 |
| 单元测试 | ✅ 快 | 审查 |
| 重构/格式化 | ✅ 快 | 定方向 |
| 文档/注释 | ✅ 快 | 校验 |
| 核心业务逻辑 | 辅助 | ✅ 主导 |
| 架构设计 | 建议 | ✅ 决策 |
| 安全/合规 | 检测 | ✅ 把关 |
| Code Review | 辅助 | ✅ 最终 |

**AI 写代码 + 人审查决策** 是最有效的组合，不是"AI 全自动"。

## 四、代码审查（不能省）

```
AI 生成的代码 = 草稿，必须人工 review：
  - 正确性（逻辑对不对）
  - 安全（注入/越权/依赖）
  - 架构（符合规范吗）
  - 性能（有无低效）
  - 测试（有无覆盖）
→ 盲信 AI 合代码是事故源头
```
- AI 也能辅助 review（发现潜在 bug），但人最终拍板。

## 五、提效度量

### 效率指标
- **提速比**：同任务 AI 辅助 vs 纯手工时长。
- PR 频率/大小（AI 往往小步快提交）。
- 交付周期（需求到上线时长）。

### 质量指标
- 缺陷率（AI 生成代码的 bug 率）。
- 返工率（review 打回率）。
- 测试覆盖率。

**分任务看**：样板/测试提速大，核心逻辑提速有限（AI 辅助而非主导）。

## 六、安全与合规

- **依赖审查**：AI 可能引不安全/幻觉包，检查真实存在+无漏洞。
- **secrets 扫描**：AI 可能硬编码密钥，扫描拦截。
- **输入校验**：AI 可能漏边界，强制 review。
- **医药合规**：AI 写涉及患者数据/合规逻辑，必须专家审。

## 七、团队文化与流程

```
落地步骤：
1. 统一工具 + 写规则文件
2. 培训（怎么用好 AI、什么该审）
3. 试点（一个项目验证）
4. 推广 + 持续优化规则
5. 度量提效，迭代规范

文化：
  - 鼓励用 AI（不抵触）
  - 但强调审查（不盲信）
  - 分享好 prompt/规则
  - AI 是工具，责任在人
```

## 八、底层本质

AI Coding 团队规范本质是**"用工程化约束让 AI 辅助编程的产出可控、一致、可度量、安全"**。统一工具+规则注入规范，人工审查保质量，指标度量提效，人机合理分工。

**AI Coding 的价值不在'少写代码'，而在'让人专注高价值决策和审查'** —— 工具是杠杆，规范是支点，缺一不可。

## 常见考点

1. **怎么让 AI 按团队规范写代码？**——规则文件（.cursorrules/CLAUDE.md）注入编码规范/架构/禁止项/安全红线；AI 每次都读，配合 review 校验。
2. **AI 写的代码能直接合吗？**——不能，必须人工 review（正确性/安全/架构/测试）；AI 是草稿不是定稿，盲信是事故源。
3. **怎么证明 AI Coding 提效？**——提速比（同任务时长对比）+ PR 频率/交付周期 + 缺陷率，分任务类型统计，用数据说话。

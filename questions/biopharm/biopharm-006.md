# 【生物医药 AI】向量数据库怎么选型与优化（FAISS/PGVector/Milvus/ES）？

> JD 依据："FAISS、PGVector、Milvus、Elasticsearch。"

## 一、为什么需要向量库

```
query → embedding → 在百万/亿级向量中找最相似的 top-k
暴力算余弦：O(N)，N=1亿时每次查全表，扛不住
→ 向量库用 ANN（近似最近邻）算法，毫秒级返回
```

## 二、四大选型对比

| 选型 | 定位 | 规模 | 优势 | 劣势 |
|------|------|------|------|------|
| FAISS | 库（非服务） | 单机百万~千万 | 高性能、零运维、灵活 | 非服务、无分布式、无权限 |
| PGVector | PG 扩展 | 千万级 | 与业务数据一体、事务一致 | 超大规模性能掉 |
| Milvus | 分布式专用库 | 十亿级 | 海量、高可用、多索引 | 运维重、架构复杂 |
| Elasticsearch | 搜索引擎 | 亿级 | 向量+全文+结构化混合 | 向量召回略逊专用库 |

## 三、各选型深入

### 1. FAISS —— 轻量原型/中小规模
- Meta 开源的库（C++/Python），不是独立服务。
- 索引：Flat（精确）/ IVF / HNSW / IVF-PQ（压缩）。
- 适合：原型验证、单机中小规模、嵌入式集成。
- 缺点：无分布式、无权限、无事务，要自己包服务层。

### 2. PGVector —— 业务一体（中小规模首选）
- PostgreSQL 扩展，向量表和业务表在同一库。
- **核心优势**：事务一致性 —— 向量插入和元数据更新原子提交；权限/版本/标签用 SQL 直接 join 过滤。
- 索引：HNSW（推荐）/ IVFFlat。
- 适合：知识库量级在千万以内、强一致和多表联合查询的场景。
- 详见 PGVector 实战题（035）。

### 3. Milvus —— 海量分布式
- 专用分布式向量库，支持十亿级。
- 多索引（IVF/HNSW/DiskANN）、分片、高可用、多副本。
- 适合：亿级以上、高 QPS、专业运维团队。
- 代价：架构复杂（coord/node/etcd/minIO 多组件），运维成本高。

### 4. Elasticsearch —— 混合检索之王
- 8.x 支持 dense_vector + kNN 搜索。
- **核心优势**：向量 + 全文（BM25）+ 结构化过滤一体化，RRF 融合。
- 适合：需要语义+关键词混合检索、已有 ES 运维能力。
- 代价：向量召回精度略逊专用库，资源消耗大。

## 四、选型决策（四问）

```
1. 数据量？
   < 千万 → PGVector / FAISS
   千万~亿 → Milvus / ES
   > 亿 → Milvus
2. 要向量+全文混合检索吗？
   是 → ES / PGVector（也可）
3. 要和业务数据强一致吗？
   是 → PGVector
4. 运维能力？
   弱 → PGVector（复用 DB 运维）/ 托管服务
   强 → Milvus / ES 自建
```

## 五、优化要点

- **索引调参**：HNSW 的 M/efConstruction/efSearch 权衡召回与速度。
- **量化压缩**：PQ/SQ 把向量压缩，省内存换少量精度损失。
- **预过滤**：检索前按元数据（权限/版本/租户）过滤，避免后过滤不足 k 个。
- **分片与副本**：大规模按向量分片，读多加副本。
- **更新策略**：增量更新 + 定期重建索引（避免碎片）。

## 六、医药场景的选型倾向

- **中小药企知识库**（说明书/文献，百万级）：PGVector（一体、事务、权限好做）。
- **大规模文献库**（亿级 PubMed）：Milvus（规模）+ ES（混合检索）组合。
- **临床试验数据**（强结构化+权限）：PGVector（事务一致 + 行级权限）。

## 七、底层本质

向量库选型本质是**"在规模、一致性、混合检索、运维成本四维上做权衡"**。没有银弹，关键是搞清楚自己的约束（数据量、是否混合检索、是否强一致、运维能力），在四象限里定位。

**很多人误以为' Milvus 最强所以都选 Milvus'，其实中小规模用 PGVector 往往更省更稳** —— 过度设计是企业 RAG 的常见坑。

## 常见考点

1. **为什么不用 MySQL 直接存向量？**——MySQL 无原生 ANN 索引，全表扫描慢；PGVector 补上了 HNSW/IVF 索引，才可用。
2. **HNSW 和 IVF 怎么选？**——HNSW 召回高/查询快/内存大，IVF 省内存/可分布式/召回略低；看内存预算和规模。
3. **多租户怎么隔离？**——元数据过滤（tenant_id 预过滤）/ 分集合（collection per tenant）/ 分库，按规模和安全等级选。


## 结构化回答

**30 秒电梯演讲：** 聊到向量数据库怎么选型与优化，我的理解是——向量数据库选型是在'规模、运维、一致性、混合检索'四象限里挑——FAISS 轻量单机，PGVector 与业务库一体，Milvus 大规模分布式，ES 强在向量+全文混合。打个比方，像选仓库——FAISS 是随身背包（小快）、PGVector 是和办公室连一起的柜子（数据在一处）、Milvus 是大型自动化立体库（海量但要专人管）、ES 是能同时按图按文找的智能仓。

**展开框架：**
1. **FAISS** — 库（非服务），单机高性能，适合原型/中小规模
2. **PGVector** — PostgreSQL 扩展，向量与业务数据一体，事务一致
3. **Milvus** — 分布式专用向量库，十亿级规模，运维重

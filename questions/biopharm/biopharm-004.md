# 【生物医药 AI】Workflow 引擎怎么设计（DAG/状态机/人工审批）？

> JD 依据："LLM 应用开发：Agent、RAG、Workflow、Tool Calling 核心能力建设；异步编程、高并发系统设计。"

## 一、Workflow 的本质：流程工程化

```
裸 LLM：query → answer（一次性，无流程）

Workflow：query → [步骤1] → [步骤2] → [分支/审批] → [步骤N] → result
                   （可恢复、可审批、可观测、可回滚）
```

把"业务流程"从代码里抽出来，变成可配置、可调度、可监控的执行图。

## 二、核心三件套：DAG + 状态机 + 人工节点

### 1. DAG（有向无环图）：定义步骤依赖
```
[检索文献] ──┐
            ├→ [筛选] → [总结] → [审批] → [导出]
[调数据库] ──┘
```
- 支持并行（无依赖步骤并发）、分支（条件走向）、汇合（等齐再走）。
- 用 JSON/YAML 或可视化拖拽定义，不写死在代码里。

### 2. 状态机：驱动执行 + 断点恢复
每个步骤有状态：`PENDING / RUNNING / PAUSED / SUCCESS / FAILED / COMPENSATING`。
```
每步执行前：状态 = RUNNING，写 checkpoint（输入+状态）到 DB
执行完：状态 = SUCCESS，写输出
进程挂了重启：读 checkpoint，从最近 SUCCESS 步骤续跑
```
**断点恢复**是长任务（如跑几小时的文献综述）的生命线。

### 3. 人工节点（HITL）：合规审批
```
步骤 = 审批 → 状态置 PAUSED → 入待审批队列 → 通知审批人
审批人通过/驳回 → 回调恢复 → 继续/回滚
```
医药场景：用药建议、临床结论、对外报告必须人工确认后才输出。

## 三、异常处理：超时/重试/补偿

| 异常 | 处理 |
|------|------|
| 步骤超时 | 重试 N 次 → 仍失败则告警/人工介入 |
| 步骤失败 | 重试 → 失败则进入补偿 |
| 已写脏数据 | **Saga 补偿**：逆向执行已成功步骤的补偿动作 |
| 整体超时 | 终止 + 通知 + 保留现场 |

**Saga 补偿示例**：
```
步骤1: 创建草稿   补偿: 删除草稿
步骤2: 发送通知   补偿: 撤回通知
步骤3 失败 → 逆序执行：撤回通知 → 删除草稿
```

## 四、执行架构（伪代码）

```python
class WorkflowEngine:
    def run(self, workflow_def, input):
        state = self.load_or_init(workflow_def.id)  # 断点恢复
        for node in self.next_runnable(state):       # DAG 调度
            if node.type == "human":
                state.pause(node); notify(node); return  # 等审批
            try:
                out = self.execute(node, state)      # 含 LLM/工具/检索
                state.save(node, out)                # checkpoint
            except Failure:
                self.compensate(state); raise
        return state.result
```

## 五、可观测性

- **全链路 trace**：每个步骤的输入/输出/耗时/状态全记录。
- **指标**：流程成功率、平均耗时、各步骤瓶颈、人工审批等待时长。
- **告警**：失败率/耗时超阈值、审批超时堆积。

## 六、Workflow vs Agent：边界

| | Workflow | Agent |
|---|---------|-------|
| 决策 | 预定义流程（确定性） | LLM 自主规划（概率性） |
| 适用 | 合规强、流程固定（审批/报表） | 开放探索（问答/研究） |
| 可靠 | 高（每步可控） | 需校验闭环 |

**实战常结合**：外层 Workflow 控骨架（确定性步骤+审批），内层局部用 Agent 做灵活决策。

## 七、底层本质

Workflow 引擎本质是**"把业务流程从代码里解放，变成可配置、可恢复、可审批、可观测的执行系统"**。DAG 管依赖，状态机管恢复，人工节点管合规，补偿事务管回滚。

**企业级 AI 的'可靠'很大程度上靠 Workflow 而非模型本身** —— 模型负责能力，流程负责可靠。

## 常见考点

1. **为什么不用普通函数调用串联步骤？**——函数串联无法断点恢复、无法人工暂停、无法可视化配置、无法补偿回滚；流程变更要改代码。
2. **长任务怎么不丢？**——每步 checkpoint 持久化 + 幂等执行（重试不会重复副作用）+ 重启续跑。
3. **人工审批超时怎么办？**——超时升级（转上级）、超时驳回、超时默认策略（按业务定）。

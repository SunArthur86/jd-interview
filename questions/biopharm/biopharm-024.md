# 【生物医药 AI】Agent 框架怎么对比与选型（LangChain/LangGraph/Dify/AutoGen/CrewAI）？

> JD 依据："LangChain/LlamaIndex/Dify/LangGraph/AutoGen/CrewAI 至少一种 Agent 框架。"

## 一、为什么选型重要

```
选错框架的代价：
  - 太重（LangChain 全家桶）→ 简单项目过度工程
  - 太轻（Dify 低代码）→ 生产级定制不够
  - 不匹配场景（要循环却用线性链）→ 推倒重来
→ 框架决定开发效率、可控性、可维护性
```

## 二、五大框架对比

| 框架 | 抽象 | 可控性 | 学习成本 | 适用 |
|------|------|--------|----------|------|
| LangChain | 链/记忆/工具（全家桶） | 中 | 中高 | 学习/小项目/生态复用 |
| LangGraph | 图式状态机 | 高 | 高 | 复杂生产级 Agent（循环/分支） |
| Dify | 低代码可视化 | 低 | 低 | 快速搭 RAG/客服/原型 |
| AutoGen | 多 Agent 对话 | 中 | 中 | 多角色研讨/协作 |
| CrewAI | 角色分工 | 中 | 中低 | 流水线式多 Agent |

## 三、各框架深入

### 1. LangChain —— 生态全家桶
- 提供链（Chain）、记忆（Memory）、工具（Tool）、检索（Retriever）、Agent 抽象。
- 生态最全（集成几百个工具/模型/向量库）。
- **优点**：学习资源多、集成全。
- **缺点**：抽象重、过度封装、复杂场景难控、版本变动频繁。
- 适合：学习入门、复用生态、中小项目。

### 2. LangGraph —— 图式精细控制（生产级首选）
```
节点（Node）+ 边（Edge）+ 状态（State）
支持：并行、分支、循环、人工节点、断点恢复
```
- 把 Agent 抽象为状态图，精细控制每步。
- 支持复杂流程（多步推理/工具循环/人工审批）。
- **优点**：可控性强、适合生产、可恢复。
- **缺点**：学习曲线高、需自己设计图。
- 适合：生产级复杂 Agent（如医药综述/多轮诊断）。

### 3. Dify —— 低代码可视化
- 拖拽搭 RAG/Agent/Workflow，内置模型/向量库集成。
- **优点**：上手快、非开发也能搭、快速验证。
- **缺点**：深度定制弱、大规模/特殊需求受限。
- 适合：中小团队快速搭客服/知识库/原型。
- 详见 Dify 可视化题（038）。

### 4. AutoGen —— 多 Agent 对话
```
多个 Agent（AssistantAgent/UserProxyAgent）对话协作
适合：多角色研讨（如 code review/分析讨论）
```
- Agent 之间对话推进任务。
- **优点**：多视角协作自然。
- **缺点**：对话式可能发散、难收敛、token 消耗大。
- 适合：多角色研讨（医生+药师+文献员讨论）。

### 5. CrewAI —— 角色分工
```
定义 Agent（角色+目标+工具）+ Task（任务）+ Crew（编队）
按角色流水线分工执行
```
- **优点**：角色化分工直观、适合流水线。
- **缺点**：复杂控制不如 LangGraph。
- 适合：内容生产/调研/报告生成等流水线任务。

## 四、选型决策

```
1. 快速验证/非开发团队 → Dify
2. 学习/小项目/复用生态 → LangChain
3. 生产级复杂 Agent（循环/分支/审批） → LangGraph
4. 多角色研讨协作 → AutoGen
5. 角色分工流水线 → CrewAI
6. 强合规/极致性能/特殊流程 → 自研（常借鉴框架）
```

## 五、生产级现实（框架+自研结合）

```
很多生产系统不全用单一框架：
  - 用 LangGraph 做编排骨架
  - 自研关键节点（校验/审计/医药合规）
  - 用框架的集成（模型/向量库/工具）
  - 自研可观测/计量/治理
→ 框架省轮子，自研补定制和可靠
```

## 六、医药场景的选型倾向

- **医药综述/多轮诊断 Agent**（复杂流程）：LangGraph（精细可控+人工节点）。
- **企业知识库客服**（快速上线）：Dify（低代码）+ 自研合规层。
- **文献多角色研讨**：AutoGen（医生/药师/文献员对话）。
- **强合规生产级**：LangGraph 骨架 + 自研校验/审计。

## 七、底层本质

Agent 框架选型本质是**"匹配团队能力、控制需求和上线节奏"**。LangChain 全/重，LangGraph 精细可控，Dify 快但定制弱，AutoGen 多 Agent 对话，CrewAI 角色分工。

**框架是脚手架不是终极方案** —— 生产级系统往往是框架（省轮子）+ 自研（补定制和可靠）的结合。

## 常见考点

1. **LangChain 为什么生产少用？**——抽象重、过度封装、复杂场景难控、版本变动；生产更倾向 LangGraph（精细可控）或自研。
2. **Dify 能上生产吗？**——中小规模/标准场景可以；大规模/强定制/特殊合规可能受限，常做原型再迁自研。
3. **多 Agent 什么时候用？**——任务需多角色多视角（研讨/分工）时用；单线任务用单 Agent + Workflow 更可控更省 token。


## 结构化回答

**30 秒电梯演讲：** 聊到Agent 框架怎么对比与选型，我的理解是——Agent 框架选型是'按你的控制欲和场景挑'——LangChain 全家桶生态全，LangGraph 图式精细可控，Dify 低代码快搭，AutoGen 多 Agent 对话，CrewAI 角色分工。控制欲越强越偏代码框架，越想快搭越偏低代码。打个比方，像选交通工具——LangChain 是瑞士军刀（啥都有但杂），LangGraph 是手动挡跑车（精细控制但要会开），Dify 是自动驾驶（快但定制弱），AutoGen 是会议室（多 Agent 开会），CrewAI 是剧组（角色分工演戏）。按场景选车。

**展开框架：**
1. **LangChain** — 生态全（链/记忆/工具/检索），抽象重，适合学习和小项目
2. **LangGraph** — 图式状态机，精细可控循环/分支，适合复杂生产级 Agent
3. **Dify** — 低代码可视化，快速搭 RAG/Agent，定制弱

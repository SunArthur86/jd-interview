# 【生物医药 AI】向量检索进阶怎么做（ANN/HNSW/IVF/量化）？

> JD 依据："FAISS、PGVector、Milvus、Elasticsearch；向量数据库选型。"

## 一、为什么需要 ANN

```
精确检索（暴力）：query 和每个向量算余弦 → O(N)
  N=1亿 → 每次查全表，秒级，扛不住

ANN（近似最近邻）：
  用可接受的精度损失（召回率 90-99%）换巨大速度提升（毫秒级）
  → 大规模检索的唯一可行解
```

## 二、三角权衡

```
        召回率（找全正确的）
           ↑
           |    HNSW（高召回/快/内存大）
           |
           |※ 帕累托前沿
           |
           |    IVF+PQ（中召回/省内存）
           |
           +──────────→ 速度
          /
         / 内存占用
```
没有"又快又准又省"，按场景在三角中找平衡。

## 三、HNSW（分层小世界图）

### 原理
```
建图：向量组织成多层图
  顶层：稀疏，少量节点，长距离边（快速跨区域）
  底层：稠密，所有节点，短距离边（精细）

查询：从顶层入口 → 贪心导航逐层下降 → 底层精细搜索
像导航：高速→省道→小路，快速逼近目标
```

### 特点
| 维度 | HNSW |
|------|------|
| 召回 | 高（95-99%） |
| 速度 | 快（亚毫秒~毫秒） |
| 内存 | 大（存图结构） |
| 构建 | 慢（建图耗时） |
| 分布式 | 不友好（图难切分） |

### 关键参数
- `M`：每个节点的邻居数（大→召回高/内存大）。
- `efConstruction`：建图候选数（大→质量好/慢）。
- `efSearch`：查询候选数（大→召回高/慢）。

**适合**：中小规模（千万级内）、高召回、单机或少量分片。

## 四、IVF（倒排分桶）

### 原理
```
训练：k-means 把向量聚成 nlist 个桶（聚类中心）
插入：每个向量归到最近的桶
查询：找离 query 最近的 nprobe 个桶，桶内暴力搜索
```

### 特点
| 维度 | IVF |
|------|------|
| 召回 | 中（调 nprobe） |
| 速度 | 快（只搜部分桶） |
| 内存 | 省（只存向量+桶索引） |
| 分布式 | 友好（桶可分片） |

### 关键参数
- `nlist`：桶数（大→每桶小/训练久）。
- `nprobe`：查询探查桶数（大→召回高/慢）。

**适合**：大规模、需分布式、内存敏感。

## 五、量化压缩

### PQ（Product Quantization）
```
向量切段，每段独立用码本聚类
向量 → 替换为各段码本索引（短编码）
内存省 8-64 倍，精度小损
```

### SQ（Scalar Quantization）
```
FP32 → INT8（每维单独量化）
内存省 4 倍，几乎无损
```

### 组合
- **IVF+PQ**：大规模省内存首选（Milvus/FAISS 常用）。
- 先分桶再桶内压缩，兼顾速度和内存。

## 六、选型决策

```
1. 数据量？
   < 千万 → HNSW（高召回快）
   千万~亿 → IVF / HNSW 分片
   > 亿 → IVF+PQ + 分布式
2. 内存敏感？
   是 → IVF+PQ（量化压缩）
3. 需分布式？
   是 → IVF（桶分片友好）
4. 召回优先？
   是 → HNSW（调大 efSearch）
```

## 七、调参与评估

```
评估指标：
  - 召回率（ANN 结果 vs 精确结果的覆盖率）
  - 查询延迟（P99）
  - 内存占用
  - 吞吐（QPS）

调参流程：
  建评测集（标注真实近邻）
  → 扫参数（HNSW: efSearch / IVF: nprobe）
  → 画召回-延迟曲线
  → 选满足业务召回（如 95%）的最低延迟点
```

## 八、底层本质

向量检索进阶本质是**"在召回率-速度-内存三角中按场景找最优 ANN 算法"**。HNSW 图导航快准内存大，IVF 分桶省内存可分布式，PQ/SQ 压缩省空间，IVF+PQ 组合大规模首选。

**没有银弹，调参是关键** —— 同一算法不同参数性能差几倍，必须基于真实数据建评测集调参。

## 常见考点

1. **HNSW 和 IVF 怎么选？**——中小规模/高召回/单机选 HNSW；大规模/分布式/内存敏感选 IVF（+PQ）；HNSW 分片难，IVF 分桶天然适合分布式。
2. **量化损失大吗？**——SQ（INT8）几乎无损；PQ 看切段数和码本大小，常见 8x4 配置精度小损（召回降几个百分点），关键场景要评测。
3. **怎么调参？**——建评测集（标注真实近邻）→ 扫参数（efSearch/nprobe）→ 画召回-延迟曲线 → 选满足业务召回的最低延迟点，不能拍脑袋。


## 结构化回答

**30 秒电梯演讲：** 聊到向量检索进阶怎么做（ANN/HNSW/IVF/量化），我的理解是——向量检索进阶是搞懂 ANN 算法如何在'召回率、速度、内存'三角中权衡——HNSW 图导航快准但费内存，IVF 分桶省内存可分布式，量化压缩把向量变小，按场景组合。打个比方，像在图书馆找书——暴力翻每本是精确（全表扫描，慢）；HNSW 是按'相似书聚类'+目录索引一层层缩小（图导航，快但建索引费空间）；IVF 是先分区分书架再区内找（分桶，省空间）；量化是把书名缩写成代号（压缩，省地方略损精度）。

**展开框架：**
1. **三角** — 召回率 vs 查询速度 vs 内存占用
2. **HNSW** — 分层小世界图，导航式检索，快准但内存大
3. **IVF** — 倒排分桶，桶内搜索，省内存可分布式，需训练

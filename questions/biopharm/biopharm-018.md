# 【生物医药 AI】负载均衡/隔离/压测怎么做（GPU/多租户/全链路）？

> JD 依据："Linux/Git/Docker/K8s/云原生；系统性能优化；高并发系统设计。"

## 一、为什么 AI 系统要专门讲均衡/隔离/压测

```
传统后端：CPU/内存均匀，负载均衡简单
AI 系统：
  - GPU 异构（不同型号/显存）
  - 模型大小不一（7B/70B 占用差几倍）
  - 多租户共享，互相干扰
  - 请求耗时长，长尾拖垮 P99
  - 压测不能用 mock 秒回（要模拟真实耗时）
```

## 二、负载均衡（多级）

### 1. 网关层均衡
```
请求 → 负载均衡器（Nginx/LB）→ 多个推理实例
策略：轮询 / 加权轮询（按 GPU 能力）/ 最少连接 / 一致性哈希（会话亲和）
```

### 2. 推理层均衡（GPU 调度）
```
多个 GPU worker → 全局调度器按负载分发
考虑：GPU 利用率 / 显存 / 队列长度 / 模型亲和
避免热点：某 GPU 打爆，别的闲着
```
- vLLM 等推理引擎内部有 continuous batching 调度。
- 多模型时按模型路由到对应 GPU 组。

## 三、隔离（防互相干扰）

### 1. 多租户隔离
```
维度：
  数据隔离：tenant_id 贯穿（见012）
  资源隔离：每租户 GPU/连接 配额
  性能隔离：限流+优先级，防某租户拖慢全局
```
- 大租户独占资源池，小租户共享+限流。

### 2. GPU 隔离（关键）
```
一张 GPU 多模型/多租户共享时：
  时间片：分时复用（轮流用）
  MPS（Multi-Process Service）：多进程共享，软隔离
  vGPU：硬件级虚拟化，强隔离
  容器级：限制显存/算力（如 --gpus + 显存限制）
```
- 强隔离（互不影响）用 vGPU，软隔离（更高利用率）用 MPS+限制。
- 防"一个模型占满显存，别的起不来"。

### 3. 慢请求/长尾隔离
```
快请求和慢请求分队列：
  快队列（实时交互）→ 优先调度
  慢队列（批处理/复杂分析）→ 后台跑
防长尾拖垮 P99。
```

## 四、全链路压测（找真实底线）

### 压测原则
```
1. 模拟真实负载：
   - LLM 真实耗时（2-10s，或 mock 固定延迟+流式）
   - 真实请求分布（简单/复杂/长短）
   - 真实并发模式（突发/持续）
2. 全链路：网关→路由→检索→推理→返回，不只压一段
3. 找瓶颈和底线：单实例 QPS、扩容拐点、雪崩阈值
```

### 关键指标
| 指标 | 含义 |
|------|------|
| QPS | 每秒请求 |
| P50/P95/P99 延迟 | 体感（P99 看长尾） |
| GPU 利用率/显存 | 资源水位 |
| 队列堆积 | 削峰能力 |
| 错误率/限流命中 | 稳定性边界 |
| 单位成本 token/$ | 经济性 |

### 压测误区
- 用 mock 秒回 → 高估 QPS 几十倍（真实 LLM 慢）。
- 只压单接口 → 忽略链路瓶颈（如检索慢）。
- 不压长尾 → 上线被慢请求拖垮。

## 五、容量规划

```
常态 QPS → 基础资源
峰值 QPS → 弹性扩容
单实例容量（压测得）→ 实例数 = 峰值/单实例
冗余：N+1（挂一个不影响）
```
- 结合 HPA 自动扩缩（见并发优化题016）。

## 六、底层本质

负载均衡+隔离+压测本质是**"让异构资源公平高效利用、故障不扩散、容量心中有数"**。均衡抗流量，隔离防干扰，压测找底线。

**AI 系统不压测上线 = 盲人开车** —— GPU/模型/租户的异构性，使得真实容量必须靠全链路压测摸清，不能拍脑袋。

## 常见考点

1. **怎么压测 AI 系统？**——用真实模型或 mock 固定延迟（流式）模拟 LLM 耗时，全链路压，重点看 P99/队列堆积/GPU 水位/雪崩点，而非纯 QPS。
2. **多租户怎么公平？**——配额+限流+优先级+资源池隔离；大租户独占，小租户共享按配额，防互相饿死。
3. **长尾慢请求怎么办？**——隔离到慢队列 + 超时熔断 + 降级，保护快请求的 P99；分析长尾原因（复杂查询/大 prompt）针对性优化。


## 结构化回答

**30 秒电梯演讲：** 聊到负载均衡/隔离/压测怎么做（GPU/多租户/全链路），我的理解是——AI 系统的负载均衡与隔离是'把流量均匀分发、把故障和噪声隔开、用压测摸清底线'——负载均衡抗流量，多租户/GPU 隔离防互相影响，全链路压测找真实瓶颈。打个比方，像大型交通系统——红绿灯和路口调度均衡车流（负载均衡），公交道和私家车道分开不互相堵（隔离），开通前要压力测试能跑多少车（压测），才知道系统真实承载。

**展开框架：**
1. **负载均衡** — 网关层（请求分发）+ 推理层（GPU 调度）多级均衡
2. **多租户隔离** — 数据/资源/配额隔离，防互相干扰
3. **GPU 隔离** — 显存/算力/时间片隔离，防一个模型/租户占满

# 【生物医药 AI】AI 产品的高并发后端怎么做（异步/调度/资源池）？

> JD 依据："RESTful API、异步编程、高并发系统设计；系统性能优化：并发、缓存、异步任务、资源调度。"

## 一、为什么 AI 后端必须异步

```
传统 CRUD：请求 10ms 内返回，一进程几十线程够用
AI 请求：LLM 调用 1-10 秒
  → 同步模型：1000 并发 = 1000 线程都阻塞在等 LLM
  → 线程耗尽、内存爆炸、吞吐崩塌
```

AI 后端的瓶颈是**等待**，不是计算。异步让"等待"不再占用昂贵线程。

## 二、异步三层架构

```
[API 网关 / FastAPI async]
      ↓ 提交
[任务队列 / Celery + Redis]  ← 解耦、削峰、重试
      ↓ 消费
[Worker 池 + 资源池]         ← GPU/连接/模型复用
```

### 1. 异步 IO 层（API 入口）
```python
@app.post("/ask")
async def ask(req):
    # 不阻塞事件循环，LLM 调用用 async client
    result = await async_llm_client.generate(req.prompt)
    return result
```
- FastAPI/asyncio：事件循环少量线程处理海量并发 IO。
- 关键：LLM 客户端必须是 async，否则 await 仍阻塞。

### 2. 任务队列层（耗时任务）
- 长 LLM 任务、批量文档解析、训练任务 → 入队异步跑。
- 前端轮询/WebSocket/SSE 拿结果。
- 解耦：API 立刻返回 task_id，worker 慢慢消费，削峰填谷。
- 支持重试、优先级、定时（Celery beat）。

### 3. 资源池层（昂贵资源复用）
- **连接池**：DB/Redis/向量库连接复用。
- **GPU 推理池**：多请求共享 GPU，批处理提升利用率。
- **模型实例池**：常驻模型避免重复加载。

## 三、协程 vs 线程 vs 进程

| 模型 | 适用 | 说明 |
|------|------|------|
| 协程（asyncio/goroutine） | IO 密集 | 等待时不占线程，撑海量并发 |
| 线程 | IO 密集（阻塞库） | 比进程轻，但有 GIL（Python） |
| 进程 | CPU/GPU 密集 | 绕开 GIL，独立内存 |

**AI 后端组合**：API 层协程（asyncio）+ 推理层进程/池（GPU）+ 队列连接两者。

## 四、资源调度（GPU 是稀缺资源）

```
请求 → [推理队列] → [批处理器] → [GPU 池]
          ↑优先级      ↑continuous batching
```
- **批处理**：攒多个请求一次推理（vLLM continuous batching），吞吐倍增。
- **优先级**：实时请求 > 后台任务，避免长任务饿死交互。
- **多模型分时复用**：一张 GPU 跑多模型（显存够时），或卸载冷模型。
- **限流**：GPU 队列长度上限，超限拒绝/降级。

## 五、背压与稳定性

- **背压**：worker 满了，队列不再接收，向上游反压（HTTP 429），防止雪崩。
- **超时与重试**：LLM 调用设超时，失败指数退避重试，不无限挂起。
- **熔断**：下游故障率超阈值，熔断直接降级，保护系统。

## 六、底层本质

高并发 AI 后端本质是**"把等待从昂贵资源里剥离"**。异步 IO 让线程不阻塞，任务队列削峰解耦，资源池复用 GPU/连接，背压限流防雪崩。

**核心认知转变：AI 后端的瓶颈是 IO 等待而非计算，所以解法是异步化+池化，而非堆机器**。

## 常见考点

1. **为什么 Python 有 GIL 还能做高并发？**——IO 密集场景用 asyncio（GIL 在 await 时释放），瓶颈在网络不在 CPU；CPU/GPU 密集用多进程绕开。
2. **任务队列怎么保证不丢？**——消息持久化（RabbitMQ 持久队列/Redis AOF）+ 消费确认（ack）+ 幂等执行。
3. **怎么压测 AI 后端？**——重点测 P99 延迟和吞吐，模拟 LLM 真实耗时（mock 固定延迟），观察队列堆积和资源池水位。

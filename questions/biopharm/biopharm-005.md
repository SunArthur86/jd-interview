# 【生物医药 AI】多模型路由怎么做（场景/成本/fallback）？

> JD 依据："AI 产品前后端开发：多模型路由、缓存；系统性能与成本控制。"

## 一、为什么需要路由

```
全用大模型：质量好但慢且贵（一个 query 几毛钱，扛不住量）
全用小模型：便宜快但复杂问题答不好
→ 路由：按场景动态选，帕累托最优
```

## 二、三维度权衡

| 维度 | 大模型 | 小模型 |
|------|--------|--------|
| 质量 | 强推理/复杂 | 简单任务够用 |
| 延迟 | 慢（秒级） | 快（百毫秒） |
| 成本 | 贵（差几十倍） | 便宜 |

任务难度是长尾分布：80% 是简单问题，20% 才需大模型。路由把这 20% 识别出来。

## 三、场景路由（核心）

```python
def route(query, context):
    scene = classify_scene(query)      # 意图：闲聊/检索/分析/合规
    complexity = estimate_complexity(query, context)  # 简单/中/复杂

    if scene == "闲聊" or complexity == "简单":
        return "qwen-turbo"            # 便宜快
    if complexity == "复杂" or scene == "合规分析":
        return "claude-sonnet"         # 强推理
    if scene == "结构化抽取":
        return "local-small"           # 本地小模型，省成本
    return "gpt-4o-mini"               # 默认均衡
```

**判定要轻**：
- 规则（关键词/标签）最快；
- 轻量分类器（小 BERT/小 LLM）次之；
- 不能用大模型来判定路由（路由比推理还贵就亏了）。

## 四、fallback 链（保可用）

```
请求 → 主供应商（claude）
        ├ 超时/限流/5xx → 备供应商（gpt）
        ├ 仍失败 → 自部署开源模型
        └ 都不行 → 规则兜底回复 + 告警
```

- 网关层做健康检查 + 自动切换，对 Agent 透明。
- 流式响应可断点续接（切供应商后重生成）。

## 五、成本控制

| 手段 | 效果 |
|------|------|
| 小模型前置 | 80% 简单问题省 90% 成本 |
| 结果缓存 | 相同/相似 query 命中缓存（语义缓存） |
| token 预算 | 每租户/会话设上限，超限降级 |
| prompt 压缩 | 压缩上下文减少输入 token |
| 批处理 | 非实时任务合并请求享批价 |

**监控**：按租户/场景/模型统计 token 与费用，设预算告警，超限自动降级。

## 六、路由判错的兜底

- 简单问题用了大模型 → 浪费成本（可接受）。
- 复杂问题用了小模型 → 质量差（用户不满）。
- **防御**：小模型回答时附"低置信度"信号 → 触发升级到大模型重答。

## 七、底层本质

多模型路由本质是**"用工程手段在模型异构性中找最优匹配"**。把"一个模型扛所有"换成"按需匹配"，在质量、延迟、成本三角中实现帕累托最优。

**这是 AI 产品成本可控的关键工程** —— 没有路由，规模化 AI 服务的成本会失控。

## 常见考点

1. **路由判定会不会拖慢响应？**——判定用轻量分类器（ms 级），远小于推理耗时；判定与首次请求可并行（乐观先用小模型）。
2. **语义缓存怎么做？**——query embedding 找相似历史问答，相似度超阈值直接返回（带校验），需注意缓存失效（知识库更新要清）。
3. **怎么平衡质量和成本？**——A/B 实验测各场景不同模型的质量-成本曲线，找拐点；动态调整路由策略。

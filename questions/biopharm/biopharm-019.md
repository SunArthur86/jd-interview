# 【生物医药 AI】预案/回滚怎么做（含钱学森工程理论）？

> JD 依据："高可用/高可靠；云原生；系统性能优化。"

## 一、为什么需要预案和回滚

```
无预案：
  凌晨系统挂 → 全员被叫醒 → 临时排查 → 手忙脚乱 → 故障扩大
  上线出 bug → 全量用户受影响 → 无法快速恢复 → 事故

有预案有回滚：
  故障按 SOP 从容处置 → 5 分钟回滚恢复 → 影响最小
```

## 二、预案体系

### 1. 故障分级
```
P0：核心功能全挂（如 AI 服务不可用）→ 立即响应
P1：部分功能/部分租户受影响
P2：体验下降/非核心异常
```

### 2. SOP（标准处置流程）
每个故障有对应 SOP：
- 症状识别（告警/指标）
- 止损动作（回滚/降级/切流）
- 排查步骤（日志/trace）
- 恢复验证
- 复盘

### 3. 值班与 on-call
- 7x24 值班 + 升级机制（一线→二线→主管）。
- 告警分级通知（P0 电话，P2 工单）。

### 4. 演练（混沌工程）
- 定期注入故障（杀实例/断网/模型挂），验证预案有效。
- 不演练的预案是纸上谈兵。

## 三、回滚（快速恢复）

### 四类回滚
| 类型 | 方法 | 速度 |
|------|------|------|
| 版本回滚 | 上一版本镜像/二进制 | 秒~分钟 |
| 配置回滚 | 配置中心版本回退 | 秒级 |
| 灰度回滚 | 关闭灰度流量 | 秒级 |
| 数据回滚 | DB 快照/时间点恢复 | 分钟~小时 |

### 关键设计
- **无状态服务**：任意时刻可换版本，秒级回滚。
- **向前兼容**：DB schema 向前兼容，回滚旧版本仍能读新数据。
- **feature flag**：新功能藏 flag 后，运行时开关，秒级回滚无需重部署。
- **危险操作前备份**：大批量数据变更前先快照。

### 回滚解决不了的故障
- 数据已脏（被错误写入污染）→ 需数据修复。
- 下游依赖挂（如 LLM 供应商）→ 需 fallback/降级。
- 配置被外部改坏 → 需追查。
→ 回滚是主要手段但非万能，还需降级/数据修复配合。

## 四、灰度发布（控影响）

```
新版本发布流程：
  ① 内部测试（金丝雀 1%）
  ② 小流量灰度（5% → 20% → 50%）
  ③ 全量
每步观察指标（错误率/延迟/质量），异常即停/回滚。
```
- 让问题在小范围暴露，影响最小。
- 结合 A/B 测质量（AI 生成质量不能只看技术指标）。

## 五、钱学森工程理论：四标尺

评估一个工程方案（含预案/回滚/架构）要四维都站得住：

| 标尺 | 含义 | AI 场景举例 |
|------|------|-------------|
| 技术可行 | 能不能做出来 | 自部署 70B 要多少 GPU，显存够不够 |
| 经济合理 | 值不值得做 | 自部署 vs API 的成本拐点，ROI 正不正 |
| 周期短 | 多久能落地 | 自研引擎 3 个月 vs 用 LangGraph 2 周 |
| 协调运转 | 各部分能否配合 | 网关/路由/推理/监控各组件能否协同 |

**只看技术可行会做出'能跑但贵/慢/难协调'的方案**。成熟工程师四维权衡。

## 六、变更管理

```
变更前：评审（影响/预案/回滚）+ 审批
变更中：灰度 + 可观测 + feature flag
变更后：验证 + 监控 + 复盘
```
- 每次变更有对应回滚预案。
- 高危变更在低峰期、有人值守。

## 七、底层本质

预案回滚本质是**"把故障应急从即兴发挥变成有备而来"**。预案（分级+SOP+演练）保证从容，回滚（四类）保证快速恢复，灰度（控影响）保证风险可控，钱学森四标尺保证方案靠谱。

**这是'高可用'的最后一道防线** —— 故障一定会来，区别在于'有预案从容 5 分钟恢复' vs '无预案手忙脚乱 5 小时'。

## 常见考点

1. **回滚和降级区别？**——回滚是"退回上一版本/状态"，降级是"故障时牺牲非核心保核心"；回滚针对变更引发的问题，降级针对运行时故障，常配合用。
2. **怎么保证回滚可行？**——无状态服务、向前兼容 schema、feature flag、变更前备份、定期演练回滚流程（避免紧急时才发现回不了）。
3. **钱学森四标尺怎么用？**——每个方案四维打分，任一维度不达标就重设计；如自部署大模型技术可行但经济不合理（量不够）就该用 API。

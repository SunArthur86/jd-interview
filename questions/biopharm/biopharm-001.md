# 【生物医药 AI】企业级 AI Agent 架构怎么做到高性能/高可用/高可靠？

> JD 依据："负责 AI Agent 系统架构设计、开发与优化（高性能/高可用/高可靠）；负责企业级 AI Agent 产品的架构设计、研发落地。"

## 一、Agent 的本质：从"调 API"到"工程系统"

裸调 LLM：
```
user_input → LLM → response
（无规划、无工具、无校验、无恢复、无审计）
```

企业级 Agent：
```
user_input
   ↓
[1] 意图理解 + 规划（Plan）：拆解任务，决定调用哪些工具/知识
   ↓
[2] 执行（Tool/Retrieval）：并行调工具、检索 RAG 知识库
   ↓
[3] 生成（Generate）：基于工具结果 + 知识，LLM 生成答案
   ↓
[4] 校验（Verify）：事实核查 / 合规校验 / 引用回溯
   ↓
[5] 返回 + 状态持久化（审计/断点恢复）
```

每一环都是对裸 LLM 的"工程化补丁"。

## 二、三大目标的拆解

| 目标 | 含义 | 关键手段 |
|------|------|----------|
| 高性能 | 低延迟、高吞吐 | 异步并发、流式、缓存、小模型路由、批处理 |
| 高可用 | 服务不中断 | fallback 链、多模型多供应商、熔断降级、容灾 |
| 高可靠 | 结果正确可追溯 | 生成-校验-重试闭环、引用回溯、审计日志、人工兜底 |

## 三、可靠性核心：生成-校验-重试闭环

```python
def reliable_agent_answer(query, max_retry=3):
    for attempt in range(max_retry):
        plan = planner(query)                  # 规划
        evidence = execute_tools(plan)         # 调工具/检索
        answer = llm.generate(query, evidence) # 生成
        report = verifier(answer, evidence)    # 校验：事实/引用/合规
        if report.passed:
            return answer, report.citations
        # 失败 → 带反馈重试（self-refine）
        query = f"{query}\n上次问题：{report.issues}"
    return fallback_human_review(answer)       # 兜底：人工
```

**医药场景的特殊校验**：引用必须可回溯到具体文档段落（合规要求），不能"编造文献"。

## 四、编排引擎：DAG + 状态机 + 人工节点

```
[意图分类] → [检索知识] ──┐
                         ├→ [生成] → [校验] → [人工审批] → [返回]
[调外部工具] ────────────┘
```

- **DAG**：定义节点依赖，支持并行（检索和工具同时跑）。
- **状态机**：每步状态持久化，进程挂了能从断点恢复（长任务必备）。
- **人工审批节点**：医药合规场景，关键决策（如用药建议）必须人工二次确认后才输出。

## 五、高可用：fallback 与降级

```
主模型（claude/gpt）挂 → 备供应商（通义/文心）→ 自部署开源模型 → 规则兜底回复
```

- 网关层做健康检查 + 自动切换，对上层 Agent 透明。
- 降级策略：复杂任务降级为"只检索不生成"或"返回草稿+提示人工确认"。

## 六、底层本质

企业级 Agent 本质是**"用工程约束把概率性模型能力变成确定性业务服务"**。模型决定能力上限，编排工程决定可靠性下限。90% 的工作量在编排、校验、监控、容灾，而非"调通一个模型"。

**这是"AI 能力"到"企业产品"的鸿沟跨越**——医药等强合规领域尤其看重可追溯、可审计、可兜底。

## 常见考点

1. **Agent 怎么处理幻觉？**——RAG 提供可引用证据 + 生成后事实校验 + 引用回溯 + 关键场景人工兜底。
2. **长任务怎么断点恢复？**——状态机持久化每步状态（DB），重启后从最近 checkpoint 续跑。
3. **怎么控制 Agent 成本？**——场景路由（简单问题用小模型）+ 结果缓存 + token 预算 + 工具调用上限。


## 结构化回答

**30 秒电梯演讲：** 聊到企业级 AI Agent 架构怎么做到高性能/高可用/高可靠，我的理解是——企业级 AI Agent 是把 LLM 的'单次生成'升级为'可规划、可调工具、可校验、可回滚、可审计'的工程系统——用编排引擎把无状态模型变成可信赖的业务服务。打个比方，像工厂流水线——LLM 是核心机床，但还要有调度员（规划）、物料员（工具/知识）、质检员（校验）、安全员（合规），才能稳定产出合格产品。

**展开框架：**
1. **三大支柱** — 高性能（低延迟/高吞吐）、高可用（容灾/fallback）、高可靠（结果可校验/可回滚）
2. **Agent = 规划（Plan）** — Agent = 规划（Plan）+ 工具（Tool）+ 记忆（Memory）+ 校验（Verify）+ 编排（Orchestrate）
3. **编排引擎** — DAG/状态机驱动，支持人工审批节点和长任务

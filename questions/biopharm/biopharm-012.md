# 【生物医药 AI】RAG 知识库平台架构怎么设计（多租户/权限/规模）？

> JD 依据："AI 产品前后端开发：知识库、向量数据库；企业级 AI Agent 产品的架构设计。"

## 一、从单库 RAG 到 RAG 平台

```
单库 RAG（demo 级）：
  一个文档库 → 切分 → 向量库 → 检索 → 生成
  （单租户、无权限、小规模、无治理）

RAG 平台（企业级）：
  多租户 × 多知识库 × 海量文档 × 权限治理 × 规模化索引 × 审计
```

## 二、分层架构

```
┌──────────────────────────────────────────┐
│  服务层    检索 API / 引用回溯 / 审计     │
├──────────────────────────────────────────┤
│  检索层    多路召回 → 融合 → 重排 → 过滤   │
├──────────────────────────────────────────┤
│  索引层    解析→切分→Embedding→入库（异步）│
├──────────────────────────────────────────┤
│  接入层    文档治理（生命周期/版本/权限）   │
├──────────────────────────────────────────┤
│  存储      向量库（Milvus/PGV）+ 业务库    │
└──────────────────────────────────────────┘
横切：多租户 / 权限 / 计量 / 监控
```

### 1. 接入层（文档治理）
- 文档生命周期（录入→审核→发布→下线）、版本、元数据（见008）。
- 权限模型：租户/部门/角色/敏感级。

### 2. 索引层（规模化建库）
```
文档变更 → 异步队列 → 解析 → 切分 → Embedding → 写向量库
                      （独立 worker 池，不影响在线检索）
```
- 增量更新（chunk 哈希对比，只重建变化块）。
- 读写分离：建库写副本，检索读主。

### 3. 检索层（多路+重排+过滤）
- 多路召回（向量+BM25+图谱）→ RRF 融合 → Cross-Encoder 重排。
- **权限预过滤**：检索前按 tenant/kb_id/role/sensitivity 过滤。

### 4. 服务层
- 检索 API、引用回溯（每个结果带 doc_id+version+段落）、审计日志。

## 三、多租户隔离（核心）

```
租户维度：
  共享集群 + tenant_id 贯穿 → 成本低，多数够用
  独立集合/库 → 强隔离，敏感数据
  独立集群 → 最高隔离，超大租户

权限落到 chunk 元数据：
  {tenant_id, kb_id, dept, role, sensitivity}
检索：WHERE tenant=? AND kb IN (?) AND role 可见 AND sensitivity 允许
```
- **预过滤**（检索前）优于后过滤（检索后可能不足 k 个）。
- 医药患者数据：核心物理隔离 + 通用知识逻辑隔离。

## 四、多知识库与路由

```
一个租户有多个库：
  药品说明书库 / 临床文献库 / 法规库 / 内部 SOP 库

用户提问 → 意图分类 → 路由到对应库（或跨库检索后融合）
"某药副作用"  → 药品库 + 文献库
"提交注册流程" → SOP 库
```
- 场景路由提升精度（避免跨库噪声）。
- 跨库检索用融合排序（加权或 RR)。

## 五、规模化挑战与解法

| 挑战 | 解法 |
|------|------|
| 文档海量（亿级 chunk） | 分布式向量库（Milvus）、分片、量化压缩 |
| 持续大量更新 | 异步建库流水线、增量索引、读写分离 |
| 高 QPS 检索 | 副本、缓存（热点 query）、HNSW 调参 |
| 多租户公平 | 按租户限流、配额、优先级调度 |

## 六、引用与审计（合规）

- 每个检索结果带 `doc_id + version + chunk + 来源URL`。
- 生成答案标注引用编号，可溯源。
- 检索和生成全程审计（谁查了什么、引用了什么版本）。

## 七、底层本质

RAG 平台本质是**"把单库检索工程化为多租户、可治理、规模化的知识服务"**。分层解耦（接入/索引/检索/服务），权限贯穿，多知识库路由，异步增量建库支撑规模化。

**单库 RAG 是'能不能检索'，RAG 平台是'能不能规模化、安全、可治理地服务多租户'** —— 这是企业级的核心差异。

## 常见考点

1. **怎么保证租户数据不串？**——chunk 元数据强制带 tenant_id，检索预过滤，写入时服务端注入（不信客户端传），加自动化测试验证隔离。
2. **亿级向量怎么检索快？**——分布式向量库分片 + HNSW/IVF 索引调参 + 副本 + 热点缓存 + 预过滤缩小候选集。
3. **知识库更新后多久能检索到？**——增量索引异步建库，秒~分钟级；关键变更可触发即时重建+缓存失效。


## 结构化回答

**30 秒电梯演讲：** 聊到RAG 知识库平台架构怎么设计（多租户/权限/规模），我的理解是——RAG 知识库平台是把'单库 RAG'升级为'多租户、可治理、可规模化的知识服务平台'——接入/索引/检索/治理分层，权限贯穿，让每个租户的安全知识都能被精准检索。打个比方，像大型连锁图书馆——不是一家书店一个库，而是总部统管（平台）、各分馆隔离（租户）、每本书有权限标签（谁能看）、统一索引但分馆互不可见，还要支持千万本书和百万读者。

**展开框架：**
1. **分层** — 接入（文档治理）→ 索引（切分/Embedding/入库）→ 检索（多路/重排）→ 服务（API/引用）
2. **多租户** — 知识库按租户隔离，权限贯穿元数据到检索
3. **多知识库** — 一个租户可有多库（药品库/文献库），按场景路由

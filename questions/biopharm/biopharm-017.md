# 【生物医药 AI】LLM 网关怎么保证稳定性（限流/降级/超时/重试）？

> JD 依据："系统性能优化；高并发系统设计；高性能/高可用/高可靠。"

## 一、LLM 网关的稳定性风险

```
LLM 调用慢（秒级）→ 请求堆积 → 线程/连接耗尽 → 雪崩
模型 API 挂/限流 → 请求失败 → 重试放大 → 打爆下游
一个租户狂调 → 占光资源 → 其他租户饿死
```
稳定性 = 故障不扩散、核心可用、优雅降级。

## 二、四件套 + 熔断

### 1. 限流（防过载）
```
多级限流：
  租户级：每分钟 N 次（令牌桶）
  模型级：GPU 并发上限
  全局级：总 QPS 上限 + 熔断
超限 → 429 Too Many Requests + Retry-After
```
- 保证多租户公平，防个别租户打爆全局。
- 算法：令牌桶（允许突发）/ 漏桶（匀速）/ 滑动窗口（精准）。

### 2. 超时（不拖死）
```
分层超时：
  连接超时：1-3s（建连慢即失败）
  读取超时：按模型预期（流式首字 5s，整体 60s）
  全链路超时：端到端上限（如 90s）
超时 → 当失败处理（重试/降级）
```
- 避免请求无限等待耗尽资源。
- 流式：首字超时 vs 整体超时分开设。

### 3. 重试（抗抖动）
```python
def call_with_retry(req, max_retry=3):
    for i in range(max_retry):
        try:
            return call(req)
        except TransientError:
            sleep(2**i + random_jitter())  # 指数退避+抖动
    raise
```
- **指数退避**：1s, 2s, 4s...（给下游恢复时间）。
- **随机抖动**：避免重试同步集中（重试风暴）。
- **幂等**：重试不能重复扣费/重复执行（带幂等键）。
- **重试上限**：超限降级，不无限重试。

### 4. 降级（保核心）
```
故障时按重要性降级：
  核心（如用药咨询）→ 降级到小模型/缓存/规则兜底
  非核心（如报表美化）→ 直接关闭/延后
  实时交互 → 返回"稍后重试"+排队
```
- 业务降级开关（feature flag）随时可关非核心。
- 预案：哪些可降级、降级到什么、谁决策。

### 5. 熔断（隔离故障）
```
下游错误率/慢调用率超阈值（如 50%）→ 熔断（直接拒绝/走降级）
持续 N 秒后半开试探 → 恢复正常或继续熔断
```
- 防止持续打挂的下游被反复冲击。
- 半开探测避免抖动。

## 三、协同流程

```
请求进来
  ↓
[限流] 超限 → 429
  ↓
[调用 LLM]
  ├ 超时 → 重试（退避）
  ├ 失败 → 重试达上限 → 熔断判断
  │        ├ 错误率高 → 熔断 + 降级
  │        └ 偶发 → fallback 备模型
  └ 成功 → 返回
```

## 四、fallback 与多供应商

```
主供应商（Claude）超时/限流 → 备（GPT）→ 自部署 → 规则兜底
网关自动切换，对业务透明
```
- 多供应商 + 自部署混合，单点故障不阻断（见推理平台题014）。

## 五、稳定性预案

- **容量规划**：常态/峰值多少 QPS，资源够不够。
- **降级预案**：每个场景的降级策略和触发条件。
- **演练**：混沌工程（故意注入故障）验证稳定性。

## 六、底层本质

LLM 网关稳定性本质是**"用限流/降级/超时/重试/熔断，把故障控制在局部，让系统优雅降级而非雪崩"**。限流防过载，超时不拖死，重试抗抖动，降级保核心，熔断隔离故障。

**这是 AI 产品'高可用'的工程底线** —— 模型会挂会慢，但产品不能崩，靠的就是这套防护。

## 常见考点

1. **限流和降级冲突吗？**——不冲突，限流是"入口控制流量"，降级是"故障时牺牲非核心保核心"；先限流挡住洪峰，挡不住的故障再降级。
2. **重试会不会加重故障？**——会（重试风暴），必须退避+抖动+上限+幂等+熔断，必要时禁重试改降级。
3. **怎么决定超时阈值？**——基于历史 P99/P999 + 业务容忍；流式按首字和整体分设；超时应短于用户耐心。

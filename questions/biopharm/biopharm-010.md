# 【生物医药 AI】AI 系统的性能与成本怎么控制？

> JD 依据："系统性能优化：并发、数据库、缓存、异步任务、资源调度、成本控制。"

## 一、成本公式

```
LLM 总成本 = Σ (token 量 × 单价) × 调用次数
```
三个杠杆：**降 token 量、降单价、降次数**。

## 二、降单价：模型路由

```
简单问题（80%）→ 小模型（qwen-turbo，单价低几十倍）
复杂问题（20%）→ 大模型（claude/gpt）
长尾超难 → 最强模型
```
- 小模型前置 + 置信度判断 + 必要时升级大模型。
- 见模型路由题（005），80% 简单问题省 90% 成本。

## 三、降次数：缓存复用

### 1. 精确缓存
```
相同 query + 相同上下文 → 直接返回历史答案
```

### 2. 语义缓存（关键）
```
query embedding → 找相似历史问答 → 相似度 > 阈值 → 返回
"阿司匹林的作用" ≈ "阿司匹林有什么功效" → 复用答案
```
- 用向量检索找相似问，命中率高。
- **注意失效**：知识库更新要清相关缓存（避免返回过时答案）。

### 3. 中间结果缓存
- 检索结果、工具调用结果、embedding 结果都缓存复用。

## 四、降 token 量：prompt 压缩

| 手段 | 效果 |
|------|------|
| 历史摘要 | 长对话压成摘要，不塞全部原文 |
| 检索精炼 | 只塞 top-3 最相关 chunk，非 top-20 |
| 去冗余 | 精简系统指令、去重复模板 |
| 结构紧凑 | JSON 用最短格式、表格转紧凑列表 |
| 上下文窗口管理 | 滑动窗口 + 摘要，不超长 |

**举例**：把 10 万字说明书塞进去 = 浪费且注意力衰减；RAG 只塞 3 个相关片段，省 99% token 还更准。

## 五、提利用率：批处理与并发

```
单请求单推理：GPU 利用率 20%
批处理（vLLM continuous batching）：攒多请求一次推理，利用率 80%+，吞吐 4 倍
```
- 推理服务用批处理（vLLM/TGI）。
- 非实时任务（批量解析/训练）用离线批，享更低单价。
- 异步并发榨干 GPU。

## 六、自部署 vs API（成本拐点）

```
调用量小 → API（按量付费，无固定成本）
调用量大且任务固定 → 自部署开源模型（7B/14B）
  单 token 成本 ≈ GPU折旧 ÷ 吞吐，远低于 API
  但要算：GPU 租金 + 运维 + 部署成本
```
- 算清**单 token 总成本（TCO）**，找拐点。

## 七、计量与监控（省钱前提）

```
按 [租户][场景][模型] 统计：
  - token 消耗（输入/输出）
  - 调用次数
  - 费用
  - 缓存命中率
  - 平均 token/请求
```
- 设**预算告警**（租户/模型超阈值）。
- 设**配额**（超限降级或限频）。
- 定期分析成本分布，找优化点（如某场景 token 异常高）。

## 八、实战组合拳

```
1. 监控：先搞清钱花在哪（哪租户/哪场景/哪模型）
2. 路由：简单问题切小模型（最大头）
3. 缓存：高频问题语义缓存（第二大头）
4. 压缩：长上下文精简（第三大头）
5. 批处理：提升 GPU 利用率
6. 预算：设租户配额，超限降级
```

## 九、底层本质

成本控制本质是**"让每一分 token 花在刀刃上"**。降单价（路由）、降次数（缓存）、降 token（压缩）、提利用率（批处理），在不损害体验前提下把成本压到最低。

**AI 产品能不能盈利，往往不在模型多强，而在成本工程做得多细** —— 同样的功能，成本差 10 倍很常见。

## 常见考点

1. **缓存和体验怎么平衡？**——语义缓存设合理相似度阈值 + 知识更新触发失效 + 关键场景（医疗）不走缓存或加校验，避免过时/错误答案。
2. **怎么说服老板上自部署？**——算 TCO：当前 API 月费 vs 自部署（GPU+运维）月费，给出量级拐点（如月调用量 > X 时自部署更省）。
3. **token 优化会不会降质量？**——会，要平衡：压缩要保留关键信息（用 RAG 精炼而非粗暴截断），A/B 测质量，找质量-成本拐点。

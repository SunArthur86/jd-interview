# 【生物医药 AI】RAG 全链路怎么设计（解析到生成）？

> JD 依据："LLM 应用开发：Agent、RAG、Workflow、Tool Calling 核心能力建设；FAISS、PGVector、Milvus、Elasticsearch。"

## 一、RAG 的本质：临时查文档

```
离线（建库）：文档 → 解析 → 切分 → Embedding → 存入向量库
在线（问答）：query → Embedding → 检索 top-k → 重排 → 拼 prompt → LLM 生成 → 引用回溯
```

LLM 不再"凭记忆答"，而是"看着检索到的文档片段答"。

## 二、七步链路详解

### 1. 文档解析（解析是上限）
- PDF/Word/HTML/扫描件 → 结构化文本。
- **医药痛点**：药品说明书、临床试验报告大量表格、化学结构图，传统解析会打散。
- 方案：多模态解析（布局识别 + 表格还原 + OCR），保留 `标题层级`、`表格行列`、`图注`。

### 2. 切分（精度核心）
| 策略 | 特点 | 适用 |
|------|------|------|
| 固定长度（500 字） | 简单但割裂语义 | 通用兜底 |
| 按句子/段落 | 保留语义边界 | 一般文档 |
| 按标题层级（结构化） | 保留文档结构 | 说明书/法规 |
| 表格按行/整表 | 保留表格完整性 | 数据表 |

**切分原则**：一个 chunk 应是"语义自洽的最小单元"，并保留元数据（来源、页码、章节、版本）。

### 3. Embedding
- 选 embedding 模型（bge/m3e/通用商业模型），文档块 → 向量。
- 注意模型与 query 用同一套。

### 4. 入库（向量库）
- FAISS（轻量/单机）/ PGVector（与业务库一体）/ Milvus（大规模/分布式）/ ES（向量+全文混合）。
- 建索引：HNSW（高召回/内存大）或 IVF（可接受稍低召回/省内存）。

### 5. 检索
- query embedding → ANN 检索 top-k（如 20-50）。
- **混合检索**：向量（语义）+ BM25（关键词）+ 知识图谱（结构化关系）多路召回并融合。

### 6. 重排（精度下限）
- 用 Cross-Encoder（如 bge-reranker）对 top-k 精排，取 top-3~5。
- 双塔检索粗筛快但不够准，Cross-Encoder 慢但准，级联用。

### 7. 生成 + 引用回溯
- 把 top 片段拼进 prompt，让 LLM `仅基于以下资料回答，并标注引用`。
- 每个生成句要能溯源到 chunk（合规必需）。

## 三、医药场景的特殊性

- **文献解析**：PubMed 论文的结构（摘要/方法/结果）需保留，便于定位证据等级。
- **说明书/法规**：多版本管理（药品说明书会修订），检索要带版本过滤。
- **引用强度**：临床证据要标注来源类型（RCT / 综述 / 病例），影响结论可信度。

## 四、底层本质

RAG 本质是**"用检索替代记忆"**——把固定在模型权重里的知识，外置成可更新、可引用、可管控的文档库。每一步（解析/切分/检索/重排）都是在把控"召回率（能不能找到）"和"精度（找到的对不对）"的权衡。

**切分决定精度上限，重排决定精度下限，引用回溯是企业落地的合规底线。**

## 常见考点

1. **为什么不直接把整个文档塞给 LLM？**——context window 有限、长上下文注意力衰减（"中间遗忘"）、成本高；RAG 只塞最相关片段。
2. **混合检索怎么融合？**——RRF（倒数排名融合）/ 加权分数融合，把向量、BM25、图谱结果统一排序。
3. **怎么评估 RAG 效果？**——召回率（检索是否含正确答案）、精度（重排后排序）、答案准确率、引用命中率（RAGAS 等评测框架）。

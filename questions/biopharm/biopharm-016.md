# 【生物医药 AI】AI 产品的并发优化怎么做（池化/缓存/扩容/异步/队列）？

> JD 依据："系统性能优化：并发、缓存、异步任务、资源调度。"

## 一、AI 高并发的特殊性

```
传统后端瓶颈：CPU 计算
AI 后端瓶颈：稀缺资源（GPU/连接）的争抢和等待
  - LLM 调用慢（秒级）
  - GPU 贵且少
  - 模型加载慢（几分钟）
  - 请求有重复和峰值
```
优化重心从"算得快"转向"复用得好、等得少、削得平"。

## 二、五板斧

### 1. 池化（复用昂贵资源）
```
连接池：DB/Redis/向量库连接复用（避免建连开销）
GPU 池：多请求共享 GPU，continuous batching
模型实例池：常驻模型，避免重复加载（加载几分钟）
```
- 资源复用率↑，单请求成本↓。

### 2. 缓存（避免重复算）
```
结果缓存：相同 query+context → 直接返回
语义缓存：相似 query → 复用（query embedding 近邻）
中间缓存：embedding/检索结果/工具结果复用
```
- 见成本控制题（010），命中率 30-60%，省大量成本。
- 注意：知识库更新要清相关缓存。

### 3. 队列（削峰解耦）
```
请求 → [队列] → [worker 池] → 执行
        ↑削峰     ↑按消费能力
```
- 耗时任务（文档解析/批量推理）入队异步跑。
- 削峰：突发流量先进队列，worker 按速率消费，不打爆下游。
- 解耦：生产消费独立伸缩。

### 4. 异步（不阻塞）
```
协程（asyncio/goroutine）：等待时不占线程，撑海量并发 IO
API 流式（SSE）：首字 200ms，体感好
长任务异步 API：提交 task_id，轮询/Webhook 取结果
```
- 详见异步题（007）。

### 5. 扩容（应对峰值）
```
HPA：按 CPU/GPU利用率/QPS/队列长度 自动扩缩
预热池：常备几个实例，避免冷启动（模型加载慢）
预测扩容：按历史规律提前扩（如工作时间高峰）
```
- AI 扩容慢（模型加载），要预热+预测，不能纯靠响应式。

## 三、五板斧的协同

```
请求洪峰来了：
  ① 缓存先挡（30-60% 命中，直接返回）
  ② 剩下的进队列（削峰）
  ③ worker 池异步消费（不阻塞）
  ④ GPU 池批处理执行（复用）
  ⑤ 队列堆积触发扩容（弹性）
```
五板斧组合，层层削峰，把洪峰变成平稳流。

## 四、优先级（先做哪个）

```
效果/成本比排序（多数场景）：
  1. 缓存（语义缓存）→ 立竿见影省成本省算力
  2. 异步 → 解决线程耗尽
  3. 池化 → 提升资源利用率
  4. 队列 → 削峰解耦
  5. 扩容 → 最后的兜底（贵）
```
先优化（缓存/异步/池化），后扩容，避免"无脑加机器"。

## 五、监控指标

- QPS / P99 延迟 / 错误率
- 缓存命中率 / 队列堆积 / worker 利用率
- GPU 利用率 / 显存 / 批大小
- 扩容频率 / 冷启动时长

## 六、底层本质

并发优化本质是**"让稀缺资源利用率最大化"**。池化复用、缓存避免重复、队列削峰、异步不阻塞、扩容兜底，五板斧把洪峰削平，让有限资源抗住高并发。

**核心理念：先优化后扩容，先复用后新增** —— 无脑加机器是最贵的解法。

## 常见考点

1. **语义缓存会不会返回错答案？**——设合理相似度阈值 + 知识更新触发失效 + 关键场景（医疗）加校验或禁缓存；用 A/B 验证命中率与质量平衡。
2. **GPU 利用率怎么提？**——continuous batching（vLLM）+ KV Cache 复用前缀 + 多模型分时复用 + 量化省显存多装请求。
3. **扩容为什么跟不上突发？**——模型加载慢（分钟级）；解法：预热池常备实例、镜像预拉、模型常驻、预测性扩容。

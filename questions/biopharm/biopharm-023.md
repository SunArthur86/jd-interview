# 【生物医药 AI】RAG 怎么进阶（召回率/重排/多跳/HyDE）？

> JD 依据："LLM 应用开发：RAG；FAISS、PGVector、Milvus、Elasticsearch。"

## 一、基础 RAG 的瓶颈

```
基础：query → embedding → 向量检索 top-k → 生成
问题：
  - query 表达不准（口语/模糊）→ 检索偏
  - 单路向量召回不全 → 漏正确文档
  - 向量相似≠真正相关 → 噪声多
  - 复杂问题需多步 → 单次检索答不了
```

## 二、查询改写/扩展（提召回）

### 1. 同义改写
```
原 query："那个治高血压的白色药片"
改写："高血压 常用药物 白色 片剂" + "降压药 白色药片"
多版本并行检索，召回更全。
```

### 2. 子问题分解
```
"药物A和药物B联用的风险" →
  子1："药物A的副作用"
  子2："药物B的副作用"
  子3："A和B的相互作用"
分别检索再合并。
```

### 3. 领域词扩展
- 用户口语 → 标准医学词（"血压高" → "高血压/动脉高压"）。
- 加同义词/缩写扩展。

## 三、多路召回（兜底全）

```
[向量检索]  ─┐
[BM25 关键词] ─┤→ RRF 融合排序 → top-k
[知识图谱]   ─┤
[结构化查询] ─┘
```
- **向量**：语义相似（"作用"≈"功效"）。
- **BM25**：关键词精确（药品名/基因名）。
- **图谱**：结构化关系（药物-靶点-疾病）。
- **RRF（倒数排名融合）**：把各路结果按排名融合，比分数融合鲁棒。

## 四、重排（提精度）

```
多路召回 top-50（宽进，重召回率）
   ↓
Cross-Encoder（bge-reranker）精排
   ↓
top-3（严出，重精度）
```
- 双塔检索（粗）快但不够准，Cross-Encoder（精）慢但准。
- 级联：召回阶段宽进保全，重排阶段严出保准。

## 五、多跳检索（解复杂问题）

```
问题："药物A通过什么靶点治疗疾病B，有什么替代药？"

跳1：检索"疾病B" → 找到靶点 X
跳2：检索"靶点X的药物" → 找到药物A
跳3：检索"靶点X的其他药物" → 找到替代药
```
- 实现：ReAct/Plan-Solve（LLM 拆步→检索→基于结果再拆→再检索）。
- 或预先构建知识图谱，支持实体跳转查询。

## 六、HyDE（Hypothetical Document Embedding）

```
原 query："阿司匹林和抗凝药联用注意什么？"
   ↓
LLM 生成"假设答案"（可能不准但有答案的语义模式）：
  "阿司匹林与华法林等抗凝药联用会增加出血风险，需监测..."
   ↓
用假设答案的 embedding 检索（比原 query 更接近答案文档）
   ↓
召回真实文档 → 生成最终答案
```
- **原理**：答案和文档语义更近（query 偏问句，答案偏陈述句）。
- **代价**：多一次 LLM 生成（成本/延迟）；假设错会带偏。
- **适用**：query 模糊但答案模式明确的场景。

## 七、其他进阶

| 技术 | 作用 |
|------|------|
| Contextual Retrieval | 给 chunk 加上下文摘要，提升检索精度 |
| Parent-Child 分块 | 检索小块，返回大块（保留上下文） |
| 长上下文重排 | 长文档内多段重排（Lost in the Middle） |
| 自查询 | LLM 生成结构化过滤条件（元数据过滤） |

## 八、底层本质

RAG 进阶本质是**"从查询、召回、排序、推理全链路系统性提升找答案概率"**。查询改写让表达准，多路召回兜底全，重排提精度，多跳解复杂，HyDE 用答案模式反查。

**基础 RAG 是'能检索'，进阶 RAG 是'检索得好'** —— 召回率和精度的提升是 RAG 工程的核心战场。

## 常见考点

1. **HyDE 什么时候好用？**——query 模糊、答案模式明确（如事实问答）时，假设答案的语义比 query 更接近文档；查询已经很精确时没必要（多此一举）。
2. **多路召回怎么融合？**——RRF（倒数排名融合）最鲁棒（不依赖分数尺度）；加权分数融合需归一化；按场景选权重。
3. **重排模型怎么选？**——bge-reranker（中文好）/ cohere-rerank（商业强）/ 自训练；衡量：重排后 top-k 命中率提升。

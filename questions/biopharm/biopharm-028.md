# 【生物医药 AI】MCP 协议工程实践怎么做（注册/发现/安全/编排）？

> JD 依据："Harness、Skills、Workflow、Tool Calling、Function Calling、MCP。"

## 一、为什么需要 MCP 工程化

```
硬编码集成：
  Agent 代码里写死调 50 个工具 → 加一个改一次代码
  N 模型 × M 工具 = N×M 适配 → 不可持续

MCP 工程化：
  工具按标准暴露+注册 → 客户端动态发现 → N+M
  注册/发现/安全/编排/治理 体系化
```

## 二、MCP 协议回顾（见003）

```
[MCP client（模型/Agent/IDE）] ←─ 协议 ─→ [MCP server（工具）]
三类能力：Tools（执行）/ Resources（数据）/ Prompts（模板）
transport：stdio（本地）/ HTTP+SSE（远程）
```

## 三、注册中心（工具元数据治理）

```
注册中心管每个工具的元数据：
{
  tool_id, name, description,
  schema（参数/返回）,
  capability（读/写/高危）,
  permissions（需什么权限）,
  version, owner, status（active/deprecated）,
  endpoint（怎么连 MCP server）
}
```
- 工具方注册，平台审核，统一目录。
- 支持版本、灰度、退役（标记 deprecated）。

## 四、发现（动态）

```python
# 客户端不硬编码工具，运行时发现
available_tools = registry.discover(
    tenant=user.tenant,
    permissions=user.permissions,
    context=task.context
)
# 把可用工具描述注入 LLM，LLM 按需调用
```
- 按租户/权限/上下文动态发现可用工具。
- 新工具注册后，客户端自动可用，无需改代码。

## 五、安全（调用前多重防护）

```
LLM 决定调工具
   ↓
[1] 白名单：该租户/用户能否调此工具（注册中心权限匹配）
   ↓
[2] 参数校验：schema 校验 + 业务校验（值域/类型）
   ↓
[3] 权限控制：数据范围（如只能查本科室患者）
   ↓
[4] 高危确认：写操作/敏感数据 → 人工二次确认
   ↓
[5] 执行 + 审计日志（谁/何时/调了什么/参数/结果）
```
- 医药高危工具（改病历/发药）必须人工确认 + 全程审计。

## 六、编排（组合多工具）

```
单工具调用简单，复杂任务需组合：
  Agent / Workflow 编排引擎
    顺序：调A → 结果喂B → 调C
    并行：A、B 同时调，结果汇合给 C
    条件：A 结果满足X 才调 B
    循环：查不到换参数重查
→ 编排引擎管依赖/状态/恢复（见004/013）
```
- MCP 管单工具的标准接入，编排引擎管多工具的组合。

## 七、治理

| 维度 | 做法 |
|------|------|
| 版本 | 工具版本化，多版本并存 |
| 灰度 | 新版本小流量验证 |
| 弃用 | deprecated 标记 + 通知 + 下线周期 |
| 监控 | 调用频率/耗时/错误率/租户分布 |
| 配额 | 按租户/工具限流，防滥用 |
| 审计 | 全调用留痕，合规追溯 |

## 八、医药工具生态示例

```
医药 MCP server 池：
  - drug_db（药品查询/相互作用）
  - clinical_guideline（临床指南检索）
  - pubmed_search（文献检索）
  - patient_record（患者病历，高危）
  - lab_interpretation（检验解读）
  - regulatory（法规查询）

Agent 按场景组合：
  "用药咨询" → drug_db + clinical_guideline
  "病例分析" → patient_record（审批）+ lab + guideline
```

## 九、底层本质

MCP 工程实践本质是**"把工具生态标准化为可注册、可发现、可安全调用、可编排、可治理的应用生态"**。注册中心管元数据，动态发现省适配，多重防护保安全，编排引擎组合复杂任务，治理保证可持续。

**这是'每对接一个工具改一次代码'到'工具自服务生态'的跨越** —— MCP 让工具像 App Store 一样可扩展，是企业级 Agent 的基础设施。

## 常见考点

1. **MCP 和 OpenAPI/REST 区别？**——OpenAPI 描述 REST 接口，MCP 是面向 LLM 的工具暴露/发现/调用协议（含 Tools/Resources/Prompts + 动态发现 + 权限），更贴合 Agent 场景；可封装 REST 为 MCP。
2. **怎么管工具权限？**——工具声明所需权限/敏感级，注册中心管，客户端按租户/角色匹配，调用前校验，高危人工确认。
3. **工具编排谁来做？**——单工具 MCP 管，多工具组合由 Agent/Workflow 编排引擎（LangGraph/自研）管依赖/状态/恢复。


## 结构化回答

**30 秒电梯演讲：** 聊到MCP 协议工程实践怎么做（注册/发现/安全/编排），我的理解是——MCP 工程实践是'让工具像应用商店的 App 一样可注册、可发现、可安全调用、可编排'——注册中心管理工具元数据，客户端动态发现，权限校验保安全，编排引擎组合多工具。打个比方，像手机应用生态——开发者把 App 上架到商店（注册），用户搜索下载（发现），系统管权限（安全），快捷指令串联多个 App（编排）。MCP 把工具生态做成这样，不是每对模型×工具都定制线。

**展开框架：**
1. **注册** — 工具按 MCP 协议暴露，注册中心管元数据（schema/权限/版本）
2. **发现** — 客户端动态发现可用工具，无需硬编码
3. **安全** — 鉴权/参数校验/权限/审计/高危确认

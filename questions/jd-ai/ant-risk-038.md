# 【蚂蚁风控】设计一个智能风控平台

> JD 依据："数字化风控服务及智能风控平台"。这是 JD 的核心目标。

## 一、智能风控平台的目标

**业务目标**：
- 多业务接入（支付、借贷、营销、海外）
- 实时决策（< 200ms）
- 高准确率（少漏报少误杀）
- 快速迭代（运营自助）

**技术目标**：
- 组件化（规则/模型/Agent 可插拔）
- 平台化（统一编排、自助运营）
- AI 化（Agent 决策、自然语言配置）
- 可观测（全链路监控、效果评估）

## 三、决策编排层（核心）

**双引擎**：
- **DAG 引擎**：传统顺序+并行+短路（毫秒）
- **Agent 引擎**：AI 自主编排（秒级，边缘场景）

**编排策略**：
```
所有事件 →
  ├─ 80% 简单场景 → DAG 引擎（毫秒）
  └─ 20% 边缘场景 → Agent 引擎（秒级）

DAG 引擎:
  名单 → 特征 → 规则 → 模型 → 决策

Agent 引擎:
  LLM 理解 → 调工具（特征/规则/模型/关系）→ 推理 → 决策
```

**编排可视化**（运营拖拽）：
```
运营在控制台拖拽节点:
  [名单] → [特征] → [规则] → [模型] → [决策]
  连线表示依赖
  短路条件可配
```

## 四、能力层（各组件）

**1. 名单服务**：黑白名单 + 布隆过滤器（毫秒）
**2. 特征服务**：见 ant-risk-022（特征平台）
**3. 规则引擎**：见 ant-risk-012（Drools/QLExpress）
**4. 模型服务**：LR/XGBoost/DNN/LLM
**5. 关系网络**：见 ant-risk-023（JanusGraph）
**6. AI Agent**：见 ant-risk-031（决策 Agent）
**7. 案件库**：历史决策记录 + 人工审核结果
**8. 审核平台**：人工复核工作台

**组件统一接入**：每个组件用 MCP 包装成标准工具，决策编排调用。

## 五、运营层（自助化）

**核心目标**：运营自助，开发少介入。

**功能**：
1. **规则配置**：DSL 拖拽 / 自然语言
2. **模型管理**：版本、灰度、回滚
3. **决策编排**：拖拽画布
4. **效果看板**：实时监控
5. **案件审核**：人工工作台
6. **A/B 实验**：策略对比

**自然语言配置**（AI 加持）：
```
运营: "夜间大额转账到陌生账户要拦截"
   ↓ LLM 翻译
DSL: if amount > 5000 && time ∈ [22,6] && relation == new → REJECT
   ↓ 自动生效（灰度）
```

## 六、AI Agent 在平台中的角色

**1. 决策 Agent**：
- 边缘场景自主决策
- 调用其他组件做工具

**2. 运营助手 Agent**：
- 自然语言翻译成 DSL
- 推荐规则、解释策略

**3. 复核 Agent**：
- 自动复核决策（双验证）
- 生成解释

**4. 自学习 Agent**：
- 分析错判案例
- 推荐规则/模型改进

## 七、自助接入（多业务）

**业务自助接入流程**：
```
新业务接入（如海外支付）:
  1. 申请接入（控制台）
  2. 配置业务标识（业务隔离）
  3. 选择规则集（复用 / 新建）
  4. 配置决策编排（DAG 拖拽）
  5. 灰度发布（10% → 全量）
  6. 监控效果
```

**业务隔离**：
- 独立规则集
- 独立模型
- 共享特征平台
- 决策日志按业务分库

## 八、灰度发布与 A/B 实验

**策略灰度**：
```
新规则上线:
  - 影子模式（不打扰线上，只评估）
  - 灰度 10%（10% 流量用新规则）
  - 灰度 50%
  - 全量

监控指标:
  - 拦截率变化
  - 误杀率
  - 漏报率
  - 用户投诉
```

**A/B 实验**：
```
对照: 规则集 A
实验: 规则集 B

按 uid hash 分桶（50/50）
对比拦截率、误杀率、收益
显著差异（p < 0.05）才切换
```

## 九、效果评估闭环

**决策回流**：
```
决策 → 落库 → 人工审核 → 标注
        ↓
      分析（错判原因）
        ↓
      迭代（规则/模型/Agent 改进）
```

**核心指标**：
- **召回率**（漏报率，越低越好）
- **精准率**（误杀率，越低越好）
- **决策 RT**（P99，< 200ms）
- **运营自助率**（< 80% 算合格）
- **AI 决策占比**（趋势上升）

## 十、可观测性

**业务大盘**：
- QPS、拦截率、误杀率
- 各业务对比
- 规则命中 Top
- 模型分数分布

**系统大盘**：
- 决策链路 RT 分布
- 各组件健康度
- Agent 决策 trace

**告警**：
- 拦截率骤变 → 业务异常
- P99 RT > 500ms → 性能问题
- Agent 幻觉率 > 5% → AI 异常

## 十一、底层本质：从"系统集成"到"平台编排"

传统风控 vs 智能风控平台：

| 维度 | 传统风控 | 智能风控平台 |
|------|---------|------------|
| 架构 | 单体应用 | 平台化（组件 + 编排） |
| 决策 | 硬编码规则 | DAG + Agent 双引擎 |
| 运营 | 改代码上线 | 控制台自助 |
| AI | 单点（模型） | 全链路（Agent/特征/Harness） |
| 迭代 | 周/月 | 天 |
| 多业务 | 各做各的 | 共享平台 |

**核心转变**：从"做系统"到"做平台"。

**这是软件工程的"平台化"思想在风控的应用**：
- 业务方像 SaaS 用户
- 风控平台像 PaaS
- 各组件像微服务

## 十二、和 AI Harness / AI Infra 的关系

```
AI Infra（底层）
  ├─ 数据平台、训练平台、推理服务
  ↓
AI Harness（中层）
  ├─ LLM 网关、工具市场、评估闭环
  ↓
智能风控平台（上层）
  ├─ 决策编排、各业务组件、运营控制台
  ↓
风控 Agent / 业务应用
```

**层次化复用**：
- AI Infra 提供 AI 基础能力
- AI Harness 把 LLM 工程化
- 智能风控平台把风控业务化

## 十三、落地演进路径

**Stage 1：传统风控**（现状）
- 规则 + 模型
- 改代码上线
- 单业务

**Stage 2：平台化**
- 规则/模型可配置
- 多业务共享
- 运营部分自助

**Stage 3：AI 辅助**
- LLM 做解释、提示
- 边缘场景 Agent 决策
- 自然语言配置

**Stage 4：AI 原生**
- Agent 主导决策
- 规则做硬约束
- 全自助运营
- 自学习闭环

**风控的当前阶段**：Stage 2-3，向 Stage 4 演进。

## 常见考点
1. **智能风控平台的核心价值**？——平台化让组件协同 + 运营自助 + 快速迭代；AI 化让决策更智能。
2. **DAG 和 Agent 引擎怎么选**？——简单场景 DAG（毫秒）、边缘场景 Agent（秒），分层组合。
3. **怎么衡量平台成熟度**？——决策 RT、迭代周期、运营自助率、AI 决策占比、多业务接入数。

**代码示例**（决策编排 DAG）：
```java
DecisionFlow flow = DecisionFlow.builder()
    .start("whitelist")                          // 名单
    .thenParallel("feature", "profile")          // 并行查特征画像
    .then("rule_eval")                           // 规则
    .then("model_infer")                         // 模型
    .then("fusion")                              // 融合
    .onShortCircuit("whitelist", r -> r.isFinal())
    .timeout(Duration.ofMillis(200))
    .fallback(e -> RiskResult.basicPass())       // 全挂兜底
    .build();

RiskResult result = flow.execute(event);
```

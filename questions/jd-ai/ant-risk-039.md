# 【蚂蚁风控】FDE 在风控 AI 项目中做什么？怎么落地一个 AI 风控解决方案？

> JD 依据："FDE"是 AI 时代的复合角色。这是 AI 转型的高级方向。

## 一、FDE 是什么角色

**FDE（Field Development Engineer / Full-Stack Dev for AI）**：懂技术 + 懂业务 + 懂客户的复合角色。

**核心职责**：
1. **理解客户需求**：客户说"要降欺诈率"，FDE 翻译成"召回率提升 X%"
2. **设计技术方案**：基于需求选 AI 能力、设计架构
3. **推动落地**：协调技术团队、客户、监管
4. **负责效果**：交付后跟踪 ROI、迭代优化

**对比其他角色**：

| 角色 | 侧重 | FDE 区别 |
|------|------|---------|
| 产品经理 | 用户需求 | FDE 更懂技术，能设计方案 |
| 解决方案架构师 | 技术方案 | FDE 更贴近客户、负责落地 |
| 技术销售 | 卖产品 | FDE 负责交付 |
| 算法工程师 | 模型 | FDE 负责全链路 |

## 二、为什么 AI 项目需要 FDE

**AI 项目的失败率**（行业数据）：
- 70-85% 的 AI 项目未能交付预期价值
- 主因：需求理解错、技术方案不匹配、监管不合规

**典型失败案例**：
```
客户：我们要用 AI 风控
算法团队：训了个 AUC 0.9 的模型
落地：模型黑盒，监管不接受 → 项目黄
```

**FDE 的价值**：在需求阶段就识别"监管要可解释"，设计方案时把规则 + 模型 + Agent 组合，保证可落地。

## 三、风控 FDE 的核心场景

**场景 1：AI 风控合规**
- 客户：银行，要求 AI 决策可解释
- FDE：设计"规则做硬约束 + LLM 做解释生成"方案
- 落地：决策带自然语言解释，监管审核通过

**场景 2：新型欺诈防御**
- 客户：支付公司，发现新型欺诈
- FDE：设计"Agent + 关系网络"方案，Agent 自主推理新型模式
- 落地：Agent 决策 + 人工复核闭环

**场景 3：智能审核**
- 客户：电商，案件审核人力贵
- FDE：设计"LLM 辅助审核"方案，自动总结、分类、推荐
- 落地：审核效率 +50%

**场景 4：跨境合规**
- 客户：海外业务，多国法规不同
- FDE：设计"分区域策略"方案，每区域独立规则 + 共享平台
- 落地：满足各国合规要求

## 四、FDE 的方案设计方法论

**1. 需求拆解（理解客户）**：
```
客户：我们要降低欺诈损失
   ↓ FDE 追问
具体：哪类欺诈？（账户盗用 / 套现 / 中介）
现状：当前欺诈率多少？损失多少？
目标：降到多少？投入预算多少？
约束：监管要求？时效要求？数据可用？
```

**2. 方案设计（技术选型）**：
```
基于需求，设计技术方案:

业务目标: 欺诈率从 0.5% 降到 0.2%
技术方案:
  ├─ 数据层: 整合事件流 + 用户画像 + 关系网络
  ├─ 决策层: 规则（硬约束）+ 模型（软判断）+ Agent（边缘）
  ├─ 监管层: 决策可解释 + 审计日志
  └─ 运营层: 控制台 + 灰度发布 + A/B

技术选型:
  - 规则引擎: Drools（成熟）/ QLExpress（轻量）
  - 模型: XGBoost（浅）+ LLM（深）
  - Agent: 自研编排 + LLM
  - 图: JanusGraph（大规模）
  - 推理: vLLM（高性能）
```

**3. 落地路径（分阶段）**：
```
Phase 1（1 个月）: 基础架构 + 规则接入
Phase 2（2 个月）: 模型部署 + 决策融合
Phase 3（2 个月）: Agent 接入 + 异步复核
Phase 4（持续）: 自学习闭环 + 优化
```

## 五、AI 风控合规（FDE 的硬骨头）

**监管对 AI 决策的要求**：

| 要求 | 实现方式 |
|------|---------|
| **可解释** | 决策带自然语言解释 + 命中规则/证据 |
| **可追溯** | 完整决策日志（输入/中间/输出） |
| **可审计** | 模型版本、训练数据可追溯 |
| **公平性** | 不歧视特定群体（性别、地域） |
| **安全** | 防注入、防越权、防数据泄露 |

**FDE 的合规设计**：

```
决策输出:
{
  "decision": "REJECT",
  "explanation": "新账户深夜大额转账给高风险账户",
  "rules_hit": [...],
  "model_score": 0.85,
  "evidence": [...],
  "model_version": "v2.3.1",
  "trace": "完整推理链路"
}

可追溯:
  - 决策日志保留 7 年（监管要求）
  - 模型版本可回溯（任何决策可重放）
  - 数据版本可追溯（DVC）
```

## 六、FDE 的项目管理

**关键节点**：
1. **需求确认**（Week 1-2）：和客户对齐目标、约束
2. **方案设计**（Week 3-4）：技术方案 + 评审
3. **POC 验证**（Week 5-8）：小规模验证
4. **落地交付**（Week 9-16）：分阶段上线
5. **效果跟踪**（持续）：ROI 评估、迭代

**沟通管理**：
- 客户：定期同步进度、风险
- 技术：明确分工、技术评审
- 监管：提前沟通、合规审查
- 内部：跨团队协调

**风险管理**：
```
风险: 监管不批准
缓解: 早期介入、可解释设计、灰度验证

风险: 模型效果不达标
缓解: POC 阶段验证、备选方案（规则）

风险: 数据质量差
缓解: 数据评估、清洗、采集补全
```

## 七、FDE 的能力模型

**1. 业务理解**（30%）：
- 风控业务全流程
- 监管法规
- 客户痛点

**2. 技术深度**（30%）：
- AI 全栈（数据/训练/推理/Agent）
- 分布式系统
- 工程化

**3. 沟通能力**（20%）：
- 客户沟通（讲技术给业务听）
- 技术沟通（讲业务给技术听）
- 跨团队协调

**4. 项目管理**（20%）：
- 需求管理
- 进度控制
- 风险管理

**学习路径**：
- 工程师 → 资深工程师（技术深度）
- → 解决方案工程师（业务理解）
- → FDE（综合）

## 八、FDE 的价值衡量

**项目级**：
- 交付率（按期按质）
- 客户满意度（NPS）
- 业务效果（ROI、欺诈率下降）

**组织级**：
- 项目交付周期
- 知识沉淀（方案库、案例库）
- 团队赋能（培训、流程）

**风控 FDE 的典型成果**：
```
项目: 某银行 AI 风控
交付:
  - 决策 RT 从 500ms → 100ms
  - 拦截率 +30%
  - 误杀率 -20%
  - 监管审核通过（可解释）

ROI:
  - 年减少欺诈损失 5000 万
  - 项目投入 500 万
  - ROI = 1000%
```

## 九、AI 转型中的 FDE（就业视角）

**市场需求**：
- AI 项目大爆发，FDE 缺口大
- 薪资高（资深 FDE 50-100 万年薪）
- 复合角色难替代（AI 替不了沟通和判断）

**风控工程师转 FDE 的优势**：
- 懂风控业务（天然业务理解）
- 懂分布式系统（技术深度）
- 懂监管（合规意识）

**需要补充**：
- AI 全栈知识
- 客户沟通技巧
- 项目管理方法论

**面试加分**：
- 能讲 AI 项目落地案例
- 理解监管对 AI 的要求
- 能设计方案（架构图、选型、阶段）

## 十、底层本质：AI 项目的"翻译官"

FDE 的本质是**"业务-技术-客户"的翻译官**：

**业务语言**：降欺诈率、提效率、合规
**技术语言**：模型、规则、Agent、平台
**客户语言**：投入产出、风险、价值

FDE 在三者之间翻译，让"技术能做的"变成"客户能用且有价值的"。

**这是 AI 时代的"系统化思维"**：
- 不只看技术（工程师视角）
- 不只看业务（PM 视角）
- 不只看客户（销售视角）
- 全链路负责（FDE 视角）

**类比传统软件**：
- 传统：项目经理 + 架构师 + 产品经理
- AI 时代：FDE（三合一）

## 十一、FDE 与 Agent 的协同（未来）

**未来形态**：FDE 用 Agent 辅助做方案设计。

```
FDE: "客户要降低欺诈率，预算 500 万，监管严"
   ↓ Agent 辅助
Agent:
  - 生成方案初稿（架构图 + 选型）
  - 评估技术可行性
  - 预估工期和成本
  - 识别合规风险
FDE: review + 调整 + 客户沟通
```

**Agent 不替代 FDE**：
- Agent 擅长生成、分析
- FDE 擅长判断、沟通、决策
- 二者协同

## 十二、案例：完整的 AI 风控项目交付

**客户**：某城商行，要求 AI 风控升级

**需求**：
- 现状：纯规则风控，欺诈率 0.5%
- 目标：降到 0.2%，时效 < 200ms，可解释

**FDE 方案**：
```
架构:
  ├─ 规则引擎（保留，硬约束）
  ├─ XGBoost 模型（决策融合）
  ├─ LLM 解释生成（监管要求）
  └─ Agent（边缘场景异步复核）

合规:
  - 决策带自然语言解释
  - 完整审计日志
  - 模型版本管理

落地:
  Phase 1（2 月）: 模型部署 + 决策融合
  Phase 2（1 月）: LLM 解释生成
  Phase 3（持续）: Agent + 自学习
```

**交付效果**：
- 欺诈率 0.5% → 0.18%（达标）
- RT < 150ms（达标）
- 监管审核通过（可解释）
- 客户满意度 NPS 9/10

## 常见考点
1. **FDE 和解决方案架构师区别**？——FDE 更贴近客户、负责落地效果；架构师偏技术方案设计。
2. **风控 AI 项目最大的合规挑战**？——可解释性（黑盒决策监管不接受），需要规则 + LLM 解释组合。
3. **FDE 怎么衡量价值**？——项目交付率、客户 NPS、业务 ROI、组织知识沉淀。

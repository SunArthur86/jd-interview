# 【蚂蚁风控】RAG 怎么用？风控系统的 RAG 应用

> JD 依据："大模型实践"。RAG 是 LLM 应用最广的模式。

## 一、为什么需要 RAG

**LLM 的局限**：
1. **知识截止**：训练数据有截止日期（不知道最新规则）
2. **不知道私有知识**：公司内部规则、案件不在训练集
3. **幻觉**：编造看起来合理但错误的事实
4. **不可追溯**：不知道答案从哪来

**RAG 的解法**：
- 把私有知识存向量库
- 提问时检索相关片段
- 拼到 prompt 让 LLM 基于片段回答
- 答案带引用可追溯

## 二、RAG 的基本流程

```
离线建库:
  文档 → 切分（chunk）→ Embedding → 存向量库

在线检索:
  问题 → Embedding → 向量检索（找相似 chunk）→ 拼 prompt → LLM 生成
```

**完整流程**：

```
1. 知识准备
   文档（PDF/Markdown/HTML）→ 切分成 chunk（如每段 500 字）

2. 向量化
   每个 chunk → embedding 模型 → 向量（如 1536 维）

3. 入库
   向量 + 原文存向量库（Milvus/Pinecone）

4. 查询时
   用户问题 → embedding → 向量库检索 Top-K 相似 chunk

5. 拼 prompt
   "基于以下资料回答: {retrieved_chunks}
    问题: {question}"

6. LLM 生成
   基于检索结果回答，带引用
```

## 三、向量库选型

| 向量库 | 特点 | 适用 |
|--------|------|------|
| **Milvus** | 开源、分布式、高性能 | 大规模生产 |
| **Pinecone** | 托管、易用 | 中小规模 |
| **Weaviate** | 开源、支持混合检索 | 通用 |
| **Faiss** | Facebook 开源、单机库 | 中小规模 |
| **Elasticsearch** | 8.0+ 支持向量 | 已有 ES 团队 |

**风控选 Milvus**：
- 开源、可私有部署
- 分布式、支撑亿级向量
- 阿里有增强版（Proxima）

## 四、风控的 RAG 应用

### 应用 1：相似案件检索
```
新事件 → embedding → 案件库检索 Top-5 相似历史案件
拼 prompt: "这是新事件: {event}
           历史相似案件: {cases}
           决策:"
LLM: 基于历史案件推理新事件
```

### 应用 2：规则库问答（运营助手）
```
运营: "什么场景该用规则 R045？"
→ 检索规则库（找到 R045 的描述、适用场景、案例）
→ LLM: "R045 是设备指纹关联多账号规则，适用于..."
```

### 应用 3：风控知识助手
```
客服: "这个用户为什么被拦截？"
→ 检索该用户决策日志 + 规则库
→ LLM: "用户被拦截因为命中 R001（异地登录）和 R045（设备关联）"
```

### 应用 4：欺诈话术识别
```
聊天文本 → embedding → 话术库检索
→ 命中"代付话术" → 标记风险
```

## 五、高级 RAG 技术

**朴素 RAG 的问题**：
- 检索不准（embedding 不匹配）
- chunk 切分差（重要信息被切断）
- 查询和文档语义差距

**改进技术**：

### 1. 查询改写（Query Rewriting）
```
原始问题: "这个用户有问题吗"
改写: "查询用户 A 的风险评分、历史决策、关联账号"
→ 改写后再检索，更精准
```

### 2. HyDE（Hypothetical Document Embeddings）
```
问题 → LLM 先生成"假设答案"
用假设答案的 embedding 检索（比问题本身更接近文档）
```

### 3. 重排序（Re-ranking）
```
向量检索召回 Top-50（粗排）
用 cross-encoder 重排到 Top-5（精排）
精度大幅提升
```

### 4. 多跳检索（Multi-hop）
```
问题需要多步推理:
  "用户 A 关联的中介是谁？"
  1. 查 A 的关联（找 B）
  2. 查 B 是否是中介
→ 多次检索
```

### 5. 混合检索
```
向量检索（语义）+ 关键词检索（精确）
合并结果（RR Fusion）
→ 既懂语义又精确
```

## 六、chunk 切分策略

**切分影响检索质量**：

**朴素切分**：
```
按固定长度切（如 500 字），按句号边界
问题：可能切断语义
```

**结构化切分**：
```
按文档结构（章节、段落、列表）
问题：需要解析文档结构
```

**语义切分**（推荐）：
```
用 embedding 检测语义边界
语义连贯的一段切成一个 chunk
```

**重叠切分**：
```
chunk 之间重叠 50-100 字
避免边界信息丢失
```

**风控规则库的切分**：
```
按规则单元切（每条规则一个 chunk）
含：规则 ID、描述、适用场景、案例
```

## 七、Embedding 模型选择

| 模型 | 特点 |
|------|------|
| OpenAI text-embedding-3 | 通用、强、贵 |
| BGE-M3（智源） | 中文好、开源 |
| M3E | 中文开源 |
| Cohere embed | 多语言 |

**风控选择**：
- 中文场景：BGE-M3 或 M3E
- 自建部署（数据敏感）

## 八、评估 RAG 系统

**检索质量**：
- **召回率**：相关文档是否被召回
- **精确率**：召回的文档是否相关
- **MRR**（Mean Reciprocal Rank）：相关文档排名

**生成质量**：
- **答案准确性**：是否基于检索结果
- **答案完整性**：是否覆盖关键信息
- **幻觉率**：是否编造未检索到的内容

**RAGAS 框架**：
- 自动化评估 RAG 系统
- 包含检索和生成的多维度指标

```

                       ↓
┌──────────────────────────────────────────────────┐

                       ↓
┌──────────────────────────────────────────────────┐

```

## 十、性能优化

**1. 检索缓存**：
- 相似问题复用结果
- Redis 缓存查询 embedding

**2. 异步预检索**：
- 用户输入时实时检索（边输入边查）
- 减少最终响应时间

**3. 增量更新**：
- 新规则/案件增量入库
- 不重建全库

**4. 多级缓存**：
- 答案缓存（同问题直接返回）
- 检索结果缓存
- Embedding 缓存

## 十一、底层本质：检索 + 生成的解耦哲学

RAG 的核心思想是**"知识的外置"**：

**传统 LLM**：知识在参数里（训练时固化）
**RAG**：知识在外部库（可动态更新）

**这是软件工程"代码 vs 数据"的分离**：
- LLM 参数 = 代码（推理逻辑）
- 向量库 = 数据（知识）
- 数据可独立更新，不用改代码（重训）

**好处**：
- 知识更新快（不用重训 LLM）
- 可控（删除某知识 = 删 chunk）
- 可追溯（答案带引用）

**代价**：
- 检索延迟（+100-500ms）
- 检索质量影响生成质量

## 十二、RAG vs 微调

| 维度 | RAG | 微调 |
|------|-----|------|
| 知识更新 | 快（更新库） | 慢（重训） |
| 适合 | 知识频繁变 | 固定模式 |
| 成本 | 低（不用训） | 高（需要数据+算力） |
| 可控性 | 高（删 chunk） | 低（重训才改） |
| 性能 | 检索 +100ms | 无额外延迟 |

**风控的选择**：
- 规则/案件：RAG（频繁更新）
- 风控决策模式：微调（固定模式，如"什么样的交易是欺诈"）

**组合**：RAG（实时知识） + 微调（基础能力）

## 常见考点
1. **RAG 检索不准怎么办**？——优化 embedding 模型、改 chunk 策略、加 re-ranking、查询改写。
2. **RAG 和微调什么时候选**？——知识更新频繁选 RAG，固定模式选微调，最佳是组合。
3. **向量库为什么用近似检索**？——精确 KNN 在亿级向量下不可行（O(n)），近似检索（HNSW/IVF）O(log n) 牺牲一点精度换百倍速度。

**代码示例**（RAG 实现）：
```python
def rag_answer(question, top_k=5):
    # 1. Embedding
    q_emb = embedding_model.encode(question)

    # 2. 向量检索
    chunks = milvus.search(
        collection="risk_knowledge",
        vector=q_emb,
        top_k=top_k * 2  # 多召回，后续重排
    )

    # 3. Re-ranking
    reranked = cross_encoder.rank(question, chunks)[:top_k]

    # 4. 构造 prompt
    context = "\n".join([c.text for c in reranked])
    prompt = f"基于以下资料回答:\n{context}\n\n问题: {question}\n回答（带引用）:"

    # 5. LLM 生成
    answer = llm.complete(prompt)
    return answer
```

# 【巨剧核 AI 陪伴】长上下文与会话管理怎么设计？

> JD 依据："负责对话服务，包括上下文管理、会话状态；熟悉长上下文、会话管理、记忆摘要。"

## 一、为什么不能全塞

裸做法的陷阱：
```
把全部历史塞 prompt → 第 200 轮时 token 爆炸
后果：超窗口报错 / 费用爆炸 / 延迟飙升 / 模型注意力稀释（lost in the middle）
```

根本矛盾：**窗口有限（固定 token）vs 对话无限（时序增长）**。

## 二、三层窗口模型

```
┌─────────────────────────────────────────┐
│  Token 预算（如 8K）                      │
│ ┌─────────────────────────────────────┐ │
│ │ System：角色人设 + 剧情设定（必留）    │ │
│ ├─────────────────────────────────────┤ │
│ │ 长期记忆摘要：用户画像/关键事实（压缩）│ │
│ ├─────────────────────────────────────┤ │
│ │ 近期对话原文：最近 N 轮（滑动窗口）   │ │
│ ├─────────────────────────────────────┤ │
│ │ 当前用户输入                          │ │
│ └─────────────────────────────────────┘ │
│           输出预留（如 1K）               │
└─────────────────────────────────────────┘
```

## 三、滑动窗口（近期原文）

保留最近 N 轮原文，保证短时连贯：
```
recent = dialog[-N:]   # N 由预算反推
```

**N 怎么定**：可用 token = 窗口 - 人设 - 摘要 - 输入 - 输出预留 - 余量；N = 可用 token / 平均单轮 token。

## 四、压缩与摘要（中期）

当窗口逼近阈值，把较旧的原轮次压缩：
```
策略 A 滚动摘要：旧 10 轮 → LLM 压成 1 段（"用户和小雨聊了工作压力，用户是程序员，情绪低落"）
策略 B 关键信息抽取：抽取实体/偏好/事件 → 结构化存表
策略 C 混合：结构化抽取 + 自然语言摘要并存
```

**增量摘要**：只压新增部分，拼接旧摘要，避免每轮全量重算（省钱省时）。

## 五、长期记忆与向量召回（远期）

被压缩/丢弃的细节进向量库，需要时召回注入：
```
用户："我上次说的那本《三体》后来怎样了？"
→ 向量检索历史 → 命中相关片段 → 注入当前上下文
```

（详见记忆系统题 boss-ai-003）

## 六、token 预算调度器

```python
def assemble_context(persona, memory_summary, recent, user_input, budget):
    # 按优先级分配，留 15% 余量
    must = persona + memory_summary          # 必留
    remain = budget - len(must) - len(user_input) - OUTPUT_RESERVE - MARGIN
    # 贪心塞近期，塞不下就压缩最旧的一轮
    while token_len(recent) > remain and recent:
        oldest = recent.pop(0)
        memory_summary = incremental_summarize(memory_summary, oldest)
        remain = budget - len(persona+memory_summary) - len(user_input) - RESERVE
    return persona + memory_summary + recent + user_input
```

## 七、底层本质

长上下文管理的本质是**"在有限容量下做信息取舍的调度问题"**——把'对话历史'从'线性增长的负担'重构为'分层调度的资源'：近期保真、中期压缩、远期索引。

**容量换质量、延迟换记忆深度**，是陪伴产品体验与成本的核心 trade-off。

## 常见考点

1. **超长上下文模型（128K/1M）还需要窗口管理吗？**——需要。一是成本（token 按量计费），二是'lost in the middle'注意力衰减，三是延迟。窗口管理是经济性和质量双重需要。
2. **摘要会不会越压越失真？**——会。缓解：保留结构化关键事实（不压）、定期全量重摘要校准、用户可主动触发'回忆'。
3. **多角色/群聊场景上下文怎么管？**——按发言者分轨 + 各自摘要 + 聚合窗口；群聊还要做相关性筛选，避免无关发言占窗口。


## 结构化回答

**30 秒电梯演讲：** 聊到长上下文与会话管理怎么设计，我的理解是——长上下文管理是在有限 token 窗口里，决定哪些对话该留、哪些该压缩、哪些该丢，让角色既不失忆又不爆 token 的工程博弈。打个比方，像一本戏的场记本——舞台只能摆有限道具（token 窗口），导演要决定哪些最近台词铺在台上（滑动窗口）、哪些旧情节压成一页摘要（压缩）、哪些直接归档（长期记忆）。

**展开框架：**
1. **窗口三层** — 近期原文（滑窗）+ 中期摘要（压缩）+ 远期向量召回（记忆）
2. **滑动窗口** — 保留最近 N 轮原文，N 由模型窗口和角色需求定
3. **压缩策略** — 滚动摘要（旧轮次压缩成一段）/ 关键信息抽取（人名/偏好/事件）

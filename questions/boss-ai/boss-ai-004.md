# 【巨剧核 AI 陪伴】IM/消息系统怎么设计？

> JD 依据："熟悉消息/IM；负责消息通知、私聊群聊、消息可靠性。"

## 一、IM 的不可靠挑战

```
用户 → 网络 → 服务端 → 角色（LLM）→ 服务端 → 网络 → 用户
任何一环都可能：丢包 / 延迟 / 断连 / 乱序 / 重复
```

用户期望：**不丢、不重、有序、实时**。这是不可靠网络上的确定性承诺。

## 二、连接层

```
主：WebSocket（双向、低延迟、长连接）
降级：HTTP 长轮询 / SSE（防火墙穿透）
心跳：客户端 30s ping，服务端 3 次未收判定断连
重连：指数退避（1s/2s/4s/8s，上限 30s）
```

**AI 陪伴特色**：角色回复是流式的，WebSocket 适合把 LLM 的 token 流实时推给前端（边生成边显示）。

## 三、消息可靠性（不丢不重有序）

```
发送：
client → msg(msg_id, session_seq) → server
server 持久化 → 回 ack(msg_id)
client 未收 ack → 重传（带相同 msg_id）

接收：
server → msg(msg_id, session_seq) → client
client 收到 → 回 ack
server 未收 ack → 重发
client 按 msg_id 去重，按 session_seq 排序投递
```

**三件套**：
- **全局唯一 msg_id**：去重（幂等）
- **会话内单调递增 seq**：排序
- **持久化后才 ack**：不丢

## 四、离线消息与推送

```
用户离线 → 消息持久化到「未读队列」（Redis ZSet，按 seq）
       → 触发推送（APNs/FCM/小米/华为厂商通道）
用户上线 → 拉取未读（按 seq 范围分页）→ 清队列
```

**离线 1000 条未读**：不一次性推，按 seq 范围分页拉取 + 客户端合并；只推有摘要的最近 N 条。

## 五、私聊 vs 群聊

**私聊（1v1，用户 ↔ 角色）**：直接写收件箱，简单。

**群聊（多角色/多用户剧情）**：
```
写扩散：消息写入每个成员收件箱
  优：读快（直接读自己箱子）
  缺：写放大（千人群写 1000 次）
  适用：小群（< 500 人）

读扩散：消息写一份到群时间线，成员拉取时合并
  优：写少
  缺：读要合并多源
  适用：大群

混合：小群写扩散 + 大群读扩散
```

AI 陪伴场景群聊通常是小规模（用户 + 几个角色），写扩散为主。

## 六、AI 陪伴的特殊消息类型

- **流式回复**：LLM 边生成边推（chunk 消息，带 same reply_id 合并）
- **主动触发消息**：角色主动找用户（定时/事件驱动，走同一投递链路）
- **富媒体**：语音/图片/视频（信令走 IM，大文件走对象存储 + URL）
- **剧情事件**：故事进度变更广播给会话内所有角色

## 七、底层本质

IM 的本质是**"用应用层协议补足网络的不可靠，把'尽量送达'升级为'确定性必达'"**：

- 不丢 = 持久化 + ack + 重传
- 不重 = 唯一 ID 去重（幂等）
- 有序 = 单调 seq
- 实时 = 长连接 + 流式

这是把**不可靠的物理通道**包装成**可靠的服务承诺**，是所有 IM 系统共同的第一性原理（QQ/微信/钉钉底层都一样）。

## 常见考点

1. **WebSocket 和 HTTP 长轮询怎么选？**——WS 双向低延迟首选；HTTP 兼容性/穿透更好做降级；SSE 适合服务端单向推送（如 LLM 流式输出）。
2. **消息怎么保证全局有序？**——很难保证跨会话全局有序；单会话内用 seq 有序即可，跨会话用时间戳近似（接受 NTP 时钟偏差）。
3. **亿级消息怎么存？**——按 user_id 分库分表 + 冷热分离（热数据 Redis/MySQL，冷归档 OSS/ClickHouse），消息 ID 用 Snowflake。


## 结构化回答

**30 秒电梯演讲：** 聊到IM/消息系统怎么设计，我的理解是——IM 消息系统是在不可靠网络上，把'用户↔角色'的文字/语音消息可靠、有序、实时地送达，并支持群聊/离线推送的工程系统。打个比方，像邮局系统——信件（消息）要挂号（确认）、盖戳（有序号）、走分拣（路由）、到家没人要暂存（离线消息）、重要信件还能加急（推送）。

**展开框架：**
1. **连接** — WebSocket 长连接（双向/低延迟）+ 降级 HTTP 轮询/SSE
2. **可靠性** — 客户端 seq + 服务端 ack + 离线消息存储 + 重传
3. **顺序** — 单会话内单调递增 seq，按 seq 投递

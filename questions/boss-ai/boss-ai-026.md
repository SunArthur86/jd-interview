# 【巨剧核 AI 陪伴】AI Harness / LLMOps 全链路怎么建？

> JD 依据："AI Harness/LLMOps 全链路：网关、推理、评估、迭代闭环。"

## 一、LLMOps vs DevOps

```
DevOps（传统软件）：
  代码 → CI/CD → 部署 → 监控 → 迭代
  特点：确定性逻辑，监控主要看性能指标

LLMOps（AI 应用）多了三环：
  [1] 评估（Evaluation）：质量怎么衡量（概率输出，没标准答案）
  [2] 漂移监控：模型/数据分布会变化（不像代码固定）
  [3] 数据闭环：用户反馈/标注 → 改进模型/prompt
```

## 二、全链路架构

```
┌──────────────────────────────────────────┐
│ [1] 网关层                                 │
│   统一接入 / 路由 / 限流 / 成本治理 / 鉴权   │
├──────────────────────────────────────────┤
│ [2] 推理层                                 │
│   API 调用 / 自部署（vLLM）/ 缓存 / 池化    │
├──────────────────────────────────────────┤
│ [3] 应用层                                 │
│   prompt 管理 / 记忆 / 工具 / 编排          │
├──────────────────────────────────────────┤
│ [4] 评估层                                 │
│   离线评测集 / 在线 A/B / LLM-as-Judge     │
├──────────────────────────────────────────┤
│ [5] 监控层                                 │
│   三支柱 + 质量监控 + 漂移检测              │
├──────────────────────────────────────────┤
│ [6] 数据闭环                               │
│   日志 → 标注 → 训练 → 上线               │
└──────────────────────────────────────────┘
```

## 三、网关层

```
职责：
  - 统一接入：业务通过一个网关调所有模型
  - 路由：按场景/成本路由
  - 限流：分 tier + 全局
  - 成本治理：预算/配额/降级
  - 鉴权：API key/权限
  - 可观测：调用日志/成本归因
  - fallback：故障切换

实现：自研 LLM Gateway（深度定制）/ LiteLLM / OpenRouter
```

## 四、推理层

```
模式：
  API：弹性大、贵、数据出域
  自部署：成本可控、数据私有、要运维
  混合：自部署小模型 + API 强模型

优化：
  vLLM continuous batching（GPU 利用率 ×5-10）
  PagedAttention（KV Cache 复用）
  量化（INT8/INT4 降规格）
  缓存（FAQ / 语义缓存）
  池化（连接池 / 模型实例池）
```

## 五、评估层（LLMOps 核心）

```
[1] 离线评测集
    业务关键场景抽样 + 人工标注
    每条：(input, context, expected_output, 评分维度)
    每次 prompt/模型变更回归

[2] 在线 A/B
    流量分桶对比新旧版本
    业务指标（留存/时长）+ 质量分

[3] LLM-as-Judge
    用强模型给输出打分（多维：人设/安全/有用/共情）
    大规模自动评估

[4] 人工评估
    关键场景抽样人工评
    LLM-Judge 校准

[5] 用户反馈
    点赞/踩/投诉 → 负样本挖掘
```

## 六、监控层

```
三支柱 + AI 特色：

性能：延迟（首 token/全量）/ 吞吐 / 错误率
业务：留存 / 时长 / 互动 / 付费
成本：tokens / 单次成本 / 路由成本分布
质量：人设一致性分 / 满意度 / 投诉率 / 审核拦截率

漂移检测（LLMOps 特色）：
  数据漂移：输入分布变化（用户问题类型变了）
    监控：embedding 距离 / 关键词分布 / query 长度分布
  概念漂移：用户期望变化（同样问题期望变了）
    监控：满意度趋势 / 投诉类型变化
  模型漂移：底层模型更新导致输出风格变化
    监控：版本绑定 / 切换前后质量对比
  
漂移显著 → 告警 → 评估 → 调整 prompt/模型
```

## 七、数据闭环

```
日志采集
   ↓
样本筛选（高价值/异常/低分）
   ↓
人工标注（input/output/期望/问题）
   ↓
数据集累积
   ↓
[1] Prompt 迭代：基于 bad case 改 prompt
[2] 微调：累积足够数据 SFT/DPO 小模型
[3] RAG 优化：补充知识库
[4] 评测集扩充：补强覆盖
   ↓
上线验证（A/B）
   ↓
回流闭环
```

这是 LLMOps 区别 DevOps 的核心——**数据持续驱动改进**。

## 八、版本与实验管理

```
版本化对象：
  - Prompt（每版有指标基线）
  - 模型（API 版本/自部署 checkpoint）
  - 配置（路由/审核/降级策略）
  - 评测集（持续扩充）

实验：
  - MLflow / WandB 记录每次实验（prompt/模型/指标）
  - 对比历史版本
  - 灰度上线 + 自动回滚
```

## 十、底层本质

AI Harness/LLMOps 的本质是**"把不确定的、会漂移的、数据驱动的 LLM 应用，纳入工程化治理"**：

- 网关 = 接入可控
- 推理 = 服务高效
- 评估 = 质量可量
- 监控 = 漂移可发现
- 闭环 = 持续改进

这是把'LLM 应用'从'调 API 的玩具'升级为'可工程化运营的产品'——是 AI 产品规模化、长期化、商业化运营的基石。

## 常见考点

1. **小公司要全套 LLMOps 吗？**——不必。按阶段：早期用 LangSmith/LangFuse 起步（追踪+评估）；中期加 prompt 管理 + A/B；规模化才上自研 + 数据闭环。先有评估再谈优化。
2. **LLM-as-Judge 可信吗？**——部分可信。适合大规模粗筛和客观维度（格式/安全）；主观维度（情感/创意）有偏差；最佳实践：LLM 粗筛 + 人工精评混合。
3. **数据闭环怎么不踩坑？**——数据质量第一（脏数据训坏模型）+ 标注一致性（多人交叉）+ 隐私脱敏 + 评测集独立（不混训练和评测）+ 小步快跑（小数据先验证）。

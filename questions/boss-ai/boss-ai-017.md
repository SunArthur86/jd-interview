# 【巨剧核 AI 陪伴】LLM 服务的限流/降级/超时/重试怎么做？

> JD 依据："负责稳定性：限流、降级；LLM 网关限流、模型降级、指数退避。"

## 一、为什么需要四道阀

```
无防护的雪崩：
  LLM 厂商变慢 → 调用堆积 → 线程池占满 → 服务无响应
  → 上游依赖也被拖 → 全链路雪崩 → 全盘崩溃

有防护：
  入口限流挡住超量
  超时切断拖慢
  重试挽救偶发
  降级保核心
  熔断隔离故障
```

## 二、限流（挡超量）

### 1. 算法
```
令牌桶（主流）：允许突发（积累的令牌可瞬间用）
漏桶：严格匀速
滑动窗口：精确计数
分布式：Redis + Lua（集群一致）
```

### 2. 多级限流
```
网关层：IP / user_id 维度（防恶意刷）
应用层：接口级（发消息/视频生成分别限）
模型层：LLM 网关 QPS / token/s（不超厂商配额）
资源层：GPU/连接池上限
```

### 3. 分 tier
```
VIP：100 req/min
付费：30 req/min
免费：10 req/min
+ 全局兜底（总 QPS 不超容量）
```

### 4. LLM 限流的特殊性
- 按 token/s（不是单纯 QPS）
- 按模型配额（OpenAI/通义分别配额）
- 限流后不直接 429（体验差），降级到便宜模型或排队

## 三、降级（保核心）

```
策略：核心保命，非核心关停

LLM 降级链：
  强模型 → 弱模型 → 兜底模板回复
  （高峰自动切弱模型）

功能降级：
  视频/图片生成暂停（成本高）
  长上下文压缩（省 token）
  主动触发消息关闭
  推荐简化（热门内容代替个性化）

触发：
  自动：GPU 利用率 > 80% / 错误率 > 5% / 成本告警
  手动：运营/告警值班一键触发

开关：
  配置中心维护，热更，分级（按场景/tier）
```

## 四、超时（切拖慢）

```
分级超时：
  连接超时：3s（建连失败快速失败）
  首 token 超时：5s（LLM 流式首字延迟）
  全量超时：30s（总时长上限）
  队列等待超时：10s（排队太久放弃）

级联超时：
  上游超时 < 下游超时
  （让上游先失败，不留悬挂请求）

超时后：
  返回错误 / 切 fallback / 重试
```

## 五、重试（救偶发）

```
原则：
  幂等性：只重试幂等操作（或带 idempotency_key）
  指数退避：1s → 2s → 4s → 8s（避免重试风暴）
  上限：最多 3 次
  抖动：加随机偏移（避免同步重试）

不重试的场景：
  非幂等（扣费/创建）
  业务错误（参数错）
  已超总超时

重试风暴：
  错误做法：1 个失败 → 重试 3 次 → 上游 QPS ×4 → 上游也挂
  正确做法：指数退避 + 抖动 + 熔断（直接不重试）
```

## 六、熔断（隔故障）

```
熔断器：
  closed（正常）→ 错误率 > 50% → open（熔断，快速失败）
  open → 等 30s → half-open（试探放少量请求）
  half-open → 成功 → closed / 失败 → open

场景：
  LLM 厂商挂了 → 熔断该厂商调用 → 切备用厂商
  DB 慢 → 熔断部分查询 → 降级到缓存

价值：
  给故障依赖喘息时间
  防止故障通过重试放大
```

## 七、LLM 网关完整防护

```
请求 → 网关
   ↓
限流（user tier + 全局）
   ↓
路由 → 主模型
   ↓
熔断检查（主模型是否 open）
   ↓ 是 → 切备用
调用（连接超时 3s + 首 token 5s + 全量 30s）
   ↓
失败 → 重试（指数退避，最多 3 次）
   ↓ 仍失败
降级（弱模型 → 模板回复）
   ↓
返回（最好情况：主模型成功；最坏：模板兜底）
```

## 八、底层本质

限流降级超时重试的本质是**"在系统边界设多重防护，把局部故障隔离吸收，不让它雪崩成全局灾难"**：

- 限流 = 拒超量（保护自己）
- 降级 = 保核心（牺牲非关键）
- 超时 = 切拖慢（防止级联阻塞）
- 重试 = 救偶发（应对瞬时）
- 熔断 = 隔故障（防止重试放大）

这是把'无防护的脆弱服务'升级为'有韧性的弹性系统'——是任何高可用 toC 服务的标配。

## 常见考点

1. **令牌桶和漏桶怎么选？**——令牌桶允许突发（适合 LLM 这种偶发突发场景），漏桶严格匀速（适合下游对均匀流量敏感的场景）。LLM 网关多用令牌桶。
2. **重试为什么会引起雪崩？**——N 个上游同时失败 → 同时重试 → 下游 QPS ×N → 下游也挂 → 全链路雪崩。缓解：指数退避 + 抖动 + 熔断。
3. **降级策略谁来定？**——业务定义（哪些核心哪些可丢）+ 工程实现（开关 + 自动触发）+ 运营配置（阈值调整）。降级是产品决策不是纯技术。

# 【巨剧核 AI 陪伴】Prompt 管理工程怎么做（版本化/A-B/模板）？

> JD 依据："熟悉 Prompt 管理；熟悉数据埋点和实验。"

## 一、为什么 prompt 需要工程化

裸做法：
```
代码里硬编码 prompt 字符串 → PM 想改要发版 → 改完不知道好坏 → 出问题无从追溯
```

prompt 是 AI 产品的**核心逻辑**（相当于传统产品的代码），但它本质是文本——易碎、难追踪、难协作。必须当代码一样工程化。

## 二、Prompt 即代码（版本化）

```
prompt 仓库（Git 或配置中心）：
  persona_xiaoyu/
    v1.0.0/  system_prompt.txt  +  meta.yaml
    v1.1.0/  system_prompt.txt  +  meta.yaml   ← 当前线上
    v1.2.0-rc/  ...
```

每个版本带：
- **变更说明**（改了什么、为什么改）
- **Code Review**（PM + 工程师双签）
- **环境隔离**（dev/staging/prod 用不同版本）
- **回滚能力**（一键切回任意历史版本）

## 三、模板化（变量插值）

```
template = """
你是「{role_name}」，性格：{personality}。
当前剧情进度：{story_progress}。
用户画像摘要：{user_profile}。
最近对话：{recent_dialog}。
用户：{user_input}
回复（保持人设，50字内）：
"""

渲染时：
prompt = render(template, {
  role_name, personality, story_progress,
  user_profile, recent_dialog, user_input
})
```

**关键**：
- 变量类型校验（防止 None/超长导致 prompt 漏字段）
- 默认值（fallback）
- 渲染后 token 预估（超窗口提前告警）
- 模板本身也版本化

## 四、A/B 测试

```
流量分桶（按 user_id hash，保证同一用户体验一致）：
  桶 A（50%）：prompt v1.1.0
  桶 B（50%）：prompt v1.2.0

指标埋点：
  - 质量分：人工评分 / LLM-as-Judge（用大模型给回复打分）
  - 业务：次日留存 / 会话时长 / 互动次数 / 付费转化
  - 成本：avg tokens / 延迟 P95

统计：跑够样本量（如每桶 1 万会话）→ t 检验/卡方 → 显著差异才下结论
```

**陷阱**：2% 的提升在小样本下可能是噪声，必须做显著性检验。

## 五、灰度发布

```
v1.2.0 发布流程：
  内部 1% → 公司员工 5% → 灰度用户 10% → 50% → 100%
每阶段观察 24-48h，核心指标劣化立即回滚
```

## 六、度量体系

| 维度 | 指标 | 采集 |
|---|---|---|
| 质量 | 人工评分 / LLM-as-Judge / 用户点赞踩 | 抽样 + 全量 |
| 业务 | 留存 / 时长 / 互动 / 付费 | 埋点 |
| 性能 | P95 延迟 / 吞吐 | 监控 |
| 成本 | avg tokens / 单次成本 | 计费 |
| 安全 | 审核拦截率 / 投诉率 | 审核日志 |

每个 prompt 版本都关联一套指标，迭代有数据支撑。

## 七、协作流程

```
PM 提需求（角色语气更温柔）
  → 工程师在 prompt 仓库起分支改 v1.2.0-rc
  → 自动评测（跑评测集对比基线）
  → Code Review（PM 评审）
  → staging 环境验证
  → 灰度上线 + A/B
  → 数据显著后全量
```

## 八、底层本质

Prompt 管理工程的本质是**"把核心逻辑（prompt）纳入软件工程体系"**：

- 版本化 = 可追溯
- 模板化 = 可复用
- A/B = 可度量
- 灰度 = 可控风险

这是把**手艺（prompt 写作）**升级为**工程（prompt 工程）**，让 prompt 迭代不再依赖灵感，而是依赖数据、流程和工具——是 AI 时代区别'调 API 玩具'和'可运营产品'的分水岭。

## 常见考点

1. **prompt 变更为什么要 A/B 而不是直接看留存？**——留存受多因素影响（节日/竞品/版本），不控制变量无法归因；A/B 控制单一变量才能说'是 prompt 改的好'。
2. **LLM-as-Judge 可信吗？**——部分可信。适合做粗筛和大规模评测，但有人设/主观题会偏；最稳是 LLM 粗筛 + 人工精评的混合。
3. **prompt 太多变不过来怎么办？**——自动化：评测集回归（每次改动自动跑）、prompt 自动优化（DSPy/AutoPrompt）、建立 prompt 组件库（人设/边界/格式可拼装）。


## 结构化回答

**30 秒电梯演讲：** 聊到Prompt 管理工程怎么做（版本化/A-B/模板），我的理解是——Prompt 管理工程是把'写在代码里的字符串'升级为'版本化、可 A/B、可回滚、可度量、可协作'的产品资产，让 prompt 像代码一样可工程化迭代。打个比方，像把厨房秘方从'大厨脑里的口诀'变成'带版本号的菜谱档案'——每版配方有编号、有食材清单（变量）、有评分（效果指标）、不好吃能回退到上一版。

**展开框架：**
1. **Prompt 即代码** — 版本化（Git/DB）、Code Review、环境隔离
2. **模板化** — 变量插值（人设/记忆/输入）+ Jinja 式语法 + 强类型校验
3. **A/B 测试** — 流量分桶 → 不同 prompt → 业务指标对比 → 显著性检验

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：prompt 放代码里还是配置中心？您更想看哪个方向？

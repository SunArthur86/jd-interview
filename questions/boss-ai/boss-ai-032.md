# 【巨剧核 AI 陪伴】视频生成任务系统怎么设计？

> JD 依据："负责视频生成任务系统：文生视频、异步任务、进度、质量。"

## 一、视频生成的工程挑战

```
特点：
  - 耗时长（30s - 几分钟，远超 HTTP 超时）
  - 耗资源（GPU 重，成本高）
  - 易失败（模型不稳定、GPU 故障）
  - 要计费（用户付费生成）
  - 要可控质量（不能瞎生成）

不能：
  - 同步等（超时）
  - 不管进度（用户焦虑）
  - 不计费（亏钱）
  - 失败不重试（体验差）
```

必须建专门的**任务系统**。

## 二、任务模型

```
Task {
  task_id: 全局唯一
  user_id, role_id, scene
  type: TEXT_TO_VIDEO | IMAGE_TO_VIDEO | VIDEO_EDIT | ...
  input: { prompt, image_url, ref_video_url, params }
  params: { resolution, duration, fps, style, ... }
  status: CREATED|QUEUED|RUNNING|SUCCESS|FAILED|CANCELED
  progress: 0-100
  steps: [{ name, status, started_at, ended_at, error }]
  result: { video_url, thumbnail_url, metadata }
  cost: { gpu_time, money }
  priority, idempotency_key
  created_at, timeout_at
}
```

## 三、算子 DAG（分阶段）

```
简单文生视频：
  [prompt 解析] → [视频生成] → [后处理（超分/补帧）] → [编码] → [存储]

复杂场景（数字人视频）：
  [LLM 生成剧本] → [TTS 配音] ─────┐
                                    ├─→ [合成] → [输出]
              [角色渲染（按剧本/TTS）]─┘

每个算子：
  - 独立服务
  - 标准接口
  - 超时/重试策略
  - 资源需求（GPU 规格）
```

## 四、任务全生命周期

```
[1] 提交
    用户/业务提交任务
    幂等检查（重复提交返回旧 task_id）
    预扣费（锁额度）
    入队
    返回 task_id（同步）

[2] 排队
    按 priority 排序
    VIP 高优先 / 普通 / 批量低
    等待 GPU 资源

[3] 调度执行
    调度器分配 GPU
    worker 按 DAG 执行算子
    每步状态持久化

[4] 进度上报
    每步完成上报
    实时推前端（WebSocket）
    webhook 回调业务
    轮询兜底

[5] 完成
    成功：产物存 OSS，回调，计费实结
    失败：记错误，按策略重试/降级/退款
    超时：标记失败，不无限等
```

## 五、进度推送

```
进度算法：
  分阶段加权：
    prompt 解析（5%）→ 视频生成（70%）→ 后处理（15%）→ 编码（5%）→ 存储（5%）
  细粒度：
    视频生成内部上报（如 30/100 帧完成）

ETA 估算：
  历史平均时长 × 剩余进度

推送方式：
  WebSocket 实时（在线用户）
  webhook 回调（业务系统）
  轮询兜底（前端定时查）

前端体验：
  进度条 + 当前阶段描述 + ETA
  "正在生成...60% （渲染中，预计还需 30 秒）"
```

## 六、质量管控

```
生成质量（多维）：
  清晰度（分辨率/码率）
  时长（符合要求）
  一致性（人物/场景不漂移）
  创意度（prompt 还原度）
  美学（构图/色彩）

自动评估：
  客观指标（清晰度/帧率）
  AI 评分模型（美学/一致性）

质量门禁：
  低于阈值 → 自动重生（限次数）
  → 仍不行 → 用最佳结果 + 标记
  → 通知用户可重试

用户反馈：
  满意度（点赞/踩）
  投诉（问题分类）
  回流改进模型
```

## 七、资源调度

```
GPU 池：
  分池（生成池/渲染池/合成池）
  按任务类型/优先级调度

策略：
  优先级（VIP 先）
  公平调度（防单用户垄断）
  抢占（VIP 可抢普通）
  弹性扩缩（队列堆积触发）

成本控制：
  按 user/scene 配额
  闲时复用（跑批量任务）
  模型量化降规格
  分时定价（闲时便宜）
```

## 八、计费

```
维度：
  算子类型 × 时长 × 资源规格 × 数量

模式：
  预扣（创建任务锁额度）
  实结（完成按实际算）
  退还（失败按比例退）

归因：
  cost 按 task → user → scene → role
  支持用户/场景维度账单
  支持成本分析（哪类任务烧钱）

定价策略：
  按用户 tier（VIP 折扣）
  按时段（闲时便宜）
  按质量规格（4K 贵）
```

## 九、可靠性与恢复

```
持久化：
  每步状态写 DB
  中间产物存 OSS（可复用）

恢复：
  worker 故障 → 任务重调度
  算子失败 → 重试/换算子/降级
  服务重启 → 加载 RUNNING 任务从最后成功步骤继续

幂等：
  算子执行带 idempotency_key
  重试不重复扣费/不重复算

超时：
  任务级（10-30min）
  算子级（每步独立）
  超时标记失败，不无限等
```

## 十、底层本质

视频生成任务系统的本质是**"把 AI 魔法变成可运营的工业流水线"**：

- 任务化 = 解耦提交和执行
- 算子 DAG = 复杂流程可描述
- 进度推送 = 用户可预期
- 质量管控 = 可信赖
- 计费 = 可商业化

这是 Sora/Runway/Pika/Midjourney 等所有 AI 生成产品的共同底座——区别于实时对话（流式），视频生成是**异步任务范式**。

## 常见考点

1. **任务排队太久用户骂街怎么办？**——预期管理（提交时给预估等待）+ VIP 优先 + 弹性扩容 + 部分场景支持实时简化版（低质量预览）+ 错峰激励（闲时便宜）。
2. **生成的视频不一致（人物变了）怎么办？**——用参考图/视频锚定（image-to-video 比 text-to-video 一致性强）+ 多帧联合推理 + 角色 ID embedding + 后处理一致性检查。
3. **GPU 任务怎么弹性扩缩？**——K8s cluster autoscaler + GPU 节点池 + 按队列堆积触发 + 缩容安全驱逐（等任务完成）+ 跨可用区调度。

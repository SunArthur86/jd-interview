# 【巨剧核 AI 陪伴】可观测性、治理、跨团队推进怎么做？

> JD 依据："负责稳定性：日志、监控、报警；可观测性、稳定性治理、跨团队推进；AI 决策可追溯。"

## 一、为什么可观测性是地基

```
没有可观测性：
  线上慢 → 不知道哪里慢
  出故障 → 不知道为什么
  用户投诉 → 复现不了
  AI 出问题 → 不知道是 prompt/模型/记忆哪的锅
  监管问 → 拿不出证据

有可观测性：
  实时看清状态
  故障快速定位
  AI 决策可追溯
  跨团队协作有共同语言
```

**看不见就管不住**。可观测性是一切稳定性治理的地基。

## 二、可观测性三支柱

### 1. 日志（Logs）—— 事件留痕
```
特点：离散事件、高基数、可检索
采集：Filebeat / Fluentd → ES / Loki
AI 场景：
  每个 LLM 调用记一条：
    { trace_id, user_id, role_id, model, prompt_version, 
      input_summary, output_summary, tokens, latency, audit_result }
  保留 90 天+（合规要求）
检索：按 user/time/role/model 自由查询
```

### 2. 指标（Metrics）—— 聚合状态
```
特点：时序、低基数、可聚合
采集：Prometheus / 自研
关键指标：
  黄金四指标：延迟（P50/P95/P99）/ 流量（QPS）/ 错误率 / 饱和度
  AI 特色：
    首 token 延迟（流式体验）
    token 成本 / 单会话
    模型路由命中率 / fallback 触发率
    审核拦截率 / 投诉率
    人设一致性分 / 用户满意度
告警：阈值 + 异常检测
```

### 3. 链路（Traces）—— 因果追踪
```
特点：请求级、跨服务、树形
采集：OpenTelemetry 标准
AI 场景一次对话的 trace：
  对话请求
   ├─ 编排引擎（50ms）
   │   ├─ 取角色人设（10ms）
   │   ├─ 召回记忆（30ms）
   │   │   ├─ 向量查询（20ms）
   │   │   └─ 结构化查询（8ms）
   │   └─ 装配 prompt（5ms）
   ├─ LLM 调用（2000ms）★ 瓶颈
   ├─ 工具调用（500ms）
   ├─ 审核（100ms）
   └─ 写记忆（50ms，异步）

价值：一眼看出瓶颈、定位故障环节、理解依赖关系
```

## 三、SLO（服务等级目标）

```
SLI（指标）：用什么衡量好坏
  - 首 token 延迟 P95 < 3s
  - 错误率 < 1%
  - 审核通过率 > 95%

SLO（目标）：SLI 要达到多少
  - 首 token 延迟 P95 < 3s 的比例 ≥ 99%
  - 错误率 < 1% 的比例 ≥ 99.5%

SLA（承诺）：违反了赔什么（合同级）

错误预算：
  SLO 99% → 每月可不可用 43min
  用完错误预算 → 冻结新功能发布，专注稳定性
  错误预算驱动 release（不是拍脑袋）
```

**多一个 9 的成本是指数级**：99% → 99.9% 容易，99.9% → 99.99% 要冗余架构 + 多活，要权衡 ROI。

## 四、AI 决策可追溯（特色）

监管要求（生成式 AI 服务管理办法）：**生成内容可追溯、可审计**。

```
每条 AI 输出留痕：
  - user_id / role_id
  - 模型（哪个厂商哪个版本）
  - prompt 版本
  - 输入摘要 + 输出摘要
  - 工具调用记录
  - 审核结果
  - 时间戳
  - trace_id（关联全链路）

应用：
  - 用户投诉角色输出问题 → 按时间/user 查日志 → 定位 prompt 版本/模型
  - 监管核查 → 提供完整生成链路证据
  - 内部质量分析 → 哪些 prompt/模型出问题多
```

## 五、稳定性治理（基于 SLO）

```
治理流程：
  [1] 定义 SLO（业务+工程共识）
  [2] 监控 SLI（实时）
  [3] 错误预算管理
      预算够 → 允许激进迭代
      预算耗尽 → 冻结 release，专注修稳定性
  [4] 故障复盘 → 改进 SLI/SLO
  [5] 季度 review（调整 SLO 目标）

价值：把'稳定性 vs 迭代速度'从吵架变成数据驱动决策。
```

## 六、跨团队推进

```
可观测性数据是跨团队共同语言：
  - 业务：看用户指标（留存/时长/付费）
  - 算法：看 AI 指标（质量分/成本/路由）
  - 工程：看系统指标（延迟/错误/饱和）
  - 运维：看资源指标（CPU/GPU/磁盘）
  - 客服：查用户日志（投诉定位）

统一的指标平台 + 共享看板 → 各团队看到同一份数据 → 协作不扯皮
```

## 七、典型可观测平台架构

```
数据采集层：
  日志：Filebeat（每服务）
  指标：Prometheus（pull）/ OTel Collector
  链路：OpenTelemetry SDK

数据存储层：
  日志：ES / Loki（成本低）
  指标：Prometheus / VictoriaMetrics
  链路：Jaeger / Tempo

展示告警层：
  Grafana（统一看板）
  Alertmanager（告警）
  PagerDuty（值班调度）

AI 特色：
  LLM 调用专属看板（LangSmith/LangFuse/自研）
  prompt 版本对比 / 模型对比 / 成本归因
```

## 八、底层本质

可观测性的本质是**"把复杂系统从黑盒变白盒，让看不见的变可见、感觉的变数字、即兴的变治理"**：

- 三支柱 = 状态可见（日志/指标/链路）
- SLO = 质量可量化（数字驱动）
- AI 追溯 = 决策可审计（合规）
- 治理 = 迭代可控制（错误预算）
- 跨团队 = 协作有共识（共同数据）

这是把'调 AI'升级为'运营 AI'的核心能力——AI 时代因为模型是黑盒、决策是概率性的，可观测性比传统服务更重要。**AI 决策可追溯**更是合规底线，是 AI 产品能否合规商用化的硬要求。

## 常见考点

1. **三支柱数据量大成本高怎么办？**——日志采样（高频低价值事件采样存）+ 冷热分层（热 ES 冷 OSS）+ 指标降采样（历史数据粗粒度）+ 链路采样（1-10%）；ROI 评估，不是全量。
2. **SLO 怎么和业务对齐？**——从用户视角定 SLI（不是工程视角），如'首字延迟'比'CPU 利用率'更贴用户感受；定期和业务 review SLO 是否还合理。
3. **AI 决策追溯怎么平衡隐私和审计？**——输入输出脱敏存（不存原文，存摘要/哈希）+ 敏感字段加密 + 访问审计（员工查追溯数据也要被审计）+ 严格权限分级。

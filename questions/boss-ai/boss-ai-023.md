# 【巨剧核 AI 陪伴】RAG 在陪伴产品怎么用（角色/剧情/用户）？

> JD 依据："熟悉 RAG、向量检索；角色知识库、剧情设定、用户画像。"

## 一、AI 陪伴的 RAG 场景

```
场景 1：角色设定检索
  用户："小雨你小时候最喜欢干嘛"
  检索：角色档案中"童年"相关片段
  注入：LLM 基于设定回答

场景 2：剧情设定检索
  用户："我们上次玩到哪了"
  检索：用户当前剧情进度 + 周边剧情细节
  注入：LLM 复述剧情并引导继续

场景 3：用户画像检索
  用户："我之前跟你说过的工作烦心事"
  检索：用户记忆中"工作"相关事件
  注入：LLM 引用用户历史共情回应
```

## 二、三类知识库

| 知识库 | 内容 | 来源 | 更新 |
|---|---|---|---|
| 角色知识库 | 人设/背景/喜好/关系/口吻 | 创作者设定 | 偶尔（版本化） |
| 剧情知识库 | 剧情/分支/事件/角色参与 | 编剧/UGC | 版本化 |
| 用户画像库 | 用户偏好/历史事件/承诺 | 对话抽取 | 持续（高频） |

用户画像库 ≈ 记忆系统（见记忆题），RAG 在这里是其检索能力。

## 三、RAG 流程

```
┌────────────────────────────────────────┐
│ 离线建库                                 │
│  原始文档 → 切块 → 嵌入 → 入向量库         │
└────────────────────────────────────────┘
                ↓
┌────────────────────────────────────────┐
│ 在线检索                                 │
│  query → embedding                       │
│        ↓                                 │
│  向量召回（top-N）+ 关键词召回（BM25）     │
│        ↓                                 │
│  重排（cross-encoder rerank）             │
│        ↓                                 │
│  取 top-K                                │
└────────────────────────────────────────┘
                ↓
┌────────────────────────────────────────┐
│ 注入 + 生成                               │
│  prompt + 检索结果 + 用户输入 → LLM        │
└────────────────────────────────────────┘
```

## 四、建库（切块与嵌入）

```
切块策略：
  角色：按设定字段（基本资料/性格/喜好/关系）切
  剧情：按章节/场景切
  用户：按事件/主题切（已在记忆系统持久化）

嵌入：
  bge-large-zh / OpenAI text-embedding-3 / 自训练
  中文场景推荐 bge 系列

存储：
  Milvus / Qdrant / pgvector（中小规模）
  按 user_id/role_id/类别 分 collection
```

## 五、检索（多路融合）

```
向量召回：
  query embedding → ANN 检索（HNSW）→ top-N
  优：语义匹配；缺：可能召回形似神不似

关键词召回（BM25）：
  query 分词 → 倒排索引 → top-N
  优：精确匹配专有名词；缺：不懂语义

混合检索：
  向量 + 关键词 并行召回 → 融合排序
 互补，覆盖更全

重排：
  cross-encoder（bge-reranker 等）对小集合精确打分
  比 bi-encoder（向量检索）更准但更慢
```

## 六、召回四要：全/准/新/相关

```
全：相关内容不漏（多路召回提升）
准：召回内容真的相关（重排 + 阈值过滤）
新：最新信息（用户画像高频更新；时间衰减旧记忆）
相关：语义对（embedding 模型质量 + 多 query 重写）

平衡：
  召回多了 → 注入太多 → prompt 爆炸 + 注意力稀释
  召回少了 → 漏关键信息 → 回答错
  要调阈值和 top-K
```

## 七、注入策略

```
prompt 结构：
  [角色人设]
  [检索到的相关知识 top-K]
  [用户记忆摘要]
  [近期对话]
  [用户输入]
  → LLM

强指令：
  "请严格基于以下设定/记忆回答，不要编造："
  防止 LLM 忽略检索结果瞎编
```

## 八、记忆 vs RAG（区别和联系）

```
记忆（个性化、可演进）：
  按 (user, role) 隔离
  会更新（事实演进）
  高度个性化
  → 用户画像库本质是记忆

RAG（通用知识、可复用）：
  跨用户共享（角色设定/剧情库）
  相对静态（版本化更新）
  通用知识
  → 角色知识库/剧情库是 RAG

技术栈同源（向量检索 + 嵌入 + 注入）
场景和治理不同
```

## 九、评估

```
检索层：
  召回率（相关内容是否被召回）
  准确率（召回的是否真的相关）
  延迟（P95）

端到端：
  人工评分（基于检索结果回答得好不好）
  LLM-as-Judge
  业务指标（满意度/留存）

回归：
  建评测集，每次模型/嵌入/prompt 改动回归
```

## 十、底层本质

RAG 在陪伴产品的本质是**"给角色装私人图书馆，按需翻书而不是背书"**：

- 建库 = 知识可管理
- 检索 = 按需取用
- 注入 = 让 LLM 基于事实回答
- 多路 + 重排 = 全且准

这是把'LLM 自带知识'扩展为'LLM + 外部知识'——是角色显得'懂自己/懂剧情/懂用户'的核心技术，是 AI 陪伴区别于通用聊天机器人的关键工程。

## 常见考点

1. **RAG 检索不准怎么优化？**——多路召回（向量+关键词）+ 重排（cross-encoder）+ query 改写（LLM 生成变体）+ 调嵌入模型 + 优化切块（更细粒度）+ 用户反馈回流。
2. **大量知识库怎么治理？**——分类管理（角色/剧情/用户分库）+ 版本化 + 权限（创作者/管理员）+ 过期清理 + 元数据标签。
3. **RAG 会让回复变慢吗？**——会。向量检索亚秒级 + 重排几十毫秒 + 注入后 prompt 变长（推理稍慢）；优化：并行召回 + 缓存 + 异步预召回 + 控 top-K。

# 【巨剧核 AI 陪伴】音视频 AI 生成任务平台架构怎么设计？

> JD 依据："负责短剧视频服务端、AI 生成任务；熟悉音视频或 AI 生成异步任务。"

## 一、平台整体架构

```
┌────────────────────────────────────────────────────┐
│ 业务接入：对话/剧情/社区触发任务                       │
├────────────────────────────────────────────────────┤
│ 任务平台 API                                        │
│   submit / query / cancel / callback                │
├──────────────┬─────────────┬────────────────────┤
│ 任务调度器     │ 算子编排引擎  │ 状态机 + 持久化       │
│ 优先级/队列/路由│ DAG 执行     │ task → step → 状态   │
├──────────────┴─────────────┴────────────────────┤
│ GPU 资源池（K8s 集群）                              │
│  TTS 池 / 渲染池 / 视频合成池 / 通用池              │
├───────────────────────────────────────────────────┤
│ 存储：任务 DB / 中间产物 OSS / 进度 Redis / 日志     │
└───────────────────────────────────────────────────┘
```

## 二、任务模型与状态机

```
Task {
  task_id, user_id, role_id, type, priority, idempotency_key
  input, params
  status: CREATED|QUEUED|RUNNING|SUCCESS|FAILED|TIMEOUT|CANCELED
  steps: [{ name, status, started_at, ended_at, error }]  ← 算子级状态
  progress, result_url, cost_tokens
  created_at, timeout_at
}
```

状态机 + 步骤明细 = 任何时刻知道任务跑到哪一步、能不能续。

## 三、算子 DAG 编排

复杂任务是算子组合：
```
"语音输入 → 数字人视频回复"

[ASR]──→ [LLM]──→ [TTS]──┐
                          ├──→ [合成]──→ [输出]
                  [渲染]──┘
```

DAG 描述：
- 节点 = 算子（带类型、参数、超时、重试策略、资源需求）
- 边 = 数据依赖
- 支持并行（TTS 和渲染可并行）

**框架**：Temporal（持久化执行）/ Airflow（批处理）/ 自研。AI 任务推荐 Temporal（断点恢复强）。

## 四、GPU 资源调度

```
资源池：
  按算子类型分池（TTS 池/渲染池/合成池）
  或通用池 + 多队列

调度策略：
  优先级（VIP 高/普通/批量低）
  公平调度（防单用户占满）
  抢占（VIP 可抢普通任务的 GPU）
  弹性扩缩容（队列堆积触发扩节点）
  亲和性（同类任务集中调度，热缓存）

实现：K8s + 自定义调度器 + 设备插件
```

## 五、可靠性与恢复

```
持久化：
  每步状态写 DB
  中间产物存 OSS（断点可重用）
  
恢复：
  节点故障 → 任务重调度到健康节点
  算子失败 → 按策略重试/换算子/降级/整体失败
  服务重启 → 加载 RUNNING 任务 → 从最后 SUCCESS 步骤继续

幂等：
  算子执行带 idempotency_key
  重试不会产生副作用（如不重复扣费）
```

## 六、进度推送与回调

```
[1] 同步返回：submit → 立即返回 task_id
[2] 异步进度：
    WebSocket 实时推送（在线用户）
    webhook 回调（业务系统）
    轮询兜底（query API）
[3] 结果：完成 → 推 result_url + 通知（IM/Push）
```

进度算法：DAG 节点完成数加权 + 算子内部细粒度上报（如 50/100 帧渲染完）。

## 七、计费与成本

```
计费维度：
  算子类型 × 执行时长 × 资源规格 × 数量
  例：数字人渲染 1min × 1GPU × 单价 = X 元

归因：
  cost 按 task_id → user_id → 角色会话 → 业务场景
  支持 user 维度账单、场景维度成本分析

预扣 + 实结：
  创建任务锁额度
  完成按实际算
  失败退还（部分失败按比例）
```

## 八、可观测

```
任务级：
  生命周期 trace（每步耗时）
  成功率/失败原因分布
  P50/P95 端到端时延
资源级：
  GPU 利用率/队列深度/扩缩容事件
  各算子成本占比
业务级：
  各场景任务量/转化/ROI
```

## 九、底层本质

音视频平台架构的本质是**"把一次性 API 调用，重构为可调度、可恢复、可计费、可观测的工业流水线"**：

- 任务抽象 = 解耦提交和执行
- DAG 编排 = 复杂流程可描述
- GPU 调度 = 稀缺资源最优利用
- 状态机 = 可恢复 + 可追溯
- 计费归因 = 可运营

这是把'AI 模型能力'转化为'可商业化产品'的核心工程，是 Sora/Midjourney/数字人/短剧生成产品的共同底座。

## 常见考点

1. **GPU 任务怎么扩容？**——K8s cluster autoscaler + GPU 节点池；按队列堆积触发扩容；缩容要安全驱逐（等任务完成）；预留缓冲应对突发。
2. **同一任务重复提交怎么防？**——idempotency_key 去重（短时内同 input+params 返回旧结果）；前端按钮防抖；用户配额限制。
3. **长任务（>10min）怎么不超时？**——任务级超时（10-30min）+ 算子级超时（每步独立）+ 中间产物持久化（断了能续）+ 流式进度（用户感知在动）。

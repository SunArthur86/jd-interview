# 【巨剧核 AI 陪伴】AI 陪伴产品整体架构怎么设计？

> JD 依据："负责产品服务端核心架构，包括用户、角色、故事、消息、记忆、社区、发布系统全链路。"

## 一、整体架构分层

```
┌──────────────────────────────────────────────────────┐
│ 接入层：App / Web / 小程序 / SDK                       │
├──────────────────────────────────────────────────────┤
│ 网关层：API Gateway / 鉴权 / 限流 / 路由                │
├──────────────────────────────────────────────────────┤
│ 业务服务层（微服务，按能力拆）：                          │
│  ┌────────┬────────┬────────┬────────┬────────┐      │
│  │ 用户系统│ 角色系统│ 故事系统│ 消息系统│ 社区系统│      │
│  ├────────┴────────┴────────┴────────┴────────┤      │
│  │ 发布系统 / 通知系统 / 支付 / 审核           │      │
│  └────────────────────────────────────────────┘      │
├──────────────────────────────────────────────────────┤
│ 编排层（Agent 引擎）：上下文组装 + 模型路由 + 工具 + 审核│
├──────────────────────────────────────────────────────┤
│ 模型层：LLM 网关 / TTS / ASR / 数字人 / 视频生成        │
├──────────────────────────────────────────────────────┤
│ 存储层：MySQL / Redis / 向量库 / 对象存储 / Kafka       │
└──────────────────────────────────────────────────────┘
```

## 二、六大子系统职责

### 1. 用户系统
- 账号 / 资料 / 偏好 / tier（免费/付费/VIP）
- 用户画像（影响角色互动）
- 关系网络（用户↔角色关系演进：陌生→熟悉→亲密）

### 2. 角色系统
- 角色元数据（人设/外貌/声音/边界）
- 角色版本化（人设迭代、A/B）
- 角色与用户的"关系状态"（亲密度/剧情进度）
- 角色市场（创作者生态：UGC/AIGC 角色）

### 3. 故事系统
- 剧情库（短剧/互动剧情模板）
- 剧情进度（用户玩到哪了）
- 分支选择（多结局）
- 剧情↔角色绑定（一个剧情可有多角色）

### 4. 消息系统（IM）
- 1v1 私聊（用户↔角色）
- 群聊（用户 + 多角色 / 多用户剧情）
- 流式回复（LLM token 流）
- 离线推送（见消息系统题）

### 5. 记忆系统
- 短期（会话原文）
- 长期（结构化事实 + 向量片段）
- 召回（影响 Agent 编排）
- 见记忆系统题

### 6. 社区系统
- 内容发布（用户生成的剧情/角色/截图）
- Feed 流（关注/推荐）
- 互动（点赞/评论/分享）
- 创作者中心

## 三、编排层（Agent 引擎）—— 中枢

```
用户消息 → 消息系统 → Agent 引擎：
   [1] 取角色人设 + 剧情进度（角色/故事系统）
   [2] 取用户画像 + 召回记忆（用户/记忆系统）
   [3] 组装 prompt + 模型路由（模型层）
   [4] 工具调用（如搜索剧情/查关系）
   [5] 输出审核
   [6] 写记忆 + 更新关系 + 返回消息（消息系统）
```

编排层是**业务逻辑中枢**，但本身无状态（状态都在各子系统），可水平扩容。

## 四、三条数据流

```
1. 实时对话流（毫秒-秒级）：
   用户 → 接入 → 编排 → LLM → 审核 → 消息回推
   要求：低延迟、流式

2. 异步生成流（秒-分钟级）：
   用户触发 → 任务队列 → TTS/数字人/视频 → 回调通知
   要求：可靠、可恢复

3. 数据分析流（异步批）：
   埋点 → Kafka → 数仓 → 计算 → BI/推荐/实验
   要求：全量、可追溯
```

三流分离，互不阻塞。

## 五、关键设计：解耦与演进

- **角色 vs 故事解耦**：一个角色可参演多故事，一个故事可有多角色（多对多）
- **消息 vs 编排解耦**：消息系统不感知 AI 逻辑，只负责投递（角色也是消息发送方）
- **记忆 vs 编排解耦**：记忆作为独立服务，编排按需调用，记忆演进不影响编排
- **社区 vs 主链路解耦**：社区故障不影响 1v1 对话

## 六、多租户与权限

```
数据隔离：
  会话/记忆按 (user_id, role_id) 隔离
  角色市场 UGC 按创作者隔离
权限：
  用户/创作者/管理员分权
  数据权限（行级过滤）
  AI 工具调用前校验
```

## 七、扩展性：新增功能怎么加

例如新增"角色直播"：
- 接入层：加直播入口
- 业务层：新增直播服务（复用消息/角色系统）
- 编排层：扩展主动触发能力（角色主动开播通知）
- **不需要改主链路** —— 这就是好架构的标志。

## 八、底层本质

AI 陪伴架构的本质是**"按业务能力拆解 + 编排层中枢调度 + 三流分离 + 解耦演进"**：

- 拆解 = 各子系统独立演进
- 编排 = 业务逻辑统一调度
- 三流 = 实时/异步/数据互不阻塞
- 解耦 = 故障隔离、可扩展

这是把'一个 AI 应用'升级为'可支撑亿级关系经营的平台'——核心是**关系建模**（用户↔角色是多维演进的关系网络，不是简单的 CRUD）。

## 常见考点

1. **六大系统都自研 vs 用 SaaS？**——差异化能力（角色/故事/记忆/编排）自研是护城河；通用能力（IM/支付/审核）可借 SaaS 快速起步，后期替换。
2. **编排层无状态怎么保持对话连贯？**——状态存各子系统（消息存会话历史、记忆存长期、角色存人设），编排每次从子系统加载组装，本身无状态。
3. **亿级关系网络（用户↔角色）怎么存？**——图数据库（Neo4j/JanusGraph）或关系表分库分表；按 user_id 分片；关系状态存 Redis（亲密度等高频字段）。

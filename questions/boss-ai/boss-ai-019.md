# 【巨剧核 AI 陪伴】预案、回滚与变更管理怎么做？

> JD 依据："负责稳定性：预案、回滚。钱学森工程理论：技术可行、经济合理、周期短、协调运转。"

## 一、为什么必须有预案

```
无预案：故障发生 → 值班慌乱 → 瞎试 → 错操作放大故障 → 长时间不可用
有预案：故障发生 → 告警分级 → 按 SOP 处置 → 一键回滚 → 快速恢复
```

故障是必然的，**慌乱放大故障是管理问题**。

## 二、预案体系

### 1. 故障分级
```
P0（电话告警）：
  服务全挂 / 隐私泄漏 / 监管事件
  响应：< 5min，全体值班 + 主管

P1（IM 告警）：
  核心指标劣化（错误率 > 5% / 留存跌）
  响应：< 15min，值班同学

P2（日报）：
  非核心异常 / 长期趋势
```

### 2. SOP（标准处置手册）
```
每类故障一份 SOP：
  - 症状（怎么发现）
  - 影响（哪些功能/用户）
  - 排查步骤（查什么日志/监控）
  - 处置（按顺序的操作）
  - 升级（搞不定找谁）
  - 事后（复盘）

例：LLM 主模型挂
  症状：错误率飙升
  排查：查 LLM 网关日志，确认是哪一家挂
  处置：1. 切 fallback 厂商 2. 限流降级 3. 必要时熔断主模型
  升级：5min 未恢复 → 通知主管
```

### 3. 演练（混沌工程）
```
定期注入故障：
  杀节点 / 网络延迟 / 依赖挂 / CPU 打满
验证：
  - 监控是否发现
  - 自动降级是否生效
  - 值班 SOP 是否有效
  - 回滚是否可用

不演练的预案 = 没有。
```

## 三、回滚通道

```
四类回滚：
  [1] 代码回滚：发版 → 发现 bug → 重新部署上一版镜像
  [2] 配置回滚：配置中心历史版本 → 一键切回
  [3] 灰度逆转：灰度比例从 50% 切回 0%
  [4] 数据回滚：DB 备份恢复 / 反向脚本

前置条件：
  - 镜像版本保留（不删历史 tag）
  - 配置版本化（配置中心支持）
  - feature flag 开关（业务功能可秒级关）
  - DB 变更前向兼容（不破坏旧版本运行）
```

**最难的是数据回滚**：要前向兼容设计（schema 加字段不删字段）+ 关键变更前备份 + 反向脚本。

## 四、变更管理（防患于未然）

```
变更原则：
  - 可灰度：1% → 10% → 50% → 100%
  - 可观测：灰度期间核心指标看板
  - 可回滚：发现问题立即切回
  - 可关闭：feature flag 控制功能开关

变更窗口：
  - 不在高峰发版
  - 不在节假日前夕（无人值班）
  - 重大变更需审批 + 值班盯

发布平台：
  - 自动化流水线（CI/CD）
  - 灰度控制台
  - 一键暂停/回滚
```

## 五、钱学森工程四准则

钱学森的系统工程理论提出方案的**四条评估标尺**——这套准则完美适用于工程方案决策：

```
1. 技术可行：方案技术上能实现吗？依赖是否具备？
   例：用自部署大模型替代 API → 技术可行吗？GPU 够吗？

2. 经济合理：成本-收益比合理吗？
   例：全量切最强模型 → 成本爆炸 → 不合理

3. 周期短：能在要求时间内完成吗？
   例：自建 GPU 集群要几个月 → 应急用不上

4. 协调运转：和现有系统/团队/流程协调吗？
   例：新方案要换语言栈 → 团队不熟 → 协调成本高
```

**举例：选 LLM 接入方案**

| 方案 | 技术可行 | 经济合理 | 周期短 | 协调运转 |
|---|---|---|---|---|
| 全用 GPT-4 API | ✓ | ✗（贵） | ✓ | ✓ |
| 全自部署 Llama | ✓ | △（GPU 贵） | ✗（采购月） | △（要运维） |
| 混合（API+自部署） | ✓ | ✓ | ✓ | ✓ |

四准则都打勾的方案，才是工程上正确的选择——不是技术上最先进，而是综合最优。

## 六、AI 陪伴的特色预案

```
模型类：
  - LLM 厂商挂 → fallback 厂商 + 降级到便宜模型
  - 模型输出质量劣化 → 版本回滚 prompt + 灰度切回
  - 角色人设崩（某 prompt 版本导致） → 紧急回滚 prompt 版本

数据类：
  - 记忆系统挂 → 降级（短期内存兜底，长期空记忆继续）
  - 向量库性能差 → 降级（少召回或跳过）

业务类：
  - 审核系统挂 → 高风险场景暂停（保守）+ 人工加审
  - GPU 不够 → 限流 + 降级（切便宜模型）
  - 角色爆火 → 弹性扩容 + 限流 + 降级
```

## 七、事后复盘

```
复盘四步：
  [1] 时间线：什么时候发现/处置/恢复
  [2] 根因：5 whys 深挖（不是表面原因）
  [3] 改进：技术（加固）+ 流程（SOP）+ 组织（值班/权限）
  [4] 跟踪：改进项 owner + deadline，定期 review

文化：
  - 对事不对人（不甩锅）
  - 重视改进而非追责
  - 每次故障都是系统进化的机会
```

## 八、底层本质

预案回滚的本质是**"把故障应急从即兴发挥升级为有备而来"**：

- 预案 = 想清楚（事前）
- SOP = 写下来（标准化）
- 演练 = 练熟了（肌肉记忆）
- 回滚 = 逃生通道（兜底）
- 复盘 = 持续进化（学习）

钱学森四准则（技术可行/经济合理/周期短/协调运转）是评估**任何工程方案**的通用标尺——好方案不是技术最炫的，而是综合最优的。这是把'工程师思维'升级为'系统工程思维'。

## 常见考点

1. **回滚比修复更快时怎么办？**——优先回滚（先恢复服务），修复后重新灰度；不要在故障现场调试（用户在等）。
2. **数据回滚有什么风险？**——可能丢数据（用户已提交的）/ 影响其他依赖此数据的系统 / 反向脚本本身有 bug；优先前向兼容设计避免数据回滚。
3. **AI 模型版本回滚怎么做？**——prompt 版本在配置中心（一键切回）/ 模型路由配置（切回旧模型）/ 灰度比例逆转；feature flag 控制 prompt 版本绑定。

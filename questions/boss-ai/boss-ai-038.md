# 【巨剧核 AI 陪伴】FDE 视角：AI 陪伴项目从需求到落地怎么设计？

> JD 依据："FDE 解决方案：AI 陪伴项目从需求到落地全链路。"

## 一、FDE 视角

```
普通工程师视角：
  给我具体技术问题，我解
  关注：技术深度

FDE 视角：
  给我业务问题，我设计端到端方案
  关注：业务价值 + 技术选型 + 工程实施 + 持续运营

FDE 要会：
  - 业务理解（客户要什么）
  - 方案设计（怎么解）
  - 技术广度（什么工具能用）
  - 沟通协作（和技术/产品/客户）
  - 落地推进（从方案到上线）
```

## 二、第一步：需求理解（业务驱动）

```
客户说："我要做个 AI 陪伴 App"
你要问：
  - 目标用户是谁？（年龄/画像/场景）
  - 核心价值主张？（情感陪伴 / 娱乐 / 教育 / 心理）
  - 差异化？（vs Character.ai / 星野 / 豆包）
  - 商业模式？（订阅/内购/广告）
  - 关键指标？（DAU/留存/时长/付费率）
  - 预算/时间约束？
  - 合规要求？（备案/数据本地化）

输出：业务需求文档（BRD）
  - 用户故事
  - 核心场景
  - 成功标准（指标）
  - 约束条件
```

## 三、第二步：方案选型（每层决策）

```
从需求反推每层选什么：

[1] 接入层
  App（iOS/Android）+ Web + 小程序？
  → 选 React Native 跨平台 + Web

[2] 业务架构
  AI 陪伴核心：用户/角色/故事/消息/记忆/社区
  → 见 011 整体架构题

[3] 编排层
  自研 vs LangGraph？
  → AI 陪伴场景自研（人设/记忆定制深）

[4] 模型层
  API vs 自部署？哪家？
  → 起步 OpenAI/通义 API，规模化混合
  → 见模型路由题

[5] 存储层
  MySQL（业务）/ Redis（缓存/会话）/ 向量库（记忆）
  → pgvector 起步，规模化 Milvus

[6] 音视频
  TTS/ASR/数字人自研还是供应商？
  → 起步供应商（成熟），规模化自研核心

[7] 部署
  云（AWS/阿里）vs 自建？
  → 公有云起步，规模化混合云
```

**选型原则（钱学森四准则）**：
- 技术可行（依赖是否具备）
- 经济合理（成本-收益）
- 周期短（能在要求时间内）
- 协调运转（和团队/现有系统协同）

## 四、第三步：架构设计

参考 011 题的整体架构。

核心原则：
```
分层 + 解耦 + 可演进

分层：接入/网关/业务/编排/模型/存储
解耦：各子系统独立演进、独立扩容、故障隔离
可演进：新功能不需要改主链路
```

## 五、第四步：MVP 落地（小步快跑）

```
MVP（最小可行产品）原则：
  最小核心功能 → 最快验证 → 最少投入

阶段 0：原型验证（2-4 周）
  - Prompt 角色测试
  - 核心对话流
  - 体验调研

阶段 1：MVP（2-3 月）
  - 1-3 个核心角色
  - 1v1 文字对话
  - 基础记忆（短期）
  - 简单 UI
  目标：验证用户是否愿意用

阶段 2：增强（3-6 月）
  - 语音对话（TTS/ASR）
  - 长期记忆
  - 角色/剧情扩展
  - 社区
  目标：留存和时长

阶段 3：规模化（6-12 月）
  - 数字人/视频
  - 多角色群聊
  - 创作者生态
  - 商业化
  目标：付费和规模

不要一上来全做——MVP 验证方向再扩展。
```

## 六、第五步：运营迭代（闭环）

```
上线只是开始：
  数据采集 → 分析 → 假设 → 实验 → 迭代

闭环：
  [埋点] 用户行为数据
  [实验] A/B 各功能/prompt/模型
  [评估] 业务指标 + AI 质量
  [迭代] 优化 prompt/模型/算法/产品
  [监控] 漂移/故障/SLO

数据驱动决策：
  留存不够 → 分析流失节点 → 实验 → 改进
  时长不够 → 分析互动深度 → 实验 → 改进
```

## 七、典型方案示例

```
客户：新兴公司，预算 500w，6 个月做出 MVP

方案：
  接入：React Native（双端）+ Web
  架构：Node.js/TypeScript 微服务 + K8s
  编排：自研 Agent 引擎（TypeScript）
  模型：起步通义 API（中文好+便宜），规模化混合自部署
  存储：MySQL + Redis + pgvector（起步）
  音视频：阿里云智能语音（起步）
  部署：阿里云（合规友好）
  审核：阿里云内容安全 + 自研人设审核
  监控：Prometheus + Grafana + LangFuse

成本估算：
  人力：30 人 × 6 月 = ~ 500w
  云资源：~ 50w
  API/数据：~ 50w

里程碑：
  M1：原型（Prompt/对话流）
  M2：MVP（1v1 文字+语音）
  M3：增强（记忆+多角色）
  M4：上线 + 数据闭环
```

## 八、关键决策点（举几个例子）

### 决策 1：API 还是自部署？
```
因素：
  QPS、成本、隐私、运维能力
MVP：API（弹性、无运维）
规模化：核心场景自部署 + API 备用
```

### 决策 2：通用 LLM 还是微调？
```
通用：起步快，效果不差
微调：差异化（人设/语气/特定任务）
MVP：通用 + 强 prompt
有数据后：关键场景微调
```

### 决策 3：自研还是 SaaS？
```
核心差异化能力（编排/记忆/角色）：自研
通用能力（IM/支付/审核）：SaaS 起步，后替换
```

## 九、风险与应对

```
技术风险：
  模型效果不达预期 → A/B 多模型/prompt 优化
  成本失控 → 限流/降级/路由优化

业务风险：
  用户不感兴趣 → 早期 MVP 验证
  留存差 → 数据驱动迭代

合规风险：
  内容审核 / 备案 / 数据隐私
  → 早期介入，预留预算

竞争风险：
  同质化 → 差异化（人设/记忆/情感）
```

## 十、底层本质

FDE 解决方案的本质是**"把零散技术点串成端到端可落地的方案，从业务需求出发到持续运营闭环"**：

- 需求驱动 = 业务价值优先
- 方案选型 = 每层择优（不是最先进是合适）
- MVP = 小步验证降风险
- 闭环 = 数据驱动持续改进

这是从'工程师'升级为'解决方案架构师/FDE'的核心能力——技术深度+业务理解+方案设计+落地推进+运营迭代的综合。前面 37 题的技术点，最终都通过 FDE 视角串成一个完整可落地的方案。

## 常见考点

1. **客户预算不够怎么砍？**——按价值排序：核心对话/记忆 > 多模态 > 社区；先做高 ROI 部分；用 SaaS 替代自建；阶段性投入（先 MVP 再扩）。
2. **怎么估算项目周期？**——按功能点拆解（角色/记忆/音视频各多少 person-month）+ 阶段并行 + 缓冲（20-30%）；参考历史项目。
3. **方案差异化怎么做？**——人设深度（自研 Agent）/ 长期记忆（让角色真懂你）/ 多模态（数字人）/ 创作者生态（UGC）/ 情感计算（共情质量）；不是卷参数而是卷体验。

# 【巨剧核 AI 陪伴】记忆与上下文系统架构怎么设计？

> JD 依据："负责记忆系统、长期记忆、会话状态；熟悉向量检索、分层记忆。"

## 一、整体架构

```
┌────────────────────────────────────────────────────┐
│ 业务层（编排/对话/推荐）通过 MemoryService 读写          │
├────────────────────────────────────────────────────┤
│ MemoryService API                                   │
│   write(user, role, content) / recall(user, role, query) │
├──────────────────┬─────────────────────────────────┤
│ 写入管线           │ 召回引擎                         │
│ 抽取→去重→合并→分发│ 多路召回→融合排序→top-K          │
├──────────────────┴─────────────────────────────────┤
│ 存储层（多存储组合）                                  │
│  ┌──────────┬──────────┬──────────┬──────────┐      │
│  │ Redis    │ MySQL    │ 向量库    │ 对象存储  │      │
│  │ 短期原文  │ 结构化事实│ 语义片段  │ 原始对话  │      │
│  └──────────┴──────────┴──────────┴──────────┘      │
└────────────────────────────────────────────────────┘
```

## 二、分层存储设计

### 1. 短期记忆（Redis）
```
key: session:{user_id}:{role_id}
value: ZSet（按 seq 排序的最近 N 轮原文）
TTL: 24h-7d
用途：高频访问、当前会话上下文
```

### 2. 结构化事实（MySQL + 分库分表）
```
user_profile: 用户画像（年龄/偏好/性格/关系节点）
role_facts: (user_id, role_id) 维度的关键事实
  字段：fact_key, fact_value, confidence, ts, version, expired
关系网络：user↔role 关系深度、事件节点（可入图库）

按 user_id 分库分表，亿级用户扛得住
```

### 3. 向量片段库（Milvus/Qdrant）
```
collection: memory_chunks
  字段: id, user_id, role_id, embedding, content, ts, importance, ...
  索引: HNSW（ANN）
分区: 按 user_id 分片（避免跨用户扫描）
规模: 亿-百亿级向量
```

### 4. 原始归档（对象存储 + 数仓）
```
所有原始对话冷归档（OSS/S3）
数仓做长期分析（用户行为/模型评估/合规追溯）
```

## 三、写入管线

```
对话轮次
   ↓
[1] 关键信息抽取（LLM 调用）
    提取：实体/偏好/承诺/情绪事件/关系变化
   ↓
[2] 过滤
    去寒暄/重复/无信息量
   ↓
[3] 去重合并
    与已有事实比对：相同→跳过/补充；矛盾→软更新（新版本）
   ↓
[4] 打标
    ts, importance, source_turn, (user_id, role_id), PII 脱敏
   ↓
[5] 分发写入
    结构化 → MySQL
    片段 + embedding → 向量库
    原文 → 短期 Redis + 冷归档
```

**异步管线**：写入对主对话流不阻塞（用户不等写入完成），写完即可。

## 四、召回引擎

```
recall(user, role, query):
   ↓
并行走多路：
  [向量召回] query embedding + filter(user, role) → top-K 片段
  [结构化查询] SELECT 用户画像/最近承诺/关键事件
  [时效召回] 最近 24h/7d 的重要事件
  [剧情召回] 当前剧情进度相关的设定
   ↓
融合排序：
  score = α·相关性 + β·重要性 + γ·时效性 + δ·情感权重
   ↓
top-N 返回 → 注入编排层 prompt
```

**性能优化**：
- 向量库 ANN（HNSW/IVF）→ 亚秒级
- 热门 query 结果缓存（如用户基础画像）
- 召回条数有上限（防 prompt 爆炸）
- 召回失败降级（空记忆也能继续对话）

## 五、记忆演进机制

```
更新策略：
  软更新：新事实覆盖旧，旧标记 expired（保留版本历史）
  矛盾处理：带置信度 + ts，下次召回按"最新可信"返回
  衰减：久未访问降权（类似 PageRank + 时间衰减）
  重要性保护：高重要性（承诺/关系节点）不衰减

例：
  旧：{怕黑: true, ts: 2024}
  新：{怕黑: false, ts: 2026}
  → 召回时返回新版本 + "曾经怕黑，现在不怕了"
```

## 六、隔离与安全

```
严格隔离：
  所有查询带 (user_id, role_id) 过滤
  禁止跨用户召回（防泄漏）
  角色间默认隔离（A 角色不见 B 角色的隐私）
  （可选）共享空间：用户基础画像可跨角色（用户同意下）

PII 保护：
  写入前脱敏（身份证/电话/银行卡）
  敏感字段加密存储
  日志脱敏展示

合规：
  被遗忘权：用户删除 → 级联删记忆（含向量库）
  数据可导出
  日志保留 90 天+
```

## 七、容量与性能

| 维度 | 短期 | 结构化 | 向量 |
|---|---|---|---|
| 容量 | 7d 滚动 | 亿级用户 | 百亿向量 |
| QPS | 10w+ | 1w | 1k-1w |
| 延迟 | ms | ms-10ms | 10-100ms |

**扩容**：Redis 集群 / MySQL 分库分表 / 向量库分片，按 user_id 水平扩。

## 八、底层本质

记忆系统架构的本质是**"用多存储组合 + 召回引擎，重建人脑的分层记忆机制"**：

- 分层 = 容量/访问频率/精确度的最优匹配
- 多路召回 = 兼顾语义/精确/时效
- 演进 = 像人一样记忆会更新
- 隔离 = 隐私合规

这是把'LLM 无状态'的根本缺陷，用工程系统补足——是 AI 陪伴产品**情感深度**的核心基础设施，是区别'一次性聊天机器人'和'真正陪伴'的分水岭。

## 常见考点

1. **记忆系统挂了对话能进行吗？**——应该能。短期记忆（Redis）必须有降级（内存兜底）；长期召回失败用空记忆继续；记忆系统故障不阻塞主链路（降级而非熔断）。
2. **向量库怎么选 + 亿级怎么扛？**——Milvus/Qdrant 支持分布式分片；按 user_id 分片（避免广播查询）；HNSW 索引精度-速度可调；定期 compaction。
3. **如何评估记忆系统好不好？**——召回准确率（人工标注）+ 召回延迟 P95 + 写入吞吐 + 用户主观评分（"角色懂我"）+ 记忆演进合理性。

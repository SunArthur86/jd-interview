# 【巨剧核 AI 陪伴】多模型路由怎么设计（路由/fallback/成本）？

> JD 依据："负责模型路由；熟悉多模型编排经验；结果返回。"

## 一、为什么需要路由

```
全用强模型（GPT-4）：体验好，但成本爆炸、延迟高、高峰扛不住
全用弱模型（GPT-3.5）：便宜快，但复杂场景质量崩、用户流失
固定单模型：上述两个问题的混合
```

本质矛盾：**不同请求难度不同 + 不同模型各有优劣**，必须按需匹配。

## 二、路由决策维度

```
请求 → 提取特征：
  ┌─────────────────────────────────┐
  │ 场景：闲聊 / 情感支持 / 剧情推进 / 工具任务 │
  │ 复杂度：简单 / 中等 / 复杂推理         │
  │ 延迟敏感度：实时（流式）/ 异步         │
  │ 成本预算：用户 tier 决定              │
  │ 历史特征：同类 query 的 token 消耗     │
  └─────────────────────────────────┘
   ↓
决策 → 选模型
```

## 三、决策方式

```
L1 规则路由（最快）：
  if 场景 == "闲聊" and 长度 < 20:  → 小模型
  if 场景 == "情感支持":             → 强共情模型
  if 场景 == "工具任务":             → 强 function calling 模型

L2 分类模型路由（更准）：
  训练一个小分类器（轻量 BERT / 小 LLM）预测难度等级 → 按等级选模型

L3 强化学习路由（进阶）：
  以（质量分 - 成本）为 reward，学习每个请求的最优路由策略
```

实战：L1 兜底 + L2 主力，L3 少量头部产品用。

## 四、Fallback 链（可用性）

```
请求 → 主模型（GPT-4）
       ├─ 超时/限流/5xx → 备模型（Claude）
       │                  ├─ 失败 → 兜底（通义/自部署小模型）
       │                  │         └─ 失败 → 模板回复（"我有点累，稍后再聊"）
```

每级设：
- 超时阈值（主 3s，备 5s）
- 熔断（错误率 > 阈值自动降级一段时间）
- 自动切换 + 告警

## 五、成本控制

```
配额层：
  免费 user：每天 100 条，全走小模型
  付费 user：无限量，强模型优先

预算层：
  单请求 token 上限（防止 prompt 爆炸）
  单用户日预算（超了降级）

降级层（高峰/成本告急）：
  全网切便宜模型 / 拒绝长上下文请求 / 关闭高成本功能（视频生成）

监控：
  实时成本 dashboard → 超阈值告警 → 自动触发降级策略
```

## 六、典型路由表

| 场景 | 主模型 | 备 | 兜底 | 理由 |
|---|---|---|---|---|
| 简单闲聊 | qwen-turbo | gpt-4o-mini | 模板 | 便宜快 |
| 情感支持 | claude-sonnet | gpt-4o | qwen | 共情强 |
| 剧情推理 | gpt-4o | claude-opus | - | 复杂推理 |
| 工具调用 | gpt-4o | claude | - | function calling 稳 |
| 内容生成 | 通义/文心 | gpt-4o | - | 中文好、便宜 |

## 七、可观测

```
路由日志：每个请求 → 路由特征 → 选了哪个模型 → 成本 → 质量反馈
分析：
  - 各模型命中率 / 成本占比
  - 错误率 / 延迟分布
  - 质量回归（A/B 看路由策略变化对留存影响）
```

## 八、底层本质

模型路由的本质是**"把固定模型升级为按需匹配的资源调度"**：

- 体验：难题用强模型，简单事用便宜模型
- 成本：钱花在刀刃上
- 可用：fallback 链扛住故障
- 弹性：高峰降级保命

这是把'调一个 API'升级为'运营一个模型组合'——是 AI 产品从'能用'走向'可规模化盈利'的核心工程能力。

## 常见考点

1. **路由判错怎么办（简单题用了强模型）？**——浪费成本但不影响体验；通过日志回流优化路由策略；引入反馈机制（用户满意度回流训练分类器）。
2. **自部署 vs API 怎么选？**——自部署成本可控、数据不出门，但需要 GPU 运维、扩容慢；API 弹性大但贵且数据出域；高 QPS + 隐私敏感场景偏自部署小模型 + API 强模型混合。
3. **怎么评估路由策略好不好？**——综合指标：质量（满意度/留存）+ 成本（单会话均价）+ 可用（成功率/延迟），不能单看一个维度。


## 结构化回答

**30 秒电梯演讲：** 聊到多模型路由怎么设计（路由/fallback/成本），我的理解是——模型路由是按场景/复杂度/成本/可用性，把请求分发到最合适的模型（强模型做难题、弱模型兜底简单活），在体验、成本、稳定性间找最优解。打个比方，像医院分诊——小感冒去全科（便宜快的小模型）、疑难杂症去专家（强模型）、专家不在有备用专家（fallback）、所有人都挂专家号医院得亏死（全用强模型成本爆炸）。

**展开框架：**
1. **路由维度** — 场景/复杂度/成本预算/延迟敏感度/可用性
2. **决策方式** — 规则（快）+ 小模型分类（准）+ 强化学习（进阶）
3. **fallback 链** — 主→备→兜底小模型，按 SLA 和成本排序

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：复杂度怎么判？您更想看哪个方向？

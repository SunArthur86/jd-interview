# 【巨剧核 AI 陪伴】LLM 推理池化、缓存、异步、扩容怎么做？

> JD 依据："负责稳定性：缓存、队列；高并发高可用。"

## 一、为什么 AI 服务的并发优化特别难

```
传统服务：CPU/内存便宜，加机器即可
AI 服务：
  - LLM API 按量计费（贵）
  - 自部署 GPU 卡贵 + 周期长（采购要几周）
  - LLM 单次调用慢（秒级）
  - 流量突发（角色爆火、活动）
固定按峰值备货 → 大部分时间资源闲置 → 成本爆炸
```

必须用**池化 + 缓存 + 异步 + 扩容**把资源利用率压榨到极致。

## 二、LLM 推理池化

### 1. 连接池
```
LLM API 连接：
  HTTP 长连接 keep-alive（避免 TCP 重建）
  连接池预热（启动建好若干连接）
  按上游 API 限流配置（不超配额）
  健康检查 + 故障剔除
```

### 2. 自部署模型推理池
```
vLLM/TGI 推理服务：
  continuous batching（多请求合并批处理）
  PagedAttention（KV Cache 复用）
  大幅提升 GPU 利用率（数倍）
```

### 3. 请求合并
```
短时间内的多请求合并批推理：
  例：1000 个独立 chat 请求 → 合并成 batch → 一次推理
  适合：批处理任务、非实时场景
```

## 三、结果缓存

### 1. 精确缓存（FAQ 型）
```
key = hash(persona_id + user_id + input_normalized + params)
value = LLM 回复
TTL：30min-1h

场景：
  "你是谁" / "你多大了" / "你叫什么"
  角色介绍类、固定信息类
命中率：极高（30-50% 不夸张）
```

### 2. 语义缓存（进阶）
```
query embedding 相似度 > 0.95 → 命中
风险：要严格过滤
  - 时效敏感问题（"今天天气"）不缓存
  - 情感/付费场景不缓存
  - 用户级隔离（不跨用户共享）
```

### 3. 热点缓存
```
热门角色人设 / 当前剧情进度 / 用户基础画像 → Redis
减少 DB 和重复计算
```

## 四、异步队列削峰

```
实时对话（用户在等）：
  → 直接处理，但限流
  → 超限排队 + 进度提示

异步任务（视频/TTS/批量）：
  → 入队（Kafka/RabbitMQ）
  → Worker 按 GPU 容量消费
  → 削峰填谷

主动消息（角色定时）：
  → 错峰发送（避开高峰）
  → 批量合并
```

## 五、弹性扩容

```
触发条件：
  GPU 利用率 > 70%
  队列堆积 > 阈值
  P95 延迟 > SLA
  预测（活动前手动扩）

扩容对象：
  无状态服务：K8s HPA（按 CPU/QPS 自动扩）
  GPU 节点：Cluster Autoscaler + GPU 节点池
  数据库/Redis：分片/读写分离

缩容：
  低于阈值 + 安全期 → 缩
  要等任务完成（不抢活）
```

**滞后问题**：
- K8s 启新 Pod 要 30s-几分钟
- GPU 节点要几分钟-十几分钟
- 弥补：常驻缓冲容量 + 限流保命 + 预热

## 六、批量推理（continuous batching）

```
传统：1 req → 1 inference（GPU 利用率低）
vLLM 式：
  持续收集并发请求 → 合并成 batch
  新请求随时加入，完成的随时退出
  GPU 一直满载

效果：单卡 QPS 提升 5-10 倍
```

## 七、典型容量规划

```
假设：DAU 100w，人均 50 次对话 = 5000w 次/天
峰值 QPS（晚 8-10 点占 30%）：~ 5w QPS

资源估算：
  LLM API：按配额+成本估算
  自部署：vLLM 单卡 ~100 QPS → 500 卡
  + 缓存命中 30% → 实际 ~350 卡

成本控制：
  - 闲时缩容到 30%
  - 闲时跑批量任务（不浪费）
  - 优先级调度（VIP 优先）
```

## 八、底层本质

AI 高并发优化的本质是**"把昂贵稀缺资源的利用率压榨到极致"**：

- 池化 = 复用（不浪费空闲）
- 缓存 = 去重（不重复算）
- 异步 = 削峰（不平峰备货）
- 扩容 = 弹性（按需供给）

这是把'按峰值固定备货'重构为'按需弹性供给 + 最大化复用'——是 AI 产品从'烧钱 Demo'走向'可盈利商业'的核心工程能力。

## 常见考点

1. **缓存命中率怎么算够好？**——看场景：FAQ 类 50%+ 正常；闲聊 20-30%；情感/创作 5% 可能就有效（按成本算 ROI）。要持续优化 query 归一化。
2. **LLM 池化和 DB 连接池一样吗？**——思路类似但更复杂：LLM 调用是长时延（秒级），要支持流式、超时、并发合并、上游限流感知，比 DB 池难得多。
3. **GPU 扩容太慢怎么办？**——保留缓冲容量（不缩到 0）+ 限流保护（防雪崩）+ 降级（切便宜模型）+ 跨可用区调度 + Serverless GPU（如 Modal/Replicate 按需起）。

# 【巨剧核 AI 陪伴】消息与社区系统架构怎么设计？

> JD 依据："负责消息系统、社区、发布系统；熟悉用户/内容/消息/IM/社区/Feed。"

## 一、两大子系统

```
┌─────────────────────────────────────────┐
│ 用户                                     │
│  ↓ ↓                                     │
│ IM 系统（实时、私密、双向）                │
│   - 1v1 私聊：用户↔角色                  │
│   - 群聊：用户+多角色 / 多用户剧情         │
│   - 流式回复（LLM token 流）              │
├─────────────────────────────────────────┤
│ 社区系统（异步、公开、单向+互动）          │
│   - 发布：角色内容 / UGC / 剧情截图       │
│   - Feed：关注流 / 推荐流                 │
│   - 互动：点赞 / 评论 / 分享 / 收藏       │
├─────────────────────────────────────────┤
│ 通知中心（统一汇聚 + 多通道下发）          │
└─────────────────────────────────────────┘
```

## 二、IM 系统（详见消息系统题 004）

核心要点：
- **连接**：WebSocket 长连接 + HTTP/SSE 降级
- **可靠**：唯一 ID 去重 + seq 排序 + ack + 重传
- **离线**：消息持久化 + APNs/FCM 推送
- **流式**：LLM 输出 token 流实时推
- **群聊**：小群写扩散，大群读扩散

**AI 陪伴特色**：
- 角色是消息发送方（统一消息模型）
- 主动触发（角色定时/事件驱动发消息）
- 富媒体（语音/图片/视频，信令走 IM，文件走对象存储）

## 三、Feed 流系统

### 1. 发布
```
内容类型：角色动态 / 用户 UGC / 剧情截图 / 短视频
发布流程：
  创建内容 → 审核（AIGC + UGC 双向审）→ 入库 → 触发分发
```

### 2. 分发模型

**写扩散（推模式）**：
```
发布时：把内容推到所有粉丝的 inbox
优：读快（直接读自己 inbox）
缺：写放大（百万粉丝写百万次）
适用：粉丝 < 1000 的小角色/普通用户
```

**读扩散（拉模式）**：
```
发布时：只写到角色 outbox
粉丝刷 Feed 时：拉取关注角色 outbox 合并 + 排序
优：写少
缺：读放大（关注多要合并多源）
适用：百万粉丝大角色
```

**混合**：
```
小角色（粉丝 < 阈值）：写扩散
大角色（粉丝 > 阈值）：读扩散
粉丝拉取时：合并 inbox（写扩散来的）+ 大角色 outbox（读扩散拉）
```

### 3. 排序

```
关注流：按时间倒序（timeline）
推荐流：
  召回（粗排）→ 排序模型（精排）→ 重排（多样性/广告）
模型：协同过滤 / 双塔 / 深度模型（DIN/DeepFM）
特征：用户偏好 + 内容属性 + 上下文 + 历史互动
```

## 四、互动系统

```
点赞 / 评论 / 分享 / 收藏

存储：
  计数（Redis）：like_count, comment_count, share_count
  明细（MySQL）：user_like(cid, uid, ts)

评论树：
  parent_id（支持嵌套）
  热度排序（点赞+回复加权）
  分页（cursor 翻页）

通知：
  被点赞/评论 → 触发通知中心 → IM 红点 / 推送
```

## 五、通知中心

```
统一接入各类通知源：
  - IM 消息
  - 互动（点赞/评论/关注）
  - 任务完成（音视频生成）
  - 系统（活动/营销）

通道：
  - App 内红点 / 弹窗
  - Push（APNs/FCM/厂商）
  - 短信（重要场景）
  - 邮件（罕见）

策略：
  - 频次控制（防骚扰）
  - 优先级排序（重要先发）
  - 折叠（同类合并）
  - 用户偏好（可关）
```

## 六、AI 陪伴特色：角色互动闭环

```
用户给角色点赞
   ↓
事件 → 通知角色引擎（角色"感知"到被点赞）
   ↓
Agent 决策：是否回应？怎么回应？
   ↓
角色主动发消息："谢谢你支持我呀～"
   ↓
走 IM 系统投递
```

这是 AI 陪伴区别于普通社区的核心——**角色是活跃的参与者**，会感知、决策、主动互动。

## 七、大 V 角色的扛量设计

百万粉丝角色发动态：
```
错误做法：写扩散 → 瞬间百万次写入 → 数据库挂
正确做法：
  [1] 走读扩散（只写一份到 outbox）
  [2] 分批预热缓存（CDN）
  [3] Feed 服务缓存热门角色内容
  [4] 粉丝拉取时合并，CDN 边缘加速
  [5] 限流防雪崩（突发刷量）
```

## 八、底层本质

消息与社区系统的本质是**"用解耦的两套系统，分别承载实时私密通讯（IM）和异步公开传播（Feed）"**：

- IM = 双向、实时、可靠、低延迟
- Feed = 单向（发→收）、异步、可排序、可扩散
- 互动 = 把单向变成双向（评论/点赞建立回路）
- 通知 = 统一汇聚事件，多通道下发

AI 陪伴的特殊性在于**角色是第一公民**——既是 IM 的发送方，也是 Feed 的发布者，还能感知互动主动回应。这让社区从'人发人看'升级为'人机共演'。

## 常见考点

1. **IM 和 Feed 用同一个数据库吗？**——通常分开。IM 重实时和可靠（Redis+分库分表 MySQL），Feed 重吞吐和分析（MySQL+数仓+ES 搜索）；隔离故障域。
2. **如何防角色被恶意刷评论？**——限流（按 user/IP/接口）+ 内容审核前置 + 异常行为检测（频次/相似度）+ 风控规则。
3. **Feed 个性化推荐怎么冷启动？**——新用户用基础画像（注册信息/初始选择）+ 热门内容 + 探索-利用策略；快速积累互动后转个性化。

# 【巨剧核 AI 陪伴】记忆系统怎么设计（短期/长期/摘要/向量检索）？

> JD 依据："负责用户记忆、长期记忆；熟悉记忆摘要、向量检索；分层记忆。"

## 一、为什么需要记忆系统

```
无记忆：每次对话像初见 → 没有情感深度，用户流失
全塞 prompt：token 爆炸 + 注意力稀释（见会话管理题）
正确做法：分层存储 + 按需召回
```

陪伴产品的**核心情感价值 = 被记住**。记忆系统就是把这种能力工程化。

## 二、分层记忆架构

```
┌──────────────────────────────────────────────────┐
│ 短期记忆（工作记忆）                                │
│   存：最近 N 轮原文（Redis / 内存）                 │
│   特点：高频、低容量、原文保真                       │
├──────────────────────────────────────────────────┤
│ 长期记忆 - 结构化（事实表）                          │
│   存：用户画像/偏好/承诺/关系节点（MySQL/图库）      │
│   特点：精确查询、可更新、带时间戳和置信度            │
├──────────────────────────────────────────────────┤
│ 长期记忆 - 向量片段（情景记忆）                      │
│   存：压缩后的对话片段 embedding（向量库）           │
│   特点：语义召回、模糊匹配、容量大                   │
└──────────────────────────────────────────────────┘
```

## 三、写入流程（记什么、怎么记）

```
对话轮次
   ↓
[1] 关键信息抽取（LLM）：人名？偏好？承诺？情绪事件？
   ↓
[2] 过滤：去寒暄/重复/无信息量
   ↓
[3] 去重合并：与已有事实比对，矛盾则更新（新覆盖旧 + 记录版本）
   ↓
[4] 打标：时间戳、重要性分数、来源轮次、(user_id, role_id)
   ↓
[5] 分发写入：结构化进事实表，片段进向量库
```

**抽取示例**：
```
用户："我最近在学吉他，不过还是弹不好"
抽取 → {skill: 吉他, level: 初学, sentiment: 挫败, ts: ...}
```

## 四、召回流程（怎么想起来）

用户输入 → 同时走多路召回 → 融合排序 → 注入上下文：
```
                    ┌─→ 向量召回（语义相关 top-K 片段）
user_input embedding┤
                    ├─→ 结构化查询（用户画像/最近承诺）
                    └─→ 时效召回（最近 24h 关键事件）

融合排序分数 = α·相关性 + β·重要性 + γ·时效性 + δ·情感权重
取 top-N 注入 prompt
```

**不是每次全搜**：向量库用 ANN（HNSW/IVF）亚秒级；加缓存（热门 query 命中）；低优先级场景异步预召回。

## 五、更新与矛盾处理

```
旧：{怕黑: true, ts: 2024-01}
新：{怕黑: false, ts: 2026-07}
→ 软更新：新记录覆盖，旧记录保留为历史版本（带 expired 标记）
→ 角色下次说"你现在不怕黑了呀"，体现记忆演进
```

**衰减机制**：久未访问的记忆降权（类似 LRU + 时间衰减），但高重要性（承诺/关系节点）不衰减。

## 六、隔离与安全

- 记忆严格带 `(user_id, role_id)` 维度，召回必带过滤条件
- 多角色之间记忆隔离（A 角色不能泄露 B 角色聊过的隐私）
- 敏感信息（密码/身份证）不入记忆，写入前脱敏

## 七、底层本质

记忆系统的本质是**"把'记忆'从模型内部能力外化为可管理的工程系统"**：

- **存什么** = 信息抽取 + 取舍
- **怎么找** = 向量 + 结构化 + 时效的多路召回
- **怎么不乱** = 矛盾处理 + 衰减 + 隔离

这是把'人脑的回忆机制'用数据库 + 检索重建出来，让 LLM 拥有**可演进、可信赖、可规模化的长期关系能力**。

## 常见考点

1. **记忆和 RAG 有什么区别？**——RAG 是'查外部知识'（一次性问答），记忆是'查关于这个用户的历史'（持续演进、需更新去重、强隔离）。记忆是 RAG 的'个性化、可变、带身份'特化。
2. **向量召回不准怎么办？**——加结构化过滤先粗筛、混合检索（向量+关键词）、重排序模型（cross-encoder rerank）、用户反馈回路纠偏。
3. **记忆会让角色越来越慢吗？**——会。控制：召回 top-K 有上限、记忆定期归档冷存储、热门记忆缓存、低优先级异步召回。


## 结构化回答

**30 秒电梯演讲：** 聊到记忆系统怎么设计（短期/长期/摘要/向量检索），我的理解是——记忆系统是给无状态的 LLM 装一个'大脑海马体'——短期记最近发生的事，长期记关键事实，并能在需要时回忆起来，让角色拥有'记得你'的能力。打个比方，像人脑——工作记忆（最近几秒对话）放前台，海马体（今天的关键事）做暂存，大脑皮层（几年前的关键记忆）做长期存储，靠'联想'（向量检索）把旧事想起来。

**展开框架：**
1. **三层** — 短期（会话原文）/ 长期（结构化事实 + 向量片段）/ 元记忆（用户画像）
2. **写入** — 对话 → 抽取关键信息 → 去重/合并/打时间戳 → 入库
3. **召回** — 相关性（向量）+ 时效性（近优先）+ 重要性加权排序

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：记忆存什么不存什么？您更想看哪个方向？

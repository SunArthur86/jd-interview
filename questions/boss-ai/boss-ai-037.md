# 【巨剧核 AI 陪伴】AI Infra 基础设施怎么建？

> JD 依据："熟悉 AI Infra 基础设施：GPU 调度、模型仓库、特征、监控漂移。"

## 一、AI Infra 的角色

```
上层：AI 产品（陪伴/对话/生成）
中层：AI 应用工程（编排/记忆/工具/审核）
基层：AI Infra（算力/模型/数据/监控）  ← 本题

价值：
  - 不直接生产产品功能
  - 决定产品的规模化能力、效率、稳定性
  - 是大厂 AI 的核心竞争力
```

## 二、GPU 调度（算力层）

### 1. 池化
```
共享池：
  所有 GPU 集中池化
  按任务需求分配
  优：利用率高
  缺：隔离弱

分池：
  按业务/任务类型分池
  LLM 池 / 渲染池 / 训练池
  优：隔离强
  缺：利用率不均

混合：
  核心业务独占池
  通用任务共享池
  闲时共享
```

### 2. 调度策略
```
优先级：
  在线推理（高，实时）
  离线训练（中）
  批处理（低，可等）

公平调度：
  按团队/用户配额
  防止单方垄断

抢占：
  高优先级可抢低优先级 GPU
  抢占时优雅停止（保存 checkpoint）

bin-packing：
  多个小任务合并到一张卡
  提升利用率

队列：
  不同任务类型独立队列
  按队列调度
```

### 3. 弹性扩缩
```
触发：
  GPU 利用率 > 70%
  队列堆积
  预测（活动前手动扩）

扩：
  Cluster Autoscaler 加 GPU 节点
  几分钟-十几分钟

缩：
  安全期 + 任务完成
  不抢活

成本：
  闲时降配（量化）
  Spot 实例（容错任务）
  跨可用区
```

### 4. 多租户
```
按租户配额：
  CPU/GPU/存储/调用次数
  防互相干扰

隔离：
  资源（分池/虚拟化）
  数据（schema/库）
  故障（一挂不全挂）
```

## 三、模型仓库（模型资产层）

### 1. 版本化
```
模型 = 一等公民，要版本管理：
  model_id, version, training_data, hyper_params,
  metrics, created_at, status
  
每次训练记录：
  数据集版本
  代码版本（git commit）
  超参
  评测指标
  工件（weights）
```

### 2. 实验追踪
```
工具：MLflow / WandB / 自研

记录每次实验：
  参数（model/data/hyper）
  指标（accuracy/loss/业务）
  工件
  日志

价值：
  对比实验（这次 vs 上次）
  复现（拿到 commit + 数据版本）
  选最优
```

### 3. 部署管理
```
模型生命周期：
  训练 → 评测 → 灰度部署 → 全量 → 监控 → 迭代/回滚

部署模式：
  蓝绿（blue-green）
  金丝雀（canary）
  影子（shadow，旁路验证）

服务化：
  vLLM/TGI 服务化
  版本路由（A/B 不同模型）
  热切换（不停机）
```

### 4. 回滚
```
模型质量劣化 / 故障 → 回到上版本
- 版本保留（不删 history）
- 一键回滚（路由切回）
- 数据/特征也要兼容版本
```

## 四、特征平台（数据资产层）

### 1. 解决的痛点
```
训练-服务 skew：
  训练用 Spark/Python 算特征
  上线用 Java/Go 重写
  两套实现，容易不一致 → 线上效果差

重复造轮子：
  每个团队各做特征工程
  重复劳动 + 不一致
```

### 2. 特征平台架构
```
离线层（训练）：
  Spark/Flink 算特征 → 离线存储
在线层（推理）：
  特征服务 API（低延迟）
  Redis/特征库

一致性保证：
  统一特征定义（一处定义）
  离线/在线同一逻辑（同代码/同变换）
  特征注册中心

复用：
  特征市场
  团队共享
  按需查询
```

### 3. AI 陪伴的特征
```
用户特征：
  年龄/性别/活跃度/tier
  偏好（话题/角色/内容类型）
  行为（会话频次/时长/付费）

角色特征：
  人设标签
  受欢迎度
  关系网络特征

上下文特征：
  当前剧情/场景
  时间/设备/地域

实时特征：
  最近会话情绪
  当前 GPU 负载（路由用）
```

## 五、监控漂移（质量保障层）

### 1. 漂移类型
```
数据漂移（Data Drift）：
  输入分布变了
  例：用户从问"角色设定"变成问"时事新闻"（模型不擅长）
  
概念漂移（Concept Drift）：
  同输入但期望输出变了
  例：用户对"礼貌"的标准变了（以前 OK 现在嫌冷漠）
  
模型漂移（Model Drift）：
  底层模型更新导致输出风格变
  例：OpenAI 升级 GPT-4 → 你的 prompt 效果变
```

### 2. 检测方法
```
输入分布监控：
  embedding 距离（用户 query vs 历史）
  关键词分布
  query 长度/类型分布
  统计距离（KS / PSI / JS divergence）

输出分布监控：
  长度/风格分布
  审核拦截率
  投诉类型
  情绪标签分布

业务指标监控：
  准确率（如有标注）
  满意度（点赞踩）
  留存/时长
  转化

显著偏移 → 告警 → 评估 → 调整
```

### 3. 应对策略
```
数据漂移：
  - 扩展训练数据（覆盖新分布）
  - 补充 RAG 知识库
  - fine-tune

概念漂移：
  - 用户反馈回流
  - 调整评测集
  - 重训/fine-tune

模型漂移：
  - 模型版本锁定
  - 升级前 A/B
  - prompt 适配
```

## 六、统一编排（MLOps 闭环）

```
数据 → 特征平台 → 训练 → 模型仓库 → 部署 → 服务 → 监控
                       ↑                              ↓
                       └────── 漂移检测/反馈 ←──────┘

工具链：
  调度：Airflow / Argo
  实验：MLflow / WandB
  特征：Feast / 自研
  服务：vLLM / Seldon / KServe
  监控：Evidently / WhyLabs / 自研
```

## 七、底层本质

AI Infra 的本质是**"把 AI 从'实验室实验'升级为'工业级可靠运营'的基础设施"**：

- GPU 调度 = 算力可运营
- 模型仓库 = 模型可管理
- 特征平台 = 数据可复用
- 漂移监控 = 质量可保障

这是大厂 AI 的核心竞争力——同样的模型，谁的基础设施好，谁就能更便宜、更稳定、更快迭代地运营 AI 产品。OpenAI/字节/阿里都在卷这套基础设施。

## 常见考点

1. **小公司需要全套 AI Infra 吗？**——不必。早期用云厂商全托管（OpenAI/MonoCloud）；规模化后才自建；按瓶颈投入（先 GPU 调度再特征再漂移）。
2. **GPU 利用率多少算高？**——vLLM 池化下 70-80% 算优秀；朴素推理 < 20% 是浪费；要持续优化但留余量（应对突发）。
3. **漂移检测误报多怎么办？**——调阈值（业务允许的偏移范围）+ 分级告警（关键指标严，辅助松）+ 人工 review 后再行动；建立基线（什么是正常波动）。

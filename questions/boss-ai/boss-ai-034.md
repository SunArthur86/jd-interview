# 【巨剧核 AI 陪伴】向量检索与召回怎么做（选型/ANN/重排）？

> JD 依据："熟悉向量检索与召回；向量库选型、ANN 算法、重排序。"

## 一、向量检索在 AI 陪伴的应用

```
记忆召回：
  用户："上次说的工作的事"
  检索：用户记忆向量库 → 找到相关工作事件片段

RAG：
  用户："小雨你最喜欢的猫是哪只"
  检索：角色知识库 → 找到角色设定相关片段

推荐：
  用户偏好 → 召回相似角色/剧情/内容

去重：
  新内容 → 检查相似度 → 防重复
```

## 二、嵌入（Embedding）

```
文本 → 嵌入模型 → 向量（如 768 维 / 1536 维）

模型选择：
  通用：bge-large-zh / OpenAI text-embedding-3 / Cohere
  多语种：bge-m3 / multilingual-E5
  自训练：领域适配（如陪伴对话数据微调）

要点：
  - 中英文场景 bge 系列效果好且开源
  - 维度越高表达越强但存储/计算越贵
  - 嵌入质量决定召回上限
```

## 三、精确检索为什么不行

```
线性扫：
  query 向量 vs 库中所有 N 个向量算距离
  返回 top-K

复杂度：O(N × D)
  N = 10 亿，D = 768
  一次查询要算 7.68×10^11 次乘加
  不可行（秒级-分钟级）
```

## 四、ANN 算法（近似最近邻）

```
思想：牺牲少量精度（召回率 95%+）换巨大速度（毫秒级）

主流算法：

[1] HNSW（Hierarchical Navigable Small World）
    分层图结构，从顶层稀疏入口逐层细化
    查询：O(log N)
    优：查询极快，召回率高（主流首选）
    缺：建图慢，内存占用大

[2] IVF（Inverted File）
    聚类分桶（k-means），查桶内
    查询：O(N/k)
    优：内存少，可调精度-速度
    缺：召回率略低，需调 nprobe

[3] ScaNN（Google）
    各向异性量化 + 多阶段
    优：质量速度平衡好
    缺：实现复杂

[4] PQ（Product Quantization）
    向量压缩存储
    配合 IVF 用（IVF-PQ）

实战：
  通用首选 HNSW（Milvus/Qdrant 默认）
  超大规模 + 内存敏感用 IVF-PQ
```

## 五、向量库选型

| 库 | 特点 | 适用 |
|---|---|---|
| Milvus | 分布式、大规模、功能全 | 亿级生产环境 |
| Qdrant | Rust 高性能、易用 | 中大规模 |
| pgvector | PostgreSQL 扩展 | 中小规模起步 |
| Weaviate | 内置嵌入/多模态 | 全栈方案 |
| Pinecone | SaaS 托管 | 不想运维 |
| Faiss | 库（非服务） | 嵌入应用内 |

```
按规模选：
  < 100 万：pgvector（简单，复用 PG）
  100 万-1 亿：Qdrant/Milvus 单机
  > 1 亿：Milvus 分布式 / Qdrant 集群

AI 陪伴推荐：
  起步 pgvector（小规模验证）
  规模化 Milvus/Qdrant（亿级用户记忆）
```

## 六、混合检索

```
单路向量检索局限：
  - 形似神不似（"今天天气" vs "今日气候"）
  - 专有名词不准（"OpenAI" 召回不到精确文档）
  - 缺时效/结构化过滤

混合：
  [向量召回] query embedding → ANN → top-N
  [关键词召回] query 分词 → BM25 倒排 → top-N
  [结构化查询] 过滤条件（user_id, ts, ...）
  
融合：
  RRF（Reciprocal Rank Fusion）：按排名融合
  或加权分数融合

效果：召回率提升 10-30%
```

## 七、重排序（Rerank）

```
问题：
  召回（bi-encoder）粗排 top-50
  top-1 不一定真比 top-5 相关（嵌入不够精细）

重排（cross-encoder）：
  把 query+doc 联合编码 → 精确打分
  比 bi-encoder 准但慢
  对 top-50 重排，取 top-K

模型：
  bge-reranker / Cohere Rerank / 自训练

效果：
  准确率显著提升
  仅对小集合（top-50）跑，延迟可控（几十 ms）
```

## 八、AI 陪伴的检索链路

```
用户 query
   ↓
[query 改写]（可选）LLM 生成变体扩展
   ↓
并行多路召回：
  [向量召回] query emb + filter(user, role) → ANN top-50
  [关键词召回] BM25 top-50
  [结构化] 用户画像/最近事件
   ↓
[融合排序] RRF
   ↓
[重排] cross-encoder 精排 top-50
   ↓
取 top-K（如 5-10）
   ↓
注入 prompt
```

## 九、性能与扩展

```
单机性能：
  HNSW：1 千万向量，查询 < 10ms
  QPS：单机 1k-1w（看硬件）

大规模扩展：
  分片（按 user_id / hash）
  副本（高可用 + 读扩）
  分区（按类别/时间）
  缓存（热门 query）

监控：
  召回率（与精确对比）
  查询延迟 P95/P99
  索引大小/内存
  QPS
```

## 十、底层本质

向量检索的本质是**"把语义相似性问题转化为几何近邻问题，再用 ANN 算法工程化加速"**：

- 嵌入 = 语义→几何
- ANN = 近似快（牺牲精度换速度）
- 混合检索 = 互补提全
- 重排 = 精细提准

这是把'关键词检索'升级为'语义检索'——是记忆系统、RAG、推荐、去重的共同基础设施，是 AI 产品的核心数据库能力。

## 常见考点

1. **HNSW 比线性扫快多少？**——O(N) → O(log N)，10 亿数据从秒级降到毫秒级；代价是建图慢、内存大、不精确（95%+ 召回）。
2. **嵌入模型选错怎么办？**——召回不准 → 换模型 → 要全量重嵌入重索引（贵）；选模型时要先在小集合评测，避免后期返工。
3. **向量库和 ES 怎么协同？**——ES 做关键词/结构化检索 + 向量字段（dense_vector）；或 ES + Milvus 双写，业务做混合检索；Milvus 也支持标量过滤（弱化 ES 需求）。

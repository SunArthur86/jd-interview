# 【巨剧核 AI 陪伴】Tool Calling 工程实践怎么做？

> JD 依据："熟悉 Tool Calling、工具调用；工具注册、发现、安全、编排。"

## 一、为什么需要 Tool Calling

```
纯 LLM 的局限：
  - 不知道实时信息（天气/新闻/库存）
  - 不能执行动作（订票/发消息/扣费）
  - 知识有截止日期
  - 不能精确计算（如复杂运算）

Tool Calling：
  LLM 决定调哪个工具 → 工具执行 → 结果回填 → LLM 继续生成
  LLM 从"嘴"升级为"手+嘴"
```

## 二、工具注册（标准化）

```yaml
# 工具 schema（OpenAI function calling 风格）
- name: get_weather
  description: 查询指定城市的当前天气
  parameters:
    type: object
    properties:
      city:
        type: string
        description: 城市名（中文/英文）
      unit:
        type: string
        enum: [C, F]
        default: C
    required: [city]

# AI 陪伴常见工具
- search_role_lore: 搜索角色设定（剧情/背景）
- query_user_memory: 查询用户历史记忆
- generate_image: 生成角色相关图片
- trigger_video_task: 触发音视频生成任务
- send_gift: 给用户送虚拟礼物
- advance_story: 推进剧情进度
- query_relation: 查询用户↔角色关系
```

注册中心管理所有工具 schema，业务可动态注册/下线。

## 三、工具发现与选择

```
传给 LLM 的工具列表：
  完整描述（name + description + parameters）

LLM 根据用户输入自主选择：
  用户："今天上海天气怎么样"
  LLM 输出：tool_call(get_weather, {city: "上海"})

工具太多问题：
  - 描述占 token
  - LLM 选择困难/误选
  解决：
  - 按场景过滤（聊天场景只暴露聊天工具）
  - 两阶段：先选大类，再选具体工具
  - 描述精炼 + 示例
```

## 四、执行流程

```
[1] LLM 决定调工具
    输出 tool_call(name, args)

[2] 参数校验
    - 类型/必填/枚举值校验
    - 业务规则校验（city 必须是合法城市）
    - 不通过 → 让 LLM 重新生成参数

[3] 权限校验
    - 用户 tier（免费用户不能调用 VIP 工具）
    - 角色/数据权限
    - 高危操作（支付/删除）二次确认

[4] 执行
    - 调用真实工具 API
    - 超时/重试/熔断
    - 失败处理

[5] 结果回填
    - tool 角色消息加结果
    - LLM 基于结果生成最终回复

[6] 审计
    - 记录每次工具调用（who/what/when/result）
```

## 五、安全护栏

```
白名单：
  只暴露预批准的工具
  危险操作不暴露给 LLM

参数校验：
  强类型 + 范围 + 业务规则
  防 prompt injection 注入恶意参数

权限：
  用户/角色/数据三层
  高危操作二次确认或人工审核

限频：
  防止 LLM 循环调用同一工具（如疯狂发消息）
  按 user/tool 维度限频

沙箱：
  代码执行类工具（如有）跑沙箱
  网络隔离 + 资源限制

审计：
  全部工具调用记录
  可追溯（合规）
  异常检测（行为偏离）
```

## 六、工具编排（从单工具到工具链）

```
单工具：一次调一个
  用户："天气？" → get_weather

多工具并行：
  用户："北京和上海天气"
  → 并行 get_weather(北京) + get_weather(上海)
  → 合并结果

工具链（多步推理）：
  用户："帮我订明天北京到上海的机票"
  → search_flights（搜航班）
  → check_user_payment（验证支付能力）
  → book_ticket（订票）
  → send_confirmation（发确认）
  每步基于上一步结果决策（Agent loop）

编排：和 Agent 引擎结合（见 Agent 引擎架构题）
```

## 七、错误处理

```
工具失败：
  - 参数错 → LLM 重新生成参数（带错误反馈）
  - 工具挂 → fallback 到同类工具 / 告知用户稍后
  - 超时 → 重试 / 降级
  - 权限不足 → 引导用户升级 / 改用其他方式

防死循环：
  工具调用次数上限
  连续失败 N 次放弃
  兜底模板回复
```

## 九、底层本质

Tool Calling 的本质是**"把 LLM 从'文本生成器'升级为'能调外部能力的行动者'"**：

- 注册 = 能力可声明
- 选择 = LLM 自主决策
- 执行 = 真实副作用
- 安全 = 不失控
- 编排 = 完成复杂任务

这是 LLM 从'问答'走向'Agent'的关键能力——是 ChatGPT 插件、Claude tools、Manus/Devin 等 AI Agent 产品的共同底层。AI 陪伴里，Tool Calling 让角色能查、能调、能生成、能互动，从'只会说'升级为'真能办事'。

## 常见考点

1. **工具描述对选择有多大影响？**——巨大。模糊描述 → LLM 误选/不选；清晰描述（功能+参数+示例）→ 准确选择。要把工具描述当 API 文档写。
2. **怎么防 prompt injection 攻击工具？**——参数白名单（不允任意值）+ 上下文隔离（用户输入和系统指令分轨）+ 危险操作二次确认 + 输入净化。
3. **工具调用太慢影响体验怎么办？**——并行调用（独立工具同时调）+ 流式返回（先返回文字部分，工具结果后续补）+ 缓存（重复 query 复用结果）+ 异步（长任务后台跑，先告知用户）。

# 【巨剧核 AI 陪伴】负载均衡、隔离、压测怎么做？

> JD 依据："负责稳定性：负载均衡、隔离、压测。熟悉 GPU 隔离、多租户、全链路压测。"

## 一、负载均衡（均匀分流）

### 1. 四层 vs 七层
```
四层（LVS）：基于 IP/端口，转发快，不感知协议
七层（Nginx/Envoy）：基于 HTTP/URL/Header，灵活路由
AI 场景：
  网关用七层（按 URL/tier 路由）
  内部 RPC 用四层或服务网格（Sidecar）
```

### 2. 算法
```
轮询：简单均匀
最少连接：按负载分（适合长连接/LLM 慢调用）
一致性哈希：会话保持（同 user → 同 worker，命中本地缓存）
加权：按节点容量（GPU 数/性能）
```

### 3. 健康检查
```
主动：定期 ping/health 接口
被动：调用失败累计 → 剔除节点
恢复：成功后再加回
```

## 二、隔离（故障不传染）

### 1. 资源隔离
```
连接池：LLM/DB/Redis 各自独立池（一处堵塞不传染）
线程池：核心/非核心接口分池
GPU 池：按算子类型/优先级分池
  实时对话池（VIP/低延迟）
  批量任务池（高吞吐）
  视频生成池（长任务）
```

### 2. 租户隔离
```
按用户 tier：
  VIP 独享部分资源（不与免费抢）
  免费用户限流更严

按租户（B 端）：
  资源配额（CPU/存储/调用量）
  数据隔离（schema/库级别）
```

### 3. 业务隔离
```
核心链路（实时对话）vs 非核心（视频生成/推荐）→ 独立服务/资源
写 vs 读 → 主从分离
热点 vs 冷数据 → 分库分表
故障域：按机房/可用区分组，跨可用区部署
```

## 三、GPU 隔离（AI 特色）

```
方案一：物理分池（最硬隔离）
  实时池：4 张 A100 专做对话
  批量池：8 张 A100 做异步任务
  优：互不干扰；缺：利用率可能不均

方案二：MPS（Multi-Process Service）
  多进程共享 GPU，内核级隔离
  提升利用率，但故障可能影响

方案三：时间片调度
  任务排队，按时间片轮转
  适合长任务

方案四：vGPU 虚拟化
  硬件级虚拟化（NVIDIA vGPU）
  切片给多租户

方案五：软件 quota
  按 user/tier 配额（最弱隔离）
```

实战：物理分池（核心隔离）+ 优先级调度（弹性利用）组合。

## 四、压测（容量可知）

### 1. 压测层次
```
单机压测：找单节点瓶颈（CPU/内存/GPU）
集群压测：验证扩容效果
全链路压测：模拟真实用户行为，端到端验证
```

### 2. 全链路压测
```
目标：
  - 找系统瓶颈（最先扛不住的环节）
  - 定容量（多少 QPS 安全/极限）
  - 验证降级（高压下保核心）
  - 暴露隐患（连接池不够/慢查询）

方法：
  流量录制：录真实请求 → 回放放大
  影子库：压测数据写到影子库（不污染线上）
  Mock 外部：不真打 OpenAI（太贵）
  标识透传：压测流量带标 → 各服务识别 → 隔离处理

工具：
  JMeter / Gatling / 自研（适合复杂场景）
  TCPCopy / 影子流量
```

### 3. 容量规划
```
压测得到：
  单机 QPS / 集群 QPS
  P99 延迟曲线
  资源利用率曲线
  瓶颈点（CPU/GPU/DB/网络）

推算：
  目标 QPS → 所需节点数（+冗余）
  高峰/低谷比例 → 弹性扩缩范围
  成本预算
```

## 五、典型压测场景

```
场景 1：日常晚高峰
  模拟 1w QPS 对话 → 验证 P95 < 3s → 容量足够

场景 2：爆款角色
  模拟某角色 QPS ×10 → 验证限流 + 降级生效

场景 3：厂商故障
  模拟 OpenAI 挂 → 验证 fallback 链路 → 切到备用厂商

场景 4：大促活动
  模拟 5w QPS → 验证全链路容量 + 降级预案
```

## 六、底层本质

负载均衡、隔离、压测的本质是**"让系统均匀承压、故障可控、容量可知"**：

- 均衡 = 防单点过载
- 隔离 = 防故障传染
- 压测 = 把盲发变已知

这三件套是工业级高可用系统的地基——AI 服务因为资源昂贵稀缺，这三件套比传统服务更关键（GPU 不能随便加，配额不能随便超）。

## 常见考点

1. **会话亲和（sticky session）需要吗？**——AI 场景常用：同 user 路由同 worker 命中本地缓存（人设/短期记忆）；用一致性哈希实现；故障时切换到任意 worker（无状态设计兜底）。
2. **多租户隔离怎么做成本最低？**——应用层 quota（按 user 计数限流）+ 共享资源池（不分物理）；规模大了再升级到独立资源组/vGPU。
3. **压测的影子流量怎么不污染监控？**——压测流量带特殊 Header → 各服务和监控 SDK 识别 → 单独统计不上报真实大盘。


## 结构化回答

**30 秒电梯演讲：** 聊到负载均衡、隔离、压测怎么做，我的理解是——负载均衡、隔离、压测是把流量合理分到资源、把故障隔在范围内、把容量测在上线前的三件套——让系统均匀承压、单点故障不传染、上线前知道能扛多少。打个比方，像城市规划——道路分流不堵车（负载均衡）、住宅商业工业分区互不打扰（隔离）、新桥通车前先过载重测试看能扛多少吨（压测）。

**展开框架：**
1. **负载均衡** — 四层/七层、轮询/最少连接/一致性哈希、健康检查
2. **隔离** — 资源（GPU/连接池）+ 租户（用户/tier）+ 业务（核心/非核心）多维度
3. **压测** — 单机→集群→全链路，找瓶颈、定容量、验证降级

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：一致性哈希解决什么？您更想看哪个方向？

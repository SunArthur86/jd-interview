# 【Java 后端架构师】线上事故复盘与根因分析方法

> 适用场景：JD 核心技术。一次大促期间订单系统全线超时 30 分钟，影响订单 X 万单。事故恢复了，但如果没有系统复盘，3 个月后同样的事故还会发生。架构师要做的是建立 blameless 复盘文化，用 5-Whys 逼近系统性根因，把一次性教训变成机制（监控/规范/演练），让团队从每次故障中变强而非重复踩坑。

## 一、概念层：blameless 文化与复盘框架

### blameless 的核心原则

| 传统追责式 | blameless 学习式 | 区别 |
|-----------|------------------|------|
| "谁改错了？" | "系统为什么允许这个错误进入生产？" | 追系统不追个人 |
| "为什么没测出来？" | "我们的防护网为什么没拦住？" | 追防护不追测试者 |
| 处罚当事人 | 改进系统/流程/工具 | 改进而非惩罚 |
| 当事人不敢说真话 | 当事人分享决策时上下文 | 真相优先 |

**为什么 blameless 不是"不追责"**：blameless 不是放过疏忽，而是把注意力放在"系统改进"上。如果真是能力问题（培训不足）或态度问题（违反规范），通过管理手段处理，不在复盘会公开追责——复盘会的唯一目标是学习。

### 复盘全流程

```
故障发生 → 应急响应（止损第一）→ 故障恢复
                                      │
                          ┌───────────┴───────────┐
                          ▼                       ▼
                    48h 内开复盘会          采集证据（日志/监控/变更记录）
                          │
                          ▼
              1. 还原时间线（分钟级）
              2. 5-Whys 根因分析
              3. 输出行动项（owner + deadline）
              4. 归档复盘文档
                          │
                          ▼
                    跟踪行动项闭环率
                    （周会 review 直到完成）
```

## 二、机制层：故障时间线与 5-Whys

### 2.1 故障时间线模板（复盘的基础）

```markdown
# 故障时间线：订单系统超时事故 2026-07-13

## 故障概述
- 级别：P0（核心功能不可用）
- 影响：订单创建全线超时，持续 32 分钟，影响订单约 1.2 万单
- 业务损失：约 80 万元 GMV 流失

## 时间线
| 时间 | 事件 | 来源 |
|------|------|------|
| 10:00 | 发版：库存服务 v2.3（新增批量查询接口） | 发布平台 |
| 10:05 | 库存服务 P99 从 50ms 升到 500ms | 监控 |
| 10:08 | 订单服务调用库存超时率从 0.1% 升到 30% | 监控 |
| 10:10 | 订单服务线程池打满（200/200） | 监控 |
| 10:12 | 订单服务全线超时，用户下单失败 | 告警+客诉 |
| 10:13 | oncall 收到 P0 告警 | 告警 |
| 10:15 | oncall 判断库存服务发版引起，决定回滚 | 决策 |
| 10:20 | 库存服务回滚 v2.2（发版平台耗时 5 分钟） | 发布平台 |
| 10:22 | 库存 P99 恢复 50ms | 监控 |
| 10:25 | 订单线程池释放，超时率下降 | 监控 |
| 10:32 | 全线恢复，故障结束 | 监控 |
| 10:35 | 确认恢复，关闭故障工单 | oncall |

## 时间线分析要点
- 检测延迟（MTTD）：10:08 首个异常 → 10:12 用户感知 = 4 分钟（合理）
- 定位延迟（MTTI）：10:13 收到告警 → 10:15 判断根因 = 2 分钟（快）
- 修复延迟（MTTR）：10:15 决定回滚 → 10:32 全线恢复 = 17 分钟（回滚慢）
```

### 2.2 5-Whys 根因分析（示例）

```markdown
# 5-Whys：为什么订单系统全线超时？

Why 1: 为什么订单系统超时？
→ 订单调用库存服务的线程池被打满（200/200），新请求排队超时。

Why 2: 为什么线程池被打满？
→ 库存服务 v2.3 的批量查询接口 P99 从 50ms 升到 500ms，
   每个订单请求占用线程时间从 50ms 变 500ms，并发 200 个就把池子占满。

Why 3: 为什么批量查询接口变慢？
→ 新接口用 "SELECT * FROM inventory WHERE sku_id IN (...)" 一次查 500 个 SKU，
   没走索引（sku_id 字段没建索引），全表扫描。

Why 4: 为什么没建索引的 SQL 能进入生产？
→ Code Review 没有检查 SQL 索引使用，单测用小数据量（100 行）测不出慢查询。

Why 5: 为什么 Review 和测试都没拦住？
→ 【系统性根因】缺乏 SQL 性能门禁——
   (a) Review checklist 没有"新 SQL 必须验证索引"项；
   (b) CI 没有 SQL 性能扫描（如 pt-query-digest 在 10 万行数据上跑）；
   (c) 预发环境数据量小（1000 行），慢 SQL 在预发测不出来。

→ 进一步：为什么回滚要 17 分钟？
   Why: 回滚走发布平台全量发布流程（拉镜像→滚动更新），没有"一键回滚"开关。

系统性根因总结：
1. 缺乏 SQL 性能门禁（Review + CI + 预发数据量）
2. 缺乏快速回滚机制（回滚依赖发布平台全量流程）
3. 库存服务慢调用没有熔断保护（拖垮了订单服务）
```

**5-Whys 的停止标志**：当根因是"系统性/机制性"缺陷（流程缺失、工具缺失、架构缺陷），且解决方案是"改系统"而非"下次注意"时，停止追问。不要追到"为什么招了这个人"——那是甩锅不是根因。

### 2.3 行动项输出（闭环跟踪）

```markdown
# 行动项（每项必须有 owner + deadline + 验证指标）

| # | 行动项 | 类型 | Owner | Deadline | 验证指标 |
|---|--------|------|-------|----------|---------|
| 1 | Review checklist 增加"新 SQL 验证索引"必查项 | 规范 | @张三 | 07-20 | checklist 更新，review 执行率 100% |
| 2 | CI 集成 SQL 性能扫描（SpotBugs + 自定义规则） | 工具 | @李四 | 07-30 | CI 扫描覆盖率 100%，慢 SQL 阻断合并 |
| 3 | 预发环境数据量提升到 100 万行（脱敏真实数据） | 环境 | @运维 | 08-15 | 预发慢 SQL 可检出 |
| 4 | 库存服务加熔断（错误率 30% 触发，降级返回缓存） | 架构 | @王五 | 07-25 | 熔断配置生效，故障演练验证 |
| 5 | 建立"一键回滚"开关（配置中心切换上一版本） | 工具 | @赵六 | 08-01 | 回滚时间 < 2 分钟 |
| 6 | 订单服务线程池与库存调用隔离（独立线程池） | 架构 | @王五 | 08-10 | 库存慢调用不影响订单主链路 |

闭环跟踪：每周 review 行动项进度，超期升级 owner 主管
完成率目标：> 80%（剩余 20% 可标记"已知风险接受"但要有明确理由）
```

## 三、实战层：复盘会怎么开

### 3.1 复盘会议程（60-90 分钟）

```
1. 重申 blameless 原则（2 分钟）
   - 主持人开场："这个会的目标是学习，不是追责。
     我们相信每个人在当时信息下做了最优决策。"

2. 还原时间线（15 分钟）
   - 当事人讲"发生了什么、我当时的决策依据"
   - 只陈述事实，不评判对错
   - 补充：为什么做了这个决策（信息/工具/时间约束）

3. 5-Whys 根因分析（30 分钟）
   - 集体追问"为什么"
   - 追到系统性根因为止
   - 用白板/在线文档实时记录

4. 行动项讨论（20 分钟）
   - 每个根因对应至少一个行动项
   - 行动项必须"机制化"（监控/规范/工具），不是"下次注意"
   - 现场认领 owner 和 deadline

5. 总结（5 分钟）
   - 一句话根因
   - 行动项总览
   - 感谢当事人的坦诚
```

### 3.2 高质量复盘 vs 低质量复盘

| 维度 | 低质量 | 高质量 |
|------|--------|--------|
| 根因 | "开发疏忽" | "缺 SQL 性能门禁，系统允许慢 SQL 进入生产" |
| 行动项 | "下次注意" "加强培训" | "CI 集成 SQL 扫描，7-30 上线，阻断合并" |
| 跟踪 | 写完报告就结束 | 周会跟踪，完成率 80%+ |
| 文化 | 甩锅、当事人不敢说话 | blameless、当事人分享决策上下文 |
| 沉淀 | 报告存档没人看 | 入知识库，新系统设计参考 |

## 四、底层本质：复杂系统的事故模型

软件事故的底层模型是**瑞士奶酪模型（Swiss Cheese Model）**：系统有多层防护（设计/编码/测试/Review/监控/告警），每层都有漏洞（如弱光照射）。事故发生是多个防护层的漏洞恰好连成一线——不是某一层完全失效，而是每层都漏了一点。

**为什么追责个人是错的**：当事人犯错（如写慢 SQL）只是最后一层奶酪的漏洞。前面还有 Review（为什么没查出）、CI（为什么没扫描）、预发（为什么没测出）、监控（为什么没预警）。只补最后一层（"开发注意"），漏洞还在其他层，换个错误还能穿透。根治要补所有层。

**为什么 blameless 能挖出真相**：如果追责，当事人会隐瞒决策上下文（"我当时其实怀疑过但时间紧"），复盘就失去了关键信息。blameless 让当事人敢说真话——"我当时的决策依据是 X，现在看来应该 Y"。这些信息是改进系统的关键，追责文化会丢失它们。

**为什么行动项必须机制化**："下次注意""加强培训"这类行动项等于没有行动项——人的注意力会衰减，培训会遗忘。机制化（CI 门禁/监控告警/架构守护测试）是把防护固化到系统里，不依赖人的记忆。一次复盘的终极目标是"让这类事故在机制上不再可能"。

**为什么度量闭环率**：复盘报告写得好不代表改进落地。跟踪行动项闭环率（完成/总数）才能验证复盘的真实价值。完成率低（< 50%）说明复盘流于形式，要反思复盘流程本身。完成率高（> 80%）且同类事故不再发生，说明复盘有效。

## 五、AI 架构师加问：5 个 AI 场景的事故复盘

1. **AI 系统事故复盘有什么特殊点？**
   AI 事故特殊在于"非确定性"——同输入可能不同输出，复现难。复盘要保留当时的 prompt、模型版本、随机种子、上下文，才能回放。根因可能是 prompt 变更/模型升级/数据漂移/RAG 召回异常。行动项要覆盖：prompt 版本控制、模型回归测试、eval 门禁、RAG 质量监控。

2. **LLM 幻觉导致的事故怎么复盘？**
   根因分析：幻觉来源是模型本身、prompt 不当、还是 RAG 喂了错误上下文？5-Whys 追到系统性根因：缺输出 schema 校验/缺引用校验/缺人工确认。行动项：加 schema 校验、加 faithfulness 检测、高敏操作人工确认。监控 hallucination_rate，设阈值告警。

3. **AI 系统的 MTTD/MTTR 怎么优化？**
   AI 事故检测难（输出非确定，难判断"对不对"）。优化 MTTD：在线质量监控（user_feedback_score 下降告警）、输出异常检测（schema 违规率、拒答率飙升）、成本异常检测（token_cost 飙升）。优化 MTTR：模型版本回滚（秒切上一版本）、prompt 版本回滚、降级到规则引擎。

4. **Agent 执行错误操作（如越权退款）怎么复盘？**
   这类事故严重度高（资金/数据影响）。5-Whys 追到：工具白名单不全/服务端没二次校验权限/高敏操作没人工确认。根因往往是"权限边界设计缺陷"。行动项：补全工具权限矩阵、所有写操作服务端校验、高敏操作强制人工确认、tool_permission_violation_rate 零容忍告警。

5. **怎么把 AI 事故教训沉淀为机制？**
   每个 AI 事故复盘输出机制化行动项：prompt 事故→prompt 版本控制 + eval 门禁；幻觉事故→schema 校验 + 引用校验；越权事故→权限矩阵 + 人工确认；成本事故→token 预算 + 模型路由。复盘文档入库，新 AI 系统设计时必须 review 历史事故清单，避免重复踩坑。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"blameless、时间线、5-Whys、机制化、闭环"** 五个词。

- **blameless**：不追个人追系统，不批评要学习
- **时间线**：分钟级还原"发生了什么"，是分析基础
- **5-Whys**：追到系统性根因（流程/工具/架构缺陷），不是"人为失误"
- **机制化**：行动项是监控/规范/工具，不是"下次注意"
- **闭环**：跟踪行动项完成率，完成才有价值

### 面试现场 60 秒回答

> 事故复盘的核心是 blameless 文化——不追责个人追系统缺陷。流程上，故障恢复后 48 小时内开复盘会，先还原故障时间线（分钟级，从变更到检测到定位到恢复），再用 5-Whys 逼近系统性根因。比如订单超时事故，5-Whys 从"线程池打满"追到"慢 SQL"追到"缺 SQL 性能门禁（Review+CI+预发都没拦住）"——根因是机制缺陷不是开发疏忽。行动项必须机制化：加 CI SQL 扫描、加熔断、加一键回滚，每个有 owner + deadline + 验证指标。最后跟踪闭环率，完成率 > 80% 才算复盘有效。度量 MTTD/MTTI/MTTR 持续优化应急能力。一句话：复盘的价值在让同类事故在机制上不再可能，不是写报告。

| 追问层级 | 面试官可能这样问 | 高分回答方向 |

| 目标追问 | 为什么不直接处罚犯错的人，要搞 blameless 复盘？ | 处罚个人只补了最后一层奶酪，系统漏洞还在，换个错误还会穿透。blameless 挖出系统性根因（为什么系统允许这个错误），根治。证明：追责文化的团队事故重复率高，blameless 文化的团队（如 Google SRE）事故率持续下降 |

### 现场对话示例

**面试官**：你说 5-Whys 追到系统性根因，但有时候就是开发写错了代码，这不是个人问题吗？

**候选人**：开发写错代码是直接原因，但不是系统性根因。要继续问：为什么写错的代码能进入生产？Review 为什么没拦住？测试为什么没覆盖？CI 为什么没扫描？预发为什么没测出？这一串防护层为什么全漏了？根因是"防护网有漏洞"，不是"开发写错了"。解法是补防护网（CI 门禁/测试用例/Review checklist），让下次即使写错也进不了生产。处罚这个开发，下个开发还会写错——因为防护网还是漏的。

**面试官**：复盘行动项总是完不成，怎么办？

**候选人**：三个手段。第一，行动项必须有 owner + deadline，不能是"团队负责"（等于没人负责）。第二，周会 review 进度，超期升级 owner 主管，仍不完成标记"已知风险接受"（要主管签字，把风险显性化）。第三，复盘会本身检查上一次复盘的行动项完成率——完不成不准开新复盘，强制闭环。如果行动项经常完不成，反思是不是行动项太大（拆小）或优先级不对（重新排序），不是放弃跟踪。

**面试官**：故障时大家都在救火，证据可能丢了，怎么还原时间线？

**候选人**：证据保全是关键。故障发生的第一动作是"保留现场"——dump 线程栈（jstack）、dump 堆（jmap）、保存慢查询日志、截图监控大盘、保留变更记录。这些在应急 SOP 里明确规定。如果证据已丢，靠当事人回忆 + 监控回溯（Prometheus 历史数据）+ 变更记录（发布平台）拼凑，标注"推测"vs"确认"。证据缺失本身也是一个行动项——"为什么监控没覆盖这个维度"，补监控。

## 常见考点

1. **5-Whys 怎么做？**——从表层现象开始连续追问"为什么"，每层用证据支撑（不猜测），追到系统性根因（流程/工具/架构缺陷）停止。多根因并行追。验证：反向推演"如果这个机制存在，事故还会发生吗"，回答"不会"才算到位。不要追到个人（"为什么招错人"）。
2. **blameless 是什么？**——复盘时不追责个人、不批评当事人的文化。核心理念：相信每个人在当时信息下做了最优决策，事故是系统性缺陷导致。作用：让当事人敢说真话，挖出系统性根因。不是不追责（能力/态度问题走管理手段），是不在复盘会公开追责。
3. **MTTD/MTTI/MTTR 是什么？**——MTTD（Mean Time To Detect，从故障发生到发现的平均时间，优化靠监控告警）；MTTI（Mean Time To Identify，从发现到定位根因的时间，优化靠可观测性/预案）；MTTR（Mean Time To Repair，从定位到恢复的时间，优化靠回滚/降级/应急 SOP）。三者合计是故障总时长。
4. **故障分级怎么定？**——P0（全站不可用/资金损失/数据泄露，CTO 级响应）；P1（核心功能不可用，主管级响应）；P2（部分功能异常/性能下降，oncall 响应）；P3（体验问题/个别 case，工单处理）。分级决定响应级别、复盘深度、通知范围。
5. **行动项怎么保证落地？**——owner 制（明确负责人）+ deadline（明确期限）+ 验证指标（可量化完成标准）+ 闭环跟踪（周会 review 进度，超期升级）+ 复盘会检查上次完成率。机制化行动项（监控/CI/规范）优于"下次注意"。完成率 > 80% 才算复盘有效。

## 结构化回答

**30 秒电梯演讲：** 事故复盘不是追责会，是学习会。根因分析用 5-Whys 逼近系统性根因（不是某人的失误，而是系统为什么允许这个失误发生）。复盘的价值在于把一次性教训变成机制（监控/告警/规范/演练），让同类事故不再发生。blameless 文化是前提——不追究个人，否则没人敢说真话

**展开框架：**
1. **blameless 文化** — 不追究个人，追究系统；不批评，要学习
2. **5-Whys 根因分析** — 连续追问为什么，从表层（现象）到深层（系统缺陷）
3. **故障时间线** — 分钟级还原"发生了什么"，是分析的基础

**收尾：** 以上是我的整体思路。您想继续深入聊——5-Whys 什么时候停？

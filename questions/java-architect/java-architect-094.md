# 【Java 后端架构师】团队工程规范、代码评审与质量度量

> 适用场景：JD 核心技术。一个 30 人的后端团队，没有统一的编码规范，每个人的代码风格、异常处理、日志习惯都不一样。新人来了 3 个月还看不懂老代码。线上事故复盘发现 60% 是"低级错误"（空指针、事务没加、幂等没做）。架构师要做的是建立一套规范 + review + 度量体系，让团队代码质量从"依赖个人"变成"机制保障"。

## 一、概念层：规范的三层与质量的两类指标

### 规范三层（从细到粗）

| 层级 | 内容 | 执行方式 | 工具 |
|------|------|---------|------|
| **编码规范** | 命名/格式/注释/异常处理 | 工具强制（CI 门禁） | checkstyle/SonarQube/SpotBugs |
| **设计规范** | 模块划分/接口设计/事务边界 | Code Review 把关 | Review checklist |
| **架构规范** | 技术选型/服务边界/一致性策略 | ADR 沉淀 + 架构评审 | ADR 仓库 |

### 质量度量两类指标

| 类型 | 指标 | 作用 | 例子 |
|------|------|------|------|
| **领先指标** | 过程质量 | 驱动改进（事前） | 测试覆盖率、静态扫描问题数、review 轮次/时长 |
| **滞后指标** | 结果质量 | 验证效果（事后） | 线上 bug 密度、bug 逃逸率、MTTR、变更失败率 |

**关键**：领先指标驱动日常改进（覆盖率低就补测试），滞后指标验证整体效果（bug 密度降了说明规范有效）。只看滞后指标是"事后诸葛亮"，只看领先指标可能"为了指标而指标"。

## 二、机制层：编码规范与工具自动化

### 2.1 Java 编码规范核心项（工具强制）

```xml
<!-- checkstyle.xml 核心规则 -->
<module name="Checker">
    <!-- 命名规范 -->
    <module name="TypeName"/>              <!-- 类名 UpperCamelCase -->
    <module name="ConstantName"/>          <!-- 常量 UPPER_SNAKE -->
    <module name="LocalVariableName"/>     <!-- 局部变量 lowerCamel -->

    <!-- 异常处理（高频踩坑） -->
    <module name="IllegalCatch">           <!-- 禁止 catch Exception/Throwable -->
        <property name="illegalClassNames" value="Exception, Throwable, RuntimeException"/>
    </module>
    <module name="EmptyBlock"/>            <!-- 禁止空 catch 块 -->

    <!-- 复杂度 -->
    <module name="CyclomaticComplexity">   <!-- 圈复杂度 < 10 -->
        <property name="max" value="10"/>
    </module>
    <module name="MethodLength">           <!-- 方法 < 80 行 -->
        <property name="max" value="80"/>
    </module>
</module>
```

### 2.2 SonarQube 质量门禁（CI 强制）

```yaml
# sonar-project.properties
sonar.qualitygate.wait=true              # 质量门禁不过 CI 失败

# 质量门禁规则（SonarQube 配置）：
# - Coverage on new code > 70%          新代码覆盖率
# - Duplicated lines on new code < 3%   新代码重复率
# - Maintainability Rating A            可维护性 A 级
# - Reliability Rating A                可靠性 A 级
# - Security Rating A                   安全性 A 级
# - Bugs = 0                            新代码 0 bug
```

### 2.3 CI 流水线（质量门禁串起来）

```yaml
# .gitlab-ci.yml 或 Jenkinsfile
stages:
  - lint            # checkstyle + SpotBugs（编码规范）
  - test            # 单元测试（覆盖率门禁）
  - scan            # SonarQube（质量门禁）
  - build           # 构建
  - deploy          # 部署（只有前面全过才到这）

lint:
  stage: lint
  script:
    - mvn checkstyle:check              # 不过则 CI 失败，PR 不能合并

test:
  stage: test
  script:
    - mvn test jacoco:report
    - mvn sonar:sonar                   # 覆盖率 + 质量门禁
  artifacts:
    reports:
      coverage_report:
        coverage_format: jacoco
```

## 三、实战层：Code Review Checklist

### 3.1 Review 必查项 Checklist（Java 后端）

```markdown
# Code Review Checklist

## 正确性（必查）
- [ ] 空指针防护：外部输入、DB 查询结果、RPC 返回都做了 null 检查
- [ ] 事务边界：写操作在事务内，事务范围合理（不含 RPC 调用），传播行为正确
- [ ] 幂等性：重复请求不会导致重复扣款/重复创建（幂等键/唯一约束/状态机）
- [ ] 并发安全：共享状态有锁/原子操作/CAS，无竞态条件
- [ ] 边界条件：0、负数、空集合、超长字符串、并发量峰值

## 异常处理（必查）
- [ ] 异常分类：业务异常（checked/自定义）vs 系统异常（unchecked）分开处理
- [ ] 不吞异常：catch 块有日志或重新抛出，不空 catch
- [ ] 不 catch Throwable：只 catch 具体异常类型
- [ ] 资源释放：try-with-resources 或 finally 关闭连接/流

## 日志（必查）
- [ ] 关键操作有日志（订单创建、支付、状态流转）
- [ ] 日志带 traceId 和业务 ID（orderId/userId）
- [ ] 日志级别正确（ERROR 告警、INFO 关键事件、DEBUG 调试）
- [ ] 不打印敏感信息（密码、token、身份证）

## 性能（重点查）
- [ ] 无 N+1 查询（循环里查 DB）
- [ ] 无大事务（事务里不含 RPC/慢操作）
- [ ] 缓存使用合理（热点缓存、失效策略）
- [ ] 线程池配置合理（隔离、有界队列、拒绝策略）

## 安全（必查）
- [ ] SQL 注入防护（参数化查询，不拼 SQL）
- [ ] XSS 防护（输出转义）
- [ ] 权限校验（接口有鉴权，数据有越权检查）
- [ ] 敏感数据加密（密码 bcrypt、敏感字段脱敏）

## 设计（建议查）
- [ ] 单一职责（方法/类只做一件事）
- [ ] 命名清晰（名字能表达意图，不缩写）
- [ ] 圈复杂度 < 10（超了拆分）
- [ ] 重复代码 < 3 处（超了抽象）
```

### 3.2 Review 流程与效率

```
PR 提交 → 工具自动检查（checkstyle/SonarQube/单测）
              │
              ▼ 工具全过
         人工 Review（1-2 人）
              │
      ┌───────┴───────┐
      │               │
   通过             需修改
      │               │
   合并           返回修改 → 再提交
```

**效率要点**：
- PR < 400 行（大 PR 拆分，review 质量随行数下降）
- 工具先过（让工具做机械检查，人只看逻辑/设计/业务）
- Reviewer 24 小时内响应（不阻塞开发节奏）
- 至少 1 人 approve 才能合并，核心链路 2 人

## 四、质量度量实战

### 4.1 度量看板（领先 + 滞后）

```markdown
# 质量看板（每周/每月 review）

## 领先指标（驱动改进）
| 指标 | 目标 | 当前 | 趋势 |
|------|------|------|------|
| 单元测试覆盖率（新代码） | > 70% | 65% | ↑ |
| SonarQube 问题数（新代码） | 0 | 2 | → |
| Code Review 平均轮次 | < 2 轮 | 2.5 轮 | ↓ |
| PR 平均大小 | < 400 行 | 520 行 | ↓ |
| 静态扫描 Critical 数 | 0 | 0 | → |

## 滞后指标（验证效果）
| 指标 | 目标 | 当前 | 趋势 |
|------|------|------|------|
| 线上 bug 密度（个/千行） | < 0.5 | 0.8 | ↓ |
| Bug 逃逸率（线上/总发现） | < 10% | 15% | ↓ |
| MTTR（平均修复时间） | < 2h | 3.2h | ↓ |
| 变更失败率 | < 5% | 8% | ↓ |
```

### 4.2 质量改进闭环（PDCA）

```
Plan:    发现 bug 密度 0.8 超目标，分析根因（60% 是空指针/事务遗漏）
Do:      更新 review checklist 增加"空指针/事务"必查项，工具加 SpotBugs 规则
Check:   1 个月后 bug 密度降到 0.5
Act:     把有效检查项固化进工具（SpotBugs 规则），减少人工依赖
```

## 五、底层本质：为什么规范 + review + 度量缺一不可

**为什么规范必须工具强制**：文档形式的规范等于没有——没人会背 50 条规则再写代码。工具强制（CI 门禁）让规范自动执行，开发者不需要记住规则，违反了 CI 直接红。checkstyle/SonarQube 把"规范"变成"编译错误"级别的硬约束。

**为什么 Code Review 的核心不是找 bug**：找 bug 是测试的活（单测/集成测试/E2E）。Review 的真正价值是：(1) 知识共享（代码至少两人懂，不是单点）；(2) 一致性（全团队风格统一，代码像一人写的）；(3) 早期发现设计问题（比单测更能发现"虽然能跑但设计差"的问题）；(4) 培养新人（review 意见是最好的教学）。把 review 当"找 bug 工具"是低估了它。

**为什么领先指标比滞后指标重要**：滞后指标（线上 bug 数）发现问题已经是结果，改进要等下一个周期。领先指标（覆盖率、扫描问题数）是过程指标，能在问题进入生产前预警。质量管理的重心应在领先指标（预防），滞后指标只用于验证整体趋势。

**为什么 shift-left 成本最低**：bug 在需求阶段发现修复成本 1x，编码阶段 5x，测试阶段 10x，上线后 100x+（含用户影响/紧急修复/信誉损失）。规范 + review + 单测把问题挡在编码阶段（shift-left），比上线后靠监控发现便宜得多。这是工程规范的根本经济动力。

## 六、AI 架构师加问：5 个 AI 场景的规范与质量

1. **AI 系统的 Code Review 有什么特殊检查项？**
   AI 专属 review 项：(1) prompt 是否配置化（不写死代码里）；(2) LLM 输出是否有 schema 校验；(3) 是否有降级路径（LLM 挂了怎么办）；(4) 是否做了 PII 脱敏（用户输入进 prompt 前）；(5) 是否有 token 成本控制（预算/路由）；(6) 是否有 prompt injection 防护。这些是 AI 系统的高频踩坑点。

2. **AI 代码的质量度量怎么补充？**
   传统指标（覆盖率/bug 密度）不够，AI 补充：(1) prompt 版本化率（多少 prompt 在管理平台而非代码里）；(2) eval 集覆盖率（核心场景有 eval 的比例）；(3) 降级链完整性（有 fallback 的 LLM 调用占比）；(4) 幻觉率（线上 user_feedback 差评中幻觉占比）。这些指标反映 AI 工程化成熟度。

3. **LLM 辅助写代码（Copilot/Cursor）怎么 review？**
   LLM 生成的代码风险：(1) 看似正确但有隐藏 bug（LLM 幻觉 API）；(2) 安全漏洞（LLM 可能生成有注入风险的代码）；(3) 许可证问题（可能抄了 GPL 代码）。review 要更严：所有 LLM 生成代码必须人工逐行 review，重点查 API 存在性/安全性/边界条件。工具加额外的 AI 代码扫描（如检测疑似幻觉的 API 调用）。

4. **怎么规范团队的 prompt 工程？**
   prompt 规范：(1) prompt 必须在管理平台版本化（不写死代码）；(2) 每个 prompt 必须有 eval 集；(3) prompt 变更走 review（不只是代码 review）；(4) prompt 里的变量必须 sanitize（防 injection）；(5) 高风险 prompt（涉及权限/资金）双人审批。把 prompt 当代码管，纳入 review 流程。

5. **AI 系统的质量门禁怎么设计？**
   CI 门禁补充 AI 项：(1) prompt 变更必须跑 eval（task_accuracy 不退化才合并）；(2) 新增 LLM 调用必须有降级路径（CI 检查 fallback 配置）；(3) prompt injection 测试通过（红队测试集）；(4) token 成本预算配置存在。这些门禁把 AI 工程规范自动化执行。

## 七、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"三层规范、四价值、领先滞后、shift-left"** 四个词。

- **三层规范**：编码（工具强制 checkstyle/SonarQube）、设计（review checklist）、架构（ADR）
- **四价值**：Code Review 找问题 + 知识共享 + 一致性 + 培养新人
- **领先滞后**：领先指标（覆盖率/扫描）驱动改进，滞后指标（bug/MTTR）验证效果
- **shift-left**：质量内建进代码，比上线后发现便宜 10-100 倍

### 面试现场 60 秒回答

> 工程规范我分三层落地。编码规范用工具强制——checkstyle 管命名/复杂度，SonarQube 质量门禁（覆盖率>70%、0 bug、0 漏洞），CI 不过不让合并。设计规范靠 Code Review checklist——必查正确性（空指针/事务/幂等）、异常处理、日志、安全，PR < 400 行保证 review 质量。架构规范用 ADR 沉淀决策。质量度量分领先指标（覆盖率、扫描问题数、review 轮次）驱动日常改进，滞后指标（bug 密度、逃逸率、MTTR）验证整体效果。Code Review 的核心不是找 bug 而是知识共享和一致性——代码至少两人懂不是单点。整体思路是 shift-left，把质量内建进代码，比上线后靠监控发现便宜 10-100 倍。

### 反问面试官

> 贵司工程规范是工具强制（CI 门禁）还是文档要求？工具强制的规范执行率高，文档式的容易流于形式。这决定我是建工具链还是写规范文档。

## 七、苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 为什么要规范，让高手自由发挥不更好？ | 高手自由发挥对个人效率高但对团队是灾难——代码风格不一无一致性、知识单点（高手离职代码没人懂）、新人学习成本高。规范的作用是让团队下限可控，高手在上限发挥。证明：无规范团队新人上手 3 个月，有规范团队 1 个月 |
| 证据追问 | 你怎么证明规范真的提升了质量？ | 对比规范前后的滞后指标：线上 bug 密度（应降）、bug 逃逸率（应降）、MTTR（应降）、变更失败率（应降）。同时看领先指标趋势（覆盖率升、扫描问题降）。要有上线前后 3-6 个月的数据对比 |
| 边界追问 | 规范会不会拖慢迭代速度？ | 短期会（学习成本 + review 时间），长期提速（bug 少返工少、新人快上手、代码可维护）。关键是规范要"精"不要"全"——只强制高 ROI 的规则（空指针/事务/幂等），不搞形式主义（每个 getter 都要注释）|
| 反例追问 | 什么场景规范是过度设计？ | 早期创业项目（活下来比规范重要）、实验性 POC（验证完就扔）、单人项目（自己懂就行）。规范适合多人协作、长期维护、核心链路的场景 |
| 风险追问 | 规范落地最大的风险？ | 团队抵触（觉得被限制/增加负担）。解法：不搞规范大爆发（一次出 50 条），分批落地（先核心痛点如空指针/事务），工具强制（不靠人记），解释 why（每条规范附踩过的坑），用效果建立信任 |
| 验证追问 | 怎么证明 Code Review 不是走过场？ | 抽审 review 意见质量（是否有实质内容而非"+1"）、统计 review 发现的问题数（应该有合理量）、对比有 review 和无 review 的 bug 逃逸率。走过场的 review 发现不了问题，有效 review 能拦下 30-50% 的 bug |
| 沉淀追问 | 团队规范沉淀什么？ | 编码规范文档 + checkstyle 配置、SonarQube 质量门禁配置、Review checklist 模板（分业务类型）、ADR 模板、质量看板模板、新人 onboarding 文档（规范 + checklist） |

### 现场对话示例

**面试官**：你说工具强制，但如果团队觉得 checkstyle 规则太严怎么办？

**候选人**：分情况。如果规则确实不合理（如强制每个方法必须有 javadoc，但 getter/setter 没必要），调整规则（规则服务于质量不是教条）。如果规则合理但团队不适应（如圈复杂度 < 10，有人习惯写长方法），坚持规则但给过渡期（先 < 15，逐步降到 < 10）+ 培训（教怎么拆分）。关键是规则要有 why（"圈复杂度高导致测试困难和 bug 多"），让团队理解不是管束而是帮助。落地时先选共识最强的规则（如空指针检查），用效果建立信任后再推更严的。

**面试官**：Code Review 太慢影响发版节奏，怎么办？

**候选人**：三个手段提速。第一，PR 拆小（< 400 行），小 PR review 快且质量高（大 PR reviewer 会疲劳走马观花）。第二，工具先过（checkstyle/SonarQube/单测全过才到人工），人只 review 工具查不了的（业务逻辑/设计）。第三，Reviewer SLA（24 小时内响应），阻塞发版的 review 升级（找 backup reviewer）。核心是让 review 聚焦在高价值检查上，不浪费时间在工具能做的事。

**面试官**：测试覆盖率 70%，剩下 30% 怎么办？

**候选人**：不追求 100%。覆盖率要看"覆盖的是什么"而非数字。核心链路（下单/支付/退款）覆盖率 90%+，非核心（管理后台/统计）50-60% 够用，getter/setter/POJO 不用测。剩下 30% 评估：如果是关键路径漏测，补上；如果是难测试的（如依赖外部系统的集成），用集成测试或契约测试覆盖；如果确实低风险，记录为已知缺口接受风险。目标是"关键路径全覆盖"，不是"行数 100%"。

## 常见考点

1. **Code Review 看什么？**——正确性（空指针/事务/幂等/边界）、异常处理（不吞异常/分类）、日志（traceId/级别/敏感信息）、性能（N+1/大事务）、安全（注入/越权）、设计（职责/命名/复杂度）。工具能查的让工具查（checkstyle），人查工具查不了的。
2. **测试覆盖率多少合适？**——核心链路 80%+，非核心 50-60%，整体 60-70%。追求关键路径覆盖而非行数 100%。getter/setter/POJO 不必测。覆盖率是领先指标，要看趋势。
3. **SonarQube 质量门禁是什么？**——CI 级别的质量检查：新代码覆盖率 > 70%、重复率 < 3%、0 bug、0 漏洞、可维护性/可靠性/安全评级 A。门禁不过 CI 失败，PR 不能合并。把规范变成硬约束。
4. **shift-left 是什么？**——质量左移，把问题发现在越早阶段越好（需求 < 编码 < 测试 < 上线）。越早发现修复成本越低（1x vs 100x）。手段：规范、review、单测、静态扫描，都是 shift-left。
5. **领先指标和滞后指标区别？**——领先指标是过程质量（覆盖率、扫描问题数、review 轮次），事前驱动改进；滞后指标是结果质量（bug 密度、MTTR、变更失败率），事后验证效果。质量管理重心应在领先指标（预防优于治疗）。

## 结构化回答

**30 秒电梯演讲：** 工程规范不是写文档，是把踩过的坑变成团队不会重复踩的机制。好的规范是可执行的（checklist + 工具自动化），不是挂在墙上的口号。Code Review 的核心不是找 bug（那是测试的活），而是知识共享 + 一致性 + 早期发现问题。质量度量要度行为而非结果（如测试覆盖率是行为，线上 bug 数是结果滞后指标）

**展开框架：**
1. **规范分三层** — 编码规范（工具强制）、设计规范（review 把关）、架构规范（ADR 沉淀）
2. **Code Review 四价值** — 找问题、知识共享、一致性、培养新人
3. **Review checklist** — 必查项（空指针/事务/幂等/异常/日志/安全）+ 业务项

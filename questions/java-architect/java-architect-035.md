# 【Java 后端架构师】削峰填谷、排队论与容量保护

> 适用场景：JD 核心技术。618/双 11 大促 0 点峰值流量是日常的 10-50 倍，下单/秒杀/抢券场景瞬时流量可能达到百万 QPS。架构师必须能用 Little's Law 算容量、用 MQ/线程池削峰、用多级限流保护容量，否则大促必挂。

## 一、概念层：峰值、稳态、容量上限

**流量形态的本质认识**：

```
日常流量：稳态 ~ 1万 QPS，波动小
大促流量：瞬时峰值 30万 QPS，持续 5-10 分钟
           │
  30万 QPS │       ██
           │      ████
           │     ████████
           │    ██████████████
   1万 QPS │█████████████████████████████████████
           └──────────────────────────────────────> 时间
              日常         大促峰值        日常
```

**关键认知**：峰值流量不可预测也不可承载（成本不允许按峰值配资源）。必须用"削峰"把峰值摊平到更长周期，或用"限流"拒绝超量流量。

**三种应对策略对比**：

| 策略 | 手段 | 保什么 | 牺牲什么 | 适用 |
|------|------|--------|---------|------|
| **削峰** | MQ 异步、线程池排队 | 全部请求最终处理 | RT 拉长（毫秒到秒） | 下单、营销、日志 |
| **限流** | 令牌桶/漏桶拒绝超量 | 已接收请求的体验 | 部分请求被拒 | 秒杀、抢券 |
| **降级** | 关闭非核心、返回默认 | 核心链路 | 非核心功能 | 故障兜底 |

## 二、机制层：Little's Law 与容量计算

**Little's Law 是排队论基础**：

```
L = λ × W

L：系统内平均请求数（并发数 / 队列长度）
λ：请求到达率（QPS）
W：单请求平均停留时间（处理 + 等待，秒）

例：1万 QPS、单请求 50ms
  L = 10000 × 0.05 = 500
  系统内同时有 500 个请求在处理或等待
```

**容量估算实例**（JD 下单场景）：

```
峰值目标：1万 QPS
单请求处理时间（含 DB、缓存、下游）：100ms = 0.1s
应用 Little's Law：
  L = 10000 × 0.1 = 1000
  需要并发处理 1000 个请求

单机线程池上限：200（CPU 16 核，单线程 1ms 调度 + 5ms IO）
集群机器数 = 1000 / 200 = 5 台

留 30% 余量（防 GC 抖动、慢查询、网络毛刺）：
  实际配 7 台，每台线程池 200
  限流阈值 = 5 × 200 / 0.1 × 0.7 = 7000 QPS（集群）
```

**线程池配置实例**：

```java
// 基于 Little's Law 计算的线程池
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    200,                                     // corePoolSize
    200,                                     // maxPoolSize（核心 = 最大，避免动态扩容抖动）
    60L, TimeUnit.SECONDS,                   // keepAliveTime
    new ArrayBlockingQueue<>(1000),          // 等待队列容量
    new ThreadFactoryBuilder().setNameFormat("order-pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy() // 队列满让调用方执行（背压降速，比拒绝好）
);

// 关键参数解释：
//   队列 1000：超过线程数的请求进队列等待
//   CallerRunsPolicy：队列满时让 Tomcat 线程自己执行（背压），降速而非拒绝
```

## 三、机制层：MQ 削峰工程实现

**MQ 异步削峰流程**：

```
同步链路（无削峰）：
  User → Gateway → OrderService ──> DB（写订单）
                                 ──> 库存（扣库存）
                                 ──> 营销（发券）
                                 ──> 履约（生成运单）
  RT = 各步骤总和 ≈ 500ms，峰值扛不住

异步链路（MQ 削峰）：
  User → Gateway → OrderService ──> DB（写订单 + outbox）
                                 ──> 返回（RT = 50ms）
                  MQ ──> 库存消费（慢慢扣）
                  MQ ──> 营销消费（慢慢发券）
                  MQ ──> 履约消费（慢慢生成运单）
  主链路 RT 50ms，峰值 10 倍流量能扛；下游各服务按自己节奏消费
```

**MQ 削峰代码**：

```java
@Service
public class OrderService {

    @Transactional
    public OrderResult placeOrder(OrderDTO dto) {
        // 1. 同步：核心 DB 操作（订单、库存校验）
        Order order = orderDao.insert(dto);
        inventoryDao.deduct(dto.getSkuId(), dto.getQty());
        // 2. 同事务写 Outbox 事件（保证事件必发）
        outboxDao.insert(new OrderPlacedEvent(order));

        // 3. 立即返回（不等营销/履约）
        return OrderResult.success(order.getId());
    }
}

// 营销消费：按自己节奏消费，积压 OK
@RocketMQMessageListener(topic = "order_events", consumerGroup = "marketing_group",
    consumeMode = ConsumeMode.CONCURRENTLY,
    consumeThreadMax = 50)   // 50 个消费线程
public class MarketingConsumer implements RocketMQListener<OrderPlacedEvent> {
    // ...
}
```

**MQ 削峰的关键监控**：

```
consumer_lag（积压）：消费速度跟不上生产速度
  ─ 告警阈值：积压 > 10万 或 持续 5 分钟上涨
  ─ 治理：扩消费者、增分区、降级非核心字段处理

produce_latency：生产端发送延迟
  ─ 告警阈值：P99 > 100ms（MQ broker 压力大）
  ─ 治理：扩 broker、本地批量发送

消费端 RT：单消息处理时间
  ─ 告警阈值：P99 > 业务 SLA（如 500ms）
  ─ 治理：优化慢 SQL、加缓存、异步化下游
```

## 四、机制层：容量保护三道防线

**三道防线递进设计**：

```
入口（Nginx/网关）
  └─ 第 1 道：集群限流（按 URL / 租户 / 用户）
       规则：1000 QPS/用户、10万 QPS/集群
       工具：Sentinel 集群限流 / Nginx limit_req

服务（应用层）
  └─ 第 2 道：单机限流 + 线程池隔离
       规则：单机 1万 QPS、核心接口独立线程池
       工具：Sentinel 单机限流 / Resilience4j / 自研

下游（依赖层）
  └─ 第 3 道：熔断 + 降级
       规则：DB/Redis/外部接口异常率 > 50% 触发熔断
       工具：Sentinel / Hystrix / Resilience4j
```

**Sentinel 多级限流配置**：

```java
// 网关层：集群限流（用户维度）
@SentinelResource(value = "gateway_order", blockHandler = "rateLimitHandler")
public Response order(Request req) {
    // ...
}

// 资源规则（Nacos 动态推送）
FlowRule rule1 = new FlowRule("gateway_order");
rule1.setLimitApp("default");
rule1.setGrade(RuleConstant.FLOW_GRADE_QPS);
rule1.setCount(10000);   // 集群 1万 QPS
rule1.setClusterMode(true);   // 集群限流
rule1.setClusterConfig(new ClusterFlowConfig().setThresholdType(GLOBAL));

// 服务层：单机限流（接口维度）
FlowRule rule2 = new FlowRule("OrderService.placeOrder");
rule2.setCount(1000);   // 单机 1000 QPS
rule2.setGrade(RuleConstant.FLOW_GRADE_QPS);

// 线程池隔离（防级联故障）
FlowRule rule3 = new FlowRule("OrderService.placeOrder");
rule3.setGrade(RuleConstant.FLOW_GRADE_THREAD);   // 线程数限流
rule3.setCount(200);   // 最多 200 并发线程

// 下游熔断
DegradeRule rule4 = new DegradeRule("DB_INSERT");
rule4.setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_RATIO);
rule4.setCount(0.5);            // 异常率 > 50%
rule4.setTimeWindow(30);         // 熔断 30s
rule4.setMinRequestAmount(20);   // 至少 20 请求才统计
```

## 五、实战层：大促容量规划全流程

**JD 大促容量规划 SOP**（架构师必须能讲）：

```
1. 流量预估
   历史大促峰值 × 今年增长系数 × 留余量
   例：去年峰值 50万 QPS × 1.2（增长）× 1.3（余量）= 78万 QPS

2. 单机容量压测
   阶梯加压，找 P99 拐点（P99 突增的 QPS）
   单机上限 = P99 拐点 × 0.8（再留余量）
   例：单机压测 P99 拐点 1.2万 QPS → 上限 1万 QPS

3. 集群容量计算
   集群容量 = 单机上限 × 机器数
   78万 / 1万 = 78 台机器（按可用区分散）

4. 限流阈值配置
   集群限流 = 集群容量 × 0.7（30% 余量）= 55万 QPS
   超过 55万 的流量直接拒绝（保已接收请求体验）

5. 削峰配置
   非核心链路异步化（营销、履约走 MQ）
   核心链路（下单、支付）保持同步
   MQ 消费者按可用容量配置（不超 DB 容量）

6. 预案准备
   限流阈值动态调整（Nacos 推送）
   降级开关（关闭推荐、关闭评价）
   扩容脚本（10 分钟扩 100 台）
```

**容量压测阶梯加压脚本**（Gatling 风格）：

```scala
// 阶梯加压找拐点
val scenario = scenario("Order Placement")
  .exec(http("place_order").post("/order").body(...))

setUp(
  scenario.inject(
    constantUsersPerSec(1000) during (1 minute),    // 1千 QPS 1 分钟
    constantUsersPerSec(3000) during (1 minute),    // 3千
    constantUsersPerSec(5000) during (1 minute),    // 5千
    constantUsersPerSec(8000) during (1 minute),    // 8千
    constantUsersPerSec(10000) during (1 minute),   // 1万
    constantUsersPerSec(12000) during (1 minute)    // 1.2万 —— 看 P99 是否突增
  ).protocols(httpProtocol)
)
// 看 P99 折线图，突增点即单机容量拐点
```

## 六、底层本质：为什么是排队论 + 多级限流

回到第一性：**系统处理能力有物理上限，瞬时流量不可控**。

**Little's Law 的本质**：它揭示了"稳态系统的守恒律"。系统在稳态时，进入的请求数 = 处理的请求数 + 排队的请求数。这个守恒律不依赖于具体实现（不管你是线程池、消息队列、还是 DB 连接池），都满足 L = λW。所以架构师可以用它做容量估算——已知任意两个量算第三个。这是排队论给工程的最大礼物：一个简单公式描述所有排队系统。

**削峰的本质是"时间换空间"**：把瞬时高峰（高 QPS、低 RT）摊平到更长周期（低 QPS、高 RT）。MQ 削峰把 1 万 QPS × 100ms 的同步处理，变成 1 万 QPS 写 MQ（5ms）+ 下游 1 万 QPS 消费（100ms 但并行 50 线程 = 200 QPS 实际）。吞吐不变但 RT 拉长，下游压力均匀化。

**多级限流的本质是"分而治之"**：单点限流做不到精细控制（用户级、接口级、下游级都要管）。多级限流让每层做自己最擅长的事——网关管用户/租户级、服务管接口级、下游管资源级。每层快速失败，避免级联雪崩。

**容量保护的反模式**：①无限排队（队列无限长，RT 飙到超时，最终全部失败）；②无限重试（重试放大流量，雪崩）；③不压测凭感觉配（阈值拍脑袋）。正确姿势是 Little's Law 算容量、压测验证、留余量、动态调整。

## 七、AI 架构师加问：5 个 AI 相关问题

1. **AI 推理服务的容量保护怎么做？**
   AI 推理是 GPU 密集（贵且稀缺），更要严格限流。用 Little's Law 算 GPU 并发：L = λW，1万 QPS × 单推理 200ms = 2000 GPU 并发。配 GPU 队列（按模型分池）+ 请求优先级（VIP 优先）+ 降级（高峰用小模型）。

2. **让 AI 预测大促峰值，AI 接管哪段？**
   AI 用历史大促数据 + 营销活动节奏（券时间、补贴力度）预测峰值 QPS 和峰值时间。AI 输出峰值预估区间，架构师据此配容量。AI 不负责执行（扩容、限流阈值仍人工确认）。

3. **AI 流量调度如何结合削峰？**
   AI 实时分析各服务负载、队列积压，动态调度流量——把流量从高负载路由到低负载机房、降级非核心功能、动态调整限流阈值。这是 AIOps 的典型场景，但要人工兜底（AI 误判可能放大故障）。

4. **AI 推理请求的排队策略怎么设计？**
   多优先级队列（VIP 直插队首、普通请求排队、低优先级 batch 处理）+ 公平调度（避免低优先级饿死）+ 超时丢弃（避免积压）。用 Little's Law 算队列深度，避免无限排队。

5. **AI 推理失败的重试如何避免雪崩？**
   指数退避（base × 2^n，上限 30s）+ 抖动（避免同步重试）+ 熔断（异常率 > 50% 停 30s）+ 降级（小模型或缓存结果）。重试必须有上限（3 次），超限转人工或默认值。

## 八、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"Little's Law 算容量、MQ 削峰拉长 RT、三道防线保护、留 30% 余量"**。

- **Little's Law**：L = λW，并发 = QPS × 单请求时间，容量计算的数学根基
- **削峰**：MQ 异步（主链路 RT 降到 50ms）、线程池排队（CallerRunsPolicy 背压）
- **三道防线**：网关集群限流 → 服务单机限流 + 线程池隔离 → 下游熔断降级
- **容量评估**：压测找单机拐点 × 集群数 × 0.7 = 限流阈值
- **排队 vs 拒绝**：核心业务排队保 RT、非核心拒绝保吞吐

### 拟人化理解

把系统想成 **JD 超市收银区**。Little's Law 算"几个收银台 + 多大排队区"——1万顾客/小时（λ）、单顾客 5 分钟（W），需要 833 个收银台或排队区容量（L = λW）。削峰是"高峰期把顾客放进排队区慢慢结账"（不拒单但等更久）。容量保护是"排队区满了就限流"（拒新顾客保已有顾客体验）。限流阈值留 30% 余量——按 700 容量配置，给 GC 抖动、慢查询留缓冲。

### 面试现场 60 秒回答

> 削峰填谷用 Little's Law 算容量：L = λW，1万 QPS × 100ms = 1000 并发，按 5 台机器 × 200 线程配。削峰靠 MQ 异步——主链路只写订单 DB（50ms），营销/履约/库存异步走 MQ 慢慢消费，下游按自己节奏不扛峰值。容量保护三道防线：网关集群限流（用户/租户维度）、服务单机限流 + 线程池隔离（接口维度）、下游熔断（异常率 > 50% 熔断 30s）。限流阈值 = 单机压测拐点 × 集群数 × 0.7，留 30% 余量防 GC 和慢查询。核心业务排队保 RT，非核心拒绝保吞吐，秒杀场景拒绝 + 异步通知。

### 现场对话示例

**面试官**：你怎么算下单服务的容量？

**候选人**：用 Little's Law。先定目标——峰值 1万 QPS，单请求处理 100ms（含 DB 写、缓存、下游调用）。L = 10000 × 0.1 = 1000 并发。然后压测单机——阶梯加压找 P99 拐点，假设单机拐点 1万 QPS（P99 从 100ms 突增到 800ms）。配 0.8 倍余量，单机上限 8000 QPS，需要 1000/（8000×0.01）= 实际算下来需要约 1000 并发 ÷ 单机并发（8000×0.1=800）≈ 2 台就够，但考虑大促余量配 5 台。集群限流阈值 = 5 × 8000 × 0.7 = 28000 QPS。这套数字背后是压测，不是拍脑袋。

**面试官**：MQ 削峰怎么不把库存扣超？

**候选人**：库存扣减走同步链路（不削峰），确保实时准确。削峰的是营销、履约、通知这类可延后处理的非核心链路。下单流程：① 同步写订单 + 扣库存（核心，不削峰）→ ② 同事务写 outbox 事件 → ③ 立即返回。营销/履约异步消费事件慢慢处理。库存对账定时任务兜底（每 5 分钟对账 DB 与缓存库存差异）。

**面试官**：限流触发后用户看到什么？

**候选人**：分场景。秒杀抢券场景：返回"活动太火爆，请重试"（前端指数退避重试，避免用户反复点）。下单场景：返回排队页（"您前面还有 N 人"），异步处理完成后通知。API 调用：返回 429 Too Many Requests + Retry-After 头。绝不返回 500（误导以为是 bug），要让客户端知道是限流可以重试。

## 常见考点

1. **Little's Law 怎么用？**——L = λW。已知 QPS（λ）和单请求时间（W），算并发（L）。例：1万 QPS × 100ms = 1000 并发，配 5 台机器每台 200 线程。
2. **MQ 削峰的代价？**——RT 拉长（同步 500ms → 异步 50ms 主链路 + 下游慢慢消费）、消费端要幂等、积压监控。实时性要求高的不能削峰。
3. **线程池队列满了怎么办？**——CallerRunsPolicy（让调用方执行，背压降速，推荐）、AbortPolicy（拒绝抛异常）、DiscardPolicy（丢弃，慎用）。
4. **限流阈值怎么定？**——单机压测拐点 × 集群数 × 0.7（留 30% 余量）。例：单机 1万 × 10 台 × 0.7 = 7万 QPS 集群限流。
5. **削峰 vs 限流怎么选？**——核心业务（下单）排队保 RT、非核心（营销）拒绝保吞吐、秒杀场景拒绝 + 异步通知。


## 结构化回答

**30 秒电梯演讲：** 聊到削峰填谷、排队论与容量保护，我的理解是——削峰填谷的本质是"用队列把瞬时高峰流量摊平到更长周期处理"，避免峰值打挂下游。排队论（Little's Law：L = λW）给出容量数学模型——系统稳态时，队列长度 = 到达率 × 平均处理时间。容量保护是"主动设上限"——超过系统能力的流量直接拒绝（快速失败）而非堆积拖垮全局。打个比方，像 JD 超市收银台：削峰是"高峰期把顾客放进排队区慢慢结账"（不拒单但拉长 RT）；容量保护是"排队区满了就限流"（拒单保已有顾客体验）。排队论算"几个收银台 + 多大排队区"才能让顾客等待时间可控。

**展开框架：**
1. **Little's Law** — L = λW（队列长度 = 到达率 × 等待时间），是容量计算的数学基础
2. **削峰三件套** — MQ 异步、线程池排队、信号量限并发
3. **容量保护三道防线** — 网关限流 → 服务限流 → 下游熔断

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：Little's Law 怎么用？您更想看哪个方向？

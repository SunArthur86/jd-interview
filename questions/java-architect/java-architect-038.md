# 【Java 后端架构师】SLO、错误预算与稳定性治理体系

> 适用场景：JD 核心技术。下单、支付、履约核心链路的稳定性必须量化治理——大促前压缩发布、故障后复盘定责、业务要快和运维要稳的矛盾，都要靠 SLO + 错误预算这套机制仲裁。架构师必须能定 SLO、设计错误预算消耗策略、落地告警和发布门禁。

## 一、概念层：SLI / SLO / SLA / 错误预算

**四个概念的关系**（必背）：

```
SLI（Service Level Indicator，指标）
  - 用户视角的可测量数据：请求成功率、P99 延迟、可用率
  - 例子：成功率 = 2xx请求数 / 总请求数

SLO（Service Level Objective，目标）
  - 对 SLI 的承诺值：99.9% 成功率、P99 < 500ms
  - 例子：30 天内成功率 ≥ 99.9%

错误预算（Error Budget）
  - 允许失败的额度 = 1 - SLO
  - SLO 99.9% → 错误预算 0.1%（30 天允许 0.1% 请求失败）
  - 30 天 1 亿请求 → 允许 10 万次失败

SLA（Service Level Agreement，合同）
  - 写进合同的对外承诺（违反赔钱）
  - 比 SLO 宽松（留缓冲）：SLA 99%，SLO 内部定 99.5%
```

**SLI 的黄金信号**（必背，用户视角）：

| SLI | 定义 | 例子 |
|-----|------|------|
| **可用性（Availability）** | 成功请求占比 | 成功率 = 2xx+3xx / 总请求 |
| **延迟（Latency）** | 请求处理时间分布 | P99 < 500ms |
| **吞吐（Throughput）** | 单位时间处理量 | QPS、TPS |
| **饱和度（Saturation）** | 资源使用程度 | CPU 80%、连接池 70% |
| **正确性（Correctness）** | 业务结果正确率 | 订单创建成功率（业务级 SLI） |

**为什么不用 CPU/内存做 SLO**：CPU 80% 不等于用户感知差（可能正常处理）。SLO 必须用用户视角——"请求是否成功、是否够快"，而非"机器是否繁忙"。CPU/内存是诊断指标（出问题时查根因用），不是 SLO。

## 二、机制层：SLO 与错误预算计算

**SLO 计算实例**（JD 下单服务）：

```
SLO 定义：
  成功率 ≥ 99.95%（28 天滚动窗口）
  P99 延迟 ≤ 500ms（28 天滚动窗口）

错误预算：
  成功率预算 = 1 - 99.95% = 0.05%
  28 天 1 亿请求 → 允许 5 万次失败
  每天允许 ~1785 次失败

预算消耗跟踪：
  Day 1：失败 500 次（消耗 28%）—— 正常
  Day 2：失败 3000 次（消耗 168%）—— 超预算，触发告警
  Day 3：失败 200 次（消耗 11%）—— 正常
```

**SLO 计算代码**（Prometheus 查询）：

```promql
# 成功率 SLI（28 天窗口）
sum(rate(http_requests_total{status!~"5.."}[28d]))
/
sum(rate(http_requests_total[28d]))

# 错误预算消耗率
1 -
(
  sum(rate(http_requests_total{status!~"5.."}[28d]))
  /
  sum(rate(http_requests_total[28d]))
)
>
0.0005   # 错误预算 0.05%

# 燃烧率（Burn Rate）= 实际错误率 / 错误预算
# 燃烧率 > 1 表示消耗比预期快，需告警
(
  sum(rate(http_requests_total{status=~"5.."}[1h]))
  /
  sum(rate(http_requests_total[1h]))
)
/
0.0005
```

**多窗口告警策略**（Google SRE 推荐）：

```
短窗口（5m/1h）+ 长窗口（1h/6h）组合，避免误报：

  告警条件：1h 错误率 > 14.4 倍预算 AND 5m 错误率 > 14.4 倍预算
  → 1 小时内会耗尽 2% 预算，紧急告警（页面级）

  告警条件：6h 错误率 > 6 倍预算 AND 3h 错误率 > 6 倍预算
  → 3 天会耗尽 10% 预算，warning 告警（工单级）

燃烧率（Burn Rate）表：
  14.4x → 1 小时耗 2% 预算（紧急）
  6x    → 3 天耗 10% 预算（关注）
  3x    → 3 天耗 5% 预算（提示）
  1x    → 30 天耗 100% 预算（正常）
```

## 三、机制层：错误预算驱动的发布治理

**错误预算的核心价值——发布门禁**：

```
错误预算剩余 > 50%：
  → 可正常发布新功能
  → 可激进做架构改造
  → 可做实验性变更

错误预算剩余 20%-50%：
  → 发布需更严格审批
  → 只允许修 bug，不做大改造
  → 加强灰度和监控

错误预算剩余 < 20% 或耗尽：
  → 冻结非紧急发布
  → 优先投入稳定性治理（修 bug、补压测、清债务）
  → 故障复盘必做
```

**SLO-driven 发布门禁代码**（CI/CD 集成）：

```python
# 发布前检查错误预算（CI pipeline）
def check_error_budget(service):
    remaining = query_slo_remaining(service)   # 查 Prometheus
    if remaining < 0.2:   # < 20%
        if not is_emergency_fix():
            raise Exception(f"Error budget exhausted ({remaining}), publish frozen")
        else:
            warn("Emergency fix, requires SRE approval")

    if remaining < 0.5:   # < 50%
        require_approval("team_lead")   # 需审批

    # 通过，允许发布
    return True
```

**SLO 仪表盘**（Grafana 大盘）：

```
Row 1：SLO 总览
  - 当前 28 天成功率（gauge，红黄绿）
  - 错误预算剩余百分比（gauge，< 20% 红色告警）
  - 燃烧率（趋势图）

Row 2：SLI 详情
  - 成功率趋势（28 天）
  - P99 延迟趋势（28 天）
  - 失败请求分布（按错误码、按接口）

Row 3：预算消耗分析
  - 按原因分类的预算消耗（发布抖动、故障、依赖异常）
  - 历史消耗趋势（看是否恶化）
```

## 四、实战层：SLO 制定与迭代

**SLO 制定五步法**（架构师必须能讲）：

```
Step 1：识别用户旅程（User Journey）
  - 下单：浏览商品 → 加购 → 下单 → 支付 → 出库
  - 选最关键的旅程（下单支付）

Step 2：定义 SLI（用户视角）
  - 成功率：HTTP 2xx / 总请求
  - 延迟：P99 < 500ms（用户能感知）
  - 不要用 CPU、GC（机器视角）

Step 3：定 SLO 初值（基于历史数据）
  - 看过去 90 天实际表现，取下限做初值
  - 例：过去 90 天成功率 99.96%-99.99% → SLO 定 99.9%（留缓冲）

Step 4：评估成本和用户期望
  - 用户能接受多少失败？（支付失败比搜索失败严重）
  - 提高 SLO 的成本？（99.99% → 99.999% 成本翻 10 倍）

Step 5：迭代（季度回顾）
  - SLO 太松（从不违规）→ 用户可能不满，收紧
  - SLO 太严（频繁违规）→ 团队疲惫，放松或投入治理
  - 业务变化（大促）→ 临时调整 SLO
```

**不同链路的 SLO 差异化**（JD 风格）：

| 链路 | 可用性 SLO | 延迟 SLO | 错误预算 |
|------|-----------|---------|---------|
| 支付（核心） | 99.99% | P99 < 200ms | 0.01%（极严） |
| 下单（核心） | 99.95% | P99 < 500ms | 0.05% |
| 搜索（重要） | 99.9% | P99 < 1s | 0.1% |
| 推荐（非核心） | 99.5% | P99 < 2s | 0.5% |
| 评论（非核心） | 99% | P99 < 3s | 1% |

**关键认知**：不要所有服务都定 99.99%，成本爆炸。核心链路严（支付 99.99%），非核心松（评论 99%），整体成本最优。

## 五、实战层：故障复盘与错误预算归因

**错误预算归因**（区分合理消耗和异常消耗）：

```
合理消耗（无需治理）：
  - 计划内发布抖动（每次发布消耗 0.01% 预算）
  - 已知的依赖波动（下游偶尔超时）

异常消耗（必须治理）：
  - 故障（一次大故障消耗 5% 预算）
  - 代码 bug 引入的回归
  - 容量不足（峰值超限）

错误预算消耗报表：
  | 日期       | 消耗 | 原因               | 归类     |
  | 2026-06-01 | 0.01%| 发布 v2.1 抖动     | 合理     |
  | 2026-06-05 | 3.5% | DB 主从切换故障    | 异常     |
  | 2026-06-10 | 0.5% | 下游支付超时       | 异常     |
```

**故障复盘模板**（基于 SLO）：

```
故障回顾：
  影响：成功率从 99.95% 降到 99.2%（消耗 30% 月度预算）
  根因：DB 连接池配置 50，峰值 200 并发不够，请求堆积超时
  时间线：
    14:00 流量突增
    14:05 错误率告警
    14:10 确认 DB 连接池满
    14:15 紧急扩连接池到 200
    14:20 恢复
  改进：
    1. 连接池配置按峰值容量算（一周内）
    2. 加连接池饱和度告警（连接池 > 80% 告警）
    3. 加自动扩容（HPA based on 连接池）
  SLO 视角：本次故障消耗 30% 预算，本月预算紧张，暂停新功能发布
```

## 六、底层本质：为什么是 SLO + 错误预算这套体系

回到第一性：**稳定性是工程问题不是态度问题，必须量化治理**。

**为什么不用"100% 可用"做目标**：100% 在物理上不可能（机器会故障、网络会抖动、软件有 bug），追求 100% 的成本指数级增长（从 99.9% 到 99.99% 成本翻 10 倍，到 99.999% 再翻 10 倍）。合理的目标是"用户能接受的最低水平 + 一点缓冲"，这就是 SLO。

**为什么 SLI 必须是用户视角**：用户感受的是"我下单成功了吗、快不快"，不是"CPU 多少、内存多少"。机器视角指标（CPU 80%）与用户感受不强相关——CPU 80% 可能用户毫无感知，CPU 30% 但 DB 慢则用户感知强烈。SLO 必须翻译成"用户看到什么"，否则治理方向会偏（优化 CPU 但用户依然慢）。

**错误预算的本质——把"稳定性 vs 速度"量化为共同货币**：传统组织里业务方要快（多发布新功能）、运维方要稳（少变更），矛盾靠吵架。错误预算把矛盾翻译成同一货币——"本月还能消耗多少不稳定额度"。预算充足（系统稳定）→ 业务可以激进发布；预算耗尽（系统不稳）→ 必须冻结发布优先修。这套机制让博弈可量化、可仲裁，避免"谁声音大谁赢"。

**为什么多窗口告警**：单一窗口（如只看 1 小时）会误报（瞬时抖动触发告警）或漏报（慢性恶化不触发）。多窗口组合（1h+5m，6h+3h）兼顾——短窗口抓紧急故障，长窗口抓慢性恶化。Google SRE 的多窗口告警是工业实践的事实标准。

## 七、AI 架构师加问：5 个 AI 相关问题

1. **AI 推理服务的 SLO 怎么定？**
   可用性 SLI：推理成功率（非 5xx）、P99 推理延迟。延迟 SLO 比传统服务严（用户等 AI 响应耐心低，P99 < 1s）。新增 SLI：token 吞吐（tokens/sec）、幻觉率（人工标注样本的幻觉比例）、成本（$/request）。

2. **让 AI 预测错误预算消耗，AI 接管哪段？**
   AI 用历史消耗数据预测本月预算是否够用（趋势外推 + 流量预测）。AI 出预警（"按当前消耗速度，10 天后预算耗尽"），人工决定是否冻结发布。AI 不直接执行冻结（影响业务，必须人工确认）。

3. **AI Agent 服务的 SLO 怎么治理？**
   Agent 链路长（用户 → AI → 工具 → 结果），SLI 用"用户感知成功率"（任务完成率）而非"接口成功率"。延迟 SLO 用"端到端响应时间"（用户问 → AI 答）。新增：工具调用失败率、AI 拒答率、token 超预算率。

4. **AI 故障复盘如何自动化归因？**
   AI 解析告警 + trace + 日志 + 变更记录，分类根因（代码 bug、配置错误、容量不足、依赖故障）。AI 出归因报告（根因假设 + 证据链 + 改进建议），人工 review 后归档。监控 AI 归因准确率（与人工复盘对比）。

5. **AI 服务的错误预算怎么和传统服务区分？**
   AI 服务失败更复杂（接口成功但答案错——幻觉）。错误预算分两层：技术层（接口 5xx、超时）+ 业务层（幻觉、答非所问）。业务层需要人工标注或用户反馈（点踩），成本高但必要。

## 八、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"SLI 用户视角、SLO 量化目标、错误预算共同货币、多窗口告警、预算门禁发布"**。

- **SLI**：用户视角（成功率、P99），不用机器视角（CPU）
- **SLO**：量化目标（99.9% 成功率、P99 < 500ms），28 天滚动窗口
- **错误预算**：1 - SLO，稳定性 vs 速度的共同货币
- **多窗口告警**：1h+5m 抓紧急、6h+3h 抓慢性
- **预算门禁**：预算耗尽冻结发布、预算充足可激进

### 拟人化理解

把 SLO 想成 **JD 物流时效承诺**。SLI 是"包裹送达时间"（用户视角，不是仓促繁忙度）。SLO 是"次日达 99.9%"。错误预算是"允许千分之一迟到"——这个月迟到多了（预算耗尽），停止接新客户优先优化物流；迟到少（预算充足），可以接更多客户或试新业务。SLA 是合同承诺（违反赔钱），比 SLO 宽松留缓冲。这套机制把"快 vs 稳"量化为同一货币，避免业务和运营吵架。

### 面试现场 60 秒回答

> SLO 是用户视角的可用性承诺（成功率 99.95%、P99 < 500ms），错误预算是允许失败的额度（1 - SLO = 0.05%）。SLI 必须用用户视角（成功率、延迟）而非机器视角（CPU）。错误预算是稳定性 vs 速度的共同货币——消耗快（系统不稳）冻结发布优先治理、消耗慢（系统稳）可激进发布。多窗口告警（1h+5m 抓紧急、6h+3h 抓慢性）避免误报漏报。SLO 制定基于历史数据下限 + 用户期望 + 成本权衡，核心链路严（支付 99.99%）、非核心松（评论 99%）。季度迭代——太松收紧、太严治理。

## 九、苏格拉底式面试追问

每一问先回答"为什么"，再"怎么做"，最后"如何证明"。

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 为什么不直接定 100% 可用，要定 99.9%？ | 100% 物理不可达（机器故障、网络抖动），追求成本指数增长（99.9→99.99 成本 10x）。SLO 是"用户能接受 + 缓冲"，不是无限高。证明：99.999% 可用年宕机 5 分钟，任何一次部署就超 |
| 证据追问 | 怎么证明 SLO 定的合理？ | 历史数据看实际表现下限、用户投诉率（SLO 违规时投诉应上升）、成本权衡（提高 0.1% 多花多少）。SLO 应让团队"偶尔紧张但不崩溃" |
| 边界追问 | SLO 能保证用户满意吗？ | 不完全能。SLO 是技术可用性（请求成功），用户满意还要看业务正确性（订单对不对）、体验（搜索结果好不好）。SLO 是底线，业务正确性要单独治理 |
| 反例追问 | 什么场景不上 SLO？ | 内部工具（无外部用户）、试验性服务（频繁变更不值得定 SLO）、低 QPS 服务（数据样本太少统计不显著）。SLO 适合稳定的核心服务 |
| 风险追问 | 错误预算耗尽冻结发布的风险？ | 业务方抵触（"我必须发新功能"）、长期冻结技术债更多（旧代码不更新）。治法：SLO 制定时预留发布消耗额度、冻结前预警、紧急 bug 走绿色通道、长期治理要规划 |
| 验证追问 | 怎么证明 SLO 机制真的有效？ | 看故障频率（应下降）、MTTR（平均恢复时间应缩短）、用户投诉率（应下降）、发布回滚率（应下降）、错误预算违规次数（应可控） |
| 沉淀追问 | 团队 SLO 治理沉淀什么？ | SLO 制定 SOP（识别旅程→定 SLI→定目标）、错误预算消耗看板、发布门禁自动化、故障复盘模板、季度 SLO 迭代会议、核心服务 SLO 清单 |

### 现场对话示例

**面试官**：SLI 为什么不能用 CPU？

**候选人**：因为 CPU 是机器视角，与用户感受不强相关。CPU 80% 时用户可能毫无感知（请求正常处理），CPU 30% 但 DB 慢时用户感知强烈（请求超时）。SLO 是"对用户的承诺"，必须用用户视角——"请求成功了吗、快不快"。CPU、GC、内存是诊断指标（出问题时查根因用），不是 SLO。如果用 CPU 做 SLO，团队会优化 CPU 但用户依然慢，治理方向就错了。

**面试官**：错误预算怎么驱动发布？

**候选人**：错误预算是"稳定性 vs 速度"的共同货币。剩余 > 50% 时正常发布新功能；20%-50% 时只允许修 bug、加强灰度；< 20% 或耗尽时冻结非紧急发布、优先投入稳定性治理。这套机制让业务和运维有共同语言——业务想发新功能时看错误预算，预算够就发、不够就先稳。CI/CD 集成自动检查——发布前查 Prometheus 的预算剩余，不够就拒绝发布（除非紧急修复）。这套机制把"能不能发布"从主观决策变成数据驱动。

**面试官**：故障消耗了 30% 预算怎么办？

**候选人**：先归因——是合理消耗（计划内发布）还是异常消耗（故障、bug）。故障属异常消耗，必须复盘定根因和改进措施。预算层面，本月剩余预算从 100% 降到 70%，可能触发发布门禁（如剩余 < 50% 需审批）。如果剩余 < 20%，冻结非紧急发布，团队优先投入稳定性治理（修 bug、补压测、清债务）。下月预算重置（28 天滚动），但如果连续多月消耗异常，说明系统有结构性问题，需要架构层面改造。

## 常见考点

1. **SLI/SLO/SLA/错误预算关系？**——SLI 是指标（用户视角）、SLO 是目标（99.9%）、错误预算 = 1-SLO（允许失败额度）、SLA 是合同（比 SLO 宽松留缓冲）。
2. **SLI 怎么选？**——用户视角（成功率、P99 延迟），不用机器视角（CPU、GC）。机器视角是诊断指标。
3. **错误预算怎么用？**——剩余多可激进发布、剩余少冻结优先治理。CI/CD 集成发布门禁，预算耗尽拒绝非紧急发布。
4. **多窗口告警是什么？**——短窗口（1h+5m）抓紧急故障、长窗口（6h+3h）抓慢性恶化，避免误报漏报。Google SRE 推荐燃烧率 14.4x/6x。
5. **SLO 定多高？**——基于历史数据下限 + 用户期望 + 成本权衡。核心链路严（支付 99.99%）、非核心松（评论 99%），不要全部 99.99%（成本爆炸）。


## 结构化回答


**30 秒电梯演讲：** 像 JD 物流时效承诺：SLO 是"次日达 99.9%"，错误预算是"允许 0.1% 即千分之一的包裹迟到"。这个月迟到包裹多了，就停止接新客户优先优化物流；迟到的少，可以接更多客户或试新业务。错误预算把"稳定性 vs 速度"的矛盾量...

**展开框架：**
1. **SLI（指标）→** — SLO（目标）→ SLA（合同）→ 错误预算（额度）
2. **SLO 典型** — 可用性 99.9%（年宕机 8.76h）、成功率 99.95%、P99 < 500ms
3. **错误预算 = 1** — SLO，是"可消耗的不稳定额度"

**收尾：** SLI 怎么选？

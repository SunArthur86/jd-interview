# 【Java 后端架构师】架构治理委员会如何避免形式主义

> 适用场景：JD 核心技术。一个 30 人团队引入架构委员会，结果每两周开 2 小时评审会，所有 PR 都过、决议没人跟踪、规范只挂在 wiki 上。半年后核心链路还是出了大故障——评审只是"盖章机"。架构师必须能用分级评审 + RFC/ADR + 上线门禁，让治理真正落地。

## 一、概念层：形式主义的五种典型与对症

| 形式主义表现 | 根因 | 对症机制 |
|------------|------|---------|
| 评审走过场，全通过 | 影响面没分级，小改动也走重评审，委员疲劳 | 分级评审：核心重评审，小改动 PR review |
| 决议不记录，无法追溯 | 评审没产物（没 ADR），只是聊天 | 强制 RFC + ADR，入 Git 版本化 |
| 规范只挂墙上 | 没和上线门禁挂钩 | CI 门禁：核心改动无 ADR 阻断合并 |
| RFC 没人写 | 没模板、没标准、写了没收益 | RFC 模板 + 评审 checklist + owner 制 |
| 委员会变辩论会 | 会前没预沟通，会上才暴露分歧 | 预沟通前置，会上聚焦分歧 |

## 二、机制层：分级评审 + RFC + ADR + 门禁

### 2.1 变更分级标准（决定评审级别）

```
┌────────────────────────────────────────────────────────────┐
│  Level 1：重评审（委员会 + RFC + ADR + 验证计划）           │
│  - 核心链路（下单/支付/库存主流程）改动                     │
│  - 数据迁移（表结构变更、数据格式变化）                     │
│  - 外部接口（开放 API、第三方集成）                         │
│  - 安全相关（认证、权限、加解密）                           │
│  - 中间件升级（Kafka/MySQL/Redis 大版本）                   │
├────────────────────────────────────────────────────────────┤
│  Level 2：轻评审（技术负责人 + PR review + 简要 ADR）       │
│  - 非核心服务改动                                          │
│  - 内部接口扩展（不破坏兼容）                              │
│  - 性能优化（加缓存、改索引）                              │
├────────────────────────────────────────────────────────────┤
│  Level 3：自评审（PR review 即可）                         │
│  - Bug 修复、UI 调整、文案改动                             │
│  - 单元测试补充                                            │
└────────────────────────────────────────────────────────────┘
```

**分级的核心原则**：治理的代价和收益匹配。全部重评审 = 评审疲劳（敷衍）；全部 PR review = 核心风险没把关。80% 的改动走轻流程，20% 的核心改动走重评审，把委员会的精力集中在真正高影响的决策上。

### 2.2 RFC 模板（讨论稿）

```markdown
# RFC-2026-018: 订单服务引入 Caffeine 本地缓存

## 作者 / 状态
- 作者：@张三
- 状态：Draft / Under Review / Accepted / Rejected
- 评审人：架构委员会、SRE、订单 owner

## 1. 背景与问题（为什么做）
- 订单查询接口 QPS 5 万，Redis 集群 P99 偶发抖动到 50ms（HGETALL 大 key）
- 商品详情页读放大严重，相同 SKU 1 秒内被查 100 次
- 现状指标：P99 80ms、Redis 命中率 92%、错误率 0.1%

## 2. 目标与非目标
- 目标：订单查询 P99 降到 30ms，Redis 调用量减少 70%
- 非目标：不解决写链路、不做分库分表（这是另一个 RFC）

## 3. 方案概述
引入 Caffeine 本地缓存（进程内），承接热点 SKU 的读请求。
- 缓存策略：maximumSize=10w，expireAfterWrite=5s
- 一致性：本地缓存 + Redis 二级缓存，写操作 MQ 广播失效
- 失败兜底：本地缓存穿透时回源 Redis，Redis 失败回源 DB

## 4. 替代方案（必写，证明思考过取舍）
- 方案 B：只加 Redis 集群容量 → 治标不治本，热点还是热点
- 方案 C：CDN 边缘缓存 → 商品详情适合，订单查询不适合（个性化）
- 方案 D：预计算 + 物化视图 → 复杂度高，收益不明显

## 5. 风险与兜底
| 风险 | 概率 | 兜底 |
|------|------|------|
| 本地缓存数据陈旧（5s 窗口） | 高 | 业务可接受（订单状态 5s 延迟）+ 关键状态走 DB |
| JVM 堆内存压力 | 中 | maximumSize 上限 + 监控 heap usage |
| 多实例本地缓存不一致 | 中 | MQ 失效广播 + 关键写强一致走 Redis |

## 6. 验证计划
- 灰度：10% → 50% → 100%，监控 P99/heap/命中率
- 回滚条件：P99 不降反升 / OOM 风险 / 业务成功率 < 99.9%

## 7. 时间线与 owner
- 评审通过后 2 周开发，1 周压测，1 周灰度
```

### 2.3 ADR 模板（决策记录）

```markdown
# ADR-2026-018: 订单服务引入 Caffeine 本地缓存

## Status
Accepted (2026-07-13)

## Context（背景与约束）
订单查询 P99 80ms，Redis 集群热点 SKU 抖动，影响商品详情页加载体验。
约束：不能引入新中间件（团队运维能力）、不能改写链路、5s 数据延迟业务可接受。

## Decision（决策）
采用 Caffeine 本地缓存 + Redis 二级缓存，本地缓存 maximumSize=10w、TTL=5s。

### 关键取舍
| 决策点 | 选择 | 换来什么 | 付出什么 |
|--------|------|---------|---------|
| 本地缓存 vs Redis 扩容 | 本地缓存 | 延迟降 50ms、Redis 压力降 70% | 多实例不一致、堆内存占用 |
| TTL=5s vs 1s | 5s | 命中率高（80%+）、Redis QPS 低 | 数据延迟 5s（业务可接受）|
| 失效策略 MQ 广播 vs 定时刷新 | MQ 广播 | 实时性强（<1s） | MQ 依赖、消息丢失风险 |

## Consequences（代价与收益）
- 正面：P99 预期降到 30ms，Redis QPS 减少 70%
- 负面：堆内存占用增加（监控 heap）、多实例窗口期不一致（MQ 兜底）
- 风险：见 RFC-2026-018 风险矩阵
```

### 2.4 评审会 checklist（评审委员的检查清单）

```
□ Context 是否清晰（业务问题、约束、量化指标）
□ 是否考虑过替代方案（≥2 个），并说明否决理由
□ 取舍是否讲清（每个关键决策"换来什么、付出什么"）
□ Top 5 风险是否识别 + 兜底措施
□ 验证计划是否含：成功指标、灰度策略、回滚条件
□ 影响的上下游是否知情（依赖服务、调用方）
□ 数据迁移方案是否有对账（如涉及）
□ 安全合规是否过审（涉及用户数据/资金时）
```

## 三、实战层：上线门禁与决议跟踪

### 3.1 评审门禁（让治理有强制力）

```yaml
# CI 门禁规则（伪代码）
rules:
  - name: core-service-needs-adr
    description: 核心服务改动必须有 ADR
    when:
      files_changed: ["order-core/**", "payment/**", "inventory/**"]
    check:
      adr_exists: true   # 检查 docs/adr/ 下是否有对应 ADR 文件
      adr_status: Accepted
    on_fail: block_merge

  - name: schema-change-needs-rfc
    description: 表结构变更必须有 RFC 评审通过
    when:
      files_changed: ["**/migrations/**", "**/*.sql"]
    check:
      rfc_status: Accepted
    on_fail: block_merge
```

**门禁的价值**：让"治理建议"变成"治理强制"。委员会不只是评审盖章，而是核心改动的上线前置条件——没有 ADR，CI 直接阻断合并。这是避免形式主义的最强约束。

### 3.2 决议跟踪（让评审有闭环）

```markdown
# 评审决议跟踪表

| RFC/ADR | 决议 | Owner | Deadline | 状态 | 跟踪记录 |
|---------|------|-------|----------|------|---------|
| ADR-018 | 接受 Caffeine 方案 | @张三 | 2026-08-10 上线 | 进行中 | 7-25 灰度 10%，P99 32ms 达标 |
| ADR-017 | 有条件接受：需补对账 | @李四 | 2026-08-05 补对账 | 进行中 | 7-20 对账方案评审中 |
| ADR-016 | 否决：拆分粒度过细 | @王五 | 重新评估 | 待办 | 7-30 重新提交 RFC |

闭环规则：
- 有条件接受的事项，限期闭环，否则 ADR 状态回退为 Rejected
- 决议跟踪表每周 review，超期升级 owner 主管
- 季度复盘否决率（健康值 10-20%）
```

### 3.3 季度治理复盘（委员会自身的改进）

```markdown
# 治理季度复盘（委员会自检）

## 量化指标
- 评审次数：本季度 45 次（核心 12 次，轻评审 33 次）
- 否决率：18%（健康范围 10-20%）
- ADR 数量：18 篇（vs 上季度 10 篇，决策记录更完整）
- 决议闭环率：85%（达标 >80%）
- 平均评审周期：3.5 天（目标 <5 天）

## 形式主义检查
- [ ] 是否有 PR 跳过评审直接上线？（查 CI 门禁日志）
- [ ] 否决的 RFC 是否有重新提交？（否决后无后续=治理失效）
- [ ] ADR 是否真在写取舍和风险？（抽样 5 篇人工审）
- [ ] 委员是否提前 review RFC？（看 RFC 评论时间分布）

## 改进项
1. RFC 模板增加"替代方案"必填项（避免只给一个方案）
2. 评审委员提前 3 天 review RFC，禁止会上才看
3. 上线门禁覆盖所有核心服务（当前覆盖 80%，补到 100%）
```

## 四、底层本质：为什么治理会沦为形式主义

**形式主义的根源**：治理动作（评审、规范、checklist）与真实风险脱钩。当"评审通过率 100%"成为 KPI，委员会就变成盖章机；当规范只挂 wiki 不挂门禁，规范就只是文档。

**为什么分级评审是反形式主义的核心**：人的注意力是稀缺资源。全部重评审 = 委员对每个 PR 都敷衍（看一眼就通过）= 形式主义。把 80% 改动走轻流程，委员会精力集中在 20% 核心改动上，每个评审都认真——这是"治理密度"的优化。

**为什么 ADR 是治理的灵魂**：决议不记录，下次同样的争议会反复出现（"为什么选 Caffeine 不选 Guava"），团队记忆依赖老人。ADR 让决策显性化、可追溯、可复盘。新人入职看 ADR 就能理解"为什么现在是这样设计"，而不是听口口相传。

**为什么门禁是强制力来源**：没有强制力的治理只是建议。建议会被业务压力冲垮——"这个紧急需求来不及写 ADR，先上线吧"。CI 门禁把"必须 ADR"变成代码合并的前置条件，业务压力压不动代码合并规则。这是治理落地的最后一道闸门。

**为什么预沟通比评审会重要**：评审会上当众辩论容易变面子之争（谁也不愿当场认输）。预沟通把分歧点 1v1 收敛，评审会聚焦"对齐共识 + 记录决议"。资深架构师 80% 的说服工作在会前完成，会上是"宣布共识"不是"辩论输赢"。

## 五、AI 架构师加问：5 个

1. **AI 怎么辅助架构治理？**
   AI 负责检查 RFC 完整性（替代方案写了没、风险矩阵全不全）、识别重复决策（"这个方案和 ADR-018 类似，建议参考"）、从历史 ADR 检索相似案例。决策权在委员会，AI 是辅助。

2. **AI 怎么判断"这个改动属于 Level 1 还是 Level 2"？**
   基于变更影响面：改动文件路径（核心服务/非核心）、改动行数、是否涉及数据迁移、是否改外部接口。用历史标注数据训练分类器，置信度低的 case 转人工判断。AI 只做初筛，委员会复核。

3. **让 AI 自动写 ADR 行不行？**
   不行。ADR 的价值是"取舍思考过程"，AI 只能写"决策结论"不能写"为什么这么取舍"。AI 可以辅助填模板（从 RFC 提取 Context、Decision 字段），但 Consequences（代价收益）必须人写——这是思考的产物。

4. **评审门禁用 AI 自动判断"这个 PR 是否破坏架构约束"，怎么做？**
   把架构约束写成可执行规则（如"order-core 不能直接依赖 payment-service"，用 ArchUnit 检查）+ AI 辅助判断语义级违规（如"这个改动引入了循环依赖"）。规则检查是确定性的，AI 补充语义判断，结果都给人工复核。

5. **怎么防止 AI 评审引入新形式主义？**
   AI 评审容易变成"AI 说 OK 就 OK"的新盖章机。对策：AI 只做完整性检查和风险提示，不做决策；所有 AI 输出有人工 fallback；定期抽样审计 AI 评审准确率（与人工评审对比）。AI 是辅助不是替代。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"分级、RFC/ADR、门禁、预沟通、否决率"** 五个词。

- **分级**：核心重评审、小改动 PR review，治理密度匹配影响面
- **RFC/ADR**：RFC 讨方案、ADR 记决策，是治理两件套
- **门禁**：核心改动无 ADR 阻断 CI，让治理有强制力
- **预沟通**：80% 说服在会前，会上聚焦共识
- **否决率**：健康值 10-20%，过低=走过场，过高=预沟通不够

### 拟人化理解

把架构治理想成 **交警执法**。只在路口查超速是形式主义（罚款完事，事故还在发生）；真正有效的是"路口查 + 测速探头 + 驾驶培训 + 道路设计"组合。架构委员会也一样：评审只是入口（路口查），决议跟踪（测速探头）、规范沉淀（驾驶培训）、上线门禁（道路设计）、培训宣贯（安全教育）才是完整闭环。少了哪一环都会变成形式主义。

### 面试现场 60 秒回答

> 架构治理避免形式主义的核心是"让治理动作与真实风险挂钩"。第一步分级评审——80% 改动走 PR review 轻流程，20% 核心改动（核心链路/数据迁移/外部接口）走重评审，治理密度匹配影响面。第二步用 RFC + ADR 把决策显性化——RFC 讨论方案、ADR 记决策，都入 Git 版本化。第三步最关键——评审决议与上线门禁挂钩，核心改动无 ADR 阻断 CI 合并，让治理有强制力。流程上 RFC 会前 1 周发出，委员异步 review，会上只讨论分歧（80% 说服在会前预沟通）。健康指标看否决率（10-20% 为健康），过低是走过场、过高是预沟通不够。

### 反问面试官

> 贵司架构治理是重流程（必须评审才能上线）还是重共识（评审是建议性）？如果是重共识，我会强调门禁挂钩；如果是重流程，我会强调分级评审避免疲劳。还有 ADR 是怎么沉淀的，Git 还是 wiki？

### 现场对话示例

**面试官**：你说分级评审，那核心改动的标准怎么定？

**候选人**：按变更影响面定四类：(1) 核心链路改动（下单/支付/库存主流程）——影响交易成功率；(2) 数据迁移（表结构、数据格式）——影响数据正确性；(3) 外部接口（开放 API、第三方集成）——影响兼容性；(4) 安全相关（认证、权限）——影响合规。这四类走 Level 1 重评审。其他走 Level 2 轻评审（技术负责人 + PR review）。标准写在 wiki 上，CI 门禁按文件路径自动判断（如改动 `order-core/**` 自动触发 Level 1 要求），不依赖人工判断。

**面试官**：RFC 写了没人看，怎么办？

**候选人**：核心是把 review 做成"评审前置条件"。RFC 发出后给委员 3 天异步 review 时间，每个委员必须留至少 3 条评论（可以是问题、建议、赞同），否则评审会不允许开。会上不重新读 RFC，只讨论评论里的分歧点。这样委员必须提前看，不能会上才翻。再辅以 RFC 模板标准化——替代方案、风险矩阵、验证计划必填，写不全退回重写，逼着作者认真思考。

**面试官**：评审通过了但上线还是出事故，怎么办？

**候选人**：先止损（回滚/降级），再复盘。复盘要回溯评审：风险是否识别到了？兜底是否生效？验证计划是否漏了场景？如果风险在评审中识别但兜底没起作用，更新 ADR 加固兜底；如果风险根本没识别到（黑天鹅），补充 checklist 这一项。评审通过不代表 owner 免责——owner 仍对线上负责。但委员会要反思：是不是评审太松了？否决率是不是太低（走过场信号）？治理体系本身要迭代。

## 常见考点

1. **RFC 和 ADR 区别？**——RFC（Request for Comments）是讨论稿，提方案征询意见；ADR（Architecture Decision Record）是决策记录，是 RFC 讨论后的结论。RFC → 评审 → ADR，两者构成治理两件套。
2. **委员会怎么避免走过场？**——分级评审（治理密度匹配影响面）+ RFC/ADR 显性化决策 + 上线门禁挂钩（核心改动无 ADR 阻断 CI）+ 决议跟踪闭环 + 季度复盘否决率（10-20% 健康）。少了门禁，治理只是建议。
3. **变更怎么分级？**——按影响面：Level 1 核心链路/数据迁移/外部接口/安全走重评审（委员会+RFC+ADR+验证计划）；Level 2 非核心服务/内部接口扩展走轻评审（技术负责人+PR review）；Level 3 Bug 修复/UI 改动走 PR review。CI 门禁按文件路径自动判断级别。
4. **评审会多长合适？**——核心方案 1-2 小时（含 Q&A），轻评审 30 分钟。超时说明预沟通不够（会上才暴露分歧）。评审会效率 80% 取决于会前准备（RFC 质量、委员预 review）。
5. **否决率多少健康？**——10-20%。低于 5% 说明走过场（全通过）；高于 30% 说明预沟通不够（会上才暴露分歧）或 RFC 质量太差。季度复盘否决率是委员会自检的关键指标。

## 结构化回答

**30 秒电梯演讲：** 架构治理委员会（ARB, Architecture Review Board）容易沦为形式主义——评审走过场、决议不跟踪、RFC 没人写、规范只挂墙上。让治理落地的核心是三件事：(1) 分级评审（核心链路重评审，小改动走轻流程），(2) 用 RFC + ADR 把决策显性化、可追溯，(3) 评审结果与上线门禁挂钩（核心改动无 ADR 不允许上线）。形式主义治不了架构，机制才能

**展开框架：**
1. **分级评审** — 核心链路/数据迁移/外部接口必评审；小改动走 PR review 轻流程
2. **RFC + ADR 是治理的两件套** — RFC 讨方案，ADR 记决策
3. **评审决议与上线门禁挂钩** — 评审决议与上线门禁挂钩（无 ADR 不上线），否则流于形式

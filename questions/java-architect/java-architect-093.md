# 【Java 后端架构师】技术债治理、重构节奏与业务连续性

> 适用场景：JD 核心技术。一个跑了 5 年的订单系统，代码 50 万行，核心逻辑里塞满了各种业务临时方案的 if-else，每次改需求要牵动 10 个文件，发版一次全组加班到凌晨，线上每月 2-3 次故障。新人 3 个月都看不懂不敢改。这是典型的技术债失控。架构师要做的是在不中断业务的前提下，系统性地还债，让系统重新可持续。

## 一、概念层：技术债的分类与量化

### 技术债分类（Martin Fowler 的三分法）

| 类型 | 成因 | 例子 | 治理策略 |
|------|------|------|---------|
| **故意债** | 有意识的权衡（赶进度） | "先这么写上线，之后优化" | 记录 + 计划还债 |
| **意外债** | 认知不足（当时不知道更好的做法） | 用同步调用本该异步的链路 | 学习后识别，逐步替换 |
| **腐烂债** | 长期不维护恶化 | 系统演进后原设计不再适用 | 评估是否还（可能已不用） |

### 技术债利息量化（决定优先级）

```markdown
# 技术债看板示例

| 债务 | 类型 | 利息（月） | 还债成本 | 优先级 | 状态 |
|------|------|-----------|---------|--------|------|
| 订单核心 if-else 膨胀（3000 行方法） | 腐烂 | 故障 2 次/月，改需求 5 天/次 | 2 人×3 周 | P0 | 进行中 |
| 同步调用库存（本该异步） | 意外 | 大促超时 3 次，P99 抖动 | 1 人×1 周 | P1 | 待办 |
| 日志无结构化 | 故意 | 排查 1 小时/次×10 次/月 | 1 人×3 天 | P2 | 待办 |
| 老版 PHP 营销页（流量<1%） | 腐烂 | 几乎无故障 | 重写成本高 | P3 | 战略性不还 |
```

**利息计算公式**：`利息 = 故障频率 × 单次影响（用户/金额/工时）+ 迭代拖慢成本（每次多花的时间 × 频率）`

**优先级公式**：`优先级 = 利息 ÷ 还债成本`（ROI 越高越优先）

## 二、机制层：重构节奏与业务连续性

### 2.1 持续还债节奏（每迭代 20% 带宽）

```
每个迭代（2 周）：
├── 业务需求（80% 带宽）
│   └── 正常功能开发
└── 技术债（20% 带宽，固定不被挤占）
    └── 从看板取 P0/P1 债务，小步还债
        例：本周把订单核心 3000 行方法里的"促销计算"抽成独立类
        例：下周把同步调库存改成异步 MQ
```

**关键**：20% 是硬约束，不和业务需求抢。否则业务一忙债务永远还不掉。架构师要和业务达成共识："持续投入 20% 换来长期迭代速度"，用 ROI 说服。

### 2.2 Strangler Fig 模式（绞杀者，渐进替代）

```java
// 新老系统并存，逐步迁移（以订单系统重构为例）

@RestController
@RequestMapping("/order")
public class OrderController {

    private final OldOrderService oldService;      // 旧系统
    private final NewOrderService newService;      // 新系统
    private final FeatureFlag flag;                // 灰度开关

    @PostMapping("/create")
    public Order create(OrderDTO dto) {
        // 灰度路由：按租户/用户切流到新系统
        if (flag.isEnabled("new-order", dto.getUserId(), 0.05)) {
            return newService.create(dto);          // 5% 流量走新系统
        }
        return oldService.create(dto);              // 95% 仍走旧系统
    }
}

// 新老系统双写（数据一致性保障）
@Service
public class DualWriteOrderService {

    public void create(Order order) {
        oldRepo.save(order);                         // 先写旧库
        try {
            newRepo.save(order);                     // 再写新库
        } catch (Exception e) {
            log.error("新库写入失败，对账任务会兜底", e);
            // 不阻断业务，靠对账任务最终一致
        }
    }
}
```

### 2.3 对账与灰度切流（业务连续性核心）

```java
// 对账任务：比对新老系统数据差异
@Scheduled(cron = "0 */10 * * * *")  // 每 10 分钟
public void reconcile() {
    long oldCount = oldRepo.countByCreatedAfter(tenMinAgo);
    long newCount = newRepo.countByCreatedAfter(tenMinAgo);
    double diffRate = Math.abs(oldCount - newCount) / (double) oldCount;

    metrics.gauge("dualwrite.diff_rate", diffRate);

    if (diffRate > 0.001) {  // diff > 0.1% 告警
        alertService.warn("双写差异率 {} 超阈值", diffRate);
    }
    if (diffRate > 0.01) {   // diff > 1% 自动回滚切流
        flag.disable("new-order");
        alertService.page("双写差异过大，已自动切回旧系统");
    }
}

// 灰度切流节奏
// 第 1 周：5% 流量，对账 diff < 0.01%，P99 < 200ms
// 第 2 周：20% 流量，监控核心指标
// 第 3 周：50% 流量
// 第 4 周：100% 切流，旧系统保留只读 1 个月后下线
```

## 三、实战层：一次真实的技术债治理

### 案例：订单核心链路 if-else 膨胀治理

**债务描述**：`OrderService.create()` 方法 3000 行，塞满了促销/优惠券/会员价/满减的 if-else，每次加促销规则要改这个方法，牵动 10+ 测试，发版全组加班。

**利息**：每月故障 2 次（改错 if 分支）、每次加需求 5 天（含测试）、新人 3 个月不敢碰。

**还债方案**（strangler fig + 策略模式）：

```java
// 第 1 步：识别变化点，抽象策略接口（不删旧代码）
public interface PromotionStrategy {
    boolean supports(OrderContext ctx);
    BigDecimal calculate(OrderContext ctx);
}

// 第 2 步：逐个把 if-else 分支迁移成策略类（小步，每周迁 1-2 个）
@Component
public class CouponStrategy implements PromotionStrategy {
    public boolean supports(OrderContext ctx) { return ctx.hasCoupon(); }
    public BigDecimal calculate(OrderContext ctx) { /* 从老 if 分支抽出来 */ }
}

// 第 3 步：新促销走策略引擎，老促销仍走 if-else（双轨并行）
@Service
public class OrderService {
    private final PromotionEngine engine;        // 新策略引擎
    private final LegacyPromotion legacy;        // 老逻辑

    public BigDecimal calcPromotion(OrderContext ctx) {
        if (flag.isEnabled("new-promotion", ctx.getUserId())) {
            return engine.calculate(ctx);        // 新引擎
        }
        return legacy.calcOldWay(ctx);           // 老逻辑
    }
}

// 第 4 步：灰度验证新引擎结果与老逻辑一致（diff = 0），逐步切流
// 第 5 步：全量后删除老 if-else 分支
```

**节奏**：4 周完成，每周迁 2-3 个策略类，全程业务无感（灰度 + 对账），发版不影响业务。

**效果**：加促销规则从 5 天降到 0.5 天（加个策略类即可），故障率降 70%，新人 1 周上手。

## 四、底层本质：为什么全面重构几乎总是错的

**全面重构的诱惑**：代码烂透了，推倒重来多爽。但历史数据表明，全面重构的项目失败率极高（> 50%），原因不是技术能力不足，而是：

1. **业务连续性断裂**：重构期间（通常 3-12 个月）业务不能停，新旧系统并行维护成本翻倍，团队疲于奔命。
2. **需求漂移**：重构期间业务需求仍在变，新系统还没上线就已经落后于需求。
3. **隐藏规则丢失**：老系统里藏着大量"血泪规则"（当年踩坑加的 if），推倒重来会重新踩一遍坑。
4. **沉没成本陷阱**：重构到一半发现方向错了，但已投入太多难以回头。

**渐进式重构为什么更优**：

- **业务不中断**：strangler fig 模式，新老并存，灰度切流，全程业务无感。
- **风险可控**：每步小改可验证可回滚，错了及时调整，不会一条路走到黑。
- **持续产出**：每步都有可交付的价值（某块代码变好了），不像全面重构要等很久才见效。
- **保留血泪规则**：逐步迁移时，老代码的隐藏规则会被识别并保留，不重新踩坑。

**技术债的第一性**：债务是时间维度的复利。早还利息低（代码还简单），晚还利息高（腐烂恶化）。但不还的最坏结果不是"代码烂"，而是"系统无法支撑业务"——这时被迫全面重构，代价最大。所以治理的目标是"让利息可控"，不是"还清所有债"。

## 五、AI 架构师加问：5 个 AI 场景的技术债

1. **AI 系统特有的技术债有哪些？**
   Prompt 债（写死在代码里没版本化）、模型债（锁死单一模型/版本无降级）、数据债（训练/评测数据无版本无溯源）、eval 债（无评测集上线靠感觉）、护栏债（无 schema 校验/prompt injection 防护）。这些债的利息是"线上幻觉/泄露/成本失控"，治理见 087-091 题。

2. **AI 方案快速验证后留下的债怎么还？**
   POC 阶段为快故意欠债（写死 prompt、无降级、无评测）。上线前必须评估还债清单：prompt 配置化、加降级链、建 eval 集、加护栏。如果直接把 POC 推上线，这些债会变成线上事故。治理：POC 到生产有明确的"还债检查清单"，不通过不上线。

3. **大模型迭代快（半年一代），现有 AI 架构怎么避免快速腐烂？**
   模型抽象层（Spring AI ChatModel）屏蔽模型差异，换模型只改配置不改业务代码。prompt 和模型解耦（prompt 管理平台）。eval 集随模型升级跑回归（换模型后 task_accuracy 不退化才上线）。这样模型迭代不会让架构腐烂。

4. **AI 方案重构时怎么保证业务连续性？**
   和传统重构一样走 strangler fig：新老 AI 链路并存（如规则引擎 + LLM 双跑），对账 diff（LLM 输出 vs 规则输出一致性），灰度切流（5% → 100%）。特殊点：AI 输出非确定，对账不能简单 diff，要用语义相似度或 eval 指标对比。

5. **技术债看板对 AI 系统怎么适配？**
   增加 AI 专属债务类型：模型债（模型版本过时）、prompt 债（未配置化）、eval 债（无评测）、护栏债（无安全防护）、数据债（数据过期）。利息指标用 AI 专属的：hallucination_rate、model_latency_p95、token_cost_per_request、fallback_rate。这些指标恶化就是 AI 技术债的利息在涨。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"分类、排优先级、边跑边修、strangler fig"** 四个词。

- **分类**：故意债（记录）、意外债（学习）、腐烂债（评估是否还）
- **排优先级**：利息（故障率×影响面）÷ 还债成本，ROI 高的先还
- **边跑边修**：每迭代 20% 带宽还债，不停业务
- **strangler fig**：新老并存、双写对账、灰度切流、保留回滚

### 面试现场 60 秒回答

> 技术债治理的核心是"边跑边修"不停业务。第一步分类——故意债（有意识权衡，记录还债计划）、意外债（认知不足，学习后识别）、腐烂债（评估是否还）。第二步量化优先级——利息（故障频率×影响面）÷ 还债成本，高 ROI 先还。第三步定节奏——每迭代固定 20% 带宽还债，不被业务挤占，用 ROI 说服业务给时间。第四步保证业务连续性——用 strangler fig 模式（新老并存）+ 双写对账（数据一致性）+ 灰度切流（5%→100%）+ 保留回滚。最忌讳的是全面重构推倒重来——业务连续性断裂、隐藏规则丢失、失败率超 50%。治理的目标是"利息可控"不是"还清所有债"，低利息的债可以战略性不管。

### 反问面试官

> 贵司技术债治理是靠架构师个人推动，还是有机制保障（如每迭代固定还债带宽、技术债看板）？这决定我是"呼吁还债"还是"建机制还债"。

## 七、苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 为什么不直接招人全面重构，要小步还债？ | 全面重构失败率超 50%（业务连续性断裂、需求漂移、隐藏规则丢失）。小步还债风险可控、业务不中断、持续产出。证明：业界（如 Martin Fowler）和 JD 内部案例都验证渐进式优于大爆炸式 |
| 证据追问 | 你怎么证明这笔技术债值得还？ | 算利息：故障频率（每月 2 次）×单次影响（用户数/金额）+ 迭代拖慢（每次需求多 3 天）。对比还债成本（2 人 3 周）。ROI = 利息节省 ÷ 成本，> 2 倍才值得立项 |
| 边界追问 | 什么债不该还？ | 低利息的（流量<1% 的老功能）、还债成本高于利息的（重写不如维持）、即将下线的系统。战略性不还也是决策，记录在案避免反复评估 |
| 反例追问 | 什么情况必须全面重构？ | 系统已无法支撑核心业务（利息 > 重构成本）且渐进式试过失败。这是最后手段，要配充分灰度+回滚+业务停摆许可。但即使如此也用 strangler fig 而非"停业务重写" |
| 风险追问 | 还债过程中最大的风险？ | 双写不一致导致数据错乱（新旧系统数据 diff）。兜底：对账任务（每 10 分钟比对）+ diff 告警（0.1%）+ 自动回滚（1%）。第二风险是灰度期间新系统 bug，兜底是保留快速切回旧系统的开关 |
| 验证追问 | 怎么证明还债真的有效？ | 对比还债前后：故障率（应降）、迭代速度（需求交付周期应缩短）、新人上手时间（应缩短）。这些是业务能感知的收益，不是"代码变优雅了"这种自嗨指标 |
| 沉淀追问 | 团队技术债治理沉淀什么？ | 技术债看板模板（债务+利息+优先级+进度）、strangler fig 实施模板（双写+对账+灰度）、ADR 模板（故意欠债要记录还债计划）、每迭代 20% 还债带宽的制度保障 |

### 现场对话示例

**面试官**：你说每迭代 20% 还债，业务说需求太急不给时间怎么办？

**候选人**：把债翻译成业务语言。不说"代码烂要重构"，说"这个债导致每次发版多花 2 天（你们需求慢）、每月故障 2 次（用户投诉）。投入 20% 还债，2 个月后发版快 1 倍、故障降 70%"。用 ROI 说服——还债是投资不是消费。如果业务仍不接受，退一步：先还"利息最高且还债成本最低"的那一两个（快速见效建立信任），用结果证明还债有价值，再争取更多带宽。最差也要守住"新需求不欠新债"的底线（代码 review 把关），避免债务继续增长。

**面试官**：双写期间对账发现新老系统结果不一致，怎么办？

**候选人**：先定性是 bug 还是预期差异。如果是新系统的 bug（少数 case 算错），修复 bug 后继续灰度。如果是预期差异（新系统故意改了逻辑，如修正了老系统的某个错误），要评估这个差异是否业务可接受——可接受则更新对账基准（把新逻辑作为正确答案），不可接受则调整新系统对齐老逻辑（先保持一致再优化）。关键是 diff 要分类、要可解释，不能笼统说"不一致"。diff 率持续 > 0.1% 不切流，直到搞清楚并修复。

**面试官**：strangler fig 模式，新旧并存期间运维成本翻倍，扛不住怎么办？

**候选人**：控制并存周期。每块迁移严格按"双写 → 对账达标 → 切流 → 下线旧"的节奏，每块并存不超过 1 个月。避免所有模块同时并存（那就等于全面重构了）。按优先级一块块迁，每块完成就下线对应的旧代码，运维成本是阶段性上升（一次一块）而非全局翻倍。如果团队规模确实扛不住，就拉长节奏（每块迁 2 个月）而非同时多块并进。

## 常见考点

1. **什么是技术债？**——Ward Cunningham 提出的比喻，指为了快速交付做出的技术权衡，后续需要付出额外维护成本（利息）。分故意债（有意识权衡）、意外债（认知不足）、腐烂债（长期恶化）。不是"代码烂"的委婉说法，是真实的经济权衡。
2. **strangler fig 模式是什么？**——Martin Fowler 提出的渐进式重构模式。新系统逐步替代旧系统：新功能在新系统开发，旧功能逐步迁移，旧系统被"绞杀"直至下线。比大爆炸式重构安全，全程业务无感。核心是新老并存 + 灰度切流。
3. **双写和对账怎么做？**——双写：同一操作同时写新老两个系统（先写旧再写新，新失败不阻断）。对账：定时（每 10 分钟/每小时）比对新老数据差异，diff 超阈值告警/回滚。持续对账直到 diff 稳定为零才全量切流。
4. **怎么排技术债优先级？**——ROI = 利息 ÷ 还债成本。利息 = 故障频率 × 单次影响 + 迭代拖慢成本。高利息低成本的先还（ROI 高）。低利息高成本的战略性不还。看板化跟踪。
5. **全面重构为什么容易失败？**——业务连续性断裂（重构期业务不能停）、需求漂移（重构期需求在变）、隐藏规则丢失（老代码的血泪规则被推倒）、沉没成本陷阱。渐进式重构（strangler fig）规避这些问题，失败率低得多。

## 结构化回答

**30 秒电梯演讲：** 技术债不是代码烂，而是过去为了快交付做的权衡，现在要还利息。治理的本质是分类（哪些债该还）、排优先级（按利息大小和业务影响）、定节奏（和业务迭代并行而非停下业务重构）。重构的最大风险不是技术做不出来，而是过程中业务中断——所以业务连续性是第一约束

**展开框架：**
1. **技术债分类** — 故意债（有意识的权衡）、意外债（认知不足留下）、腐烂债（长期不维护恶化）
2. **优先级 = 利息** — 优先级 = 利息（故障率 × 影响面）÷ 还债成本，高利息低成本的先还
3. **重构节奏** — 与业务并行，每迭代拨 20% 带宽还债，不停业务

**收尾：** 以上是我的整体思路。您想继续深入聊——怎么说服业务给时间还债？

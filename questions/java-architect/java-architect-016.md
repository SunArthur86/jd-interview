  essence: 微服务拆分边界的本质是"找到业务的限界上下文（Bounded Context）"——一组高内聚的业务概念和规则形成一个边界，边界内的模型一致、边界间通过明确的接口契约交互。从领域模型推导而非从数据库表拆，能避免"按表拆服务"导致的分布式大对象、跨服务事务、循环依赖。
  analogy: 像城市规划划地块。商业区、住宅区、工业区各有边界（限界上下文），区内建筑风格统一（同一模型），区间靠道路连接（接口契约）。不能把一栋楼的厨房划到隔壁地块（订单的"地址"字段不该单独成服务），也不能把住宅区和工业区混在一起（订单和库存模型不该共用一张表）。
  first_principle: 为什么要按领域模型而不是按技术分层或数据库表拆？因为业务变更是按领域维度的（商品团队改商品、订单团队改订单），按领域拆分让变更影响域收敛在单个服务内，避免一个需求改动 N 个服务。按表拆会导致一个业务概念被切碎到多个服务，每次变更跨服务协调。
  key_points:
  - 限界上下文（Bounded Context）是拆分的核心单元：一个领域内术语一致、规则一致
  - 上下文映射（Context Map）：上游/下游、防腐层（ACL）、共享内核、客户/供应商
  - 聚合根（Aggregate Root）：一致性边界，外部只能引用聚合根 ID 不能引用内部对象
  - 领域事件（Domain Event）：解耦上下文，发布订阅异步通信
  - 拆分粒度权衡：太细则 RPC 调用泛滥、太粗则失去独立部署价值
first_principle:
  problem: 给定一个复杂业务（如电商交易），如何把它切成一组"高内聚低耦合"的服务，让业务变更影响域最小化、团队协作摩擦最小化？
  axioms:
  - 业务变更按领域维度发生（商品改商品、订单改订单），拆分要对齐变更轴
  - 跨服务调用有网络/一致性代价，不能太碎
  - 团队沟通成本与接口数量正相关，要控制接口边界
  rebuild: 用 DDD 的事件风暴（Event Storming）梳理领域事件（订单已创建、库存已扣减），反向推导出命令（创建订单）、聚合（订单聚合）、限界上下文（交易上下文、库存上下文）。每个限界上下文对应一个微服务（或一组服务）。上下文间用领域事件解耦（订单服务发"订单已创建"事件，库存服务订阅扣减库存）。聚合根定义一致性边界（订单内的订单项不能独立修改，必须通过订单聚合根）。拆分粒度参考"两个披萨团队"（6-10 人维护一个服务）。
follow_up:
  - 一个限界上下文一定对应一个微服务吗？——不一定。限界上下文是逻辑边界，微服务是物理边界。一个上下文可以拆成多个服务（如订单上下文拆成订单写服务和订单查询服务，CQRS），也可以多个上下文合并成一个服务（初期业务量小，订单+支付合一，后续再拆）。拆分要考虑团队规模、流量、变更频率
  - 怎么判断拆得太细？——症状：一个用户请求要调用 5+ 个服务（RPC 链路过长）、跨服务事务需求频繁（强一致需求高）、团队 70% 时间在协调接口、循环依赖（A 调 B，B 又调 A）。解法是合并高频协作的上下文
  - 聚合根为什么不能引用其他聚合的内部对象？——为了保证一致性边界。如果订单聚合直接引用商品聚合的内部对象（如商品规格），商品规格变更可能绕过订单的校验。正确做法是订单只存商品 ID（值对象），需要商品详情时通过商品服务查询
  - 领域事件怎么保证可靠投递？——用"本地消息表 + 消息中间件"模式：业务操作和消息记录在同一事务，事务提交后异步投递消息，消费方幂等处理。保证事件不丢（见 018 题分布式事务）
  - 老系统怎么从单体演进到微服务？——绞杀者模式（Strangler Fig）：先在单体边缘加新微服务（新建功能走新服务），逐步把单体功能迁到新服务，最后单体萎缩下线。不要一次性重构（风险大）
memory_points:
  - 限界上下文 = 微服务拆分的逻辑单元（术语一致 + 规则一致）
  - 聚合根 = 一致性边界（外部只引用 ID，内部强一致）
  - 上下文映射：U/D（上游下游）、ACL（防腐层）、Shared Kernel（共享内核）
  - 领域事件 = 上下文解耦（发布订阅，本地消息表保证可靠）
  - 拆分粒度：两个披萨团队（6-10 人一服务），按变更频率和流量独立需求拆

# 【Java 后端架构师】微服务拆分边界如何从领域模型推导

> 适用场景：JD 核心技术。电商交易链路涉及订单、商品、库存、支付、营销、物流等十几个领域，拆错了就是无尽的跨服务协调和分布式事务。架构师必须能用事件风暴推导限界上下文、定义聚合根边界、设计上下文映射——这是"微服务能不能落地"的前提。

## 一、概念层：DDD 战略设计三件套

**DDD 核心概念**（战略设计，面试必背）：

| 概念 | 定义 | 电商示例 |
|------|------|---------|
| **领域（Domain）** | 业务问题空间 | 电商交易 |
| **子域（Subdomain）** | 领域的细分 | 核心子域（订单）、支撑子域（商品）、通用子域（认证） |
| **限界上下文（Bounded Context）** | 模型一致的边界 | 订单上下文、库存上下文、支付上下文 |
| **聚合（Aggregate）** | 一致性边界 | 订单聚合（订单 + 订单项 + 收货地址） |
| **聚合根（Aggregate Root）** | 聚合的入口实体 | Order 是聚合根，OrderItem 只能通过 Order 访问 |
| **领域事件（Domain Event）** | 业务发生的事实 | OrderCreated、InventoryDeducted |
| **上下文映射（Context Map）** | 上下文间的关系 | 订单（U）→ 支付（D），防腐层翻译模型 |

**限界上下文的核心标志**（判断边界的依据）：

1. **术语一致**：上下文内同名概念含义一致。如"商品"在商品上下文指 SKU 完整信息，在订单上下文指商品快照（下单时锁定价格），在物流上下文指待发货物品——三个上下文不能用同一个"商品"模型。
2. **规则一致**：上下文内业务规则集中。如"订单金额 = 商品价格 × 数量 + 运费 - 优惠"这条规则属于订单上下文，不该散落到商品和营销服务。
3. **变更同频**：上下文内的功能变更频率相近，一起部署。如订单创建和订单查询变更频率差异大，可拆成两个服务（CQRS）。

## 二、机制层：事件风暴推导限界上下文

**事件风暴流程**（实操步骤，画图必考）：

```
步骤 1：贴领域事件（橙色便签）
  ── 梳理业务流程中发生的事实
  ── 示例：用户下单 → OrderCreated → InventoryLocked → PaymentSucceeded → OrderConfirmed → Shipped → Delivered

步骤 2：贴命令（蓝色便签）
  ── 每个事件由什么命令触发
  ── 示例：CreateOrder（触发 OrderCreated）、LockInventory、ConfirmPayment

步骤 3：贴参与者（黄色便签）
  ── 谁执行命令（用户、系统、定时任务）
  ── 示例：用户（CreateOrder）、支付网关（ConfirmPayment）

步骤 4：贴外部系统/读模型（粉色/绿色）
  ── 示例：支付网关（外部系统）、订单详情页（读模型）

步骤 5：识别聚合（虚线框圈）
  ── 命令作用的一致性单元
  ── 示例：CreateOrder 作用于 Order 聚合（含订单项）

步骤 6：横向切分限界上下文（粗线）
  ── 按业务语义聚类
  ── 示例：
       ┌──────────────┬──────────────┬──────────────┐
       │  订单上下文   │  库存上下文   │  支付上下文   │
       │  Order 聚合   │  Inventory聚合│  Payment聚合 │
       │  CreateOrder  │  LockInv     │  ConfirmPay  │
       │  OrderCreated │  InvLocked   │  PaySuccess  │
       └──────────────┴──────────────┴──────────────┘

步骤 7：定义上下文映射
  ── 上下文间如何交互
```

**上下文映射关系**（必考）：

| 关系 | 含义 | 示例 |
|------|------|------|
| **上游/下游（U/D）** | 上游提供 API，下游依赖 | 订单（U）→ 支付（D），但实际支付通知订单是反向上游 |
| **防腐层（ACL，Anti-Corruption Layer）** | 下游建翻译层隔离上游模型污染 | 订单上下文建 ACL 翻译遗留系统的字段 |
| **共享内核（Shared Kernel）** | 两个上下文共享一部分模型 | 订单和物流共享"地址"值对象 |
| **客户/供应商（Customer/Supplier）** | 协商式 U/D，下游有话语权 | 营销（客户）要求订单（供应商）暴露优惠券核销 API |
| **各行其道（Conformist）** | 下游被迫顺从上游模型 | 对接第三方支付，被迫用其模型 |
| **开放主机服务（OHS）+ 发布语言（PL）** | 上游提供标准开放 API + 事件 schema | 订单服务发布 OpenAPI + 事件 schema 给所有下游 |

## 三、实战层：电商交易的拆分案例

**典型电商交易拆分**（参考京东/阿里实践）：

```
┌─────────────────────────────────────────────────────────────────┐
│                       电商交易域                                  │
├──────────────┬──────────────┬──────────────┬──────────────┬─────┤
│  用户上下文   │  商品上下文   │  交易上下文   │  库存上下文   │ ... │
│  User        │  Product     │  Order       │  Inventory   │     │
│  UC1: 注册   │  PC1: 上架   │  OC1: 下单   │  IC1: 锁定   │     │
│  UC2: 登录   │  PC2: 改价   │  OC2: 取消   │  IC2: 释放   │     │
│  UC3: 改地址 │  PC3: 下架   │  OC3: 查询   │  IC3: 扣减   │     │
│              │              │              │              │     │
│  User 聚合   │  Product聚合 │  Order聚合   │  Inventory聚合│    │
│              │  SKU 聚合    │  (含订单项)   │  (按 SKU)    │     │
│              │              │              │              │     │
│  聚合根:User │ 聚合根:Product│聚合根:Order │ 聚合根:Inv   │     │
├──────────────┼──────────────┼──────────────┼──────────────┼─────┤
│  支付上下文   │  营销上下文   │  物流上下文   │  评价上下文   │     │
│  Payment     │  Promotion   │  Logistics   │  Review      │     │
│  Pay 聚合    │  Coupon聚合  │  Shipment聚合│  Review聚合  │     │
└──────────────┴──────────────┴──────────────┴──────────────┴─────┘

上下文间通过领域事件解耦：
  OrderCreated → Inventory（锁定）, Promotion（核销优惠券）
  PaymentSucceeded → Order（确认）, Logistics（发货）, Promotion（积分）
  Shipped → Order（更新状态）, Review（待评价）
```

**聚合根设计代码示例**：

```java
// Order 聚合根（一致性边界）
@Entity
@Table(name = "orders")
public class Order extends BaseAggregateRoot {
    @EmbeddedId
    private OrderId id;                    // 值对象 ID
    private UserId userId;                 // 引用其他聚合的 ID（不是对象）
    private OrderStatus status;
    private Money totalAmount;
    private List<OrderItem> items;         // 聚合内实体，外部不能直接访问
    private Address shippingAddress;       // 值对象

    // 业务规则封装在聚合根内（不暴露 setter）
    public void confirm() {
        if (status != OrderStatus.PENDING_PAYMENT) {
            throw new IllegalStateException("只有待支付订单能确认");
        }
        this.status = OrderStatus.CONFIRMED;
        // 发布领域事件（解耦下游）
        registerEvent(new OrderConfirmedEvent(this.id));
    }

    public void addItem(ProductId productId, int quantity, Money price) {
        // 业务规则校验在聚合根内
        if (this.status != OrderStatus.DRAFT) {
            throw new IllegalStateException("草稿状态才能加商品");
        }
        this.items.add(new OrderItem(productId, quantity, price));
        recalculateTotal();
    }

    // 外部只能引用 OrderId，不能引用 OrderItem
    // 这保证了订单项的修改必须经过 Order 聚合根
}
```

**上下文间通信（领域事件 + ACL）**：

```java
// 订单上下文发布事件
@Service
public class OrderApplicationService {
    @Transactional
    public void createOrder(CreateOrderCmd cmd) {
        Order order = orderFactory.create(cmd);
        orderRepository.save(order);
        // 通过本地消息表保证可靠投递（见 018 题）
        eventPublisher.publish(new OrderCreatedEvent(order.getId(), order.getItems()));
    }
}

// 库存上下文订阅事件（建 ACL 防腐层翻译）
@Component
public class InventoryEventListener {
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        // ACL：把订单上下文的 OrderItem 翻译成库存上下文的 SKU 锁定请求
        List<SkuLockRequest> requests = orderItemAclTranslator.translate(event.getItems());
        inventoryService.lock(requests);
    }
}
```

## 四、实战层：拆分粒度的判断与反模式

**拆分粒度判断矩阵**：

| 维度 | 倾向独立服务 | 倾向合并 |
|------|------------|---------|
| **变更频率** | 各自独立频繁变更 | 一起变更 |
| **团队归属** | 不同团队负责 | 同一团队 |
| **流量规模** | 流量差异大（独立扩容） | 流量相近 |
| **一致性需求** | 最终一致可接受 | 强一致需求 |
| **部署节奏** | 独立部署节奏 | 同步部署 |

**反模式（必答）**：

1. **按数据库表拆**（最常见错误）
   - 错误：order 表、order_item 表、user 表各一个服务
   - 问题：order_item 必须随 order 一起变更（强一致），拆成两个服务引入分布式事务
   - 正确：order 和 order_item 在同一聚合，同一服务

2. **分布式大对象**
   - 症状：用户请求要调用 5+ 服务才能组装一个页面
   - 解法：用 BFF（Backend For Frontend）聚合，或合并高频协作的服务

3. **循环依赖**
   - 症状：A 调 B，B 又调 A
   - 解法：用领域事件解耦，或重新划分边界（可能是同一上下文）

4. **贫血模型 + 服务堆积**
   - 症状：Service 类全是过程式代码，Entity 只是 POJO
   - 解法：把业务规则收敛到聚合根（充血模型）

5. **共享数据库**
   - 症状：多个服务连同一个库，绕过接口直接改表
   - 解法：每服务独立库，跨服务只能通过 API/事件

**绞杀者模式（单体演进）**：

```
阶段 1：单体边缘加新服务（新建功能走新服务）
  ┌─────────────┐
  │   Monolith  │  ←── 新功能不走这里
  │  （老功能）  │
  └─────────────┘
        +
  ┌─────────────┐
  │  New Service │  ←── 新功能走这里
  └─────────────┘

阶段 2：逐步迁移老功能（按上下文）
  ┌─────────────┐    ┌─────────────┐
  │  Monolith   │    │  Order Svc  │  ←── 订单迁出
  │  （剩余）    │    │  Product Svc│  ←── 商品迁出
  └─────────────┘    └─────────────┘

阶段 3：单体萎缩下线
  ┌─────────────┐
  │  全部微服务   │
  └─────────────┘
```

## 五、底层本质：为什么按领域而不是按技术或表拆

回到第一性：**软件架构要对齐业务变更轴，而不是技术分层轴**。

- **业务变更按领域发生**：商品团队改商品逻辑、订单团队改订单流程，不会出现"改所有服务的 DAO 层"这种需求。按领域拆分让变更影响域收敛在单个服务，团队独立迭代。
- **限界上下文对齐康威定律**：康威定律说"系统结构反映组织沟通结构"。按领域拆服务，对应按业务域组团队（商品团队、订单团队），团队内高频沟通、团队间低频协作，沟通成本最低。如果按技术层组团队（前端、后端、DBA），每个需求要跨团队协调，摩擦大。
- **聚合根保证一致性边界**：聚合内的强一致（订单和订单项必须一起变更）用本地事务保证，聚合间的最终一致用领域事件保证。这种"分级一致性"避免了分布式事务的复杂性。
- **领域事件实现解耦**：上下文间通过事件通信，发布方不关心谁订阅（库存服务挂了不影响订单创建），实现真正的独立部署和容错。

**拆分粒度的本质权衡**：微服务的收益（独立部署、独立扩容、技术栈自由）和代价（网络延迟、分布式事务、运维复杂度）的平衡点。粒度太细则 RPC 链路长、一致性难保证；粒度太粗则失去独立部署价值。经验法则：一个服务一个团队（6-10 人），一个团队负责一个或少数几个上下文。

## 六、AI 架构师加问：5 个 AI 相关问题

1. **AI 能自动识别限界上下文吗？**
   能辅助。用 LLM 分析需求文档/代码/数据库 schema，提取领域名词（高频出现的相关词），聚类成候选上下文。但最终边界要架构师结合业务语义判断（AI 不懂业务变更频率和团队归属）。AI 输出候选方案，人工 review 调整。

2. **AI Agent 的多步骤编排怎么对应 DDD 聚合？**
   Agent 的每个"工具调用"对应一个上下文的命令（如 search 工具=查询上下文，calc 工具=计算上下文）。Agent 编排逻辑本身是一个"应用服务"，协调多个上下文，但不持有业务状态（状态在聚合根）。Agent 的可逆操作用领域事件 + 补偿事务（Saga）保证最终一致。

3. **怎么用 AI 辅助事件风暴？**
   AI 解析会议纪要/PRD，自动提取领域事件（动词过去式 + 名词）、命令、参与者。输出事件风暴便签草稿，团队在其上修改补充。AI 还能检测遗漏（如识别到 Payment 但没识别 Refund），提示补充。

4. **微服务拆分后，AI 怎么辅助跨服务问题定位？**
   AI 分析全链路 trace（如 SkyWalking 数据），识别慢调用集中在哪两个服务之间；分析日志关联异常（订单服务报错 + 库存服务超时）；输出根因假设 + 推荐排查路径。但根因确认要人工（AI 不懂业务上下文）。

5. **怎么防 AI 误拆服务导致拆得过碎？**
   AI 建议拆分时必须输出"拆分前后调用链路对比"（拆前 3 跳，拆后 7 跳就是过碎）、"跨服务事务需求清单"（强一致需求多的不该拆）、"团队归属映射"。人工 review 这些指标后再决策。AI 还要监控拆分后的 RPC 调用图密度，超阈值预警。

## 七、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"限界上下文、聚合根、上下文映射、领域事件、拆分粒度"**。

- **限界上下文**：术语一致 + 规则一致 + 变更同频 = 拆分单元
- **聚合根**：一致性边界，外部只引用 ID
- **上下文映射**：U/D、ACL（防腐层）、Shared Kernel、OHS+PL
- **领域事件**：上下文解耦，本地消息表保证可靠
- **粒度**：两个披萨团队（6-10 人一服务），按变更频率/流量/团队拆

### 拟人化理解

把微服务拆分想成**城市规划**。限界上下文是功能分区（商业区、住宅区），区内建筑风格统一（同一模型），区间靠道路连接（接口契约）。聚合根是一栋楼的大门——外人来访只能走大门登记（通过聚合根操作），不能翻窗户进某个房间（直接改订单项）。领域事件是城市广播——住宅区发生事件（订单创建），商业区听到自己响应（库存扣减），但事件发布方不关心谁在听。上下文映射是区与区的关系——商业区是上游（提供商品），住宅区是下游（消费商品），中间建翻译层（ACL）防止商业区术语污染住宅区。

### 面试现场 60 秒回答

> 微服务拆分从领域模型推导，核心是找限界上下文。用事件风暴梳理领域事件→命令→聚合→上下文。限界上下文的标志是术语一致、规则一致、变更同频。每个上下文对应一个或一组服务。上下文内用聚合根定义一致性边界，外部只引用聚合根 ID，聚合内强一致用本地事务。上下文间用领域事件解耦，本地消息表保证可靠投递。上下文关系用上下文映射描述：U/D、ACL 防腐层、共享内核、OHS。拆分粒度参考两个披萨团队（6-10 人一服务），按变更频率和流量独立需求拆，避免按表拆（强一致需求高的不该拆）和拆得过碎（RPC 链路过长）。老系统演进用绞杀者模式，逐步迁移不要一次性重构。

### 反问面试官

> 贵司微服务是按业务域拆（订单/商品/库存各自独立）还是按团队拆？现在有多少个服务、有没有遇到拆得过碎（一个请求调 10+ 服务）的问题？如果有，我会聊 BFF 聚合和上下文合并。

### 现场对话示例

**面试官**：你怎么判断订单和订单项该在一个服务还是拆开？

**候选人**：核心看一致性需求。订单项是订单聚合的一部分——创建订单时必须同时插入订单和订单项（要么全成功要么全失败），这是强一致需求。如果拆成两个服务，创建订单要跨服务事务（Saga 或 TCC），引入额外复杂度和延迟。从 DDD 角度，订单项没有独立的业务生命周期（不能脱离订单单独存在），它不是独立聚合，是 Order 聚合内的实体。所以订单和订单项必须在同一服务，共用数据库表，本地事务保证强一致。判断依据是：A 能否独立存在？如果能（如商品可以脱离订单存在），就是独立聚合，可能拆服务；如果不能（订单项脱离订单无意义），就是聚合内实体，不拆。

**面试官**：那商品和订单为什么能拆？

**候选人**：因为商品和订单是不同的限界上下文，术语和规则不一致。商品在商品上下文是完整的 SKU 信息（含规格、库存、价格），在订单上下文是下单时的商品快照（锁定价格和规格，防止后续改价影响历史订单）。两个上下文用不同的模型（商品上下文用 Product 聚合，订单上下文用 OrderItem 引用 ProductId + 快照字段）。变更频率也不同——商品频繁改价/上下架，订单是用户行为驱动，独立部署收益大。一致性上，订单创建时只需查询商品信息（最终一致可接受，快照已锁定关键字段），不需要强一致。所以商品和订单拆成两个服务。

**面试官**：如果订单创建时要扣库存，库存扣了但订单创建失败怎么办？

**候选人**：这是典型的分布式一致性问题。订单和库存是两个上下文，用领域事件 + Saga 模式保证最终一致。流程是：订单服务先创建订单（待确认状态）→ 发布 OrderCreated 事件 → 库存服务订阅事件扣减库存 → 扣减成功发布 InventoryDeducted → 订单服务订阅事件确认订单。如果库存扣减失败（如库存不足），发布 InventoryDeductFailed → 订单服务订阅事件取消订单。这是编排式 Saga（用事件驱动），也可以用协调式 Saga（中心协调器调各服务）。关键点：库存扣减要支持回滚（释放锁定），订单状态机要支持"待确认→已确认/已取消"的状态流转。详见 018 题分布式事务。

## 常见考点

1. **限界上下文和微服务什么关系？**——限界上下文是逻辑边界（模型一致），微服务是物理边界（独立部署）。一个上下文可以是一个服务，也可以拆成多个（如 CQRS 拆读写服务），还可以多个上下文合并成一个服务（初期业务小）。拆分要考虑团队规模、流量、变更频率。
2. **聚合根怎么设计？**——聚合根是聚合的入口实体，封装一致性规则。外部只能引用聚合根 ID（值对象），不能引用聚合内实体。聚合内修改必须通过聚合根方法（不暴露 setter）。聚合要小（只包含必须强一致的实体），大聚合会导致锁竞争和性能问题。
3. **领域事件和消息队列什么关系？**——领域事件是业务概念（OrderCreated），消息队列是技术实现（Kafka/RocketMQ）。领域事件通过消息队列投递，但要保证"业务操作和事件发布原子性"——用本地消息表（业务和消息记录在同一事务，异步投递消息）。详见 018 题。
4. **贫血模型和充血模型区别？**——贫血模型：Entity 只是 POJO（getter/setter），业务逻辑在 Service（过程式）。充血模型：Entity 封装业务规则（confirm、addItem 等方法），Service 只编排。DDD 推崇充血模型，把规则收敛到聚合根，避免 Service 膨胀。


## 结构化回答

**30 秒电梯演讲：** 聊到微服务拆分边界如何从领域模型推导，我的理解是——微服务拆分边界的本质是"找到业务的限界上下文（Bounded Context）"——一组高内聚的业务概念和规则形成一个边界，边界内的模型一致、边界间通过明确的接口契约交互。从领域模型推导而非从数据库表拆，能避免"按表拆服务"导致的分布式大对象、跨服务事务、循环依赖。打个比方，像城市规划划地块。商业区、住宅区、工业区各有边界（限界上下文），区内建筑风格统一（同一模型），区间靠道路连接（接口契约）。不能把一栋楼的厨房划到隔壁地块（订单的"地址"字段不该单独成服务），也不能把住宅区和工业区混在一起（订单和库存模型不该共用一张表）。

**展开框架：**
1. **限界上下文（Bounded Context）** — 限界上下文（Bounded Context）是拆分的核心单元：一个领域内术语一致、规则一致
2. **上下文映射（Context Map）** — 上游/下游、防腐层（ACL）、共享内核、客户/供应商
3. **聚合根（Aggregate Root）** — 一致性边界，外部只能引用聚合根 ID 不能引用内部对象

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：一个限界上下文一定对应一个微服务吗？您更想看哪个方向？

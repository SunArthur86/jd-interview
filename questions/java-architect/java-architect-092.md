# 【Java 后端架构师】架构评审怎么讲清取舍与风险

> 适用场景：JD 核心技术。一个订单系统要拆分成微服务，架构师在评审会上展示了一堆时序图和组件选型，被评委追问"为什么不保持单体""拆分过程中数据不一致怎么办""回滚方案呢"，答不上来被否。架构评审考的不是方案多漂亮，而是你有没有想清楚"为什么这样做、代价是什么、出问题怎么办"。

## 一、概念层：评审的三输出与四阶段

### 评审三输出（缺一不可）

| 输出 | 内容 | 价值 |
|------|------|------|
| **ADR（决策记录）** | Context（背景）+ Decision（决策）+ Consequences（代价收益） | 让取舍显性化，未来可追溯为什么这么选 |
| **风险矩阵** | 识别的风险 + 概率/影响 + 兜底措施 | 证明风险可控而非假装没风险 |
| **验证计划** | 成功指标 + 灰度策略 + 回滚条件 | 证明方案上线后可验证、可回退 |

### 评审四阶段（闭环）

```
1. 预沟通（评审前 1 周）
   - 和关键评委（架构师/SRE/业务）1v1 对齐争议点
   - 收敛方案选项，不在会上才暴露分歧
        ▼
2. 评审会（1-2 小时）
   - 讲：背景目标 → 方案 → 取舍 → 风险 → 验证计划
   - 答：评委追问（聚焦取舍和风险，不纠结实现细节）
   - 决：通过 / 有条件通过 / 否决 / 需重新评审
        ▼
3. 决议跟踪（评审后）
   - ADR 入 Git，风险项建跟踪任务
   - 有条件通过的事项限期闭环
        ▼
4. 上线验证（上线后）
   - 对比验证计划的指标，证明方案有效
   - 未达预期进入复盘，更新 ADR
```

## 二、机制层：ADR 模板与风险矩阵

### 2.1 ADR 模板（Architecture Decision Record）

```markdown
# ADR-042: 订单系统从单体拆分为微服务

## Status
Accepted (2026-07-13)

## Context（背景与问题）
- 订单单体应用 50 万行代码，30 人团队协作，发版互相阻塞
- 大促期间订单峰值 10 万 QPS，单体扩容时全链路抖动（GC、连接池）
- 库存、营销、支付耦合在订单里，一个慢调用拖垮全链路
- 业务要求：支持独立发版、独立扩容、故障隔离

## Decision（决策）
将订单单体拆分为 3 个微服务：order-core（下单主链路）、order-query（查询）、order-async（异步履约）。

### 关键取舍

| 决策点 | 选择 | 换来了什么 | 付出了什么 |
|--------|------|-----------|-----------|
| 拆分粒度 | 3 个服务而非 10 个 | 独立扩容+故障隔离，复杂度可控 | 分布式事务、跨服务调用 |
| 通信方式 | 同步 RPC（核心）+ MQ（异步） | 核心链路低延迟，异步解耦 | 一致性窗口（MQ 延迟） |
| 数据一致性 | 最终一致（saga + 对账） | 性能和可用性 | 强一致场景需补偿 |
| 服务发现 | Nacos | 动态扩缩容 | 运维依赖 |

### 考虑过的替代方案
- **方案 B：保持单体 + 模块化**：成本低但无法独立扩容，否决
- **方案 C：拆 10 个微服务**：隔离更彻底但运维和一致性成本爆炸，否决

## Consequences（代价与收益）
- 正面：独立发版（发版频率从 2 周→2 天）、独立扩容（大促只扩核心）、故障隔离
- 负面：引入分布式事务（saga 复杂度）、跨服务调试难、运维成本上升
- 风险：见风险矩阵 ADR-042-RISK
```

### 2.2 风险矩阵（RISC 矩阵）

```markdown
# 风险矩阵 ADR-042-RISK

| 风险 | 概率 | 影响 | 兜底措施 | 负责人 |
|------|------|------|---------|--------|
| 拆分过程中数据不一致（双写偏差） | 中 | 高 | 双写 + 对账任务 + 差异告警，超阈值自动回滚切流 | @张三 |
| saga 补偿失败导致悬空事务 | 低 | 高 | 悬空事务巡检 + 人工兜底 SOP，T+1 清零 | @李四 |
| 跨服务调用链路变长导致延迟上升 | 中 | 中 | 压测验证 P99 < 200ms，超限加缓存/并行化 | @王五 |
| 服务间依赖循环 | 中 | 中 | 依赖检查 CI + 架构守护测试 | @赵六 |
| 运维复杂度上升导致故障排查难 | 高 | 中 | 完善链路追踪 + 统一看板 + 故障演练 | @运维 |
```

### 2.3 验证计划

```markdown
# 验证计划

## 成功指标（上线后 2 周验证）
| 指标 | 基线（单体） | 目标（微服务） | 验证方式 |
|------|------------|--------------|---------|
| 发版频率 | 2 周/次 | 2 天/次 | 统计发版记录 |
| 大促扩容影响 | 全链路抖动 5min | 核心链路无感 | 大促压测对比 |
| P99 延迟 | 150ms | < 200ms | 线上监控 |
| 故障隔离 | 单点故障全链路挂 | 非核心故障不影响核心 | 故障演练 |

## 灰度策略
- 第 1 周：10% 流量灰度，双写校验 diff < 0.01%
- 第 2 周：50% 流量，监控核心指标
- 第 3 周：100% 切流，保留单体回滚能力 1 个月

## 回滚条件
- 数据一致性 diff > 0.1%（对账失败率飙升）
- P99 > 300ms 且持续 10 分钟
- 核心链路错误率 > 1%
触发任一条件，30 分钟内切回单体
```

## 三、实战层：评审会的表达技巧

### 3.1 评审会发言结构（20-30 分钟）

```
1. 背景目标（3 分钟）
   - 业务痛点（用数据说话："大促期间 3 次全链路抖动，影响订单 X 单"）
   - 目标（"独立扩容 + 故障隔离，发版频率提升 5 倍"）

2. 方案概览（5 分钟）
   - 架构图（一页纸，关键链路清晰）
   - 核心技术选型（拆分粒度、通信、一致性）

3. 取舍说明（5 分钟，重点）
   - 每个关键决策：换来什么 + 付出什么
   - 为什么不选替代方案（带对比）

4. 风险与兜底（5 分钟，重点）
   - 主动点出 Top 5 风险
   - 每个风险配兜底措施

5. 验证计划（3 分钟）
   - 成功指标 + 灰度 + 回滚

6. Q&A
```

### 3.2 高分回答 vs 低分回答

| 评委追问 | 低分回答 | 高分回答 |
|---------|---------|---------|
| "为什么不保持单体？" | "微服务是最佳实践" | "单体无法独立扩容，大促全链路抖动 3 次。但单体成本低，所以我们分阶段拆，先拆核心链路验证" |
| "数据不一致怎么办？" | "用分布式事务" | "用 saga + 对账，容忍 T+1 最终一致。强一致场景（资金）走 TCC。对账 diff > 0.1% 自动告警" |
| "回滚方案呢？" | "可以回滚" | "保留单体 1 个月，双写期间可秒切回。回滚条件明确：P99>300ms 或错误率>1%" |
| "压测了吗？" | "还没" | "POC 阶段压了核心链路，10 万 QPS 下 P99 180ms。全量压测在灰度前完成" |

## 四、底层本质：评审是置信度传递

架构评审的底层不是"证明方案完美"，而是**传递置信度**——让评委相信"你想清楚了取舍、识别了风险、有兜底预案"。评委否决一个方案，往往不是方案本身错，而是置信度不够（答不清取舍和风险）。

**为什么取舍是评审核心**：架构是约束下分配资源，没有取舍就没有架构（"全都要"是需求文档不是架构方案）。讲清取舍证明你想过了权衡，而不是拍脑袋。每个"选 A 不选 B"都要回答：A 换来了什么、付出了什么、为什么这个场景下值得。

**为什么风险要主动点出**：藏着风险不代表没风险，评委会追问。主动点出风险 + 带兜底措施，反而证明你成熟（识别风险的能力 + 应对的准备）。最差的是被评委问出风险却没对策——那是否决信号。

**为什么用证据替代感觉**："我觉得没问题"没有说服力。压测数据、POC 结果、历史事故案例、行业标杆，这些是客观证据。评审要的是"这个方案在 X 条件下验证过，指标 Y 达标"，不是"我觉得应该行"。

**为什么预沟通比评审会重要**：评审会上暴露分歧是最差的——当众辩论容易变成面子之争。预沟通把争议点提前 1v1 收敛，评审会聚焦"对齐共识 + 记录决议"，效率高且不伤和气。资深架构师 80% 的说服工作在会前完成。

## 五、AI 架构师加问：5 个 AI 场景的评审

1. **LLM 接入方案的评审，取舍怎么讲？**
   核心取舍：确定性 vs 灵活性。LLM 带来灵活性（非确定决策）但要付出确定性代价（幻觉、延迟、成本）。评审要讲清：哪些链路允许非确定（容错高）、哪些必须确定性（资金/权限走规则引擎兜底）。风险：LLM 幻觉写入核心系统，兜底是 schema 校验 + 输出白名单 + 高敏操作人工确认。

2. **AI 方案评审的验证计划特殊在哪？**
   传统方案验证"性能/可用性"，AI 方案还要验证"质量"（task_accuracy）和"成本"（token_cost_per_request）。质量验证要建 eval 集，成本验证要算 ROI（AI 节省的人工 vs AI 调用成本）。这两项是 AI 评审特有的，漏了就是方案不完整。

3. **Agent 方案的风险矩阵怎么列？**
   Agent 特有风险：(1) 工具越权（LLM 被诱导调用危险工具）→ 工具白名单 + 服务端二次校验；(2) 死循环烧钱 → 步数/token/耗时上限；(3) 幻觉决策 → 高敏操作人工确认；(4) 长任务不可恢复 → checkpoint + saga。这些风险必须在评审时识别并配兜底。

4. **AI 方案被"这个模型能力够吗"质疑怎么回应？**
   用 POC 数据回应。准备：在评测集上跑过（task_accuracy X%）、在 5% 灰度流量上验证过（user_feedback_score Y）、有降级路径（模型不够好时切规则引擎）。不能只说"GPT-4 很强"，要拿业务场景的实际数据证明。

5. **怎么评审"是否该引入 AI"本身？**
   先问业务价值：AI 解决的是确定性规则解决不了的问题吗？（如自然语言理解、非结构化内容生成）。再算 ROI：AI 调用成本 + 改造成本 vs 节省的人工/提升的转化。最后定边界：哪些走 AI（灵活决策），哪些留规则（确定性执行）。评审通过的标准不是"AI 先进"，而是"ROI 为正且风险可控"。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"取舍、风险、验证、预沟通"** 四个词。

- **取舍**：每个决策讲清"换来什么、付出什么、为什么值得"
- **风险**：主动点出 Top 5 风险 + 每个配兜底措施
- **验证**：成功指标 + 灰度策略 + 回滚条件，缺一不可
- **预沟通**：80% 说服工作在会前完成，会上聚焦共识

### 面试现场 60 秒回答

> 架构评审的核心是讲清取舍和风险，不是展示方案多牛。评审三输出：ADR（决策记录，每个关键决策讲清换来什么付出什么）、风险矩阵（主动点出 Top 5 风险 + 兜底措施）、验证计划（成功指标 + 灰度策略 + 回滚条件）。流程上，评审前 1 周预沟通对齐争议点，评审会 20-30 分钟讲背景-方案-取舍-风险-验证，评审后 ADR 入 Git、风险项跟踪、上线后验证指标。表达上用证据替代感觉——压测数据、POC 结果、历史事故案例。最忌讳的是答不清"为什么选 A 不选 B"和"出问题怎么办"，那是否决信号。

## 七、苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 为什么架构方案要评审，不能架构师自己定？ | 评审的价值是视角互补（SRE 看运维、安全看合规、业务看价值）+ 集体背书（降低单点决策风险）+ 知识共享。但评审不是民主投票，是帮架构师发现盲点，最终决策权在 owner |
| 证据追问 | 评审会上你说"压测过"，具体怎么证明？ | 出压测报告：场景（10 万 QPS 持续 30 分钟）、指标（P99 180ms、错误率 0.01%、CPU 70%）、对比基线（单体 P99 150ms，微服务略高但可接受）、瓶颈分析（DB 连接池是下一步优化点）|
| 边界追问 | 评审能保证方案不出事吗？ | 不能。评审降低风险不消除风险。评审的价值是"识别已知风险 + 兜底"，但未知风险（黑天鹅）仍会发生。所以评审必须配验证计划 + 回滚预案，出事能快速恢复 |
| 反例追问 | 什么方案不该过重评审？ | 非核心链路小改动、实验性功能、内部工具——走 PR review 轻流程即可。重评审留给：核心链路、数据迁移、外部接口、安全相关。用变更影响面定评审级别，避免评审疲劳 |
| 风险追问 | 评审通过后还是出事了，怎么办？ | 先止损（回滚/降级），再复盘。复盘要回溯评审：风险是否识别到了？兜底是否生效？验证计划是否漏了？更新 ADR 和风险矩阵。评审通过不免责，owner 仍要对线上负责 |
| 验证追问 | 怎么证明评审真的起到了把关作用？ | 统计：评审否决率（健康值 10-20%，太低说明走过场，太高说明预沟通不够）、评审后的事故率（应低于未评审的）、ADR 数量和质量（是否真在记录取舍）|
| 沉淀追问 | 团队评审规范沉淀什么？ | ADR 模板、风险矩阵模板、评审 checklist（必含：取舍、风险、验证、回滚）、评审分级标准（什么改动走什么级别）、评审决议跟踪流程 |

### 现场对话示例

**面试官**：你说评审要讲取舍，能举一个你经历过的取舍吗？

**候选人**：订单系统拆微服务时，拆分粒度是核心取舍。选 3 个服务而非 10 个——换来的是独立扩容和故障隔离够用，付出的是分布式事务和运维成本可控。如果拆 10 个，隔离更彻底但一致性成本爆炸（10 个服务间的 saga 链路太复杂）。决策依据是大促压测数据：3 个服务已能满足 10 万 QPS 核心链路隔离需求，没必要过度拆分。这个取舍记在 ADR 里，未来业务量翻倍再重新评估。

**面试官**：评审时有人坚持要拆 10 个，你怎么办？

**候选人**：不在会上辩论。会前预沟通时拿出对比：3 个 vs 10 个的运维成本（监控大盘数量、故障定位复杂度、saga 链路长度）、压测数据（3 个已满足）、行业案例（阿里中间件团队的经验也是先粗后细）。用证据说服，不是用职级压。如果对方仍坚持，请他出一份"拆 10 个的压测和运维成本评估"，用数据对数据。最终如果达不成共识，升级到技术委员会决策，但要记录分歧（ADR 里写"考虑过 10 个方案，因 X 否决"）。

**面试官**：评审通过了，上线后 P99 飙到 500ms，怎么办？

**候选人**：先看回滚条件——验证计划里写的"P99 > 300ms 持续 10 分钟触发回滚"，500ms 已超阈值，立即切回单体（保留期内的秒切能力）。止损后复盘：是压测没覆盖到的场景（如某个慢 SQL 在真实流量下放大）、还是容量估算偏差（实际 QPS 超过压测）、还是依赖服务抖动。复盘结论更新到 ADR（"拆分后 P99 上升，需先优化 X 再拆"），重新评估方案。评审通过不代表免责，我对线上负责到底。

## 常见考点

1. **ADR 是什么？**——Architecture Decision Record，架构决策记录。记录 Context（背景）、Decision（决策）、Status（状态）、Consequences（代价收益）。一页纸，放 Git 版本化，未来可追溯为什么这么选。是架构评审的标准输出。
2. **评审怎么分级？**——按变更影响面：核心链路/数据迁移/外部接口/安全相关走重评审（委员会 + ADR + 验证计划）；非核心小改动走轻评审（PR review + 技术负责人确认）。分级避免评审疲劳。
3. **评审会多长合适？**——核心方案 1-2 小时（含 Q&A），小方案 30 分钟。超时说明预沟通不够（会上才暴露分歧）。评审会效率 80% 取决于会前准备。
4. **评审被否怎么办？**——不要对抗，问清否决理由（是风险没兜底、还是取舍不合理、还是证据不足）。补证据或调整方案后重新评审。被否是评审在起作用，不是失败。
5. **评审委员会怎么组成？**——架构师 + 资深开发 + SRE/运维 + 安全（涉及时）+ 业务负责人（核心链路时）。4-7 人，视角互补又不至于低效。固定成员保证评审标准一致。

## 结构化回答

**30 秒电梯演讲：** 架构评审不是展示方案多牛，而是证明取舍合理、风险可控。好的评审输出三样东西：决策记录（ADR，为什么选 A 不选 B）、风险矩阵（识别的风险 + 兜底措施）、验证计划（怎么证明上线后有效）。讲不清取舍的评审是自嗨，列不出风险的方案是赌博

**展开框架：**
1. **评审三输出** — ADR（决策记录）、风险矩阵（风险+兜底）、验证计划（指标+灰度+回滚）
2. **取舍要讲清** — 每个选择的"换来了什么"和"付出了什么"
3. **风险要主动点出** — 不藏风险，而是带着兜底措施一起讲

**收尾：** 以上是我的整体思路。您想继续深入聊——ADR 怎么写？

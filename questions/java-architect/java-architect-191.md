# 【Java 后端架构师】遗留系统绞杀者模式改造实践

> 适用场景：JD 核心技术。一个跑了 8 年的订单单体，50 万行代码，30 人协作，没人敢动。直接重写要 2 年、风险极高；不重写每次大促就抖动。绞杀者模式是第三条路——渐进迁移，每步可回滚，把"重写项目"分解成"N 次安全的小替换"。

## 一、概念层：绞杀者模式与一次性重写的对比

| 维度 | 一次性重写 | 绞杀者模式 |
|------|----------|-----------|
| 风险 | 项目级（全有全无） | 功能级（每个单元独立） |
| 时间 | 2-3 年大项目 | 持续渐进，每季度可见产出 |
| 回滚 | 不可能（回不去） | 每步可回滚（路由切回老系统） |
| 业务中断 | 高（割接期） | 低（灰度切流） |
| 失败率 | 高（Joel Spolsky: 重写是最大错误） | 低（小步快跑） |
| 适合场景 | 系统彻底报废重建 | 核心业务在跑、不能停服 |

**核心思想**：不重写整个系统，而是引入 Facade 层，把功能按优先级一个个迁移到新系统。新功能直接在新系统实现（不再改老系统），旧功能逐步替换。终态是老系统"枯萎"（无流量）后下线。

## 二、机制层：Facade 路由与渐进迁移

### 2.2 Facade 路由实现（Spring Cloud Gateway）

```yaml
# 网关路由配置：按路径/Header 路由到新/旧系统
spring:
  cloud:
    gateway:
      routes:
        # 已迁移：订单查询走新系统
        - id: order-query-new
          uri: lb://order-query-service  # 新微服务
          predicates:
            - Path=/api/order/query/**
            - Header=X-Migrate-Version, v2  # 灰度 Header
        
        # 未迁移：订单查询默认走老系统
        - id: order-query-old
          uri: lb://legacy-order-system   # 老单体
          predicates:
            - Path=/api/order/query/**
        
        # 已迁移：库存查询全量走新系统
        - id: inventory-new
          uri: lb://inventory-service      # 新微服务
          predicates:
            - Path=/api/inventory/**
        
        # 新功能：只在老系统不存在的，直接新系统
        - id: recommendation-new
          uri: lb://recommendation-service
          predicates:
            - Path=/api/recommendation/**
```

**灰度切流**：通过 Header（`X-Migrate-Version: v2`）或 Cookie 控制流量比例。10% 用户带 v2 Header 走新系统，验证无问题后逐步放大到 50% → 100%。

### 2.3 单个功能迁移的五个阶段

```
阶段 1: 双写（新旧系统并行写）
  ┌─────────┐
  │ 客户端   │ ──→ 老系统（主写）──→ 老DB
  └─────────┘         │
                      │ MQ/CDC
                      ▼
                   新系统（影子写）──→ 新DB
  验证：新DB 数据与老DB 一致（对账 diff < 0.01%）
        ▼
阶段 2: 影子读（读流量并行，对比结果）
  ┌─────────┐
  │ 客户端   │ ──→ 老系统（返回客户端）──→ 老DB
  └─────────┘         │
                      │ 异步
                      ▼
                   新系统（影子读，不返回客户端）──→ 新DB
  验证：新老读结果 diff 率（记录但不影响用户）
        ▼
阶段 3: 灰度切流（10%→50%→100%）
  ┌─────────┐
  │ 客户端   │ ──→ Facade ──┬─→ 90% 老系统
  └─────────┘               └─→ 10% 新系统
  验证：新系统错误率/P99/业务指标达标
        ▼
阶段 4: 全量切新系统
  ┌─────────┐
  │ 客户端   │ ──→ Facade ──→ 100% 新系统
  └─────────┘
  老系统保留（应急回滚），观察 1-2 周
        ▼
阶段 5: 老系统下线
  ┌─────────┐
  │ 客户端   │ ──→ 新系统
  └─────────┘
  老系统连续 30 天零流量，灰度下线（先摘流量，再关服务）
```

### 2.4 数据迁移：CDC 实时同步

```java
// CDC（Debezium）监听老 DB binlog，实时同步到新 DB
@Configuration
public class LegacyDataSyncConfig {
    
    // Debezium 监听 MySQL binlog
    @Bean
    public io.debezium.config.Configuration debeziumConfig() {
        return io.debezium.config.Configuration.create()
            .with("name", "legacy-order-sync")
            .with("connector.class", "io.debezium.connector.mysql.MySqlConnector")
            .with("database.hostname", "legacy-mysql.jd.com")
            .with("database.serverId", "184054")
            .with("database.includeList", "legacy_order")
            .with("table.includeList", "legacy_order.orders,legacy_order.order_item")
            .with("database.history.kafka.bootstrap.servers", "kafka:9092")
            .with("database.history.kafka.topic", "schema-changes.legacy_order")
            .build();
    }
}

// 消费 CDC 事件，写入新 DB
@KafkaListener(topics = "legacy_order.legacy_order.orders")
public void syncOrder(ChangeEvent<String, String> event) {
    OrderCdcEvent cdc = parse(event.value());
    OrderNew orderNew = mapToNewSchema(cdc.getAfter());  // 老schema → 新schema
    orderNewRepository.save(orderNew);
}
```

**对账机制**（迁移期的生命线）：

```java
// 定时对账：老 DB vs 新 DB 数据一致性
@Scheduled(cron = "0 0 * * * *")  // 每小时对账
public void reconcile() {
    List<Long> orderIds = legacyRepo.findUpdatedInLastHour();
    int diff = 0;
    for (Long id : orderIds) {
        OrderLegacy legacy = legacyRepo.findById(id);
        OrderNew neu = newRepo.findById(id);
        if (!match(legacy, neu)) {
            diff++;
            alertService.sendDiffAlert(id, legacy, neu);  // 差异告警
            autoRepair(id);  // 自动修复（从老 DB 重同步）
        }
    }
    metrics.recordDiffRate((double) diff / orderIds.size());  // 监控 diff 率
}
```

**对账的红线**：diff 率 > 0.1% 暂停切流，> 1% 自动回滚到老系统。迁移期对账是数据一致性的最后一道防线。

## 三、实战层：JD 订单单体改造案例

### 3.1 改造规划（按领域优先级）

```
订单单体（50w 行，30 人协作）
├── 领域 1: 订单查询（高频读，先迁）        ← Q1
├── 领域 2: 库存（高耦合，第二批）          ← Q2
├── 领域 3: 下单主链路（核心，最慎重）      ← Q3-Q4
├── 领域 4: 营销（业务变化快，迁后迭代快）  ← Q2
└── 领域 5: 履约（异步，迁后解耦）          ← Q4

迁移顺序原则：
1. 先迁读多于写的（风险低，读可重试）
2. 先迁边界清晰的（依赖少）
3. 核心链路最后迁（积累经验后再碰）
4. 每次迁一个领域，验证 1-2 周再下一个
```

### 3.2 单领域迁移 Runbook（订单查询）

```markdown
# 订单查询迁移 Runbook

## 迁移目标
老系统的订单查询功能迁移到新 order-query-service。

## Phase 1: 双写（第 1 周）
- [ ] 新 order-query-service 部署上线（无流量）
- [ ] CDC 同步老 DB → 新 DB（全量 + 增量）
- [ ] 对账任务上线，每小时跑，监控 diff 率
- [ ] 准入标准：diff 率连续 3 天 < 0.01%

## Phase 2: 影子读（第 2 周）
- [ ] 网关异步复制读请求到新系统（不影响用户响应）
- [ ] 对比新老读结果，记录 diff 率（功能正确性）
- [ ] 准入标准：影子读 diff 率连续 3 天 < 0.1%

## Phase 3: 灰度切流（第 3-4 周）
- [ ] 网关按 Header 路由 10% 流量到新系统
- [ ] 监控：错误率、P99、业务成功率
- [ ] 逐步放大：10% → 30% → 50% → 80% → 100%
- [ ] 每个档位观察 2 天，无异常才放大
- [ ] 准入标准：100% 流量下错误率不上升、P99 不上升

## Phase 4: 全量切流（第 5 周）
- [ ] 100% 流量走新系统
- [ ] 老系统保留（应急回滚），观察 2 周
- [ ] 回滚预案：网关一键切回老系统（< 1 分钟）

## Phase 5: 老系统下线（第 8 周）
- [ ] 确认老系统订单查询连续 30 天零流量
- [ ] 灰度下线：先摘流量（路由不指向），观察 1 周
- [ ] 关闭老系统订单查询接口
- [ ] 老代码暂不删除（保留 3 个月应急）
```

### 3.3 回滚预案（每步都可回滚）

```yaml
# 回滚决策树
触发回滚的条件（任一满足）：
  - 新系统错误率 > 1% 持续 5 分钟
  - 新系统 P99 > 老系统 1.5 倍持续 10 分钟
  - 业务成功率下降 > 0.5%
  - 对账 diff 率 > 1%

回滚动作：
  1. 网关一键切回老系统（修改路由配置，< 1 分钟生效）
  2. 通知值班 SRE 和业务方
  3. 复盘：哪个环节出问题（数据不一致？新系统 bug？依赖服务？）
  4. 修复后重新进入灰度（从 10% 开始，不从 50% 继续）

回滚不能做的事：
  - 不要尝试"边修边切"（延长故障时间）
  - 不要删新系统数据（保留用于复盘）
  - 不要瞒报（事故要复盘）
```

## 四、底层本质：为什么绞杀者模式比大重写安全

**一次性大重写的失败模式**：Joel Spolsky 在《Things You Should Never Do》里说重写是开发者能犯的最大错误。原因：(1) 老系统的代码里藏着多年积累的 bug fix 和边界处理，重写会重新踩这些坑；(2) 重写期间老系统还在硬撑（没人敢改），新系统迟迟上不了，两边都没维护好；(3) 重写 2-3 年，业务早变了，新系统上线就过时。Netscape 在 1990 年代重写 Navigator，花了 5 年，期间失去整个市场给 IE。

**绞杀者模式的安全来自"可逆性"**：每个迁移单元（一个领域/一个接口）都走"双写→影子读→灰度→全量→下线"五步，每步都能回滚（网关切回老系统）。一个单元失败不影响其他单元，风险隔离在功能级。这把"重写项目"的风险从"全有全无"降到"N 次独立赌博"——即使个别失败，整体仍能推进。

**为什么 Facade 是关键**：Facade（网关/路由层）是新旧系统的"切换开关"。没有 Facade，新旧系统是两个独立入口，客户端要改调用地址，迁移成本高。有 Facade，客户端只调 Facade，内部路由透明切换，客户端无感。Facade 让"切流"成为一个配置变更（秒级），而不是代码修改（周级）。

**为什么数据迁移是最难的**：代码可以灰度切流（10% 流量试），但数据是全量的——新老 DB 必须保持一致。迁移期新老并行写（双写）+ 实时同步（CDC）+ 对账（diff 监控）才能保证。对账 diff 率是迁移能否推进的红线指标——diff 率 > 0.1% 暂停切流，说明数据同步有问题。

## 五、AI 架构师加问：5 个

1. **AI 怎么辅助绞杀者模式迁移？**
   AI 负责代码理解（分析老系统调用链、依赖图）、迁移风险评估（"这个接口被 20 个服务调用，迁移风险高"）、对账异常归因（"diff 集中在营销模块，疑似优惠券字段映射问题"）。决策（迁不迁、什么时候迁）在人。

2. **AI 能自动生成新系统的代码吗？**
   AI 能生成骨架代码（接口定义、数据模型、CRUD），但业务逻辑（订单状态机、库存扣减规则）必须人写——这些是老系统多年积累的隐性知识，AI 从代码里读不全。AI 辅助起草，人补全业务逻辑。

3. **AI 怎么帮对账？**
   AI 做异常归因——对账 diff 出现时，AI 分析 diff 模式（哪些字段不一致、哪些订单类型集中），匹配历史 diff 案例库，给出候选解释（"疑似金额舍入差异"）。AI 不自动修复（数据修复必须人确认），只做归因建议。

4. **绞杀者模式接入 LLM 做迁移知识库，怎么设计？**
   把老系统的业务规则、数据字典、历史 bug、迁移复盘做成 RAG 知识库。新人迁移新领域时提问"这个接口有哪些坑"，AI 检索返回历史案例。关键：知识库要持续更新（每次迁移复盘入库），不能是一次性整理。

5. **怎么用 AI 自动判断"老系统可以下线了"？**
   AI 监控老系统的流量、依赖、对账 diff，综合判断：流量连续 N 天为零、无其他服务依赖、对账无差异、回滚预案就绪。AI 给出"可下线"置信度，人最终决策（下线是半不可逆动作，谨慎）。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"Facade 路由、五步迁移、CDC 对账、可回滚、自然枯萎"** 五个词。

- **Facade 路由**：统一入口，内部按功能路由到新/旧系统
- **五步迁移**：双写→影子读→灰度切流→全量→下线
- **CDC 对账**：Debezium/Canal 实时同步 + 每小时对账 diff 监控
- **可回滚**：每步可秒级切回老系统（网关改路由）
- **自然枯萎**：老系统无流量后下线，不是物理删除

### 拟人化理解

把绞杀者模式想成 **绞杀榕**（名字来源）。热带雨林里，绞杀榕的种子落在老树枝上，新根慢慢环绕老树生长，抢夺阳光和养分。老树还在（业务不停），新树逐步取代（功能迁移）。最终老树枯萎倒下，新树独立成林。关键是"渐进取代"——不是一刀砍了老树（业务崩溃），而是新树长大才让老树退场。

### 面试现场 60 秒回答

> 绞杀者模式是改造遗留系统的安全方法——不一次性重写（Joel Spolsky 说重写是最大错误），而是引入 Facade 网关统一入口，按功能路由到新/旧系统。新功能直接在新系统实现，旧功能按领域优先级迁移（先读后写、先边界清晰后核心链路）。每个迁移单元走五步：双写（CDC 实时同步老 DB 到新 DB）→ 影子读（异步对比结果）→ 灰度切流（10%→50%→100%）→ 全量 → 老系统下线。每步可回滚（网关秒级切回老系统）。数据一致性靠对账——每小时跑 diff，>0.1% 暂停切流，>1% 自动回滚。终态是老系统连续 30 天零流量后自然下线，不是物理删除。这个模式把"重写项目"的风险从"全有全无"降到"功能级独立赌博"，所以比大重写安全得多。

### 现场对话示例

**面试官**：绞杀者模式听起来好，但迁移要 1-2 年，业务等不了怎么办？

**候选人**：迁移周期长不是问题，关键是"每季度有可见产出"。绞杀者模式不是"1-2 年后才有结果"，而是"每季度迁一个领域"——Q1 迁订单查询（业务立刻看到 P99 提升）、Q2 迁库存、Q3 迁下单主链路。每个领域迁移完都有独立收益，业务方看得见。如果业务说"下单链路最痛，要先迁"，那就调整优先级（虽然核心链路风险高，但可以先迁周边功能积累经验，再碰核心）。关键是和业务对齐优先级，不是技术方拍脑袋定顺序。

**面试官**：双写期间数据不一致怎么办？

**候选人**：双写一致性是对账兜底的。三个机制：(1) CDC 实时同步——老 DB 写入后毫秒级同步到新 DB，窗口期极短；(2) 每小时对账——扫描最近 1 小时变更的订单，对比新老 DB，diff 告警；(3) 自动修复——diff 出现时从老 DB 重同步到新 DB（老 DB 是 source of truth）。红线指标：diff 率 > 0.1% 暂停切流（查根因），> 1% 自动回滚（数据可能错乱）。迁移 1 年我们积累了 N 个 diff 根因案例（时区、舍入、NULL 处理），沉淀成 checklist，新领域迁移时提前规避。

**面试官**：老系统什么时候可以下线？

**候选人**：三个条件全满足才能下线：(1) 老系统该领域连续 30 天零流量（网关日志验证）；(2) 无其他服务依赖该接口（调用链分析）；(3) 对账无差异（数据已完全迁移）。下线也分步——先摘路由（网关不指向老系统），观察 1 周无问题；再关闭老系统接口（进程还在，接口不响应）；最后 3 个月后删代码（保留应急期）。每步都可回退，不一刀切。老系统"枯萎"是渐进的，不是物理删除。

## 常见考点

1. **绞杀者模式是什么？**——Martin Fowler 提出的遗留系统改造模式。引入 Facade 网关统一入口，按功能路由到新/旧系统，新功能直接在新系统实现，旧功能逐步迁移。名字来自绞杀榕（环绕老树生长，逐步取代）。核心是"用可逆的小步替换不可逆的大重写"。
2. **为什么不直接重写？**——Joel Spolsky 经典论证：重写丢失多年积累的 bug fix 和边界处理，新系统重新踩坑；重写期间老系统还在硬撑，两边都没维护好；重写 2-3 年，业务早变了。Netscape 重写 Navigator 失去市场是经典反例。
3. **迁移五步是什么？**——双写（CDC 同步老 DB 到新 DB）→ 影子读（异步对比新老读结果）→ 灰度切流（10%→50%→100%）→ 全量切新系统 → 老系统下线。每步可回滚（网关秒级切回老系统）。
4. **数据迁移怎么保证一致？**——CDC（Debezium/Canal）实时同步 binlog；双写期新老并行；每小时对账 diff 监控（>0.1% 暂停切流，>1% 自动回滚）；diff 自动修复（从老 DB 重同步）。对账是迁移期的生命线。
5. **Facade 路由怎么做？**——Spring Cloud Gateway/Nginx 按路径/Header 路由。如 `/api/order/query/**` 走新系统，`/api/v1/**` 走老系统。灰度通过 Header（`X-Migrate-Version: v2`）控制流量比例，10% 用户带 v2 走新系统。

## 结构化回答

**30 秒电梯演讲：** 绞杀者模式（Strangler Fig Pattern，Martin Fowler 提出）是改造遗留系统的安全方法——不一次性重写，而是用 Facade 路由渐进迁移：新功能在新系统实现，旧功能逐步替换，每步可回滚。名字来自热带的绞杀榕——新树环绕老树生长，逐步取代老树，最终老树枯萎。本质是用可逆的小步替换不可逆的大重写

**展开框架：**
1. **Facade 路由是核心** — 入口统一，内部按功能路由到新/旧系统
2. **渐进迁移** — 新功能在新系统，旧功能逐步替换，每步可回滚
3. **每个迁移单元要小** — 每个迁移单元要小（一个接口/一个领域），独立验证

**收尾：** 以上是我的整体思路。您想继续深入聊——为什么不直接重写？

# 【Java 后端架构师】压测模型、流量录制回放与容量评估

> 适用场景：JD 核心技术。618/双 11 大促前必须压测验证容量——下单链路能扛多少 QPS？峰值流量来了会不会挂？架构师必须能用四种压测模型找出容量拐点、用流量回放真实业务、用 Little's Law 算出集群容量、配置合理的限流和扩容预案。

## 一、概念层：四种压测模型

**四种压测模型的分工**：

| 模型 | 目的 | 方法 | 时长 | 关注指标 |
|------|------|------|------|---------|
| **基准压测（Baseline）** | 找单接口极限 | 单接口阶梯加压 | 5-10min | 单接口 QPS、P99 |
| **负载压测（Load）** | 找拐点 | 阶梯加压到性能突降 | 30-60min | 拐点 QPS、CPU、连接池 |
| **压力压测（Stress）** | 找崩溃点 | 持续超量直到崩溃 | 10-30min | 第一个崩的组件、错误率 |
| **稳定性压测（Endurance）** | 找内存泄漏 | 80% 容量长跑 | 8-24h | 内存趋势、GC、Full GC |

**单机拐点的判断**（核心技能）：

```
阶梯加压（每 2 分钟加 1000 QPS）：
  1000 QPS：P99 = 50ms,  CPU = 30%
  3000 QPS：P99 = 80ms,  CPU = 50%
  5000 QPS：P99 = 100ms, CPU = 70%
  7000 QPS：P99 = 150ms, CPU = 80%   ← 开始饱和
  8000 QPS：P99 = 500ms, CPU = 90%   ← P99 突增（5 倍）
  10000 QPS：P99 = 2000ms, 错误率 5%  ← 已崩溃

拐点 = 8000 QPS（P99 突增的点）
单机容量上限 = 拐点 × 0.8 = 6400 QPS（再留 20% 余量）
集群容量 = 6400 × 10 台 = 6.4万 QPS
限流阈值 = 6.4万 × 0.7 = 4.48万 QPS（再留 30% 余量）
```

**Little's Law 验证**：

```
已知：单请求处理时间 W = 100ms
拐点：8000 QPS（λ）
验证并发数：L = λ × W = 8000 × 0.1 = 800

实际配置：10 台 × 单机线程池 80 = 800 并发
✓ 与 Little's Law 吻合，容量评估合理
```

## 二、机制层：流量录制回放

**流量录制回放 vs 手写脚本对比**：

| 维度 | 流量回放 | 手写脚本 |
|------|---------|---------|
| 真实度 | 高（真实参数、热点 key、调用链） | 低（手造数据分布失真） |
| 维护成本 | 低（自动录制） | 高（业务变更要改脚本） |
| 业务覆盖 | 全（录制所有请求） | 部分（只测写的接口） |
| 合规风险 | 高（含真实用户数据，要脱敏） | 低（人造数据） |

**GoReplay 流量录制**（生产侧）：

```bash
# 在生产机器上启动 GoReplay，监听网卡流量
gor --input-raw :8080 \                    # 监听 8080 端口
    --output-file requests.gor \            # 录制到文件
    --http-allow-method GET \               # 只录 GET（避免写请求污染）
    --output-file-flush-interval 10s

# 录制 5 分钟大促流量
# 输出格式（.gor 文件）：
#   1 timestamp
#   GET /product/123 HTTP/1.1
#   Host: api.jd.com
#   ...
```

**流量改写**（避免冲突）：

```bash
# 回放时改写（避免与生产冲突）
gor --input-file requests.gor \
    --output-http "http://stress-test-env:8080" \  # 回放到压测环境
    --http-modify-header "X-Pressure-Test: true" \  # 加压测标
    --http-modify-url "s/\/product\/(\d+)/\/product\/pt_$1/" \  # ID 加前缀
    --http-set-header "X-Trace-Id: pt_$(uuid)" \    # 新 traceId
    --output-http-workers 100 \                      # 并发数
    --output-http-stats                              # 输出统计

# 敏感字段脱敏（用 middleware）
# 手机号 13812345678 → 138****5678
# 身份证 → 脱敏
# 真实用户 ID → pt_原始ID（避免与生产冲突）
```

**TCPCopy（内核级流量复制）**：

```bash
# TCPCopy 复制 TCP 包，更底层（不依赖应用协议）
# 适合非 HTTP 协议（如数据库、MQ）

# 在线上机器（tcpcopy 客户端）
tcpcopy -x 8080-压测环境IP:8080 -l /var/log/tcpcopy.log

# 在压测环境（intercept 进程，回送 ACK）
intercept -i eth0 -F tcp and src port 8080 -d
```

## 三、机制层：全链路压测架构

**JD 全链路压测架构**（架构师必须能画）：

```
压测平台
  │
  ├── 流量录制（GoReplay/TCPCopy）
  │     生产机器 ──录制──> 请求文件（脱敏改写）
  │
  ├── 压测引擎（JMeter/Gatling/自研）
  │     压测平台 ──发起──> 压测流量（带压测标 X-Pressure-Test: true）
  │
  └── 全链路透传
        网关 ──识别压测标──> 透传到所有下游
        服务 ──识别压测标──> 写影子库表（不污染生产）
        MQ ──识别压测标──> 写影子 topic
        缓存 ──识别压测标──> 写影子 key（前缀 pt_）

压测数据隔离：
  生产 DB：orders 表（真实数据）
  影子 DB：orders_pt 表（压测数据，独立表空间或独立库）

  生产 Redis：key = user:123
  影子 Redis：key = pt_user:123（前缀隔离）

  生产 MQ：topic = order_events
  影子 MQ：topic = order_events_pt
```

**全链路压测标透传代码**：

```java
// 网关识别压测流量
@Component
public class PressureTestFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String ptFlag = exchange.getRequest().getHeaders().getFirst("X-Pressure-Test");
        if ("true".equals(ptFlag)) {
            // 标记为压测流量，透传到所有下游
            PressureTestContext.markAsPressureTest();
            ServerHttpRequest req = exchange.getRequest().mutate()
                .header("X-Pressure-Test", "true")
                .build();
            return chain.filter(exchange.mutate().request(req).build());
        }
        return chain.filter(exchange);
    }
}

// 服务识别压测标，路由到影子资源
@Service
public class OrderService {

    public Order createOrder(OrderDTO dto) {
        if (PressureTestContext.isPressureTest()) {
            // 压测流量：写影子表 + 影子 MQ + 影子缓存
            return createOrderShadow(dto);
        }
        return createOrderReal(dto);
    }

    private Order createOrderShadow(OrderDTO dto) {
        Order order = shadowOrderDao.insert(dto);   // 影子表 orders_pt
        shadowMqProducer.send("order_events_pt", event);   // 影子 topic
        shadowRedis.set("pt_user:" + dto.getUserId(), data);  // 影子 key
        return order;
    }
}
```

## 四、实战层：容量评估全流程

**JD 大促容量评估 SOP**（必背）：

```
Step 1：流量预估
  历史大促峰值 × 今年增长系数 × 余量
  例：去年峰值 50万 QPS × 1.2（增长）× 1.3（余量）= 78万 QPS

Step 2：单机容量压测
  阶梯加压找拐点
  单机上限 = P99 拐点 × 0.8
  例：单机拐点 1.2万 QPS → 上限 9600 QPS

Step 3：集群容量计算
  集群容量 = 单机上限 × 机器数
  78万 / 9600 ≈ 82 台机器（按可用区分散）

Step 4：配置限流和预案
  集群限流 = 集群容量 × 0.7 = 55万 QPS
  超过 55万 拒绝（保已接收请求体验）
  扩容预案：峰值超 55万 时 10 分钟扩 20 台（容量 + 14万）

Step 5：全链路压测验证
  录制大促流量回放，压到 78万 QPS
  验证：成功率 > 99.9%、P99 < SLA、无雪崩、无 OOM

Step 6：稳定性压测
  80% 容量（62万 QPS）持续 8 小时
  验证：内存平稳、GC 稳定、无 Full GC、无连接泄漏
```

**JMeter/Gatling 压测脚本**：

```scala
// Gatling 阶梯加压（Scala DSL）
val httpProtocol = http
  .baseUrl("http://stress-test-env:8080")
  .header("X-Pressure-Test", "true")   // 压测标

val scn = scenario("Order Placement")
  .exec(http("place_order")
    .post("/order")
    .body(StringBody("""{"userId":"${userId}","amount":99.5}"""))
    .check(status.is(200)))

setUp(
  scn.inject(
    // 阶梯加压找拐点
    constantUsersPerSec(1000).during(120),    // 1千 QPS 2 分钟
    constantUsersPerSec(3000).during(120),    // 3千
    constantUsersPerSec(5000).during(120),    // 5千
    constantUsersPerSec(7000).during(120),    // 7千
    constantUsersPerSec(8000).during(120),    // 8千（看 P99）
    constantUsersPerSec(10000).during(120)    // 1万（看崩溃）
  ).protocols(httpProtocol)
    .throttle(
      reachRps(10000).in(60),    // 1 分钟内加到 1万 RPS
      holdFor(300)                // 保持 5 分钟
    )
)
```

**容量评估报表**（压测产出）：

```
| 服务        | 单机拐点 | 单机上限(×0.8) | 实例数 | 集群容量 | 限流阈值(×0.7) |
|------------|---------|----------------|--------|----------|----------------|
| 网关        | 3万     | 2.4万          | 30     | 72万     | 50万           |
| 订单服务    | 1.2万   | 9600           | 82     | 78万     | 55万           |
| 库存服务    | 1.5万   | 1.2万          | 40     | 48万     | 34万           |
| DB（主库）  | -       | 5万 QPS        | 主从   | -        | -              |

瓶颈分析：
  - 库存服务集群容量 48万 < 订单服务 78万
  - 大促时库存先满，限流要从订单层提前拦截
  - 解决：库存扩到 60 台（容量 72万）或库存异步化
```

## 五、实战层：压测避坑指南

**常见压测陷阱**：

```
陷阱 1：JMeter 客户端先崩
  现象：压测机 CPU 100%，被测服务还好
  原因：单台 JMeter 扛不住高压测流量
  解法：分布式压测（多台 JMeter 控制台+agent）或用 Gatling（异步 IO 更省资源）

陷阱 2：缓存命中率虚高
  现象：压测 QPS 很高，上线后实际低
  原因：压测数据重复（同一个 userId 反复查），缓存命中 100%
  解法：压测数据要分散（模拟真实用户分布，缓存命中率 70-80%）

陷阱 3：连接池打满但不报错
  现象：QPS 上不去，但日志无错
  原因：DB 连接池满，请求排队等待连接（表现为 RT 高而非错误）
  解法：监控连接池活跃数（hikaricp_connections_active），RT 飙升先查连接池

陷阱 4：GC 停顿被平均
  现象：平均 RT 低，但偶发超时
  原因：GC 停顿几秒，平均 RT 看不出来
  解法：看 P99/P99.9 而非平均，监控 GC 停顿分布

陷阱 5：写压测污染生产
  现象：压测后生产多了奇怪订单
  原因：写压测未隔离，直接写生产库
  解法：写压测必须影子库表 + 压测标透传 + MQ 影子 topic
```

## 六、底层本质：为什么是压测而非容量计算公式

回到第一性：**系统容量无法用公式精确算，必须压测量化**。

**为什么公式算不准**：系统容量受多个变量影响——CPU 核数、内存、磁盘 IO、网络、JVM GC、DB 连接池、下游依赖。这些变量的组合复杂（如 GC 停顿与对象分配速率相关、连接池满与下游 RT 相关），无法用简单公式表达。Little's Law 给理论估算（L=λW），但实际拐点必须压测找（GC、连接池等瓶颈点公式算不出）。

**为什么流量回放比手写脚本真**：手写脚本的参数分布与真实业务不符（如手造的 userId 均匀分布，但真实业务有热点——大商家订单集中）。这导致缓存命中率、热点 key 冲突、DB 分片倾斜都与真实不符。流量回放录真实请求，参数分布、调用链、热点 key 都真实，压测结果更可信。

**为什么读压测可在生产、写压测必须隔离**：读操作幂等不改变状态，在生产压读不影响数据正确性（只是多消耗资源）。写操作改变状态（INSERT/UPDATE），在生产压写会污染数据（多出测试订单、扣错库存）。所以写压测必须影子库表（独立表空间）+ 压测标透传（全链路识别并路由到影子资源）。

**为什么 0.7 系数而非 1.0**：压测环境与生产有差异（压测时无 GC 抖动、无慢查询突发、无网络毛刺），生产实际容量比压测低。留 30% 余量（× 0.7）应对这些突发。设 1.0 限流等于压测峰值，生产遇到突发（GC 3 秒）就会雪崩。

## 七、AI 架构师加问：5 个 AI 相关问题

1. **AI 推理服务的压测怎么做？**
   压测要模拟真实 prompt 分布（长 prompt + 短 prompt 混合）、并发请求对 GPU 显存的压力、批处理（batch）对吞吐的影响。关键指标：tokens/sec（吞吐）、P99 首字延迟（流式）、显存峰值、GPU 利用率。AI 压测比传统服务复杂（请求是非均匀的，prompt 长度影响吞吐）。

2. **让 AI 自动分析压测瓶颈，AI 接管哪段？**
   AI 解析压测数据（CPU、RT、GC、连接池、慢 SQL）、识别瓶颈（CPU 瓶颈、IO 瓶颈、连接池瓶颈、下游瓶颈）。AI 出优化建议（扩容、调连接池、加缓存、异步化），人工 review。比人工分析快，但建议要验证（AI 可能漏看依赖关系）。

3. **AI Agent 链路压测怎么设计？**
   Agent 链路长（用户 → AI → 工具 → 结果），压测要覆盖全链路。模拟不同任务类型（简单查询、复杂推理、工具调用密集）、不同工具调用模式（单工具、多工具串行、多工具并行）。监控每个环节的 RT 和错误率，定位最慢的环节。

4. **AI 预测容量怎么做？**
   AI 用历史大促数据 + 营销活动节奏预测峰值 QPS、峰值时间、热点商品。比人工估算准（考虑更多变量），但要做置信区间（AI 可能低估）。预测输出峰值预估区间，架构师按上界配容量。

5. **AI 推理的弹性扩容怎么结合压测？**
   压测找单 GPU 容量（如单卡 100 QPS），按峰值预测算 GPU 数（如 1万 QPS 需 100 卡）。弹性扩容基于实时指标——GPU 利用率 > 80% 自动扩、< 30% 自动缩。GPU 扩容慢（加载模型要分钟级），要预留预热时间，不能等满了才扩。

## 八、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"四压测模型、P99 找拐点、流量回放真、写压测隔离、容量 ×0.7"**。

- **四模型**：基准（单接口极限）、负载（阶梯找拐点）、压力（找崩溃点）、稳定性（8h 长跑验泄漏）
- **拐点判断**：P99 突增、CPU > 80%、连接池打满，对应 QPS 即容量上限
- **流量回放**：GoReplay/TCPCopy 录制 + 改写（时间戳/IDs/traceId）+ 脱敏
- **写压测隔离**：影子库表 + 压测标透传 + 影子 MQ topic + 影子 Redis key
- **容量公式**：单机拐点 × 0.8 × 机器数 × 0.7 = 限流阈值

### 拟人化理解

把压测想成 **JD 大促前仓库压力测试**。基准压测是"测单个分拣员极限速度"（单接口）、负载压测是"逐步加包裹量找传送带拐点"（阶梯加压看 P99 突增）、压力压测是"超量灌入看哪里先崩"（找瓶颈组件）、稳定性压测是"持续 8 小时高负载看员工会不会累垮"（验内存泄漏和 GC）。流量录制回放是"录昨天真实订单今天重放"（比手造假订单更真，参数分布、热点 key 都真实）。容量评估是"测出每个分拣员能扛 100 单/分钟，10 个分拣员 = 1000 单/分钟，留 30% 余量配 700 单/分钟的接单上限"。

### 面试现场 60 秒回答

> 压测四模型：基准找单接口极限、负载阶梯加压找拐点、压力超量找崩溃点、稳定性 8 小时长跑验内存泄漏。单机拐点看 P99 突增 + CPU 80% + 连接池满，按 0.8 折扣得单机容量。集群容量 = 单机容量 × 机器数，限流阈值再 × 0.7 留余量。流量回放用 GoReplay 录制生产请求，改写时间戳和 IDs 避免冲突，脱敏敏感字段。写压测必须隔离——影子库表（orders_pt）+ 压测标透传（X-Pressure-Test: true）+ 影子 MQ topic + 影子 Redis key 前缀。Little's Law 验证：L = λW，1万 QPS × 100ms = 1000 并发，配 10 台 × 100 线程吻合。压测陷阱：JMeter 客户端先崩（用分布式）、缓存命中率虚高（数据要分散）、连接池满不报错（监控活跃数）。

### 现场对话示例

**面试官**：你怎么找单机拐点？

**候选人**：阶梯加压。从 1000 QPS 开始，每 2 分钟加 1000，监控三个指标：P99 延迟、CPU 利用率、DB 连接池活跃数。拐点的特征是 P99 突增（如从 100ms 突增到 500ms，5 倍跳跃），同时 CPU 从线性增长变平（已饱和），连接池活跃数接近上限（排队等连接）。这个拐点 QPS 就是单机容量上限的参考。再按 0.8 折扣得实际配置容量——拐点 8000 QPS，配 6400 QPS 单机上限。10 台集群 = 6.4万 QPS，限流阈值再 × 0.7 = 4.48万。这套数字背后是阶梯压测的真实数据，不是拍脑袋。

**面试官**：流量回放和手写脚本哪个好？

**候选人**：流量回放更真。手写脚本的问题：①参数分布失真（手造 userId 均匀分布，真实业务有热点——大商家订单集中）；②业务覆盖不全（只测写的接口，漏掉边界接口）；③维护成本高（业务变更要改脚本）。流量回放录真实请求，参数分布、调用链、热点 key 都真实，压测结果更可信。但流量回放要处理：①脱敏（用户数据合规，手机号身份证脱敏）；②改写（时间戳改当前、ID 加压测前缀避免冲突）；③写请求隔离（回放写流量必须路由到影子库表）。工具上读压测用 GoReplay（应用层）或 TCPCopy（内核层），写压测必须配合影子环境。

**面试官**：写压测怎么不污染生产？

**候选人**：三层隔离。第一，影子库表——压测数据写到 orders_pt 表（独立表空间或独立库），与生产 orders 表物理隔离。第二，压测标透传——网关识别 X-Pressure-Test: true 头，全链路透传到所有服务，每个服务识别后路由到影子资源。第三，影子中间件——MQ 写影子 topic（order_events_pt）、Redis 写影子 key（pt_user:123 前缀）、ES 写影子索引。这样压测流量从入口到存储全程与生产数据物理隔离，绝不交叉。验证方法：压测后查生产 orders 表，不应有任何 pt_ 前缀的压测订单。

## 常见考点

1. **四种压测模型？**——基准（单接口极限）、负载（阶梯找拐点）、压力（找崩溃点）、稳定性（8h 长跑验泄漏）。
2. **单机拐点怎么找？**——阶梯加压，看 P99 突增 + CPU 饱和 + 连接池满。按 0.8 折扣得单机容量。
3. **流量回放怎么做？**——GoReplay/TCPCopy 录制生产请求，改写时间戳/IDs/traceId，脱敏敏感字段，回放到压测环境。
4. **写压测怎么隔离？**——影子库表（orders_pt）+ 压测标透传（X-Pressure-Test）+ 影子 MQ topic + 影子 Redis key 前缀。
5. **容量怎么算？**——单机拐点 × 0.8 × 机器数 × 0.7 = 限流阈值。Little's Law 验证：L = λW。

## 结构化回答

**30 秒电梯演讲：** 压测是用模拟流量找出系统容量的物理上限。压测模型分基准/负载/压力/稳定性四类；流量录制回放（录制真实请求 + 修改时间戳/IDs 回放）比手写脚本更接近真实业务；容量评估用 Little's Law（L=λW）+ 单机拐点 + 集群数算出可承载 QPS，再按 0.7 系数配限流

**展开框架：**
1. **压测四模型** — 基准（单接口极限）、负载（阶梯找拐点）、压力（超量找瓶颈）、稳定性（长时间）
2. **单机拐点** — P99 突增、CPU > 80%、连接池打满，对应 QPS 即容量上限
3. **流量回放** — 录制生产请求（GoReplay/TCPCopy）→ 改写时间戳/IDs → 回放到压测环境

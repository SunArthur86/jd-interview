# 【Java 后端架构师】G1、ZGC、Shenandoah 如何做低延迟选型

> 适用场景：JD 核心技术。交易/风控/推荐链路堆动辄几十上百 G，一次 200ms 的 STW 就是线上故障。低延迟 GC 不是炫技，是金融级 SLA 的硬约束。

## 一、概念层：三大低延迟 GC 的核心机制对比

| 维度 | G1 | ZGC | Shenandoah |
|------|----|----|-----------|
| **默认版本** | JDK 9+ | JDK 21 分代 GA（JDK 15 生产可用） | JDK 12+（Red Hat / OpenJDK） |
| **堆分区** | Region（1-32MB，约 2048 个） | Region（ZPage，2MB/32MB 动态） | Region（与 G1 类似） |
| **停顿来源** | 疏散阶段 STW（软目标可预测） | 仅根扫描快照 STW，**与堆无关** | 仅根扫描 STW，**与堆无关** |
| **并发化技术** | 并发标记 + 并发疏散（部分 STW） | **染色指针 + 读屏障** | **Brooks 转发指针 + 读/写屏障** |
| **停顿量级** | 50-200ms | **<1ms（百 G 堆）** | **<10ms** |
| **吞吐代价** | 基线 | -5~15% | -5~10% |
| **堆开销** | 无额外 | 无额外（染色指针） | +5-10%（转发指针） |
| **典型场景** | 通用低延迟 | 超低延迟 + 大堆 | OpenJDK 生态低延迟 |

**染色指针（ZGC 核心，面试必画）**：

```
64 位指针布局（JDK 21 分代 ZGC）：
┌────────────────────────────────────┬──────┬──────┬──────┐
│          可用地址位（对象地址）        │ Final│ Remap│ Mark │
│                                    │ 0/1  │ 0/1  │ 0/1  │
└────────────────────────────────────┴──────┴──────┴──────┘
                                      ↑      ↑      ↑
                                   重定位  转发表  标记位
                                   完成?   查询?   存活?

业务线程访问引用 → 读屏障检查染色位 → 若 Mark0/Remap0 触发自愈 → 透明返回正确地址
```

**转发指针（Shenandoah Brooks）**：

```
对象头
┌──────────────┐
│ forwardedPtr │ ──▶ 指向自身（未移动）/ 新副本（正在移动）
├──────────────┤
│  实际数据     │
└──────────────┘
每次访问多一次间接跳转，但实现简单、不依赖平台特性
```

## 二、机制层：ZGC 一次并发回收的工作流

```
业务线程 ──────────────────────────────────────────────►
  │   │  STW1     │并发标记   │STW2    │并发疏散+重定位 │并发重映射
  │   │  初始标记  │          │再标记   │              │
  ▼   ▼           ▼          ▼        ▼              ▼
停顿 <1ms        ~10ms       <1ms    并发（业务照跑）  并发
```

关键点（架构师必须答得出来）：

1. **STW1 / STW2 只做根扫描的快照**，时间只跟 GC Roots 数量（线程数、栈帧深度）有关，与堆大小无关。
2. **并发标记**遍历对象图，用染色指针的 Mark0/Mark1 位记录存活。
3. **并发疏散**：选定 Region，把存活对象拷贝到新 Region，更新转发表。
4. **读屏障自愈**：业务线程下次访问旧引用，读屏障发现 Remap 位为 0，查转发表返回新地址并回写——业务线程无感。
5. **并发重映射**：修正剩余旧引用（懒执行，合并到下次标记）。

JDK 21 **分代 ZGC** 的突破：把堆切成 Young（分代） + Old，Young GC 只扫新生代，分配速率高场景吞吐提升 10%+，停顿仍 < 1ms。

## 三、实战层：选型决策树与参数

**选型决策树**：

```
堆大小 ?    P99 目标 ?
  │            │
  ▼            ▼
< 8G  ─────────► G1 足够（停顿 < 100ms 容易达成）
8-32G ─────────► G1 + 精调，或 Shenandoah（如果 OpenJDK）
> 32G 或 P99 < 10ms ─► ZGC（JDK 21+ 分代）
任何堆 + 旧 JDK ────► G1 是兜底
```

**G1 低延迟参数**（交易服务常用）：

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100            # 软目标，G1 据此选 Region 数
-XX:G1HeapRegionSize=16m            # 大堆调大，减少 Region 数
-XX:InitiatingHeapOccupancyPercent=35  # 提前并发标记（默认45）
-XX:G1ReservePercent=15             # 预留防疏散失败
-XX:G1NewSizePercent=20             # 新生代下限，避免过小频繁 YGC
-XX:ConcGCThreads=4                 # 并发标记线程，一般 = CPU/8
-XX:ParallelGCThreads=16            # STW 阶段线程，一般 = CPU/4
```

**ZGC 参数**（JDK 21 大堆低延迟）：

```bash
-XX:+UseZGC
-XX:+UseZGenerational               # JDK 21+ 分代 ZGC
-XX:SoftMaxHeapSize=28g             # 软上限，超了才主动 GC
-XX:ZUncommitDelay=300s             # 内存归还 OS 延迟
-XX:ConcGCThreads=8                 # 并发回收线程
# ZGC 不接受 MaxGCPauseMillis——停顿本就和堆无关
```

**验证选型是否成立的压测命令**：

```bash
# JFR 看 GC 事件分布（JDK 自带）
jcmd <pid> JFR.start name=gc duration=5m settings=profile filename=gc.jfr
# jfr print --events jdk.GCPhasePause gc.jfr  → 看每次 STW 毫秒数

# GC 日志
-XX:+UseZGC -Xlog:gc*:file=gc.log:time,level,tags
# 关注 "Pause" 行：ZGC 的 Pause Mark Start 0.012ms（亚毫秒）
```

**真实场景案例**：风控决策服务，堆 64G，G1 下 Mixed GC 偶发 400ms 停顿导致 P99 抖到 800ms。切 ZGC（JDK 21 分代）后，停顿稳定 < 2ms，P99 降到 30ms，吞吐下降约 8%（用 CPU 余量换的）。代价是 CPU 从 60% 升到 72%。

## 四、底层本质：为什么 ZGC 能做到与堆无关

回到第一性原理：**传统 GC 停顿 = 根扫描时间 + 对象移动时间 + 引用修正时间**。后两项都要求 STW，且和存活对象量成正比，于是堆越大停顿越长。

ZGC 的本质突破是用**染色指针把引用修正摊到并发阶段**：

- 每个引用在 64 位地址里留出几个 bit 记录"这个对象是否正在被标记/移动"。
- 业务线程访问引用时，**读屏障**（JVM 内联的几条指令）检查染色位，发现对象已移动就查转发表、修正引用、返回新地址——这一步对业务透明。
- 于是"移动对象"和"修正引用"都不需要 STW，STW 只剩下"根扫描快照"——这只跟 GC Roots 数量有关，跟堆大小无关。

这就是低延迟 GC 的第一性原理：**用每次访问多花几条指令（读屏障税），换取停顿从 O(堆) 降到 O(根)**。当堆足够大、P99 要求足够严，这笔交易就划算；否则 G1 的吞吐优势更香。

## 五、AI 架构师加问：5 个 AI 相关问题

1. **让 AI 推荐低延迟 GC 参数，AI 接管哪段？**
   AI 接管 GC 日志解析、分配速率画像、历史相似服务参数推荐；真正改 JVM 参数必须人工审批——一次错误的 IHOP 可能让并发标记跟不上触发退化的 Full GC。

2. **怎么评测 GC Copilot 的效果？**
   沉淀黄金样本：历史 GC 事故的 jstat/JFR 快照 + 专家修复动作。离线 eval：推荐参数是否复现专家路径；在线灰度：单实例对比 `jfr jdk.GCPhasePause` 的 P99、`old_gc_count`、`allocation_rate_mb_s`。

3. **AI 调用 jcmd/JFR 怎么控制风险？**
   命令白名单（`JFR.start`/`GC.class_histogram` 只读免审批，`GC.run` 写操作需审批）；JFR 录制设大小上限防磁盘打满；所有 tool_call 记录 pid + 操作人 + 回滚参数。

4. **GC 知识库做 RAG 怎么分版本？**
   按 JDK 版本（8/17/21）和 GC 类型（G1/ZGC/Shenandoah）打标——JDK 21 的分代 ZGC 参数和 JDK 15 单代完全不同，检索前先按运行时版本过滤，避免把单代参数推荐给分代场景。

5. **怎么防止 AI 误推荐低延迟 GC？**
   输出必须带代价说明（吞吐 -X%、CPU +Y%）；高吞吐批处理服务即使堆大也优先 G1/Parallel，AI 要根据服务画像（吞吐 vs 延迟）拒答或给出 G1 备选；监控 fallback_rate 和人工接管率。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"染色指针、转发指针、决策树、读屏障税、验证指标"** 五个词。

- **染色指针**：ZGC 把标记/重定位位编码进引用本身
- **转发指针**：Shenandoah 用对象头多一个指针指向新副本
- **决策树**：堆<32G 选 G1，堆大或 P99<10ms 选 ZGC
- **读屏障税**：低延迟 GC 的代价是每次访问多几条指令
- **验证指标**：`jfr jdk.GCPhasePause` + `allocation_rate_mb_s`

### 拟人化理解

把 GC 当**高速公路养护队**。G1 是"半夜封一条道施工"——影响可控但停顿看路段。ZGC 是"开着导航的养护车"——给每辆车（引用）装 GPS 标签（染色指针），车开过施工点时导航自动绕行（读屏障自愈），全程不封路。

### 面试现场 60 秒回答

> 选型不看品牌看 P99 目标和堆规模。堆 32G 以内、P99 容忍 200ms，G1 足够，调好 MaxGCPauseMillis 和 IHOP。堆几十上百 G、P99 要 10ms 以内，上 ZGC（JDK 21 分代），代价是吞吐降 5-15%、CPU 多用 20%——这笔账要和业务算清楚。Shenandoah 在 OpenJDK 生态里是 ZGC 的替代，机制是 Brooks 转发指针，停顿接近但多占 5-10% 堆。验证不是靠感觉，是 `jfr --events jdk.GCPhasePause` 看每次停顿是否真 < 1ms，`allocation_rate_mb_s` 看并发标记是否跟得上分配速率。最容易翻车的是切了 ZGC 没留 CPU 余量，并发线程抢业务 CPU 反而更慢。

### 反问面试官

> 贵司这条链路是吞吐优先（离线/批处理）还是延迟优先（交易/风控）？如果是吞吐优先我会坚持 G1 或 Parallel；如果是延迟优先、堆又大，我会推 ZGC 但先做 CPU 预算评估。

### 现场对话示例

**面试官**：你说 ZGC 停顿 < 1ms，那是怎么做到的？

**候选人**：核心是染色指针把引用修正并发化。传统 GC 移动对象要 STW 修所有引用，ZGC 在 64 位指针里留几个 bit 记录对象的标记和重定位状态，业务线程访问引用时读屏障检查染色位，发现对象已移动就查转发表自愈。所以 STW 只剩根扫描快照，只跟 GC Roots 数量有关，跟堆大小无关——这是停顿与堆解耦的根本原因。

**面试官**：那代价是什么？

**候选人**：读屏障税。每次指针解引用多几条指令，吞吐降 5-15%，CPU 多用 20%+。所以选型要算 CPU 预算：如果机器 CPU 已经 80%，切 ZGC 会把业务线程挤慢。另一个代价是 JDK 版本要求，分代 ZGC 要 JDK 21+，老版本只能用单代，分配速率高场景吞吐上不去。

**面试官**：上线后你怎么验证？

**候选人**：不是看"感觉快了"。我用 `jcmd pid JFR.start settings=profile` 录 5 分钟，`jfr print --events jdk.GCPhasePause` 看每次 STW 毫秒数，确认 P99 真的 < 1ms。同时看 `allocation_rate_mb_s` 是否压垮并发回收——如果分配速率太高，并发标记会跟不上，堆会持续增长最终触发退化 Full GC。这个我会提前压测，定一个分配速率上限告警。

## 常见考点

1. **G1 和 CMS 区别？**——G1 基于 Region 化堆 + 混合回收，可预测停顿；CMS 基于分代 + 标记清除有碎片，JDK 14 已移除。新项目只选 G1/ZGC。
2. **ZGC 为什么 JDK 21 才分代？**——早期 ZGC 单代，并发标记成本随存活对象线性增长；分代后 Young GC 只扫新生代，高分配速率场景吞吐提升 10%+，停顿仍 < 1ms。
3. **Shenandoah 和 ZGC 谁更好？**——停顿接近，机制不同：Shenandoah 用 Brooks 转发指针（多占 5-10% 堆，不依赖平台），ZGC 用染色指针（不占堆但需 64 位地址空间 + OS 支持）。ZGC 生态更主流。
4. **低延迟 GC 下还会 Full GC 吗？**——理论上 ZGC 无 Full GC，但极端情况（分配速率远超回收速率、并发失败）会退化为 STW 串行回收，此时几乎等于 OOM。监控 `allocation_rate_mb_s` 是关键。


## 结构化回答

**30 秒电梯演讲：** 聊到G1、ZGC、Shenandoah 如何做低延迟选型，我的理解是——低延迟 GC 的本质是把"对象移动"和"业务线程执行"并发化——读屏障/写屏障接管引用修正，让 STW 只剩下几次极短的初始标记和再标记，从而把"停顿"从"跟堆大小成正比"变成"跟堆大小无关"。打个比方，像搬家：G1 是"分区小搬家公司"挑值钱的房间先搬；ZGC 是"染色搬家法"——给每个家具贴颜色标签，搬家工人和你在房间里走动同时进行，靠标签识别哪些家具正在搬。你在屋里几乎感觉不到停顿。

**展开框架：**
1. **G1** — Region 化堆 + 混合回收，停顿目标软可预测，JDK 9+ 默认
2. **ZGC** — 染色指针 + 读屏障，停顿 < 1ms 且与堆无关，JDK 21 Generational GA
3. **Shenandoah** — Brooks 转发指针 + 并发 evac，停顿与堆无关

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：ZGC 为什么 JDK 21 才分代？您更想看哪个方向？

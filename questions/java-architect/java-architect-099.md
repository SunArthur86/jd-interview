# 【Java 后端架构师】从单体到微服务的演进路线

> 适用场景：JD 核心技术。一个电商系统从最初的 PHP 单体起步，5 年后变成 50 万行 Java 代码的巨型单体，30 人团队在一个代码库里协作，发版互相阻塞，大促时一个慢功能拖垮全站。架构师面临的问题是：要不要拆微服务？怎么拆？拆成什么样？什么时候拆？这是架构师最常遇到的演进决策。

## 一、概念层：演进路径与拆分触发点

### 架构演进的四个阶段

```
阶段 1：单体          阶段 2：模块化单体     阶段 3：服务化        阶段 4：微服务
（all-in-one）        （代码解耦）          （核心拆分）          （全面服务化）
┌──────────┐         ┌──────────┐         ┌────┬────┬────┐    ┌──┬──┬──┬──┬──┐
│ Order    │         │ Order    │         │Order│Pay │Inv │    │O │P│I│U│S│
│ Payment  │  →      │ Payment  │   →     │(独立)│(独立)│(独立)│ → │...每个独立...
│ Inventory│         │ Inventory│         │ 共享DB       │    │  各自独立DB  │
│ User     │         │ User     │         └──────────────┘    └──────────────┘
│ Search   │         │ Search   │         一个进程多服务       多进程多服务多库
└──────────┘         └──────────┘
一个进程一个库        一个进程一个库
团队 < 5人            团队 5-15人           团队 15-30人         团队 > 30人
```

**核心原则**：不是"越往后越好"，是"匹配当前规模"。小团队用微服务是过度设计（分布式复杂度 > 收益），大团队用单体是协作灾难。

### 拆分触发点（什么时候拆）

| 触发点 | 单体的痛 | 微服务的解 | 判断标准 |
|--------|---------|-----------|---------|
| **团队协作瓶颈** | 30 人改一个代码库，冲突多、发版阻塞 | 按服务分团队，独立开发部署 | 团队 > 15 人，发版等待 > 2 天 |
| **独立扩容需求** | 大促全链路扩容，非热点也跟着扩 | 只扩热点服务（如订单不扩搜索） | 部分模块 QPS 是其他 10 倍 |
| **故障隔离需求** | 一个慢功能打满线程池全站挂 | 故障隔离在单服务内 | 发生过"局部故障拖垮全局"事故 |
| **技术栈异构需求** | 全栈必须统一（Java） | 各服务可选合适技术（如搜索用 Go） | 有明确的异构收益 |
| **独立迭代需求** | 业务线发版节奏不同被迫对齐 | 各服务独立发版 | 业务线发版频率差异 > 3 倍 |

**不拆的理由**（保持单体）：团队小、业务稳定、流量低、没有上述痛点。不要"为了微服务而微服务"。

## 二、机制层：按 DDD 拆服务与数据演进

### 2.1 按业务能力拆分（DDD 领域驱动）

**正确拆法**：按业务领域（Bounded Context）拆。

```
电商系统按 DDD 拆分：

订单域（Order Context）        支付域（Payment Context）
- 创建订单                     - 发起支付
- 订单状态流转                 - 支付回调处理
- 订单查询                     - 退款

库存域（Inventory Context）    用户域（User Context）
- 扣减/回滚库存                - 用户信息
- 库存查询                     - 用户鉴权

营销域（Marketing Context）    搜索域（Search Context）
- 促销/优惠券                  - 商品搜索
- 价格计算                     - 推荐排序

每个域 = 一个微服务（或一组微服务）
域内高内聚，域间通过明确的接口通信
```

**错误拆法**：按技术层拆（OrderController 一个服务、OrderService 一个服务、OrderDao 一个服务）。这会造成服务间调用爆炸（一个请求跨 3 个服务），且没有业务意义。

### 2.2 数据演进（拆分最难的部分）

```
数据拆分三步走（降低一致性风险）：

步骤 1：代码解耦，共享数据库（过渡期）
┌─────────────┐ ┌─────────────┐
│ Order Svc   │ │ Payment Svc │
│ 只操作      │ │ 只操作      │
│ order 表    │ │ payment 表  │
└──────┬──────┘ └──────┬──────┘
       │                │
       ▼                ▼
  ┌─────────────────────────┐
  │    共享数据库（monolith） │   ← 还是一个库，但代码边界清晰
  │  order | payment | inv   │
  └─────────────────────────┘
  价值：代码解耦，为物理拆分做准备，业务无感知

步骤 2：数据库物理拆分（按服务归属）
┌─────────────┐ ┌─────────────┐
│ Order Svc   │ │ Payment Svc │
└──────┬──────┘ └──────┬──────┘
       ▼                ▼
  ┌──────────┐     ┌──────────┐
  │ Order DB │     │Payment DB│   ← 各自独立库
  └──────────┘     └──────────┘
  难点：跨库查询（如订单 JOIN 用户）、跨库事务
  解法：跨库查询用数据冗余/CQRS/数据聚合服务
       跨库事务用 saga/消息最终一致

步骤 3：按需分库分表（单服务数据量大时）
  Order DB → 按 userId 分 16 库 × 8 表 = 128 个分片
  价值：单服务内部水平扩展
```

## 三、实战层：strangler fig 渐进拆分

### 拆分节奏（以订单系统为例）

```markdown
# 微服务拆分演进路线（6 个月）

## 阶段 1（M1）：模块化单体（代码解耦）
- 单体内按 DDD 划分模块（order/payment/inventory 包）
- 模块间通过接口调用（不直接依赖实现类）
- 数据库表按模块归属打标签（order_xxx/payment_xxx）
- 价值：代码边界清晰，为拆分做准备，零业务风险

## 阶段 2（M2-M3）：拆第一个服务（试点，低风险）
- 选择：搜索服务（非核心链路、读为主、故障可降级）
- 用 strangler fig：新搜索服务部署，网关灰度切流
- 数据同步：MySQL → ES（CDC 或定时同步）
- 价值：验证拆分方法（部署/监控/运维），建立信心

## 阶段 3（M4-M5）：拆核心链路（订单/支付/库存）
- 核心：同步 RPC（订单→库存→支付）+ 异步 MQ（履约/通知）
- 数据：各自独立库，跨服务查询用数据冗余或聚合 API
- 一致性：saga + 对账任务兜底
- 灰度：5% → 50% → 100%，每步对账 diff < 0.01%
- 价值：核心目标达成（独立扩容、故障隔离、独立发版）

## 阶段 4（M6）：下线旧单体 + 沉淀
- 旧单体保留只读 1 个月后下线
- 沉淀：拆分 SOP、服务模板（starter）、监控模板、接入规范
- 价值：团队能力提升，后续服务化可复用

## 贯穿全程：基础设施先行
- M1 之前：服务注册发现（Nacos）、配置中心、链路追踪、统一日志
- 没有基础设施就拆微服务 = 把单体的简单问题变成微服务的复杂问题
```

### strangler fig 灰度切流代码

```java
// 网关层灰度路由
@Component
public class OrderGatewayFilter {

    public OrderResponse route(OrderRequest req) {
        // 灰度：按用户/租户/百分比切流
        if (grayRouter.shouldRouteToMicroservice(req.getUserId(), "order-ms", 0.05)) {
            try {
                OrderResponse msResult = orderMicroservice.create(req);
                // 对账（灰度期双跑校验一致性）
                OrderResponse monolithResult = oldMonolith.query(req.getTempId());
                reconcileService.compare(msResult, monolithResult);
                return msResult;
            } catch (Exception e) {
                log.warn("微服务异常，降级单体", e);
                return oldMonolith.create(req);   // 降级
            }
        }
        return oldMonolith.create(req);
    }
}
```

## 四、底层本质：康威定律与分布式复杂度

### 康威定律（架构映射组织）

**"设计系统的组织，其架构会映射组织的沟通结构。"**

含义：如果 30 人在一个代码库里开发，无论怎么"设计架构"，最终都会变成一个耦合的巨型单体（因为人都在互相沟通、互相依赖）。要拆微服务，必须同时调整组织结构——按服务边界划分团队（订单团队、支付团队、库存团队），团队边界和服务边界对齐。

**反向康威定律**：如果你想强制某种架构，先调整组织结构。想让订单和支付解耦？先把订单团队和支付团队拆开（物理隔离沟通），架构自然会解耦。只拆代码不拆团队，微服务会退化回"分布式单体"（代码拆了但人还在一起改，依然耦合）。

### 分布式复杂度（微服务的代价）

微服务不是免费的午餐，它用**分布式复杂度**换取**独立扩展和故障隔离**：

| 复杂度 | 单体 | 微服务 | 应对 |
|--------|------|--------|------|
| 网络不可靠 | 无（进程内调用） | 有（RPC 超时/重试/熔断） | Resilience4j + 降级 |
| 数据一致性 | 单机事务 | 分布式事务（saga/TCC） | 最终一致 + 对账 |
| 服务发现 | 无（直接调用） | 需要（Nacos/Eureka） | 注册中心 |
| 配置管理 | 统一配置 | 分布式配置 | 配置中心（热更新） |
| 可观测性 | 单机日志 | 分布式链路 | OpenTelemetry + traceId |
| 部署运维 | 一个包 | N 个服务 | K8s + CI/CD |
| 故障排查 | 单机 dump | 跨服务 trace | 全链路追踪 |

**核心权衡**：微服务的收益（独立扩展/故障隔离/团队自治）是否大于代价（分布式复杂度 + 运维成本）。团队小、业务简单时，代价 > 收益，保持单体。团队大、业务复杂时，收益 > 代价，值得拆。

### 演进而非重构的第一性

**"演进式架构"**：架构应该随业务规模"长出来"，而非"设计出来"。种子（单体）适合幼苗期（小团队小业务），随着生长逐渐分叉（模块化→服务化→微服务）。跳过中间阶段（种子直接长成树）会因为根基不稳而倒塌——没有模块化基础就拆微服务，会拆出耦合的"分布式大泥球"。

strangler fig 模式的本质是"让新老系统并存，逐步替换"——这比"推翻重来"风险低得多（业务连续性保障），且每步可验证（发现问题及时调整）。演进式架构承认"我们无法完美预测未来"，所以保持每步可回退的灵活性。

## 五、AI 架构师加问：5 个 AI 场景的架构演进

1. **AI 能力接入单体还是独立服务？**
   AI 能力（LLM/RAG/Agent）建议独立服务（AI Gateway + AI Services），不嵌入业务单体。原因：(1) AI 调用慢且贵（秒级延迟、token 成本），和业务单体混在一起会拖垮主链路；(2) AI 能力迭代快（模型/prompt 升级），独立服务可单独迭代不影响业务；(3) AI 能力可复用（多个业务调用同一个 AI Gateway）。

2. **AI 系统从单体到微服务有什么特殊考量？**
   AI 服务的拆分维度：按模型能力（LLM 服务/embedding 服务/RAG 服务/Agent 服务分开）。特殊考量：(1) GPU 资源不能像 CPU 那样无限水平扩（贵且稀缺），要按 GPU 容量规划；(2) 模型版本管理（A/B/灰度不同模型版本）；(3) AI 服务的 SLO 不同（延迟秒级，不是百毫秒级）。

3. **RAG 服务怎么和业务单体协同？**
   RAG 作为独立检索服务，业务单体通过 RPC 调用。接口：retrieve(query, userContext) → chunks。权限过滤在 RAG 服务内做（pre-filter），业务单体传 userContext。知识库索引更新走异步（CDC），RAG 服务内部维护索引新鲜度。

4. **Agent 系统的微服务边界怎么划？**
   Agent 本身是一个服务（Agent Executor），但它调用的工具（function）可能是其他服务。边界：Agent 编排逻辑独立（Agent Service），工具是各业务服务的 API（Order Service 的 queryOrder、Payment Service 的 refund）。Agent 通过 RPC/MQ 调用工具，和微服务架构天然契合——每个工具调用就是一个跨服务 RPC。

5. **AI 平台（MLOps）的架构演进？**
   AI 平台演进：单体（一个 notebook 跑所有）→ 拆分（数据pipeline/训练服务/推理服务/评测服务独立）→ 平台化（统一 AI Gateway + 模型仓库 + 评测平台 + 监控）。拆分驱动：模型迭代频率高（训练/推理要解耦）、多团队共用（需要平台化）、成本控制（GPU 资源统一调度）。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"触发点、DDD、数据三步、strangler fig、康威定律"** 五个词。

- **触发点**：团队协作/独立扩容/故障隔离/技术异构/独立迭代，有痛点才拆
- **DDD**：按业务能力（领域边界）拆，不按技术层拆
- **数据三步**：代码解耦（共享库）→ 库拆分（独立库）→ 表拆分（分库分表）
- **strangler fig**：新老并存 + 灰度切流 + 保留回滚，渐进非重构
- **康威定律**：架构映射组织，拆代码同时拆团队

### 面试现场 60 秒回答

> 从单体到微服务是渐进演进，不是一次性重构。第一步判断要不要拆——看触发点：团队协作瓶颈（30 人一个代码库发版阻塞）、独立扩容需求（部分服务 QPS 是其他 10 倍）、故障隔离需求（局部故障拖垮全局）。没有这些痛点保持单体，不为微服务而微服务。第二步怎么拆——按 DDD 业务能力拆（订单域/支付域/库存域），不按技术层拆。先做模块化单体（代码解耦，零风险），再拆试点服务（非核心，验证方法），最后拆核心链路。第三步数据最难——三步走：代码解耦（共享库，只操作自己的表）→ 库拆分（各服务独立库）→ 表拆分（分库分表）。跨库一致性用 saga + 对账兜底。第四步用 strangler fig 模式——新老并存、灰度切流（5%→100%）、保留回滚，业务不中断。第五步基础设施先行——注册发现/配置中心/链路追踪/统一日志在拆分前就绪，否则把简单问题变成复杂问题。核心：康威定律，拆代码同时拆团队，团队边界和服务边界对齐。

### 反问面试官

> 贵司当前是单体、模块化单体，还是已经微服务化了？团队规模多大？这决定我是推拆分、推模块化、还是优化现有微服务治理。

## 七、苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 微服务有那么多好处，为什么不一开始就微服务？ | 微服务有分布式复杂度代价（网络/一致性/运维），初期业务不确定、团队小、流量低时，单体更简单高效。演进式架构：随规模长出来。证明：Netflix/Amazon 也是从单体起步逐步拆的，没有一开始就微服务 |
| 证据追问 | 你怎么知道拆微服务真的解决了问题？ | 对比拆分前后的量化指标：发版频率（应升）、故障隔离（单服务故障不扩散）、独立扩容（大促只扩核心）、团队协作效率（跨团队冲突减少）。如果拆了没改善，说明拆分维度错或没配套组织调整 |
| 边界追问 | 微服务解决不了什么？ | 解决不了代码质量（差代码拆成微服务还是差）、解决不了业务逻辑复杂（复杂度只是转移）、解决不了团队管理问题（人不行换架构也没用）。微服务解决的是"规模带来的协作和扩展瓶颈"，不是银弹 |
| 反例追问 | 什么情况你会把微服务合并回单体？ | 服务拆太细运维成本爆炸（N 个服务的监控/部署/故障）、服务间调用链路过长（一个请求跨 10 个服务延迟高）、团队规模缩小（小团队维护 N 个服务负担重）。合并同质的细粒度服务（如把订单查询/创建/取消合回一个订单服务）|
| 风险追问 | 拆分过程中最大的风险？ | 数据一致性问题（拆库后跨库事务处理不当导致数据错乱）。兜底：对账任务（持续比对数据差异）+ 灰度切流（发现不一致立即回滚）+ saga 补偿（每步可回滚）。第二风险是基础设施不就绪就拆（没有服务发现/链路追踪，拆完一团乱麻）|
| 验证追问 | 怎么证明拆分是成功的？ | 全链路压测（各服务 P99 达标、整体链路无瓶颈）、灰度数据一致性（对账 diff < 0.01%）、业务无感知（无客诉增加）、运维就绪（监控/告警/部署/回滚全链路通）。四项达标才算拆分成功 |
| 沉淀追问 | 拆分后沉淀什么？ | 服务拆分 SOP（从模块化到独立的步骤）、服务模板 starter（新服务快速创建）、监控/部署/告警模板、领域模型文档（DDD 边界）、微服务接入规范、数据一致性对账框架 |

### 现场对话示例

**面试官**：你说按业务能力拆，但订单要查用户信息、查商品信息，这些跨域查询怎么办？

**候选人**：跨域查询有三种模式。第一，数据冗余——订单服务保存必要的用户信息快照（userId、昵称、手机号脱敏），不实时查用户服务。订单创建时从用户服务拉一次，后续用冗余数据。适合：对实时性要求不高。第二，API 聚合——订单服务调用用户服务/商品服务的 API 拼装结果。适合：需要实时数据但量不大。配合熔断降级（用户服务挂了返回缓存的或默认的）。第三，CQRS/数据聚合层——专门建一个聚合查询服务，订阅各域的数据变更（CDC），维护一个聚合视图。适合：复杂查询场景（如订单详情页要展示订单+用户+商品+物流）。选哪种看实时性要求和查询复杂度。核心是避免跨域 JOIN（微服务的反模式）。

**面试官**：拆分后服务间调用链路变长，延迟上升怎么办？

**候选人**：三个手段。第一，减少同步调用——能异步的用 MQ（如订单创建后通知营销/物流，不需要同步等结果），核心链路才同步。第二，并行化——独立的调用并行发（如同时查用户和查商品，而非串行），CompletableFuture 编排。第三，缓存——跨服务查询的结果缓存（用户信息缓存 5 分钟，减少 RPC）。监控链路 P99，超 SLO 就优化。如果优化后仍不达标，重新评估服务边界——可能拆得太细，考虑合并同域服务。

**面试官**：团队只有 8 个人，适合微服务吗？

**候选人**：大概率不适合。8 人团队维护微服务的运维成本（N 个服务的部署/监控/故障排查/版本兼容）会拖垮开发效率。建议保持模块化单体——代码按模块解耦（清晰的包结构 + 模块间接口调用），但部署仍是一个应用。这样既有代码层面的清晰边界（未来拆分容易），又没有微服务的分布式复杂度。等团队长到 15-20 人、业务复杂度上升、出现明确的扩容/隔离需求时，再拆。康威定律：8 人团队的服务边界就是团队本身——一个服务一个团队。

## 常见考点

1. **微服务和 SOA 区别？**——SOA（面向服务架构）用 ESB（企业服务总线）做集成，服务间通过 ESB 通信，偏重企业级集成。微服务去中心化，服务间点对点通信（RPC/MQ），强调独立部署和轻量通信。微服务是 SOA 的精细化实践，去掉了 ESB 这个中心。
2. **康威定律是什么？**——"设计系统的组织，其架构会映射组织的沟通结构。"含义：架构和组织结构对应。团队按服务边界划分（订单团队管订单服务），团队边界 = 服务边界。反向应用：想改变架构先改变组织结构。
3. **怎么定微服务的边界？**——按 DDD（领域驱动设计）的限界上下文（Bounded Context）拆。每个上下文对应一个业务领域（订单/支付/库存），领域内高内聚，领域间通过明确接口通信。不按技术层（Controller/Service/Dao）拆——那是分层不是分服务。
4. **strangler fig 模式是什么？**——渐进式替换旧系统的模式。新系统逐步替代旧系统：新功能在新系统开发，旧功能逐步迁移，旧系统被"绞杀"直至下线。比大爆炸式重构安全——新老并存、灰度切流、业务不中断。
5. **分布式事务怎么处理？**——优先避免跨服务事务（重新设计边界）。必须跨则：saga（最终一致，每步可补偿，适合大部分业务）、TCC（强一致，try-confirm-cancel，复杂但严格，适合资金）、本地消息表（最终一致，简单）。大部分场景最终一致 + 对账兜底够用，强一致是少数且成本高。

## 结构化回答

**30 秒电梯演讲：** 从单体到微服务不是一次性拆分，是按痛点驱动、按能力边界、按风险可控的渐进演进。拆分的触发点不是微服务时髦，而是单体出现了具体的扩展瓶颈（团队协作、独立扩容、故障隔离、技术异构）。拆分的核心难点不是技术（RPC/MQ/注册中心），是数据——如何拆分耦合的数据库且保证一致性

**展开框架：**
1. **演进路径** — 单体 → 模块化单体 → 服务化 → 微服务（不是一步到位）
2. **拆分触发点** — 团队协作瓶颈、独立扩容需求、故障隔离需求、技术栈异构需求
3. **拆分维度** — 按业务能力（领域驱动 DDD）拆，不按技术层（controller/service/dao）拆

**收尾：** 以上是我的整体思路。您想继续深入聊——微服务拆多细合适？

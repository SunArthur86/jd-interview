# 【Java 后端架构师】如何答好系统设计题（高分框架）

> 适用场景：JD 核心技术。系统设计面试（"设计一个秒杀系统""设计一个短链服务""设计一个朋友圈 Feed 流"）是架构师面试的核心环节。高分的关键不是"会做"，而是"按框架做"——结构化作答让面试官能跟着思路走、每一环都能打分。混乱作答（想到哪写到哪）= 低分。

## 一、概念层：6 步框架

### 1.1 框架总览

### 1.2 每步的核心问题与陷阱

| 步骤 | 核心问题 | 常见陷阱 |
|------|----------|----------|
| 需求澄清 | "这个系统要做什么？用户多少？延迟要求？" | 不问需求闷头做（假设错全盘错） |
| 容量估算 | "QPS 多少？存多少数据？带宽多少？" | 不估算直接说"用 Redis"（量级决定选型） |
| 接口设计 | "核心 API 有哪些？" | 只列接口不列参数/响应 |
| 数据模型 | "表怎么设计？怎么分片？缓存怎么用？" | 只画表不画索引/分片键 |
| 扩展性 | "怎么扛 10 倍流量？机房挂了怎么办？" | 只说"加机器"不说分片策略 |
| 权衡 | "为什么选 A 不选 B？代价是什么？" | 只给一个方案不讨论替代（显得没深度） |

## 二、机制层：以"设计一个秒杀系统"为例

### Step 1: 需求澄清

```
候选人：我先确认几个需求，确保理解一致。

功能需求：
1. 商品秒杀——用户在特定时间抢购限量商品
2. 下单——抢到后 15 分钟内支付，超时取消
3. 库存——每个 SKU 有限量（如 1000 件）

非功能需求：
1. 并发——秒杀瞬间流量是平时的 100 倍
2. 一致性——不能超卖（业务底线）
3. 可用性——秒杀期间不能挂

约束：
1. 用户量——假设 1 亿注册用户，活动峰值 100 万同时在线
2. 商品——单场秒杀 SKU 数 < 100
3. 库存——单 SKU 库存 100-10000 件

面试官，这个范围对吗？需要补充吗？
```

**关键**：每个假设都和面试官确认。不确认闷头做 = 偏离需求。

### Step 2: 容量估算

```
候选人：基于需求做容量估算。

QPS：
- 峰值同时在线 100 万
- 秒杀瞬间（开抢后 1 秒）假设 30% 用户点击 = 30 万 QPS
- 平时浏览商品详情约 1 万 QPS

存储：
- 订单数据：假设 10% 抢中 = 10 万订单/场
- 每订单 1KB，单场 100MB，年 1000 场 = 100GB/年
- 历史订单 5 年 = 500GB（单表扛不住，需分表）

带宽：
- 商品详情页 50KB，1 万 QPS 浏览 = 500MB/s 入口带宽
- 静态资源走 CDN，源站只承担动态请求

结论：
- 峰值 30 万 QPS → 单机扛不住（单机 Redis 约 10 万 QPS），需集群 + 分片
- 订单 500GB → 单表性能下降，需按 user_id 分表
```

**关键**：数字用于后续选型。30 万 QPS 决定"不能用单机"，500GB 决定"必须分表"。

### Step 3: 接口设计

```
候选人：核心接口列一下。

1. 商品详情（浏览）
   GET /api/v1/seckill/products/{skuId}
   Response: { skuId, name, price, stock, startTime, status }

2. 秒杀下单（核心）
   POST /api/v1/seckill/orders
   Body: { skuId, userId }
   Response: { orderId, status, payDeadline }
   限流：单用户 1 次/SKU，全局令牌桶

3. 查询订单状态
   GET /api/v1/orders/{orderId}
   Response: { orderId, skuId, status, payDeadline }

4. 支付回调
   POST /api/v1/orders/{orderId}/pay
   Body: { paymentId, amount }

5. 超时取消（定时任务/MQ 延迟）
   内部接口，不暴露

面试官，这些接口覆盖需求吗？
```

### Step 4: 数据模型

```
候选人：数据模型分三块——DB、缓存、消息队列。

【DB 表设计】

-- 库存表（按 sku_id 分表）
CREATE TABLE inventory (
  sku_id BIGINT PRIMARY KEY,
  total_stock INT,
  available_stock INT,
  locked_stock INT,  -- 下单未支付
  version INT,       -- 乐观锁
  INDEX idx_sku (sku_id)
);

-- 订单表（按 user_id 分表，避免单表热点）
CREATE TABLE seckill_order (
  order_id BIGINT PRIMARY KEY,  -- 雪花 ID
  user_id BIGINT,
  sku_id BIGINT,
  status TINYINT,  -- 0待支付 1已支付 2已取消
  create_time DATETIME,
  pay_deadline DATETIME,
  INDEX idx_user (user_id),
  INDEX idx_sku_status (sku_id, status)
) PARTITION BY HASH(user_id);

【Redis 缓存设计】

-- 库存预热（开抢前加载）
SET seckill:stock:{skuId} 1000  -- 原子扣减用 DECR

-- 用户去重（防刷）
SET seckill:user:{skuId}:{userId} 1 EX 3600

-- 订单 ID 预生成（避免 DB 生成成为瓶颈）
LIST seckill:orderId:pool:{skuId}

【消息队列】

Topic: seckill-order-created
- 分区：按 sku_id 分（同 SKU 顺序消费）
- 消费者：异步创建 DB 订单、发支付提醒
作用：削峰——Redis 扣减成功后发 MQ，DB 异步落库

【数据流向】

用户请求 → Redis 扣库存（DECR）→ 发 MQ → 异步落 DB 订单
                ↓
          扣减失败（库存≤0）→ 返回"已售罄"
```

### Step 5: 扩展性与高可用

```
候选人：画一下整体架构，讲扩展性和高可用。

┌──────────┐    ┌──────────┐    ┌───────────┐
│  CDN     │───▶│  Nginx   │───▶│ Gateway   │
│ 静态资源 │    │ 负载均衡 │    │ 限流/鉴权 │
└──────────┘    └──────────┘    └─────┬─────┘
                                      │
                                      ▼
                              ┌───────────────┐
                              │ SeckillService│  ← 应用层集群（可水平扩展）
                              │  (多实例)     │
                              └──┬─────────┬──┘
                                 │         │
                    ┌────────────▼──┐  ┌──▼──────────┐
                    │ Redis Cluster │  │ MQ (Kafka)  │
                    │ 库存扣减/去重  │  │ 削峰/异步    │
                    └───────────────┘  └──┬──────────┘
                                          │
                              ┌───────────▼───────────┐
                              │ OrderConsumer (异步)  │
                              │ 落库 DB / 发支付提醒   │
                              └───────────┬───────────┘
                                          │
                              ┌───────────▼───────────┐
                              │ MySQL 分库分表        │
                              │ (订单按 user_id 分)   │
                              └───────────────────────┘

扩展性：
- 应用层——无状态，水平扩展（加机器即可）
- Redis Cluster——分片，按 sku_id 分布库存
- MySQL——订单按 user_id 分表（32 库 32 表），避免单表热点

高可用：
- 多活——Redis/MySQL 主从，机房级故障切换
- 降级——Redis 挂了降级到 DB（牺牲性能保正确）
- 限流——Gateway 令牌桶，超量请求直接拒绝（保护后端）
- 熔断——下游（支付/库存）挂了快速失败，不堆积请求

监控：
- QPS/延迟/错误率（Prometheus + Grafana）
- 库存水位告警（库存 < 10% 预警）
- MQ 积压告警（消费延迟 > 1 分钟告警）
```

### Step 6: 权衡取舍

```
候选人：讨论几个关键权衡。

【权衡 1：库存扣减放 Redis 还是 DB？】
- Redis 扣减：性能高（10 万 QPS），但宕机会丢未落库的扣减（需持久化兜底）
- DB 扣减：强一致，但性能低（单机几千 QPS），扛不住 30 万峰值
- 选择：Redis 扣减 + MQ 异步落 DB。代价：极端情况（Redis 宕机）可能少量库存不一致，用对账补偿

【权衡 2：订单同步落库还是异步？】
- 同步：用户立即看到订单，但 DB 是瓶颈（30 万 QPS 打死 DB）
- 异步：用户先拿到"抢购成功"，订单稍后生成。代价：用户需轮询订单状态
- 选择：异步。用户体验稍差（延迟 1-2 秒看到订单），但保护 DB

【权衡 3：超卖容忍度】
- 强一致（CP）：全程 DB 锁，绝不超卖，但性能差
- 最终一致（AP）：Redis 扣减可能短暂超卖，对账修正
- 选择：业务底线是不超卖（用户投诉 + 赔付），选 CP——Redis 原子 DECR 保证不超卖，DB 对账兜底

【备选方案：如果用 DB 扛？】
- 分库分表 + 乐观锁，单 SKU 库存分片（如 1000 库存拆 10 片各 100）
- 优点：强一致、无 Redis 依赖
- 缺点：实现复杂、性能仍不如 Redis
- 适用：库存量级小（< 1 万）且强一致要求极高的场景

面试官，这些权衡合理吗？有想深入的方向吗？
```

## 三、实战层：答题技巧与避坑指南

### 3.1 高分技巧

| 技巧 | 说明 | 示例 |
|------|------|------|
| **主动确认** | 每步做完和面试官对齐 | "我估算 QPS 30 万，这个量级你认可吗？" |
| **量化思维** | 所有决策有数字支撑 | "30 万 QPS 单机扛不住（单机 Redis 10 万），所以集群" |
| **画图** | 架构图比文字清晰 | 边说边画数据流向、组件关系 |
| **权衡** | 没有完美方案，列优劣势 | "Redis 快但可能丢，DB 慢但强一致，选 Redis+对账" |
| **先框架后细节** | 先整体后局部 | 先画整体架构，再深入某个组件 |
| **承认不懂** | 诚实比硬编好 | "Kafka 我没深入用过，但根据理解是 X 用法，对吗？" |

### 3.2 常见失分点

| 失分点 | 描述 | 对策 |
|--------|------|------|
| **不问需求** | 假设错全盘错 | 前 3 分钟必须澄清需求 |
| **不算容量** | 选型无依据 | QPS/存储/带宽都要算 |
| **只给一个方案** | 显得没深度 | 至少讨论 2 个备选 + 权衡 |
| **追求完美** | 过度设计（小系统上 K8s） | 量级定方案，不过度 |
| **闷头做** | 不和面试官互动 | 每步确认，像协作而非独白 |
| **跳步骤** | 直接写代码不设计 | 严格走 6 步框架 |
| **不画图** | 纯口头描述 | 至少画一张整体架构图 |

## 四、底层本质：为什么框架比答案重要

**系统设计面试考的不是"最终方案对不对"，而是"思考过程是否结构化"**。面试官想看你：(1) 能不能把模糊需求（"设计秒杀系统"）拆解成具体问题（QPS 多少？库存多少？延迟要求？）；(2) 能不能量化（30 万 QPS 而不是"很高"）；(3) 能不能权衡（Redis 快但可能丢，DB 慢但强一致）。这些能力是架构师的核心——日常工作就是把模糊业务需求变成具体技术方案。

**为什么每步要确认**：闷头做 30 分钟发现理解错需求 = 全盘重来 = 0 分。架构师日常工作也是——接到需求先和 PM 确认（"这个 QPS 是峰值还是平均？""延迟要求 100ms 还是 1s？"），确认后再设计。面试官用"需求澄清"环节考察这个习惯。

**为什么量化重要**："QPS 很高用 Redis" vs "峰值 30 万 QPS，单机 Redis 10 万，需 3 节点集群 + 分片"——后者展示工程 sense。架构师日常工作都是量化的（容量规划/扩容决策/成本估算），不会量化 = 拍脑袋决策。

**为什么权衡是加分项**：没有完美方案——Redis 快但可能丢数据，DB 强一致但慢。架构师的价值就是在约束下做最优权衡。只给一个方案不讨论替代 = 显得"只会背答案"。讨论 2-3 个备选 + 权衡 = 展示"能独立思考"。

## 五、AI 架构师加问：5 个

1. **AI 时代系统设计题会变吗？**
   会。LLM 能生成代码后，"怎么写"贬值，"为什么这么设计"升值。面试官会更看重权衡取舍、容量估算、边界场景——这些 AI 难以替代的判断力。答题要更突出"为什么"而非"怎么做"。

2. **AI 辅助画架构图会影响评分吗？**
   面试中通常手画（白板/纸），用 AI 画反而减分（看不出你的思考）。但日常工作中 AI 画图（PlantUML/Mermaid/Structurizr）提效——口头描述需求，AI 生成图，人改细节。

3. **AI 能替你做系统设计吗？**
   能做初稿（"设计秒杀系统"AI 给出完整方案），但不能替代判断——业务约束（"我们预算有限""合规要求 X"）需要人输入，权衡取舍（"接受 0.1% 超卖换 10 倍性能"）需要人决策。AI 是初稿工具，人是决策者。

4. **LLM 系统本身的设计题（如设计 ChatGPT）怎么答？**
   框架不变，但数据模型换——向量库（embedding 存储）、RAG（检索增强）、token 缓存、模型路由（小模型兜底/大模型精答）。扩展性考虑 GPU 资源调度、推理 batching。权衡：延迟（流式 vs 等全量）vs 成本（GPU 贵）。

5. **面试官用 AI 出题/判卷，你怎么应对？**
   AI 判卷看关键词覆盖（QPS/缓存/分库分表/限流），所以框架的 6 步都要走全（关键词不漏）。但 AI 难判"思考深度"——所以权衡取舍要讲透（不只是提一句"考虑过 X"，要分析优劣）。人机协同判卷下，既要关键词全又要深度够。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"需求、估算、接口、数据、扩展、权衡"** 六个词。

- **需求**：澄清功能/非功能/约束（前 3 分钟）
- **估算**：QPS/存储/带宽量化（数字定选型）
- **接口**：REST API 列出（路径/参数/响应）
- **数据**：表/缓存/MQ 设计（分片键/索引）
- **扩展**：分库分表/多活/限流（画架构图）
- **权衡**：CAP/成本/复杂度对比（讨论备选）

每步做完和面试官确认再下一步。

### 拟人化理解

把系统设计面试想成 **盖房子**。不能直接砌墙（写代码），要先问"盖几层？住几人？预算多少？地震带吗？"（需求/估算），再画图纸（接口/数据模型），再选结构（扩展性/高可用），最后权衡"用钢结构贵但快，用混凝土便宜但慢"（取舍）。跳步 = 返工。面试官是甲方——每步和他对齐，盖完发现理解错 = 推倒重来。

### 面试现场 60 秒回答

> 系统设计题我用 6 步框架。第一步需求澄清——问清功能/用户量/延迟要求，和面试官确认范围。第二步容量估算——算 QPS/存储/带宽，数字用于选型（30 万 QPS 决定不能用单机）。第三步接口设计——列 REST API（路径/参数/响应）。第四步数据模型——表结构/分片键/Redis key/MQ topic。第五步扩展性与高可用——分库分表/读写分离/多活/限流降级，画整体架构图。第六步权衡取舍——CAP/成本/复杂度对比，讨论 2-3 个备选方案。每步做完和面试官确认再下一步。重点在数据模型和扩展性（占 50% 时间），这是架构师核心。追求"结构化 + 量化 + 权衡"，不追求"唯一正确答案"。

### 反问面试官

> 贵司系统设计面试更看重什么——是方案完整性（6 步走全）还是深度（某个点深挖）？这决定我答题侧重——完整性优先就均匀分配时间，深度优先就挑一个点（如分片策略）深讲。

### 现场对话示例

**面试官**：你刚才说 Redis 扣库存，如果 Redis 挂了呢？

**候选人**：三个层次应对。第一层高可用——Redis Cluster 主从切换，单节点故障自动迁移，不影响整体。第二层降级——如果整个集群挂了（机房级故障），降级到 DB 扣减（乐观锁），牺牲性能保正确——此时 QPS 降到几千，用限流（令牌桶）控制入口流量，避免打死 DB。第三层兜底——Redis 持久化（AOF + RBD），重启后从磁盘恢复未落库的扣减。最坏情况（Redis 挂且持久化丢）用对账补偿——定时任务对 Redis 库存和 DB 订单做对账，发现不一致（Redis 多扣）则回补库存。业务上接受"极端情况秒杀暂停 10 分钟（降级切换）"，但不接受"超卖"。

**面试官**：你说分库分表按 user_id 分，那按 sku_id 查所有订单怎么办？

**候选人**：这是分库分表的经典问题——分片键选择。我选 user_id 是因为写热点分散（不同用户的订单分布在不同库）。按 sku_id 查订单是次要场景（运营看某商品卖了多少），两种解法：(1) 异构索引——写订单时同时写一份按 sku_id 分片的索引表（冗余存储），查询走索引表；(2) 离线汇总——订单落库后发 MQ，消费者按 sku_id 汇总到统计表（如每 SKU 每小时销量），查询走统计表。推荐方案 2——实时性要求不高（秒杀结束后看数据），避免双写一致性问题。如果一定要实时，用方案 1 但要处理双写一致（分布式事务/对账）。

**面试官**：时间快到了，你觉得还有什么没说到？

**候选人**：回顾一下 checklist——需求/估算/接口/数据/扩展/权衡都覆盖了。可能还可以补充的是监控告警（QPS/延迟/库存水位/MQ 积压的阈值）和预案（超卖怎么办、Redis 挂了怎么办的 Runbook）。如果还有时间，可以深入讲讲秒杀的防刷策略（验证码/设备指纹/黑名单）。您觉得哪个方向值得展开？

## 常见考点

1. **系统设计题的 6 步框架是什么？**——需求澄清（功能/非功能/约束）→容量估算（QPS/存储/带宽）→接口设计（REST API）→数据模型（表/缓存/MQ）→扩展性与高可用（分库分表/多活/限流）→权衡取舍（CAP/成本/复杂度）。每步做完和面试官确认再下一步。
2. **需求澄清要问什么？**——功能需求（做什么）、非功能需求（QPS/延迟/可用性/一致性）、约束（用户量/数据量/预算/合规）。前 3 分钟必须澄清，否则闷头做发现理解错 = 全盘错。
3. **容量估算怎么算？**——从用户量推导：用户数 × 日活率 × 人均操作 = 日操作数；日操作数 ÷ 86400 = 平均 QPS；峰值 QPS = 平均 × 5-10 倍。存储：日增量 × 天数。带宽：QPS × 响应大小。数量级对就行。
4. **权衡取舍怎么答？**——列出 2-3 个备选方案，对比优劣势（性能/一致性/成本/复杂度），说明选择理由。例：Redis 扣库存快但可能丢，DB 扣库存强一致但慢，选 Redis+对账——业务接受最终一致换性能。
5. **常见失分点是什么？**——(1) 不问需求闷头做；(2) 不算容量直接选型；(3) 只给一个方案不讨论替代；(4) 过度设计（小系统上 K8s）；(5) 闷头做不互动；(6) 跳步骤直接写代码；(7) 不画图纯口头描述。

## 结构化回答

**30 秒电梯演讲：** 系统设计题高分的关键不是会做，而是按框架做。混乱作答（想到哪写到哪）= 面试官无法评估。高分框架是 6 步：需求澄清→容量估算→接口设计→数据模型→扩展性与高可用→权衡取舍。每一步都要做完再说下一步，且每一步都和面试官确认（我估算 QPS 是 1 万，这个量级你认可吗？）。不确认闷头做 = 偏离需求

**展开框架：**
1. **6 步框架** — 需求澄清→估算→接口→数据模型→扩展性→权衡
2. **每步确认** — 做完一步和面试官对齐，不闷头做
3. **量化思维** — QPS/存储/带宽都要有数字（数量级估算）

**收尾：** 以上是我的整体思路。您想继续深入聊——面试官打断问细节怎么办？

  problem: 当单库单表数据量到亿级、TPS 到万级时，如何拆分数据让系统继续可扩展？

  - 单机磁盘/连接/TPS 有物理上限，无法无限扩
  - 大部分业务请求带用户维度（查某用户的订单），可按用户分片
  - 跨分片查询代价高（要聚合多库结果），要尽量单分片查询
  rebuild: 按业务维度分片——用户数据（订单/地址）按 user_id 哈希分库分表（同一用户数据在一个分片，避免跨库）。路由算法用 hash(user_id) % N（N 是分片数）。扩容用倍数翻倍（2→4→8，N 翻倍时只需迁移一半数据，路由算法 hash % N 不变）。全局 ID 用雪花算法（64 位：时间戳+机器号+序列号），趋势递增利于 B+ 树插入，全局唯一不依赖 DB。跨分片查询用广播表（小表如配置广播到所有分片）或绑定表（主表和从表按同一维度分片，JOIN 不跨库）。

  - 分库分表后主键怎么生成？——用全局唯一 ID。雪花算法（时间戳 41bit + 机器号 10bit + 序列号 12bit，每秒 400 万 ID，趋势递增）。号段模式（DB 批量发号，每次取一段缓存）。不推荐 UUID（无序导致 B+ 树页分裂频繁，插入性能差）
  - 分库分表后怎么做跨库 JOIN？——避免跨库 JOIN（设计时让相关表按同维度分片，绑定表）。必须跨库时：业务层聚合（分别查再合并）、广播表（小表冗余到所有库）、Elasticsearch 宽表（异构索引）。生产尽量设计成分片内完整查询
  - 扩容怎么做？——倍数扩容（2 库→4 库）。2 库时 hash(user_id) % 2，扩到 4 库时 hash(user_id) % 4。原来在库 0 的数据（hash%2=0），扩容后一半到库 0（hash%4=0）一半到库 2（hash%4=2），只需迁移一半。如果原来分片数是 2 的幂，扩容翻倍数据迁移量最小（50%）
  - 一致性哈希是什么？——把哈希值空间组织成虚拟环，节点和 key 都映射到环上，key 顺时针找最近的节点。扩容时只影响新节点附近的一小段 key（迁移量少），不命中用虚拟节点解决数据倾斜。Redis Cluster 用的不是一致性哈希是哈希槽（16384 槽）
  - 分库分表中间件选型？——ShardingSphere（Apache 顶级项目，JDBC/Proxy 两种模式）、MyCat（代理模式，已少用）。JDBC 模式轻量（应用直连 DB），Proxy 模式独立部署（应用连 Proxy）

# 【Java 后端架构师】分库分表路由、扩容与全局 ID

> 适用场景：JD 核心技术。订单表年增 50 亿行，单库存不下、查询慢、连接打满。架构师必须能设计分片策略、规划扩容路径、选全局 ID 方案——这是亿级数据系统的必经之路。

## 一、概念层：为什么要分库分表

**单机瓶颈与对应方案**：

| 瓶颈 | 表现 | 方案 |
|------|------|------|
| **磁盘容量** | 单表 5000 万行后磁盘紧张 | 分表（水平拆分） |
| **连接数** | 单库连接数几千上限，业务多了不够用 | 分库（多实例） |
| **写入 TPS** | 单机写入几千 TPS 上限 | 分库（写分散） |
| **单表性能** | 千万行后索引退化、查询慢 | 分表（单表行数控制） |
| **锁冲突** | 热点行锁竞争 | 分表（热点分散） |

**分库 vs 分表**：

| 维度 | 分库 | 分表 |
|------|------|------|
| 解决问题 | 连接数、TPS、IO | 单表数据量、查询性能 |
| 数据位置 | 不同数据库实例 | 同库或不同库的不同表 |
| 典型场景 | 业务高并发 | 数据量大 |

**垂直拆分 vs 水平拆分**：

```
垂直拆分（按字段/模块）：
  订单库：order, order_item
  商品库：product, sku
  用户库：user, address
  ── 按业务域拆，每个库独立部署

水平拆分（按行）：
  orders_0: user_id % 4 == 0 的订单
  orders_1: user_id % 4 == 1 的订单
  orders_2: user_id % 4 == 2 的订单
  orders_3: user_id % 4 == 3 的订单
  ── 按维度拆，分散到多个表/库

通常先垂直（按业务域拆库），再水平（单库数据量大再分表）
```

## 二、机制层：路由策略

**路由维度选择**（必考）：

| 维度 | 路由方式 | 适用 | 问题 |
|------|---------|------|------|
| **用户维度（user_id）** | hash(user_id) % N | C 端业务（查我的订单） | 不同用户数据量不均 |
| **时间维度** | 按月/天分表 | 日志、流水 | 热点集中在最新表 |
| **地理维度** | 按地区分库 | 多地部署 | 跨地查询慢 |
| **订单维度** | hash(order_id) % N | 查订单详情 | 查某用户所有订单要跨库 |

**用户维度路由示例**（最常用）：

```java
// 16 库 × 4 表 = 64 个分片
// 路由：先算库号，再算表号
public ShardLocation route(Long userId) {
    int hash = userId.hashCode();
    int dbIndex = hash % 16;           // 0-15 库
    int tableIndex = (hash / 16) % 4;  // 0-3 表
    return new ShardLocation("db_" + dbIndex, "orders_" + tableIndex);
}

// 查询：SELECT * FROM orders WHERE user_id = 12345
// ── 路由到 db_9.orders_2，单分片查询，高效
```

**跨分片查询的处理**（必考）：

```sql
-- 场景：查某用户所有订单（按 user_id 分片，单分片查询）
SELECT * FROM orders WHERE user_id = 12345;
-- 只查一个分片，高效

-- 问题场景：查某个商品的所有买家（跨所有分片）
SELECT * FROM orders WHERE product_id = 67890;
-- 不带 user_id，要广播到所有 64 个分片查询，再合并结果（慢）

-- 解决方案 1：绑定表（主表和从表按同维度分片）
-- orders 按 user_id 分片，order_item 也按 user_id 分片（冗余 user_id 字段）
-- JOIN 时两个表在同一分片，不跨库
SELECT * FROM orders o JOIN order_item oi ON o.id = oi.order_id
WHERE o.user_id = 12345;
-- 同分片 JOIN，高效

-- 解决方案 2：广播表（小表冗余到所有分片）
-- product 表（商品信息）数据量小，冗余到所有库
-- JOIN 时本地有 product 数据，不跨库

-- 解决方案 3：ES 宽表（异构索引）
-- 把订单数据同步到 ES，支持任意维度查询
-- DB 负责强一致写入，ES 负责复杂查询
```

## 三、机制层：扩容策略

**倍数扩容**（画图必考）：

```
初始：2 库 4 表（db_0, db_1，各 2 表）
  user_id hash % 2：
    hash=0 → db_0
    hash=1 → db_1

扩容到 4 库 8 表：
  user_id hash % 4：
    hash=0 → db_0（原 db_0 的一半，hash%4=0 的）
    hash=1 → db_1（原 db_1 的一半，hash%4=1 的）
    hash=2 → db_2（原 db_0 的另一半，hash%4=2 的）← 从 db_0 迁移
    hash=3 → db_3（原 db_1 的另一半，hash%4=3 的）← 从 db_1 迁移

迁移量：每个旧库迁移 50% 数据到新库
路由算法：hash % N，N 从 2 变 4，算法不变只改模数

关键：分片数必须是 2 的幂（2,4,8,16...），扩容翻倍迁移量最少（50%）
  ── 如果从 3 库扩到 5 库，迁移量会更大（要重新分布）
```

**扩容步骤**（生产流程）：

```
1. 准备新库：部署 4 个新库实例（db_0', db_1', db_2, db_3）
2. 双写：应用同时写新旧库（新数据两边都有）
3. 数据迁移：把旧库的历史数据按新路由迁移到新库
   ── 用数据同步工具（如 DataX、Canal）按 hash 分批迁移
4. 数据校验：对比新旧库数据量、抽样数据一致性
5. 切读：应用读切到新库（灰度，先切 10% 流量验证）
6. 停旧库：确认新库稳定后，停旧库写，下线旧库

整个过程：双写 + 迁移 + 校验 + 灰度切读，不停服扩容
```

**一致性哈希**（了解）：

```
传统哈希（hash % N）：N 变化时几乎所有数据要重新分布（迁移量大）
一致性哈希：把哈希空间组织成环（0 ~ 2^32-1），
           节点和 key 都映射到环上，key 顺时针找最近节点
           扩容时只影响新节点附近的一小段 key（迁移少）

问题：节点少时数据不均（倾斜）→ 用虚拟节点（每节点映射多个虚拟点）
应用：Redis 不用一致性哈希（用哈希槽），Memcached 用一致性哈希
```

## 四、实战层：全局 ID 方案

**全局 ID 需求**：分库分表后各分片独立生成 ID，需要全局唯一。且 ID 最好是趋势递增（利于 B+ 树插入，避免页分裂）。

**方案对比**（面试必考）：

| 方案 | 唯一性 | 递增 | 性能 | 依赖 | 适用 |
|------|--------|------|------|------|------|
| **UUID** | 全局唯一 | 无序 | 高 | 无 | 不推荐（B+ 树页分裂） |
| **自增主键（单库）** | 单库唯一 | 递增 | 高 | DB | 单库（分库后冲突） |
| **号段模式** | 全局唯一 | 趋势递增 | 极高 | DB（批量取号） | 高并发（美团 Leaf） |
| **雪花算法** | 全局唯一 | 趋势递增 | 极高 | 时钟 | 推荐（主流） |
| **Redis INCR** | 全局唯一 | 递增 | 高 | Redis | 中等并发 |

**雪花算法（Snowflake）**（画图必考）：

```
64 位 long 型 ID 结构：
┌──────────┬──────────┬───────────┬──────────────┐
│ 1 bit    │ 41 bit   │ 10 bit    │ 12 bit       │
│ 符号位   │ 时间戳    │ 机器 ID    │ 序列号        │
│ (固定 0) │ (毫秒)    │ (1024 台)  │ (每毫秒 4096) │
└──────────┴──────────┴───────────┴──────────────┘

- 时间戳：41 bit，可用 69 年（2^41 ms / 1000 / 3600 / 24 / 365）
- 机器 ID：10 bit，1024 台机器（可拆 5 bit 机房 + 5 bit 机器）
- 序列号：12 bit，每毫秒每机器可生成 4096 个 ID

总容量：1024 台机器 × 每毫秒 4096 = 每秒 400 万 ID
特点：趋势递增（时间戳在高位），全局唯一（机器号区分），不依赖 DB

问题：时钟回拨（机器时间往回拨会导致重复 ID）
解决：1. 拒绝生成（回拨期间报错）
      2. 等待（等时间追上）
      3. 用序列号兜底（回拨量小于序列号空间时）
```

**雪花算法代码**：

```java
public class SnowflakeIdGenerator {
    private final long epoch = 1609459200000L;   // 起始时间（2021-01-01）
    private final long machineIdBits = 10L;
    private final long sequenceBits = 12L;
    private final long maxMachineId = ~(-1L << machineIdBits);   // 1023
    private final long sequenceMask = ~(-1L << sequenceBits);    // 4095

    private final long machineId;
    private long sequence = 0;
    private long lastTimestamp = -1L;

    public synchronized long nextId() {
        long currentTimestamp = System.currentTimeMillis();

        // 时钟回拨处理
        if (currentTimestamp < lastTimestamp) {
            long offset = lastTimestamp - currentTimestamp;
            if (offset <= 5) {
                try { Thread.sleep(offset); } catch (InterruptedException e) {}
                currentTimestamp = System.currentTimeMillis();
            } else {
                throw new RuntimeException("时钟回拨超过 5ms");
            }
        }

        if (currentTimestamp == lastTimestamp) {
            sequence = (sequence + 1) & sequenceMask;
            if (sequence == 0) {   // 序列号用尽，等下一毫秒
                currentTimestamp = waitNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0;
        }

        lastTimestamp = currentTimestamp;

        return ((currentTimestamp - epoch) << (machineIdBits + sequenceBits))
             | (machineId << sequenceBits)
             | sequence;
    }
}
```

## 六、底层本质：数据本地性与扩展性权衡

回到第一性：**分库分表的本质是"用数据分散换扩展性，代价是跨分片查询复杂"**。

- **数据本地性原则**：同一用户的频繁关联数据（订单、地址、评价）应在同一分片，避免跨库 JOIN 和聚合。路由维度选最高频查询条件（如 user_id），让 80%+ 查询单分片命中。
- **扩容的数学基础**：分片数用 2 的幂（2,4,8,16），扩容翻倍时迁移量恰好 50%（hash % 2N 时，原 hash % N 的每个分片正好分裂为两个）。非 2 的幂扩容迁移量大（如 3→5 要重新分布 80%+）。
- **全局 ID 的递增性**：B+ 树索引按 ID 排序，ID 无序（如 UUID）会导致插入位置随机，触发频繁页分裂（数据搬移）。趋势递增（雪花）让新数据追加到索引末尾，页分裂少，插入性能好。
- **一致性的妥协**：分库分表后无法用单库事务，跨库操作要分布式事务（Saga/可靠消息，见 018 题）。所以设计时要保证"单分片内事务完整"，跨分片用最终一致。

## 七、AI 架构师加问：5 个 AI 相关问题

1. **AI 推理结果表怎么分片？**
   按 user_id 分片（查询"某用户的推理历史"是高频场景）。如果按场景查询多（如查"某模型的所有结果"），建 ES 宽表异构索引。大字段（embedding 向量）不存 MySQL，存向量数据库。

2. **让 AI 设计分片策略，AI 接管哪段？**
   AI 分析查询日志统计高频查询条件 → 推荐路由维度（高频字段）→ 评估跨分片查询比例。AI 还能预测数据增长，建议扩容时机（单分片超 1000 万行时）。最终分片策略要架构师结合业务确认。

3. **扩容时怎么用 AI 辅助？**
   AI 监控各分片数据量分布（是否倾斜）→ 推荐扩容时机 → 规划迁移批次（按 hash 范围分批，避免一次性迁移打爆 DB）。AI 还能校验迁移数据一致性（抽样对比）。

4. **AI 生成的分片键怎么验证合理？**
   静态分析：检查查询条件是否都带分片键（不带的就是跨分片查询，要优化）。压测：模拟生产查询，统计跨分片查询比例（应 < 20%）。监控：上线后统计单分片查询率，低说明分片键选错。

5. **向量数据的"分片"和 MySQL 一样吗？**
   不一样。向量库（Milvus）按集合分片（类似表），内部用 HNSW/IVF 索引。查询是 ANN（近似最近邻），天然要查所有分片（因为相似向量可能分布在各分片）。所以向量库的分片主要是容量扩展，查询性能靠索引优化而非分片路由。

## 八、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"分库分表、用户维度路由、倍数扩容、雪花 ID、绑定表"**。

- **分库**（连接/TPS）vs **分表**（数据量），通常一起做
- **路由**：hash(user_id) % N，C 端业务按用户分片
- **扩容**：倍数翻倍（2→4→8），分片数用 2 的幂，迁移 50%
- **全局 ID**：雪花算法（时间戳+机器号+序列号，趋势递增）
- **跨库 JOIN**：绑定表（同维度分片）、广播表、ES 宽表
- **中间件**：ShardingSphere（JDBC 模式轻量）

### 拟人化理解

把分库分表想成**连锁仓储**。单仓装不下时，按"收件人地区"分到多个仓（按维度分片）。北京仓放北京用户的货（路由），查"我的订单"只查北京仓（单分片高效）。扩容时北京仓再拆东城西城仓（倍数扩容，地区编号不变只是细分），迁移一半货到新仓。每个包裹有全国唯一物流单号（雪花 ID，不会因多仓重复）。查"某商品的所有买家"要问所有仓（跨分片广播），要么建个商品索引（ES 宽表）。

### 面试现场 60 秒回答

> 分库分表解决单库单表瓶颈——分库突破连接数和 TPS，分表突破单表数据量。路由按用户维度 hash(user_id) % N，让同一用户的数据在一个分片（数据本地性）。扩容用倍数翻倍（2→4→8），分片数必须是 2 的幂，翻倍时迁移量恰好 50%。全局 ID 用雪花算法（41bit 时间戳 + 10bit 机器号 + 12bit 序列号），趋势递增利于 B+ 树插入，全局唯一不依赖 DB，每秒 400 万 ID。跨库 JOIN 靠绑定表（主从表按同维度分片）、广播表（小表冗余）、ES 宽表（异构索引）。中间件用 ShardingSphere，JDBC 模式应用直连 DB 性能好，业务代码无感知。设计原则：80%+ 查询单分片命中，跨分片用最终一致。

### 反问面试官

> 贵司核心表（订单/交易）是分库分表了还是单库？分了多少库表？扩容是倍数翻倍还是一致性哈希？全局 ID 用雪花还是自研（如美团 Leaf 号段）？跨分片查询有没有遇到性能问题？

## 九、苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 为什么不分库分表，加机器垂直扩展不行吗？ | 用物理上限说话：单机磁盘有上限（TB 级）、连接数有上限（几千）、写入 TPS 有上限（几千）。亿级数据 + 万级 TPS 单机扛不住。垂直扩展（加 CPU/内存）到一定程度边际成本陡增（高端服务器贵），水平扩展（分片）用普通服务器性价比高 |
| 证据追问 | 怎么判断该分库分表了？ | 量化指标：单表行数 > 1000 万、单表磁盘 > 100GB、查询 P99 > 1s 且索引优化无效、单库连接数打满、写入 TPS 到瓶颈。不是"感觉慢"，是指标证明单机到极限 |
| 边界追问 | 分库分表能解决所有性能问题吗？ | 不能。解决不了：跨分片查询慢（要异构索引）、分布式事务复杂（要 Saga）、运维复杂（N 个库备份监控）、热点分片（某用户数据特别多）。分库分表是"最后手段"，先优化索引/SQL/缓存，不到万不得已不分 |
| 反例追问 | 什么场景不该分库分表？ | 数据量小（百万级，索引优化够）、强一致需求高（分布式事务代价大）、跨维度查询多（路由维度难选）、运维能力不足（分库分表运维复杂）。先垂直拆分（按业务域拆库），水平拆分是最后手段 |
| 风险追问 | 分库分表上线后最大风险？ | 主动点出：跨分片查询性能差（广播全库）、扩容迁移数据风险、分布式事务复杂、分片键选错（大量跨分片查询）、热点分片（某用户数据倾斜）、运维复杂（N 倍监控备份） |
| 验证追问 | 怎么证明分片策略合理？ | 统计单分片查询率（应 > 80%）；监控各分片数据量分布（是否倾斜）；跨分片查询 P99（应在容忍范围）；扩容演练（验证迁移流程） |

### 现场对话示例

**面试官**：为什么分片数要用 2 的幂？

**候选人**：因为扩容翻倍时迁移量最少。假设 2 库，hash(user_id) % 2，数据分布：hash%2=0 在库 0，hash%2=1 在库 1。扩到 4 库，hash(user_id) % 4：原来 hash%2=0 的数据，hash%4 可能是 0 或 2（各占一半）；原来 hash%2=1 的数据，hash%4 可能是 1 或 3（各占一半）。所以库 0 的一半数据要迁到库 2，库 1 的一半要迁到库 3，总迁移量 50%。如果分片数不是 2 的幂，如 3 库扩到 5 库，hash%3 和 hash%5 的映射关系复杂，几乎每个分片的数据都要重新分布，迁移量可能 80%+。2 的幂翻倍的好处是 hash%2N 与 hash%N 有数学关系（hash%2N = hash%N 或 hash%N + N），迁移量恰好 50%。所以设计分片数时预留 2 的幂（16, 32, 64），扩容路径清晰。

**面试官**：雪花算法时钟回拨怎么办？

**候选人**：时钟回拨是雪花算法的主要风险——机器时间往回拨（NTP 同步、手动调时间），可能导致生成重复 ID（同一毫秒同一序列号）。三种处理方案。第一，拒绝生成——回拨期间直接报错，等时间追上（简单但影响可用性）。第二，等待——如果回拨量小（< 5ms），线程 sleep 等时间追上再生成（平衡可用性）。第三，用序列号兜底——回拨量小于序列号空间（12bit = 4096）时，回拨期间用更大的序列号避免冲突（复杂）。生产实践：机器部署 NTP 服务同步时间（预防回拨）；回拨检测 + 自动报警（回拨超 5ms 告警）；关键业务用号段模式兜底（雪花挂了用号段）。另外，机器号分配要用 Zookeeper/配置中心管理（避免手动配置重复），机器下线后回收机器号。

**面试官**：分库分表后怎么做分页查询？

**候选人**：这是分库分表的经典难题。单库分页是 LIMIT offset, size，跨库分页要"每个库取 offset+size 条，内存合并取前 size 条"。如 LIMIT 100, 10，4 个库各取 110 条，内存合并排序取 100-109 条。问题：深分页时每个库要取很多（LIMIT 10000, 10 取 10010 条 × 4 库）。三种优化。第一，禁止跨库深分页——产品限制（如最多翻 100 页），或要求带分片键（查"我的订单"单库分页无此问题）。第二，游标分页——记住上一页最后一条的（时间戳, id），下一页 WHERE (create_time, id) < (上一页值) ORDER BY create_time DESC LIMIT 10，不用 OFFSET。第三，ES 宽表——把数据同步到 ES，用 ES 做复杂查询和分页，DB 只负责写入。生产首选带分片键的查询（单库分页），跨库分页用 ES 兜底。

## 常见考点

1. **分库分表和分区表区别？**——分区表是单库内按规则（范围/列表/哈希）把表拆成多个分区，对应用透明，仍是单库（连接数/TPS 瓶颈不解决）。分库分表是跨多个数据库实例，真正分散 IO 和连接。分区表适合中等数据量（千万级），分库分表适合亿级。
2. **一致性哈希和哈希槽区别？**——一致性哈希把节点和 key 都映射到环上（0~2^32-1），key 顺时针找最近节点，扩容时只影响相邻段。哈希槽（Redis Cluster）固定 16384 个槽，节点负责一段槽，扩容时迁移槽。哈希槽迁移粒度可控（按槽），一致性哈希用虚拟节点解决倾斜。
3. **号段模式和雪花算法怎么选？**——号段模式（DB 批量取号）依赖 DB 但性能极高（内存发号），适合超高并发且有 DB 保障的场景；雪花算法不依赖 DB（去中心化），适合分布式多机部署。两者可结合——雪花为主，号段兜底（雪花时钟回拨时用号段）。
4. **分库分表后事务怎么办？**——跨库事务用分布式事务方案（Saga/TCC/可靠消息，见 018 题）。设计原则：把强相关的数据放同一分片（单库事务），跨分片用最终一致。如订单和订单项按 user_id 同分片（单库事务），订单和库存跨服务用 Saga。

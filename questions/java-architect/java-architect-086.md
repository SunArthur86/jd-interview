  rebuild: 五步——第一，定义指标（核心：点击率 CTR；护栏：加载延迟、崩溃率）。第二，计算样本量（基线 CTR=5%，期望提升 10% 即 5.5%，α=0.05，power=0.8，算出每组需 15 万样本）。第三，分流（用户 ID 哈希分桶，50% A、50% B，同一用户始终同组）。第四，收集数据并统计检验（t 检验，p-value < 0.05 且效果方向正确）。第五，决策（显著提升且护栏无恶化→全量 B；否则保留 A 或调优 B）。期间监控 SRM（实际比例 50:50，失调说明 bug）。
follow_up:
  - 怎么保证分流的随机性？——用户 ID + 实验层 ID 做哈希（MurmurHash3），哈希值 mod 1000 得到桶号（0-999）。哈希均匀分布保证随机性。分桶结果缓存（Redis）保证一致性（同一用户下次查还是同桶）
  - 多个实验同时跑怎么互不干扰？——实验层正交。每个实验在一个独立的层（layer），用户在不同层独立分流（用层 ID + 用户 ID 哈希）。实验 A 在 layer1，实验 B 在 layer2，互不影响（正交）。类似分层抽样
  - 样本量怎么算？——基于基线转化率 p、期望提升 Δp、显著性 α（0.05）、功效 power（0.8）。公式 n = 16 * p*(1-p) / Δp^2（近似）。或用在线计算器（evanmiller.org/ab-testing）。样本不足时实验不可信（检测不出差异）

# 【Java 后端架构师】灰度实验平台与 A/B Testing

> 适用场景：JD 核心技术。推荐算法 v2 想替换 v1，首页改版想验证效果，价格策略想对比收益。架构师必须设计科学的 A/B 实验平台——分流、指标、统计检验、护栏监控。

## 一、概念层：A/B 测试的科学基础

**A/B 测试核心流程**（面试必画）：

```
1. 假设                  2. 分流                3. 实验
   "B 比 A 好"            用户ID哈希分桶           收集指标数据
   （假设检验）            50% A / 50% B           核心指标 + 护栏

4. 统计检验              5. 决策
   t检验 / p-value        p<0.05 且护栏稳 → 全量 B
   置信区间               否则 → 保留 A 或调优 B
```

**指标体系**（核心 + 护栏）：

| 指标类型 | 作用 | 例子 | 决策权重 |
|---------|------|------|---------|
| 核心指标 | 决定方案好坏（决策用） | CTR（点击率）、转化率、GMV | 高（看是否显著提升） |
| 护栏指标 | 保底（防止方案有副作用） | 崩溃率、页面延迟、流失率、客诉率 | 否决权（恶化则否决） |
| 辅助指标 | 诊断（理解为什么好/差） | 各环节漏斗、停留时长、跳出率 | 参考（辅助理解） |

## 二、机制层：分流与实验层正交

**分流哈希算法**（同一用户始终同一组）：

```java
public class ExperimentRouter {

    /**
     * 分流：用户ID + 实验层ID 哈希分桶
     * @param userId  用户ID
     * @param layerId 实验层ID（保证不同实验层正交）
     * @param buckets 桶配置（如 ["A:50", "B:50"]）
     * @return 分组（A 或 B）
     */
    public String route(String userId, String layerId, List<Bucket> buckets) {
        // 1. 用户ID + 层ID 拼接后哈希（MurmurHash3，均匀分布）
        String key = userId + ":" + layerId;
        int hash = MurmurHash3.hash32(key.getBytes());
        int bucketNum = Math.abs(hash) % 1000;   // 桶号 0-999

        // 2. 累计权重找分组（A 占 0-499，B 占 500-999）
        int cumulative = 0;
        for (Bucket bucket : buckets) {
            cumulative += bucket.getWeight() * 10;   // 50% → 500
            if (bucketNum < cumulative) {
                return bucket.getName();   // 返回 A 或 B
            }
        }
        return buckets.get(buckets.size() - 1).getName();
    }
}

// 使用：推荐算法实验（layer=reco）
String group = router.route("user123", "reco",
    List.of(new Bucket("A", 50), new Bucket("B", 50)));
// user123 在 reco 层始终是同一组（一致性）
```

**实验层正交**（多实验互不干扰）：

```
实验层架构（Layer / Domain）：

┌─────────────────────────────────────────────────┐
│ UI 层（layer=ui）                                │
│ 实验A：红色按钮 vs 蓝色按钮                       │
│ 用户ID + "ui" 哈希分桶                            │
├─────────────────────────────────────────────────┤
│ 推荐层（layer=reco）                              │
│ 实验B：算法v1 vs 算法v2                          │
│ 用户ID + "reco" 哈希分桶（独立于 ui 层）          │
├─────────────────────────────────────────────────┤
│ 价格层（layer=price）                             │
│ 实验C：满减 vs 优惠券                            │
│ 用户ID + "price" 哈希分桶（独立）                 │
└─────────────────────────────────────────────────┘

正交性：同一用户在 ui 层可能是 A（红），在 reco 层可能是 B（v2），
      在 price 层可能是 A（满减）。三组实验独立分流，互不干扰。
      类似分层抽样——每层独立随机，组合后覆盖所有情况。
```

**互斥实验 vs 正交实验**：

| 类型 | 规则 | 场景 | 例子 |
|------|------|------|------|
| 互斥实验 | 同一实验域内，用户只能进一个实验 | 同类实验（会互相影响） | 两个推荐算法实验（都改推荐，不能同时） |
| 正交实验 | 不同实验层，用户在每层独立分流 | 不同类实验（不互相影响） | UI 实验 + 推荐实验（改的东西不同，可同时） |

## 三、机制层：统计假设检验

**样本量计算**（实验前必算）：

```java
public class SampleSizeCalculator {

    /**
     * 计算单组所需样本量（双比例 z 检验）
     * @param p1        基线转化率（如 0.05）
     * @param mde       最小可检测效应（如 0.1，即提升 10%）
     * @param alpha     显著性水平（0.05）
     * @param power     统计功效（0.8）
     * @return 每组样本量
     */
    public static int calculate(double p1, double mde, double alpha, double power) {
        double p2 = p1 * (1 + mde);           // 实验组预期转化率
        double pAvg = (p1 + p2) / 2;

        // z 分位数
        double zAlpha = NormalDistribution.inverseCumulativeProbability(1 - alpha / 2);  // 1.96
        double zBeta = NormalDistribution.inverseCumulativeProbability(power);            // 0.84

        // 样本量公式：n = (zα/2 + zβ)² * 2 * p_avg*(1-p_avg) / (p2-p1)²
        double delta = p2 - p1;
        double n = Math.pow(zAlpha + zBeta, 2) * 2 * pAvg * (1 - pAvg) / (delta * delta);

        return (int) Math.ceil(n);
    }
}

// 例子：基线 CTR=5%，期望提升 10%（即 5.5%），α=0.05，power=0.8
// calculate(0.05, 0.10, 0.05, 0.8) → 每组约 15 万样本
// 日活 100 万，分流 50%，每天 50 万样本，实验需 15万/50万 ≈ 0.3 天
// 但为了覆盖周期波动，至少跑 7 天（覆盖一周内的行为模式）
```

**统计检验代码**（实验后判断显著性）：

```java
public class ABTestAnalyzer {

    /**
     * 双比例 z 检验（A/B 两组转化率对比）
     */
    public static TestResult twoProportionZTest(
            long conversionsA, long samplesA,    // A 组：转化数、样本数
            long conversionsB, long samplesB) {  // B 组：转化数、样本数

        double pA = (double) conversionsA / samplesA;   // A 组转化率
        double pB = (double) conversionsB / samplesB;   // B 组转化率
        double pPool = (double) (conversionsA + conversionsB) / (samplesA + samplesB);

        // 合并标准误
        double se = Math.sqrt(pPool * (1 - pPool) * (1.0/samplesA + 1.0/samplesB));

        // z 统计量
        double z = (pB - pA) / se;

        // p-value（双尾）
        double pValue = 2 * (1 - NormalDistribution.cumulativeProbability(Math.abs(z)));

        // 提升幅度
        double lift = (pB - pA) / pA;

        // 95% 置信区间
        double margin = 1.96 * se;
        double[] ci = {pB - pA - margin, pB - pA + margin};

        return new TestResult(pA, pB, lift, pValue, ci,
            pValue < 0.05 && lift > 0   // 显著且正向提升
        );
    }
}

// 例子：A 组 5000/100000=5%，B 组 5500/100000=5.5%
// z=2.24, p-value=0.025, lift=10%, CI=[0.001, 0.009]
// 结论：显著提升（p<0.05），可全量 B
```

**SRM（样本比例失调）检测**：

```java
public class SRMChecker {

    /**
     * 卡方检验检测样本比例失调
     * 预期 50:50，实际比例是否偏离
     */
    public static boolean hasSRM(long samplesA, long samplesB, double expectedRatio) {
        long total = samplesA + samplesB;
        long expectedA = (long) (total * expectedRatio);
        long expectedB = total - expectedA;

        // 卡方统计量
        double chiSquare = Math.pow(samplesA - expectedA, 2) / expectedA
                         + Math.pow(samplesB - expectedB, 2) / expectedB;

        // p-value（自由度=1）
        double pValue = 1 - ChiSquareDistribution.cumulativeProbability(chiSquare, 1);

        // p < 0.001 表示严重失调（SRM）
        return pValue < 0.001;
    }
}

// 例子：预期 50:50，实际 A=49500, B=50500（实际 49.5:50.5）
// chiSquare=100, p-value<0.001 → SRM！分流有 bug，实验结论无效
// 原因排查：哈希不均？实验组崩溃？定向分流失衡？
```

**业务集成代码**（推荐场景）：

```java
@Service
public class RecommendService {

    @Autowired private ExperimentRouter router;
    @Autowired private RecoAlgorithmV1 algoV1;
    @Autowired private RecoAlgorithmV2 algoV2;
    @Autowired private MetricReporter metricReporter;

    public List<Item> recommend(String userId) {
        // 1. 分流判断用户在推荐实验的分组
        String group = router.route(userId, "reco",
            List.of(new Bucket("A", 50), new Bucket("B", 50)));

        // 2. 根据分组走不同算法
        List<Item> items;
        if ("A".equals(group)) {
            items = algoV1.recommend(userId);
        } else {
            items = algoV2.recommend(userId);
        }

        // 3. 埋点上报（曝光、分组）
        metricReporter.reportExperimentExposure(userId, "reco", group);
        // 后续点击、转化事件也带 group，用于指标计算

        return items;
    }
}
```

## 五、底层本质：为什么 A/B 测试要这么设计

回到第一性：**A/B 测试的科学基础是随机对照试验（RCT），工程上是保证随机性、一致性、正交性**。

- **随机性的必要性**：随机分流保证两组除方案外其他条件相同（用户分布、时段、设备等），差异只能由方案引起。如果分流有偏（如 VIP 都进 A 组），差异可能是用户属性而非方案。哈希算法（MurmurHash3）均匀分布保证随机性，但要验证（SRM 检测、分群均衡性检查）。
- **一致性的必要性**：同一用户必须始终进同一组（不能这次 A 下次 B，用户体验混乱且数据污染）。用 userId + layerId 哈希保证——同一用户同一层哈希结果固定。分流结果可缓存（Redis）减少计算，但缓存要和哈希一致（缓存失败回退到哈希重算）。
- **正交性的必要性**：多实验同时跑时，如果都用同一个分流（userId 哈希），实验间会相关——实验 A 的 A 组和实验 B 的 A 组是同一批人，无法区分效果。正交（每层独立哈希）保证实验间统计独立，可以单独分析每个实验的效果。互斥实验（同一域内）用于会互相影响的实验（如两个推荐算法），正交实验（不同层）用于不互相影响的实验（如 UI + 推荐）。
- **统计检验的必要性**：直觉判断不可靠——5% vs 5.5% 看着有差异，但小样本时可能是随机波动。统计检验量化"差异是真实的概率"——p-value < 0.05 表示 95% 置信度差异不是随机的。但要配合样本量（样本不足检测不出差异，即功效不足）和 MDE（最小可检测效应，太小需要巨量样本）。
- **护栏指标的必要性**：核心指标可能提升（CTR 涨了），但护栏可能恶化（崩溃率涨了、延迟涨了）。没有护栏可能"为追求核心指标牺牲用户体验"。护栏有否决权——即使核心指标显著提升，护栏恶化也不能上线。典型护栏：崩溃率、页面延迟、留存率、客诉率。
- **SRM 检测的必要性**：预期 50:50 但实际 49:51，说明分流有 bug 或实验组有问题（如 B 组崩溃率高导致用户退出计数减少）。SRM 存在时两组不可比，实验结论无效。卡方检验自动检测，SRM 告警立即停止实验排查。

**为什么用实验层而不是简单分流**：简单分流（所有实验用同一 userId 哈希）会导致实验相关——实验 A 的 A 组和实验 B 的 A 组是同一批人。实验层正交（每层独立哈希 userId + layerId）保证实验独立。这是 Google Overlap 模型、Microsoft ExP 的核心设计。

**为什么实验要跑满一个周期**：用户行为有周期性（工作日 vs 周末、白天 vs 夜晚）。跑 1 天只看到一天的样本分布，跑 7 天覆盖完整周期。提前停止（peeking）会虚高显著性（多次检验增加假阳性）。Sequential testing（序贯检验）可提前停止但要校正（如 Bonferroni）。

## 六、AI 架构师加问：5 个

1. **AI 自动分析实验结果？**
   能辅助。AI 分析实验数据，不只给 p-value，还解释"为什么 B 好或差"——分群分析（B 对新用户好、对老用户差）、漏斗分析（B 在哪一步提升）。比单纯统计检验更有洞察。但决策（是否上线）要人工，AI 可能过度解读噪声。

2. **AI 自动调实验参数？**
   AI 能根据实验数据自动调流量比例（B 表现好，自动从 10% 调到 50%）。类似 bandit 算法（多臂老虎机）。但风险是过早收敛（过早押注 B，错过 A 的后期追赶）。生产用固定分流 + 人工决策更稳。

3. **AI 做因果推断替代 A/B？**
   能补充。A/B 消耗流量（实验组可能差），因果推断（如 uplift modeling）从历史数据推断效果，不需实验。但因果推断假设多（无未观测混杂），准确性不如 A/B。适合无法 A/B 的场景（如价格实验，不能对用户收不同价）。

4. **LLM 时代 A/B 怎么做？**
   LLM 应用（如推荐话术）的 A/B 挑战是输出不确定性（同一输入 LLM 可能给不同输出）。要做多次采样取平均，或固定 seed。指标更难定义（话术好坏是主观的，用人工标注或 LLM-as-judge）。

5. **AI 检测实验数据异常？**
   AI 能检测指标异常波动（如某天 B 组 CTR 骤降，可能是 bug 或外部事件）。比阈值规则更准（能识别"慢漂移"）。但根因分析（为什么降）要人工排查。

## 七、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"五步走、三指标、三性"**。

- **五步走**：假设 → 分流 → 实验 → 统计检验 → 决策
- **三指标**：核心（决策）、护栏（否决）、辅助（诊断）
- **三性**：随机性（哈希均匀）、一致性（同用户同组）、正交性（多实验独立）

### 拟人化理解

把 A/B 测试想成**医学临床试验**。把病人随机分两组（分流），一组吃新药（实验组 B），一组吃老药（对照组 A）。观察疗效指标（转化率 CTR）和副作用护栏（崩溃率、流失率）。用统计检验判断"疗效差异是真效果还是运气"（p-value < 0.05）。最后决定新药上市（全量 B）还是弃用。多药物试验不能混着吃（互斥），不同症状的药可以同时（正交）。

### 面试现场 60 秒回答

> A/B 测试五步——假设、分流、实验、统计检验、决策。第一步定义假设——"推荐算法 v2 比 v1 的 CTR 高"。定义指标——核心是 CTR，护栏是崩溃率和延迟（否决权），辅助是漏斗各环节。第二步分流——用户 ID + 实验层 ID 哈希（MurmurHash3）mod 1000 分桶，50% A 50% B。分流有三性保证——随机性（哈希均匀）、一致性（同一用户始终同组，userId+layerId 固定哈希）、正交性（不同实验在不同层独立分流，互不干扰）。第三步实验前算样本量——基于基线 CTR（5%）、MDE（10%）、α（0.05）、power（0.8）算出每组 15 万样本，至少跑 7 天覆盖周期。第四步统计检验——双比例 z 检验，p-value < 0.05 且效果方向正确才算显著。同时 SRM 检测（卡方检验，预期 50:50 实际失调则实验无效）。第五步决策——核心指标显著提升且护栏不恶化→全量 B；否则保留 A 或调优。护栏实时监控，B 崩溃率超阈值自动熔断（kill switch 回退 A）。平台架构：配置中心 + SDK（分流+埋点）+ 数据处理（Kafka+Flink+ClickHouse）+ 分析看板。

### 反问面试官

> 贵司有自研 A/B 平台还是用开源（如 GrowthBook）？日实验量多少？分流是 SDK 还是网关？这决定我聊架构设计还是统计分析方法。

### 现场对话示例

**面试官**：同一用户为什么必须始终进同一组？

**候选人**：一致性，两个原因。第一，用户体验——如果用户这次看到红色按钮（A），下次看到蓝色（B），体验混乱（以为是 bug）。特别是 UI 实验，频繁切换让用户困惑。第二，数据污染——如果用户在 A 和 B 之间切换，他的行为（点击、转化）既受 A 影响又受 B 影响，无法归因。统计检验假设"两组用户是固定的"，切换会破坏假设。实现上用 userId + layerId 哈希——同一用户同一层的哈希结果固定，始终同一组。分流结果可缓存（Redis）减少重复计算，但缓存失败时回退到哈希重算（幂等）。

**面试官**：多实验同时跑会不会互相影响？

**候选人**：取决于实验层设计。同一实验域（互斥组）内的实验会互相影响——比如两个推荐算法实验（都改推荐逻辑），用户只能进一个，不能同时。不同实验层（正交）的实验不互相影响——比如 UI 实验（layer=ui）和推荐实验（layer=reco），用户在每层独立分流，可以同时是 UI 的 A 和 reco 的 B。正交的关键是每层用 layerId + userId 哈希，层间哈希独立。验证正交性——做交叉分析（UI 的 A 组在 reco 层的 A/B 比例应该 50:50，不相关）。如果发现相关（UI 的 A 偏向 reco 的 A），说明正交设计有 bug。

**面试官**：实验跑到一半发现 B 组崩溃率高怎么办？

**候选人**：熔断。护栏指标（崩溃率）实时监控，超阈值自动触发 kill switch——立即停止实验，全量回退 A。护栏有否决权，即使核心指标（CTR）提升，崩溃率涨也不能上线。熔断机制：Flink 实时计算 B 组崩溃率，超过基线（A 组）2 倍触发告警，平台自动把 B 的流量比例从 50% 调到 0%（秒级）。同时通知实验负责人排查（B 算法有 bug）。这是 A/B 平台的安全网——允许试错但自动止损。极端情况（B 导致数据损坏），需要数据修复 + 通知受影响用户。

## 常见考点

1. **A/B 分流怎么保证随机和一致？**——用户 ID + 实验层 ID 哈希（MurmurHash3）mod 1000 分桶。哈希均匀保证随机，userId+layerId 固定保证一致（同一用户同一层始终同组）。SRM 检测验证随机性。
2. **什么是实验层正交？**——不同实验在不同层（layer），每层独立分流（userId + layerId 哈希）。实验间统计独立，可单独分析效果。互斥实验（同一域内）用于会互相影响的实验。
3. **样本量怎么算？**——基于基线转化率、MDE（最小可检测效应）、α（0.05）、power（0.8）。公式 n = 16*p*(1-p)/Δp²（近似）。样本不足功效不够，检测不出差异。
4. **p-value < 0.05 就能上线吗？**——不能只看 p-value。还要护栏稳定（崩溃率/延迟不恶化）、样本量充足、SRM 无失调、分群均衡、置信区间不含 0。护栏有否决权。
5. **什么是 SRM？**——样本比例失调（预期 50:50 实际 49:51）。卡方检验 p<0.001 判定 SRM。原因：分流 bug、实验组崩溃、定向分流失衡。SRM 存在时实验无效，必须排查。

## 结构化回答

**30 秒电梯演讲：** A/B 测试的本质是用随机分流 + 统计假设检验，科学判断方案 A 和 B 谁更好。核心三要素——分流（用户随机分桶保证实验组和对照组可比）、指标（核心指标 + 护栏指标）、统计显著性（p-value < 0.05 才算可信结论）。难点是样本量计算、分流一致性（同一用户始终进同一组）、SRM（样本比例失调）检测、交互效应（多实验互扰）

**展开框架：**
1. **分流** — 用户 ID + 实验层哈希分桶（mod 1000），同一用户始终进同一组
2. **实验层正交** — 不同实验在不同层（layer），互不干扰（正交分流）
3. **指标体系** — 核心指标（决策用，如转化率）+ 护栏指标（保底，如崩溃率/流失率）

**收尾：** 以上是我的整体思路。您想继续深入聊——怎么保证分流的随机性？

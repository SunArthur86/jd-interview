# 【Java 后端架构师】架构图、ADR 与技术决策记录怎么写

> 适用场景：JD 核心技术。架构师换团队后，新人面对一堆代码问"为什么订单服务用 saga 不用 TCC""为什么选 RocketMQ 不选 Kafka"，没人答得上来——决策上下文丢失。架构图和 ADR 是解决"架构知识传递和沉淀"的两支笔，缺一不可。

## 一、概念层：C4 模型与 ADR 四要素

### 1.1 C4 模型（架构图的四层）

C4（Context/Container/Component/Code）是 Simon Brown 提出的架构图分层方法，核心思想是"像 Google Maps 一样分层缩放"：

| 层级 | 内容 | 受众 | 示例 |
|------|------|------|------|
| **Level 1: Context（系统上下文）** | 系统与外部依赖（用户、第三方）的关系 | 业务方、高管 | 用户 → 订单系统 → 支付网关 |
| **Level 2: Container（容器）** | 系统内的进程/服务/数据库（"容器"≠Docker） | 运维、架构师 | 订单服务(Java) + MySQL + Redis + Kafka |
| **Level 3: Component（组件）** | 单个服务内的模块/包 | 开发 | 订单服务内：下单模块/查询模块/履约模块 |
| **Level 4: Code（代码）** | 类图、时序图（UML） | 作者自己、新人 | OrderService.createOrder() 时序 |

**C4 的核心价值**：不分层的架构图要么太粗（业务看不懂细节）要么太细（高管看不懂全貌）。C4 让你按受众分层画，每层聚焦关键信息。

### 1.2 ADR 四要素（决策记录的骨架）

ADR（Architecture Decision Record）由 Michael Nygard 提出，是轻量级的决策记录格式：

| 要素 | 内容 | 价值 |
|------|------|------|
| **Status** | Proposed/Accepted/Deprecated/Superseded | 决策的生命周期状态 |
| **Context（背景）** | 问题、约束、量化指标 | 为什么需要做这个决策 |
| **Decision（决策）** | 选了什么 + 替代方案否决理由 | 决策本身 + 思考过程 |
| **Consequences（后果）** | 正面/负面影响、风险 | 决策的代价和收益 |

**ADR 的核心价值**：3 年后的新人看 ADR 就能理解决策上下文（为什么这么选、考虑过什么替代方案、代价是什么），而不是靠口口相传或猜测。决策可能错，但记录必须真实。

### 2.4 ADR 完整模板（含替代方案）

```markdown
# ADR-042: 订单服务用 Saga 而非 TCC 做分布式事务

## Status
Accepted (2026-07-13)

## Context（背景与问题）
订单创建涉及 3 个服务：订单（建订单）、库存（扣库存）、支付（冻结金额）。
需要跨服务事务保证一致性。

约束：
- 业务容忍最终一致（T+1 对账可接受）
- 性能要求：下单 P99 < 200ms
- 团队能力：TCC 需要业务代码侵入（try/confirm/cancel），团队学习成本高
- 现状指标：强一致方案下下单 P99 350ms，超业务要求

## Decision（决策）
采用 Saga 模式（事件驱动 + 补偿事务），不用 TCC。

### 关键取舍
| 决策点 | Saga | TCC | 选择理由 |
|--------|------|-----|---------|
| 一致性 | 最终一致 | 强一致 | 业务可接受最终一致（对账兜底）|
| 性能 | P99 180ms（无锁等待）| P99 350ms（资源预留）| Saga 满足 <200ms 要求 |
| 代码侵入 | 中（补偿逻辑）| 高（try/confirm/cancel）| 团队能力匹配 Saga |
| 隔离性 | 弱（中间状态可见）| 强（资源预留）| 业务可接受中间状态 |

### 替代方案否决理由
- **TCC**：性能不达标（P99 350ms > 200ms 要求），代码侵入高，否决
- **2PC（XA）**：性能更差（同步阻塞），且 MySQL XA 有可靠性问题，否决
- **本地消息表 + MQ**：适合简单场景，订单涉及 3 服务多步补偿，复杂度失控，否决

## Consequences（后果）
### 正面
- 性能：P99 从 350ms 降到 180ms（达业务要求）
- 可用性：Saga 异步，单服务故障不阻塞下单（补偿事务兜底）

### 负面
- 中间状态可见：订单创建中库存已扣但支付未冻结，用户可见"扣了库存没付款"
- 补偿逻辑复杂：每个步骤要写补偿（cancelOrder/rollbackInventory/unfreezePayment）

### 风险与兜底
- 补偿失败 → 悬空事务巡检 + 人工兜底 SOP（T+1 清零）
- 中间状态用户投诉 → 前端展示"处理中"+ 超时查询接口

## 修订历史
- 2026-07-13: Accepted（@张三）
- 2026-07-20: 补充补偿失败兜底（@李四）
```

### 2.5 ADR 状态机

```
            Proposed（提议中）
                 │
        评审通过  │  评审否决
                 │
        ┌────────┴────────┐
        ▼                 ▼
    Accepted           Rejected
        │
        │  新 ADR 取代
        │  假设失效
        ▼
    Superseded ──指向──> 新 ADR-xxx
        │
        │  完全废弃
        ▼
    Deprecated
```

**关键原则**：ADR 不删除，只改状态。即使决策废弃，历史记录保留（"我们曾经这么想过"），这对未来决策有参考价值。

## 四、底层本质：为什么架构图和 ADR 缺一不可

**架构图解决"当下沟通"**：架构是抽象的，不画图就只能口述。但图也有坑——画太粗（业务看不懂全局）、画太细（开发看不到全貌）。C4 的分层让"按受众画图"成为可能：业务看 Context（懂全局）、运维看 Container（懂部署）、开发看 Component（懂模块）。一张图画给所有人看 = 没人看。

**ADR 解决"未来追溯"**：架构是"约束下的取舍"，但取舍的上下文（为什么这么选）如果不记录，3 年后会丢失。新人接手时，看到"saga 模式"但不知道"为什么不用 TCC"——可能是性能要求，可能是团队能力，可能是历史包袱。ADR 记录这个上下文，让未来的人能判断"假设是否还成立"（如果业务现在要求强一致了，TCC 的否决理由就不成立了，需要重新决策）。

**为什么 ADR 必写替代方案否决理由**：替代方案是决策的"反例"。如果不写否决理由，未来无法评估是否需要重新决策。例：ADR 说"选 saga 不选 TCC"，但没写"否决理由是性能要求 <200ms"。3 年后业务说"现在要强一致"，团队不知道当年为什么否决 TCC，可能重新踩坑。写了否决理由（"TCC 性能 350ms 超标"），未来就能判断"如果性能要求放宽，TCC 可以重新评估"。

**为什么图和 ADR 要入 Git**：文档不入 Git 就和代码脱节——代码改了文档没改（过期），或文档在 wiki 代码在 Git（双向不同步）。入 Git 的好处：(1) 版本化（历史可追溯）；(2) 代码审查时同步审文档（图/ADR 改动和代码改动一起 review）；(3) CI 可检查（核心改动必须引用 ADR，图必须和代码同步）。

## 五、AI 架构师加问：5 个

1. **AI 能自动生成架构图吗？**
   AI 能从代码反向生成 Component/Code 层图（分析类依赖），但 Context/Container 层需要人画（业务抽象 AI 做不了）。AI 辅助：从 import 依赖生成模块图、从 Spring Bean 依赖生成组件图。人 review 修正。

2. **AI 能自动写 ADR 吗？**
   AI 能写 Context（从代码注释/RFC 提取背景）和 Decision（从 commit history 提取决策），但 Consequences（代价收益）和"替代方案否决理由"必须人写——这是思考的产物，AI 没有业务上下文。AI 辅助起草，人 review 补全。

3. **怎么用 AI 让 ADR"活起来"（不变成死文档）？**
   AI 定期扫描 ADR 的假设是否还成立：读当前代码/配置/业务文档，对比 ADR 的 Context（"性能要求 P99 <200ms"），如果业务要求变了（现在要 <100ms），AI 提醒"ADR-042 的假设可能失效，建议重新评估"。把 ADR 从静态文档变成动态知识。

4. **AI 辅助架构图"图即代码"，怎么保证图和代码同步？**
   CI 集成 AI 检查：扫描代码结构（Spring Bean、Feign client、数据库表），对比 DSL 图，发现不一致告警（"代码新增了 OrderQueryService 但图里没有，请更新"）。AI 提示更新点，人决定怎么改。

5. **新人入职，AI 怎么帮他用 ADR 快速理解架构？**
   AI RAG：把所有 ADR + 架构图 + 关键代码注释做成知识库，新人提问"为什么订单用 saga"，AI 检索 ADR-042 返回答案（含 Context/Decision/Consequences）。比新人自己翻 50 篇 ADR 高效。但 AI 必须给引用（ADR 编号），让新人能溯源验证。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"C4 分层、ADR 四要素、替代方案、入 Git、状态机"** 五个词。

- **C4 分层**：Context（业务看）/Container（运维看）/Component（开发看）/Code（作者看）
- **ADR 四要素**：Status / Context / Decision（含替代方案否决）/ Consequences
- **替代方案**：必写否决理由，否则未来无法判断是否要重新决策
- **入 Git**：图和 ADR 与代码同生命周期，CI 检查同步
- **状态机**：Proposed → Accepted → Deprecated/Superseded，不删除

### 拟人化理解

把架构图和 ADR 想成 **建筑工程**。架构图是建筑效果图（沟通给业主看，分层：效果图/平面图/施工图），ADR 是设计计算书（给未来维修工程师看，"为什么承重墙这么设计，不能拆"）。没有效果图的建筑业主不敢买（看不懂），没有计算书的建筑 3 年后没人敢改（不知道为什么这么设计）。两者一起构成"建筑可维护性"。

### 面试现场 60 秒回答

> 架构图我用 C4 模型分层画——Level 1 Context（系统边界给业务看）、Level 2 Container（进程/服务给运维看）、Level 3 Component（模块给开发看）、Level 4 Code（类图给作者看）。每层只画关键信息不堆细节，避免"一张图给所有人看=没人看"。ADR 用四要素：Status（状态机 Proposed/Accepted/Deprecated/Superseded）、Context（背景约束）、Decision（决策+替代方案否决理由）、Consequences（代价收益）。最关键是"替代方案否决理由"必写——3 年后新人能判断假设是否还成立、是否需要重新决策。图和 ADR 都入 Git 和代码同生命周期，CI 门禁核心改动必须引用 ADR，避免文档和实现脱节。

### 反问面试官

> 贵司架构文档是 wiki 还是 Git？ADR 有标准模板和编号规范吗？这决定我推文档规范的方式——wiki 我会先迁 Git，已有规范我会复用。

## 七、苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 为什么花时间写 ADR，直接写代码注释不行吗？ | 代码注释记"怎么做"，ADR 记"为什么这么选"。3 年后看代码知道是 saga 模式（怎么做），但不知道为什么不用 TCC（为什么）——后者才是决策上下文。ADR 补这个空缺 |
| 证据追问 | 怎么证明 ADR 真有用？ | 看新人入职后能否通过 ADR 理解历史决策（"为什么订单用 saga"），不用靠口口相传；看决策回溯效率（出事故时能否快速找到相关 ADR 评估假设是否失效）；看 ADR 引用频率（核心改动是否都引用了 ADR）|
| 边界追问 | 所有决策都写 ADR 吗？ | 不是。ADR 写"架构级"决策（选型、模式、跨服务接口），不写"实现级"（变量命名、设计模式）。后者写代码注释就够。一般按"3 年后还会有人问为什么"判断——会问就写 ADR |
| 反例追问 | 什么情况 ADR 反而是负担？ | (1) 团队 5 人以下、单一产品——口口相传够，ADR 是过度形式化；(2) 决策频繁变化（试错期）——ADR 写完就废弃，维护成本高。ADR 适合需要长期维护的中大型系统 |
| 风险追问 | ADR 写错了怎么办？ | 不修改原 ADR，新建 ADR 标记 Superseded 指向新 ADR。保留历史（"我们曾经这么想过"），这对未来决策有参考价值。如果原 ADR 有事实错误（如性能数据写错），可以在原文加勘误注释，但保留原文 |
| 验证追问 | 怎么保证 ADR 不变成死文档？ | (1) CI 门禁核心改动引用 ADR；(2) 季度 review ADR 状态（Accepted 的假设是否还成立）；(3) 新人入职必读 ADR（培训）；(4) 事故复盘回溯相关 ADR。让 ADR"活"在团队日常工作流里 |
| 沉淀追问 | 团队文档规范沉淀什么？ | C4 DSL 模板、ADR 模板（含四要素）、ADR 编号规范、ADR 索引 README、CI 门禁规则、新人 ADR 入门指南、图即代码工作流（Structurizr/PlantUML）|

### 现场对话示例

**面试官**：你说 ADR 必写替代方案否决理由，为什么这么重要？

**候选人**：替代方案否决理由是决策的"反例"，让未来能判断是否要重新决策。比如 ADR-042 选 saga 否决 TCC，如果只写"选 saga"，3 年后业务要强一致，团队不知道当年为什么否决 TCC，可能重新调研踩坑。如果写"否决 TCC 是因为性能 P99 350ms 超标（业务要求 <200ms）"，未来就能判断：如果性能要求放宽（业务能接受 350ms），TCC 可以重新评估；如果性能要求更严（<100ms），saga 都不够，要找新方案。否决理由把"决策的边界条件"显性化，是 ADR 最有价值的部分。

**面试官**：架构图画了没人维护，很快就过期，怎么办？

**候选人**：核心是"图即代码"——架构图用 DSL（Structurizr/PlantUML）写在 Git 里，和代码同生命周期。CI 检查图和代码同步：扫描代码结构（Spring Bean、Feign client），对比 DSL 图，发现不一致告警（"代码新增 OrderQueryService 但图里没有"）。这样图改不了代码（CI 阻断），代码改了图也要改（CI 提醒），双向锁定。比 wiki 上的截图图（改代码忘改图）可靠得多。

**面试官**：C4 四层都要画吗？工作量大。

**候选人**：不一定全画。Context 和 Container 是必画的（给业务和运维沟通必需），Component 画核心服务（订单/支付这些复杂服务），Code 层一般不画（UML 用得少，代码本身就是"图"）。画图原则是"按需"——如果某层的信息在代码或注释里已经清楚，就不画图。图的价值是"沟通和沉淀"，不是为了画而画。一张 Context + 一张 Container + 核心服务的 Component，通常够用。

## 常见考点

1. **C4 模型是什么？**——Simon Brown 提出的架构图分层方法：Context（系统上下文，给业务看）、Container（容器=进程/服务/数据库，给运维看）、Component（服务内模块，给开发看）、Code（类图，给作者看）。核心是"按受众分层画"，像 Google Maps 一样分层缩放。
2. **ADR 四要素？**——Status（状态：Proposed/Accepted/Deprecated/Superseded）、Context（背景问题约束）、Decision（决策+替代方案否决理由）、Consequences（正面/负面影响、风险）。一页纸，入 Git 版本化。
3. **ADR 为什么必写替代方案否决理由？**——替代方案是决策的"反例"，让未来能判断假设是否还成立、是否要重新决策。不写否决理由，3 年后新人不知道当年为什么否决，可能重新踩坑。
4. **架构图怎么避免过期？**——图即代码（DSL 写在 Git），CI 检查图和代码同步（扫描代码结构对比图，不一致告警）。图和代码同生命周期，双向锁定。
5. **ADR 和代码注释区别？**——代码注释记"怎么做"（实现细节），ADR 记"为什么这么选"（决策上下文）。ADR 是架构级决策（选型、模式），注释是实现级细节（变量、逻辑）。两者互补不替代。

## 结构化回答

**30 秒电梯演讲：** 架构图和 ADR 是架构师的两支笔——架构图画给当下看（沟通工具），ADR 写给未来看（决策档案）。好的架构图分层（C4 模型：Context/Container/Component/Code），每层面向不同受众；好的 ADR 记为什么这么决策（Context/Decision/Consequences + 替代方案否决理由），让 3 年后的新人能理解决策上下文。两者一起构成架构可追溯性

**展开框架：**
1. **C4 模型** — Context（系统边界）/Container（进程/服务）/Component（模块）/Code（类）四层
2. **ADR 四要素** — Status（状态）/Context（背景）/Decision（决策+替代方案）/Consequences（代价收益）
3. **架构图分层画** — 高层给业务看，低层给开发看，不分层=没人看

**收尾：** 以上是我的整体思路。您想继续深入聊——C4 模型是什么？

# 【Java 后端架构师】线上容量红线与大促保障体系

> 适用场景：JD 核心技术。双 11/618 大促峰值流量是日常的 50-100 倍，没有容量评估就是赌博，没有预案就是事故。架构师必须能用容量公式算资源、用全链路压测验证、用预案体系兜底。

## 一、概念层：容量公式与三脚架

### 1.1 容量评估公式

```
峰值 QPS = 日均 QPS × 大促系数 × 时段系数

例：订单服务
  日均 QPS：5,000
  大促系数：双 11 = 30 倍（基于历史数据 + 今年 GMV 目标）
  时段系数：0 点峰值 = 10 倍（全天的集中度）
  → 峰值 QPS = 5,000 × 30 × 10 = 150 万 QPS

所需机器数 = (峰值 QPS × 安全系数) ÷ 单机容量
  安全系数：1.5-2（留余量应对突发）
  单机容量：压测得出（如单机 2000 QPS，P99 < 100ms）
  → 机器数 = (150w × 1.5) ÷ 2000 = 1,125 台
```

**安全系数为什么是 1.5-2**：业务可能超预期（GMV 超目标）、流量可能抖动（瞬间尖峰）、机器可能故障（部分宕机）。留 50-100% 余量是工程实践，太低（1.0）= 无冗余，太高（3+）= 浪费成本。

### 1.2 大促保障三脚架

```
        容量评估（提前算）
             │
             │
   ┌─────────┼─────────┐
   │         │         │
   ▼         ▼         ▼
压测验证   预案体系   监控告警
(实际打)   (兜底)    (实时看)
   │         │         │
   └─────────┼─────────┘
             │
             ▼
       大促当天保障
```

**三脚架缺一不可**：
- 容量评估算错 → 资源不够或浪费
- 压测不做 → 算的对不对不知道（理论值 vs 实际值）
- 预案没有 → 出事手忙脚乱
- 监控不全 → 出事不知道

## 二、机制层：全链路压测与容量红线

### 2.1 全链路压测架构（影子流量）

```
                    压测平台
                       │
                  生成影子流量
                  (带 X-Pressure: true Header)
                       │
                       ▼
                ┌────────────┐
                │  网关       │ ← 识别影子流量，透传 Header
                └──────┬─────┘
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │ 订单服务 │   │ 库存服务 │   │ 支付服务 │
   └────┬────┘   └────┬────┘   └────┬────┘
        │             │             │
        ▼             ▼             ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │ 影子库   │   │ 影子库   │   │ 影子库   │  ← 影子数据隔离
   │(pressure│   │(pressure│   │(pressure│     不污染真实数据
   │  _db)   │   │  _db)   │   │  _db)   │
   └─────────┘   └─────────┘   └─────────┘
```

**影子流量的实现**：
- 压测请求带特殊 Header（`X-Pressure: true`）
- 网关识别后透传，所有服务识别后路由到影子库
- 影子库是独立的数据库/表（`pressure_orders` vs `orders`），压测数据不污染生产
- 压测用户是虚拟用户（压测账号），不影响真实用户

### 2.2 容量红线（告警阈值）

```yaml
# 容量红线配置（每服务独立配置）
service: order-service
sla:
  p99_latency_ms: 200        # P99 < 200ms
  error_rate: 0.1%           # 错误率 < 0.1%
  success_rate: 99.9%        # 成功率 > 99.9%

capacity_watermarks:
  cpu:
    warning: 60%             # CPU > 60% 告警
    critical: 70%            # CPU > 70% 严重（红线）
    auto_scale: 75%          # CPU > 75% 自动扩容
  memory:
    warning: 70%
    critical: 80%
  disk:
    warning: 60%
    critical: 70%
  db_connection_pool:
    warning: 70%
    critical: 85%

traffic_watermarks:
  qps:
    current: 1500000         # 当前 QPS
    capacity: 1800000        # 容量（压测得出）
    utilization: 83%         # 利用率（current/capacity）
    warning: 80%             # 利用率 > 80% 告警
    critical: 90%            # 利用率 > 90% 限流
```

**红线的意义**：超过 critical 水位，系统在崩溃边缘（雪崩风险）。监控大盘实时显示水位，值班 SRE 一眼看清哪些服务告急。

## 三、实战层：预案体系与 Runbook

### 3.1 预案分类与分级

```
按动作分类：
┌─────────────────────────────────────────────┐
│ 1. 限流（Limit）—— QPS 超容量时主动拒绝    │
│    机制：Sentinel/网关限流，按接口/用户/租户│
│    生效：秒级                              │
├─────────────────────────────────────────────┤
│ 2. 降级（Degrade）—— 关非核心保核心        │
│    机制：feature flag / 配置中心开关       │
│    例：关推荐、关评论、关积分              │
│    生效：秒级（配置推送）                  │
├─────────────────────────────────────────────┤
│ 3. 扩容（Scale）—— 加机器提容量            │
│    机制：K8s HPA / 弹性扩容                │
│    生效：分钟级（拉镜像+启动+预热）        │
├─────────────────────────────────────────────┤
│ 4. 切流（Failover）—— 路由到备用机房       │
│    机制：DNS/GLB 切换                     │
│    生效：分钟级（DNS TTL + 健康检查）      │
└─────────────────────────────────────────────┘

按紧急度分级：
P0 预案（秒级生效）：限流、降级 —— 自动触发或 oncall 一键
P1 预案（分钟级生效）：扩容 —— oncall 决策
P2 预案（分钟级生效）：切流 —— 需主管批准（影响大）
```

## 四、底层本质：为什么大促是工程不是赌博

**容量评估的第一性是"用历史数据预测未来"**。大促峰值不是凭感觉——"我觉得今年双 11 会有 100w QPS"是赌博，"历史双 11 是日常的 30 倍 × 今年 GMV 目标增长 20% = 36 倍系数，日均 5w QPS × 36 = 180w QPS 峰值"是工程。历史数据 + 业务目标 + 安全系数 = 可量化的容量需求。

**为什么全链路压测不可省**：容量公式算的是理论值（单机 2000 QPS × 1000 台 = 200w QPS），但实际可能达不到——网络带宽瓶颈、DB 连接池打满、下游依赖拖累。全链路压测在生产环境用影子流量打一遍，才能发现真实瓶颈（"算的是 200w 实际只能扛 150w，瓶颈在 DB 连接池"）。压测是把"理论容量"校正为"实际容量"。

**为什么预案必须演练**：预案写在 wiki 上不等于能用。Chaos Engineering 的核心理念——主动注入故障验证系统韧性。演练发现的问题（脚本有 bug、人不熟练、依赖服务没准备好）在平时修，比大促当天手忙脚乱强 100 倍。Netflix 的 Chaos Monkey 每天随机 kill 生产节点，就是强制演练。

**为什么降级顺序要提前定**：大促当天资源紧张时，不可能临场决定"保什么弃什么"——决策延迟 + 情绪压力 = 错误决策。提前定降级顺序（核心交易 > 用户功能 > 个性化），大促当天只需"按预案执行"，不需要临场思考。这是把"紧急决策"前置为"预案准备"。

## 五、AI 架构师加问：5 个

1. **AI 怎么辅助容量评估？**
   AI 做流量预测——用历史大促数据 + 业务目标 + 外部信号（天气/热点/竞品活动）训练预测模型，输出峰值 QPS 的置信区间。AI 预测比人工估算准（考虑更多变量），但最终决策在人（业务方确认业务目标）。

2. **AI 怎么辅助预案执行？**
   AI 做异常检测（提前 1 分钟预测 CPU 将超红线，主动扩容而非被动触发）+ 预案推荐（"当前是库存瓶颈，建议执行限流 P0 预案"）。但执行需人工确认（oncall 一键），AI 不自动执行（影响大）。

3. **AI 能替代全链路压测吗？**
   不能完全替代。压测验证的是真实系统的瓶颈（含未知依赖），AI 只能基于已知模型预测，未知瓶颈（第三方服务抖动、网络分区）压测才能发现。AI 辅助生成压测脚本、分析压测结果、推荐优化点，不替代压测本身。

4. **大促值班，AI 怎么帮 oncall？**
   AI 做异常归因——告警发生时，AI 快速分析（"库存服务 P99 飙高，疑似下游 DB 慢查询，关联到 5 分钟前的配置变更"），给 oncall 候选解释和处置建议。AI 是辅助决策，oncall 最终决策和执行。

5. **怎么用 AI 预测"哪些服务大促可能出问题"？**
   历史数据训练：哪些服务历史上大促告急过、当前容量水位、依赖复杂度、最近变更频率。AI 综合给"大促风险评分"，高分服务重点保障（提前扩容、加预案、值班关注）。

## 六、记忆口诀与面试现场表达

### 1 分钟记忆口诀

抓 **"容量公式、全链路压测、容量红线、预案分级、必须演练"** 五个词。

- **容量公式**：(峰值 QPS × 安全系数 1.5-2) ÷ 单机容量 = 机器数
- **全链路压测**：影子流量 + 影子库，生产环境验证实际容量
- **容量红线**：CPU<70%、内存<80%、P99<SLA、错误率<0.1%
- **预案分级**：P0 限流降级（秒级）/ P1 扩容（分钟级）/ P2 切流（分钟级）
- **必须演练**：预案不演练=没有预案，Chaos 注入验证

### 拟人化理解

把大促保障想成 **春运火车站**。容量评估是"提前算客流"（公式），全链路压测是"试运行加车次"（验证算得对），容量红线是"候车室人数上限告警"（监控），预案是"客流超限时的应急措施"（限流进站/加开临客/分流到其他站）。春运当天不能等客流来了才准备——大促也一样，提前 2 个月算容量、1 个月压测、2 周演练。

### 面试现场 60 秒回答

> 大促保障是工程不是赌博。第一步容量评估——峰值 QPS = 日均 × 大促系数（历史 30 倍）× 时段系数（峰值 10 倍），机器数 = (峰值 × 安全系数 1.5) ÷ 单机容量。第二步全链路压测——生产环境用影子流量（带 X-Pressure Header）+ 影子库压测，验证容量算得对，找真实瓶颈。第三步容量红线——CPU<70%、内存<80%、P99<SLA、错误率<0.1%，超红线告警。第四步预案体系——P0 限流降级（秒级，自动触发）、P1 扩容（分钟级，oncall 决策）、P2 切流（分钟级，主管批准），每个预案有 Runbook（触发条件/执行动作/验证方法/回滚）。最关键：预案必须演练至少 2 轮，Chaos 注入验证，不能等大促当天才第一次执行。降级顺序提前定（核心交易 > 用户功能 > 个性化），大促当天按预案执行不临场思考。

### 现场对话示例

**面试官**：容量公式算的是理论值，实际达不到怎么办？

**候选人**：所以必须全链路压测校正。容量公式说单机 2000 QPS × 1000 台 = 200w QPS，但实际压测可能只能扛 150w——瓶颈可能在 DB 连接池（应用扩了 DB 没扩）、网络带宽（机房出口限制）、下游依赖（第三方服务限流）。压测就是把这些瓶颈暴露出来，校正"理论容量"为"实际容量"。压测发现 150w 时 DB 连接池打满，那就加 DB 连接池或加 DB 实例，再压到 200w。这是个迭代过程——压测 → 找瓶颈 → 优化 → 再压测，直到达到预期容量。

**面试官**：预案演练会不会影响线上业务？

**候选人**：演练要在低峰期（凌晨 2-4 点）做，影响最小。演练用影子流量（不影响真实用户），但预案执行（如限流）会短暂影响真实流量——所以演练限流预案时只限 1% 流量，验证"限流生效+恢复"的闭环，不真把流量限到影响业务。Chaos 注入（kill 节点）在多副本部署下影响可控（其他副本接管）。核心是"演练规模从小到大"——先演练 1% 流量，再演练 10%，验证预案可靠后再大促用。

**面试官**：大促当天库存服务挂了，怎么决策？

**候选人**：按预案分级决策。如果是 CPU/内存超红线（P0），自动触发限流降级（秒级生效），oncall 收到告警确认。如果限流后还不行（P1），oncall 决策扩容（K8s HPA 自动扩 + 手动加机器，分钟级）。如果整个机房挂了（P2），升级主管批准切流到备用机房（分钟级）。决策原则：保核心交易（下单/支付），弃非核心（推荐/评论）。降级顺序提前定（大促前评审过），当天按预案执行不临场思考。同时通知业务方（影响评估），战情室每 30 分钟同步状态。

## 常见考点

1. **容量公式怎么算？**——峰值 QPS = 日均 QPS × 大促系数（历史数据，如双 11 是日常 30 倍）× 时段系数（峰值时段集中度，如 0 点是日均 10 倍）。所需机器数 = (峰值 QPS × 安全系数 1.5-2) ÷ 单机容量（压测得出）。安全系数留余量应对突发。
2. **全链路压测怎么做？**——生产环境用影子流量（带 X-Pressure Header）+ 影子库（pressure_db 隔离压测数据），不污染真实数据。从 10% 预期峰值逐步打到 100%，找瓶颈（CPU/DB/网络/下游依赖），验证容量算得对。
3. **容量红线怎么设？**——CPU < 70%（留 30% 余量防雪崩）、内存 < 80%、磁盘 < 70%、P99 < SLA（如 200ms）、错误率 < 0.1%。超红线告警，超 critical 水位触发预案。
4. **预案怎么分级？**——P0 限流/降级（秒级生效，自动触发或 oncall 一键，如 Sentinel QPS 上限）、P1 扩容（分钟级，oncall 决策，K8s HPA）、P2 切流（分钟级，主管批准，DNS/GLB 切换备用机房）。
5. **预案为什么要演练？**——不演练的预案在真出事时执行不出来（脚本有 bug、人不熟练、依赖没准备好）。Chaos Engineering 理念：主动注入故障验证韧性。演练发现的问题在平时修，比大促当天手忙脚乱强。大促前至少演练 2 轮。

## 结构化回答

**30 秒电梯演讲：** 大促保障的核心是用容量公式提前算、用压测验证算得对、用预案兜底算错了。容量红线 = (峰值 QPS × 安全系数) ÷ 单机容量，安全系数 1.5-2 倍。预案要分类（限流/降级/扩容/切流）、分级（P0/P1/P2）、可演练（不能等真出事才第一次执行）。大促不是赌博，是工程

2. **全链路压测** — 生产环境压测（影子库/影子流量），验证容量
3. **容量红线** — CPU < 70%、内存 < 80%、P99 < SLA、错误率 < 0.1%

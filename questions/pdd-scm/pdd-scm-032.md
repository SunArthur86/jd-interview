# 【拼多多供应链】LLM 在商品/采购的具体应用？

> JD 依据："大模型业务探索"。

## 一、商品方向

**1. 标题生成**：
```
输入: {"品牌":"苹果","型号":"iPhone 15","规格":"256G 黑色"}
LLM 生成: "Apple iPhone 15 256GB 黑色 5G 全网通 新品"
```

**2. 属性抽取**：
```
输入: "这款华为 Mate60 Pro 12+512G 雅丹黑 卫星通话"
LLM 抽取: {"品牌":"华为","型号":"Mate60 Pro","内存":"12G","存储":"512G","颜色":"雅丹黑","卖点":"卫星通话"}
```

**3. 文案优化**：LLM 生成营销文案 + 违禁词过滤。

## 二、采购方向

**1. 谈判辅助**：
```
采购员输入诉求: "压价 5%，理由是竞品更便宜"
LLM 生成谈判邮件: "尊敬的 XX，关于近期采购合作..."
```

**2. 合同审查**：
```
LLM 抽取条款: 付款条件/违约责任/交期/质量标准
标注风险: "付款条件过于苛刻（预付 100%）"
人工复核
```

**3. 供应商沟通**：自动分类邮件 + 起草回复。

## 三、客服/工单

**工单分类**：
```
用户反馈: "买的手机壳颜色发错了"
LLM 分类: 售后-发错货 → 路由对应处理组
```

**知识问答**：RAG 检索帮助文档，自动回答。

## 四、成本与性能

```
日常文本（标题/工单）: 小模型（GLM-Flash），0.001 元/次
复杂场景（合同/谈判）: 大模型（GLM-Plus），0.05 元/次
缓存: 相似输入复用结果
```

## 五、底层本质

LLM 在供应链的价值是**"自动化文本脑力劳动"**：
- 商品文案：生成
- 属性抽取：理解
- 工单分类：理解
- 合同审查：理解+判断

这是 LLM 最成熟的应用方向——比决策类场景风险低、价值明确。

## 常见考点
1. **LLM 生成的商品文案合规吗**？——LLM 生成 + 违禁词词库过滤 + 人工抽检。
2. **怎么评估 LLM 效果**？——人工标注准确率 + 业务指标（文案点击率提升）。
3. **LLM 替代运营吗**？——辅助（提效），不替代（决策仍人做）。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：商品文案你用 LLM 生成，但供应商或运营可以手写。LLM 生成文案的价值是什么，比人写好在哪？**

价值在"规模 + 一致性 + 成本"：
1. **规模**——拼多多日新增千万 SKU，每个都要文案，人工写不过来（要几万个运营）。LLM 批量生成，一天出百万条。
2. **一致性**——人工文案风格各异（有的啰嗦有的简略），LLM 按统一 prompt 生成，风格一致（都符合"卖点突出 + 简洁"的品牌调性）。
3. **成本**——LLM 生成一条文案 0.001 元（GLM-Flash），人工写一条 1-5 元（运营时薪），成本降 1000 倍。
4. **多语言/多版本**——LLM 一键生成多语言版本（英文/东南亚语）和 A/B 测试版本（长版/短版），人工要重写。
代价是"质量参差"（LLM 偶尔生成不通顺或违规文案），需要违禁词过滤 + 人工抽检。但对"标准化描述"（如规格、卖点）足够。

### 第二层：证据与定位

**Q：运营反馈某批 LLM 生成的商品文案"违禁词"（如"最佳""第一"），被平台下架。你怎么定位是 prompt 没约束、违禁词库不全，还是 LLM 不遵守？**

三步排查：
1. **看 prompt**——检查文案生成的 prompt 是否包含"禁用绝对化用语（最佳/第一/最优）"的约束。如果 prompt 漏了，LLM 不知道要避开。
2. **看违禁词库**——`违禁词.txt` 或 DB 表是否包含触发的词（如"最佳"）。如果词库没这个词，过滤层漏过。广告法违禁词库要持续更新（监管新增词）。
3. **看 LLM 输出**——拉这批文案的 LLM 原始输出（生成日志），确认是否真的包含违禁词。如果 LLM 输出"最优质"，违禁词库没"最优质"（只有"最佳"），是词库不全。如果 LLM 输出"最佳"但 prompt 明确禁止，是 LLM 不遵循（幻觉）。
定位后修复：prompt 加强约束 + 词库补词 + LLM 输出后强制过滤（正则/字符串匹配，不依赖 LLM 自觉）。

### 第三层：根因深挖

**Q：LLM 抽取商品属性（从供应商文案提取"颜色/材质/规格"）准确率只有 85%，很多 SKU 抽错。根因是什么？**

根因通常是"输入质量差 + 任务定义模糊"：
1. **供应商文案乱**——供应商提供的文案格式不一（有的表格、有的纯文本、有的图文混排），LLM 解析难。解法是输入预处理（清洗 HTML、提取纯文本、结构化表格）。
2. **属性定义不明确**——"材质"可能写"纯棉""棉 100%""全棉"，LLM 不知道是否同一属性。解法是给 LLM 属性的候选值（如材质候选：纯棉/涤纶/混纺），让 LLM 选而不是自由生成。
3. **Few-shot 不足**——LLM 没有"正确抽取的示例"参考，格式输出不稳定。解法是 prompt 加 few-shot（3-5 个正确示例），让 LLM 学习输出格式。
4. **模型能力**——小模型（GLM-Flash）对复杂文本理解弱，准确率低。复杂抽取任务用大模型（GLM-Plus），准确率能到 95%+。
定位：抽错的样本分类（输入乱/属性模糊/模型弱），针对性修复。

**Q：那为什么不直接用规则/正则提取属性，确定性更高？**

正则适合"格式固定"的提取（如"颜色：红"这种 key-value 结构），但供应商文案是"自由文本"：
1. **格式多变**——"红色纯棉 T 恤""T 恤 红 色 纯棉""材质:棉 颜色:红"，正则要写无数个 pattern 覆盖，维护难。
2. **语义理解**——"透气舒适"隐含"材质是透气面料"，正则提取不出（没明确"材质"字段），LLM 能理解语义推断。
3. **新属性**——供应商突然写"抗菌"（新属性），正则没配过提取不出，LLM 能识别新属性（基于语义）。
所以正则适合"结构化字段"，LLM 适合"非结构化文本"。实际是"LLM 抽取 + 正则校验"组合——LLM 抽出候选，正则验证格式（如颜色必须是"红/蓝/绿"枚举），不合规的标人工复核。

### 第四层：方案权衡

**Q：合同审查你用 LLM 抽取条款（付款条件/违约责任/交期）+ 标注风险。但合同涉及法律风险，LLM 标注错了可能漏过风险条款。怎么权衡"LLM 自动化"和"法律安全"？**

法律场景必须"LLM 辅助 + 人工复核"：
1. **LLM 做初筛**——LLM 抽取所有条款，标注潜在风险（如"付款条件: 预付 100%"标"风险: 预付比例过高"），生成风险清单。人工只复核清单（不用通读合同全文），效率高。
2. **风险分级**——低风险（如交期条款小差异）LLM 自动通过，中风险人工抽检，高风险（如违约金 > 100 万）强制人工复核。LLM 不能直接放行高风险。
3. **可追溯**——LLM 标注的每条风险带"依据"（引用合同原文 + 判断理由），人工复核时能快速验证 LLM 是否标对。
4. **持续改进**——人工复核的结果（"这条 LLM 标错了"）反馈给 LLM 微调或 prompt 优化，持续提升准确率。
策略是"LLM 提效 + 人工兜底风险"，不是"LLM 全自动"。

**Q：为什么不直接用专业的合同审查软件（如 LawGeex、Kira），它们针对法律场景训练，比通用 LLM 准？**

专业软件和通用 LLM 各有优劣：
1. **专业软件准但贵**——LawGeex 等按合同量收费（如 50 元/份），千万级采购合同成本高。通用 LLM 便宜（0.05 元/份）。
2. **专业软件覆盖窄**——只擅长特定合同类型（如劳动合同），供应链的采购合同（框架协议/执行单）可能不在覆盖范围。通用 LLM 灵活，能处理各种合同。
3. **专业软件不灵活**——固定规则，新风险条款（如新增的"数据合规条款"）要等厂商更新。LLM 通过 prompt 调整快速适配。
所以策略是"通用 LLM 做初筛（覆盖广、成本低）+ 专业律师团队复核高风险合同"。只有合同量极大（如银行）才用专业软件，供应链用 LLM + 人工复核性价比更高。

### 第五层：验证与沉淀

**Q：你怎么证明 LLM 生成的文案/抽取的属性质量达标（准确率、合规率）？**

多维评估：
1. **人工标注准确率**——随机抽 1000 条 LLM 抽取的属性，人工标注对错，准确率应 > 95%（低于阈值调 prompt 或换大模型）。
2. **合规率**——LLM 生成的文案过违禁词过滤后，合规率应 100%（违禁词 = 0）。上线前全量过违禁词库，不合规的重生成。
3. **业务指标**——文案生成后的"商品详情页转化率"（用户点击/下单率）应不低于人工文案（A/B 测试）。如果 LLM 文案转化率低 20%，说明质量差（不够吸引人），调 prompt。
4. **持续监控**——上线后监控"用户投诉文案错误"的工单数，异常增多说明 LLM 质量退化（如模型版本更新），及时排查。

**Q：怎么让团队的 LLM 应用规范统一（prompt 管理、模型选型、评估标准）？**

沉淀 LLM 工程规范：
1. **Prompt 版本管理**——prompt 像 code 一样管理（Git 仓库），每次改 prompt 记录变更 + 评估结果（准确率是否提升），可回滚。
2. **模型选型矩阵**——按任务复杂度选模型：简单分类（工单）用 GLM-Flash（便宜），复杂生成（文案）用 GLM-Plus（质量高）。文档给选型指南。
3. **评估 benchmark**——每个 LLM 应用维护测试集（1000 条标注数据），prompt/模型变更时跑 benchmark，准确率退化不让上线。
4. **调用监控**——LLM 调用的 `latency`、`error_rate`、`cost`（token 消费）监控，超阈值告警（如成本突增可能是 bug 导致重复调用）。

# 【拼多多供应链】怎么用 AI 代码助手改造研发流程？

> JD 依据：JD 8"AI/LLM"标签，AI 编程是转型基础。

## 一、工具组合

| 工具 | 类型 | 适用 |
|------|------|------|
| Cursor | IDE 集成 | 日常开发（补全+Agent） |
| Claude Code | 终端 Agent | 自动化、批量改、CI/CD |
| Copilot | 插件 | 轻量补全 |

## 二、供应链实战场景

**1. 多文件重构**：
```
Prompt: "@codebase 把供应链项目里所有 Date 改成 LocalDateTime，
        保持格式，加 import，mvn compile 验证"
Cursor 自动遍历修改+编译验证
```

**2. 调试**：
```
Claude Code: 读 error log → 定位 → 修复 → 编译验证
```

**3. 测试**：
```
Prompt: "@StockService.deduct 写单元测试，覆盖正常/库存不足/并发场景"
```

**4. 遗留代码理解**：
```
Prompt: "解释这个 10 年前的库存引擎类逻辑"
Cursor: 总结+标注复杂点+建议优化
```

## 三、提效数据

- 编码时间 70% → 30%
- 单测编写大幅减少
- 文档几乎零成本
- 整体效率 +30-50%

## 四、必须人工 Review

AI 会：
- 改错关联文件（升级 Spring Boot 漏改配置）
- 误删"看起来没用"的代码（降级开关）
- 不懂业务规则改错阈值

**教训**：AI 改完必须逐行 review + 编译 + 测试。

## 五、供应链特殊坑

- 敏感规则/价格算法：用企业版（私有部署）
- 关联文件遗漏：必须编译验证
- 配置误删：关键配置加注释

## 六、底层本质

AI 编程是**"意图到代码翻译的自动化"**：
- 1950s：机器码
- 1970s：高级语言
- 2000s：IDE
- 2020s：自然语言→代码（AI）

让程序员聚焦"想什么"而非"怎么敲"。

## 常见考点
1. **AI 代码助手能替代程序员吗**？——不会，会用 AI 的取代不会用的。
2. **AI 代码最大问题**？——理解不深（关联遗漏）+ 业务不懂 + 边缘 case 遗漏。
3. **敏感代码怎么用 AI**？——企业版私有部署 + 关键逻辑人工写。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你用 AI 代码助手（Cursor/Claude Code）改造研发流程，提效 30-50%。但供应链的核心代码（库存扣减/价格算法）涉及业务规则和敏感数据，为什么敢用 AI 改？**

核心代码不用 AI 改，只让 AI 改"标准化、低风险"的代码：
1. **AI 改的**——单元测试、CRUD 模板代码、遗留代码理解、批量重构（如 Date→LocalDateTime）、文档生成。这些是"规则明确 + 可验证（编译/测试）"的，AI 改完跑测试就能确认对错。
2. **人工写的**——库存扣减逻辑、价格算法、促销规则、风控策略。这些涉及"业务语义 + 边界判断 + 合规"，AI 不懂业务（如"预售允许超卖"这种规则），容易改错。必须人写。
3. **敏感数据隔离**——供应链的价格算法、供应商信息是商业机密，不能用云端 AI（数据泄露风险）。用企业版私有部署（如自建 CodeGeeX/Copilot Enterprise），代码不出内网。
所以策略是"AI 辅助 + 人工把关核心"，不是全盘 AI 化。

### 第二层：证据与定位

**Q：AI 批量重构（Date→LocalDateTime）后，某服务启动失败（ClassNotFoundException）。你怎么定位是 AI 改漏了文件、改错了 import，还是编译配置问题？**

三步排查：
1. **看 AI 的 diff**——AI 重构会生成 diff（改了哪些文件），review diff 看是否有遗漏（如改了 Service 没改对应的 DTO）。如果某文件还用 Date 但 import 被删了，是 AI 遗漏。
2. **看编译错误**——`mvn compile` 的错误信息会明确指出哪个文件/类找不到。如果是 `java.lang.ClassNotFoundException: java.util.Date` 但代码用了 `Date`，是某文件没改成 LocalDateTime 但 import 删了。
3. **看关联影响**——Date→LocalDateTime 不只是改类名，还要改序列化（Jackson 注解）、数据库映射（MyBatis TypeHandler）、前端 API 格式。如果只改了类名没改序列化，运行时反序列化失败。AI 可能漏改这些关联点。
定位：用 `grep -r "Date" src/` 全局搜，确认是否还有 Date 残留；编译 + 单测全跑，找所有报错点。

### 第三层：根因深挖

**Q：AI 改完后单测全绿、编译通过，但线上发现 AI 把一个"看起来没用的降级开关"删了，导致故障时没降级。根因是什么？**

根因是"AI 不懂业务语义，误删看似无用的代码"：
1. **降级开关看起来没用**——`if (ConfigCache.get("degrade_xxx", false)) return fallback;` 在平时（开关关）不执行，AI 分析时可能判断为"死代码"删除。但这个开关是"故障时的应急降级"，平时不用但关键时刻必须存在。
2. **AI 删的理由**——AI 的 diff 说明可能是"移除未使用的条件分支"或"简化逻辑"。AI 不理解"这是运维开关，不是业务逻辑"。
3. **单测覆盖不到**——单测通常测正常路径（开关关），不测"开关开的降级路径"（要 mock 配置中心），所以单测绿但降级丢了。
根因：AI 不能区分"业务死代码"和"运维开关代码"。预防：
- 关键配置/开关加注释 `// 运维开关，勿删`，AI 看到注释会保留。
- Code Review 重点看 AI 删的代码（diff 里 `-` 的行），确认不是误删。
- 降级路径也要单测覆盖（mock 开关开的场景），CI 强制。

**Q：那为什么不直接禁止 AI 删除代码（只允许新增/修改），彻底避免误删？**

禁止删除会让 AI 重构不彻底：
1. **重构要删旧代码**——Date→LocalDateTime 要删旧的 `import java.util.Date`，禁止删除会导致 import 冗余/冲突。
2. **优化要删低效代码**——如果 AI 要替换"循环拼接字符串"为 StringBuilder，要删旧的循环，禁止删除会导致两套代码并存。
所以不能一刀切禁止删除，要"允许删除 + 严格 review"。策略：
- AI 的 diff 里所有删除行（`-`）必须人工 review，确认不是业务/运维代码。
- 关键代码（开关/配置/降级）加注释标记，AI 看到标记保留。
- Code Review 的红线：AI 删的代码如果涉及 `if/switch/配置读取`，强制人工确认。

### 第四层：方案权衡

**Q：AI 写单元测试很快，但单测质量参差（有的 case 覆盖不全）。怎么权衡"AI 写测试快"和"测试有效性"？**

AI 写测试要"快 + 人工补强"：
1. **AI 写基础测试**——AI 能快速生成"正常路径 + 基本边界"的测试（如 `deduct(库存够)` + `deduct(库存不够)`），覆盖 70% 的 case。这比人工从零写快 10 倍。
2. **人工补边界**——复杂的边界 case（如并发扣减、幂等、超时降级）AI 容易漏，人工补充。AI 生成的测试当作"起点"，不是"终点"。
3. **覆盖率检查**——CI 强制覆盖率（如 > 80%），AI 写完跑覆盖率，没覆盖的分支人工补。用 JaCoCo 看哪些分支没覆盖。
4. **变异测试**——用 PIT 等工具做变异测试（改代码看测试能否发现），AI 写的测试如果"变异后仍通过"（说明测试没真正验证），要人工强化。

**Q：为什么不直接让人工写所有单测（质量可控），而是用 AI 再人工补？**

人工写单测的成本高：
1. **时间**——人工写一个 Service 的完整单测要几小时（覆盖所有分支），AI 几分钟生成基础版本。时间差 10 倍。
2. **枯燥**——单测是"低创造性劳动"（写 case + 断言），程序员不愿意写（往往拖延或敷衍），AI 代劳让人聚焦业务逻辑。
3. **一致性**——人工写的单测风格各异（命名/结构不一），AI 按统一模板生成，一致性好。
所以策略是"AI 写 70% 基础 + 人工补 30% 边界"，总体效率提升 + 质量可控。纯人工写慢且易偷工减料，纯 AI 写覆盖不全，两者结合最优。

### 第五层：验证与沉淀

**Q：你怎么证明 AI 代码助手真的提效了（不是主观感觉）？**

量化指标：
1. **开发效率**——对比"AI 辅助前 vs 后"的"单需求开发周期"（从领需求到上线）。如库存优化需求，AI 前平均 5 天，AI 后 3 天，提效 40%。
2. **代码质量**——AI 生成代码的 bug 率（上线后 bug 数 / 代码行数），应不高于人工代码（否则 AI 没意义）。用 SonarQube 扫 AI 代码的 code smell。
3. **单测覆盖率**——AI 写单测后，项目整体覆盖率应提升（如从 50% 到 80%），且变异测试通过率 > 80%。
4. **研发满意度**——问卷调研团队对 AI 工具的满意度（是否愿意继续用），主观感受是采纳率的关键。

**Q：怎么让团队规范使用 AI 代码助手（提效但不引入风险）？**

沉淀规范：
1. **使用指南**——明确"AI 可以改什么（测试/模板/重构）+ 不能改什么（核心业务/敏感逻辑）"，新人培训必读。
2. **Review 红线**——AI 生成的代码必须逐行 review，重点看"删除的行"和"改动的业务逻辑"。关键路径（库存/支付）的 AI 代码要架构师 + 业务专家双 review。
3. **敏感代码隔离**——价格算法/风控规则/供应商信息等敏感代码，用企业版私有 AI 或禁止 AI 接触（物理隔离的代码库）。
4. **AI 代码标注**——commit message 标注"AI-assisted"，便于追溯（如果出 bug，知道是 AI 改的，针对性 review）。
5. **持续培训**——团队定期分享"AI 使用技巧 + 踩坑案例"（如某次 AI 误删降级开关的教训），提升整体使用水平。

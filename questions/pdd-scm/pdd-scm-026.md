# 【拼多多供应链】高并发五件套怎么用？

> JD 依据："高并发系统的开发和调优"。

## 一、五件套详解

### 1. 池化
```java
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    64, 128, 60, SECONDS,
    new LinkedBlockingQueue<>(2000),  // 有界！
    new ThreadFactoryBuilder().setNameFormat("scm-%d").build(),
    new CallerRunsPolicy());
```

### 2. 缓存（多级）
```java
Product p = caffeine.getIfPresent(id);          // L1 本地（50% 命中）
if (p == null) p = redis.get("p:" + id);         // L2 分布式（95% 命中）
if (p == null) { p = mysql.findById(id); redis.set(...); }
```

### 3. 扩容
```yaml
# K8s HPA
minReplicas: 50
maxReplicas: 500
metrics: [{ type: Resource, resource: { name: cpu, target: { averageUtilization: 60 }}}]
```
前提：服务无状态（状态外置 Redis/HBase）。

### 4. 异步
```java
// 并行查询
CompletableFuture<Stock> sf = supplyAsync(() -> stockService.get(sku), pool);
CompletableFuture<Price> pf = supplyAsync(() -> priceService.get(sku), pool);
CompletableFuture.allOf(sf, pf).get(50, MS);  // 并行，50ms 超时

// MQ 异步
kafka.send("order-created", event);  // 不阻塞主流程
```

### 5. 队列
```
大促订单洪峰 → Kafka（入队）→ 消费者稳定处理（削峰）
```

## 二、供应链组合应用

```
商品详情页查询：
  池化（连接池）+ 缓存（Caffeine+Redis 98%命中）+ 异步（并行查库存/价格/评价）
  → 单接口 RT 从 200ms → 30ms
```

## 三、底层本质

吞吐量 = 并发数 / 处理时间。五件套：
- 分子（并发）：池化 + 扩容
- 分母（时间）：缓存 + 异步
- 削峰（突发）：队列

## 常见考点
1. **缓存一致性**？——Cache Aside（先 DB 后删缓存）+ 延迟双删。
2. **池化为什么必须上界**？——无界导致 OOM（Executors 的坑）。
3. **异步化代价**？——失去强一致，需最终一致补偿。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：高并发五件套（池化/缓存/扩容/异步/队列）你都用上了，但它们的优先级是什么？资源有限时先上哪个？**

优先级按"ROI（性价比）从高到低"：
1. **缓存（最高 ROI）**——命中率 95% 意味着 95% 请求不打 DB，立竿见影降延迟和 DB 压力。成本低（Redis 集群），效果大。
2. **池化**——线程池/连接池避免频繁创建销毁，是基础设施，必须配。
3. **异步**——把同步调用（Feign RPC）改异步（MQ），解耦 + 削峰，但引入最终一致复杂度，改造量大。
4. **队列**——削峰填谷，但引入消息延迟，适合"容忍延迟"的场景（如下单后通知物流）。
5. **扩容（最后）**——加机器是"暴力解"，成本高（机器钱），且不能解决单点瓶颈（如热点 key）。只有前四件套用尽还扛不住，才扩容。
所以策略是"缓存 + 池化打底，异步 + 队列解耦，扩容兜底"，不是一上来就加机器。

### 第二层：证据与定位

**Q：商品详情页接口 RT 从 30ms 涨到 200ms，你用了五件套但性能还是退化。怎么定位是哪件套失效？**

逐层排查（从快到慢）：
1. **看缓存命中率**——Caffeine `cache.hitRate()` 和 Redis `hit_rate`，如果从 98% 降到 80%，缓存失效（可能是 TTL 太短或 key 设计问题，大量 miss 打 DB）。这是最常见原因。
2. **看池化状态**——线程池 `pool.active` 和连接池 `druid.active`，如果都满（排队），是池子太小或下游慢。
3. **看异步超时**——CompletableFuture 的 `get(50, MS)` 如果频繁超时（超时计数 > 0），是某个异步任务（如查评价）慢，拖累整体。arthas trace 看哪个 future 慢。
4. **看队列积压**——如果详情页依赖某个 MQ 消费的结果（如实时销量），`consumer_lag` 大说明消费跟不上，数据延迟导致接口等数据。

### 第三层：根因深挖

**Q：缓存命中率从 98% 降到 80%，根因是什么？**

几种可能：
1. **缓存预热失败**——大促前预热脚本没跑或跑了但 key 策略变了（如从 `product:{id}` 改成 `p:{id}`），缓存空的，大量 miss。查预热日志。
2. **TTL 太短**——某次优化把 TTL 从 30 分钟调成 5 分钟（减少一致性窗口），命中率骤降（频繁回源）。查 Caffeine 配置的 `expireAfterWrite`。
3. **热点 key 集中失效**——某类目下所有商品的缓存同时过期（如都设 10:00 过期），10:00 时大量 miss 打 DB（雪崩）。根因是 TTL 没加随机抖动（`TTL + random(60)`）。
4. **缓存被穿透**——大量请求查不存在的 id（如爬虫扫 id），布隆过滤器没配或失效，每个请求打 DB。查访问日志的 id 分布。
根因定位：看 Redis 的 `miss_count` 和访问日志，判断是"全量 miss"（预热问题）还是"特定 id miss"（穿透问题）还是"时间集中 miss"（雪崩）。

**Q：那为什么不直接把缓存 TTL 设成无限（永不过期），彻底避免 miss？**

永不过期引入"数据不一致"——商品改价后缓存还是旧价，用户看到错误价格。折中方案：
1. **逻辑过期 + 异步刷新**——缓存不设 TTL（物理永不过期），但 Value 里带 `expire_at` 字段。读请求发现 `expire_at` 过了，返回旧值同时异步触发刷新（后台更新缓存）。用户永远拿到缓存（不 miss），数据滞后一个刷新周期（如 5 分钟）。
2. **分级 TTL**——热 key（top 1 万）长 TTL（1 小时）+ 定时刷新；冷 key 短 TTL（5 分钟）。用有限的内存最大化命中率。
3. **最终一致兜底**——Canal 监听 binlog 主动刷新缓存（改价秒级生效），不依赖 TTL 过期。
所以不是"永不过期 vs TTL"，而是"主动刷新 + 逻辑过期"结合，既不 miss 又保一致。

### 第四层：方案权衡

**Q：异步化（CompletableFuture 并行查库存/价格/评价）你设了 50ms 超时。但评价服务偶尔慢（300ms），超时后走降级（不显示评价）。用户反馈"评价时有时无"，怎么办？**

超时降级的体验问题，权衡方案：
1. **分级超时**——核心数据（库存/价格）超时 50ms（必须快），非核心（评价/推荐）超时 200ms（容忍慢）。评价偶慢 300ms 超 200ms 才降级，减少降级频率。
2. **骨架屏/懒加载**——评价接口异步加载，详情页先渲染（商品信息完整），评价区域显示"加载中"，1 秒内返回就填充，超时显示"暂无评价"。用户感知更平滑（不是"时有时无"而是"稍后加载"）。
3. **降级体验优化**——降级时显示"部分评价"（本地缓存的最近评价）而不是空，至少有内容。

**Q：为什么不直接去掉超时，等所有异步任务都返回再响应（保证数据完整）？**

去掉超时的风险是"木桶效应"——整体 RT 由最慢的任务决定。评价服务慢 300ms，整个详情页就要 300ms+，大促时用户感知卡顿。超时降级是"核心数据优先 + 非核心可降级"，保证核心体验（看到商品和价格）快速响应。代价是非核心（评价）偶有缺失，但比"整体卡顿"体验好。电商详情页的标准做法是"核心同步 + 非核心异步懒加载"，不是全部同步等齐。

### 第五层：验证与沉淀

**Q：你怎么证明五件套的组合真的把接口 RT 从 200ms 降到 30ms、不是某次流量低碰巧？**

对比验证 + 归因分析：
1. **A/B 验证**——同流量下，开了五件套的版本 RT=30ms，关了缓存的版本 RT=200ms，直接对比（shadow 流量双跑）。
2. **逐项归因**——逐个关掉五件套（关缓存、关异步、降线程数），看 RT 变化，量化每件套的贡献。如：关缓存 RT 涨到 150ms（缓存贡献 120ms），关异步再涨到 200ms（异步贡献 50ms）。
3. **压测验证**——恒定 QPS 压测，RT 稳定在 30ms（P99），说明是五件套效果不是流量波动。

**Q：怎么让团队规范使用五件套，而不是各自为政？**

沉淀规范 + SDK：
1. **缓存 SDK**——`@Cacheable(key="product:{id}", ttl=1800, jitter=60)` 注解，统一多级缓存（Caffeine + Redis）+ 随机 TTL + 互斥重建，业务方加注解即生效。
2. **异步框架**——封装 `AsyncExecutor`（预配置线程池 + 超时 + 降级），`AsyncExecutor.parallel(stockFuture, priceFuture).get(50, MS, fallback)`，不裸用 CompletableFuture。
3. **容量规范**——每个服务的线程池/连接池/缓存容量按"压测峰值 × 1.5"配置，写入 Nacos 动态可调。扩容阈值（HPA）按 CPU/QPS 自动触发，不靠人工。
4. **Code Review checklist**——所有外部调用（DB/Redis/RPC）必须有超时 + 降级；所有缓存必须有 TTL + 上限；所有线程池必须有界 + 命名 + 监控。

## 结构化回答

**30 秒电梯演讲：** 单机处理能力有限，如何在不无限堆硬件下应对突发流量？简单说就是——高并发五件套——池化（复用资源）、缓存（少算少 IO）、扩容（加机器）、异步（不阻塞）、队列（削峰），分别从复用/减少/扩展/不等待/缓冲五个角度提升吞吐。

**展开框架：**
1. **池化** — 池化：线程池/连接池/对象池（必须有界）
2. **缓存** — 缓存：Caffeine → Redis → MySQL 多级
3. **扩容** — 扩容：垂直（升硬件）vs 水平（加机器，无状态）

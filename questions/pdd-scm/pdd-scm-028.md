# 【拼多多供应链】负载均衡、隔离、压测怎么做？

> JD 依据："高可用、高并发环境下的研发"。

## 一、负载均衡

```
DNS → VIP（LVS）→ Gateway → 服务实例
```

**算法**：
- 轮询、随机（简单）
- 最少连接（处理时长不均）
- 一致性哈希（缓存场景）
- 同机房优先（减少跨机房 RT）

## 二、隔离

```
物理级（最强）：不同业务不同集群
集群级：K8s namespace
线程级：每下游独立线程池
信号量级：计数隔离
```

**供应链隔离**：
```java
@Bean("corePool") Executor corePool() { ... }   // 下单/库存（核心）
@Bean("auxPool") Executor auxPool() { ... }      // 推荐/评价（非核心）
```

## 三、全链路压测

```
1. 录制线上流量
2. 构造影子流量（× N 倍）
3. 影子库隔离（不污染生产）
4. 全链路监控
5. 找瓶颈 → 扩容 → 验证
```

**容量规划**：
```
日常 5万 QPS，双 11 峰值 100万
冗余 1.5 倍 → 设计容量 150万
单机 5000 QPS → 300 实例
```

## 四、底层本质

三者是分布式系统运维基础：
- 均衡：效率最大化（避免热点）
- 隔离：稳定性保障（爆炸半径）
- 压测：容量可知（数据驱动）

## 常见考点
1. **一致性哈希**？——节点增减只影响相邻段（缓存场景减少 miss）。
2. **线程池隔离 vs 信号量**？——线程池强（独立资源）开销大；信号量轻共享线程。
3. **压测怎么不打扰用户**？——影子流量 + 影子库 + 影子 MQ。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：负载均衡你用轮询（RoundRobin），但大促时发现某些实例 CPU 90%、另一些 30%。轮询不是均匀分配吗，为什么不均衡？**

轮询"请求数均匀"但"负载不均匀"——不同请求的耗时不同（查爆款商品 100ms，查冷门商品 5ms），轮询把请求均匀分配，但慢请求恰好集中到某实例，该实例 CPU 高。根因是"请求耗时差异大"。解决：
1. **加权响应时间（WRR）**——LB 根据实例的历史 RT 动态调权重，RT 长的实例少分配请求。Nacos 的 `WeightedRule` 支持。
2. **最少连接（LeastConnections）**——LB 把请求发给"当前活跃连接数最少"的实例，天然均衡负载（慢实例连接多，少分配）。
3. **一致性哈希**——按用户/skuId 哈希到固定实例，把"热点 sku 的查询"分散（不同 sku 不同实例），避免单实例热点。

### 第二层：证据与定位

**Q：大促压测时某服务（订单服务）RT 暴涨，但单实例 RT 正常。你怎么定位是 LB 问题、实例间不均衡，还是连锁反应？**

逐层排查：
1. **看实例负载分布**——Prometheus 看 `instance_cpu` 和 `instance_qps`，如果某实例 CPU 90% QPS 1000、另一实例 CPU 30% QPS 200，是 LB 不均衡（请求没均匀分配）。
2. **看 LB 策略**——检查 Nacos/Gateway 的 LB 配置，如果是 RoundRobin 但实例健康检查失效（某实例假死但仍被路由），慢实例拖累。
3. **看依赖**——如果各实例负载均衡但 RT 暴涨，可能是下游（如 DB）成瓶颈，所有实例都在等 DB，RT 都涨。看 DB 的 `SHOW PROCESSLIST` 和 `slow_query`。

### 第三层：根因深挖

**Q：定位发现 LB 正常（请求均匀），但某实例 RT 比其他高 5 倍。根因是什么？**

单实例慢的根因通常是"环境差异"或"本地状态"：
1. **GC 差异**——某实例恰好频繁 Full GC（jstat -gcutil 看 FGC），处理暂停。其他实例 GC 正常。根因可能是该实例的缓存或对象分配异常（如本地缓存更大）。
2. **JIT 未预热**——新扩容的实例刚启动，JIT 还没编译热点代码（解释执行慢），RT 高。压测前要预热（发空请求触发 JIT）。
3. **网络/磁盘差异**——某实例所在宿主机网络抖动（跨机房调用慢）或磁盘 IO 满（日志写慢）。看该实例的 `network_latency` 和 `disk_io`。
4. **连接池差异**——某实例的 DB/Redis 连接池配置不同（如连接数少），排队等待。看 `druid.pool.active` 和 `maxActive`。
根因定位：对比慢实例和正常实例的 jstack/jstat/网络指标，找差异点。

**Q：那为什么不直接把慢实例从 LB 摘掉，用正常的？**

摘掉慢实例是临时止血，但要小心：
1. **容量减少**——摘掉 1 个实例，剩余实例要扛更多 QPS，可能连锁打挂。
2. **根因没解**——慢实例重启后可能还是慢（如果是配置或宿主机问题）。
正确做法：先定位根因（GC/JIT/网络），如果是配置问题（如连接池小）远程修复；如果是宿主机问题（网络/磁盘），迁移实例；确认根因后再决定是否重启。摘实例只在"确认有冗余容量且根因不可远程修复"时做。

### 第四层：方案权衡

**Q：隔离你用线程池隔离（核心业务用 corePool，非核心用 auxPool）。但线程池多了，线程数膨胀（corePool 200 + auxPool 200 = 400 线程），内存和上下文切换开销大。怎么权衡？**

线程池隔离 vs 信号量隔离的权衡：
1. **线程池隔离（强隔离，开销大）**——核心和非核心各自线程池，非核心挂了（auxPool 满）不影响核心（corePool 独立）。代价是线程多（400 个 × 1MB 栈 = 400MB 内存），上下文切换开销。
2. **信号量隔离（轻量，共享线程）**——所有请求共享主线程池（如 Tomcat 线程），用信号量计数限制非核心的并发（如 auxService 最多 50 并发）。代价是非核心超限会占用主线程（隔离性弱）。
3. **混合策略**——核心链路（下单/支付）用线程池隔离（强保护），非核心（推荐/评价）用信号量（轻量）。corePool 100 线程（核心专用），非核心走信号量限并发，总线程数控制。

**Q：为什么不直接用物理隔离（核心和非核心不同集群）？**

物理隔离是最强的（不同机器、不同 DB），但成本高：
1. **机器成本**——要维护两套集群（核心集群 + 非核心集群），机器数翻倍。
2. **运维复杂**——两套集群的部署、监控、扩缩容都要独立管理。
3. **资源利用率低**——非核心集群平时空闲（夜间无推荐流量），核心集群忙，资源不能共享。
所以策略是"核心链路物理隔离（保绝对安全）+ 非核心逻辑隔离（线程池/信号量，共享资源省成本）"。只有"绝对不能挂"的核心（如支付）才物理隔离，其他逻辑隔离够。

### 第五层：验证与沉淀

**Q：全链路压测你怎么保证"影子流量不打扰真实用户"又不失真？**

影子流量的关键是"隔离但真实"：
1. **影子库**——压测流量写影子库（如 `order_shadow` 表），不写生产库。但影子库的数据量要和生产一致（否则压测不真实）。用 TPC（数据克隆）或 binlog 同步保持影子库和生产库数据量一致。
2. **影子 MQ**——压测消息发到影子 topic（`order_shadow_events`），不影响真实消费。但消费者的处理逻辑要和生产一致（同样的代码，不同的 topic）。
3. **影子缓存**——压测流量读写影子 Redis key（`shadow:product:{id}`），不污染生产缓存。但命中率要模拟真实（不能让影子 key 把热 key 挤出缓存）。
4. **流量标记**——压测请求 header 带 `x-pressure-test: true`，各中间件（DB/Redis/MQ）识别后路由到影子资源。统一 SDK 自动加标记，业务无感。

**Q：怎么让团队的负载均衡/隔离/压测配置规范统一？**

沉淀规范：
1. **LB 策略规范**——核心服务用"加权响应时间"（动态均衡），缓存场景用一致性哈希（减少 miss），默认轮询（简单场景）。新服务上线选 LB 策略要 review。
2. **隔离规范**——核心链路（下单/支付）强制线程池隔离，非核心用信号量。线程池配置（core/max/queue）按压测容量预设，禁止裸用。
3. **压测准入**——大促前所有核心服务必须过全链路压测（影子流量 × 2 倍峰值），找出瓶颈并扩容。压测报告（瓶颈点、扩容后 QPS/RT）作为上线依据。
4. **容量画像**——每个服务的容量画像（单机 QPS/RT/资源占用）维护在 wiki，扩缩容参考，不拍脑袋。

## 结构化回答

**30 秒电梯演讲：** 多机如何流量均衡、业务互不影响、容量可知？简单说就是——负载均衡分散流量（避免热点）、隔离限制爆炸半径（所以障不传染）、压测验证容量上限（数据驱动扩容），三者让系统均衡分布、所以障可控、容量可知。

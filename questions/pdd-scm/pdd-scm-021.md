# 【拼多多供应链】设计拼多多级供应链架构（千万商品、亿订单）

> JD 依据："主导系统架构设计"。这是 P7+ 架构题。

## 一、整体架构

```
┌──────────────────────────────────────────────────────────────┐
│                    接入层（网关、限流）                       │
├──────────────────────────────────────────────────────────────┤
│                    业务层（App/H5/小程序）                   │
├──────────────────────────────────────────────────────────────┤
│                    中台能力层                                │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐    │
│  │商品中心│ │库存中心│ │订单中心│ │营销中心│ │结算中心│    │
│  └────────┘ └────────┘ └────────┘ └────────┘ └────────┘    │
├──────────────────────────────────────────────────────────────┤
│                    领域服务层（供应链专属）                   │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐    │
│  │采购服务│ │仓储服务│ │物流服务│ │供应商  │ │对账服务│    │
│  └────────┘ └────────┘ └────────┘ └────────┘ └────────┘    │
├──────────────────────────────────────────────────────────────┤
│                    数据中台                                  │
│   Flink 实时（销量/库存预警）+ Spark 离线（T+1 报表）        │
├──────────────────────────────────────────────────────────────┤
│                    基础设施（K8s + 多机房 + 多仓）           │
└──────────────────────────────────────────────────────────────┘
```

## 二、中台化设计

**商品中心**：服务所有业务线（自营/第三方/跨境），统一 SPU/SKU 模型。
**库存中心**：统一现货/预售/在途/冻结模型，多仓库存统筹。
**订单中心**：统一订单状态机，适配不同业务（普通/拼团/预售）。

**复用价值**：新业务线（如多多买菜）接入只需配业务规则，复用中台能力。

## 三、异步化（领域事件）

```
下单 → OrderCreatedEvent
   ├─ 库存中心：冻结库存
   ├─ 营销中心：核销优惠券
   └─ 风控：异步复核

出库 → ShippedEvent
   ├─ 结算：生成账单
   ├─ 物流：跟踪运单
   └─ 用户：发货通知
```

本地消息表保证事件不丢，下游幂等消费。

## 四、多仓多活

```
华东仓（上海）─ 服务华东用户（次日达）
华南仓（广州）─ 服务华南用户
华北仓（北京）─ 服务华北用户
跨境仓（保税区）─ 跨境业务

下单时：按用户地址 + 商品库存位置 → 智能分仓
```

**分仓算法**：优先本地仓 → 备选邻近仓 → 跨区调拨（成本最高）。

## 五、容量估算

```
商品：千万级 SPU，亿级 SKU
订单：日均亿级，峰值百万 QPS
库存：亿级 SKU × 多仓 = 百亿 cell
GMV：日百亿级

存储：MySQL 分库分表 + HBase（明细）+ ES（搜索）+ Redis（缓存）
计算：Flink（实时）+ Spark（离线）
```

## 六、大促保障

1. **提前备货**：大促前 1 月按预测备货到各仓
2. **弹性扩容**：K8s HPA 大促前扩 3 倍
3. **降级预案**：非核心（推荐/评价）异步或关闭
4. **全链路压测**：影子流量验证容量

## 七、底层本质

供应链架构本质是**"用复用（中台）+ 解耦（异步）+ 就近（多仓）应对规模和地域复杂性"**。

**钱学森工程理论体现**：
- **技术上可行**：用成熟组件（Spring Cloud/Flink/MySQL）
- **经济上合理**：中台复用降低边际成本；多仓降低物流成本
- **研制周期短**：中台让新业务快速接入
- **能协调运转**：领域事件让各环节松耦合协同

## 常见考点
1. **中台和微服务区别**？——中台是业务能力复用（战略），微服务是技术拆分（战术）。
2. **多仓库存怎么统筹**？——库存中心统一视图（各仓库存+在途+冻结），分仓独立扣减。
3. **供应链最难的是什么**？——一致性（跨服务）+ 时效（物流）+ 成本（仓储/物流优化）。

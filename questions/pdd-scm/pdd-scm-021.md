# 【拼多多供应链】设计拼多多级供应链架构（千万商品、亿订单）

> JD 依据："主导系统架构设计"。这是 P7+ 架构题。

## 一、整体架构

```
┌──────────────────────────────────────────────────────────────┐
│                    接入层（网关、限流）                       │
├──────────────────────────────────────────────────────────────┤
│                    业务层（App/H5/小程序）                   │
├──────────────────────────────────────────────────────────────┤
│                    中台能力层                                │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐    │
│  │商品中心│ │库存中心│ │订单中心│ │营销中心│ │结算中心│    │
│  └────────┘ └────────┘ └────────┘ └────────┘ └────────┘    │
├──────────────────────────────────────────────────────────────┤
│                    领域服务层（供应链专属）                   │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐    │
│  │采购服务│ │仓储服务│ │物流服务│ │供应商  │ │对账服务│    │
│  └────────┘ └────────┘ └────────┘ └────────┘ └────────┘    │
├──────────────────────────────────────────────────────────────┤
│                    数据中台                                  │
│   Flink 实时（销量/库存预警）+ Spark 离线（T+1 报表）        │
├──────────────────────────────────────────────────────────────┤
│                    基础设施（K8s + 多机房 + 多仓）           │
└──────────────────────────────────────────────────────────────┘
```

## 二、中台化设计

**商品中心**：服务所有业务线（自营/第三方/跨境），统一 SPU/SKU 模型。
**库存中心**：统一现货/预售/在途/冻结模型，多仓库存统筹。
**订单中心**：统一订单状态机，适配不同业务（普通/拼团/预售）。

**复用价值**：新业务线（如多多买菜）接入只需配业务规则，复用中台能力。

## 三、异步化（领域事件）

```
下单 → OrderCreatedEvent
   ├─ 库存中心：冻结库存
   ├─ 营销中心：核销优惠券
   └─ 风控：异步复核

出库 → ShippedEvent
   ├─ 结算：生成账单
   ├─ 物流：跟踪运单
   └─ 用户：发货通知
```

本地消息表保证事件不丢，下游幂等消费。

## 四、多仓多活

```
华东仓（上海）─ 服务华东用户（次日达）
华南仓（广州）─ 服务华南用户
华北仓（北京）─ 服务华北用户
跨境仓（保税区）─ 跨境业务

下单时：按用户地址 + 商品库存位置 → 智能分仓
```

**分仓算法**：优先本地仓 → 备选邻近仓 → 跨区调拨（成本最高）。

## 五、容量估算

```
商品：千万级 SPU，亿级 SKU
订单：日均亿级，峰值百万 QPS
库存：亿级 SKU × 多仓 = 百亿 cell
GMV：日百亿级

存储：MySQL 分库分表 + HBase（明细）+ ES（搜索）+ Redis（缓存）
计算：Flink（实时）+ Spark（离线）
```

## 六、大促保障

1. **提前备货**：大促前 1 月按预测备货到各仓
2. **弹性扩容**：K8s HPA 大促前扩 3 倍
3. **降级预案**：非核心（推荐/评价）异步或关闭
4. **全链路压测**：影子流量验证容量

## 七、底层本质

供应链架构本质是**"用复用（中台）+ 解耦（异步）+ 就近（多仓）应对规模和地域复杂性"**。

**钱学森工程理论体现**：
- **技术上可行**：用成熟组件（Spring Cloud/Flink/MySQL）
- **经济上合理**：中台复用降低边际成本；多仓降低物流成本
- **研制周期短**：中台让新业务快速接入
- **能协调运转**：领域事件让各环节松耦合协同

## 常见考点
1. **中台和微服务区别**？——中台是业务能力复用（战略），微服务是技术拆分（战术）。
2. **多仓库存怎么统筹**？——库存中心统一视图（各仓库存+在途+冻结），分仓独立扣减。
3. **供应链最难的是什么**？——一致性（跨服务）+ 时效（物流）+ 成本（仓储/物流优化）。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你设计拼多多级供应链架构用了"中台化"（商品/库存/订单中心共享给所有业务线）。但业内很多人说中台已经失败（阿里拆中台），你怎么证明供应链中台不是跟风？**

中台失败的是"为了复用而强抽象"（如阿里早期把所有业务强塞共享业务中台，导致一个改动影响所有业务线，迭代慢）。供应链中台成立的依据是"业务模型确实一致"——自营/第三方/跨境的商品都是 SPU/SKU 结构，库存都是"现货+冻结+在途"模型，订单都是状态机。这些领域模型跨业务线高度一致，中台复用是合理的。区别在于"中台只共享领域模型和数据，不共享业务逻辑"——自营的促销规则和跨境的关税逻辑各自实现，不塞中台。判断标准是：如果三个业务线都要改同一个领域模型（如加"保质期"属性），中台改一次受益三次；如果各自不同，就不进中台。

### 第二层：证据与定位

**Q：大促时整个供应链架构的瓶颈在哪？你怎么定位是商品中心、库存中心还是订单中心的问题？**

用全链路追踪 + 黄金指标定位：
1. **看入口**——网关的 `order_create_p99` 如果飙高，先看是哪个下游慢。SkyWalking 追踪 order_create 的 span，如果 `order→inventory` 这一段占比 80%，库存中心是瓶颈。
2. **看各中心的黄金指标**——商品中心（`product_query_p99`）、库存中心（`stock_deduct_p99` + `redis_hit_rate`）、订单中心（`order_insert_qps` + `db_lock_wait`）。哪个指标异常飙高就是瓶颈。
3. **看资源**——如果某中心的 DB CPU 100%（`SHOW PROCESSLIST` 大量活跃连接），是该中心 DB 到瓶颈；如果 Redis 内存满或 CPU 高，是缓存层瓶颈。拼多多大促常见瓶颈是"库存 Redis 单分片热点"（爆款 SKU 都 hash 到一个 slot），用监控 `redis_hotkey_count` 定位。

### 第三层：根因深挖

**Q：定位到库存中心是瓶颈（stock_deduct_p99 飙到 500ms），根因是 Redis 热点（某爆款 SKU 单分片 QPS 10 万）。为什么不直接给 Redis 加机器就能解决？**

加 Redis 机器不解决热点问题——热点是一个 key 的 QPS 远超单分片承载（Redis 单实例 10 万 QPS），加机器只是加分片数，但热点 key 还是落在同一个分片（hash 不变）。根因解法：
1. **客户端分片（hot key 拆分）**——把 `stock:1001` 拆成 `stock:1001_0` 到 `stock:1001_9` 10 个 key，各存 1/10 库存，扣减时随机选 key，把单 key 的 10 万 QPS 分散到 10 个分片各 1 万。
2. **本地缓存预扣**——对超热 key（top 100），应用实例本地预扣（AtomicLong），定期从 Redis 同步额度，把 Redis 的 10 万 QPS 降到"同步额度"的低频（如每秒 1 万）。淘宝就是用这个方案扛双 11。
3. **读写分离 + 多级**——热点 key 的读走本地缓存（Caffeine），写仍走 Redis，减少 Redis 读压力。

**Q：那为什么不直接把热点 SKU 的库存扣减改成纯 DB 乐观锁（UPDATE ... WHERE qty>=?），避免 Redis 热点？**

DB 扛不住。DB 乐观锁对热点 SKU 的行锁串行化，10 万 QPS 扣同一行，行锁排队 + 大量失败重试，DB 直接打挂。Redis 是内存操作，单分片 10 万 QPS 是极限但能撑（DB 几千就挂）。所以 Redis 是性能层，DB 是兜底层，热点 SKU 必须走 Redis（拆分热点），不能退回 DB。DB 只用于冷门 SKU 和对账。

### 第四层：方案权衡

**Q：你的多仓多活架构（华东/华南/华北分仓就近发货）在大促时遇到跨仓调拨延迟。怎么权衡"就近发货省物流成本"和"跨仓调拨保证有货"？**

这是个优化问题，不是非此即彼：
1. **库存前置**——大促前根据预测（销量预测算法）提前把热销 SKU 铺到各仓（华东仓备 10 万、华南仓备 8 万），大促时就近发货不调拨。预测准确是关键，用历史大促数据 + 季节性模型预测。
2. **跨仓调拨兜底**——某仓售罄时从有货仓调拨，但调拨有物流时效（1-2 天）。用户下单时如果本仓无货，前端展示"预计 X 天送达"（如实告知），用户可选等或换其他商品。
3. **总仓统筹**——设一个"总仓"或"虚拟仓"统筹库存，某分仓无货时总仓调拨。总仓库存是各分仓的"缓冲池"。
权衡标准是"物流成本 vs 缺货损失"——如果跨仓调拨成本 < 缺货导致的订单流失损失，调拨；否则宁可让用户等或流失。

**Q：为什么不全国一个大仓，彻底避免跨仓调拨？**

单仓的物流时效和成本不可接受：
1. **时效差**——全国用户从单仓（如华东）发货，华南用户要 2-3 天；多仓就近发货华南用户次日达。电商时效是核心竞争力。
2. **运费高**——单仓发货全国，远距离运费高（华南订单从华东发运费比华南仓发贵 30%）。
3. **容量瓶颈**——大促单仓处理能力（分拣/打包/出库）有限，多仓分担。单仓大促日发千万单，分拣线扛不住。
4. **单点风险**——单仓因天气/疫情停摆，全国发货都停。多仓互备。
所以多仓是物流时效 + 成本 + 容量 + 风险分散的最优解，跨仓调拨是它的代价，用库存前置和预测降低调拨频率。

### 第五层：验证与沉淀

**Q：你怎么证明这个架构能扛大促（千万商品、亿订单、百亿 GMV）？**

多层验证：
1. **全链路压测**——大促前用 2 倍预期峰值压测（影子流量 + 影子库），验证各中心（商品/库存/订单）的 QPS 和延迟。压测发现瓶颈（如库存 Redis 单分片热点），提前优化。
2. **混沌演练**——主动注入故障（kill 库存中心某实例、断网、DB 慢），验证熔断/降级/多活的 failover 是否生效。确保"单点故障不雪崩"。
3. **大促实战**——大促当天实时监控核心指标（订单创建成功率、库存扣减延迟、物流发货时效），大盘异常立即降级（如关闭非核心功能）。
4. **事后复盘**——大促后复盘架构瓶颈，沉淀到下年优化。拼多多双 11 后会出"架构稳定性报告"，记录瓶颈点和优化项。

**Q：怎么让这个复杂架构可持续演进，不变成大泥球？**

架构治理机制：
1. **架构决策记录（ADR）**——每个重大架构决策（如"库存用 Redis 还是 DB"）写 ADR 文档，记录背景/决策/后果，新人能理解决策脉络。
2. **架构守护测试**——ArchUnit 测试依赖方向（如"领域服务不能直接依赖中台实现"），CI 强制，防止架构腐蚀。
3. **定期架构 review**——每季度架构 review，评估"中台能力是否还是共享模型""是否有过度耦合"，及时重构。架构不是一次性设计，是持续演进。

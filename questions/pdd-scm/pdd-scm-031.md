# 【拼多多供应链】怎么用 AI Agent 改造供应链？

> JD 依据：JD 8 提到"AI/LLM"标签，供应链智能化是趋势。

## 一、传统供应链的决策痛点

- **补货量**：靠人工经验 + 固定公式，难考虑大促/季节性/突发事件
- **异常处理**：缺货/延误/质量问题，每个都要人工判断
- **供应商选择**：靠历史关系，难全面评估

固定规则难穷尽这些"判断"场景。

## 二、Agent 架构

```
决策 Agent（LLM）
   ↓ 调用工具（MCP）
┌──────────────────────────────────┐
│ 工具层                           │
│  ├─ 库存查询 Tool               │
│  ├─ 销量预测 Tool               │
│  ├─ 供应商评估 Tool             │
│  ├─ 采购下单 Tool               │
│  ├─ 物流跟踪 Tool               │
│  └─ 历史案例检索 Tool（RAG）   │
└──────────────────────────────────┘
```

## 三、场景：智能补货

```
Agent 思考 1: SKU=12345 库存低于安全线，需补货
Agent 行动 1: 调库存 Tool + 销量预测 Tool
Agent 观察 1: 当前库存 500，未来 30 天预测销量 2000，提前期 7 天

Agent 思考 2: 需补 (2000×37/30 - 500) = 1967，向上取整 2000
Agent 行动 2: 调供应商评估 Tool（找最优供应商）
Agent 观察 2: 供应商 A 评分 9.2、提前期 5 天、价格最低

Agent 决策: 向供应商 A 下单 2000，理由附预测和评估
```

## 四、场景：异常自处理

```
事件: 供应商 B 到货延迟 3 天
Agent:
  1. 评估影响（哪些订单受影响）
  2. 检索历史类似案例（RAG）
  3. 生成方案（催货/换供应商/调拨其他仓/通知用户）
  4. 执行或人工确认
```

## 五、实时性挑战与解法

**分层**：
- 80% 简单补货：规则引擎（毫秒）
- 20% 复杂决策（大促/异常）：Agent（秒级异步）

**预计算**：离线 Agent 跑全量 SKU 算"建议补货量"，缓存 Redis。

## 六、防幻觉

1. **工具结果约束**：Agent 只基于工具返回数据决策
2. **双 Agent 验证**：决策 Agent + 复核 Agent 交叉验证
3. **关键场景人工确认**：大金额采购必须人工审批
4. **可解释**：决策带推理 trace + 证据引用

## 七、底层本质

Agent 改造供应链的本质是**"从规则自动化到决策智能化"**：
- 传统：人写规则，机器执行
- Agent：人给目标，机器自主推理决策

供应链的判断场景多、数据丰富（库存/销量/物流），是 AI Agent 落地的优质场景。

## 常见考点
1. **Agent 推理慢怎么实时**？——分层（简单规则/复杂 Agent 异步）+ 预计算 + 缓存。
2. **Agent 替代人吗**？——不替代，是人机协同（Agent 建议，人审批关键决策）。
3. **怎么评估 Agent 效果**？——补货准确率（vs 人工）、缺货率、库存周转。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：智能补货你用 AI Agent（ReAct 推理 + 工具调用），但传统规则引擎（库存 < 安全线就补货）简单可靠。为什么要引入 Agent 这种复杂方案？**

规则引擎能处理"已知模式"（库存低就补），但处理不了"复杂决策"：
1. **多因素权衡**——补货不仅要看库存，还要看销量预测、供应商提前期、大促系数、仓储容量。规则引擎写死规则（`if 库存 < 预测 × 1.2 then 补`），无法适应动态因素（如大促前提前备货、供应商延迟时换源）。Agent 能综合多工具返回的数据推理。
2. **异常处理**——供应商延迟、物流中断等异常，规则引擎只能报错让人工处理，Agent 能检索历史案例（RAG）生成方案（催货/换源/调拨）。
3. **可解释**——Agent 的决策带推理 trace（"库存 500，预测 2000，供应商 A 最优"），人能审计；规则引擎只有 if-else，异常时难追溯。
所以策略是"80% 常规补货用规则引擎（快、稳），20% 复杂决策用 Agent（智能、灵活）"，不是全盘 Agent 化。

### 第二层：证据与定位

**Q：Agent 生成的补货决策错了（某 SKU 补了 10 万，实际只需要 1 万）。你怎么定位是工具返回数据错、推理逻辑错，还是幻觉？**

看 Agent 的推理 trace：
1. **看工具返回**——Agent 的 trace 记录每步工具调用的结果。如果"销量预测 Tool 返回 100 万"（实际日均 1000），工具数据错（预测模型 bug 或输入异常）。查预测模型的输入和输出。
2. **看推理过程**——如果工具数据对（预测 2000），但 Agent 算出补 10 万，是推理数学错（如把 2000 × 50 倍系数算错成 2000 × 50 = 10 万）。看 LLM 的推理步骤，定位计算错误。
3. **看幻觉**——如果 Agent 引用了"不存在的供应商"或"编造的历史案例"，是幻觉（LLM 自由发挥）。解法是约束 Agent 只基于工具返回的数据决策，不自由生成。
定位方法：Agent 必须输出完整 trace（思考链 + 工具调用 + 观察 + 决策），人工对照 trace 找错在哪步。

### 第三层：根因深挖

**Q：Agent 推理对了（补 2000），但调采购下单 Tool 时传了 20000（多了一个 0）。根因是什么？**

根因是"工具调用参数生成错"。LLM 生成工具参数时可能出错（多 0、少 0、单位错）。几种情况：
1. **数值精度**——LLM 生成数字时偶尔出错（如 2000 写成 20000），尤其是大数字。解法是工具参数用结构化 schema（JSON Schema）约束 + 数值范围校验（如 qty 必须 < 100000），超范围拒绝。
2. **单位混淆**——LLM 把"箱"和"件"搞混（1 箱 = 10 件），补货 2000 件传成 2000 箱（= 2 万件）。解法是工具参数带单位字段，明确 `qty_unit=piece`。
3. **Tool 描述不清**——Tool 的描述（`deduct_stock(sku, qty)`）没说明 qty 的单位/范围，LLM 猜错。解法是 Tool 描述用清晰的 schema（`qty: integer, 1-10000, unit: piece`），加示例。
根因定位：看 Agent 的工具调用日志（参数值），对比预期值，找差异。预防：工具调用前做参数校验 + 关键操作（金额 > 10 万）人工确认。

**Q：那为什么不直接用确定性代码（Python 脚本）调工具，避免 LLM 生成参数的不确定性？**

确定性代码适合"流程固定"的场景（步骤已知），但 Agent 的价值是"流程自适应"：
1. **动态决策**——补货流程不是固定的（可能先查库存，也可能先查供应商产能，取决于情况），LLM 根据上一步结果决定下一步，代码写不死。
2. **异常处理**——遇到供应商延迟，Agent 能"临时决定"换方案（如换供应商），代码要穷举所有异常分支（难维护）。
3. **自然语言交互**——运营用自然语言问 Agent（"帮我补一下服饰类目缺货的 SKU"），LLM 理解意图 + 调度工具，代码做不到自然语言交互。
所以策略是"关键参数（金额/数量）用代码校验 + 兜底，流程编排用 Agent 灵活决策"。不是二选一，是 Agent 编排 + 代码校验。

### 第四层：方案权衡

**Q：Agent 推理慢（一次补货决策 5 秒），但供应链补货要实时（库存低就补）。怎么权衡"Agent 智能慢"和"实时性要求"？**

分层策略：
1. **80% 常规用规则引擎（毫秒）**——库存 < 安全线，按固定公式补（`补货量 = 预测 × 1.2 - 库存`），规则引擎秒级触发，不等 Agent。
2. **20% 复杂用 Agent（秒级异步）**——大促备货、异常处理、多供应商比价等复杂场景，Agent 异步跑（消息触发），结果写"建议表"，采购经理审核后执行。不阻塞实时链路。
3. **预计算**——离线 Agent 每天跑全量 SKU 算"建议补货量"，结果缓存 Redis。实时链路查缓存（毫秒），不实时跑 Agent。只有缓存 miss（新 SKU）才实时跑 Agent。

**Q：为什么不直接用更快的模型（如 GPT-4o-mini）降低 Agent 延迟？**

快模型（小模型）推理快但能力弱：
1. **复杂推理能力差**——补货决策要多步推理（库存 + 预测 + 供应商 + 成本），小模型容易出错（算错、漏步骤）。大模型（如 GPT-4）推理准但慢（5-10 秒）。
2. **工具调用不稳定**——小模型调工具时参数容易错（如传错 sku_id），大模型更稳定。
3. **幻觉更严重**——小模型更容易编造数据（如幻觉供应商），大模型更遵循指令。
所以权衡是"复杂决策用大模型（慢但准）+ 常规用规则引擎（快且稳）"，不是全用快模型。Agent 延迟通过"异步 + 预计算 + 分层"解决，不是靠降级模型质量。

### 第五层：验证与沉淀

**Q：你怎么证明 Agent 改造后补货决策真的比人工/规则引擎好（准确率高、成本低）？**

AB 测试对比：
1. **Agent vs 规则引擎**——同样 SKU，一组用规则引擎补货，一组用 Agent 补货，对比"补货准确率"（补的量 vs 实际销量）。Agent 应比规则引擎缺货率低 30%+（考虑了预测和供应商因素）。
2. **Agent vs 人工**——采购经理人工补货 vs Agent 建议补货（人审核后执行），对比"库存周转率"（Agent 应提升 10-20%，因为补货更精准不积压）。
3. **成本核算**——Agent 的推理成本（LLM API 调用费）vs 节省的人力成本 + 减少的缺货损失。Agent 月成本（如几万元 API 费）应远低于节省的缺货损失（几十万）。

**Q：怎么让 Agent 在生产环境稳定运行（不幻觉、不误操作）？**

多重防护：
1. **工具结果约束**——Agent 只能基于工具返回的数据决策，禁止自由生成（prompt 约束 + 输出 schema 校验）。
2. **双 Agent 验证**——决策 Agent 生成方案后，复核 Agent（独立调用）验证合理性，不一致触发人工审核。
3. **关键操作人工确认**——金额 > 10 万的采购、涉及淘汰供应商的操作，必须人工审批后执行，Agent 不能自动执行。
4. **决策日志可审计**——所有 Agent 决策（推理 trace + 工具调用 + 最终方案）记录日志，可追溯。定期抽审（如每天抽 10 个决策），评估质量。
5. **回退机制**——Agent 故障（如 LLM 服务不可用）时自动回退到规则引擎，保证业务不中断。

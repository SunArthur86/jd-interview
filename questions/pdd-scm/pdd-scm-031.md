# 【拼多多供应链】怎么用 AI Agent 改造供应链？

> JD 依据：JD 8 提到"AI/LLM"标签，供应链智能化是趋势。

## 一、传统供应链的决策痛点

- **补货量**：靠人工经验 + 固定公式，难考虑大促/季节性/突发事件
- **异常处理**：缺货/延误/质量问题，每个都要人工判断
- **供应商选择**：靠历史关系，难全面评估

固定规则难穷尽这些"判断"场景。

## 二、Agent 架构

```
决策 Agent（LLM）
   ↓ 调用工具（MCP）
┌──────────────────────────────────┐
│ 工具层                           │
│  ├─ 库存查询 Tool               │
│  ├─ 销量预测 Tool               │
│  ├─ 供应商评估 Tool             │
│  ├─ 采购下单 Tool               │
│  ├─ 物流跟踪 Tool               │
│  └─ 历史案例检索 Tool（RAG）   │
└──────────────────────────────────┘
```

## 三、场景：智能补货

```
Agent 思考 1: SKU=12345 库存低于安全线，需补货
Agent 行动 1: 调库存 Tool + 销量预测 Tool
Agent 观察 1: 当前库存 500，未来 30 天预测销量 2000，提前期 7 天

Agent 思考 2: 需补 (2000×37/30 - 500) = 1967，向上取整 2000
Agent 行动 2: 调供应商评估 Tool（找最优供应商）
Agent 观察 2: 供应商 A 评分 9.2、提前期 5 天、价格最低

Agent 决策: 向供应商 A 下单 2000，理由附预测和评估
```

## 四、场景：异常自处理

```
事件: 供应商 B 到货延迟 3 天
Agent:
  1. 评估影响（哪些订单受影响）
  2. 检索历史类似案例（RAG）
  3. 生成方案（催货/换供应商/调拨其他仓/通知用户）
  4. 执行或人工确认
```

## 五、实时性挑战与解法

**分层**：
- 80% 简单补货：规则引擎（毫秒）
- 20% 复杂决策（大促/异常）：Agent（秒级异步）

**预计算**：离线 Agent 跑全量 SKU 算"建议补货量"，缓存 Redis。

## 六、防幻觉

1. **工具结果约束**：Agent 只基于工具返回数据决策
2. **双 Agent 验证**：决策 Agent + 复核 Agent 交叉验证
3. **关键场景人工确认**：大金额采购必须人工审批
4. **可解释**：决策带推理 trace + 证据引用

## 七、底层本质

Agent 改造供应链的本质是**"从规则自动化到决策智能化"**：
- 传统：人写规则，机器执行
- Agent：人给目标，机器自主推理决策

供应链的判断场景多、数据丰富（库存/销量/物流），是 AI Agent 落地的优质场景。

## 常见考点
1. **Agent 推理慢怎么实时**？——分层（简单规则/复杂 Agent 异步）+ 预计算 + 缓存。
2. **Agent 替代人吗**？——不替代，是人机协同（Agent 建议，人审批关键决策）。
3. **怎么评估 Agent 效果**？——补货准确率（vs 人工）、缺货率、库存周转。

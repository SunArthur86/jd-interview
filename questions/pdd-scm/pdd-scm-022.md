# 【拼多多供应链】设计商品中心（千万 SPU、亿 SKU）

> JD 依据："商品货品领域研发"。

## 一、整体架构

```
写请求 → MySQL（分库分表，按 spu_id hash）
              ↓ binlog
         Canal → Kafka
              ├→ ES（搜索索引）
              ├→ Redis（删缓存）
              └→ 下游（推荐/广告/订单）

读请求 → Caffeine（本地）→ Redis（分布式）→ MySQL
搜请求 → ES（全文+聚合）
```

## 二、存储设计

**MySQL 分库分表**：
```yaml
ShardingSphere:
  tables:
    spu:
      actualDataNodes: ds${0..15}.spu_${0..15}
      databaseStrategy: { hash: { column: id } }
      tableStrategy: { hash: { column: id } }
    sku:
      actualDataNodes: ds${0..31}.sku_${0..31}   # SKU 更多，分更细
```

**ES 索引**：
```json
{
  "title": { "type": "text", "analyzer": "ik_max_word" },
  "category_id": { "type": "keyword" },
  "price": { "type": "double" },
  "sales": { "type": "long" },
  "attributes": { "type": "nested" }
}
```

## 三、读写分离

**读（高频）**：多级缓存命中率 98%。
**写（低频）**：MySQL 主库，Canal 异步同步。

**缓存一致性**：Cache Aside + Canal 兜底。

## 四、商品同步下游

```
商品变更 → binlog → Canal → Kafka
   ├→ 推荐服务（更新推荐池）
   ├→ 广告服务（更新广告库存）
   ├→ 搜索服务（ES 索引）
   └→ 订单服务（缓存商品信息）
```

## 五、底层本质

商品中心是**"主数据管理"**——CRUD 基础上叠加缓存、搜索、同步三层能力。核心矛盾是"读多写少"和"多下游同步"，解法是读写分离（缓存）+ 异步同步（Canal）。

## 常见考点
1. **商品改价怎么秒级生效**？——binlog → Canal → 删 Redis + 更 ES + 通知下游，秒级。
2. **SKU 价格怎么存**？——SKU 表独立 price 字段（不同规格不同价）。
3. **千万商品 ES 索引多大**？——单文档 1KB × 千万 = 10GB，分 5 分片每片 2GB。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：商品中心你把 SPU 分 16 库、SKU 分 32 库（SKU 分更细）。为什么 SKU 要比 SPU 分更多库？SKU 不是属于 SPU 的吗，为什么不和 SPU 同库？**

因为 SKU 数量是 SPU 的 10-100 倍（一个 SPU 平均 10 个 SKU），数据量和访问压力都更大：
1. **数据量**——千万 SPU 对应亿级 SKU，SKU 表是 SPU 表的 10 倍。SPU 分 16 库每库约 60 万行，SKU 如果也分 16 库每库 600 万行，单库压力大。分 32 库每库 300 万行，更均衡。
2. **访问压力**——商品详情页查 SPU 1 次（标题/类目），查 SKU 多次（所有规格列表），SKU 的 QPS 是 SPU 的几倍，需要更多分库分担。
3. **代价是跨库 JOIN**——查 SPU 和它的 SKU 要跨库（SPU 在 ds3，SKU 可能在 ds17）。解法是"SKU 表冗余 spu_id 字段，按 spu_id 分库"（让 SPU 和它的 SKU 落同一库），而不是按 sku_id 分。这样分库键一致，同 SPU 的数据同库，无跨库 JOIN。题目说"SKU 分 32 库"应该也是按 spu_id 分，只是库数更多（SPU 16 库映射到 SKU 32 库要设计路由）。

### 第二层：证据与定位

**Q：商品详情页打开慢（2 秒），你怀疑是商品中心的分库查询慢。怎么定位是哪个库的问题？**

分布式追踪 + 分库指标：
1. **看 trace**——SkyWalking 追踪商品详情请求，看 `product_center.query` 这个 span 耗时，以及它内部调了哪个分库（如 `SELECT * FROM spu_3 WHERE id=?`）。
2. **看各分库的慢查询**——每个分库的 `slow_query_log`，找耗时 > 500ms 的 SQL。如果 spu_3 库慢，可能是这个库的某个 SPU 数据异常（如 SKU 特别多，JSON 字段几 MB）。
3. **看 DB 连接池**——某分库的连接池（Druid）`active` 满，说明这个库压力大（可能是热点 SPU 集中在这库）。Druid 的监控页面能看各库的连接池状态。

### 第三层：根因深挖

**Q：定位发现 spu_3 库慢，某 SPU 的 SKU 列表查询要 1 秒。根因是什么？**

根因大概率是"单 SPU 的 SKU 数量异常多 + JSON 字段大"：
1. **SKU 过多**——某服饰 SPU 有 1000 个 SKU（颜色×尺码×版型组合），`SELECT * FROM sku WHERE spu_id=?` 返回 1000 行，每行带 attributes JSON（几 KB），总共几 MB 数据，网络传输 + 反序列化慢。
2. **JSON 字段大**——attributes JSON 包含完整属性树（含图片 URL 列表），单行 10KB，1000 行就是 10MB。
解法：
1. **SKU 分页**——商品详情不一次查所有 SKU，按销售属性筛选（如"选黑色后查对应 SKU"），配合 ES 做属性筛选。
2. **JSON 精简**——attributes 只存必要属性（颜色/内存），图片 URL 等大字段拆到独立表或 CDN，按需查。
3. **SKU 列表缓存**——整个 SPU 的 SKU 列表缓存 Redis（JSON 字符串），详情页直接读缓存。

**Q：那为什么不直接限制每个 SPU 的 SKU 数（如最多 100 个），从源头避免？**

业务上不合理：
1. **服饰类目天然 SKU 多**——颜色(5) × 尺码(8) × 版型(2) = 80 SKU，加上季节款可能 100+，限制 100 会砍掉正常商品。
2. **拼团/套餐 SKU**——组合商品（如"手机+壳+膜"套餐）SKU 数组合爆炸，可能几百个。
所以 SKU 数量不能硬限制，要在"存储和查询"层面优化（分页、精简 JSON、缓存），而不是限制业务。

### 第四层：方案权衡

**Q：千万商品 ES 索引 10GB，你分 5 分片。但大促时商品频繁改价改库存，ES 写入压力大（每秒几万次 update）。怎么权衡"分片数"和"写入性能"？**

ES 分片数影响写入和查询：
1. **分片多 = 写入并发高**——每个分片独立写入，5 分片比 1 分片写入 QPS 高 5 倍。但分片多也增加 cluster 协调开销（每条写入要路由到正确分片）。
2. **分片多 = 单分片小**——查询时多个分片并行（scatter-gather），但分片太多（如 50 个）协调开销反而拖慢查询。
3. **权衡**——10GB 数据分 5 片（每片 2GB）是合理的（推荐单分片 10-50GB）。大促写入压力大可以"临时增加副本"（replica，读负载分担）或"按时间分索引"（如按月滚动，`product_202607`），每个索引分片数保持 5。
改价改库存的实际优化是"减少 ES 写入"——库存这种高频变更的不全量同步 ES（ES 只存价格和基础信息，库存走 Redis 单独查），降低 ES 写入压力。

**Q：为什么不直接把商品的所有信息（含实时库存）都放 ES，查询一次拿到全？**

ES 不适合高频更新：
1. **写入延迟**——ES 写入后要 refresh（1 秒）才能搜到，实时性差。库存每秒扣减几万次，ES 更不上。
2. **segment merge 开销**——频繁 update 触发 segment 合并，CPU 和 IO 开销大，影响查询性能。
3. **一致性窗口**——Canal 同步有延迟，ES 的库存值可能比 Redis 旧，展示"有货"实际已售罄，用户下单失败体验差。
所以 ES 只存"相对稳定"的数据（标题/类目/属性/价格），实时性数据（库存）走 Redis 单独查，详情页"ES 拿基础信息 + Redis 拿实时库存"，组合查询。这是电商标准做法。

### 第五层：验证与沉淀

**Q：你怎么证明商品中心能支撑千万 SPU、亿 SKU，查询性能达标？**

压测 + 监控：
1. **容量压测**——灌入 1 千万 SPU + 1 亿 SKU 模拟数据，压测 `WHERE spu_id=?` 和 `WHERE sku_id=?` 的 QPS 和 P99。目标 P99 < 50ms（分库后单库数据量可控）。
2. **分库均衡监控**——各分库的 `spu_count` 和 `qps` 应该均衡（偏差 < 10%），如果某库倾斜（如 spu_3 承载 30% 流量），说明分片键设计有问题（如 SPU id 生成不均匀），要重新分片。
3. **ES 同步延迟**——监控 Canal→ES 的延迟（`es_sync_lag`），正常 < 5 秒；大促时写入压力大可能涨到 30 秒，评估业务能否接受（商品改价的延迟容忍度）。

**Q：怎么让商品中心的数据分片可演进（未来涨到 10 亿 SKU 怎么扩容）？**

分片可演进设计：
1. **一致性哈希或分片路由表**——不写死 `spu_id % 16`，用路由表（`shard_route` 表记录 spu_id 在哪个库），扩容时迁移部分数据到新库，改路由表，不改动业务代码（业务读路由表决定访问哪个库）。
2. **双写灰度扩容**——扩容时新库双写（写旧库 + 新库），读仍走旧库，验证数据一致后切读到新库，最后停旧库。零停机扩容。
3. **分片键不可变约定**——分片键（spu_id）一旦分配不能改（改了数据要跨库迁移），所有查询必须带分片键（不带分片键的查询要扫所有库，禁止）。Code Review 确保所有 SQL 带 `spu_id` 条件。

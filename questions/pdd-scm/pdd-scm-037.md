# 【拼多多供应链】AI Infra 怎么设计？

> JD 依据：JD 9 大模型工程师核心。

## 一、AI Infra 四层

```
数据层（采集/标注/版本 DVC）
   ↓
训练层（分布式训练 PyTorch/DeepSpeed）
   ↓
推理层（vLLM 部署/量化/GPU 调度）
   ↓
监控层（漂移/效果/迭代）
```

## 三、训练层

**分布式训练**：
| 并行 | 适用 |
|------|------|
| 数据并行 | 模型单卡放得下 |
| 张量并行 | 单卡放不下（切矩阵） |
| Pipeline 并行 | 超大模型（切层） |

**Checkpoint**：定期保存，故障可恢复。

**实验管理**：MLflow / W&B（参数/metrics/模型追踪）。

## 六、底层本质

AI Infra 是**"让 AI 从手工作坊到工业流水线"**：
- 数据平台统一管理
- 训练流水线自动化
- 部署标准化
- 监控持续化

**Java 工程师的优势**：分布式系统、性能优化、工程化——AI Infra 本质是系统工程。

## 常见考点
1. **AI Infra 和 MLOps 区别**？——MLOps 偏流程（CI/CD），AI Infra 偏平台（基础设施）。
2. **训练和推理为什么用不同 GPU**？——训练需大显存（A100）；推理重吞吐（T4 性价比）。
3. **怎么处理模型漂移**？——监控数据/概念漂移 → 触发重训。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：AI Infra 你设计了训练池（A100）和推理池（T4）分离。为什么不混用（同一批 GPU 既训练又推理），省机器？**

训练和推理的负载特征完全不同：
1. **训练重显存 + 算力**——训练要存模型参数 + 梯度 + 优化器状态（是推理的 3-4 倍显存），需要 A100（80GB 显存）。推理只存模型参数 + KV cache，T4（16GB）够。混用的话训练任务占 A100，推理任务抢不到资源。
2. **训练批处理 + 长任务**——训练跑几小时到几天，批量提交，可排队。推理是实时请求（毫秒响应），不能等。混用会让推理请求被训练任务阻塞。
3. **成本优化**——训练用 A100（贵，按需启停，训练完释放），推理用 T4/A10（便宜，常驻服务）。混用要么全用 A100（推理浪费贵卡），要么全用 T4（训练跑不动）。
所以分离是"按负载特征选硬件 + 按成本优化"，混用两边都不优。

### 第二层：证据与定位

**Q：推理服务延迟突增（P99 从 200ms 涨到 2 秒），你怎么定位是 GPU 满、模型慢，还是并发太高？**

分层排查：
1. **看 GPU 利用率**——`nvidia-smi` 看 GPU 的 `utilization` 和 `memory`。如果 utilization 100% + memory 90%，GPU 打满，是并发太高（请求排队）。如果 utilization 低但延迟高，是模型本身慢（如 batch 太大或量化问题）。
2. **看队列长度**——vLLM 的 `running_queue_size` 和 `waiting_queue_size`，如果 waiting 涨（请求积压），是并发超 GPU 处理能力。
3. **看单请求耗时**——Trace 单个请求的"prefill（首 token）+ decode（生成 token）"耗时。如果 prefill 慢（输入 token 多），是 prompt 太长；如果 decode 慢（输出 token 多），是生成长。

### 第三层：根因深挖

**Q：GPU 打满（utilization 100%），但加并发也不提升吞吐（QPS 卡在 50）。根因是什么？**

根因是"GPU 成为瓶颈，加并发只是排队不提升吞吐"：
1. **batch 满了**——vLLM 的 Continuous Batching 会把多个请求合并成一个 batch 同时处理，GPU 算力固定，batch 满后新请求排队。吞吐瓶颈是 GPU 的 FLOPS（浮点运算能力），加并发只是让请求在队列等，不增加吞吐。
2. **显存限制 KV cache**——LLM 推理的 KV cache 占显存（每个请求的上下文都要存），显存满了 vLLM 不能接新请求（要等旧请求完成释放 KV cache）。即使 GPU 算力有余，显存限制并发数。
3. **解法**：
   - **量化**——INT4/INT8 量化降低模型显存占用，留更多显存给 KV cache，提升并发。
   - **扩 GPU**——加 GPU（tensor parallel），吞吐线性提升。
   - **限制 context 长度**——限制单请求的 prompt 长度（如 4K token），减少 KV cache 占用，提升并发。

**Q：那为什么不直接用更大的 GPU（如 H100）单卡扛，不搞分布式推理？**

H100 单卡扛不动高并发 LLM：
1. **显存仍不够**——LLM（如 70B 模型）参数就占 140GB（FP16），单张 H100（80GB）放不下。必须张量并行（切到多卡）。
2. **吞吐瓶颈**——单卡 GPU 的算力有限（即使是 H100），高并发时请求排队。多卡（tensor parallel）能并行处理更多请求。
3. **成本**——H100 单卡 3-4 万美元，4 张 H100 = 12-16 万美元。而 8 张 A10（总显存相当）成本约 4 万美元，性价比更高。只有大模型（70B+）才必须 H100，中等模型（7B-13B）用 A10/L20 足够。
所以分布式推理（多张中端卡）比单张大卡更经济 + 更灵活（可按需扩缩 GPU 数）。

### 第四层：方案权衡

**Q：模型漂移监控你用 PSI（总体稳定性指数）检测数据漂移，PSI > 0.25 触发重训。但重训成本高（几小时 GPU），怎么权衡"漂移检测敏感"和"重训成本"？**

分级响应策略：
1. **轻度漂移（PSI 0.1-0.25）**——告警但不重训，人工分析原因（如某特征分布变了但模型仍可用）。可能是数据季节性变化（如换季），不必须重训。
2. **中度漂移（PSI 0.25-0.5）**——触发增量重训（用最近数据 fine-tune，不从头训），耗时短（几十分钟），成本低。定期（每周/每月）增量重训兜底。
3. **重度漂移（PSI > 0.5）**——触发全量重训（重新训练），耗时长（几小时），但要立即做（模型已严重失效）。如市场结构变化（疫情后消费习惯彻底变）。
4. **效果监控兜底**——PSI 只是"输入漂移"，还要看"输出效果"（如预测准确率 AUC/MAPE）。如果 PSI 低但准确率掉（概念漂移，输入分布没变但输入-输出关系变了），也要重训。
所以不是"PSI 高就重训"，而是"PSI + 效果指标"双信号决定是否重训 + 重训级别（增量 vs 全量）。

**Q：为什么不直接定期重训（如每周），不搞漂移检测，简单省事？**

定期重训的问题：
1. **浪费成本**——如果模型没漂移（数据稳定），每周重训是浪费 GPU（白跑）。漂移检测能避免无效重训。
2. **响应慢**——如果漂移发生在重训周期中间（如周一来，下次重训在周日），模型失效 6 天。漂移检测能即时触发，响应快。
3. **掩盖问题**——定期重训不分析"为什么漂移"（如某特征源故障），只是机械重训。漂移检测的告警能暴露数据 pipeline 问题（如特征工程 bug 导致分布变）。
所以策略是"漂移检测触发 + 定期重训兜底"——漂移检测处理异常变化（即时响应），定期重训处理常态衰减（如模型慢慢退化）。两者结合，不是二选一。

### 第五层：验证与沉淀

**Q：你怎么证明 AI Infra（训练/推理/监控）稳定可靠，不影响业务？**

多维验证：
1. **推理可用性**——推理服务的 `success_rate` > 99.9%、`p99_latency` < 500ms（业务可接受）。GPU 故障时自动切换备节点（k8s 调度），业务无感。
2. **训练可靠性**——训练任务的 `success_rate` > 95%（偶尔失败可重试），Checkpoint 机制保证故障恢复（从最近 checkpoint 续训，不从头）。
3. **漂移检测有效**——注入"数据漂移"（故意改特征分布），验证 PSI 告警是否触发；注入"概念漂移"（改标签），验证效果监控是否告警。演练保证检测机制灵敏。
4. **成本可控**——月度 GPU 成本监控（训练 + 推理），按业务线分摊，超预算告警。推理服务的"单次调用成本"（GPU 成本 / QPS）应持续下降（优化量化/vLLM）。

**Q：怎么让 AI Infra 的使用规范统一（资源申请、模型部署、监控）？**

沉淀平台 + 规范：
1. **GPU 调度平台**——统一管理 GPU 资源池，业务方提交训练/推理任务到平台（声明资源需求），平台调度（k8s + GPU operator），禁止业务方直连 GPU 机器。
2. **模型部署标准化**——推理服务统一用 vLLM/Triton 部署，提供标准镜像 + 配置模板，业务方只提供模型权重，平台自动部署 + 监控。
3. **监控标配**——所有模型服务必须上报 `latency / qps / gpu_utilization / drift_psi`，接入 Prometheus + Grafana 大盘，异常告警。
4. **成本治理**——各业务线的 GPU 配额（如每月 1000 GPU 小时），超配额限流或收费（内部结算），倒逼业务方优化（如量化、batch、用便宜卡）。

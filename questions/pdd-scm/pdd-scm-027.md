# 【拼多多供应链】高可用三件套（限流/降级/超时重试）怎么设计？

> JD 依据："线上系统稳定性和维护经验"。

## 一、限流

**令牌桶**（允许突发）：
```
固定速率放令牌（1000/s），请求拿令牌，无则拒绝
桶满丢新令牌 → 允许攒令牌应对突发
```

**漏桶**（匀速）：
```
请求入桶，恒定速率漏出（处理）
水满拒绝 → 强制匀速
```

**Sentinel 配置**：
```java
@SentinelResource(value = "query_product", blockHandler = "fallback")
public Product query(long id) { ... }
```

## 二、降级

```java
public Product query(long id) {
    try {
        return productService.get(id);          // 正常
    } catch (Exception e) {
        return cachedProduct(id);                // 降级：读缓存
    }
}
// 极端：return Product.default();               // 兜底默认
```

降级开关（配置中心一键）：
```java
@NacosValue("${scm.degrade.recommend:false}", autoRefreshed = true)
private boolean degradeRecommend;
```

## 三、超时分级

```yaml
feign:
  client:
    config:
      default: { connect-timeout: 1000, read-timeout: 2000 }
      product-service: { read-timeout: 50 }      # 严格下游
```

## 四、重试

```java
@Retryable(maxAttempts = 3, backoff = @Backoff(delay = 100, multiplier = 2))
public Product callProduct(long id) { ... }
```

**幂等前提**：非幂等（扣款）不能盲目重试，需业务幂等 token。

## 五、供应链实战

```
商品详情请求:
  ├─ 网关限流（单 UID 10/s）
  ├─ 超时分级（商品 50ms / 库存 30ms / 评价 50ms）
  ├─ 评价服务挂 → 降级隐藏评价模块
  └─ 全挂 → 兜底返回基础商品信息
```

## 六、底层本质

高可用三件套承认**"故障必然"**，从被动应对转主动防御：
- 限流：保护自己边界
- 超时重试：防被外界拖死
- 降级：故障时保核心价值

## 常见考点
1. **熔断和限流区别**？——限流主动（按 QPS 拒绝）；熔断被动（按失败率切断）。
2. **重试风暴**？——上游重试放大下游压力 → 必须配熔断器。
3. **降级和熔断关系**？——熔断触发降级（熔断后走 fallback）。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：限流你用令牌桶（Sentinel），但拼多多大促时某些爆款商品的查询 QPS 是平时的 100 倍。如果限流阈值设平时的 10 倍，大促时大量请求被拒；设 100 倍，平时资源浪费。怎么定阈值？**

阈值不能设固定值，要"动态 + 分场景"：
1. **基线 + 弹性**——平时阈值 = 历史 P99 QPS × 2（留余量），大促时手动/自动调高（如大促前配置中心一键调到平时的 50 倍）。Sentinel 支持规则动态推送（Nacos）。
2. **分级限流**——不同接口不同阈值：核心接口（商品详情）阈值高（保护用户体验），非核心（推荐/广告）阈值低（先降级非核心保核心）。
3. **自适应限流**——用 Sentinel 的自适应算法（如 BBR 模式），根据系统负载（CPU/RT）自动调阈值。负载低放宽，负载高收紧，不预设固定值。
所以策略是"平时基线 + 大促弹性 + 自适应兜底"，不是一刀切。

### 第二层：证据与定位

**Q：大促时商品详情接口大量返回 429（限流拒绝），用户投诉"进不去"。你怎么定位是限流阈值设低了、正常流量被误杀，还是真有爬虫攻击？**

三步定位：
1. **看限流维度**——Sentinel 的限流维度（QPS/并发数/关联资源）。如果是"全局 QPS"触发，看实际 QPS 是否超阈值（`sentinel_flow_count`）；如果是"单 UID 限流"触发（某 UID 1 秒查 100 次），看该 UID 是否正常用户（爬虫特征：高频 + 无操作间隔）。
2. **看流量分布**——网关日志按 UID/IP 聚合，如果 TOP 10 IP 贡献了 50% 流量，是爬虫/刷单，要走风控不是限流。如果流量均匀分布（每个 UID 几次），是正常用户涌入，要提阈值。
3. **看下游容量**——如果限流是为了保护下游（如 DB），看 DB 的 `active_connections` 和 `slow_query_count`，如果 DB 已经满了（连接池满、CPU 100%），限流是对的（保下游不挂），不能提阈值，要扩容下游。

### 第三层：根因深挖

**Q：定位发现是正常用户流量被误杀（限流阈值设低了），但你不敢贸然提阈值（怕打挂下游）。根因是什么？**

根因是"限流阈值和下游容量不匹配"。限流阈值的依据应该是"下游能扛多少"，不是拍脑袋。正确做法：
1. **压测定容量**——大促前压测下游（DB/Redis），找到其极限 QPS（如 DB 扛 5 万 QPS），限流阈值设容量的 70%（3.5 万），留 30% 余量。
2. **按下游瓶颈设阈值**——如果瓶颈是 DB 连接池（50 个），每个请求占 1 个连接 10ms，理论 5000 QPS，限流设 3500 QPS。如果提限流到 5000，DB 连接池满，请求排队超时。
3. **扩容下游**——如果业务要求 5 万 QPS 但 DB 只能扛 3.5 万，要先扩容 DB（加从库/加连接池）再提阈值，不能只提阈值不扩容。
所以根因不是"阈值设低了"，是"下游容量没扩到位就要求高阈值"。限流是"保护下游"的，不是"限制业务"的。

**Q：那为什么不直接把限流去掉，让下游自己扛（DB 连接池满了自然拒绝）？**

去掉限流会雪崩：
1. **连接池满 → 超时 → 重试 → 更大压力**——DB 连接池满后，新请求等连接（超时），上游重试，放大流量，DB 更扛不住。
2. **慢查询放大**——高并发时 DB 锁竞争加剧，原本 10ms 的查询变 500ms，更多请求堆积，恶性循环。
3. **故障扩散**——DB 挂了，所有依赖它的服务都挂（雪崩）。
限流的作用是"在下游崩溃前主动拒绝部分请求，保住核心链路"。拒绝 30% 请求（返回 429）比"100% 请求因 DB 挂而超时"好得多。限流是"弃车保帅"。

### 第四层：方案权衡

**Q：重试你配了 `maxAttempts=3, backoff=exponential`（指数退避）。但下游服务慢（不是挂），重试会让下游更慢。怎么权衡"重试提高成功率"和"重试放大压力"？**

重试是双刃剑，关键是"什么时候该重试，什么时候不该"：
1. **区分错误类型**——网络抖动/超时（瞬时）重试；业务错误（如库存不足）不重试（重试也是同样结果）；下游熔断中不重试（重试无意义，等恢复）。
2. **退避策略**——指数退避（100ms → 200ms → 400ms）给下游恢复时间，不是立即重试（立即重试放大压力）。
3. **重试预算**——设重试比例上限（如总请求的 10%），超了不再重试（直接失败），防止重试风暴。Spring Retry 的 `RetryTemplate` 支持 `circuit breaker`（重试到一定次数触发熔断，停止重试）。
4. **幂等前提**——只有幂等接口（GET 查询）能安全重试；非幂等（POST 下单）必须带幂等 token，否则重试导致重复下单。

**Q：为什么不直接把超时设长一点（如 10 秒），不重试，等下游慢慢返回？**

长超时的风险是"线程被占用"：
1. **线程池耗尽**——每个请求等 10 秒，100 个请求占满线程池，新请求被拒。
2. **级联超时**——上游等下游 10 秒，上游的上游也可能超时（层层超时），整条链路卡住。
3. **用户体验差**——用户等 10 秒早就离开了。
短超时（200ms）+ 快速失败 + 降级（返回缓存）比长超时好——快速给用户一个"次优但可用"的结果（缓存数据），比让用户等 10 秒看到错误好。电商接口的标准是"50ms 内必须返回（成功或降级）"，不长等。

### 第五层：验证与沉淀

**Q：你怎么证明限流/降级/熔断的配置合理（不过度限流也不打挂下游）？**

压测 + 演练验证：
1. **容量压测**——逐步加压（1000 → 5000 → 10000 QPS），找到系统的"拐点"（RT 开始飙涨、错误率上升的 QPS），限流阈值设在拐点的 70%。
2. **故障注入演练**——主动 kill 下游服务，验证熔断是否在预期时间触发（如 5 秒内失败率 > 50% 熔断），降级 fallback 是否返回正确兜底数据。如果熔断没触发（仍往下调用），配置有问题。
3. **大促实战监控**——大促时实时看 `reject_rate`（限流拒绝率）、`degrade_count`（降级次数）、`circuit_breaker_state`（熔断器状态），拒绝率 > 5% 说明阈值偏低（或下游容量不足），要调。

**Q：怎么让团队的高可用配置规范统一、不漏配？**

沉淀高可用 SDK + 规范：
1. **统一注解**——`@HaResource(fallback="xxxFallback", timeout=50, retry=2, rateLimit=1000)`，一个注解配齐超时/重试/限流/降级，业务方不裸写 try-catch。
2. **配置模板**——按接口类型（核心/非核心）预设超时和限流阈值（核心：50ms/高阈值；非核心：200ms/低阈值 + 可降级），新接口套模板。
3. **CR checklist**——所有 Feign 调用必须有超时配置；所有外部依赖必须有熔断器；所有非核心接口必须有降级 fallback。SonarQube 扫"Feign 调用无超时配置"报 blocker。
4. **大促前演练**——大促前必须跑全链路故障注入（kill 各下游、模拟慢响应），验证降级/熔断都生效，否则不让上线。

## 结构化回答

**30 秒电梯演讲：** 资源有限、下游可能挂、网络会抖，如何让系统在压力下仍提供核心服务？简单说就是——限流在入口截流（防过载）、降级在所以障兜底（保核心）、超时重试在调用层保护（防级联），三者是高可用"防御工事"。

**展开框架：**
1. **限流** — 限流：令牌桶（突发）/ 漏桶（匀速）/ 滑动窗口
2. **降级** — 降级：兜底返回、关闭非核心、读旧数据
3. **超时** — 超时：连接/读/总，必须分级

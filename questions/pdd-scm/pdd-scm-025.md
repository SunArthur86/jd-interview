# 【拼多多供应链】设计结算对账系统（日百亿 GMV）

> JD 依据："结算"是供应链核心环节。

## 一、对账体系

```
实时监控（Flink）          T+1 全量对账（Spark）
   ↓                          ↓
差异告警（秒级）            平账/差异报告（天级）
```

## 二、三单匹配（付款前）

```java
public boolean match(Long poId) {
    PurchaseOrder po = poService.get(poId);        // 采购单
    ArrivalNote arrival = arrivalService.get(poId); // 到货单
    Invoice invoice = invoiceService.get(poId);    // 发票

    return po.amount == arrival.amount          // 金额一致
        && po.qty == arrival.qty                // 数量一致
        && po.amount == invoice.amount;          // 发票金额一致
}
// 三单一致才允许付款
```

## 三、T+1 全量对账

```
订单系统数据 vs 仓储系统数据 vs 财务系统数据
   ↓ 三方比对
一致 → 平账
不一致 → 差异表 → 自动调账 or 人工核查
```

**Spark 批处理**：
```sql
SELECT o.order_id, o.amount as order_amt, w.amount as wh_amt, f.amount as fin_amt
FROM orders o
FULL OUTER JOIN warehouse w ON o.order_id = w.order_id
FULL OUTER JOIN finance f ON o.order_id = f.order_id
WHERE COALESCE(o.amount,0) != COALESCE(w.amount,0)
   OR COALESCE(w.amount,0) != COALESCE(f.amount,0);
```

## 四、实时差异监控（Flink）

```java
// 库存负数监控
stockStream.filter(s -> s.qty < 0).addSink(new AlertSink());

// 金额突变监控
orderStream
    .keyBy(o -> o.skuId)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .aggregate(new SumAggregator("amount"))
    .filter(amt -> amt > threshold)  // 5 分钟金额异常
    .addSink(new AlertSink());
```

## 五、差异处理

| 差异类型 | 处理 |
|---------|------|
| 时序差（跨日订单） | 自动延后一天对账 |
| 退货未同步 | 触发逆向同步 |
| 价格变更未同步 | 财务补录 |
| 复杂差异 | 人工核查工单 |

## 六、底层本质

对账本质是**"多源数据的交叉验证"**——单一系统可能有 bug 或延迟，多系统交叉才能保证资金绝对正确。这是金融级系统的" defense in depth"（纵深防御）。

**钱学森工程理论**：技术上可行（成熟大数据栈）、经济上合理（自动对账省人力）、协调运转（订单/仓储/财务三系统协同）。

## 常见考点
1. **对账性能怎么保证**？——Spark 批处理（TB 级）+ 分区裁剪 + 增量对账。
2. **怎么防止重复付款**？——付款单唯一（业务幂等）+ 三单匹配校验。
3. **实时和离线对账区别**？——实时抓关键异常（秒级）；离线全量兜底（天级）。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：结算对账你做了"T+1 全量对账 + 实时差异监控"两层。财务最关心的是"不能多付也不能少付"，为什么 T+1 全量对账能保证这一点？**

T+1 全量对账的价值是"完整覆盖 + 确定性结论"。实时监控只能抓"已知模式异常"（如金额突变、库存负数），对"未知 bug 导致的系统性偏差"无能为力（如某版本代码把所有订单金额算少了 1 分，实时监控看不出，因为是均匀偏差）。T+1 全量对账把订单/仓储/财务三个系统的每一条记录做 FULL OUTER JOIN，任何一条不一致都暴露。它的确定性在于"三方数据完全对齐才算平账"，不依赖预设规则。财务的资金安全靠这个"兜底"，实时监控只是"早发现"。

### 第二层：证据与定位

**Q：T+1 对账跑出来有 1000 条差异（订单系统和财务系统金额不一致）。你怎么定位是时序差、同步丢数据，还是计算 bug？**

分类排查：
1. **时序差（跨日订单）**——最常见。订单 23:59 创建，仓储 00:01 同步，跨日了。判断：差异订单的 `create_time` 是否接近凌晨（23:30-00:30），如果是，大概率时序差，延后一天对账自动消除。查这 1000 条里有多少是时序差（应占 80%+）。
2. **同步丢数据**——Canal 或 MQ 丢消息导致财务系统缺这条记录。判断：`SELECT * FROM finance WHERE order_id=?` 如果查不到，是同步丢失。看 Canal 的 `consumer_lag` 和丢消息日志。
3. **计算 bug**——三方都有记录但金额不同。判断：对比订单的 `amount` 和财务的 `amount`，如果差值是固定的（如都差 0.01 元），是精度 bug（如 double 精度丢失，要改 BigDecimal）；如果差值随机，是计算逻辑 bug（如优惠券扣减规则不一致）。

### 第三层：根因深挖

**Q：差异里有 50 条是"订单金额比财务多 0.01 元"，固定差值。根因是什么？**

根因大概率是"精度处理不一致"：
1. **double vs BigDecimal**——订单系统用 `double` 算金额（如 10.10 元存成 10.0999999），财务系统用 `BigDecimal(10.10)`，对比时 10.0999999 != 10.10，差 0.0000001，四舍五入后显示差 0.01。
2. **舍入策略不一致**——订单用 `HALF_UP`（四舍五入），财务用 `DOWN`（截断），同一计算结果不同。如 10.105 元，订单算 10.11，财务算 10.10。
3. **税率计算精度**——含税金额 = 不含税 × (1 + 税率)，如果中间步骤精度不一致（订单先算税后保留 2 位，财务全程保留 4 位），结果差 0.01。
根因定位：查订单和财务的金额计算代码，确认都用 `BigDecimal` 且 `setScale(2, HALF_UP)`。统一精度规范后差异消除。

**Q：那为什么不直接用整数（分）存金额（如 1010 分 = 10.10 元），彻底避免浮点精度问题？**

用"分"（整数）存金额是最佳实践，但题目场景可能没这么做。改成整数存储的好处：
1. **无精度丢失**——整数运算精确，`1010 - 100 = 910` 永远精确。
2. **性能好**——整数运算比 BigDecimal 快（BigDecimal 是对象，有 GC 开销）。
3. **显示时转换**——前端显示时 `/ 100` 转成元，存储和计算用分。
如果订单和财务都用"分"存储，0.01 的差异根除。但改造老系统成本高（数据库 schema 改、所有代码改），所以新增系统用"分"，老系统用 BigDecimal 统一精度，逐步迁移。拼多多新业务用"分"，老业务用 BigDecimal。

### 第四层：方案权衡

**Q：T+1 全量对账用 Spark 跑 TB 级数据，耗时 4 小时。财务要"上午 9 点前出对账报告"，但数据量增长后跑不完。怎么优化？**

三个方向：
1. **增量对账**——不全量比对，只比"昨天变更的订单"（`WHERE update_time >= yesterday`）。Spark 读取增量分区（分区裁剪），数据量从 TB 降到 GB，耗时从 4 小时降到 30 分钟。全量对账降频到每周一次兜底。
2. **预计算**——白天实时把订单/仓储/财务数据按 `order_id` 聚合到汇总表（如每天一行：`order_count, total_amount`），T+1 对账时只比对汇总表（几万行），不对明细（几十亿行）。明细对账只在汇总不一致时下钻。
3. **列式存储 + 引擎**——把对账数据存 Iceberg/Hudi（列式 + 索引），用 Trino/Presto 查（比 Spark 快几倍）。Spark 适合 ETL，Trino 适合即席查询，对账是即席场景。

**Q：为什么不直接用 MySQL 做对账（JOIN 三张表），非要上 Spark？**

MySQL 扛不住对账的数据量：
1. **数据量**——日百亿 GMV 意味着每天几亿订单，三张表 JOIN 是几亿 × 几亿，MySQL 跑几小时甚至跑不出（内存不足、临时表爆）。
2. **资源占用**——对账是大查询（全表扫 + JOIN），占用 MySQL 大量 CPU/IO，影响在线业务（下单/查询被拖慢）。
3. **扩展性**——MySQL 单机，Spark 是分布式（几十台机器并行），TB 级数据 Spark 是标配。
所以策略是"在线业务用 MySQL（OLTP，低延迟），对账分析用 Spark/Trino（OLAP，大数据量）"，数据从 MySQL 同步（如离线导出到 Hive/Iceberg）后做对账。这是 HTAP（混合事务分析）的分库分栈方案。

### 第五层：验证与沉淀

**Q：你怎么证明对账系统准确（没漏报差异、没误报）？**

两个验证：
1. **注入测试**——故意造差异（如改一条订单的金额，不改财务），看对账能否抓出。如果抓不出，是对账逻辑漏（如 JOIN 条件错、WHERE 过滤掉）。拼多多每月跑一次"差异注入演练"，验证对账敏感度。
2. **误报率**——对账报告的差异里，"真差异"占比应 > 90%（其余是时序差等假阳性）。如果真差异占比 < 50%，说明对账规则太敏感（大量误报），财务要花时间排查假阳性，效率低。调规则（如时序差自动延后）降低误报。

**Q：怎么让对账系统可持续演进，数据量涨 10 倍还能跑？**

架构预留：
1. **分区设计**——所有对账表按天分区（`PARTITION BY dt`），增量对账只读昨天分区，数据量增长不影响单次对账耗时（只看单天量）。
2. **引擎可替换**——对账逻辑用 SQL（Spark SQL / Trino SQL），引擎可替换（Spark→Trino→Doris），业务 SQL 不变。如果 Spark 扛不住，换引擎即可。
3. **分层对账**——日级对账（增量，30 分钟）+ 周级对账（全量，兜底）+ 月级审计（财务复核）。数据量增长时，日级增量对账仍快（只看当天），全量对账调频（从每天降到每周）。

# 【拼多多供应链】采购到结算的全流程怎么设计？

> JD 依据："大供应链体系包括商品、供应商、仓配、运营和结算"。

## 一、正向全流程

```
采购 → 到货 → 质检 → 入库 → 上架 → 销售 → 出库 → 对账 → 结算
 │       │      │      │      │      │      │      │      │
采购单  到货单  质检单  库存+   SKU    订单   运单   账单   付款
                     现货    上架                 对账   凭证
```

## 二、各环节聚合根 + 状态机

**采购单**：
```
新建 → 已审批 → 已下单 → 部分到货 → 全部到货 → 已关闭
```

**到货单**：
```
待到货 → 已到货 → 质检中 → 质检通过 → 入库中 → 已入库
                  质检失败 → 退货
```

**库存**：
```
在途（采购未到货）→ 现货（可售）→ 冻结（下单未发货）→ 已出库
```

**结算**：
```
待对账 → 对账中 → 对账一致 → 待付款 → 已付款 → 已关账
                 对账差异 → 人工核查
```

## 三、领域事件串联

```
采购到货 → ArrivalEvent → 仓储服务消费 → 质检入库
质检通过 → QcPassedEvent → 库存服务消费 → 加现货
下单 → OrderCreatedEvent → 库存服务消费 → 冻结库存
出库 → ShippedEvent → 结算服务消费 → 生成账单
```

**落地**（本地消息表 + MQ）：
```java
@Transactional
public void arrival(ArrivalCmd cmd) {
    arrivalDao.insert(arrival);                    // 业务
    outboxDao.insert(new Outbox("ArrivalEvent"));  // 事件（同事务）
}
```

## 四、对账系统

**T+1 全量对账**：
```
库存系统数据 vs 仓储系统数据 vs 财务系统数据
  → 三方一致：平账
  → 三方差异：告警 + 人工核查
```

**实时差异监控**：库存负数、金额对不上等实时告警。

## 五、逆向流程（退货/退款）

```
用户退货 → 退货单 → 质检 → 退货入库（加现货）→ 退款 → 反向结算
```

逆向是正向状态机的镜像，复用质检/入库/结算能力。

## 六、底层本质

供应链本质是**"价值流的信息映射"**——商品实物流动（采购→仓储→配送）伴随信息流（单据状态）和资金流（对账结算）三流合一。

**DDD 的价值**：每个环节是独立限界上下文（采购域、仓储域、结算域），通过领域事件松耦合，让长流程可演进、可扩展。

**和传统 ERP 区别**：电商供应链是"事件驱动 + 最终一致"（高并发），ERP 是"过程驱动 + 强一致"（内部管理）。

## 常见考点
1. **状态机怎么实现**？——状态枚举 + 状态转移表（合法性校验）+ Spring StateMachine 框架。
2. **对账怎么保证准确**？——T+1 全量三方对账 + 实时差异监控 + 差异自动调账规则。
3. **供应链核心难点**？——多部门多系统协作（一致性）+ 海量数据（性能）+ 复杂业务规则（正确性）。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：采购到结算的全流程你用领域事件串联（采购到货事件触发仓储入库），而不是在一个大事务里调所有服务。为什么？大事务不也能保证一致性吗？**

大事务在跨服务跨部门场景不可行：
1. **事务时间太长**——从采购到货到质检入库到结算，跨度几小时到几天（到货可能分批、质检要人工），这么长的事务锁资源（DB 连接、行锁）浪费，且一旦某步失败全回滚（已入库的货退回去？）。
2. **跨服务事务不可能**——采购系统、仓储系统、结算系统是独立微服务独立库，没有一个数据库能管所有，XA 跨库性能极差。
3. **领域事件 + 最终一致**——每个环节是独立聚合根（独立事务），完成后发事件通知下游，下游异步处理。某步失败就补偿（如质检失败退货）。这让长流程"分而治之"，每环节独立成功失败，整体最终一致。这是供应链（长流程）和支付（短流程强一致）的根本区别。

### 第二层：证据与定位

**Q：某采购单显示"全部到货"但库存系统没有加现货（库存还是 0）。你怎么定位是到货事件没发、MQ 丢了，还是库存消费失败？**

四段排查（和 015 题类似但针对采购场景）：
1. **看采购单状态**——`SELECT status FROM purchase_order WHERE id=?`，确认是"全部到货"。看 `purchase_event_log` 是否有 `ArrivalEvent` 记录。
2. **看 outbox**——`SELECT * FROM outbox WHERE event_type='ArrivalEvent' AND biz_id=?`，status 是 SENT 还是 PENDING。PENDING 说明没投递（扫描任务问题）。
3. **看 MQ 消费**——库存服务的消费日志搜 `onArrival purchase_id=?`，确认是否消费。如果消费了但库存没加，看 `stock_log` 是否有这条到货的加库存记录。
4. **看库存状态**——到货后库存应该从"在途"转"现货"，`SELECT * FROM stock WHERE sku_id=? AND type='IN_TRANSIT'` 看在途库存是否清零、现货是否增加。如果在途库存清零但现货没增，是状态转移的 bug。

### 第三层：根因深挖

**Q：定位发现库存服务消费了 ArrivalEvent，但库存没加（stock_log 没记录）。根因是什么？**

几种可能：
1. **幂等冲突误判**——库存消费做了幂等（`SETNX processed:{arrival_id}`），如果之前消费过（部分成功——SETNX 成功但加库存失败），重试时被幂等挡住。看消费日志的"已处理"记录，如果是幂等挡住，要手动清幂等键重放。
2. **质检未通过**——到货后要质检通过才加现货库存。如果到货事件触发了质检流程，但质检结果 FAILED（商品破损），库存不会加现货而是走退货。查 `qc_result` 表的质检结果。
3. **SKU 映射错误**——采购单的 SKU 和库存系统的 SKU 映射断了（如采购用供应商 SKU 编码，库存用内部 SKU 编码，映射表缺记录）。加库存时 `WHERE sku_id=?` 找不到记录，静默失败。查 `sku_mapping` 表。
根因定位：看库存消费的具体异常日志（`try { addStock(); } catch` 的 catch 块有没有吞异常）。

**Q：那为什么不直接在采购到货时同步调库存服务（Feign RPC）加库存，失败就回滚到货状态？**

同步调用的问题：
1. **耦合**——采购系统要等库存系统返回才能完成到货，库存慢或挂了采购也失败，不应该。
2. **回滚不合理**——到货是物理事实（货已经到了仓库），不应该因为"库存系统加库存失败"就回滚到货状态。正确做法是到货状态确认（物理事实），库存加现货失败走重试/人工补偿（信息流问题）。
3. **长链路超时**——采购到货→质检→入库→加库存是长流程，同步调用链长，任意环节慢都阻塞。
事件驱动的价值是"物理事实确认（到货）和信息流处理（加库存）解耦"，到货状态代表"货到了"，加库存失败不代表"货没到"，用重试/对账修复信息流即可。

### 第四层：方案权衡

**Q：对账系统你做了 T+1 全量对账（每天凌晨对比三方数据）。但大促时 T+1 发现差异已经晚了（可能已经超卖）。要不要改成实时对账？**

实时对账和 T+1 对账各有适用场景，不是二选一：
1. **T+1 全量对账**——覆盖所有数据，发现"慢性不一致"（如某 SKU 长期少 1 件），准确性高。适合兜底。
2. **实时差异监控**——对关键指标（库存负数、金额异常波动）实时监控（Flink 流式处理库存变更事件），发现"急性错误"（如扣减 bug 导致超卖）立即告警。适合快速响应。
3. **组合策略**——实时监控关键指标（`stock < 0`、`amount_abnormal`）+ T+1 全量对账兜底。大促时加重实时监控（如每 5 分钟对账热销 SKU），平时 T+1 够。不要追求全量实时（数据量大、成本高），分场景。

**Q：为什么不直接用强一致（XA 两阶段提交）保证采购-库存-结算的数据始终一致，不用对账？**

XA 在供应链完全不可行：
1. **跨服务跨库**——采购、仓储、库存、结算是独立微服务独立库，XA 要一个全局事务管理器（如 Seata AT 模式）协调所有库，锁资源时间长（几小时事务），DB 连接占满。
2. **性能灾难**——XA 的 prepare + commit 两阶段，每步都要等所有参与者，长流程（采购到结算跨天）的 XA 事务会让数据库不可用。
3. **CAP 限制**——分布式系统下，网络分区时 XA 会阻塞（所有参与者等协调者），供应链这种长流程用 XA 任何网络抖动都卡死。
所以供应链只能用最终一致（事件驱动 + 对账兜底），强一致（XA）只在"同库短事务"用（如订单+订单项同库）。对账不是"兜底"，是最终一致架构的"必要组件"——它验证数据最终是否真的一致，发现并修复异常。

### 第五层：验证与沉淀

**Q：你怎么证明供应链全流程的事件链路没断、数据最终一致？**

对账体系验证：
1. **链路完整性**——每天对账：`采购到货单数 = 入库单数 = 库存增加记录数 = 结算账单数`，各环节的 `count` 应该匹配（考虑退货/损耗）。差值 > 阈值告警。拼多多每天凌晨跑这个"四流对账"。
2. **金额对账**——`SUM(采购单.金额) = SUM(入库单.金额) = SUM(账单.金额)`，金额三流合一，差值 < 0.01 元（四舍五入误差）正常，超阈值告警。
3. **链路追踪**——用 SkyWalking 追踪采购单号的全链路（到货→质检→入库→加库存→结算），任一 span 缺失说明链路断了。

**Q：怎么让团队维护这套长流程状态机的正确性？**

沉淀机制：
1. **状态机框架**——用 Spring StateMachine 或自研状态机引擎，所有状态转移走框架（声明 `transitions` 配置），框架校验合法性（非法跳转抛异常），禁止代码里直接 `setStatus()`。
2. **状态机可视化**——状态转移图用 PlantUML 维护，和代码同步，新人看图理解流程。每次新增状态/转移，先改图再改代码，图和代码不一致 CI fail。
3. **逆向流程复用**——退货/退款等逆向流程复用正向的状态机和事件链路（反向触发），不重新写一套，减少维护成本。Code Review 确保"退货入库"走的是"入库"的同一套代码，不是 copy 的另一份。

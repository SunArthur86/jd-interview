# 【拼多多供应链】预案、回滚、应急响应怎么做？用钱学森理论审视

> JD 依据："对系统稳定性和潜在问题高度敏感"。

## 一、钱学森工程理论在高可用的应用

钱学森提出工程系统四原则，完美适配高可用设计：

| 原则 | 高可用体现 |
|------|----------|
| **技术上可行** | 预案要可执行（不能纸上谈兵） |
| **经济上合理** | 降级成本可控（不能为 0.01% 故障投入 100% 成本） |
| **研制周期短** | 故障恢复快（MTTR < 5 分钟） |
| **能协调运转** | 跨团队应急有序（不能各干各的） |

## 二、预案分级

```
P0（电话）：核心服务不可用 → 5 分钟响应
P1（短信）：性能严重降级 → 15 分钟
P2（IM）：单实例异常 → 30 分钟
```

**预案文档**：
```markdown
# R-001: 商品中心不可用
触发: 商品查询 5xx > 10%
步骤:
  1. 通知 oncall
  2. 降级：切缓存兜底（关闭 DB 读）
  3. 排查 jstack/jmap
  4. 不可修复则回滚最近发布
  5. 恢复观察 30 分钟
```

## 三、回滚类型

| 类型 | 方式 | 注意 |
|------|------|------|
| 代码 | kubectl rollout undo | 验证数据兼容 |
| 配置 | Nacos 版本回滚 | 立即生效 |
| 数据 | binlog 逆操作 | 最难，要备份 |
| 流量 | DNS 切备机房 | 秒级 |

## 四、应急五步

```
发现（监控告警）
  ↓
通知（建事故群，召集）
  ↓
定位（看监控/trace/日志）
  ↓
止损（降级/回滚/切流）—— 5 分钟内必须有动作
  ↓
恢复 + 复盘
```

## 五、供应链实战预案

**大促供应链保障**：
```
T-30: 全链路压测 + 扩容 + 预案演练
T-7:  缓存预热 + 监控大盘
T-1:  降级非核心（推荐/评价）
当天: 实时监控 + 一键扩容/降级
```

## 六、底层本质

高可用的本质是**"接受故障必然，追求快速恢复"**。钱学森四原则是审视工程系统合理性的框架：
- 技术可行：预案真能跑
- 经济合理：成本和收益匹配
- 周期短：MTTR 最小化
- 协调运转：团队协同有序

## 常见考点
1. **回滚最大的坑**？——数据兼容性（schema 变更后老代码不认）。
2. **怎么验证预案有效**？——混沌工程（主动注入故障）。
3. **应急第一原则**？——先止损（降级/回滚）而非找根因。

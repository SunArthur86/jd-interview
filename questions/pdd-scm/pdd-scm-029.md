# 【拼多多供应链】预案、回滚、应急响应怎么做？用钱学森理论审视

> JD 依据："对系统稳定性和潜在问题高度敏感"。

## 一、钱学森工程理论在高可用的应用

钱学森提出工程系统四原则，完美适配高可用设计：

| 原则 | 高可用体现 |
|------|----------|
| **技术上可行** | 预案要可执行（不能纸上谈兵） |
| **经济上合理** | 降级成本可控（不能为 0.01% 故障投入 100% 成本） |
| **研制周期短** | 故障恢复快（MTTR < 5 分钟） |
| **能协调运转** | 跨团队应急有序（不能各干各的） |

## 二、预案分级

```
P0（电话）：核心服务不可用 → 5 分钟响应
P1（短信）：性能严重降级 → 15 分钟
P2（IM）：单实例异常 → 30 分钟
```

**预案文档**：
```markdown
# R-001: 商品中心不可用
触发: 商品查询 5xx > 10%
步骤:
  1. 通知 oncall
  2. 降级：切缓存兜底（关闭 DB 读）
  3. 排查 jstack/jmap
  4. 不可修复则回滚最近发布
  5. 恢复观察 30 分钟
```

## 三、回滚类型

| 类型 | 方式 | 注意 |
|------|------|------|
| 代码 | kubectl rollout undo | 验证数据兼容 |
| 配置 | Nacos 版本回滚 | 立即生效 |
| 数据 | binlog 逆操作 | 最难，要备份 |
| 流量 | DNS 切备机房 | 秒级 |

## 四、应急五步

```
发现（监控告警）
  ↓
通知（建事故群，召集）
  ↓
定位（看监控/trace/日志）
  ↓
止损（降级/回滚/切流）—— 5 分钟内必须有动作
  ↓
恢复 + 复盘
```

## 五、供应链实战预案

**大促供应链保障**：
```
T-30: 全链路压测 + 扩容 + 预案演练
T-7:  缓存预热 + 监控大盘
T-1:  降级非核心（推荐/评价）
当天: 实时监控 + 一键扩容/降级
```

## 六、底层本质

高可用的本质是**"接受故障必然，追求快速恢复"**。钱学森四原则是审视工程系统合理性的框架：
- 技术可行：预案真能跑
- 经济合理：成本和收益匹配
- 周期短：MTTR 最小化
- 协调运转：团队协同有序

## 常见考点
1. **回滚最大的坑**？——数据兼容性（schema 变更后老代码不认）。
2. **怎么验证预案有效**？——混沌工程（主动注入故障）。
3. **应急第一原则**？——先止损（降级/回滚）而非找根因。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：应急第一原则是"先止损再找根因"。但如果不找根因就回滚，可能回滚后问题还在（根因是数据或配置）。为什么还强调先止损？**

止损的目的是"快速恢复用户服务"，不是"解决问题"。一次故障的成本 = 影响用户数 × 持续时间。每多花 1 分钟找根因，用户就多受 1 分钟影响。先止损（回滚/降级）把"持续时间"压到最短，用户恢复后再慢慢排查根因。至于"回滚后问题还在"的情况，通过止损过程收集的信息（回滚时间点、回滚是否生效）反而能更快定位根因——如果回滚到 1 小时前版本，问题消失，说明是这段时间内的变更（代码/配置）导致。所以"先止损"不冲突"找根因"，止损是"收集信息的第一步"。

### 第二层：证据与定位

**Q：凌晨收到 P0 告警"商品中心 5xx > 10%"，你登系统发现是某次发布后开始的。你怎么确认是这次发布的问题、回滚能恢复？**

三步快速判断：
1. **看告警时间线**——告警 03:00 开始，看发布记录，如果 02:55 有发布（代码或配置），高度疑似。看发布的 diff（代码改动/配置变更）。
2. **看错误特征**——`grep 5xx access.log | head 100`，看错误的异常类型。如果是 `NullPointerException` 且堆栈指向新代码，坐实是发布引入。如果是 `ConnectionRefused`（下游挂了），可能是下游故障不是发布问题。
3. **灰度验证**——如果不确定，先在 1 个实例回滚（kubectl rollout undo --to-revision=X 单实例），看该实例是否恢复。如果恢复，全量回滚。这比直接全量回滚安全（避免误回滚引入新问题）。

### 第三层：根因深挖

**Q：回滚后问题消失，但根因是"新代码的 SQL 在大促数据量下慢查询（平时没事）"。为什么测试环境没发现？**

根因是"测试环境数据量不够"。测试环境的商品表只有几万行，新 SQL（如 `SELECT * FROM product WHERE category_id=? ORDER BY sales LIMIT 1000000,10` 深分页）在小数据量下毫秒级返回，测试通过。但生产千万行数据，深分页扫几百万行，慢查询超时。解决：
1. **压测前必须有生产级数据量**——测试环境用生产脱敏数据（或 TPC 生成千万级数据），新 SQL 必须在千万级数据下跑 EXPLAIN，`rows` 超阈值（如 > 10000）不让上线。
2. **SQL 审核平台**——DBA 用 Yearning/Archery 审核，模拟生产数据量跑 EXPLAIN，type/rows/filesort 不达标不让上线。
3. **灰度发布**——代码先灰度 10% 流量，观察新 SQL 的慢查询计数，无异常再全量。这次故障如果在灰度阶段能发现（10% 流量的慢查询告警），不至于全量故障。

**Q：那为什么不直接禁止深分页 SQL 上线，彻底避免？**

禁止深分页是规范，但要落地：
1. **代码扫描**——SonarQube 规则扫描 `LIMIT \d{5,}`（5 位以上数字的 LIMIT，即深分页），报 critical，review 确认是否必要。
2. **SQL 规范**——文档明确"C 端列表用游标分页（`WHERE id > last_id`），后台列表 LIMIT 不超 10000"，超了必须用 search_after 或 ES。
3. **Code Review**——所有带 LIMIT 的 SQL review 时看 offset 是否可控（用户能翻到第几页）。
但不能"一刀切禁止"——后台导出（如运营导商品）可能需要大 offset，这种走 ES 的 scroll 或离线导出，不是 MySQL。分场景处理。

### 第四层：方案权衡

**Q：你的预案分级 P0（5 分钟响应）/ P1（15 分钟）/ P2（30 分钟）。但实际故障往往"不知道多严重"（刚告警时判断不准）。怎么避免"P2 告警按 P2 处理，结果发现是 P0"？**

预案分级要"动态升级"：
1. **初始按影响定级**——告警时按"影响范围"定级（如商品查询 5xx > 10% 是 P0，单实例挂是 P2）。
2. **自动升级规则**——告警持续恶化时自动升级（如 P2 告警持续 5 分钟或影响面扩大到 50% 用户，自动升 P0）。Prometheus + Alertmanager 的 `inhibit_rules` 和 `group_wait` 支持动态升级。
3. **人工升级机制**——oncall 判断后可手动升级（"这个 P2 看着像 P0"），立即召集 P0 响应。宁可过度响应（P2 当 P0 处理）也不能漏判（P0 当 P2 拖延）。
所以分级是"起点"，不是"终点"，故障演进中动态调整。

**Q：为什么不所有告警都按 P0 处理（最快响应），省得判断？**

P0 响应代价大——P0 要电话通知 oncall、建事故群、可能半夜拉人起床。如果所有告警都 P0：
1. **疲劳轰炸**——oncall 每天被几十个 P0 告警轰炸，真 P0 来了反而麻木（"狼来了"）。
2. **人力浪费**——每次 P0 响应要召集多人，大量低优告警按 P0 处理浪费团队时间。
3. **干扰研发**——半夜 P0 告警吵醒 oncall，如果都是误报（如监控抖动），严重影响团队状态。
所以分级是"资源合理分配"——P0 给真正影响用户的故障（少而精），P2 给单实例/小范围异常（自动恢复或低优处理）。告警要"准而少"，不是"多而全"。

### 第五层：验证与沉淀

**Q：你怎么证明预案真的有效（不是纸上谈兵），故障时能按预案执行？**

混沌工程验证：
1. **定期演练**——每月一次故障注入演练（kill 商品中心某实例、注入慢 SQL、断网），团队按预案响应，记录"响应时间/止损时间/是否按预案执行"。拼多多双 11 前必跑全链路故障演练。
2. **预案有效性检查**——演练后检查预案的每一步是否可执行（如"回滚到 revision X"的 revision 是否存在、"切换备机房"的 DNS 是否能切）。预案过期（revision 被清理）要更新。
3. **复盘改进**——每次演练或真实故障后做 blameless 复盘（不追责，聚焦改进），预案的不足（如漏了某个降级开关）记录并改进。

**Q：怎么让团队的应急响应能力持续在线（不至于半年不故障忘了怎么处理）？**

三件事：
1. **oncall 轮值 + 交接**——oncall 每周轮值，交接时 review 本周告警和预案，保持敏感度。
2. **定期演练**——每月一次小范围演练（注入一个故障），每季度一次全链路演练（模拟大促故障），保持团队的应急肌肉记忆。
3. **预案 wiki + 自动化**——预案文档维护在 wiki（可搜索），关键步骤自动化（如"一键降级""一键回滚"的脚本/平台），oncall 不用记步骤，按平台指引执行。自动化程度越高，对人的记忆依赖越低，应急越可靠。

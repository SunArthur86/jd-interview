# 【拼多多供应链】可观测性、稳定性治理、跨团队推进怎么做？

> JD 依据："积极推动落地解决"（跨团队推进能力）。

## 一、可观测性三支柱

```
Metrics（聚合指标）   → 看趋势、告警（Prometheus）
Logs（事件日志）       → 看细节（ELK）
Traces（调用链路）     → 定位环节（SkyWalking）

协同: 告警(Metrics) → 定位环节(Trace) → 看细节(Log)
```

**供应链关键指标**：
```
业务: order_qps, order_p99_rt, 库存准确率, 对账差异率
系统: jvm_gc, db_slow_query, redis_hit_rate
```

## 二、稳定性治理（SLO）

```
SLI（指标）: P99 RT、可用性、错误率
SLO（目标）: P99 < 200ms、可用性 99.99%
错误预算 = 1 - SLO = 0.01%（月停 4.3 分钟）
```

**错误预算管理**：
- 预算未用完 → 可大胆发布（创新）
- 预算用完 → 停止发布（稳定优先）

## 三、跨团队推进（钱学森"协调运转"）

**1. 接口契约**：
```yaml
# 契约文件（OpenAPI/Protobuf），双方共同维护
openapi: 3.0
paths:
  /inventory/deduct:
    post:
      requestBody: { skuId: long, qty: int }
      responses: { 200: { success: bool } }
```
契约先行，避免"你改了我不知道"。

**2. 联合演练**：
- 大促前联合演练（商品+库存+订单+结算）
- 模拟故障（杀实例）验证降级

**3. 变更协同**：
```
变更日历: 标注各团队发布计划
封网期: 大促/节假日前禁止非紧急变更
联合发布: 跨服务依赖的变更协同上线
```

**4. 事故复盘（ blameless ）**：
- 不追责个人，找系统漏洞
- 5Why 根因分析
- 改进措施跟踪到闭环

## 四、供应链治理实战

**库存准确率 SLO**：
```
SLI: 库存 Redis vs DB 差异率
SLO: 差异率 < 0.01%
告警: 超 0.01% 自动告警 + 对账
```

**跨团队接口（商品→订单）**：
```
商品团队改 SKU 模型 → 提前 2 周通知订单团队
→ 双方约定迁移方案 → 灰度上线
```

## 五、底层本质

可观测性 + 治理 + 跨团队是钱学森**"能协调运转"**原则的落地：
- **看得清**（可观测性）：系统状态透明
- **管得住**（治理）：可用性量化管理
- **协同好**（跨团队）：多团队有序协作

这是大规模工程系统的管理基础——技术再好，管理跟不上也会出问题。

## 常见考点
1. **SLO 和 SLA 区别**？——SLO 是内部目标（团队承诺），SLA 是外部合同（违约赔偿）。
2. **错误预算怎么用**？——预算内可激进发布（创新），超预算保守（稳定）。
3. **跨团队最大难点**？——信任和沟通，解法是契约先行 + 联合演练 + blameless 复盘。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：可观测性你配了 Metrics/Logging/Tracing 三支柱。但监控面板几十个、告警几百条，oncall 经常"告警疲劳"（忽略告警）。为什么不减少监控，只看核心指标？**

减少监控不是解法，"告警治理"才是。告警疲劳的根因是"告警太多 + 告警不准"。解法：
1. **分级告警**——核心指标（order_success_rate < 99%）才 P0 告警（电话），辅助指标（GC 频率、慢查询数）只记 dashboard 不告警。oncall 只被核心告警打扰。
2. **告警去重 + 抑制**——同一故障可能触发多个告警（GC 慢 → RT 高 → 错误率高），用 Alertmanager 的 `inhibit_rules` 聚合（一个根因告警，抑制下游告警），oncall 只收 1 条。
3. **告警 SLO**——告警本身设 SLO：`告警准确率 > 90%`（90% 的告警是真问题），低准确率的告警（如某 CPU 阈值经常误报）调阈值或删。
所以不是"减少监控"（监控要全），而是"减少告警"（告警要准）。

### 第二层：证据与定位

**Q：用户反馈下单偶发失败，但监控的 `order_success_rate` 是 99.9%（正常）。你怎么定位这 0.1% 的偶发失败？**

用 Tracing 定位长尾：
1. **看错误 trace**——SkyWalking/Jaeger 按 `status=error` 过滤 trace，看失败请求的完整链路。如果失败的请求都卡在某个下游（如库存扣减超时），定位到该下游。
2. **看错误日志**——ELK 搜 `order_id` 或 `error`，看失败的具体异常。如果是 `RedisConnectionException`，是 Redis 偶发网络抖动；如果是 `optimistic_lock_conflict`，是并发冲突。
3. **看长尾分布**——`order_p999_rt` 如果飙高（平时 200ms，偶发 2 秒），说明 0.1% 的请求卡在某个环节。arthas trace 看慢请求的耗时分布。
0.1% 的偶发问题要看"长尾"（P99/P999），不是平均值。平均值正常不代表没有问题。

### 第三层：根因深挖

**Q：定位发现偶发失败是"Redis 连接超时"（1 秒内几次连接拒绝）。Redis 本身没挂，为什么偶发连接拒绝？**

Redis 连接超时的根因通常不是 Redis 本身，是"连接池配置或网络"：
1. **连接池太小**——高峰期并发请求多，连接池（如 Lettuce/Jedis 的 `maxActive=50`）打满，新请求等连接超时。看连接池的 `pool_waiting_count`，如果 > 0，是池子不够。
2. **网络抖动**——某时刻网络丢包或延迟飙升（如宿主机网络拥堵），Redis 连接超时。看网络监控（`tcp_retransmit_count`）。
3. **Redis 阻塞**——Redis 执行了慢命令（如 `KEYS *`）阻塞主线程，新连接处理不了。看 Redis 的 `slowlog`（`SLOWLOG GET 10`）。
根因定位：看连接池监控 + 网络 + Redis slowlog，交叉对比失败时刻。

**Q：那为什么不直接把连接池调到无限大（不限制），彻底避免连接等待？**

无限连接池有害：
1. **Redis 端拒绝**——Redis 单线程，连接数太多（如几万）时，处理连接握手开销大，反而变慢。Redis 默认 `maxclients=10000`，超了直接拒绝。
2. **内存浪费**——每个连接占内存（如 10KB），几万连接占几百 MB，应用内存浪费。
3. **级联超时**——如果下游（Redis）慢，无限连接池会让应用创建更多连接涌过去，Redis 更慢，恶性循环。
正确做法是"有限连接池 + 合理超时 + 熔断"——连接池按峰值并发设（如峰值 200 并发 × 1.5 = 300 连接），超时 50ms（快速失败），Redis 慢时熔断走降级（返回缓存）。

### 第四层：方案权衡

**Q：错误预算管理（SLO 99.99% = 月停 4.3 分钟）你怎么推行到团队？业务方要 100% 可用，你说"允许 4.3 分钟故障"他们不接受。**

错误预算不是"允许故障"，是"量化管理稳定性"。推行策略：
1. **从成本切入**——100% 可用意味着冗余投入巨大（多机房热备、7×24 值班），成本是 99.99% 的 5-10 倍。问业务方"多花 5 倍成本换 0.01% 可用性值吗"，大多数业务会选 99.99%。
2. **分场景定 SLO**——核心链路（下单/支付）SLO 99.99%，非核心（推荐/评价）SLO 99.9%。不是所有功能都要 99.99%，按业务重要性分级。
3. **错误预算是"创新空间"**——预算内（没故障）可以激进发布（快速迭代新功能），预算耗尽（故障多）停止发布（稳定优先）。这让团队有"节奏感"，不是无脑追求 100%。
所以错误预算是"用数据驱动稳定性投入"，让业务方理解"99.99% 性价比最高"，而不是纠结 0.01%。

**Q：为什么不直接追求 100% 可用（技术上的极限）？**

100% 可用在分布式系统不可能（数学上）：
1. **网络分区**——CAP 下，网络分区时要么牺牲可用要么牺牲一致，不可能两者兼得。
2. **硬件故障**——磁盘损坏、网络断、机房断电，物理故障不可消除。
3. **软件 bug**——任何复杂系统都有 bug，不可能零缺陷。
所以追求 100% 是"不切实际的目标"，反而让团队疲于奔命（过度冗余、不敢发布）。错误预算的智慧是"接受故障必然，用预算管理频率和影响"，让团队在"稳定性"和"迭代速度"之间平衡。Google SRE 的核心理念就是"错误预算让开发和稳定不再对立"。

### 第五层：验证与沉淀

**Q：你怎么证明可观测性体系真的能"看清系统状态"，故障时能快速定位？**

混沌演练验证：
1. **注入故障看发现时间**——主动注入故障（kill 实例、慢 SQL、Redis 超时），测量"告警触发时间"（MTTD）和"oncall 定位根因时间"。目标 MTTD < 1 分钟，定位 < 5 分钟。如果超时，是监控盲区（没告警）或 dashboard 不清晰。
2. **监控覆盖率**——统计"核心链路的 trace 覆盖率"（所有关键接口是否有 Tracing），目标 > 95%。关键业务指标（order_qps/success_rate）是否有 dashboard + 告警，目标 100%。
3. **告警准确率**——统计"告警触发的真故障比例"（告警中真问题的比例），目标 > 90%。低准确率的告警调阈值或删。

**Q：怎么让跨团队协作（接口契约/联合演练/复盘）可持续，不流于形式？**

机制保障：
1. **契约测试自动化**——接口契约（OpenAPI/Protobuf）用 Pact 或 Spring Cloud Contract 做消费者驱动测试，任何一方改契约，另一方的 CI 自动跑测试，不通过不让发布。契约不只是文档，是可执行的测试。
2. **联合演练制度化**——大促前必须联合演练（商品+库存+订单+结算），演练报告作为上线依据，不演练不让上。演练模拟真实故障（杀实例、断网），验证跨团队的降级/熔断协同。
3. **blameless 复盘文化**——故障复盘不追责个人（"谁写的 bug"），找系统漏洞（"为什么 review 没发现/为什么测试没覆盖"）。改进措施有 owner 和 deadline，下次复盘检查闭环。拼多多复盘文化就是"对事不对人"，让团队敢报告问题（而不是隐瞒）。

**30 秒电梯演讲：** 分布式系统跨多服务团队，如何看清状态、量化可用性、协同有序？简单说就是——可观测性（Metrics+Logs+Traces）让系统从黑盒变白盒，稳定性治理（SLO+错误预算）让可用性可量化，跨团队推进（接口契约+联合演练）让协作有序。

# 【拼多多供应链】分布式事务怎么解决？下单扣库存跨服务怎么保证一致？

> JD 依据："分布式系统的设计和应用"。

## 一、分布式事务方案对比

| 方案 | 一致性 | 性能 | 业务侵入 | 适用 |
|------|--------|------|---------|------|
| XA（2PC） | 强 | 差（锁久） | 小 | 传统金融 |
| TCC | 强 | 中 | 大（写 Try/Confirm/Cancel） | 扣款 |
| Saga | 最终 | 好 | 中（写补偿） | 长流程 |
| 本地消息表 | 最终 | 好 | 小 | 异步通知 |
| 事务消息（RocketMQ） | 最终 | 好 | 中 | 电商 |

## 二、本地消息表（供应链最常用）

```java
@Transactional
public void createOrder(OrderReq req) {
    orderDao.insert(order);                    // 业务表
    outboxDao.insert(new Outbox(               // 消息表（同事务）
        "OrderCreatedEvent", toJson(order)));
}
// 定时扫描 outbox，投递 MQ，成功删除
```

下游（库存服务）订阅，幂等消费：
```java
public void onOrderCreated(OrderEvent e) {
    if (!redis.setnx("processed:" + e.orderId, "1", 24, HOURS)) return;
    stockService.deduct(e.items);
}
```

## 三、TCC（强一致，扣款场景）

```
Try：预留资源（冻结库存）
   ↓ 成功
Confirm：确认（扣减冻结库存）
   ↓ 失败
Cancel：回滚（解冻）
```

业务侵入大（每个操作写三个方法），适合核心扣款。

## 四、Saga（长流程）

```
下单 → 扣库存 → 生成支付 → 通知物流
失败补偿：取消支付 → 回库存 → 取消订单
```

每步有对应补偿，任一失败反向补偿。

## 五、供应链实战

**下单扣库存**（本地消息表 + 幂等）：
```
订单服务：createOrder() + 写消息表（同事务）
   ↓ MQ
库存服务：幂等消费，扣库存（失败重试，重试到成功或告警）
```

**退款回库存**（同上，反向事件）。

**采购到货入库**（Saga）：
```
采购到货 → 仓储入库 → 更新库存 → 通知采购方
失败：回滚入库 → 标记到货失败
```

## 六、底层本质

分布式事务本质是 **CAP 取舍**：
- 强一致（XA）：锁久，牺牲可用性
- 最终一致（柔性事务）：异步，牺牲一致性窗口

供应链选最终一致——业务可容忍秒级不一致，换高并发。

## 常见考点
1. **TCC 和 2PC 区别**？——2PC 资源层（DB 锁）；TCC 业务层（业务代码实现 Try/Confirm/Cancel）。
2. **本地消息表为什么可靠**？——业务和消息同事务，要么都成功要么都失败；定时投递保证最终送达。
3. **RocketMQ 事务消息原理**？——半消息 + 本地事务 + 回查，和本地消息表思路类似但 MQ 原生支持。

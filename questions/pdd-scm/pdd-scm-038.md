# 【拼多多供应链】设计智能供应链中台

> JD 依据：JD 8 业务中台方向。

## 一、中台架构

```
业务层（自营/跨境/买菜/多多买菜）
   ↓ 自助接入
中台能力层
  ├─ 基础能力：商品/库存/订单/采购/结算
  └─ AI 能力：销量预测/智能补货/异常检测/供应商评估
   ↓
数据中台（特征/训练/推理）
```

## 二、AI 能力作为中台服务

```yaml
services:
  - name: sales_forecast
    type: AI
    input: { sku_id, days: 30 }
    output: { forecast, confidence }
    sla: { rt: 100ms, accuracy: mape<0.2 }

  - name: auto_replenish
    type: AI_AGENT
    input: { sku_id }
    output: { order_qty, supplier, reason }

  - name: anomaly_detect
    type: AI
    input: { metric, window }
    output: { is_anomaly, score }
```

## 三、自助接入

新业务（如多多买菜）接入：
1. 控制台选能力（商品/库存/预测）
2. 配业务规则（补货策略/异常阈值）
3. 灰度发布 → 全量

## 四、自然语言配置（AI 加持）

```
运营: "叶菜类每天补货，库存低于 100kg 自动下单"
LLM 翻译 → 中台规则配置
```

## 五、评估闭环

```
AI 决策（补货量）→ 落库 → 实际销量对比 → 标注（对/错）
   → 模型迭代 → 灰度上线
```

## 六、底层本质

智能中台本质是**"AI 能力的平台化复用"**——从"每业务线自建 AI"到"中台提供 AI 服务，业务调用"。这是工业化的规模效应。

## 常见考点
1. **中台和微服务区别**？——中台是业务能力复用（战略），微服务是技术拆分（战术）。
2. **怎么衡量中台价值**？——复用率、迭代速度、成本节约。
3. **AI 怎么融入中台**？——AI 能力（预测/补货）作为服务，业务按需调用。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：智能供应链中台你把 AI 能力（销量预测/补货/异常检测）封装成中台服务，各业务线调用。为什么不各业务线自建 AI（自营/跨境/买菜各自一套），中台不是会有"一刀切"的问题吗？**

中台 vs 自建的权衡在于"规模效应 vs 定制化"：
1. **中台价值**——AI 能力的底层逻辑一致（预测用同样模型，补货用同样算法），中台建一次，各业务线复用，省研发成本（不用各建一套）。同时数据打通（各业务线的销量数据汇总训练，模型更准）。
2. **定制化需求**——各业务线有特殊规则（如多多买菜是生鲜短保质期，补货逻辑和自营服饰不同）。如果中台完全"一刀切"，满足不了特殊场景。
3. **解法**——中台提供"基础能力 + 可配置规则"。预测服务是统一的（底层模型一致），但补货策略是可配置的（各业务线配自己的安全库存系数、补货频率、供应商）。中台不强制"一刀切"，而是"核心统一 + 边缘灵活"。
判断标准：如果某能力跨 3+ 业务线复用，进中台；如果只 1 个业务线用，自建。

### 第二层：证据与定位

**Q：多多买菜接入中台后，叶菜类补货经常缺货（叶菜保质期短，预测不准）。你怎么定位是中台预测模型不适合生鲜、配置不对，还是数据不足？**

三步排查：
1. **看预测准确率**——叶菜类的预测 MAPE，对比自营服饰的 MAPE。如果叶菜 MAPE 40%（服饰 15%），是模型不适合生鲜（叶菜受天气/损耗/短保质期影响大，通用预测模型没考虑）。
2. **看配置**——买菜的补货配置（安全库存系数、补货频率）是否针对叶菜调整。如果配置沿用服饰的（安全库存 1.2 倍），不适合叶菜（叶菜要低库存快周转，安全库存 0.5 倍 + 每天补货）。
3. **看数据**——买菜叶菜的历史销量数据是否足够（至少 3 个月），数据少模型训不准。如果是新接入（数据不足），用相似品类迁移学习。
定位：分环节看（预测准不准 / 配置合不合理 / 数据够不够），针对性修复。

### 第三层：根因深挖

**Q：叶菜类预测 MAPE 40%，根因是模型没考虑"短保质期损耗"。模型预测卖了 1000 份，实际有 200 份损耗（变质扔掉），导致补货不够。怎么解决？**

根因是"预测目标定义不清"：
1. **区分销量 vs 补货量**——预测应该是"实际需求"（用户想买多少），不是"实际销售"（扣掉损耗后的）。如果模型用"实际销售"（800 份）训练，预测出 800，但实际需求是 1000（200 损耗），补货 800 就缺货。
2. **加损耗特征**——模型输入加"历史损耗率"特征（如叶菜平均损耗 20%），预测时加回损耗量。预测"需求 1000 = 销售 800 + 损耗 200"，补货按需求补。
3. **分阶段预测**——先预测"进货量"（考虑损耗，多备 20%），再预测"销售"（实际卖出）。两个模型分工，不是混在一起。
根因：生鲜类目的预测必须显式建模"损耗"（变质/过期/损坏），不能用通用模型（默认无损耗）。

**Q：那为什么不直接在补货环节加固定损耗系数（如统一加 20%），不依赖模型？**

固定损耗系数的问题：
1. **损耗率动态变**——叶菜损耗受天气（高温损耗高）、运输（颠簸损耗）、存储（冷库故障）影响，不是固定 20%。固定系数要么过度备货（正常天气损耗 10%，备了 20% 浪费），要么备货不足（异常天气损耗 30%，备 20% 缺货）。
2. **品类差异**——叶菜（损耗 20%）、根茎类（损耗 5%）、肉类（冷藏损耗 2%），品类间损耗差异大，统一系数不合理。
所以策略是"模型预测损耗率（动态）+ 人工兜底异常"——模型根据天气/品类/运输条件预测损耗率，正常情况按模型；异常情况（如冷库故障）人工调整。固定系数只作 fallback（模型不可用时用）。

### 第四层：方案权衡

**Q：中台的 AI 服务你做了 SLA 约束（如预测服务 RT < 100ms、MAPE < 0.2）。但 AI 模型的准确率不可控（受数据质量影响），怎么保证 SLA？**

AI 服务的 SLA 和传统服务不同（确定性 vs 概率性），要分层承诺：
1. **性能 SLA（可控）**——RT、可用性这些工程指标可控（优化推理 + 冗余），承诺 RT < 100ms、可用性 99.9%，没达到是工程问题，可追责。
2. **质量 SLA（概率性）**——准确率（MAPE）受数据/模型限制，只能"尽力而为"。承诺"MAPE 月均 < 0.2"（不是每次都 < 0.2），个别 case 超了不违约。
3. **降级兜底**——AI 服务故障时降级到规则引擎（如预测挂了用历史均值补货），保证业务不中断。SLA 承诺"AI 不可用时降级到规则，RT 仍 < 100ms"。
4. **持续优化**——质量 SLA 不达标时，触发模型迭代（加特征/换模型），持续提升。业务方理解"AI 是概率性的，会越来越好但不是 100%"。

**Q：为什么不把 AI 服务做成同步强依赖（业务必须等 AI 返回才决策），保证决策最准？**

强依赖的风险是"AI 故障 = 业务中断"：
1. **可用性下降**——AI 服务 99.9% 可用（月停 43 分钟），强依赖意味着业务也跟着停。弱依赖（AI 挂了降级到规则）则业务仍可用。
2. **延迟敏感**——下单链路要快（50ms 内响应），AI 预测 100ms 会拖慢。弱依赖让 AI 异步（预测结果缓存，下单时读缓存不实时调 AI）。
3. **故障隔离**——AI 模型更新/重训可能导致短时不可用（或质量波动），强依赖会波及业务。弱依赖隔离故障。
所以策略是"核心链路弱依赖 AI（读缓存/降级规则），非核心链路可强依赖（如离线补货建议生成）"。AI 是"增强"不是"必需"，业务在没有 AI 时仍能运行（用规则兜底）。

### 第五层：验证与沉淀

**Q：你怎么证明智能中台的价值（复用率、降本增效）？**

中台价值指标：
1. **复用率**——中台 AI 服务被多少业务线调用（如 5 个业务线用预测服务）、调用量（日均 API 调用数）。复用率越高，中台摊薄成本越值得。
2. **降本**——对比"各业务线自建 AI"vs"中台统一提供"的研发成本。中台建一套（10 人月），5 业务线复用省了 40 人月（5 × 10 - 10）。
3. **增效**——新业务线接入中台的时间（如多多买菜接入预测服务用 1 周 vs 自建预测用 3 个月），提速业务上线。
4. **业务效果**——各业务线用中台 AI 后的业务指标（如缺货率降 30%、周转率提升 15%），证明 AI 真有效。

**Q：怎么让智能中台可持续演进（不只服务现有业务，还能支撑新业务）？**

平台化设计：
1. **能力抽象**——AI 能力抽象为标准接口（`predict(input) → output`），新业务接入只配置（不需要改中台代码）。如新业务"多多医药"接入预测，只需配"药品类目"，复用预测服务。
2. **插件化扩展**——中台支持插件（如自定义预测模型、自定义补货规则），业务线有特殊需求时开发插件接入，不污染中台核心。
3. **自助接入平台**——业务方通过控制台自助接入（选能力 + 配规则 + 灰度发布），不需要中台团队人工支持，降低接入成本。
4. **能力市场**——中台维护"能力目录"（有哪些 AI 服务、各服务效果、适用场景），业务方像逛超市一样选能力。新能力（如"智能定价"）开发后上架，全公司可见可用。

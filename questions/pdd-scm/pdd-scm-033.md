# 【拼多多供应链】AI Harness 怎么工程化？让 LLM 生产级运行

> JD 依据：JD 9 大模型工程师对应。

## 一、AI Harness 架构

```
上层（Agent/应用）
   ↓
AI Harness 平台
  ├─ LLM 网关（多模型路由+限流+成本）
  ├─ 推理服务（vLLM 集群）
  ├─ 工具市场（MCP Server）
  ├─ 可观测（Trace/Metrics）
  └─ 评估平台（离线+在线）
   ↓
基础设施（GPU 集群、向量库、知识库）
```

## 二、LLM 网关

```java
public class LLMGateway {
    public Completion complete(Request req) {
        String model = router.select(req.getScenario());  // 场景路由
        rateLimiter.tryAcquire(req.getUserId());            // 限流
        quotaChecker.check(req);                            // 配额
        try {
            return callWithFallback(model, req);            // 带降级
        } catch (Exception e) {
            return fallback(req);                           // 降级规则
        }
    }
}
```

- 路由：简单场景 → 便宜模型；复杂 → 强模型
- 成本：实时 token 统计 + 预算告警
- 降级：主模型挂 fallback 到备模型

## 三、推理优化

- **vLLM/PagedAttention**：KV cache 分页管理，吞吐 5-10 倍
- **量化**：INT8/INT4，速度 +2-3 倍
- **批处理**：Continuous Batching，GPU 持续满载

## 四、工具管理（MCP）

```yaml
tools:
  - name: query_inventory
    mcp_server: scm-tools
  - name: create_purchase
    mcp_server: scm-tools
```

Agent 按需调用，统一注册发现。

## 五、评估闭环

```
LLM 决策 → 落库 → 人工标注（对/错）→ 微调 → 灰度上线
```

## 六、底层本质

AI Harness 是 LLM 时代的"操作系统"——上层（Agent）不关心底层（模型/GPU/工具），底层细节由 Harness 管理。

## 常见考点
1. **LLM 推理最有效优化**？——vLLM/PagedAttention（吞吐 5-10 倍）+ 量化。
2. **多模型怎么路由**？——按场景（简单/复杂）+ 成本（便宜/贵）+ 可用性。
3. **怎么评估 LLM 决策质量**？——人工标注测试集 + A/B + 业务指标。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：AI Harness 你设计了 LLM 网关（路由 + 限流 + 降级）。为什么不直接让业务方调 LLM API（如 GLM SDK），要中间加一层网关？**

网关的价值是"统一治理"：
1. **成本控制**——没有网关，各业务方直接调 LLM API，token 消费无法统一统计和限额。某业务 bug 导致无限调用，账单爆炸。网关统一计费 + 预算告警 + 按业务线限额。
2. **模型路由**——不同场景用不同模型（简单任务用便宜模型，复杂用强模型）。没有网关，业务方各自选模型（可能都选最贵的），成本高。网关按场景路由，自动选性价比最优的模型。
3. **降级容灾**——主模型（如 GLM-Plus）挂了，网关自动 fallback 到备模型（如 GLM-Flash 或其他厂商），业务无感。没有网关，业务方各自处理容灾，重复建设。
4. **统一监控**——所有 LLM 调用的 latency/error_rate/cost 统一上报，大盘可视化。没有网关，各业务方自己埋点，数据格式不一。
所以网关是"LLM 的 API Gateway"，和微服务的 API Gateway 一样的治理逻辑。

### 第二层：证据与定位

**Q：业务方反馈 LLM 调用慢（10 秒），但你看网关监控，P99 才 2 秒。怎么定位是网关问题、模型慢，还是业务方误报？**

分环节排查：
1. **看网关 trace**——LLM 网关记录每次调用的 `gateway_start`、`model_start`、`model_end`、`gateway_end`。如果 `model_end - model_start` = 8 秒（模型推理慢），是模型问题；如果 `gateway_start - 业务调用时间` = 5 秒（网关排队），是网关限流/排队。
2. **看限流**——如果业务方的调用被限流（`rate_limiter` 拒绝后重试），累计耗时会长。看网关的 `reject_count` 和 `retry_count`。
3. **看业务方埋点**——业务方报"10 秒"，可能是从"发请求"到"收到响应"的端到端时间，包含网络（业务方→网关→模型）往返。如果业务方在跨机房调用，网络 RTT 可能占几秒。让业务方看 trace（全链路追踪），定位耗时在哪个环节。

### 第三层：根因深挖

**Q：定位是模型推理慢（model 推理 8 秒）。但这个场景之前只要 2 秒，为什么变慢了？**

模型推理变慢的根因：
1. **prompt 变长**——业务方在 prompt 里塞了更多 context（如 RAG 检索的文档从 3 篇增到 10 篇），输入 token 翻倍，推理慢。看 `prompt_tokens` 指标，如果从 1000 涨到 5000，是输入变长。
2. **并发高排队**——GPU 推理是批处理（Continuous Batching），高并发时请求在 GPU 队列排队。看 GPU 的 `queue_size` 和 `batch_size`，如果队列堆积，是容量不足，要扩 GPU。
3. **模型版本变更**——运维升级了模型（如从 INT8 量化改成 FP16 精度），推理慢但质量好。看模型版本和量化配置。
4. **长输出**——LLM 输出的 token 数变多（如从 100 token 涨到 1000 token），生成时间是输出长度的线性函数（每个 token 自回归生成）。看 `completion_tokens`。
根因定位：看 token 数（输入 + 输出）和 GPU 队列，判断是输入变长 / 输出变长 / 排队。

**Q：那为什么不直接给业务方限制 prompt 长度（如最多 1000 token），避免输入过长导致慢？**

限制 prompt 长度要分场景：
1. **简单任务（分类/抽取）**——限制 1000 token 够（输入文本不长），强制限制合理。
2. **复杂任务（RAG 问答/合同审查）**——需要长 context（如 10 篇文档），限制 1000 token 会截断关键信息，任务做不了。
所以策略是"按场景配额"——简单任务限 1000 token（便宜快），复杂任务限 8000 token（贵慢但必要）。网关按场景路由，不同场景不同 token 上限和模型。同时监控"无效 context"（如 RAG 检索了无关文档），优化检索质量减少 context 长度，而不是硬限制。

### 第四层：方案权衡

**Q：推理优化你用 vLLM（PagedAttention）+ 量化（INT8）。但量化会损失精度（模型变笨），怎么权衡"速度"和"质量"？**

量化的权衡是按场景分：
1. **简单任务用量化模型**——工单分类、属性抽取等结构化任务，INT8 量化的精度损失可接受（准确率降 1-2%），但速度提升 2-3 倍，成本降。用 INT8。
2. **复杂任务用 FP16/BF16**——合同审查、复杂推理等对精度敏感的任务，量化可能导致明显能力下降（如漏抽风险条款），用全精度。
3. **A/B 验证**——量化模型上线前，用测试集（1000 条标注数据）对比量化前后的准确率，降 < 2% 才上线。如果降 5%+，该场景不适合量化。
4. **混合部署**——部署两套模型（INT8 量化 + FP16 全精度），网关按场景路由。简单任务走量化（快），复杂走全精度（准）。

**Q：为什么不直接用最强的 GPU（如 H100）跑全精度模型，不量化，既快又准？**

H100 贵 + 供不应求：
1. **成本**——H100 单卡几万美元，部署几十张卡成本百万美元级。供应链 AI 不是核心（不像自动驾驶），ROI 不划算。
2. **供应**——H100 产能受限，大厂优先囤货，中小企业难买到。
3. **功耗**——H100 功耗高（700W），数据中心电费和散热成本高。
所以策略是"消费级 GPU（如 A10/L20）+ 量化优化"——用便宜的卡 + 软件优化（vLLM/量化）达到接近 H100 的性价比。只有大模型预训练（必须 H100 集群）才用顶级卡，推理服务用中端卡 + 优化足够。

### 第五层：验证与沉淀

**Q：你怎么证明 AI Harness（网关 + 推理优化 + 评估闭环）让 LLM 应用稳定且高效？**

多维指标：
1. **可用性**——LLM 调用的 `success_rate` > 99.9%（网关降级兜底），`p99_latency` < 3 秒（推理优化生效）。
2. **成本**——单次调用平均成本（`total_cost / total_calls`），优化后应下降（量化 + 路由便宜模型）。月度账单监控，超预算告警。
3. **质量**——评估闭环的"人工标注准确率"应 > 90%（业务可接受），且持续提升（微调/prompt 优化）。
4. **故障演练**——主动 kill 主模型（如 GLM-Plus），验证网关是否秒级 fallback 到备模型，业务无感。

**Q：怎么让团队的 LLM 应用开发规范统一（网关接入、prompt 管理、评估）？**

沉淀 LLM 工程平台：
1. **统一接入**——所有业务方必须通过 LLM 网关调用（禁止直连 SDK），网关统一计费/限流/监控。SonarQube 扫"直连 LLM SDK"的代码报 blocker。
2. **Prompt 平台**——prompt 集中管理（版本化 + 评估），业务方在平台编辑 prompt，平台自动跑测试集评估，通过后才上线。
3. **评估自动化**——每个 LLM 应用接入评估流水线（CI 跑测试集），prompt/模型变更自动评估，准确率退化阻断上线。
4. **成本看板**——各业务线的 token 消费看板，超预算自动限流（防止某业务 bug 拖垮全公司预算）。

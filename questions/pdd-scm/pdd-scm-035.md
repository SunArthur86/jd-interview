# 【拼多多供应链】RAG 怎么用在供应链？

> JD 依据："大模型业务探索"。

## 一、RAG 流程

```
离线建库:
  文档（规则/流程/案例）→ 切分 chunk → Embedding → Milvus 向量库

在线检索:
  问题 → Embedding → 检索 Top-K → 拼 prompt → LLM 生成（带引用）
```

## 二、供应链 RAG 应用

**1. 采购规则问答**：
```
运营: "单笔采购超过 100 万需要谁审批？"
RAG 检索采购规则文档 → LLM: "根据《采购管理制度》第 5 条，超 100 万需 VP 审批"
```

**2. 历史案例检索**：
```
事件: "供应商 A 到货延迟"
RAG 检索历史类似案例 → LLM: "历史 3 次类似案例处理方案：催货/换供应商/通知用户"
```

**3. 运营知识助手**：
```
客服: "这个 SKU 为什么下架了？"
RAG 检索下架记录 + 规则 → LLM: "因质量投诉超阈值，按规则 X 下架"
```

## 三、高级 RAG

- **查询改写**：原始问题改写更精准
- **HyDE**：LLM 先生成假设答案再检索
- **重排序**：向量召回 Top-50 → cross-encoder 精排 Top-5
- **混合检索**：向量（语义）+ 关键词（精确）

## 四、防幻觉

决策必须引用证据：
```json
{
  "answer": "需要 VP 审批",
  "evidence": ["采购管理制度.docx#section5"],
  "confidence": 0.95
}
```

## 五、底层本质

RAG 是**"知识外置"**——LLM 参数存推理能力，知识存外部向量库，解耦让知识可独立更新不用重训。

## 常见考点
1. **RAG 检索不准怎么办**？——优化 embedding 模型 + chunk 策略 + 重排序 + 查询改写。
2. **RAG 和微调选哪个**？——知识频繁变 RAG；固定模式微调；最佳组合。
3. **向量库为什么近似检索**？——精确 KNN 亿级不可行，HNSW 近似 O(log n)。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：供应链运营问 RAG 助手"单笔采购超 100 万需要谁审批"，你用 RAG 检索文档而非把审批规则写进 prompt。为什么不直接 prompt 里写规则（few-shot），要外挂向量库？**

RAG 和 prompt 写死的区别在"知识更新"：
1. **规则会变**——审批制度每季度更新（金额阈值/审批人调整），prompt 写死每次改要改代码 + 重新部署。RAG 更新向量库（重新 embedding 新文档）即可，不动代码。
2. **规则量大**——采购管理有几百条规则（各品类/各金额/各地区），全塞 prompt 会超 context 限制（token 爆炸），且 LLM 在长 prompt 里"找不到"相关规则。RAG 只检索相关的 3-5 条，精准。
3. **多文档源**——规则散落在多个文档（采购制度/财务制度/合规手册），prompt 无法统一管理。RAG 统一索引所有文档，检索时跨文档找。
所以策略是"推理能力用 LLM 参数，知识用 RAG 外挂"，知识可独立更新不重训模型。

### 第二层：证据与定位

**Q：运营反馈 RAG 回答错了（问"采购超 100 万审批"，答了"超 50 万的审批流程"）。你怎么定位是检索没找对文档、LLM 理解错，还是文档本身错？**

三步排查：
1. **看检索结果**——RAG 系统记录每次检索的"召回文档"（Top-5）。如果召回的是"50 万审批"的文档（不是"100 万"的），是检索不准（embedding 没区分 50 万 vs 100 万）。看召回文档的相关性。
2. **看 LLM 生成**——如果召回的是对的（含 100 万审批规则），但 LLM 生成答案错了（答 50 万），是 LLM 幻觉（没遵循检索内容）。看 LLM 的 prompt 是否有"只基于检索内容回答"的约束。
3. **看文档本身**——如果文档里写的就是错的（如采购制度文档版本旧，还是 50 万阈值，实际已改成 100 万），是文档没更新。看文档版本和更新时间。
定位方法：RAG 必须输出"检索的文档片段 + LLM 答案"，人工对照看错在哪步。

### 第三层：根因深挖

**Q：检索召回的文档不准（找到的是"50 万审批"不是"100 万"）。根因是 embedding 模型差、chunk 策略错，还是查询没改写？**

三个可能：
1. **Chunk 太大**——如果文档按大段落切分（如一整章作为一个 chunk），一个 chunk 里同时含"50 万"和"100 万"的规则，embedding 混淆，检索时两者都召回。解法是按"规则条目"切分（每条规则一个 chunk，含金额/审批人/条件），精准。
2. **Embedding 模型差**——通用 embedding 模型（如 text-embedding-ada-002）对数字不敏感（50 和 100 的语义向量接近），检索混淆。解法是用专门微调的 embedding 模型（或在 query 里强化数字"金额大于 100 万"）。
3. **查询没改写**——用户问"超 100 万审批"，原始 query 直接 embedding 可能召回不准。解法是查询改写（LLM 先把问题改写为"采购金额超过 100 万元的审批流程和审批人"），更精准。
根因定位：看 chunk 大小（一个 chunk 含多少规则）、embedding 模型、查询改写，针对性优化。

**Q：那为什么不直接用关键词检索（Elasticsearch 搜"100 万 审批"），确定性更高？**

关键词检索适合"精确匹配"（如搜"100 万"这个字面词），但有局限：
1. **语义理解差**——用户问"大额采购要谁批"（没说具体金额），关键词检索搜不到（没"100 万"字面）。向量检索能理解"大额"语义，召回"100 万审批"文档。
2. **同义词**——文档写"百万级采购"，用户问"100 万"，关键词不匹配。向量能识别同义。
3. **组合查询**——用户问"服饰类目超 100 万且紧急的采购审批"，关键词检索难处理多条件组合，向量检索能综合语义。
所以最佳实践是"混合检索"（Hybrid Search）：关键词检索（精确匹配金额/编号）+ 向量检索（语义匹配），两者结果融合重排序。纯关键词漏语义，纯向量漏精确，混合最优。

### 第四层：方案权衡

**Q：RAG 的知识库要频繁更新（采购制度每季度改），你用重新 embedding 全量文档的策略。但全量重建慢（几小时），期间 RAG 服务不可用。怎么权衡"知识更新及时"和"服务可用"？**

增量更新 + 版本切换：
1. **增量索引**——向量库（如 Milvus/Qdrant）支持增量插入，只 embedding 变更的文档（不是全量），分钟级完成。前提是文档有版本标记，能 diff 出变更部分。
2. **双索引切换**——重建时新建一个索引（v2），旧索引（v1）继续服务。v2 建好后，原子切换（别名指向 v2），零停机。类似 ES 的 reindex + alias 切换。
3. **定期重建 + 增量补充**——每季度全量重建一次（保证索引健康），平时增量更新（单文档变更实时 embedding）。两者结合。

**Q：为什么不直接用微调（fine-tune）把规则知识写进模型参数，不用外挂 RAG？**

微调和 RAG 适用不同场景：
1. **微调适合"固定模式"**——如让模型学会供应链领域的术语（SKU/SPU/补货），这些是稳定的语言模式，微调让模型"懂行话"。
2. **RAG 适合"频繁变知识"**——审批规则/价格政策每季度变，微调要重训（成本高、周期长），RAG 更新向量库即可（秒级）。
3. **组合最优**——微调让模型懂供应链语义（提升检索和理解准确率）+ RAG 外挂具体规则知识（实时更新）。拼多多供应链助手就是微调基座（理解力强）+ RAG（规则新）。
所以不是二选一，是"微调基座 + RAG 知识"组合。纯微调知识更新难，纯 RAG 语义理解弱。

### 第五层：验证与沉淀

**Q：你怎么证明 RAG 系统的回答准确、运营信任它？**

多维评估：
1. **检索准确率**——人工标注 1000 个问题的"正确文档"，RAG 召回的 Top-5 里含正确文档的比例（Recall@5），应 > 95%。低于阈值优化 embedding/chunk/检索策略。
2. **回答准确率**——人工评估 1000 个回答的"正确性 + 完整性 + 是否引用证据"，准确率应 > 90%。LLM 回答必须带 evidence 引用（文档名 + 章节），便于人工核验。
3. **用户满意度**——运营给回答打分（thumbs up/down），满意度应 > 85%。低分回答分析原因（检索错/理解错/文档缺），持续优化。
4. **A/B 测试**——RAG 助手 vs 人工客服，对比"回答准确率 + 响应时间"。RAG 应准确率接近人工 + 速度快 10 倍。

**Q：怎么让团队的 RAG 应用质量可持续提升（不一次性上线就不管）？**

闭环机制：
1. **反馈闭环**——用户对回答的反馈（thumbs down + 修正答案）收集，用于优化（重新 embedding 错误文档 + 微调 prompt）。每周分析 bad case。
2. **知识库治理**——知识库文档定期 review（过期文档更新 + 缺失文档补充），专人维护（供应链各领域的 SME 负责本领域文档）。
3. **评估自动化**——维护测试集（1000 个标注的 Q&A），RAG 任何变更（embedding 模型/chunk 策略/prompt）跑测试集，准确率退化阻断上线。
4. **持续优化**——按季度评估 RAG 效果，引入新技术（如更好的 embedding 模型、重排序、查询改写），持续提升准确率。

## 结构化回答

**30 秒电梯演讲：** LLM 不知道供应链私有知识（规则/流程/案例），如何让 LLM 基于内部知识回答？简单说就是——RAG 让 LLM 基于供应链私有知识回答——检索商品规则/采购流程/历史案例拼到 prompt，LLM 基于事实生成，解决 LLM 不知道内部业务的问题。

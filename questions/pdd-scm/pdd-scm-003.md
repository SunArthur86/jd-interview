# 【拼多多供应链】DDD 怎么落地？供应链的领域模型怎么设计？

> JD 依据："应用软件设计原则、设计模式、领域驱动等设计和研发高质量的业务系统"。

## 一、为什么供应链需要 DDD

供应链的复杂性：
- **概念交叉**："商品"在采购=待采品，在仓储=SKU，在销售=SPU，在结算=计价品
- **业务规则多**：下单扣库存、退款回库存、预售锁库存
- **跨团队协作**：商品团队、库存团队、订单团队各自维护系统

不用 DDD 的问题：所有逻辑堆在 Service 层，概念混乱，改一处影响全局。

## 二、战略设计：限界上下文

```
供应链域（子域划分）
├─ 商品上下文（核心）：SPU/SKU/类目/属性
├─ 库存上下文（核心）：现货/预售/在途/冻结
├─ 采购上下文（支撑）：采购单/供应商/到货
├─ 仓储上下文（支撑）：入库/出库/调拨
├─ 订单上下文（核心）：订单/订单项/状态机
├─ 结算上下文（支撑）：对账/发票/付款
└─ 物流上下文（支撑）：运单/轨迹/签收
```

**上下文映射**（关系）：
- 订单 → 库存：订单创建触发库存扣减（领域事件）
- 采购 → 仓储：采购到货触发出库（领域事件）
- 商品 ← 订单：订单引用商品 SKU（防腐层 ACL 转换）

## 三、战术设计：聚合根

**聚合根 = 一致性边界**。一个聚合内的修改必须原子，跨聚合用最终一致。

```
订单聚合根
├─ Order（聚合根，实体）
│   ├─ orderId（值对象）
│   ├─ status（枚举：待支付/已支付/已发货/已完成）
│   └─ 地址（值对象）
├─ OrderItem（实体，属于订单聚合）
│   ├─ skuId
│   ├─ quantity
│   └─ price
└─ 业务规则：订单总金额 = Σ 订单项金额；状态变更必须按状态机

商品聚合根（独立聚合）
├─ Product（聚合根）
│   ├─ spuId
│   ├─ skus[]  （实体）
│   └─ 属性（值对象）
```

**关键原则**：
- 外部只能持有聚合根引用（`orderId`），不能直接持有 `OrderItem`
- 修改聚合内的操作必须通过聚合根方法（`order.addItem()` 而非 `orderItem.setQty()`）

## 四、领域事件解耦

```java
// 订单聚合根发布事件
public class Order extends AggregateRoot {
    public void pay() {
        this.status = PAID;
        // 发布领域事件（同事务写入本地事件表）
        DomainEvents.publish(new OrderPaidEvent(this.orderId, this.items));
    }
}

// 库存上下文订阅（异步，最终一致）
@EventHandler
public class StockEventHandler {
    public void on(OrderPaidEvent event) {
        for (Item item : event.getItems()) {
            stockService.deduct(item.getSkuId(), item.getQuantity());
        }
    }
}
```

**落地**：本地消息表 + MQ（保证事件不丢）：
```java
@Transactional
public void pay(Long orderId) {
    orderDao.updateStatus(orderId, PAID);
    outboxDao.insert(new Outbox("OrderPaidEvent", payload));  // 同事务写事件表
}
// 定时扫 outbox 投递 MQ，成功删除
```

## 五、供应链 DDD 落地的难点

1. **聚合粒度难定**：订单+订单项是一个聚合，但订单+物流是一个吗？（通常拆开，物流独立聚合）
2. **跨聚合事务**：下单扣库存不能强一致（跨服务），用Saga/TCC补偿
3. **历史系统改造**：从贫血模型到充血模型要重构，建议新业务先试点
4. **团队认知**：DDD 学习曲线陡，需要培训

## 六、和传统三层的对比

| 维度 | 三层架构 | DDD |
|------|---------|-----|
| 领域逻辑 | 堆在 Service | 抽到 Domain 层（充血模型）|
| Service | 胖（业务+技术） | 瘦（只编排，无业务）|
| 一致性 | 事务包一大堆 | 聚合内强一致，跨聚合最终一致 |
| 演进 | 改一处影响全局 | 上下文独立演进 |

## 七、底层本质：复杂度的"分而治之"

DDD 本质是**"用业务语言驱动技术建模"**：
- 业务方说"下单扣库存" → 技术方建模为"订单聚合发布 OrderPaidEvent，库存聚合订阅扣减"
- 概念有边界（限界上下文）
- 一致性有边界（聚合根）
- 协作有契约（领域事件）

**这是软件工程"关注点分离"在业务建模的体现**——让复杂的供应链系统像乐高积木一样，每块（上下文）职责清晰、接口明确。

## 常见考点
1. **聚合和聚合根区别**？——聚合是一组相关对象（一致性边界），聚合根是聚合的入口实体。
2. **DDD 一定比三层架构好吗**？——不是。简单 CRUD 用 DDD 过度设计；复杂业务领域才值得。
3. **领域事件怎么保证不丢**？——本地消息表（业务+事件同事务）+ MQ + 消费幂等。

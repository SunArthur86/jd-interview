# 【拼多多供应链】设计库存中心（亿 SKU、高并发扣减、防超卖）

> JD 依据："高并发系统开发"。

## 一、库存模型

```
SKU 维度库存:
  可售库存 = 现货库存 - 冻结库存 + 预售库存
  在途库存 = 采购未到货
```

| 类型 | 含义 |
|------|------|
| 现货 | 仓库实际有 |
| 冻结 | 已下单未发货 |
| 预售 | 按产能可超卖 |
| 在途 | 采购在途 |

## 二、整体架构

```
扣减请求 → Redis（Lua 原子扣减）
              ↓ 成功
           MQ（异步）→ DB（持久）+ 库存日志
              ↓
           定时对账（Redis vs DB）
```

## 三、Redis 原子扣减（Lua）

```lua
local stock = tonumber(redis.call("get", KEYS[1]))
if stock and stock >= tonumber(ARGV[1]) then
    redis.call("decrby", KEYS[1], ARGV[1])
    return 1  -- 成功
end
return 0  -- 库存不足
```

Lua 保证"判断+扣减"原子，防超卖。

## 四、热点 SKU 分桶

某爆款秒杀，单 key 被万级并发打爆。**分桶**：
```
SKU=1，库存 1000
  → 拆 10 桶：sku:1:bucket:0..9，各 100
  → 扣减按 uid hash 路由桶
  → 桶不足时从其他桶借调
```

## 五、防超卖的多重保障

1. **Redis Lua**：原子判断+扣减
2. **DB 兜底**：下单时 `UPDATE stock SET qty=qty-1 WHERE qty>=1`（影响行数 0 则失败）
3. **超时回滚**：下单未支付超时，归还冻结库存

## 六、底层本质

库存中心本质是**"高并发下的计数正确性"**：
- Redis 保证实时性（快）+ 原子性（不超卖）
- DB 保证持久性（权威）
- 分桶解决热点（ scalability）

**和银行账户系统同构**——都是"并发记账 + 对账保证一致"。

## 常见考点
1. **Redis 挂了库存怎么办**？——Redis 持久化（AOF）+ DB 兜底校验 + 快速重启从 DB 恢复。
2. **分桶的桶不够怎么办**——动态合并/借调，或单桶直接走 DB（牺牲性能保正确）。
3. **库存负数怎么防**——Lua 扣减前判断 + DB 约束 `CHECK qty >= 0`。

## 苏格拉底式面试追问

> 这组追问不是背答案，而是模拟面试官层层逼近本质。每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：库存中心你设计了"Redis 实时扣减 + DB 持久 + 定时对账"三层。为什么不直接用 DB 扣减（UPDATE stock SET qty=qty-1 WHERE qty>=1），DB 本身就能保证原子性和防超卖？**

DB 单点扛不住大促并发。DB 乐观锁对热点 SKU 的行锁串行化——万级并发扣同一 SKU，999 个等行锁，QPS 被压到几百，大促直接超时。Redis 是内存操作，单实例 10 万 QPS，Lua 脚本保证原子性。所以 Redis 做实时扣减（性能层），DB 做持久化（兜底层），对账保证最终一致。DB 直接扣减只在"低并发场景"（如 B2B 批量采购）够用，C 端大促必须 Redis 前置。

### 第二层：证据与定位

**Q：大促时出现超卖（库存变负数），用户下单成功但发不出货。你怎么定位是 Redis Lua 没生效、分桶逻辑 bug，还是 DB 同步丢数据？**

三段排查：
1. **看 Redis 库存值**——`GET stock:1001`，如果是负数，是 Lua 扣减逻辑 bug（`stock >= qty` 判断没生效）。`SCAN stock:1001:*` 看分桶，如果某桶负数，是分桶借调逻辑 bug。
2. **看超卖计数**——`GET oversell_count:1001:{date}`，埋点统计超卖次数。如果大促期间这个计数 > 0，确实超卖了。
3. **看扣减日志**——`stock_log` 表按 `sku_id=1001` 查扣减记录，`SUM(qty)` 对比 Redis 的初始库存，差值就是超卖量。看是哪几个 order_id 的扣减导致超卖，它们的来源（秒杀活动？正常下单？）。

### 第三层：根因深挖

**Q：Redis 库存值是负数（-5），Lua 脚本里明明有 `if stock >= qty then` 的判断。根因是什么？**

几种可能：
1. **分桶借调竞态**——分桶扣减时，某桶不足要从其他桶借调。如果借调逻辑不是原子的（先查其他桶余额，再转账），两个线程同时借调同一桶，都看到余额 100，各借 80，实际只有 100，超借。解法：借调也用 Lua 原子操作（整个"查+扣"在一个 Lua 脚本）。
2. **Lua 脚本被改了**——上线时 Lua 脚本更新，新版本漏了 `stock >= qty` 判断（如改成 `stock >= 0`）。`SCRIPT GET <sha>` 看当前运行的脚本内容。
3. **库存初始化错**——大促前预热库存时，设了错误的初始值（如本应 1000 设成 100），扣到负数是"正常扣减但初始值错"。看库存初始化日志。
根因定位：把 Lua 脚本的执行日志（Redis 的 `MONITOR`）拉出来，看超卖时刻的扣减序列，反推是哪一步漏了判断。

**Q：那为什么不直接给 Redis 的 key 加一个 lua 守护（定期检查 `stock < 0` 就告警冻结扣减），最快止损？**

告警冻结是临时止血，但不解决根因。而且"定期检查"有窗口期——两次检查之间已经超卖了。正确的止损是"发现超卖立即停止该 SKU 的扣减"（下架该 SKU），然后排查根因。守护脚本（cron 每秒检查 `stock:* < 0`）是补充手段，但不能替代 Lua 逻辑的正确性。根治要修 Lua 脚本的判断逻辑，确保 `stock >= qty` 一定生效（加单元测试覆盖边界值）。

### 第四层：方案权衡

**Q：热点 SKU 分桶你按 uid hash 路由，但如果某桶（bucket_0）先扣完，其他桶还有余，用户被路由到 bucket_0 会失败。怎么权衡"分桶均匀"和"用户体验"？**

分桶的核心矛盾是"分布均匀（性能）vs 跨桶借调（正确性）"。三种策略：
1. **固定桶（简单但体验差）**——用户永远路由到固定桶（uid hash），桶空了就失败，即使其他桶有货。用户体验差（"明明有货我买不到"）。
2. **跨桶借调（复杂但体验好）**——桶空时自动从其他桶借调（Lua 内实现"查其他桶余额 + 转账"）。用户无感知，但 Lua 逻辑复杂，借调竞态要小心（全用 Lua 原子）。
3. **动态合桶（折中）**——当某桶余量 < 阈值（如 < 10），把该桶的剩余库存合并到下一个桶，标记该桶"已售罄"，后续路由跳过它。减少借调频率。
推荐方案 2（跨桶借调），因为用户体验是电商核心，Lua 原子性能也能接受。拼多多秒杀就是跨桶借调，单 SKU 扛百万 QPS。

**Q：为什么不直接用 Redis Cluster 的自动分片，让 Cluster 把热点 key 分散到多节点？**

Redis Cluster 的分片是"按 key 的 hash 分到固定 slot"，一个 key 只在一个 slot（一个节点）。热点 key（如 `stock:1001`）永远在一个节点，Cluster 不会自动分散（因为分散就破坏了 key 的原子性——一个 key 不能跨节点原子操作）。所以"业务层分桶"（`stock:1001_0` 到 `stock:1001_9`，10 个 key 自然分布到不同 slot）是唯一解，Cluster 帮不上。Cluster 的价值是"整体容量和吞吐扩展"（加节点），不是"单热点 key 分散"。

### 第五层：验证与沉淀

**Q：你怎么证明库存中心的防超卖方案在大促时真的生效、没有超卖事故？**

三层验证：
1. **超卖计数器**——Lua 扣减时如果 `stock < 0`，`INCR oversell_count:{sku_id}:{date}`，每天看这个指标必须 = 0。大促后跑全量对账 `SELECT SUM(qty) FROM stock_log WHERE sku_id=X` 对比库存，偏差 = 0。
2. **混沌演练**——压测环境造万级并发扣同一 SKU（库存 1000，10000 个请求各扣 1），最终 Redis 库存必须 = 0 而不是负数；分桶场景测"某桶空后借调"，验证不超卖。
3. **对账巡检**——每小时跑 Redis vs DB 对账（`redis_stock - db_stock`），差值应 = 0（最终一致）；差值 > 0 告警，可能是 MQ 消费延迟（暂态）或丢数据（故障）。

**Q：怎么让团队维护库存扣减的正确性，不再引入超卖 bug？**

沉淀三道防线：
1. **扣减 SDK 统一**——封装 `StockClient.deduct(skuId, qty, orderId)`，内部强制走 Lua（含幂等 + 分桶 + 借调），业务方调 SDK 不能绕过去手写 SQL 或 Lua。SDK 有完整的单元测试（覆盖边界值、并发场景）。
2. **Code Review 红线**——任何库存相关的 SQL/Lua 修改必须架构师 + DBA 双人 review，重点看"判断逻辑（>=）"和"原子性（是否在事务/Lua 内）"。
3. **大盘监控**——`stock_deduct_fail_rate`（扣减失败率）、`oversell_count`（超卖数）、`redis_db_diff`（Redis 与 DB 差），任一异常飞书告警值班，大促期间 24 小时值守。
